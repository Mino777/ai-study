## 아쉬운 것

### MDX 파싱 에러 1건 — 사전 체크 안 돌림

`pg-trgm-gin-index-for-like-search.mdx` 작성 시 `**CREATE INDEX USING gin (<col> gin_trgm_ops)**` 에서 `<col>` 이 HTML void element로 해석돼 빌드 실패.

솔루션 `docs/solutions/mdx/2026-04-12-jsx-parsing-traps-curly-angle.md` 에 `<숫자` / `{...}` 패턴은 있었지만 **`<소문자>` (void element) 패턴은 미박제**. 오늘 패턴 #3 으로 보강.

**교훈**: 솔루션 문서는 과거 사례 커버리지만큼만 보호한다. 새 패턴 만나면 바로 확장.

### 연결 보강을 박제 후 따로 처리

10 엔트리 박제 먼저 → 5 커밋 만든 뒤 사용자 피드백 "기존 문서도 같이 업데이트" 받고 연결 보강 진행. 다음 번엔 박제 시점에 동시에 connections 보강하면 커밋이 주제별로 더 깔끔.

다만, 10 엔트리 묶음에선 연결 그래프가 복잡해서 **한 번에 쓰기 어렵다**는 반대 주장도 있음. 박제 → 보강 2 사이클이 실용적일 수 있음.
