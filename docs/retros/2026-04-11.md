## 잘된 것

### ✅ 한 흐름으로 R1→R4 끝까지 달렸다
중간에 끊지 않고 schema → 컴포넌트 → 샘플 데이터 → 대시보드 → SRS → 파이프라인 통합 → 문서까지 한 번에. 매 라운드 끝날 때마다 `npm run build`로 검증했기 때문에 되돌릴 일이 없었음.

### ✅ 손으로 짠 YAML escape를 버리고 js-yaml로 전환
초기 `yamlEscape()` 함수가 따옴표/백슬래시 엣지 케이스에서 깨진 걸 로컬 roundtrip 테스트에서 바로 포착 → 즉시 `js-yaml`로 교체. 만약 Gemini API 호출하고 PR 만들어진 뒤에 발견했으면 훨씬 비싼 실수였을 것.

### ✅ 사전 validation이 빌드를 지켰다
`scripts/validate-content.mjs`가 Mermaid subgraph 에러를 push 직전 rebuild에서 잡아줌. 이 검증 스크립트 자체가 **이전 compound 스프린트의 자산**이었음 — 복리 효과가 실제로 발생한 순간.

### ✅ localStorage만으로 SRS 구현
서버 인증·DB 없이 클라이언트만으로 학습 루프가 완결됨. 미래에 로그인/동기화 추가해도 같은 데이터 구조 그대로 확장 가능.

## 잘된 것

### ✅ 사고 → 시스템화 → dogfooding 사이클을 *같은 날* 클로즈
보통 사고 이후 "다음에 조심해야지"로 끝나는데, 오늘은 사고 → 메모리 박제 → 슬래시 커맨드 게이트화 → 첫 적용까지 한 세션 안에 끝냈다. 두 번째 엔트리에서 같은 사고가 0건이었다는 것이 회로가 닫혔다는 증거.

### ✅ 메모리와 슬래시 커맨드의 *이중 박제*
같은 규칙을 두 곳(Claude 메모리 + `.claude/commands/`)에 두는 게 처음에는 중복처럼 보였지만, 둘은 본질적으로 다른 자산이다:
- **메모리** = Claude의 기억(개인). 새 세션에 자동 로드.
- **슬래시 커맨드** = 프로젝트 파일(공적). 사람이 의식적으로 호출.

한쪽이 빠져도 다른 쪽이 잡아준다. 특히 슬래시 커맨드는 git에 들어가므로 *영구 자산* — 메모리와 달리 사라지지 않는다.

### ✅ Compound Engineering 엔트리에 검증 메모를 본문 자체에 남김
"어떤 도구로 무엇을 검증했는지"를 엔트리 본문 끝에 남기니, 미래에 누가 봐도 신뢰도를 직접 평가할 수 있다. 이 패턴 자체가 다음 엔트리에도 강제될 만한 가치가 있음.

### ✅ 가장 큰 메타 학습: "인용구는 권위가 자동 부여되는 형식이라 AI가 편향된다"
