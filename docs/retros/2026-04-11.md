## 잘된 것

### ✅ 한 흐름으로 R1→R4 끝까지 달렸다
중간에 끊지 않고 schema → 컴포넌트 → 샘플 데이터 → 대시보드 → SRS → 파이프라인 통합 → 문서까지 한 번에. 매 라운드 끝날 때마다 `npm run build`로 검증했기 때문에 되돌릴 일이 없었음.

### ✅ 손으로 짠 YAML escape를 버리고 js-yaml로 전환
초기 `yamlEscape()` 함수가 따옴표/백슬래시 엣지 케이스에서 깨진 걸 로컬 roundtrip 테스트에서 바로 포착 → 즉시 `js-yaml`로 교체. 만약 Gemini API 호출하고 PR 만들어진 뒤에 발견했으면 훨씬 비싼 실수였을 것.

### ✅ 사전 validation이 빌드를 지켰다
`scripts/validate-content.mjs`가 Mermaid subgraph 에러를 push 직전 rebuild에서 잡아줌. 이 검증 스크립트 자체가 **이전 compound 스프린트의 자산**이었음 — 복리 효과가 실제로 발생한 순간.

### ✅ localStorage만으로 SRS 구현
서버 인증·DB 없이 클라이언트만으로 학습 루프가 완결됨. 미래에 로그인/동기화 추가해도 같은 데이터 구조 그대로 확장 가능.

## 아쉬운 것

### ⚠️ 같은 날 AI 생성 콘텐츠 이슈 2건 연속 발생
- `context-compression.mdx` — Mermaid subgraph 공백 (로컬 rebuild가 잡아줌)
- `agent-architectures.mdx` — `<br>` 태그 (Vercel 배포 실패로 잡힘, 2026-04-09 솔루션 재발)

**뼈아픔**: 2026-04-09에 이미 `br-tag-compile-error` 솔루션 문서를 썼지만, 그 지식이 **Gemini 프롬프트에는 반영되지 않았다**. 솔루션 문서는 "Claude가 다음에 볼 때 알게 되는" 정적 자산인데, AI 자동 파이프라인에 피드백으로 들어가지 않으면 같은 실수를 반복한다.

### ⚠️ Vercel 배포 실패를 사용자가 먼저 알림
로컬 빌드는 통과(17 → 18 엔트리 기준)했지만, push 후 원격에 새 엔트리(`agent-architectures`)가 하나 더 들어왔고 그건 내가 rebuild하지 않았다. 사용자의 "배포 실패했는데?" 메시지로 비로소 알아챔.

→ **교훈**: `git pull --rebase` 이후에는 반드시 rebuild. 특히 AI 자동 생성 파이프라인이 돌아가는 프로젝트에서는 "내가 push하지 않은 콘텐츠"가 언제든 끼어들 수 있음.

### ⚠️ 테스트 부족
Quiz 컴포넌트·SRS 로직·YAML 직렬화 모두 단위 테스트 없음. Roundtrip은 수동으로 일회성으로만 확인. 다음 스프린트에 테스트 커버리지 향상(P3) 올려야 함.
