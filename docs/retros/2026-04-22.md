## 잘된 것

- **문제 발견 전 사전 설계**: worktree 격리는 사용자 질문("git worktree 사용하는거 맞지?") 이전에 병렬 git index 오염 문제를 먼저 분석하고 설계. 사용자가 알기 전에 이미 구조 제안.
- **bash 3.2 호환 버그 즉시 탐지**: `${$(cmd)}` macOS bash 비호환 → 코드 작성 직후 실행 없이 문법 분석으로 발견. 중간 변수로 수정.
- **단방향→양방향 통신 완전성**: `notify_architect()` + `watch_inbox()` 조합으로 워커가 아키텍트를 폴링 없이 깨울 수 있는 완전한 푸시 구조 완성.
- **토큰 절감 근거 제시**: "많이 클까?"라는 막연한 질문에 파일별 토큰 수 측정 + 워커 수 × 스프린트 수 추정치로 구체적 수치 응답.

## 아쉬운 것

- **`pane-base-index 1` 환경 대응을 사후에**: `tmux-setup-full`이 `main.0/.1/.2/.3` 인덱스를 가정했는데, `pane-base-index 1` 환경에서 오동작. 초기 설계 시 `pane-base-index` 환경 변수를 확인했어야 했으나 확인 안 함. 픽스 커밋이 별도로 필요했음.
- **`create_worker_worktree` 출력 오염 미예측**: 함수가 상태 메시지와 경로를 같은 stdout으로 출력 → 호출부에서 `| tail -1` 패치. 함수 설계 시 stdout/stderr 분리 원칙을 처음부터 적용했어야 했음.
- **통합 테스트 없음**: 7개 기능 추가 후 실제 `tmux-setup-full` + `dispatch-layer` + `watch-layer` 전체 플로우를 시뮬레이션하지 않음. 각 함수는 코드 리뷰로 검증했지만 통합 실행은 사용자 환경에 의존.

## Anti-Rationalization 체크

- **판단 실수**: `pane-base-index` 환경 차이를 "동작할 것"이라고 추정한 것이 판단 실수. 외부 환경 의존 코드는 조건 확인 후 작성해야.
- **테스트 없이 추정**: `create_worker_worktree | tail -1` 패치도 실행 없이 추정. 스크립트 함수는 로컬 실행 1회로 확인 가능했음.
- **자기 귀인**: "예상보다 환경이 달라서" 아님 → "pane-index 환경 다양성을 추정하지 않고 작성한 내 판단 실수".

## 다음에 적용할 것

1. **tmux 함수 작성 시 `pane-base-index` 체크 먼저**: `tmux show-options -g pane-base-index` 결과 분기.
2. **bash 함수 stdout 규약**: 경로/결과 값은 `>&2`로 로그, stdout은 caller가 캡처할 값만. `create_worker_worktree` 패턴 반복 방지.
3. **dispatch 후 통합 시나리오 1회 드라이런**: 실제 tmux 없어도 각 단계 입출력을 순서대로 검증하는 간단한 스크립트 작성.
