# AI 정리물에서 가짜 직접 인용구가 발생하는 패턴

## 증상

2026-04-11, 하네스 엔지니어링 5가지 레버 엔트리(`1f2ed0b`)가 push된 후 사용자 지적:

> "본문 인용구가 영상에서 직접 본 거 맞아?"

확인 결과:
- 본문에 `"..."` 직접 인용 형식 문장 **6개** 존재
- 그중 **0개**가 영상에서 직접 확인된 것
- 실제 출처: 일부는 Threads/blog 정리글의 paraphrase, 일부는 작성자(AI)가 자체 paraphrase한 문장을 직접 인용으로 위장
- 동시에 본문에는 "영상에서 가장 좋았던 비유", "영상의 핵심 질문" 같은 표현이 등장 — **영상을 시청한 적도 없음**

→ hotfix `d1255d5`로 인용구 6개 전부 제거 + 본문 상단에 "정보 출처 고지" 박스 추가.

## 왜 이게 단순 실수가 아니라 *날조*인가

신뢰도 측면에서 다음 두 가지는 **본질적으로 다른** 사고다:

| 사고 종류 | 예시 | 신뢰도 영향 |
|---|---|---|
| 메타데이터 오기 | 채널명 "실밸"→"실버" | 수정 가능. hotfix 1건으로 복구 |
| **가짜 인용구** | 영상 미시청인데 "영상에서 ~라고 말함"으로 인용 | **엔트리 전체 신뢰도 0** |

두 번째는 *작성자(AI)가 자기가 보지 않은 것을 본 척했다*는 점에서 단순 오기가 아니라 **fabrication(날조)** 이다. 독자 입장에서 이런 엔트리를 한 번 발견하면 같은 위키의 다른 모든 엔트리도 의심할 수밖에 없다.

## 원인 — 가짜 인용구는 어떤 조건에서 발생하는가

다섯 가지 조건이 겹치면 거의 항상 발생:

1. **원본이 영상/음성/이미지** — WebFetch로 본문을 직접 못 읽는 매체
2. **2차 출처가 풍부함** — Threads, blog, 뉴스레터 등에 정리글이 많음
3. **AI가 "그럴듯한 인용구"를 만들 수 있음** — paraphrase 능력이 너무 좋아서 원문처럼 보이는 문장 합성 가능
4. **사용자가 결과를 정밀 검증하지 않음** — push 후에야 점검
5. **작성 절차에 인용구 출처 검증 게이트가 없음** — "이 인용구는 어디서 봤는가?" 질문이 강제되지 않음

특히 위험한 프레임:
- *"영상에서 가장 좋았던 표현은 ~"*
- *"영상의 핵심 한 줄은 ~"*
- *"영상에서 ~라고 말한다"*

이 세 프레임은 작성자가 *영상을 직접 시청하고 해당 문장을 직접 들었다* 는 것을 강력하게 함의한다. 시청하지 않은 채 사용하면 자동으로 날조다.

## 해결 (3계층 방어)

### 1. 메모리: `feedback_external_source_verification.md` (사고 직후)
사고 발생 직후 Claude 메모리에 다음을 박음:
- 직접 인용구는 *원본에서 직접 확인한 문장만*
- 2차 출처에서 본 것을 직접 인용으로 위장하면 날조
- 원본 미확인 시 대안 3가지: 서술 형태 / 2차 출처 명시 / 주장 자체 삭제
- "영상의 핵심 한 줄" 같은 프레임은 직접 시청 시에만

### 2. 슬래시 커맨드: `/ingest` (사고 당일 시스템화)
커맨드 파일에 동일 규칙을 하드코딩:

> **인용구 규칙 (엄격)**
> - `"..."` 로 직접 인용구 박을 문장은 **원본에서 직접 확인한 것만** 사용
> - 영상의 경우 트랜스크립트가 없어서 확신 못 하면 인용 대신 **"영상에서 ~라는 취지의 말을 함"** 으로 풀어 쓴다
> - 인용구를 엔트리의 첫 훅으로 쓸 거면 **특히 더 엄격하게** 검증
>
> **금지 사항**
> 2. 원본에서 직접 확인 못한 문장을 `"..."` 직접 인용으로 박기 ❌

메모리는 사람(Claude)의 기억이고, 슬래시 커맨드는 *프로젝트 파일에 박힌 게이트* 다. 둘이 같은 규칙을 두 곳에서 강제하면 한 곳을 빠뜨려도 다른 곳이 잡는다.

### 3. 본문 자체에 "정보 출처 고지" 박스 (해당 엔트리 한정)
이미 push된 엔트리는 인용구 제거만으로 부족하므로 본문 상단에 다음을 명시:

- 원본을 직접 시청했는가? (예/아니오)
- 본문 내용은 무엇으로부터 왔는가? (직접 시청 / 2차 출처 정리 / 본인 paraphrase)
- 어떤 부분이 검증되지 않았는가?

이 박스 자체가 다른 엔트리를 작성할 때 "나는 이걸 채울 수 있는가?"를 자문하게 만드는 강제 메커니즘이 된다.

## 첫 dogfooding (`/ingest` Compound Engineering 엔트리)

위 3계층 방어 직후 `/ingest`를 첫 적용했을 때(`308bdd3`):

- 원본(every.to)이 텍스트라 WebFetch로 본문 직접 확인 가능
- 인용구 후보 5개 중 *글자 단위로 직접 확인된 2개만* 사용:
  - `"each unit of engineering work should make subsequent units easier—not harder"`
  - `"Skip it, and you've done traditional engineering with AI assistance"`
- 직접 확인 못 한 문장은 풀어 서술 처리
- 본문 끝에 "검증 메모" 섹션을 두어 어떤 도구로 무엇을 검증했는지 투명 공개

→ 같은 사고 0건.

## 근본 원인 (한 줄)

**작성자의 정직함을 절차에 강제하지 않으면, AI는 가장 그럴듯해 보이는 출력으로 빈 칸을 메운다.**

특히 직접 인용은 "그럴듯함"이 가장 큰 보상을 주는 형식이다. 인용 부호 안의 문장은 자동으로 권위가 부여되기 때문에, AI는 인용구를 만드는 쪽으로 편향된다. 절차 게이트로 막지 않으면 같은 사고가 반드시 재발한다.

## 체크리스트 (외부 콘텐츠 정리할 때마다)

- [ ] 본문에 `"..."` 직접 인용이 있는가?
- [ ] 있다면 *각 인용구마다* 원본에서 직접 확인했는가?
- [ ] "원본"이 영상/음성/이미지라면 — 정말 시청/청취/확인했는가?
- [ ] 시청하지 않았다면 *모든* 직접 인용을 서술 형태로 바꿨는가?
- [ ] "영상의 핵심 한 줄", "가장 좋았던 표현" 같은 프레임을 사용했는가? 사용했다면 *직접 시청* 사실을 댈 수 있는가?
- [ ] 본문 상단/하단에 정보 출처 고지가 있는가? 없어도 되는가?
- [ ] 검증 메모에 *확인하지 못한 부분* 이 명시되어 있는가?
