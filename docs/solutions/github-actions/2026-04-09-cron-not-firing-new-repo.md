# GitHub Actions cron이 새 레포에서 실행되지 않는 문제

## 증상

새로 생성한 레포지토리에 `daily-lesson.yml` 워크플로우를 추가했고, schedule trigger를 `cron: "0 0 * * *"` 로 설정했음. 하지만 예약된 시간에 자동으로 실행되지 않고, 7번의 모든 실행이 workflow_dispatch(수동 트리거)로만 기록됨.

- 레포 생성일: 2026-04-07
- 워크플로우 추가일: 2026-04-07 (같은 날)
- 워크플로우 상태: "active" (비활성화 아님)
- Fork 여부: 원본 레포 (fork 아님)
- Secrets: 설정됨
- 최근 push: 존재함

하지만 cron schedule trigger는 자동으로 실행되지 않음.

## 원인

**GitHub Actions의 schedule trigger는 새로 생성된 레포지토리에서 활성화되는 데 24~48시간이 소요됨.** 이것은 GitHub의 알려진 동작 방식(known behavior)이다.

새 레포에서:
1. 워크플로우 파일을 푸시했을 때
2. GitHub는 즉시 스케줄러를 등록하지 않음
3. 내부 스케줄링 시스템이 새 레포를 인식하고 등록할 때까지 대기 필요
4. 이 과정에서 24~48시간 지연 발생

## 해결

1. **cron 표현식 변경**: `"0 0 * * *"` → `"5 0 * * *"` (1분 시간 변경)
   - 이렇게 하면 GitHub이 push 시점에 스케줄러를 강제로 다시 등록함
   - 실제 실행 시간이 자정에서 자정 5분으로 변경되지만, 스케줄이 활성화됨

2. **수동 실행**: 변경 후 workflow_dispatch로 즉시 테스트하여 문제 상황 확인

3. **결과**: cron 변경 후 자동 실행이 정상 동작함을 확인

## 근본 원인

GitHub Actions의 스케줄 시스템은 분산 인프라로 구성되어 있으며, 새 레포지토리가 모든 스케줄링 노드에 등록되는 데 시간이 걸림. 이는:

- **새 레포의 콜드스타트 문제**: GitHub 내부의 시간 기반 스케줄러가 새로운 레포지토리를 모든 스케줄링 워커에 동기화하는 과정
- **설정 푸시 딜레이**: 워크플로우 파일을 푸시한 즉시 모든 스케줄러가 인식하지 못함
- **스케줄 재등록 필요**: cron 표현식을 변경하면 GitHub이 스케줄러를 우선적으로 갱신함

따라서 단순히 시간만 변경하면 문제가 해결되는 것처럼 보이지만, 실제로는 **GitHub의 스케줄 재등록 메커니즘을 트리거하는 것**.

## 체크리스트

새 레포에 예약된 워크플로우를 추가할 때:

- [ ] 워크플로우 파일을 main 브랜치에 푸시
- [ ] **즉시 workflow_dispatch로 수동 테스트** (스케줄링이 활성화되었는지 확인)
- [ ] 48시간 대기
- [ ] **48시간 후에도 cron이 실행되지 않으면**, cron 표현식을 약간 변경하고 push
  - 예: `"0 0 * * *"` → `"5 0 * * *"`
  - 예: `"0 0 * * *"` → `"1 0 * * *"`
- [ ] 변경 후 다시 수동 workflow_dispatch 실행해서 동작 확인
- [ ] 다음 스케줄 시간에 자동 실행 모니터링
- [ ] **프로덕션 레포는 cron 시간을 충분히 떨어지게 설정** (예: 자정 vs 자정 5분)하여 서버 부하 분산

**팁**: 개발 환경에서는 테스트를 위해 cron을 짧은 간격으로 설정 후, 프로덕션 배포 시 의도된 시간으로 변경하기.
