# /ingest 슬래시 커맨드 — 외부 URL을 검증된 학습 엔트리로 가공하는 7단계 게이트

## 문제

외부 URL(유튜브 영상, 블로그, 논문, 트위터 스레드)을 학습 엔트리로 빠르게 정리하고 싶다. 그런데 매번 수작업으로 하면 다음 두 종류의 사고가 반복된다.

1. **메타데이터 오기재** — oembed/검색 결과 한 곳만 믿고 제목·저자·채널명을 잘못 적음. 1글자 차이("실밸"↔"실버")가 가장 위험.
2. **인용구 날조** — 원본을 직접 보지 않은 채로 `"..."` 형식 직접 인용을 본문에 박음. 이건 *작은 것* 이 아니라 신뢰도를 0으로 만드는 근본 사고.

2026-04-11 첫 사고(하네스 5레버 엔트리, `1f2ed0b` → `0221d4b` → `d1255d5`)를 통해 **사람이 매번 같은 절차를 기억해서 지키는 것은 불가능**하다는 게 명확해졌다. 절차 자체를 시스템화해야 한다.

## 증상 (사고가 어떻게 보였는가)

```
1. 사용자: "이 유튜브 영상 정리해줘"
2. AI: oembed로 메타데이터 받고 → 곧바로 본문 작성 → 커밋
3. 사용자: "채널명/제목이 틀렸는데?"
4. AI: hotfix 커밋 1 (제목·채널명 수정)
5. 사용자: "본문 인용구도 영상에서 직접 본 거 맞아?"
6. AI: 아니오 (가짜 인용구 6개 발견)
7. AI: hotfix 커밋 2 (인용구 6개 제거 + "정보 출처 고지" 추가)
```

같은 엔트리에 hotfix가 두 번 — 메타데이터 1번, 인용구 1번. 둘 다 *작성 시점에* 막을 수 있는 사고였다.

## 해결: `/ingest` 슬래시 커맨드 (`.claude/commands/ingest.md`)

URL을 받은 순간부터 엔트리 생성까지를 **7단계 게이트**로 강제한다. 어느 게이트라도 통과 못 하면 작업 중단.

### Phase 0: 사전 점검
- 이미 존재하는 엔트리에 인용된 URL인가? (중복)
- 접근 가능한 형식인가? (private 링크면 중단)

### Phase 1: 메타데이터 확보 — **최소 2회 교차 검증**
도메인별 핸들러:

| 도메인 | 1차 | 2차 |
|---|---|---|
| youtube.com | oembed | WebSearch로 제목·채널명 재확인 |
| velog/medium/github.io | WebFetch 본문 | `<meta>` 태그 vs 본문 제목 |
| arxiv | WebFetch 요약 | abs/ 페이지 vs PDF 제목 |
| twitter/threads | WebFetch | WebSearch 정리글 |

**🚨 의미 모순 체크 (Critical)**:
- 제목과 목표 카테고리가 의미적으로 일치하는가?
- 두 소스 사이에 *한 글자라도* 차이가 있는가? — 1글자 차이가 가장 위험
- 의심 시그널 하나라도 발견되면 **즉시 중단 + 사용자에게 보고**. 추측으로 진행 금지.

### Phase 2: 본문 보강 — **최소 2개 독립 소스**
영상/음성처럼 직접 읽기 어려운 매체는:
1. WebSearch로 동일 주제 텍스트 정리 포스트 2개 이상 확보 (같은 저자 글 X — 독립 소스 원칙)
2. **두 소스에서 동일하게 등장하는 개념만 "검증된 사실"** 로 간주
3. 한 소스에만 나오는 주장은 직접 인용 금지 — 풀어 쓰거나 생략

**인용구 규칙 (엄격)**:
- `"..."` 직접 인용은 **원본에서 직접 글자 단위로 확인한 문장**만 허용
- 트랜스크립트가 없어 확신 못 하면 인용 대신 *"~라는 취지의 말을 함"* 으로 풀어 쓴다
- 첫 훅으로 쓸 인용구는 특히 더 엄격하게

### Phase 3: 카테고리 매핑
사용자 명시 → 키워드 매칭 추천 → Phase 1의 의미 모순 체크 한 번 더 → 애매하면 사용자 확인.

### Phase 4: 초안 작성
고정된 frontmatter 템플릿 + 고정된 본문 섹션 순서(왜 지금 / 핵심 개념 / 구조 / 실전 팁 / 내 프로젝트 / 자기 점검 / 출처). 1인칭 시점 + 내 해석 섹션 필수.

### Phase 5: 검증
`npm run build` → MDX/Mermaid validation. 실패 시 최대 3회 수정 재시도. dangling connection 체크.

### Phase 6: 커밋 제안
`학습: <짧은 제목>` 패턴. **push는 사용자 명시 승인 대기.** (`feedback_commit_deploy.md`)

### Phase 7: 보고
- 생성 파일 경로
- 카테고리 선택 근거
- 교차 검증에 사용한 소스 리스트 (도구까지: "oembed로 받음", "WebSearch로 교차")
- 확인하지 *못한* 부분 명시
- 애매한 부분 있으면 사용자 확인 요청

### 금지 사항 (커맨드에 명시적으로 박음)
1. 단일 소스 메타데이터만 믿고 본문에 인용 ❌
2. 원본에서 직접 확인 못한 문장을 `"..."` 로 박기 ❌
3. 제목-카테고리 의미 모순을 감지하고도 진행 ❌
4. "본문이 맞으면 제목 정도는 괜찮다"는 합리화 ❌
5. 사용자 승인 없이 push ❌

## 첫 dogfooding (`308bdd3`)

`/ingest https://news.hada.io/topic?id=26560` → Compound Engineering 엔트리 생성.

| 단계 | 실제로 한 일 | 결과 |
|---|---|---|
| Phase 0 | `Grep`으로 URL 중복 체크 | 0건 |
| Phase 1 | hada WebFetch + every.to 원본 WebFetch + WebSearch | 제목/저자/날짜 3소스 일치 |
| Phase 2 | every.to 원본 직접 본문 fetch + WebSearch로 추가 every.to 글 확인 | 4단계 루프·5단계 ladder·인용구 모두 직접 확인 |
| Phase 3 | harness-engineering — 의미 모순 없음 | ✓ |
| Phase 4 | 1인칭 시점, "이미 /compound로 실천 중" 프레임 | 단순 번역본 아님 |
| Phase 5 | `npm run build` 1회 통과 | 21 entries, dangling 0 |
| Phase 6 | 사용자 승인 후 push | ✓ |
| Phase 7 | 검증 메모를 엔트리 본문에도 남김 | ✓ |

**핵심**: 첫 적용에서 hotfix 0건. 사고 → 시스템화 → dogfooding → 통과 사이클이 한 번에 닫혔다.

## 근본 원인

사람의 기억력에 의존하는 절차는 *반드시* 실패한다. 특히 다음과 같은 조건이 겹치면 100% 실패:

1. 절차가 5단계 이상
2. 검증 항목이 한국어/영어 1글자 차이 같은 미세 차이
3. AI가 "그럴듯한" 추측으로 빈 칸을 메울 수 있음
4. 사용자가 매번 결과를 정밀 검증하지 않음

해결 방향은 두 가지가 있다:
- **(A) 더 잘 기억하기** ← 안 된다
- **(B) 절차를 실행 게이트로 강제** ← `/ingest`가 한 일

게다가 게이트 자체를 슬래시 커맨드 파일로 박아두면 *프로젝트의 영구 자산*이 된다. 사람이 메모리를 잃어도, 새 세션이 시작돼도, `/ingest`만 호출하면 동일한 검증이 다시 작동한다. 이게 Compound Engineering이 말하는 "다음 작업을 더 쉽게 만드는" 자산화의 전형이다.
