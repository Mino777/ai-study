## 0. 한 줄 포지셔닝

"이건 'LLM으로 iOS 개발을 빠르게'가 아니라, **LLM 에이전트가 프로덕션에서 저지르는 실패를 체계적으로 박제·측정·차단하는 자기개선형 운영체제**입니다. 저는 이걸 toy 데모가 아니라, PR 리뷰 게이트조차 없는 실상용 앱에 에이전트를 태우면서 회귀를 코드 레벨 가드로 막는 실전 인프라로 설계했습니다."

## 1. 왜 plugin + MCP로 외부화(앱 repo에서 분리)했나

### 상황
초기엔 하네스 자산(스킬·에이전트·훅·박제)이 전부 앱 repo의 `.claude/` 안에 살았습니다. 그러니까 도구를 고친 커밋과 앱 기능 커밋이 히스토리에서 뒤섞였습니다. 리뷰할 때 "이게 앱 변경이야 도구 변경이야"를 매번 골라내야 했고, 앱 repo의 커밋 히스토리가 오염됐습니다.

### 결정
하네스를 **별도 repo + Claude Code 플러그인**으로 외부화하고, 박제(팀 지식)는 **MCP 서버**로 관리했습니다. 그리고 무엇이 어느 repo에 쌓이는지를 "경계 매트릭스"로 못박았습니다.

- **하네스 repo**: 메커니즘 — agents/skills/commands/hooks/scripts, MCP 서버, 운영룰(CLAUDE.md 본체), 박제 메모리, 하네스 ADR·회고. MCP add_memory로 **자동 PR + auto-merge**(사용자 개입 0).
- **앱 repo**: 검증 대상과 함께 진화하는 것 — 앱 Swift 코드, 검증 yaml, SPEC, 앱 ADR. 일반 PR(코드와 함께 리뷰).

원칙 한 줄: **"검증/명세 자산은 검증 대상(앱)과 함께 진화, 하네스 메커니즘은 하네스 repo."** 그리고 marketplace + `plugin.json` semver로 배포하니까 `/plugin update`만으로 팀 전원이 최신 도구를 받습니다.

### 트레이드오프
- 얻은 것: 앱 repo가 도구 변경에 오염 안 됨. 도구를 소프트웨어 제품처럼 버전·배포·롤백 가능.
- 낸 것: **2-repo 동기화 복잡도**. version bump를 깜빡하면 marketplace가 "이미 설치됨"으로 skip해서 stale cache가 발생합니다. 이건 실제로 겪었고, 그래서 "자산 변경 시 version bump 강제" 규칙을 훅·ADR로 박제해 방어했습니다.

### 예상 꼬리질문
**Q. 그냥 monorepo로 두는 게 동기화 부담이 없지 않나?**
A. 두 자산의 **변경 주기(cadence)와 리뷰 정책이 완전히 다릅니다.** 앱 코드는 사람 리뷰를 거쳐야 하는 baseline이고, 하네스 지식은 사람 리뷰 병목 없이 auto-merge로 즉시 전파돼야 팀 전체에 복리로 쌓입니다. 이 둘을 한 repo에 두면 auto-merge 정책과 리뷰 정책이 충돌합니다. 분리의 본질은 파일 위치가 아니라 **거버넌스 경계**입니다.

**Q. 정직하게, 아직 미완결인 부분은?**
A. 있습니다. 플러그인이 완전 self-contained가 아직 아니라서, 다수 파일이 앱 repo의 스크립트 경로를 하드코딩 참조하고 있고 `${PLUGIN_ROOT}` 자기참조 치환이 `.md` 본문에는 안 먹습니다(env는 훅/MCP 서브프로세스에만 주입되니까요). 그래서 단일 추상화 변수 + 폴백 패턴으로 점진 이행 중이고, 이걸 ADR로 추적하고 있습니다. **자기 결함을 ADR로 스스로 박제 중이라는 게 오히려 시스템 건강 신호**라고 봅니다.

## 2. 왜 메모리를 auto-PR로(팀 공유 박제) 만들었나

### 상황
개인 Claude 세션이 겪은 환각·버그·해결책은 그 세션이 끝나면 증발합니다. 다음에 다른 팀원(혹은 내 다른 세션)이 똑같은 벽에 부딪히면 처음부터 다시 삽질합니다. 로컬 메모리 파일은 그 세션에만 갇혀 있어서 팀 자산이 못 됩니다.

### 결정
박제를 **git-네이티브 팀 지식 그래프**로 만들었습니다. `add_memory` MCP 호출 → 로컬 `memory/*.md` 파일 쓰기 + 인덱스 갱신 → `bot/memory/<slug>` 브랜치 → REST로 **PR 자동 생성 → squash auto-merge → 브랜치 자동 삭제** → 모든 팀원 세션이 main에서 즉시 surface. 사용자 개입 0입니다.

메타데이터는 지식 그래프 최신 연구를 차용했습니다: **bi-temporal**(`valid_from` / `invalidated_by` / `superseded_by`) — 삭제 대신 무효 마킹, 대체(superseded)와 세계-무효(invalidated)를 구분. `last_verified` 자동 스탬프로 stale 탐지. 인덱스는 **200줄 budget**으로 관리(에이전트가 200줄 이후를 truncate하므로), 근접 중복은 canonical 허브로 접습니다.

### 트레이드오프
- 얻은 것: 팀 지식이 git으로 버전관리·리뷰 가능. 실수가 자동으로 팀 영구 자산이 됨. 팀·세션이 늘수록 복리.
- 낸 것: 인증 env 셋업 필요, **write serialization** 필요(병렬 add_memory가 rebase 충돌). 이건 cross-process 락(원자적 mkdir)으로 직렬화해서 해결했고, 그 자체를 ADR로 박제했습니다.

### 예상 꼬리질문
**Q. auto-merge면 쓰레기 지식이 무검증으로 팀에 퍼지지 않나? 신뢰성 리스크 아닌가?**
A. 그래서 **merge 게이트를 사람 리뷰가 아니라 코드로** 옮겼습니다. add_memory 파이프라인 안에 방어 장치가 folding돼 있습니다 — 위키 형식 강제 검증(feedback/project는 본문에 "왜"+"어떻게 적용" 라벨 필수, 없으면 throw), 의미 기반 near-duplicate 가드(신규 저장 전 유사도 측정 → 경고, 자동 덮어쓰기 안 함), 공유 토큰 3개 이상이면 related 자동 링크(orphan 원천 차단). 즉 **품질 게이트를 리뷰 시점에서 write 시점으로 left-shift**한 겁니다. 사람 병목을 제거하되 품질 하한선은 결정적 검증으로 유지합니다.

**Q. 이 add_memory 코드 자체의 신뢰성은? MCP 서버가 멈추면?**
A. 이 코드는 자기 실패 경험을 방어 장치로 흡수한 결과물입니다. git commit이 silent fail하던 문제 → user.email fail-fast, 네트워크 hang으로 stdio MCP 전체가 멈추던 문제 → 20~30초 timeout + AbortSignal, 인증 401 사고 → email 형식 fail-fast, 로컬 unpushed 커밋을 auto-PR이 흡수하던 문제 → 방어 로직. 이 방어 하나하나가 다시 박제로 인덱스에 존재합니다. **분산 시스템에서 흔한 실패 모드(부분 실패, 타임아웃, 동시성, 인증)를 하나씩 코드 게이트로 닫은** 케이스입니다.

## 3. 왜 멀티에이전트 오케스트레이션인가 (그리고 왜 후퇴시켰나)

### 상황
초기엔 tmux 4-pane 워커풀로 여러 에이전트를 병렬 가동하고 send-keys로 IPC를 흉내냈습니다. 그런데 이게 실패의 온상이었습니다 — send-keys 전송 유실, pane-grep으로 워커 상태를 잘못 판정, 기동 hang, 컨텍스트 도미노(한 워커 오염이 연쇄).

### 결정
**두 단계로 결정했습니다.**

(1) 멀티에이전트 토폴로지를 **SOLO + 온디맨드 격리 fan-out**으로 후퇴시켰습니다(ADR로 확정, 구 워커풀 관련 52개 파일 물리 삭제). 메인 세션이 직접 grep/Read/Edit/build를 하는 SOLO 오케스트레이터 겸 실행자이고, **무겁거나 병렬인 작업만** 온디맨드로 worktree 격리 에이전트를 2~3개 보수적으로 fan-out합니다. tmux는 IPC가 아니라 레이아웃+persistence용일 뿐이고, 세션 간 조율은 git(worktree/branch) + 공유 박제(MCP) + 사람으로 합니다.

(2) 대신 살아남은 멀티에이전트 패턴은 **역할 라우팅과 검증 분리**에 집중했습니다:
- **모델 비용 라우팅**: 17개 에이전트가 frontmatter로 model을 명시. 읽기전용 룰기반(탐색/리뷰/UI검증)은 저비용 모델, 다단계 구현은 중간 모델, 아키텍처 판단만 최상위 모델. 비용을 역할에 맞게 계층화.
- **Generator–Critic 분리**: 코드 리뷰가 Tier1(결정적 grep, FAIL=BLOCKER) / Tier2(LLM 판단, FAIL=WARNING)로 이분. Tier2는 verdict 전에 **반증우선**(먼저 FAIL 근거를 찾고 없으면 PASS)으로 agreement bias를 회피.
- **2중 안전 사후감사**: merge를 수행한 에이전트의 "유실 0" 자기주장을 신뢰하지 않고, **독립** 감사 에이전트가 반증우선으로 재검증.

### 트레이드오프
- 얻은 것: 단순·관측가능·재현가능. 실패 클래스 원천 제거.
- 낸 것: 자동 병렬 오케스트레이션의 이론적 처리량. "인지 대역폭은 병렬화 안 된다"는 판단하에, 1인~소수 팀 규모에서 A2A 자동 메시징은 오버엔지니어링이라 도입 안 했습니다.

### 예상 꼬리질문
**Q. 멀티에이전트를 자랑하다가 결국 SOLO로 후퇴한 거면, 실패 아닌가?**
A. 반대로 이게 **엔지니어링 성숙도**라고 봅니다. 결정 근거가 우리 자체 실패 taxonomy 25건+ 에다가 외부 문헌(멀티에이전트 실패 연구)까지 **3-소스 수렴**이었습니다. 화려한 아키텍처를 유지하는 게 아니라, 측정된 실패율을 근거로 복잡도를 걷어낸 겁니다. 그리고 완전 포기가 아니라, **신뢰성이 검증되는 격리 fan-out만 남기고 관측 불가능한 IPC를 버린** 선택적 후퇴입니다.

**Q. 하네스 자산을 바꿀 때 이게 정말 개선인지 어떻게 아나? 회귀는?**
A. 하네스 자체 변경(운영룰/verifier 프롬프트/스킬/훅)을 골든셋(GT-1~22)으로 돌려 **Wilson score 95% 신뢰구간**으로 개선/회귀/노이즈를 판정합니다. 신뢰구간이 겹치면 "채택 금지". 그리고 gold-solution 주입 selfcheck로 LLM 없이 파이프라인을 결정적으로 검증합니다. **프롬프트 튜닝을 측정 기반 실험으로** 다룬 겁니다.

**Q. subagent가 "완료했다"고 거짓말하면?**
A. SubagentStop 훅에 done-lie 검증이 있습니다. builder 계열 에이전트가 "빌드 성공/완료"를 주장하며 종료했는데 transcript 마지막 빌드 결과가 실패면 **exit 2로 stop을 차단**합니다. 15줄 bash, 0토큰, "빌드 깨진 채 done 거짓말"을 결정적으로 막는 최고 ROI 가드입니다.

## 4. 왜 훅 기반 가드(PreToolUse 차단)인가

### 상황
"메모리에 규칙을 적어두면 에이전트가 지키겠지"는 안 통합니다. LLM은 CLAUDE.md에 적힌 룰을 깜빡합니다. 특히 위험한 동작 — 시크릿 파일 편집, 박제 파일 직접 쓰기(auto-PR 우회), 광역 grep으로 "검색 0건"을 "기능 부재"로 오판 — 은 문서 규칙만으로는 못 막습니다.

### 결정
규칙을 **문서가 아니라 코드로**, 라이프사이클 훅에 배선했습니다. 46개 claude-hooks가 PreToolUse/PostToolUse/SubagentStop/SessionStart마다 걸립니다. exit code 규약은 **0=silent 성공, 1=재시도, 2=차단**.

핵심은 **차단(exit 2)은 소수의 안전 footgun에만, 나머지는 warn(exit 0)** 이라는 규율입니다("성공은 silent, 실패만 표면화"). 실제 차단 사례:
- **file-protection**: `.env`/시크릿 plist/인증서 편집 차단 + **`memory/*.md` 직접 쓰기 차단**(박제는 MCP add_memory만 — auto-PR 파이프라인 우회 방지 + repo 경계 강제).
- **grep-hygiene**: 광역 grep / 빌드 산출물 미제외 footgun을 차단해 전용 탐색 도구로 유도.
- **AI 주석 enforce**: AI가 실질 코드 4줄+ 추가하면서 "왜" 주석 0개면 exit 2로 되돌림.

warn 계열은 더 나은 도구로 유도(전체 파일 통째 Read → 부분 Read/코드그래프)하거나 관련 지식을 자동 surface합니다.

### 트레이드오프
- 얻은 것: 에이전트가 룰을 어겨도 **행동 레벨에서 물리적으로 차단**. 리뷰어 없이도 안전 하한선 유지.
- 낸 것: 훅 유지보수 부담 + 세션 블로킹 위험. 그래서 훅 규칙을 못박았습니다 — 실행 5초 이내, timeout 30초 이하, stderr 최소화(컨텍스트 오염 방지).

### 예상 꼬리질문
**Q. 훅이 너무 많으면 에이전트가 계속 막혀서 오히려 생산성이 떨어지지 않나?**
A. 그래서 **차단과 경고를 엄격히 분리**했습니다. 진짜 차단은 되돌릴 수 없거나 프로덕션에 나가면 사고인 것(시크릿, 박제 우회, 안전 위반)에만 걸고, 나머지는 전부 non-blocking warn입니다. 훅이 시끄러우면 에이전트가 무시하게 되니까, 차단은 희소해야 신호가 삽니다. 이건 "성공은 silent" 원칙 — 잘 될 땐 아무 소리도 안 나야 합니다.

**Q. 어떤 규칙을 문서에 두고 어떤 걸 훅으로 승격하나? 판단 기준은?**
A. **격상 사다리**로 결정론화했습니다. 같은 결함이 — 1회면 박제만(탐색 비용 감수) → 2회면 에이전트/스킬에 inline 박제 → 3회면 CLAUDE.md HARD RULE → 4회면 **훅 코드 fix 의무** → fix 후에도 재발하면 즉시 구조 ADR. 즉 규칙을 훅으로 올리는 건 감이 아니라 **반복 횟수라는 측정치**가 트리거합니다. "환각 1회는 탐색 비용이라 OK, 2회 발생은 박제 갱신 실패 = 운영 결함"이 원칙입니다.

**Q. 훅이 우회되면(worktree 격리 서브에이전트가 세션 훅을 안 거치면)?**
A. 그 케이스를 위해 커밋 직전 self-check 스킬로 코드 레벨 안전 회귀를 이중으로 막습니다. 세션 훅과 스킬 레벨 가드를 겹쳐서, 어느 한 경로가 뚫려도 다른 층이 잡게 **defense-in-depth**로 설계했습니다.

## 5. 왜 JIT retrieval / 토큰 최적화인가

### 상황
두 가지 문제가 있었습니다. (1) 에이전트가 코드를 맨손 grep→read 루프로 찾으면 느리고, "검색 0건"을 "기능 없음"으로 오판해 환각합니다(실제로 BLE 관련 "기능 없음" 2회 오답이 근본원인이었습니다). (2) 579개 박제 + 152 솔루션을 전부 컨텍스트에 넣으면 context window가 폭발합니다.

### 결정
**JIT(Just-In-Time) 검색**으로, 필요한 순간에 필요한 조각만 꺼냅니다.

- **탐색**: 전용 도구(캐시+시맨틱, LLM 미사용이라 환각 0)로 0.1~0.5초에 위치 특정. 맨손 grep 금지를 HARD RULE로. "검색 0건 ≠ 기능 부재"를 명시 룰로.
- **지식 검색 3계층**: BM25 랭킹 박제 검색(IDF+길이 정규화) / 그래프 인지 검색(매칭 노드 + 이웃 1~2 hop, self-contained라 격리 에이전트에서도 안전) / 셸 통합 검색기(온톨로지 쿼리 확장 — 한글 키워드를 도메인 동의어로 OR 확장해 단일 grep 0건 환각 방지).
- **컨텍스트 격리**: 광역 탐색은 서브에이전트로 격리하고 메인엔 요약만 반환(측정치: 격리 9K vs 누적 15K 토큰, 약 40% 절감).
- **인덱스 200줄 budget**: 에이전트가 truncate하는 한계를 정면으로 엔지니어링. 활성 박제만 인덱스에, 이력성은 아카이브(파일은 보존해 JIT로 계속 검색됨).

### 트레이드오프
- 얻은 것: 초 단위 위치 특정 + 환각 0 탐색 + 컨텍스트 절감.
- 낸 것: 인덱스 stale 가능성. 그래서 SessionStart 훅이 wiki-index를 백그라운드로 rebuild하고, 박제엔 last_verified 스탬프 + decay 추적(안 쓰이고 오래된 인덱스 항목을 아카이브 후보로 제안)을 붙였습니다.

### 예상 꼬리질문
**Q. 박제만 쌓으면 뭐하나, 정작 필요할 때 안 꺼내지면?**
A. 그게 대부분 지식 시스템의 실패 지점입니다 — "쌓기"는 하는데 "막혔을 때 자동으로 튀어나오기"에 실패합니다. 그래서 **활용을 훅으로 강제**했습니다. PostToolUse 훅이 에러 패턴 / 같은 명령 10분 내 3회 반복 / 탐색 churn을 감지하면 관련 박제 top-3을 **자동으로 컨텍스트에 주입**합니다("추측·재시도 전에 이 박제부터 확인"). 사람이 "혹시 전에 이런 적…"을 기억할 필요가 없습니다. 노이즈 가드로 90초 cooldown + 저점수 skip + non-blocking을 겁니다. **축적↔활용 폐루프를 훅으로 닫은** 게 핵심입니다.

**Q. 왜 벡터 임베딩 RAG를 안 쓰고 BM25/그래프인가?**
A. 세 가지 이유입니다. (1) 코드/도메인 용어는 정확 매칭이 중요해서 BM25의 어휘 정합이 오히려 유리합니다. (2) 박제는 `[[slug]]` 링크로 명시적 그래프를 이루니까, 임베딩 유사도보다 **명시적 이웃 확장**이 더 정확하고 설명 가능합니다. (3) self-contained여야 worktree 격리 에이전트에서도 외부 의존 없이 돕니다. 규모(579개)에서 brute-force가 1~2ms라 벡터 인프라의 복잡도를 정당화할 이유가 없었습니다. **규모에 맞는 가장 단순한 검색**을 골랐습니다.

## 6. 오케스트레이터 / 워커 경계는 어떻게 그었나

### 상황
"누가 무엇을 하고, 무엇을 신뢰하고, 무엇을 격리하나"의 경계가 흐리면 컨텍스트 오염과 책임 소재 붕괴가 일어납니다.

### 결정
**세 축으로 경계를 그었습니다.**

(1) **실행 경계 (오케스트레이터 vs 워커)**: 메인 세션 = SOLO 오케스트레이터 겸 실행자. 가벼운 작업(1~5줄)은 직접, 광역 탐색은 격리 Explore 서브에이전트, 신규 피쳐 풀사이클은 격리 builder 에이전트로 fan-out. 작업 크기로 라우팅합니다. 격리 에이전트는 CLAUDE.md를 상속 못 하므로 핵심 룰("환각 2회 금지" 등)을 각 에이전트 파일에 **인라인 재박제**합니다.

(2) **신뢰 경계 (Generator vs Critic)**: 만드는 에이전트와 검증하는 에이전트를 분리합니다. merge를 한 에이전트의 자기주장을 독립 감사 에이전트가 반증우선으로 재검증(2중 안전). 리뷰는 결정적 검사(BLOCKER)와 판단 검사(WARNING)를 분리.

(3) **거버넌스 경계 (하네스 repo vs 앱 repo)**: §1의 경계 매트릭스. 하네스 자산은 auto-merge(개입 0), 앱 자산은 일반 PR(코드와 함께 리뷰). file-protection 훅이 이 경계를 물리적으로 강제.

### 트레이드오프
- 얻은 것: 책임 소재 명확 + 컨텍스트 오염 격리 + 신뢰성(자기검증 금지).
- 낸 것: 격리는 컨텍스트 공유를 포기하는 것이라, 에이전트 간 미묘한 조율은 사람이 git과 박제를 통해 이어줘야 합니다.

### 예상 꼬리질문
**Q. 오케스트레이터가 곧 실행자면 단일 장애점 아닌가?**
A. 1인~소수 팀 규모에선 **관측 가능성이 처리량보다 중요**하다고 판단했습니다. 자동 A2A 오케스트레이터를 두면 처리량은 늘지만, 어디서 뭐가 틀어졌는지 사람이 못 봅니다(구 워커풀에서 실측한 실패). SOLO는 사람이 모든 결정 지점을 관측·개입할 수 있고, 병렬이 필요한 순간만 격리 fan-out으로 확장합니다. **확장은 fan-out으로, 신뢰는 SOLO로** 분리한 겁니다.

**Q. 격리 에이전트에 룰을 인라인 재박제하면 CLAUDE.md와 drift하지 않나?**
A. 맞습니다, 그게 비용입니다. 그래서 정합성 감사를 자동화하는 방향으로 가고 있고(스킬↔라우팅 정합성 체크), 핵심 룰은 소수만 인라인하고 나머지는 스킬 진입점을 얇게 유지해 로직 중복을 줄입니다. 완벽하진 않지만, **격리의 이득(컨텍스트 오염 차단)이 drift 비용보다 크다**고 봅니다.

**Q. 이 경계 설계를 처음부터 알았나, 아니면 배웠나?**
A. 전부 실패에서 역산했습니다. silent-nil(서버 키와 한 글자 달라도 에러 0으로 조용히 nil — 컴파일·mock QA·diff 다 통과하고 실서버에서만 버그), mock 프로덕션 유출, 워커풀 도미노 — 각각을 겪고 나서 3-오라클 계약 감사, ship-radar 회귀 게이트, SOLO 후퇴로 경계를 다시 그었습니다. **경계는 설계가 아니라 사고에서 배운 것**이고, 그 학습 과정 자체가 박제·ADR로 히스토리에 남아 있습니다.

## 마무리 — 격상 시키는 한 문장

"제가 이 하네스에서 증명한 건 특정 도구 사용법이 아니라, **AI 에이전트를 프로덕션에 안전하게 태우는 신뢰성 엔지니어링**입니다. 실패를 1급 시민으로 다루고(taxonomy → 박제 → 훅 → ADR 격상 사다리), 품질 게이트를 리뷰에서 write 시점으로 left-shift하고, 신뢰 경계를 Generator-Critic으로 분리하고, 변경을 신뢰구간으로 측정하는 것 — 이건 분산 시스템·DevEx·신뢰성 관점 그대로이고, 어떤 조직이 LLM 에이전트를 도입하든 재사용 가능한 아키텍처 레벨의 자산입니다."
