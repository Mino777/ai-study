## Q1. "이거 결국 셸 스크립트랑 마크다운 모음 아닌가요? 뭐가 엔지니어링이죠?"

**함정**: 규모(파일 수)로 반박하게 유도 → "파일 많다"는 임팩트가 아님. 양이 아니라 설계 원리를 물은 것.

**강한 답변**:
스크립트가 176개인 건 결과지 본질이 아닙니다. 본질은 세 가지 설계 결정입니다. 첫째, **실패를 자산으로 접는 폐루프**입니다. 같은 실수가 반복되면 격상 사다리(1회 박제 → 2회 에이전트 인라인 → 3회 CLAUDE.md HARD RULE → 4회 hook 코드 게이트 → 5회 구조 ADR)를 타고 점점 강한 강제 장치로 승격됩니다. 즉 문서가 아니라 **실패 빈도가 가드 코드의 강도를 결정**하는 측정 기반 시스템입니다. 둘째, **지식을 git-네이티브 팀 자산으로 승격**한 것입니다 — 개인 로컬 메모리가 아니라 `add_memory` MCP 호출이 자동 PR + squash auto-merge로 팀 공유 repo에 반영됩니다. 셋째, **하네스 자체를 소프트웨어 제품처럼 검증**합니다 — MCP 서버에 단위 테스트, 자산 변경은 Wilson score 95% 신뢰구간 회귀셋(GT-1~22)으로 개선/회귀/노이즈 판정. 신뢰구간이 겹치면 채택 금지입니다. "스크립트 모음"은 이 세 루프를 못 만듭니다.

**one-liner**: 파일 수가 아니라 "실패 → 측정 → 코드 게이트 승격" 폐루프가 엔지니어링입니다.

## Q2. "LLM은 환각을 합니다. 하네스가 만든 코드나 박제를 어떻게 신뢰하죠?"

**함정**: "신뢰합니다/안 합니다" 이분법 유도. 정답은 "신뢰하지 않도록 설계했다".

**강한 답변**:
신뢰하지 않는 걸 전제로 설계했습니다. 세 층위 방어가 있습니다. **탐색 층**: 맨손 grep→read 루프를 HARD RULE로 금지하고, 코드 위치 특정은 캐시+시맨틱 기반 전용 도구(`code-locate`, `ask-bot`)로 대체합니다 — 이 도구들은 LLM을 쓰지 않으므로 위치 특정에 환각이 0입니다. 답변에는 file:line 출처와 신뢰도가 붙습니다. **검증 층**: Generator-Critic을 분리합니다. 코드 리뷰는 Tier1(결정적 grep, FAIL=BLOCKER)과 Tier2(LLM 판단, FAIL=WARNING)로 이분하고, Tier2는 반증우선(falsification-first) 프롬프트로 FAIL 근거를 먼저 찾은 뒤 없으면 PASS — LLM-as-judge의 agreement bias를 회피합니다(실측 failure detection +25%p). **강제 층**: 서브에이전트가 "완료"라고 거짓말하면 `done-lie-on-subagent-stop` 훅이 transcript 마지막 빌드 결과를 확인해 실패면 exit 2로 종료를 차단합니다. 15줄 bash로 "빌드 깨진 채 done 거짓말"을 결정적으로 막습니다.

**one-liner**: LLM을 신뢰하는 게 아니라, LLM이 틀렸다는 전제로 결정적 게이트를 깔았습니다.

## Q3. "메모리를 자동 PR로 auto-merge 한다고요? 사람 리뷰 없이 쓰레기가 팀 전체에 퍼지는 거 아닌가요?"

**함정**: auto-merge = 무검증이라는 프레임. 실제로는 write-time 검증이 다층.

**강한 답변**:
auto-merge 전에 **write-time 게이트**가 여러 겹입니다. 첫째, `validateWikiFormat`이 형식을 강제합니다 — description 필수, feedback/project 타입은 본문에 `Why:` + `How to apply:` 라벨이 라인 단위 정규식으로 있어야 하고, 없으면 throw. 빈 박제·산문 우연매칭을 원천 차단합니다. 둘째, `findNearDuplicates`가 저장 전 기존 박제와 slug+name+desc 유사도를 측정해 근접중복이면 경고를 반환하고, 자동 덮어쓰기는 안 합니다(명시 upsert만). 셋째, slug은 kebab-case 강제, legacy underscore는 기존 파일 존재 시 in-place upsert만 허용해 중복본 양산을 막습니다. 넷째, `suggestRelated`가 공유 토큰 3개 이상 박제를 related에 자동 링크해 orphan을 막습니다. 그리고 이게 앱 코드가 아니라 **별도 지식 repo**에 쌓이기 때문에 프로덕션 코드 릴리스에 영향이 없습니다. 품질이 낮은 박제가 들어와도 200줄 인덱스 budget과 decay 추적(access count 기반 ARCHIVE 후보 제안)으로 자연 도태됩니다.

**one-liner**: auto-merge는 write-time 형식·중복·링크 게이트를 통과한 뒤에만 일어납니다 — "무검증"이 아니라 "결정적 검증 후 무마찰 전파"입니다.

## Q4. "여러 세션이 동시에 add_memory 하면 git 충돌 나잖아요. 그건 어떻게?"

**함정**: 분산 쓰기 동시성 문제를 놓쳤을 거라는 가정.

**강한 답변**:
실제로 겪은 문제고, ADR로 박제했습니다. 병렬 `add_memory`가 rebase 충돌을 내던 걸 **cross-process 락**으로 직렬화합니다 — `.git/gma-addmemory.lock`을 `mkdir` 원자성으로 획득(ADR-006)하고, try/finally로 모든 경로에서 릴리스를 보장합니다. 각 쓰기는 pull --rebase → 고유 timestamp 브랜치(`bot/memory/<slug>-<ts>`) → commit → push → PR → squash merge → 브랜치 삭제 순으로 원자적으로 진행됩니다. 추가로 네트워크 hang이 stdio MCP 전체를 멈추던 문제를 simpleGit `block:30000`, Bitbucket fetch `AbortSignal.timeout(20000)`으로 막았고, git user.email 미설정 시 commit이 silent fail하던 최악 케이스를 fail-fast throw로 바꿨습니다. 이 방어 장치 하나하나가 대응하는 박제로 존재합니다 — 시스템이 자기 동시성 실패로부터 배운 흔적이 코드에 접혀 있습니다.

**one-liner**: 동시성은 mkdir 원자 락으로 직렬화하고, 그 결정을 ADR로 박제했습니다.

## Q5. "훅이 40개 넘게 걸려 있으면 개발이 느려지지 않나요? 매 동작마다 게이트면 짜증날 텐데."

**함정**: 게이트 = 마찰 = 생산성 저하 프레임.

**강한 답변**:
훅의 대부분은 **차단이 아니라 warn(exit 0)**입니다. 진짜 차단(exit 2)은 소수의 안전 footgun에만 한정합니다 — 광역 grep(`.build` 미제외로 검색 0건을 기능 부재로 오판), AI 코드 "왜" 주석 누락, 민감 파일 직접 편집, 박제 파일 직접 쓰기, 빌드 깨진 채 done 거짓말. 나머지는 "더 나은 도구를 유도"하거나 "관련 지식을 surface"하는 비차단입니다. 설계 원칙이 명시적입니다: **성공은 silent, 실패만 표면화**(Codex 원칙). 또 룰로 실행 시간 5초 이내, hook timeout 30초 이내를 강제해 세션 블로킹을 막습니다. 오히려 훅이 생산성을 **높입니다** — `recall-on-error`가 에러·반복·탐색 churn을 감지해 과거 해결책을 자동 주입하니, 개발자가 "전에 이런 적 있었나"를 기억할 필요가 없습니다. 마찰을 넣는 게 아니라 재작업을 없애는 겁니다.

**one-liner**: 훅은 대부분 silent warn이고, 차단은 재작업 비용이 큰 footgun에만 겁니다 — 마찰이 아니라 재작업 제거입니다.

## Q6. "혼자 쓰는 장난감 아닌가요? 팀 임팩트를 어떻게 증명하죠?"

**함정**: 개인 생산성 도구로 축소. "팀"이라는 단어에 근거를 요구.

**강한 답변**:
설계 자체가 개인 도구가 아니라 팀 인프라입니다. 결정적 증거는 **박제가 개인 로컬이 아니라 공유 git repo에 auto-PR로 반영되고, 다른 팀원 세션이 검색으로 그걸 surface받는다**는 겁니다. plugin `autoUpdate:true` 또는 `harness_pull`로 전 팀원이 동기화됩니다. 신규 개발자는 `install-dynamic.sh` + 온보딩 5단계로 clone→plugin/MCP 연동까지 자기완결하고, 그 순간 3.5개월치 579개 박제(대부분 실패에서 역산한 함정)를 상속받습니다 — 즉 팀이 이미 밟은 지뢰를 신규 인원이 안 밟습니다. 커밋 히스토리가 온통 `bot/memory/...` auto-merge PR인 것 자체가 팀 전파가 실제로 돌고 있다는 로그입니다. 물론 현재 운영 모델은 SOLO 오케스트레이터이지만, 그건 "1인이 쓴다"가 아니라 "1인 세션이 오케스트레이터이고 지식은 팀 전체가 공유한다"는 구조입니다.

**one-liner**: 박제가 auto-PR로 공유 repo에 쌓이고 전 팀원 세션에 surface되니, 이건 개인 도구가 아니라 팀 institutional memory입니다.

## Q7. "그럼 왜 멀티 워커 병렬 오케스트레이션을 버리고 SOLO로 후퇴했죠? 확장성 포기 아닌가요?"

**함정**: SOLO = 원시적·비확장. "네이티브 에이전트 팀도 못 굴리나" 압박.

**강한 답변**:
후퇴가 아니라 **실패 taxonomy 기반 의도적 결정**이고 ADR-008로 박제했습니다. 구 4-pane tmux 워커풀은 send-keys IPC / pane-grep 상태판정 / 역할 priming이 실패 클래스(전송 실패·관측 오판·기동 hang·컨텍스트 도미노)의 근원이었고, 관련 52개 파일을 물리 삭제했습니다. 근거는 우리 실패 사례 25건+ 와 외부 문헌(MAST, Cognition) 3-소스 수렴입니다. 대신 **SOLO + 온디맨드 격리 fan-out**을 씁니다 — 무겁거나 병렬인 작업만 worktree 격리 에이전트로 보수적 2~3개 스폰합니다. 근거는 Addy Osmani의 "인지 대역폭은 병렬화되지 않는다"입니다. 세션 간 조율은 취약한 send-keys가 아니라 git(worktree/branch) + 공유 박제(MCP) + 사람으로 대체했습니다. 즉 병렬성을 버린 게 아니라, **조율 채널을 IPC에서 git으로 바꿔 관측 가능하고 결정적으로** 만든 겁니다. 1인 규모에서 A2A 자동 메시징은 오버엔지니어링이라 미도입했습니다.

**one-liner**: 취약한 send-keys IPC를 git+박제로 대체한 의도적 결정이지 확장성 포기가 아닙니다 — 실패 25건 + 외부 문헌 3소스로 뒷받침됩니다.

## Q8. "토큰 비용은요? 훅마다 recall 주입하고 서브에이전트 fan-out하면 비용 폭발할 텐데."

**함정**: 자동화 = 토큰 낭비. 비용 인지 없음을 의심.

**강한 답변**:
비용은 세 가지로 관리합니다. **모델 라우팅**: 에이전트 17개가 역할별로 모델을 명시 라우팅합니다 — read-only 룰기반 작업(탐색/리뷰/UI검증)은 haiku 기본(비용 약 15배 절감), 다단계 추론/구현은 sonnet, 아키텍처 판단만 opus. **컨텍스트 격리**: 광역 탐색은 서브에이전트로 격리해 메인엔 요약만 반환합니다 — 실측 "격리 9K vs 누적 15K 토큰"으로 약 40% 절감. ios-explorer는 파일 dump 0줄, 경로:라인 + 핵심 1~3줄만 반환해 context rot를 막습니다. **JIT 검색**: 박제는 전체를 로드하지 않고 필요 순간에만 BM25로 top-N을 꺼냅니다. recall 훅도 noise 가드(90초 cooldown, score<3 skip, non-blocking)로 주입을 절제합니다. 그리고 훅 자체는 bash라 0토큰입니다 — `done-lie` 훅의 설계 근거가 "15줄 bash, 0토큰, done 거짓말 차단 최고 ROI"입니다. 캐시+시맨틱 탐색 도구도 LLM 미사용이라 탐색에 토큰이 안 듭니다.

**one-liner**: haiku 라우팅 + 컨텍스트 격리(40% 절감) + JIT + bash 훅(0토큰)으로, 자동화가 오히려 토큰을 아낍니다.

## Q9. "실측 숫자를 대는데, 커밋 날짜가 전부 하루이틀에 몰려 있던데요. 이거 조작 아닌가요?"

**함정**: 정직성 함정. 데이터 이상을 숨겼는지 테스트.

**강한 답변**:
좋은 지적이고, 저도 그 이상 징후를 인지하고 명시적으로 표시했습니다. git 커밋 날짜가 전부 최근 하루이틀에 찍힌 건 **history rewrite 흔적**입니다 — 앱 repo에서 하네스 자산을 분리해 별도 repo로 옮기는 과정에서 히스토리가 재작성됐습니다. 그래서 커밋 날짜는 신뢰하지 말라고 리포트에도 적었습니다. 실제 타임라인의 근거는 **retro 파일 날짜(가장 이른 게 약 3.5개월 전)와 박제 메타의 last_verified/valid_from 날짜**입니다 — 이건 rewrite에 안 딸려가는 내용 기반 신호입니다. 또 초기 스냅샷 문서(skills 41 / agents 12 / memory 257 / solutions 104)와 현재(59/17/579/152)를 비교하면 성장 궤적이 일관됩니다. 숫자를 파는 게 아니라 "어떤 숫자는 신뢰하고 어떤 숫자는 신뢰하지 말라"까지 구분해서 파는 게 제 방식입니다.

**one-liner**: 커밋 날짜는 repo 분리 과정의 rewrite 아티팩트라 신뢰 안 하고, 타임라인은 retro/박제 메타 같은 내용 기반 신호로 교차검증합니다.

## Q10. "실패하면요? 훅이나 MCP 서버가 죽으면 개발이 아예 막히는 single point of failure 아닌가요?"

**함정**: 자동화 의존 = 취약성. fallback 부재를 의심.

**강한 답변**:
fail-open과 fallback을 설계에 넣었습니다. **훅은 fail-open**입니다 — `done-lie` 훅은 신호가 모호하면 통과시키도록 강하게 fail-open, recall 훅은 non-blocking(exit 0)이라 실패해도 세션을 안 막습니다. 진짜 차단(exit 2)은 안전 footgun에만 걸리므로 훅이 죽어도 개발이 멈추는 게 아니라 게이트가 느슨해질 뿐입니다. **MCP 서버는 다층 fallback**입니다 — MCP `search_memory`가 안 되면 셸 폴백인 `recall.sh`(852줄)가 memory + docs + git log를 통합 검색합니다. MCP 서버 자체도 esbuild 단일 번들(`dist/index.js`가 git 커밋됨)이라 node만 있으면 npm install 없이 실행되어 의존성 실패 표면이 최소입니다. `wiki_query`는 shell 의존이 없는 self-contained라 worktree 격리 에이전트에서도 동작합니다. **탐색 도구도 폴백**: codegraph가 안 되면 부분 Read로 degrade합니다. 즉 자동화가 죽으면 수동으로 우아하게 내려앉지, 전부 멈추지 않습니다.

**one-liner**: 훅은 fail-open, MCP는 셸 폴백(recall.sh)과 단일 번들로 degrade — SPOF가 되지 않도록 우아한 저하를 설계했습니다.

**함정**: 축적 = 노이즈 증가 = 검색 품질 저하. 스케일 한계.

**강한 답변**:
정확히 그 문제를 지식 그래프 최신 연구로 제도화했습니다. **200줄 budget**: Claude Code가 200줄 이후 자동 로드를 truncate하므로, 활성 박제만 인덱스(MEMORY.md)에 올리고 완료/이력성은 MEMORY-ARCHIVE.md로 이동합니다 — 파일은 보존되어 JIT 검색으로 계속 surface됩니다(ADR-004). **canonical 허브**: 근접중복 클러스터는 `*-canonical` 허브 한 줄로 등록하고 변종은 허브가 링크합니다(contextual-retrieval 경량판). **decay 추적**: recall이 매칭 slug의 access count와 last_used를 사이드카 JSON에 기록하고(박제 본문 미변경 → rebase 충돌 회피), 안 쓰이고 오래된 박제를 ARCHIVE 후보로 제안합니다. **BM25 랭킹**: 검색은 전량 로드가 아니라 IDF+길이 정규화로 top-N만 반환. **bi-temporal 메타**: valid_from / invalidated_by / superseded_by(Graphiti·Zep 차용)로 오래된 지식을 삭제 대신 무효 마킹해 stale를 격하합니다. 즉 인덱스는 200줄로 유지되고 파일은 무한 축적되는 분리 구조입니다.

**one-liner**: 인덱스는 200줄 budget + canonical 허브 + decay로 유한하게, 파일은 무한 축적하되 BM25 JIT로만 꺼냅니다 — 축적과 검색을 분리했습니다.

**함정**: 도메인 특화 = 이식 불가 = 일반화 실패.

**강한 답변**:
두 층으로 분리돼 있어서 이식 가능합니다. **도메인 특화 층**(카셰어링 도메인 온톨로지, RIBs/ReactorKit 레이어 규칙, silent-nil 계약 감사)은 이 앱에 묶이지만, **메커니즘 층**은 도메인 무관입니다 — 격상 사다리, 박제 auto-PR 파이프라인, Generator-Critic 반증우선 검증, Wilson 신뢰구간 회귀셋, 200줄 budget + BM25 JIT, 훅 기반 fail-open 게이트, 모델 라우팅 원칙. 이 메커니즘들은 어떤 iOS 팀에도, 나아가 어떤 코딩 에이전트 워크플로에도 이식됩니다. 실제로 경계 매트릭스(harness vs app repo 분리)가 이 이식성을 강제합니다 — 앱 종속 자산과 하네스 메커니즘을 물리적으로 다른 repo에 두니까, 메커니즘만 떼어 새 프로젝트에 심을 수 있습니다. 면접 관점에서 제가 재사용 가능한 자산으로 파는 건 카셰어링 도메인이 아니라 "**LLM 에이전트를 프로덕션에 안전하게 태우는 측정 기반 방법론**"입니다.

**one-liner**: 도메인 특화 층(온톨로지/RIBs)과 도메인 무관 메커니즘 층(격상 사다리/auto-PR/신뢰구간 게이트)이 repo 경계로 분리돼 있어, 메커니즘은 어떤 에이전트 워크플로에도 이식됩니다.

## Q20. "박제를 쌓는 건 알겠는데, 정작 필요할 때 안 꺼내지면 무용지물 아닌가요? '쌓기'와 '쓰기'는 다른 문제인데."

**함정**: 대부분의 지식 시스템이 실패하는 지점(축적은 되나 활용 실패)을 정확히 지적.

**강한 답변**:
정확히 그게 이 시스템의 핵심 차별점입니다 — **축적↔활용 폐루프를 훅으로 닫았습니다**. 대부분의 지식 시스템은 쌓기는 하지만 "필요 순간에 자동으로 꺼내기"에 실패합니다. 사람이 "혹시 전에 이런 적…"을 기억해야 하니까요. 여기선 `recall-on-error` 훅이 세 신호를 자동 감지합니다: ① 에러 패턴(BUILD FAILED / Exception / non-zero exit) ② 같은 command 10분 내 3회 반복 ③ 탐색 churn(grep/find 3회+ 또는 빈 결과). 감지되면 recall.sh --json으로 top-3 박제를 system-reminder로 주입합니다 — "추측·재시도 전에 이 박제부터 확인하고 적용." 즉 개발자나 에이전트가 기억할 필요 없이, 막히는 순간 관련 지식이 튀어나옵니다. UserPromptSubmit 단계의 `recall-on-prompt`도 프롬프트 관련 과거 박제를 선제 surface합니다. 활용을 사람 의지에 맡기지 않고 **훅으로 강제**한 게 novel한 지점입니다. 비전 주석 그대로 "에이전트가 막히면 백과사전 찾듯 검색 → 학습 → 즉시 적용"입니다.

**one-liner**: 활용을 사람 기억에 맡기지 않고 recall-on-error 훅이 에러·반복·탐색churn을 감지해 관련 박제를 자동 주입합니다 — 쌓기와 쓰기의 간극을 훅으로 메웠습니다.

## Q21. "auto-merge 파이프라인이 Bitbucket 인증에 의존하네요. 인증 하나 틀어지면 전체가 멈추는 취약점 아닌가요?"

**함정**: 외부 의존성(Bitbucket REST) = 취약점.

**강한 답변**:
그 취약점을 실제로 겪었고 fail-fast로 방어했습니다. 함정은 REST API가 username 자리에 email을 요구한다는 것입니다(git push HTTPS username과 다름). 과거 default가 잘못돼 401 사고가 났고, 그래서 default를 **제거**하고 `resolveUsername()`이 email 형식(@ 미포함)이 아니면 즉시 상세 안내와 함께 throw하도록 했습니다 — 가장 흔한 401 원인을 진단 메시지로 표면화합니다. 401/403/409 각각 명시 에러를 냅니다. 그리고 인증이 실패해도 **로컬 파일 쓰기와 인덱스 갱신은 이미 완료된 상태**이고, PR/merge만 실패합니다 — try/finally가 main으로 복귀시키고 orphan 브랜치를 정리합니다. 즉 인증 실패는 "팀 전파 지연"이지 "지식 유실"이 아닙니다. 로컬 박제는 남아 있고, 검색(search_memory/recall.sh)은 인증 불필요한 로컬 파일 기반이라 계속 동작합니다. 인증은 전파 채널의 문제일 뿐 시스템 전체의 SPOF가 아닙니다.

**one-liner**: 인증 실패는 401 fail-fast로 진단되고, 로컬 박제·로컬 검색은 인증 없이 동작하므로 "전파 지연"이지 "지식 유실"이 아닙니다.

## Q22. "결국 이 모든 게 당신 개인 워크플로에 최적화된 거고, 다른 사람은 이 규율을 안 지킬 텐데요. 강제가 되나요?"

**함정**: 규율 의존 시스템은 사람이 안 지키면 무너진다. 사회적 강제 실패.

**강한 답변**:
그래서 규율을 **문서가 아니라 코드로 강제**한 게 핵심 설계입니다. "메모리 룰"만 있으면 LLM이든 사람이든 깜빡합니다 — 그래서 격상 사다리 4단계가 반복 실패를 hook 코드 게이트로 승격합니다. 실례: 박제 직접 쓰기 금지는 부탁이 아니라 file-protection.sh의 exit 2 차단이고, AI 코드 "왜" 주석은 posttooluse 훅이 exit 2로 되돌리며, 광역 grep footgun은 grep-hygiene 훅이 차단하고, 빌드 깨진 채 done 거짓말은 done-lie 훅이 종료를 막습니다. 이건 개인 규율에 의존하지 않습니다 — 사용자가 잊거나 무시해도 훅이 행동 레벨에서 강제합니다. 게다가 이 규율은 개인 세션만이 아니라 **worktree 격리 서브에이전트에도 적용**됩니다(preflight-safety가 세션 훅 우회 커밋을 코드 레벨로 재차단). 규율이 사람 의지가 아니라 라이프사이클 훅에 배선돼 있으니, "누가 안 지킨다"는 곧 "exit 2로 막힌다"입니다.

**one-liner**: 규율을 문서가 아니라 exit 2 훅 게이트로 코드화해서, 사람이 잊거나 무시해도 행동 레벨에서 강제됩니다 — 개인 의지에 의존하지 않습니다.
