# 카셰어링 앱 Reverse System Design — 풀 워크스루

> **포맷**: 모바일 시스템 디자인 면접(5~10분 구두 답변)을 문서화한 것.
> **관점**: "제가 만든(=운영/기여한) 시스템"으로 아키텍처를 재구성해 설명.
> **대상 시스템**: 전국 단위 카셰어링/모빌리티 iOS 앱 (약 320K LOC, 50개 SPM 모듈, 화면 100개+, RIBs + ReactorKit + RxSwift).
> **트레이드오프 언어**: 되돌리기 어려운 결정(one-way door) vs 언제든 되돌릴 수 있는 결정(two-way door)을 매번 소리내어 구분.

---

## 0. 오프닝 — 30초 프레이밍

"이 앱은 단순 예약 앱이 아니라 **물리 세계(차량 하드웨어)와 실시간으로 결합된 모빌리티 플랫폼**입니다. 소프트웨어 버그가 '앱이 하얗게 뜬다'가 아니라 '주차장에서 차 문이 안 열린다'는 현장 장애가 되는 도메인이에요. 그래서 시스템 디자인의 핵심 축은 세 가지입니다 — (1) **BLE 디지털 카키(차량 제어)의 실시간 암호화 통신**, (2) **다중 진입 채널(딥링크)이 100개 넘는 화면으로 수렴하는 네비게이션 상태머신**, (3) **50개 모듈을 4레이어로 쪼갠 모듈러 모노레포로 복잡도를 의도적으로 구조화**한 것. 이 세 축을 중심으로 요구사항부터 딥다이브까지 훑겠습니다."

면접 프레임 순서대로 갑니다: ①요구사항 → ②기능 → ③비기능 → ④데이터모델 → ⑤API → ⑥하이레벨 → ⑦딥다이브 → ⑧트레이드오프 → ⑨마무리.

## ① 요구사항 정의 (Requirements)

먼저 스코프를 좁힙니다. 카셰어링 앱은 비즈니스 라인이 폭발적이라, 면접에서라면 "어디까지 다룰까요?"를 먼저 확인하고 들어갑니다.

**기능 요구사항 (무엇을 하는가)**
- 회원/게스트가 차량을 **예약 → 운행 → 반납 → 정산**하는 풀 사이클
- **디지털 카키**: 예약한 차량을 앱으로 잠금/해제/트렁크/비상등·경적 제어 (BLE 근거리 + 서버 원격 2경로)
- **다양한 렌탈 상품**: 단기, 편도 렌탈(출발지≠반납지), 편도 특가딜, 차량 배달/픽업, 구독형 패스
- **위치 기반 매물 탐색**: 지도에서 근처 차량/특가 매물 찾기, 반납 위치 확정
- 로그인(소셜 4종 + 자체), 회원가입, 신분증/카드 **OCR** 등록, 기업(B2B) 라인
- **딥링크 진입**: 푸시/어트리뷰션/유니버설링크/웹뷰/소셜콜백 → 특정 화면 직행

**논스코프 (명시적으로 뺀 것)**
- 결제 PG 자체 로직 — 웹뷰(PG) + 서버 책임으로 위임. 앱은 카드 등록 + 정산 내역 표시만.
- 백엔드 매칭/요금 계산 알고리즘 — 서버 영역.

이 스코프 확정이 **two-way door**입니다. 면접관이 "카키 빼고 예약만 보자"고 하면 언제든 좁힐 수 있어요. 반대로 아래 나올 **모듈 4레이어 분리, RIBs 채택**은 한번 정하면 320K 라인을 되돌려야 하는 **one-way door**입니다.

## ② 기능 상세 (Core Features)

가장 복잡도가 높은 순으로 3개만 깊게:

**1) 디지털 카키 (BLE 차량 제어)** — 최고 난도.
앱이 차량 하드웨어와 직접 통신합니다. 자체 BLE 패킷 프로토콜 스택을 구현했어요 — 커스텀 프레이밍(STX/ETX, 길이·목적지·메시지타입·응답코드 파싱), AES-CBC 암호화 채널, 핸드셰이크로 세션 수립, LRC 오류검출. 제어 명령은 비트마스크(잠금/해제/트렁크/경적/비상등), 재시도 3회, 재연결 디바운스. GATT UUID 3종(Service/Write/Notify) 고정. 별도 텔레매틱스 SDK로 **서버 원격제어 2번째 경로**도 병존 — BLE가 실패해도 서버로 문을 열 수 있게.

**2) 렌탈 상품 라인 (도메인 폭증)** — 편도/편도특가딜/배달·픽업/구독/기업/게스트가 한 앱에. 각기 예약~반납 흐름이 달라서, 편도 라인만 봐도 `편도` + `편도특가딜` **2벌의 4레이어 모듈 세트**로 통째 복제돼 있습니다. 비즈니스 라인별 도메인 격리를 위해 의도적으로 중복을 감수한 거예요.

**3) 딥링크 네비게이션** — 5개 진입 채널(외부 URL스킴/유니버설링크/푸시/어트리뷰션 SDK/소셜콜백)이 **120개+ 목적지 enum(`AppMenu`)**으로 수렴. 뒤 ⑦에서 딥다이브합니다.

## ③ 비기능 요구사항 (Non-Functional) — 모바일 특화

여기가 모바일 SD의 진짜 승부처입니다. 웹과 달리 **오프라인/실시간/배터리/하드웨어**를 다뤄야 해요.

| 축 | 요구 | 이 앱의 대응 |
|---|---|---|
| **실시간성** | 차 문 열림은 즉각적, 지연 = 현장 장애 | BLE 근거리 직접 제어 + 서버 원격 폴백 2경로 |
| **오프라인** | 지하주차장·통신 음영에서도 카키 동작 | BLE는 네트워크 불필요(근거리). 단, 데이터 캐싱은 최소 (뒤 트레이드오프에서 자백) |
| **보안** | 토큰·디지털키·신분증 = 탈취 시 물리 자산 위험 | 저장소 AES256 암호화, BLE 채널 AES-CBC, OCR 온디바이스 |
| **위치 실시간** | 근처 매물/반납위치 실시간 | 위치확정 UI를 독립 모듈로 분리할 만큼 비중 큼 |
| **크로스플랫폼 parity** | AOS와 동시 개발, 서버 계약 공유 | 서버 payload 포맷/상태코드를 양 플랫폼이 공유, 프로세스로 parity 담보 |
| **배터리/BLE 수명** | 백그라운드 BLE 연결 유지 | 재연결 디바운스 + 세션 상태머신 |

핵심 비기능 결정 하나를 소리내면 — **"디지털 카키는 네트워크에 의존하면 안 된다"**. 이게 BLE 근거리 통신을 앱이 직접 구현한 이유예요. 서버 원격만 있으면 통신 음영에서 차를 못 엽니다. 대가는 앱이 저수준 BLE 프로토콜·암호화·재시도를 전부 책임진다는 것. **one-way door**입니다 — 프로토콜 스택을 걷어내려면 하드웨어 펌웨어까지 바꿔야 하니까.

## ④ 데이터 모델 (Data Model)

레이어별로 데이터가 어떻게 변형되는지가 핵심입니다. **DTO → Entity → State → ViewState** 4단 변형.

```
[서버 JSON]
   │  Decodable 디코딩
   ▼
[DTO]  (응답DTO 모듈, 191개 파일)          ← 거의 전 필드 Optional
   │  init(dto:) 매핑, nil을 ?? 기본값으로 흡수
   ▼
[Entity]  (Domain 레이어)                  ← 도메인 순수 모델
   │  UseCase → Repository 경유
   ▼
[State]  (ReactorKit, SSOT)               ← 화면 내부 단일 진실
   │  reduce() 순수함수로만 변경
   ▼
[ViewState / ViewPulse]                    ← View 노출용, 내부 State 캡슐화
   │  @Pulse = 1회성 이벤트(에러토스트) 분리
   ▼
[SwiftUI View / UIKit]
```

**데이터 모델 설계 결정 3가지, 각각 트레이드오프 자백:**

**(a) 표준 응답 봉투 `BaseResponse<T>`** — 서버 공통 필드(페이징, 감사필드, `errCode`, `errMsg`, `data: T?`)를 제네릭으로 래핑. `validate()`가 `errCode != 0`이면 에러 throw, `data == nil`이면 별도 에러. → **장점**: 응답 계약 통일. **트레이드오프**: `data` 필드가 없으면 무조건 throw라, 서버에 "data 필드 꼭 내려달라"고 계약해야 함.

**(b) 전 필드 Optional + CodingKeys 미보유 (~64%)** — 이건 제가 면접에서 **솔직하게 약점으로 꺼내는** 부분입니다. DTO의 대부분 프로퍼티가 Optional이고, 191개 중 약 36%만 CodingKeys를 명시해요. 디코딩 실패가 throw가 아니라 **조용한 nil(silent-nil)**로 흡수됩니다. 서버가 키를 리네임해도 컴파일/런타임 어디서도 안 잡히고 빈 문자열/0으로 흘러가요.
→ **왜 이렇게 됐나**: 크로스플랫폼 + 잦은 서버 변경 환경에서 "일단 안 죽는" 방어적 선택. **대가**: `필드가 안 온 것`과 `의도적 기본값`이 구분 안 됨. **어떻게 방어하나**: Entity 매핑 레이어에서 `?? 기본값`으로 흡수하고, **개발 하네스(별도 플러그인)에 계약 감사 스킬**을 두어 swagger+AOS+실응답 3-oracle 대조로 CI에서 잡습니다. 즉 **코드가 아니라 프로세스로 회귀를 막는** 구조. 이건 **two-way door** — 점진적으로 CodingKeys를 채워가며 개선 중.

**(c) State / ViewState 경계 분리** — 내부 도메인 State가 View로 새지 않게 `ViewState`로 매핑. 1회성 이벤트는 `@Pulse`로 분리. → State 재방출 시 에러 알럿이 중복으로 뜨는 고전적 Rx-MVI 버그를 구조적으로 차단. 성숙도 신호입니다.

## ⑤ API / 네트워킹 (API Design)

**타입 세이프 요청 계약 — `Requestable` 프로토콜:**

```swift
protocol Requestable: Hashable {
  associatedtype Output: Decodable   // ← 응답 DTO를 컴파일타임 바인딩
  var endpoint: URL { get }
  var method: HTTPMethod { get }
  var query: QueryItems { get }
  var header: HTTPHeader { get }
}
```

각 API가 이 프로토콜을 채택한 struct. `Output` associatedtype로 **요청↔응답이 컴파일타임에 묶여** 타입 세이프합니다.

**전송 계층**: Alamofire를 async/await로 감싸고, 그 위에 **Rx 브릿지**를 얹었습니다. 이게 이 앱 네트워킹의 핵심 설계 결정이에요.

```
[Requestable]
   │
   ▼
async send() ──(Alamofire, withCheckedThrowingContinuation)──► T.Output
   │
   ▼  Observable.create { Task { ... } }
[Observable<T.Output>]  ← 앱 전 계층이 Rx라 async를 다시 Rx로 끌어올림
```

**핵심 트레이드오프 3개:**

**(a) 토큰 갱신 = Rx 브릿지가 사실상의 401 인터셉터.** async send가 `.invalidToken`/`.tokenExpired`를 throw하면, Rx 래퍼가 토큰 refresh 후 **정확히 1회 재시도**. → Alamofire의 인터셉터/리트라이어를 안 쓰고 Rx 레이어에서 수동 처리한 거예요. **대가**: async 버전을 직접 쓰면 이 리트라이가 없어서, 반드시 Rx 경유해야 함. 그리고 로직이 3곳(send/sendFull/sendLicense)에 복붙돼 있어요 — 리팩터 부채로 자백.

**(b) 동시 토큰갱신 스로틀이 락이 아니라 시간창 기반.** 마지막 갱신 시각이 1초 이내면 실제 갱신 안 하고 알림만 쏨. → 간이 뮤텍스지만 경합 구간에 레이스 가능성이 있어요. 개선 여지로 인지하고 있습니다.

**(c) 네트워크 클라이언트가 화면마다 N개 생성.** 공유 싱글톤 없이 각 RIB Builder가 개별 생성 → 각 인스턴스가 토큰갱신 알림 옵저버를 등록해서 **N중 등록** 위험. → 이건 명확한 개선 포인트. 원래는 DI 루트에서 단일 인스턴스를 주입했어야 합니다.

**에러 규약**: 네트워크 실패를 Rx `onError`로 흘리지 않고 **도메인 `Result` enum(`.success` / `.fail(message)`)으로 값-레벨 처리**. 스트림이 dispose되지 않고 살아있어서 화면이 죽지 않아요. reduce에서 `.fail(msg)`를 `errorMessage`로 흡수.

## ⑥ 하이레벨 아키텍처 (High-Level Design)

### 모듈 그래프 (SPM 50개, 4레이어 수직 슬라이스)

```
┌─────────────────────────────────────────────────────────────┐
│                     App (Composition Root)                   │
│              AppComponent = 전 Repository 생성 DI 루트         │
└───────────────────────────┬─────────────────────────────────┘
                            │
        ┌───────────────────┼───────────────────┐
        ▼                   ▼                   ▼
   [Feature 수직 슬라이스]   [공통 인프라]        [파운데이션]
                            │                   │
  ┌──────────────┐    ┌──────────┐        ┌──────────────┐
  │ Presentation │    │ Analytics│        │ Network      │
  │      │       │    │ Common   │        │  ├RESTInfra  │
  │      ▼       │    │ UserIF   │        │  ├RESTAPIs   │
  │   UseCase    │    │ Location │        │  ├RESTDTO    │
  │      │       │    │ AddrSrch │        │  └RESTMock   │
  │      ▼       │    │ OCR      │        │ Persistent   │
  │  Repository  │    │ WebView  │        │  (AES256)    │
  │      │       │    └──────────┘        │ Domains(공통)│
  │      ▼       │                        └──────────────┘
  │   Domain     │
  └──────────────┘
```

**이상적 의존성 방향**: `Presentation → UseCase → Repository → Domain`, 공통(Persistent/Analytics/Common/UserInterface)은 전 레이어 접근 가능.

Feature별로 이 4레이어를 수직 슬라이스로 물리 분리했습니다. 예: `편도` 피처는 `편도Domain / 편도Repository / 편도UseCase / 편도Presentation` 풀세트. 성숙한 피처일수록 완전 분리되고, 신생 피처는 단일 모듈로 시작 — **granularity가 피처 성숙도에 비례**합니다.

### 레이어 심층 — RIBs 트리 구조

네비게이션·DI는 Uber RIBs로, 화면 내 상태는 ReactorKit으로 합니다. **하이브리드**예요.

```
AppRoot
 └ Bridge (메인 탭 허브)
    ├ HomeTab
    ├ 편도 → SearchFind / LocationFind / ReturnLocation / CarSelect / CheckVehicle
    │        └(인증 서브플로우) LoginSelect / AccountRegist / IDCardRegist / SMSCert
    ├ 편도특가딜 → List / Detail / RouteCheck / Alarm / Filter
    ├ 배달·픽업 → LocationFind / LocationConfirm
    ├ 디지털카키 / 이용내역 / Menu / 알림함
```

각 RIB = **Router(네비) + Interactor(로직/상태) + Builder(DI)** 3종 세트. Router가 자식 Builder를 주입받아 `attachChild`/`detachChild`로 트리를 동적으로 조립·해제합니다. 화면 수 프록시로 Builder 113개, Interactor 111개, ViewController 140개.

### 데이터 흐름 — 단방향 사이클

```
[View] 사용자 탭
   │ listener.sendAction(.didTapBell)      ← View는 인텐트만 발사, State 직접변경 불가
   ▼
[Interactor = Reactor]  action Subject
   │ mutate(action) → useCase 호출 → Observable<Mutation>   ← 사이드이펙트/라우팅 여기서
   ▼
[Mutation]
   │ reduce(state, mutation) → State        ← 순수함수, 변경 지점 단 1곳
   ▼
[State (SSOT)]
   │ state.map { ViewState } / { ViewPulse }
   ▼
[ViewController] 구독 → viewModel.@Published 갱신
   ▼
[SwiftUI View] 자동 리렌더
```

**여기서 제가 강조하는 설계 정체성**: RIBs Interactor 한 클래스가 **동시에 ReactorKit Reactor**입니다. 별도 Reactor 클래스를 안 두고 `PresentableInteractor`가 `Reactor` 프로토콜을 겸해요. 이걸 **Xcode 파일 템플릿에 박제**해서 21개 화면에서 일관되게 강제했습니다. 팀 스케일에서 아키텍처를 지키는 방법 — 규율에 의존하지 않고 템플릿으로 강제.

## ⑦ 딥다이브 (Deep Dive) — 딥링크 네비게이션 상태머신

면접에서 "가장 어려웠던 한 곳을 깊게"라고 하면 저는 **딥링크**를 팝니다. BLE는 화려하지만, 시스템 전체를 흔드는 복잡도는 딥링크예요.

**문제**: 5개 진입 채널이 120개+ 화면으로 수렴하는데, 진입 시점의 앱 상태(스플래시 중? 게스트? 로그인 안 됨?)와 충돌하면 안 됩니다.

```
[외부 URL스킴]  [유니버설링크]  [FCM 푸시]  [어트리뷰션 SDK]  [소셜 로그인 콜백]
      │              │            │             │                  │
      └──────────────┴────────────┴─────────────┴──────────────────┘
                              │  전부 같은 open(url) 창구로 유입
                              ▼
                    ┌──────────────────────┐
                    │  정규화 파이프라인      │
                    │ 1. 앱 기본스킴으로 치환 │
                    │ 2. 어트리뷰션 접두 제거 │
                    │ 3. in-app vs 외부 분기  │
                    └──────────┬────────────┘
                              ▼
                    ┌──────────────────────┐
                    │  상태 게이팅           │  ← 핵심 방어 로직
                    │ 스플래시/게스트/인트로  │
                    │ 상태 검사 → 화면 정리   │
                    └──────────┬────────────┘
                              ▼
                    ┌──────────────────────┐
                    │  Workflow 라우팅 (76개)│
                    │ path prefix 매칭 →     │
                    │ actionable item 트리    │
                    │ 를 비동기 walk          │
                    └──────────┬────────────┘
                              ▼
              [AppMenu enum 120+ 목적지 중 하나로 도착]
                (associated value: 위경도·rentSeq·필터 등)
```

**딥다이브 포인트 3개:**

**(1) 정규화 — 이질적 소스를 단일 내부 스킴으로.** 어트리뷰션 SDK가 붙인 접두(`xxx.attribution.io/`)를 벗기고, 소셜 콜백(카카오/구글/네이버)까지 같은 창구에서 분기. → 진입 채널이 늘어도 **하류 라우팅 코드는 안 바뀌게** 정규화 레이어로 격리. 이건 **two-way door** — 새 채널 추가가 값싸요.

**(2) 상태 게이팅 — 딥링크가 앱 상태와 싸우지 않게.** 스플래시 중에 딥링크가 오면? 게스트인데 회원 전용 화면을 요구하면? 이 방어 로직이 두껍습니다. `/guest` 경로는 아예 별도 게스트 Workflow로 분기. → 여기서 제가 실제로 짠 회귀 방지 플래그들이 있어요 — `didApplyDeepLinkFilter` 같은. **단방향 흐름 + 낙관적 로컬 갱신을 섞으면** State 재방출로 서버값이 되살아나는 회귀가 생기는데, 이걸 1회성 가드 플래그로 막았습니다. 이게 **하이브리드 아키텍처의 대가**예요 — QA 티켓 번호까지 주석에 박제해서 왜 이 플래그가 필요한지 남겼습니다.

**(3) Workflow = RIBs 비동기 라우팅.** 각 Workflow가 `init?(url:)`에서 path를 검증(안 맞으면 nil), `onStep { rootItem.waitBridge() }.onStep { moveMenu() }.commit()`으로 actionable item 트리를 비동기 walk. → 딥링크 표면을 **UI 테스트 진입점으로도 재사용**(별도 UITestRouter). 딥링크가 QA 자동화의 축이 됐어요.

**트레이드오프 자백**: 120개 enum case는 관리 부담이 큽니다. 파일 상단에 "각 화면이 어떤 URI로 진입하는지" 수동 문서화가 필요할 정도예요. 이걸 데이터 드리븐(라우팅 테이블)으로 뺄 수도 있었지만, associated value의 타입 안전성을 위해 enum을 유지했습니다. **되돌리기 어려운(one-way) 선택은 아니지만** 리팩터 비용이 큰 상태.

## ⑧ 트레이드오프 종합 (Trade-offs) — 소리내어 자백

면접에서 제일 점수 따는 구간. **약점을 먼저 꺼내고, 왜 그 선택을 했는지 방어**합니다.

| 결정 | 선택 | 트레이드오프 | door |
|---|---|---|---|
| **상태관리** | RIBs+ReactorKit+Rx 3중 결합 | 학습곡선 가파름, 보일러플레이트 많음 ↔ 대규모 팀에서 단방향 강제 | one-way (핵심) |
| **Rx vs async** | async는 경계에만, 즉시 Rx 재흡수 | "전면 마이그레이션 안 함"의 명시적 선택 ↔ 일관성 유지 | two-way (점진 가능) |
| **UIKit vs SwiftUI** | 신규는 SwiftUI+ViewModel, 레거시는 UIKit+RxCocoa 병존 | 2세대 바인딩 혼재 ↔ 점진 전환 | two-way (진행 중) |
| **DTO 계약** | 전 필드 Optional + silent-nil | 서버 변경에 안 죽음 ↔ 계약 위반이 조용히 통과 | two-way (프로세스로 방어) |
| **캐싱/오프라인** | remote-only, SSOT 사실상 없음 | 단순함 ↔ 오프라인 미지원, 네트워크 의존 | two-way |
| **영속성** | UserDefaults+AES256, Keychain 미사용 | 구현 단순 ↔ 키/IV 하드코딩, 토큰 보안 약함 | **개선 필요 (자백)** |
| **DI** | 컨테이너 없이 RIBs 생성자 주입 | 컴파일타임 그래프 강제 ↔ Component 비대, 클라이언트 N개 생성 | two-way |
| **모듈 분리** | 50개 SPM 4레이어 | 빌드 유닛 격리, 병렬 개발 ↔ 문서-코드 갭, 스텁 유령 모듈 | one-way |

**가장 정직하게 꺼내는 2개:**

**(a) 영속성에 Keychain을 안 쓰고 UserDefaults+AES256를 썼다.** 게다가 AES 키/IV가 소스에 하드코딩. → "왜 이렇게 됐고, 어떻게 고칠 건가"를 설명할 수 있으면 오히려 플러스입니다. 토큰은 Keychain으로 옮기는 게 정석. 이건 레거시 결정이고 개선 대상으로 인지 중.

**(b) Clean Architecture를 지향했지만 실제 의존성 그래프는 엄격한 단방향이 아니다.** Domain이 Network/DTO를 역참조하고, Presentation이 UseCase를 우회해 Repository를 직접 import하는 지점이 있어요. → 이걸 아는 게 중요합니다. **그래서 개발 하네스에 결합도 감사 스킬(code-analyzer/coupling-analysis)을 상시 두어** 레이어 위반을 CI로 감시합니다. "이상과 현실의 갭을 인지하고 자동화로 방어한다"는 게 시니어의 신호예요.

## ⑨ 마무리 (Wrap-up) — 30초 클로징

"정리하면, 이 앱의 시스템 디자인은 **세 가지 원칙**으로 요약됩니다.

1. **물리 세계와의 실시간 결합을 최우선** — 디지털 카키를 네트워크 비의존 BLE로 직접 구현하고 서버 원격을 폴백으로 둔 것. 오류 허용치가 낮은 도메인이라 이중화가 필수였습니다.

2. **복잡도를 우발적이 아니라 의도적으로 구조화** — 8개 비즈니스 라인을 50개 모듈 4레이어로 물리 분리하고, RIBs+ReactorKit 단방향 흐름을 템플릿으로 강제해 규모를 억제했습니다.

3. **이상과 현실의 갭을 자동화로 방어** — DTO silent-nil, 레이어 위반 같은 구조적 부채를 코드로 완벽히 막지 못하는 지점은, 개발 하네스의 계약 감사·결합도 감사 스킬로 프로세스 레벨에서 잡습니다.

만약 처음부터 다시 짠다면 세 가지를 바꾸겠습니다 — **토큰을 Keychain으로**, **네트워크 클라이언트를 DI 루트 단일 인스턴스로**, **DTO에 CodingKeys를 강제**. 나머지 큰 뼈대(RIBs+모듈 분리)는 이 규모에선 여전히 옳은 선택이라고 봅니다."

### 부록 — 면접관 예상 꼬리질문 & 대비

- **"Rx를 왜 안 걷어내나?"** → async는 구현 디테일, Rx는 아키텍처 계약. 전 계층 단방향 파이프라인의 일관성이 마이그레이션 비용보다 가치가 큼. 경계(네트워크/검증)만 async, 즉시 Rx 재흡수.
- **"모듈이 정말 50개 필요한가?"** → 성숙도 기반 granularity. 신생 피처는 단일 모듈로 시작, 성숙하면 4레이어 분리. 빌드 유닛 격리로 병렬 개발 + 컴파일 격리 이득.
- **"BLE 실패하면?"** → 재시도 3회 + 서버 원격제어 2번째 경로 + LRC 오류검출. "문 안 열림"은 즉각 현장 장애라 이중화 필수.
- **"오프라인 지원 없는 게 문제 아닌가?"** → 카키는 BLE라 오프라인 동작. 데이터 화면은 remote-only가 맞음(실시간 매물/요금은 캐시하면 오히려 위험). 필터 목록만 인메모리 read-through 캐시.
- **"팀에서 아키텍처를 어떻게 지키나?"** → 규율이 아니라 강제. Xcode 템플릿 박제 + swiftlint force-unwrap error + 하네스 CI 감사 스킬.
