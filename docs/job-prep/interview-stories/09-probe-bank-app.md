## A. 아키텍처 선택의 정당화 ("왜 X 안 썼나")

### Q1. RIBs를 왜 골랐나? MVVM-C나 TCA로 갔으면 더 단순했을 텐데.

**함정 포인트**: "Uber가 써서" 같은 권위 답. 또는 RIBs 단점(보일러플레이트)을 인정 못 하는 방어.

**강한 답변**:
"세 가지 도메인 제약이 RIBs를 밀었습니다. 첫째, 이 앱은 **화면이 100개 넘고 진입 경로가 딥링크·푸시·유니버설링크·소셜로그인·웹뷰 5채널**로 폭발합니다. RIBs는 비즈니스 로직(Interactor)과 뷰(ViewController)를 분리하고, **화면이 없는 로직 전용 RIB**을 트리에 붙일 수 있어서 '로그인 여부 게이팅', '게스트 워크플로' 같은 뷰 없는 흐름을 1급 시민으로 다룹니다. MVVM-C의 Coordinator는 이걸 뷰 밖 별도 객체로 억지로 빼야 해요.

둘째, **부모-자식 생명주기와 DI가 컴파일 타임에 강제**됩니다. `attachChild`/`detachChild`로 자식 RIB의 메모리 수명이 트리에 묶이고, `Dependency` 프로토콜을 부모가 만족 못 시키면 빌드가 깨집니다. 50모듈 30명 팀에서 이 '누락하면 컴파일 실패'가 런타임 크래시보다 훨씬 쌉니다.

셋째, TCA는 당시(그리고 지금도) 대규모 레거시 UIKit 코드베이스에 점진 도입이 어렵습니다. 우린 이미 RxSwift가 2,191개 `disposed(by:)`로 깔린 상태였고, ReactorKit이 Rx 위에서 단방향 상태를 이미 주고 있었어요. RIBs는 이 Rx 백본과 자연스럽게 맞물립니다.

**단점은 인정합니다**: 화면 하나당 Builder/Interactor/Router/ViewController 4파일 보일러플레이트. 그래서 우린 이걸 **Xcode 파일 템플릿에 박제**해서 사람이 손으로 안 짜게 했습니다. 보일러플레이트를 '자동화로 상쇄'한 거지, 없앤 척한 게 아닙니다."

**one-liner**: "RIBs는 뷰 없는 흐름·컴파일타임 DI·Rx 정합 세 가지를 동시에 줘서, 진입경로가 폭발하는 대형 모빌리티 앱에는 MVVM-C/TCA보다 총비용이 낮았고, 보일러플레이트는 템플릿으로 상쇄했습니다."

### Q2. Rx를 왜 안 걷어내나? Swift Concurrency가 표준인데 기술 부채 아닌가?

**함정 포인트**: "마이그레이션 중입니다"라고 얼버무리기 (findings상 마이그레이션 흔적 없음 = 거짓 방어). 또는 async/await를 무조건 우월하다고 단정.

**강한 답변**:
"명시적으로 **안 걷어내는 쪽을 선택**했습니다. 이건 부채 방치가 아니라 결정입니다.

전략은 '**async는 구현 디테일, Rx는 아키텍처 계약**'입니다. 하부 네트워크 전송은 이미 async/await로 현대화했어요 — Alamofire를 `withCheckedThrowingContinuation`으로 감싸고, 토큰 만료 → refresh → 재요청도 async로 자연스럽게 씁니다. 그런데 그 async 결과를 **즉시 `Observable.create` + `Task`로 Rx 세계로 재브릿지**합니다.

왜냐면 앱의 상태관리 계약이 ReactorKit 단방향 파이프라인(Action→Mutation→State)이고, 이게 21개 화면에 템플릿으로 박혀 있기 때문입니다. 여기에 async 스트림을 절반만 섞으면 **두 개의 서로 다른 동시성 모델이 한 Interactor 안에서 충돌**합니다 — 취소 전파, 스케줄러, 백프레셔가 다 달라져요. 그건 진짜 부채입니다.

그래서 우린 async를 **네트워크 경계와 복잡한 순차 검증 두 군데에만 국소 도입**하고 (`await MainActor.run` 후 `observer.onNext`로 Mutation 재주입), 나머지는 Rx로 통일했습니다. Combine은 import만 6파일, 사실상 미사용 — 세 번째 모델을 안 들이려고 의도적으로 뺐습니다.

전면 마이그레이션의 ROI를 계산해보면, 2,191개 구독 지점을 async로 바꾸는 리스크(취소·라이프사이클 회귀) 대비 얻는 게 '최신 문법'뿐입니다. **동작하는 일관성 > 유행하는 파편화**."

**one-liner**: "Rx는 걷어낼 부채가 아니라 유지할 계약입니다 — async는 네트워크·검증 경계에만 국소 도입 후 Rx로 재흡수해서, 한 코드베이스에 동시성 모델이 두 개 공존하는 진짜 부채를 막았습니다."

### Q3. UIKit과 SwiftUI가 섞여 있다. 왜 한쪽으로 안 통일했나?

**함정 포인트**: "점진 마이그레이션이라"만 반복. 하이브리드의 비용(2세대 바인딩 혼재)을 숨기기.

**강한 답변**:
"현실적 제약과 리스크 관리입니다. **레거시 화면은 UIKit + RxCocoa 직결**(VC가 `.rx`로 직접 바인딩), **신규/리뉴얼 화면은 SwiftUI + `@MainActor ObservableObject` ViewModel**을 `UIHostingController`로 감싸 RIBs에 얹습니다.

핵심은 **RIBs 트리를 갈아엎지 않고 뷰 레이어만 교체 가능**하게 만든 겁니다. RIBs ViewController가 얇은 브릿지가 되고, Interactor(=Reactor)의 `viewState`/`viewPulse` Observable을 구독해서 SwiftUI ViewModel의 `@Published`를 갱신합니다. 즉 상태 계약은 그대로 두고 렌더링만 SwiftUI로 바꾸는 거예요. 100개 화면을 한 번에 SwiftUI로 리라이트하는 건 회귀 지옥입니다.

**하이브리드의 대가는 정직하게 인정**합니다: 단방향 흐름 + SwiftUI 낙관적 로컬 갱신을 섞으면 'State 재방출 시 서버값이 되살아나는' 회귀가 납니다. 우린 이걸 `didApplyInitialRegions`, `hasUserResetRoute` 같은 **1회성 가드 플래그로 방어**하고, 주석에 QA 티켓 번호까지 박아뒀어요. 이게 하이브리드의 세금이고, 우린 그걸 구조적으로(플래그 + 코드리뷰) 관리합니다."

**one-liner**: "상태 계약(RIBs/ReactorKit)은 고정하고 렌더링만 화면 단위로 SwiftUI 전환 — 100화면 일괄 리라이트의 회귀 리스크를 피하고, 하이브리드 세금은 1회성 가드 플래그로 격리했습니다."

### Q4. DI 프레임워크(Swinject 등)를 왜 안 썼나? 수동 생성자 주입은 안 지겨운가?

**함정 포인트**: "필요 없어서"라는 게으른 답. 수동 배선의 비대함(Component가 8개 자식 Dependency conform)을 방어 못 함.

**강한 답변**:
"의도적으로 **컨테이너를 안 썼습니다**. RIBs의 `Component`/`Builder` 생성자 주입만 씁니다.

이유는 **컴파일 타임 안전성**입니다. Swinject 같은 서비스 로케이터는 의존성 해석이 런타임이에요 — 등록 안 한 타입을 resolve하면 런타임 크래시입니다. 30명이 붙는 50모듈에서 그건 프로덕션에서 터질 지뢰입니다. 반면 RIBs 생성자 주입은 **부모 `Dependency` 프로토콜을 만족 못 하면 빌드가 안 됩니다**. 의존성 그래프가 타입 시스템으로 강제돼요.

Composition Root도 단일화했습니다 — 앱 최상단 `AppComponent`가 `Component<EmptyDependency>`로 전 Repository를 만드는 루트 컨테이너입니다. 그래프의 진실은 한 군데 있어요.

**비대함은 인정**합니다: 부모 Component가 자식 8개의 Dependency 프로토콜을 동시에 conform하면서 커집니다. 이건 RIBs의 알려진 세금이에요. 하지만 이 비대함은 **'컴파일러가 검증하는 명시적 비대함'**입니다 — 그래프가 코드에 다 드러나 있어서 신입이 읽으면 뭐가 뭘 필요로 하는지 보입니다. 매직(런타임 자동 해석)의 편함보다, 명시(컴파일 강제)의 추적성을 택했습니다."

**one-liner**: "컨테이너의 런타임 해석 크래시 대신 RIBs 생성자 주입의 컴파일타임 강제를 택했습니다 — Component가 비대해지는 세금은 있지만, 그건 컴파일러가 검증하는 '보이는 비대함'이라 30명 팀에서 추적 가능합니다."

## B. 확장성 / 팀 스케일 ("팀 커지면 버티나")

### Q5. 30명이 50모듈을 만진다. 모듈 순환 의존은 어떻게 막나?

**함정 포인트**: "안 생겨요"라는 과신. 실제로 findings엔 `Common → feature Domain` 역참조라는 순환 씨앗이 있음. 이걸 모르면 감점.

**강한 답변**:
"네 가지 장치 + 하나의 감시 도구입니다.

1. **product 세분화** — 공통 Network 모듈을 4개 product(전송 인프라 / API 정의 / 응답 DTO / Mock)로 쪼갰습니다. DTO만 쓰는 Domain은 API 정의만 가져가고, 실 HTTP 전송은 인프라만 가져가요. **소비자가 필요한 표면만 의존**하니 사이클 표면이 줄어듭니다.

2. **feature Domain을 최하위 레이어로** — Repository/UseCase/Presentation이 공유하는 얇은 바닥에 둡니다.

3. **RIBs Dependency 프로토콜** — 상위가 하위에 구체 타입 대신 프로토콜 요구를 선언해서, 구체 의존이 컴파일 그래프에 직접 안 박히게 완화합니다.

4. **의도적 엣지 절단** — 실제로 무거워지는 의존은 `.package(path:)`를 **주석 처리**해서 끊어둔 흔적이 있습니다. 결합을 코드로 관리한 증거예요.

그리고 정직하게 말하면, **완벽하진 않습니다**. `Common`이 특정 feature Domain을 역참조하는 사이클 씨앗이 실재합니다 — 공통 모듈이 피처를 알면 결합이 위로 새죠. 그래서 우린 **읽기 전용 아키텍처 감사 도구(coupling/code-analyzer)를 하네스에 상시 붙여서** RIBs 레이어 위반과 SPM 사이클을 자동 탐지합니다. '사이클 없다'가 아니라 '사이클을 상시 계측하고 씨앗을 알고 있다'가 정확한 답입니다."

**one-liner**: "product 세분화 + Domain 최하위 배치 + 프로토콜 의존 + 의도적 엣지 절단으로 사이클을 줄이되, `Common→Domain` 역참조 같은 씨앗은 실재하므로 읽기전용 감사 도구로 상시 계측합니다 — '없다'가 아니라 '계측한다'."

### Q6. Clean Architecture를 표방하는데, Domain이 Network/DTO에 의존한다. 이거 위반 아닌가?

**함정 포인트**: 이걸 부정하거나 모르는 척. findings상 `OnewayDomain`, `Domains`가 실제로 Network product에 의존함 — 인정하고 실용적으로 방어해야 함.

**강한 답변**:
"위반 맞습니다. 순수 Clean Architecture라면 Domain은 아무것도 몰라야 하는데, 우리 Domain은 API 정의와 DTO를 압니다. **인정하고, 왜 그렇게 됐고 뭐가 대가인지 말씀드리겠습니다.**

원인은 **DTO→Entity 변환 경계를 Domain 안에 둔** 실용적 타협입니다. 정석은 Repository 구현이 DTO를 Entity로 바꾸고 Domain은 Entity만 아는 건데, 초기 설계에서 Entity의 `init(dto:)`를 Domain에 두면서 DTO 타입 참조가 딸려 들어왔어요.

**대가는 명확합니다**: 서버 계약(DTO)이 바뀌면 Domain이 재컴파일되고, Domain의 순수성(테스트 용이성)이 떨어집니다. 그래서 우린 이걸 **다른 레이어에서 보상**합니다 — DTO는 Presentation에 직접 노출 안 되고 반드시 Entity로 변환되며, 이때 `?? 기본값`으로 silent-nil을 흡수합니다.

만약 리팩터 예산이 있다면 제 우선순위는 **Domain에서 DTO 참조를 걷어내고 매핑을 Repository로 내리는 것**입니다. 다만 이건 순수성이라는 이상 대비, 매핑 코드 중복이라는 현실 비용이 있어서 트레이드오프를 재고 결정할 사안입니다. 지금은 '이상적이진 않지만 통제되고 있는' 상태입니다."

**one-liner**: "네, Domain이 DTO를 아는 건 Clean Architecture 위반입니다 — Entity 변환을 Domain에 둔 타협의 결과고, 대가(재컴파일·순수성 저하)를 알고 있으며 Entity 매핑 계층으로 보상 중입니다. 걷어내려면 매핑을 Repository로 내리는 게 정답 방향입니다."

### Q7. Presentation이 Repository를 직접 import한다. UseCase 경계를 우회하는 거 아닌가?

**함정 포인트**: 가이드 자체 규칙("Presentation→Repository 금지")을 코드가 어긴 상황. 모르면 감점, 정당화만 하면 순진함.

**강한 답변**:
"맞습니다, 우리 자체 가이드가 금지한 걸 실코드가 어깁니다 — Presentation이 `AccountRepository`, 전송 인프라를 직접 import하는 지점이 여럿 있어요. **규칙과 현실의 갭을 인정**합니다.

배경은 **UseCase가 선택 레이어**라는 데 있습니다. 우리 4레이어(Domain/Repository/UseCase/Presentation)에서 UseCase는 '비즈니스 로직이 있을 때만' 넣습니다. 그래서 성숙한 피처(편도)는 UseCase 풀세트지만, **UseCase가 얇거나 없는 피처는 Presentation이 Repository를 직접 부르는 지름길**이 생겼어요. UseCase가 단순 pass-through면 그걸 강제로 끼우는 게 오히려 보일러플레이트니까요.

**대가는 컴파일 결합 증가**입니다 — Presentation이 Repository 구현 모듈을 직접 알면 빌드 시간과 순환 위험이 올라갑니다. 이걸 근본적으로 풀려면 **Interface 전용 모듈**로 구현과 인터페이스를 물리 분리해야 하는데, 그 시도(`...PresentationInterface`)는 스텁만 만들고 방치된 상태입니다. 정직하게 미완입니다.

제 판단: '모든 Presentation이 UseCase를 거친다'를 교조로 강제하기보다, **경계를 넘는 지점을 감사 도구로 계측하고, 결합이 실제로 아픈 곳부터 Interface 모듈로 분리**하는 게 실용적입니다. 규칙을 100% 지키는 것보다 규칙 위반의 비용이 어디서 큰지 아는 게 중요합니다."

**one-liner**: "네, UseCase가 선택 레이어라 얇은 피처에선 Presentation이 Repository를 직접 부릅니다 — 컴파일 결합이라는 대가를 알고, Interface 모듈 분리가 정답이지만 지금은 스텁 상태라 감사 도구로 우회 지점을 계측 중입니다."

### Q10. 토큰 갱신이 시간창(1초/0.35초) 기반이다. 동시 401이 쏟아지면 레이스 안 나나?

**함정 포인트**: "락을 걸었어요"라고 거짓 방어. findings상 실제로는 시각 기반 간이 뮤텍스라 경합 구간 레이스 가능성 있음. 인정 + 개선안이 정답.

**강한 답변**:
"정확한 지적입니다. 지금은 **진짜 락이 아니라 시간창 기반 스로틀**입니다 — 마지막 갱신 시각이 1초 이내면 실제 갱신을 안 하고 0.35초 대기 후 성공 알림만 쏩니다. 여러 요청이 동시에 401을 맞아도 refresh 폭주는 안 나게 한 거죠.

**동작 원리**: async `send`가 `.invalidToken`/`.tokenExpired`를 throw하면 Rx 래퍼가 `reloadAccessToken()` 후 **정확히 1회 재시도**합니다. 시간창은 이 refresh가 N번 중복 호출되는 걸 막습니다.

**레이스 가능성 인정**: 시각 비교와 갱신 사이에 원자성이 없어서, 정확히 경계 타이밍에 두 스레드가 동시에 '갱신 필요'로 판정하는 구간이 이론상 존재합니다. 지금은 그 확률이 낮고(0.35초 대기가 대부분 흡수), refresh가 멱등에 가까워서 실무상 문제가 안 났지만, **구조적으로 안전하진 않습니다**.

**제 개선안**: (1) 진짜 직렬화 — refresh를 단일 Actor나 serial queue로 감싸서 '진행 중이면 그 결과를 await'하는 single-flight 패턴으로 바꿉니다. (2) refresh 실패 시 대기하던 요청들에 일괄 실패 전파. 이러면 시간창이라는 확률적 방어를 결정적 방어로 승격할 수 있습니다.

**one-liner**: "지금은 시간창 스로틀이라 경계 타이밍 레이스가 이론상 존재합니다 — 실무상 문제는 없었지만 구조적으로 안전하진 않아서, single-flight(Actor/serial queue) 패턴으로 승격하면 3곳 중복 로직까지 함께 해소됩니다."

해결: **동일 시그니처/로직을 테스트 타깃에 미러링**합니다. 그리고 '이게 컴파일되면 계약이 일치한다'로 검증해요. 예를 들어 어떤 할인율 계산 로직을 DTO 없이 테스트하려면, 같은 계산 함수를 Protocol Witness로 테스트에 두고 프로덕션과 시그니처가 어긋나면 빌드가 깨지게 합니다. Rx 스트림은 RxBlocking으로 동기 검증하고요.

**계층별 테스트**: UseCase/Repository/Reactor는 XCTest + Mock(`MockXxxUseCase`)으로 단위 검증. DTO는 JSON 픽스처 디코딩 → Entity 분기 프로퍼티 assert로 계약 검증. View는 ViewInspector + 스냅샷 테스트.

**UI/E2E는 Maestro**: XCUITest 대신 Maestro YAML 플로우 35개(스모크 + 회귀 + 시나리오 상태머신)를 씁니다. flakiness는 `-UITEST` 런치인자로 애니메이션을 끄고 딥링크로 화면에 직접 진입해서 제거합니다.

**정직한 한계**: XCTest 파일이 70개 수준이라 100+ 화면 대비 커버리지가 균일하진 않습니다. 그래서 커버리지를 '전 화면 유닛'이 아니라 **'계약 지점(DTO·UseCase) 유닛 + 핵심 플로우 Maestro E2E'로 층화**해서, 리소스를 회귀 위험이 큰 곳에 집중합니다."

**one-liner**: "모듈 경계가 mock을 막으면 동일 시그니처를 미러링하는 Protocol Witness로 '컴파일되면 계약 일치'를 검증하고, UI는 Maestro E2E(`-UITEST`로 flakiness 제거)로 층화 — 커버리지를 전 화면이 아니라 계약·핵심플로우에 집중합니다."

### Q15. BLE, 결제, 실물 차량 같은 건 유닛 테스트가 불가능하다. 어떻게 검증하나?

**함정 포인트**: "실기기로 해봐요"만. 하드웨어/외부 의존을 어떻게 경계로 격리해 테스트 가능하게 만드는지가 핵심.

**강한 답변**:
"'테스트 불가능한 것'을 **테스트 가능한 경계로 분리**하는 게 핵심입니다.

**BLE**: 물리 통신 자체는 유닛 테스트 못 하지만, **패킷 빌더/파서는 순수 함수**라 테스트 가능합니다. `BlePacketBuilder`가 명령을 바이트로 만드는 것, `BleResponseParser`가 응답 바이트를 파싱하는 것, LRC 계산, AES-CBC 암복호화 — 이건 입력→출력이 결정적이라 유닛 테스트로 100% 덮습니다. **불확실한 물리 통신은 얇게, 결정적 로직은 두껍게** 만들어서 테스트 표면을 최대화해요. 물리 연결은 실기기 수동 + Maestro로 검증합니다.

**결제**: 결제 정합성(멱등성 등)은 **의도적으로 앱이 아니라 웹 PG/서버 책임**으로 위임했습니다. 앱은 카드 등록(OCR)과 정산 내역 표시만 하고, 실 결제는 웹뷰 PG를 통합니다. 즉 앱 레이어에서 검증할 결제 로직 자체를 최소화했어요. 이건 '못 테스트해서 도망'이 아니라, **금전 정합성을 서버 단일 진실원에 두는** 설계 결정입니다.

**서버 계약**: 실 서버 응답 대신 **Mock 픽스처(JSON)**로 Repository를 테스트합니다. Mock은 `#if DEBUG` 가드로 릴리스 유출을 막고요(이거 실수하면 프로덕션에 mock이 새서, 가드를 17곳에 명시적으로 박았습니다)."

**one-liner**: "하드웨어·결제를 통째로 테스트하는 대신 결정적 경계(패킷 빌더/파서·LRC·AES는 순수함수 유닛 테스트, 물리통신은 얇게)로 분리하고, 결제 정합성은 서버 단일 진실원에 위임해 앱의 테스트 불가 표면 자체를 최소화합니다."

## E. 데이터 / 영속성 / 오프라인 ("오프라인·캐시·보안")

### Q16. Repository가 캐시 없이 매번 네트워크를 친다. SSOT/오프라인 전략이 없는 거 아닌가?

**함정 포인트**: "캐시 넣으면 돼요"로 단순화. 캐시 없음이 '실수'가 아니라 '도메인 특성상 선택'일 수 있음을 설명.

**강한 답변**:
"대부분 Repository가 **스테이트리스 remote 프록시**인 건 사실입니다 — network 호출 → DTO를 Entity로 map, 캐시 없음. 22개 중 인메모리 캐시가 있는 건 필터 목록 하나(read-through)뿐이에요. **이건 도메인 특성상 의도된 선택**입니다.

카셰어링의 핵심 데이터는 **실시간성이 생명**입니다 — 차량 가용성, 예약 상태, 위치, 가격. 이걸 캐시하면 '이미 예약된 차를 보여주거나 지난 가격을 보여주는' **stale이 곧 사고**입니다. 오프라인에서 예약 버튼을 누르게 하는 것 자체가 위험해요. 그래서 **서버를 단일 진실원**으로 두고, 앱은 얇은 뷰로 유지합니다.

오프라인은 **연결 에러를 명확히 전파**하는 쪽입니다 — 오프라인 상태에서 조작 가능한 척하지 않고 `connectionError`로 사용자에게 알립니다. 이게 '오프라인 편집 후 동기화'보다 이 도메인엔 안전합니다.

**캐시가 정당한 곳엔 씁니다**: 필터 목록처럼 자주 안 바뀌고 stale이 무해한 건 read-through 캐시. 이미지는 Kingfisher가 캐싱. 즉 **'캐시 안 함'이 기본이 아니라, 데이터 휘발성에 따라 캐시 여부를 결정**한 겁니다.

**개선 여지**: 캐시가 있는 필터도 TTL/무효화가 없어서 프로세스 수명 = 캐시 수명입니다. 앞으로 정말 오프라인 UX가 필요한 읽기 전용 화면(이용 내역 등)엔 명시적 TTL 캐시를 붙일 수 있습니다."

**one-liner**: "차량 가용성·가격은 stale이 곧 사고라 서버를 단일 진실원으로 두고 remote 프록시를 유지합니다 — 캐시는 stale이 무해한 곳(필터·이미지)에만 선택적으로 쓰고, 오프라인은 '편집 허용'이 아니라 '명확한 에러 전파'로 방어합니다."

### Q17. 토큰을 Keychain이 아니라 UserDefaults + 하드코딩 키 AES로 저장한다. 보안 취약 아닌가?

**함정 포인트**: 방어 불가능한 걸 방어하려다 신뢰 상실. 이건 **명확히 인정 + 개선안**이 정답. findings상 키/IV가 소스 하드코딩.

**강한 답변**:
"이건 **명확한 개선 대상이고, 변명하지 않겠습니다.** 현재 토큰(access/refresh)이 UserDefaults에 저장되고, 키·값을 AES-CBC로 암호화하지만 **키와 IV가 소스에 하드코딩**돼 있습니다. 이건 Keychain 대비 약합니다 — 바이너리에서 키를 추출하면 복호화가 가능하니까요.

**있는 그대로의 평가**: 이건 'Keychain 대신 UserDefaults를 쓰면서 고정키로 난독화한' 수준입니다. 탈옥/바이너리 분석 앞에선 방어력이 제한적입니다.

**왜 이렇게 됐나(변명 아닌 맥락)**: UserDefaults는 마이그레이션·다중 값 관리가 쉬워서 초기에 편의로 선택된 걸로 보입니다. 그 위에 AES를 얹어 '평문보다는 낫게'는 만들었지만, 위협 모델을 Keychain 수준으로 올리진 못했습니다.

**제 개선 로드맵**: (1) **토큰류 민감정보는 Keychain으로 이전** — `kSecAttrAccessibleWhenUnlockedThisDeviceOnly`로. (2) 하드코딩 키 제거, 필요한 대칭키는 **Secure Enclave 파생 또는 Keychain 보관**. (3) 기존 저장분은 마이그레이션 경로 제공. 이건 순수 개선이라 트레이드오프가 거의 없어서 우선순위 높은 부채입니다.

**참고로 서버-클라 양쪽 암호화 계약**(서버가 내려주는 enc 필드 복호화)은 별개로 잘 동작하고, 배포~강제업뎃 갭 호환까지 고려돼 있습니다 — 즉 팀이 암호화 자체를 모르는 게 아니라, **토큰 저장 위치 선택이 잘못된** 겁니다."

**one-liner**: "변명 없이 인정합니다 — 토큰이 UserDefaults + 소스 하드코딩 키 AES라 Keychain보다 약합니다. 트레이드오프가 거의 없는 순수 개선이라, 토큰류를 Keychain(+Secure Enclave 파생키)으로 이전하는 게 제 우선순위 높은 부채입니다."

### Q18. DTO의 64%가 CodingKeys 없이 전부 Optional이다. silent-nil로 서버 계약 깨져도 모르는 거 아닌가?

**함정 포인트**: 이걸 사소하게 여기기. findings상 구조적 위험으로 지목됨 — 심각성 인지 + 방어 장치 설명이 정답.

**강한 답변**:
"구조적 위험 맞습니다. DTO 191개 중 CodingKeys 보유가 36%뿐이고 프로퍼티가 거의 다 Optional이라, **서버가 키를 리네임하면 디코딩이 throw가 아니라 조용히 nil**이 됩니다. 그리고 Entity 변환에서 `?? 기본값`으로 흡수되니, **'필드가 안 온 것'과 '의도적 기본값'이 구분 안 되고** 빈 문자열/0으로 흘러갑니다. 이게 silent-nil 안티패턴이고, 최악은 잘못된 값으로 사용자에게 잘못된 정보를 보여주는 겁니다.

**왜 이 구조가 생겼나**: 크로스플랫폼(타 플랫폼 동시 개발) + 서버 계약이 자주 바뀌는 환경에서, 디코딩을 관대하게(Optional) 만들어서 '한 필드 누락으로 전체 화면이 안 뜨는' 사고를 막으려는 방어였습니다. 즉 **엄격 디코딩(하나 틀리면 throw) vs 관대 디코딩(silent-nil)의 트레이드오프**에서 관대를 택한 거예요. 둘 다 나름의 위험이 있습니다.

**방어 장치**: 코드만으로는 못 막으니 **프로세스로 담보**합니다 — 공유 DTO/필드 작업 진입 시 **3-오라클 대조(API 스펙 + 타 플랫폼 구현 + 실 서버 응답)**를 커밋 전 게이트로 강제하고, CodingKeys 명시 + decode-roundtrip 테스트로 silent-nil을 구조적으로 차단합니다. 배포 전엔 회귀 레이더가 'enc 필드 silent-nil' 같은 과거 사고 시그니처를 이번 diff에서 스캔합니다.

**정답 방향**: 신규 DTO는 CodingKeys 필수 + 진짜 필수 필드는 non-Optional로 선언해서, 계약 위반을 **디코딩 시점에 시끄럽게** 만드는 걸로 점진 전환하는 겁니다."

**one-liner**: "네, 관대 디코딩(전부 Optional)이 silent-nil 위험을 구조적으로 안습니다 — '한 필드 누락으로 화면 붕괴'를 막으려던 트레이드오프고, 코드로 못 막으니 3-오라클 계약 감사 + decode-roundtrip 게이트로 방어하며, 신규는 CodingKeys 필수·필수필드 non-Optional로 시끄럽게 전환 중입니다."

## F. 성능 / 동시성 ("성능 병목 · async 어느 계층")

### Q19. async/await를 어느 계층에 두나? 아무 데나 Task 뿌리면 안 되는데.

**함정 포인트**: "필요한 데 써요"로 모호. 계층별 원칙이 없으면 아키텍처 감각 의심.

**강한 답변**:
"명확한 원칙이 있습니다: **async는 두 경계에만, 나머지는 Rx.**

1. **네트워크 전송 계층** — 최하부 Alamofire 호출을 async로 감쌉니다. 여기가 async의 자연스러운 자리예요: I/O 대기, 토큰 만료→refresh→재요청의 순차 흐름이 async/await로 가장 읽기 쉽습니다. 하지만 이 결과를 **즉시 `Observable.create`로 Rx에 재브릿지**해서, 상위 계층은 async를 안 봅니다.

2. **Interactor 내 복잡한 순차 검증** — 여러 단계 검증이 얽힌 곳은 async로 쓰되, `await MainActor.run`으로 메인 복귀 후 `observer.onNext`로 Mutation 스트림에 재주입합니다.

3. **단발성 지연** — 토스트 자동 해제 같은 건 `Task.sleep`로 간결하게(Rx `delay`보다 읽기 쉬움).

**원칙의 핵심**: async를 **UseCase·Repository·Presentation 같은 아키텍처 계약 계층에는 안 흘립니다.** 그 계층은 Rx `Observable`이 계약이에요. async를 거기 뿌리면 취소·스케줄러·라이프사이클이 두 모델로 갈라져서 Interactor 하나에 동시성 모델이 둘 공존하는 사고가 납니다.

즉 **async는 '잎(leaf)'에만, Rx는 '줄기(trunk)'에.** 잎에서 async로 편하게 쓰고 줄기로 나오기 전에 Rx로 재흡수 — 이 규칙을 지키면 async의 가독성과 Rx 파이프라인의 일관성을 둘 다 얻습니다. MainActor 격리도 이 재브릿지 지점에서 명시적으로 처리하고요."

**one-liner**: "async는 네트워크 전송·복잡 검증·단발 지연 같은 '잎'에만 두고 반드시 Rx로 재브릿지하며, UseCase/Repository/Presentation 계약 계층엔 안 흘립니다 — 한 Interactor에 동시성 모델이 둘 공존하는 걸 막는 'async는 잎, Rx는 줄기' 원칙입니다."

---

### Q20. 네트워크 클라이언트가 화면(RIB)마다 새로 생성된다. 낭비 아닌가? NotificationCenter N중 등록도 문제고.

**함정 포인트**: 이걸 옹호만 하기. findings상 RIB Builder마다 인스턴스 생성 + 옵서버 N중 등록은 실제 리스크. 인정 + 왜 감내 가능한지.

**강한 답변**:
"정확한 관찰입니다. 전송 클라이언트가 공유 싱글톤이 아니라 **각 RIB Builder에서 개별 생성**돼서, 화면마다 인스턴스가 N개 뜹니다. 각 인스턴스가 생성자에서 토큰 갱신 NotificationCenter 옵서버를 등록하니, **토큰 갱신 알림이 오면 N개가 모두 반응**합니다.

**왜 이렇게 됐나**: RIBs의 DI 철학이 '의존성을 트리 스코프에 묶는' 거라, 클라이언트도 RIB 스코프로 생성한 자연스러운 결과입니다. 각 화면이 자기 클라이언트를 갖는 게 격리 측면에선 깔끔해요.

**실제 비용 평가**: 전송 클라이언트 자체는 가볍습니다(설정 + URLSession 참조). 메모리 낭비는 무시할 수준이에요. **진짜 리스크는 NotificationCenter N중 등록**입니다 — 토큰 갱신 시 N개가 각자 재요청/상태갱신을 시도하면 중복 refresh나 경합이 생길 수 있습니다. 다만 이건 앞서(Q10) 말한 **시간창 스로틀이 완화**합니다 — N개가 동시에 refresh를 시도해도 1초 창이 실제 갱신을 1회로 눌러요.

**개선 방향**: (1) 전송 클라이언트를 **AppComponent 스코프의 공유 인스턴스로 승격**해서 옵서버를 1개로. RIBs에서도 상위 Component가 만든 걸 자식이 주입받으면 됩니다. (2) 토큰 갱신을 옵서버 방송이 아니라 **single-flight refresh(Q10)로 중앙화**하면 N중 등록 문제가 근본 해소됩니다.

즉 지금은 '스로틀로 증상 완화 + 인스턴스 비용은 무시 가능'이라 감내 가능하지만, **구조적 정답은 클라이언트 공유 + refresh 중앙화**입니다."

**one-liner**: "네, RIB 스코프 생성이라 클라이언트가 N개 뜨고 옵서버가 N중 등록됩니다 — 인스턴스 비용은 무시 가능하고 N중 refresh는 시간창 스로틀이 눌러서 감내 가능하지만, 구조적 정답은 클라이언트를 AppComponent 스코프로 공유하고 refresh를 single-flight로 중앙화하는 것입니다."

### Q21. 화면당 State 재방출로 UI가 튀는 회귀를 플래그로 막았다는데, 그거 땜빵 아닌가?

**함정 포인트**: 플래그 방어를 '깔끔한 설계'라 우기기. 또는 그냥 부채라고 항복. 하이브리드의 본질적 대가로 설명.

**강한 답변**:
"땜빵이라기보단 **하이브리드의 본질적 세금을 국소화한** 겁니다. 정직하게 말하면 이상적 설계는 아니에요.

**문제의 뿌리**: 단방향 흐름(ReactorKit State)과 SwiftUI 낙관적 로컬 갱신을 섞으면, State가 재방출될 때 '사용자가 이미 바꾼 값'이 서버 초기값으로 되살아나는 회귀가 납니다. 예로 사용자가 경로를 리셋했는데 reactive 재발화로 서버값이 복원되는 식이죠.

**왜 플래그인가**: 이걸 근본적으로 풀려면 '사용자 의도'와 '서버 상태'를 State에서 1급으로 구분하는 상태 모델링이 필요합니다. `hasUserResetRoute`, `didApplyDeepLinkFilter` 같은 플래그가 사실 그 '사용자 의도' 비트를 명시적으로 든 겁니다 — **암묵적 재방출 규칙을 명시적 가드로 승격**한 거예요. 주석에 QA 티켓 번호까지 박아서 '왜 이 플래그가 있는지'를 추적 가능하게 했습니다.

**대가 인정**: 플래그가 늘면 State가 복잡해지고, 새 개발자가 조합을 놓칠 수 있습니다. 그래서 이건 **관리되는 부채**입니다 — 없앤 척 안 하고, 각 플래그에 근거(티켓)를 붙여 국소화합니다.

**정답 방향**: 반복되면 이 '1회성 적용' 패턴을 State에 `applied: Set<Trigger>` 같은 **1급 개념으로 일반화**하거나, SwiftUI 로컬 상태와 Reactor State의 소유권 경계를 더 엄격히 나누는 겁니다. 지금은 플래그가 그 경계를 임시로 긋고 있는 거고요."

**one-liner**: "단방향 흐름 + SwiftUI 낙관적 갱신을 섞은 하이브리드의 본질적 세금입니다 — 플래그는 '사용자 의도'라는 암묵 규칙을 티켓 근거와 함께 명시적 가드로 승격한 관리되는 부채고, 반복되면 State의 1급 개념으로 일반화하는 게 정답입니다."

## G. 문서·유령·일관성 ("설계와 현실의 갭")

### Q22. 문서엔 모듈 58개, 실제 Package.swift는 50개. 유령 디렉토리도 있다. 관리가 안 되는 거 아닌가?

**함정 포인트**: 얼버무리기. 문서-코드 드리프트를 어떻게 감지·정정하는지가 성숙도 신호.

**강한 답변**:
"드리프트 맞습니다. 문서는 58, 실제 SPM 패키지는 50이고, `...Domain`·`...PresentationInterface` 같은 **유령 디렉토리 6개**(Package.swift·Sources 없이 잔재만)가 있습니다. 리네이밍/리팩터 잔재예요.

**중요한 구분**: 이건 **'설계가 무너진 것'이 아니라 '문서가 코드를 못 따라간 것'**입니다. 코드 자체는 컴파일되고 동작합니다 — 유령 디렉토리는 빌드 그래프에 없으니 런타임 영향 0입니다. 문제는 **온보딩 신뢰도**입니다: 새 사람이 문서의 58을 믿고 유령 모듈을 실 모듈로 세면 헷갈려요.

**어떻게 관리하나**: (1) **코드를 진실원으로** 삼습니다 — 모듈 수는 문서가 아니라 `find Package.swift`가 진실입니다. (2) 문서 부패를 감지하는 **reground 프로세스** — 문서가 참조하는 코드/스크립트/링크가 실재하는지 스캔해서 낡은 문서를 갱신·archive합니다. (3) 유령 디렉토리는 정리 대상 백로그에 있습니다(삭제는 사람 승인 필요 — 혹시 모를 참조 확인 후).

**메타 교훈**: 320K LOC 장수 프로젝트에서 문서-코드 100% 동기화는 비현실적입니다. 그래서 우린 **'문서를 항상 맞추기'가 아니라 '드리프트를 감지·표면화하기'**에 투자합니다. 문서가 거짓말하는 걸 도구가 잡아내는 게, 사람이 문서를 완벽히 유지하리라 기대하는 것보다 현실적입니다."

**one-liner**: "네, 문서 58 vs 코드 50 + 유령 6개는 드리프트입니다 — 런타임 영향은 0(빌드 그래프 밖)이고, 코드를 진실원으로 삼아 문서 부패를 자동 감지(reground)하는 데 투자합니다. '문서 완벽 유지'가 아니라 '드리프트 표면화'가 대형 장수 프로젝트의 현실적 전략입니다."

### Q23. Interface 전용 모듈을 시도했다가 스텁만 남기고 방치했다. 실패한 설계 아닌가?

**함정 포인트**: 방치를 옹호하거나 숨기기. 미완 시도를 어떻게 판단하는지가 엔지니어링 성숙도.

**강한 답변**:
"미완 맞습니다. Presentation의 인터페이스만 별도 모듈로 빼서 형제 피처가 **구현 없이 참조**하게 하려던 시도(`...PresentationInterface`)가 스텁(빈 껍데기)으로 남아 있습니다. 그 결과 Presentation끼리 **구현 모듈을 직접 import**하는 지금 상태가 됐고요(Q7과 연결).

**왜 방치됐나(추정)**: Interface 모듈 분리는 컴파일 결합을 줄이지만, **모든 public 타입을 인터페이스/구현 두 곳에 유지하는 비용**이 큽니다. 즉시 아픈 문제가 아니면 ROI가 낮게 느껴져서 우선순위가 밀린 걸로 봅니다.

**이걸 어떻게 판단하나**: 저는 **미완 시도 자체를 실패로 안 봅니다** — 실패는 '방치된 걸 모르거나 숨기는 것'입니다. 우린 이게 스텁인 걸 알고, 감사에서 Presentation→구현 직접 import를 계측합니다. 즉 **알고 있는 미완**입니다.

**결정 기준**: Interface 모듈은 **컴파일 결합이 실제로 아플 때**(빌드 시간 급증, 순환 발생) 도입하는 게 맞습니다. 지금 전면 도입은 유지비만 늘리는 조기 최적화일 수 있어요. 그래서 제 판단은 '전부 분리'가 아니라 **'결합이 아픈 핵심 경계부터 선택적으로 Interface 분리'**입니다. 스텁은 그 트리거를 기다리는 상태고, 방향 자체는 유효합니다. 다만 **스텁을 코드에 남겨두면 유령이 되니**, 실제 도입 전까진 디렉토리를 정리하는 게 위생적입니다."

**one-liner**: "미완 맞지만 '아는 미완'입니다 — Interface 모듈은 유지비가 커서 컴파일 결합이 실제로 아픈 경계부터 선택 도입하는 게 맞고, 전면 도입은 조기 최적화입니다. 방향은 유효하되 스텁은 유령이 되니 정리하는 게 위생적입니다."

### Q24. PUT 요청이 dead code 때문에 POST 경로로 나간다. 이런 버그가 왜 프로덕션에 있나?

**함정 포인트**: 사소하다고 넘기거나, 반대로 과잉 사과. 버그의 실제 영향 범위를 정확히 진단하는 게 핵심.

**강한 답변**:
"실측된 버그성 흔적 맞습니다 — 요청 팩토리의 switch에서 PUT이 POST 빌더로 라우팅되고, 별도 정의된 PUT 빌더는 호출되지 않는 dead code입니다. 결과적으로 PUT이 body-JSON 형태로 나갑니다.

**왜 안 터졌나(영향 진단)**: 이게 프로덕션에서 사고가 안 난 이유는, **서버가 해당 PUT 엔드포인트를 body-JSON으로도 받아주고 있기 때문**일 가능성이 높습니다. 즉 '틀린 코드인데 서버 관대함이 가려주는' 전형적 케이스예요. 그래서 컴파일도 통과하고 런타임도 동작해서 아무도 못 봤습니다 — **증상이 없는 버그**라 발견이 늦은 겁니다.

**어떻게 발견됐나**: 사람이 코드를 통독해서가 아니라 **정적 분석/코드 감사에서 dead code(호출 안 되는 PUT 빌더)로 잡혔습니다**. 이게 감사 도구를 상시 돌리는 가치예요 — 증상 없는 버그를 증상 나기 전에 잡습니다.

**대응**: (1) switch를 고쳐 PUT을 올바른 빌더로 라우팅하고 dead code 제거. (2) 다만 **서버가 진짜로 어느 포맷을 기대하는지 계약을 먼저 확인**해야 합니다 — 지금 body-JSON으로 잘 받고 있다면, 코드를 고쳤을 때 오히려 깨질 수 있으니 3-오라클로 서버 계약을 검증하고 회귀 테스트를 붙인 뒤 고칩니다. **'버그니까 바로 고친다'가 아니라 '왜 안 터졌는지 이해하고 안전하게 고친다'**가 5년차의 접근입니다."

**one-liner**: "네 dead code로 PUT이 POST 경로로 나가지만 서버가 body-JSON을 관대하게 받아줘서 증상이 없던 버그입니다 — 코드 감사로 발견했고, 무작정 고치면 오히려 깨질 수 있어 서버 계약을 3-오라클로 확인하고 회귀 테스트를 붙인 뒤 안전하게 수정합니다."

### Q25. 크로스플랫폼(타 플랫폼 동시개발) parity를 코드가 아니라 '프로세스'로 담보한다는데, 그게 신뢰 가능한가?

**함정 포인트**: "주석에 써놨어요"로 끝. 코드 레벨 자동화 부재를 어떻게 프로세스로 보상하는지가 핵심.

**강한 답변**:
"코드 레벨 parity 자동화가 없는 건 맞습니다 — 소스엔 '타 플랫폼은 `<br>`로 개행, iOS는 `\n`으로 변경' 같은 **주석 인지**만 있고, 공유 스펙을 강제하는 코드 계약은 없습니다. 이건 한계입니다.

**왜 코드로 안 하나**: 두 플랫폼이 **같은 서버 API를 다르게 렌더링**하는 건 본질적으로 불가피합니다(개행 문자, 날짜 포맷, 상태코드 게이팅). 이걸 코드 계약으로 강제하려면 공유 스펙 레이어를 두 네이티브 코드베이스가 물어야 하는데, 그 인프라 비용이 큽니다.

**대신 프로세스로 보상**: (1) **3-오라클 계약 감사** — 공유 도메인 필드 작업 시 API 스펙 + **타 플랫폼 구현** + 실 서버 응답 셋을 대조합니다. 타 플랫폼이 이미 shipped한 동등 기능이 있으면 그걸 SoT로 추출해서 정렬합니다. (2) **parity 자가검증을 QA 왼쪽으로 이동(left-shift)** — 구현 전에 타 플랫폼 동작을 추출하고, QA 전에 parity를 셀프체크합니다. (3) 서버 헤더가 플랫폼을 구분하므로(공통 헤더 규격), 서버가 계약의 중심 진실원 역할을 합니다.

**정직한 평가**: 프로세스 담보는 사람이 게이트를 실행해야 해서 코드 계약보다 약합니다 — 사람이 건너뛰면 뚫려요. 그래서 우린 이 프로세스를 **하네스 스킬로 자동 트리거**(공유 필드 변경 감지 시 활성)해서 '사람이 기억해서'가 아니라 '도구가 강제'하는 쪽으로 올렸습니다. 완벽한 코드 계약은 아니지만, **'주석 의존'에서 '자동 트리거되는 게이트'로 성숙도를 한 단계 올린** 상태입니다."

**one-liner**: "코드 계약이 없는 건 맞고, 두 플랫폼의 렌더링 차이는 본질적으로 불가피합니다 — 대신 3-오라클 감사(스펙+타플랫폼+실응답)를 QA 왼쪽으로 이동시키고, 그 게이트를 하네스가 공유필드 변경 시 자동 트리거해 '주석 의존'을 '강제되는 프로세스'로 승격했습니다."

2. **트레이드오프 언어**: "X가 나쁘다"가 아니라 "X와 Y 사이에서 Z 이유로 X를 택했고 비용은 W". (Q1·Q8·Q16·Q19)
3. **'없다'가 아니라 '계측한다'**: 사이클·드리프트·계약위반을 완벽 차단 대신 상시 감사로 표면화. (Q5·Q13·Q22)
4. **폭발 반경 사고**: 복제·격리·킬스위치로 실패의 전파를 막음. (Q8·Q9·Q11)
5. **경계로 테스트 가능성 확보**: 테스트 불가한 걸 결정적 경계로 쪼갬. (Q14·Q15)
6. **도메인 특성이 설계를 정당화**: 실시간성·물리결합·규제가 '왜 이렇게'의 근거. (Q9·Q16)

> 면접 팁: 함정 질문일수록 **먼저 인정**하고 들어가면 방어가 아니라 대화가 된다. "네, 그거 위반 맞습니다. 왜 그런지 말씀드리면—"이 가장 강한 오프닝.
