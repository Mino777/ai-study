## 🧭 모바일 시스템 디자인 10-스텝 프레임 (DSA 5스텝의 SD 버전)

> 모든 SD 문제에 이 순서로. 정답이 아니라 **트레이드오프를 소리내는 과정**이 점수.

```
①  요구사항 명확화   기능/비기능(규모·지연·오프라인) 질문부터 (절대 바로 설계 X)
②  범위 확정         "이번 라운드에 뭘 설계? 뭘 뺄지" 합의
③  규모 추정         DAU·QPS·데이터량 대략 (냅킨 계산)
④  API 설계          클라↔서버 계약 (엔드포인트·페이로드)
⑤  데이터 모델       뭘 저장/캐시? 로컬 DB 스키마
⑥  하이레벨 그림      클라-서버-DB-캐시-CDN 박스&화살표
⑦  모바일 특화        오프라인/동기화 충돌/이미지/페이지네이션/배터리
⑧  병목 & 스케일     캐싱·CDN·페이지네이션·레이트리밋
⑨  트레이드오프       "이 선택의 대가는 X, 대안은 Y" 소리내기 ← 감별점
⑩  마무리            요약 + "더 판다면 어디" 언급
```

> DSA 5스텝처럼 매 SD 문제마다 이 프레임을 명시적으로 호명 → 압박 상황에서도 서사 자동화.

## 🅰️ iOS 아키텍처 서사 뱅크 (강점 → 면접 방어 스크립트)

> 실무 경험을 "왜 이 결정을 했나 + 트레이드오프 + 대안"으로 정제. 각 항목 = 면접 딥다이브 방어 카드.

- [ ] A1. 모듈화(SPM 멀티모듈) — 왜 쪼갰나? 빌드시간·팀분업·의존성 방향. 대가는?
- [ ] A2. Clean Architecture 계층(Domain/Repository/Presentation) — 경계와 의존성 규칙, 테스트 용이성
- [ ] A3. DI(의존성 주입) — 왜 필요, 어떻게(생성자/컨테이너), 테스트 대역
- [ ] A4. MVVM vs MVI vs TCA — 무엇을 왜 골랐나, 상태관리 철학
- [ ] A5. 반응형(Rx) → Swift Concurrency 전환 결정 — 왜, 무엇을 얻고 잃나
- [ ] A6. 화면 전환/좌표계(Coordinator/Router) — 네비게이션 결합도 낮추기
- [ ] A7. 네트워크 계층 설계 — 재시도·캐시·에러 전파·토큰 갱신
- [ ] A8. 레거시 점진 개선(Strangler) — 리스크 관리하며 리팩터
- [ ] A9. 테스트 전략 — 단위/통합/UI 피라미드, 무엇을 어디서
- [ ] A10. "왜 이 결정?" 종합 리허설 — 압박 질문 방어 (금요일 모의면접)
