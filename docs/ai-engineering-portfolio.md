# AI Engineering Portfolio — Forward Deployed Engineer

> **"AI를 쓰는 사람"이 아니라 "AI를 현장에 배치하고 제어하는 엔지니어"**
>
> 이 문서는 1인 FDE가 AI 에이전트를 무기 삼아 4개 프로덕트를 동시에 설계·구축·운영한 종합 현황이다.
> 단순 도구 나열이 아니라, **현장에서 부딪힌 문제 → 어떤 AI 제어 체계를 설계했는가**를 중심으로 정리한다.
>
> **마지막 업데이트**: 2026-04-22

---

## 1. 전장(Theater) 개요 — 1인이 4개 프로덕트를 동시 운영하는 구조

FDE의 핵심 문제: **"혼자서 이걸 다 어떻게 돌리는가?"**

| 프로젝트 | 도메인 | 기술 스택 | FDE가 풀어야 할 문제 |
|----------|--------|----------|---------------------|
| **ai-study** | AI 엔지니어링 위키 | Next.js 15 + MDX | 학습한 방법론을 다른 프로젝트에 즉시 배포 가능한 형태로 박제 |
| **moneyflow** | AI 트레이딩 | Next.js 16 + Supabase + Zustand | AI 매매 판단의 실패를 구조적으로 다음 사이클에 반영 |
| **tarosaju** | AI 타로 서비스 | Next.js 16 + Supabase + Anthropic SDK | AI 출력의 품질을 사람 없이 검증·방어 |
| **aidy** | AI 다이어리 (4레포) | Spring Boot + Swift/TCA + Compose | 서버·iOS·Android 3팀분 작업을 AI 에이전트 4개로 동시 진행 |

**해답: AI를 "도구"가 아니라 "배치 가능한 병력"으로 운용한다.**

## 2. 배치한 AI 전력(Force Structure)

### 2-1. AI 에이전트 편성

| 역할 | 모델 | 배치 방식 | 작전 범위 |
|------|------|----------|----------|
| **주력 개발 에이전트** | Claude Opus/Sonnet (Claude Code) | 터미널 직접 배치, tmux 다중 세션 | 전 프로젝트 코드 작성·리뷰·디버깅 |
| **콘텐츠 생성 에이전트** | Gemini 2.5 Flash | GitHub Actions CI 파이프라인 | ai-study 일일 레슨 자동 생성 |
| **QA 에이전트** | Claude + gstack browse | 헤드리스 브라우저 자동화 | 배포 후 시각적 QA + 성능 체크 |
| **전문가 에이전트** | Claude (에페메럴) | `/consult`로 온디맨드 소환 | 법률·보안·마케팅 등 도메인 자문 |

### 2-2. 무기 체계(Toolchain) — 외부 프레임워크 선택적 이식

FDE는 프레임워크를 통째로 쓰지 않는다. **현장에 필요한 모듈만 분해해서 이식한다.**

| 출처 | 가져온 것 (현장 검증 완료) | 버린 것 (현장 부적합) | 판단 근거 |
|------|--------------------------|---------------------|----------|
| **gstack** | QA, browse, ship, investigate, compound 등 30+ 스킬 | — | 가장 넓은 커버리지, 현장 즉시 사용 가능 |
| **Superpowers** (obra) | SDD 2-stage review, git-worktrees, brainstorming | finish-branch | 워크플로우 불일치 — 우리 compound 사이클과 충돌 |
| **Perpetual Engine** (greatsk55) | MetricsEvaluator, ConsultantFactory, MessageQueue | 전체 프레임워크 | 부분 이식이 더 가벼움. 전체 도입은 오버헤드 |

### 2-3. 자체 설계 제어 체계 (현장에서 만든 것)

외부 프레임워크로 안 풀리는 문제를 현장에서 직접 설계했다.

| 체계 | 풀어야 했던 현장 문제 | 설계한 해법 |
|------|---------------------|-----------|
| **Harness Engineering** | AI가 틀린 걸 자신있게 말한다 | 5-Layer 래퍼: Input→Context→Inference→Output→Observability 전 구간 제어 |
| **Compound Engineering** | 같은 실수가 다음 세션에서 반복된다 | Plan→Work→Review→Compound 4단계 강제 + 인시던트→다음 입력 자동 변환 |
| **Hub-Worker Model** | 4개 프로젝트 간 지식이 파편화된다 | 허브 1개(ai-study) + 워커 N개, 파일 기반 비동기 메시지 큐 |
| **LLM-First Wiki** | AI 에이전트가 위키를 못 읽는다 | 모든 엔트리에 AI Agent Directive(트리거·전제조건·실행 단계·안티패턴) 강제 |

## 3. 작전별 AI 배치 상세(Operations)

### 작전 1: ai-study — 방법론 허브 운영

**FDE 미션**: 현장에서 검증한 AI 엔지니어링 패턴을 박제하고, 다른 프로젝트에 즉시 배포할 수 있게 유지

```
[정찰] 지식 그래프 분석 → 빈 영역·약한 연결 자동 탐지
  → [생성] Gemini Flash 일일 레슨 (09:00 KST 자동)
  → [검증] Zod 스키마 + /validate-mdx + /validate-ai-output
  → [박제] MDX 엔트리 (AI Agent Directive 포함)
  → [배포] JIT 벡터 검색으로 다른 프로젝트에서 즉시 호출 (99.8% 토큰 절감)
  → [피드백] search-hits.json + shadow-benchmark로 효과 측정
```

| FDE 제어 포인트 | 구현 | 왜 필요했나 |
|----------------|------|-----------|
| **입력 검증** | Zod frontmatter (13 카테고리, confidence 1-5) | Gemini가 잘못된 카테고리/confidence를 넣어서 빌드 깨짐 반복 |
| **출력 검증** | `/validate-mdx` (Mermaid 5대 함정), `/validate-ai-output` (가짜 인용 4대 함정) | AI 생성 MDX가 Shiki+Mermaid 충돌, 존재하지 않는 URL 인용 |
| **컨텍스트 절감** | JIT 검색 (multilingual-e5-small, 384d) | 위키 전체 331K tokens → 에이전트에게 ~800 tokens만 전달 |
| **관측** | search-hits.json + weekly shadow-benchmark CI | "실제로 검색이 작동하는가"를 숫자로 확인 |

### 작전 2: aidy — 1인 4레포 동시 개발

**FDE 미션**: 서버·iOS·Android 3개 플랫폼을 AI 에이전트 4개로 동시에 개발. 사람은 스펙만 쓰고 게이트만 검증

```
[스펙 작성] api-contract.md (단일 진실 원천)
  → [작전 명령] aidy-architect가 Work Order 발행
      ├── Claude #1: aidy-server (Spring Boot + Kotlin)
      ├── Claude #2: aidy-ios (Tuist + TCA + SwiftUI)
      └── Claude #3: aidy-android (Compose + MVVM)
  → [게이트 검증] Gate 1: 스펙 준수 / Gate 2: 크로스 플랫폼 통합
  → [보고] 파일 기반 비동기 메시지 큐 (messages/ JSON)
```

| FDE 제어 포인트 | 구현 | 왜 필요했나 |
|----------------|------|-----------|
| **Spec = Law** | `api-contract.md` 단일 진실 원천 | 4개 레포에서 각각 다른 해석 → 통합 시 필드 불일치 |
| **2-Stage Review** | Stage 1: 스펙 compliance / Stage 2: 코드 품질 | AI가 "돌아가는 코드"를 만들지만 스펙과 어긋나는 경우 빈번 |
| **tmux 4-pane 동시 실행** | 4개 Claude Code 세션 병렬 | 순차 실행 시 하루 걸릴 작업을 2-3시간으로 단축 |
| **비동기 메시지 큐** | messages/ JSON (Perpetual Engine 이식) | 세션 간 상태 전달 — "서버 API 완료" → "iOS 연동 시작" |

### 작전 3: moneyflow — AI 트레이딩 운영

**FDE 미션**: AI 매매 판단의 실패를 자동으로 다음 사이클 입력으로 변환

| FDE 제어 포인트 | 구현 | 왜 필요했나 |
|----------------|------|-----------|
| **Circuit Breaker** | AI 호출 실패 시 자동 차단 + fallback | API 장애 시 무한 재시도 → 비용 폭발 방지 |
| **비용 추적** | ccusage 베이스라인 측정, 트레이드당 AI 비용 로깅 | AI API 비용이 수익을 초과하는 순간 감지 |
| **일일 Compound 순환** | 매매 결과 → 다음날 시그널 입력 | "오늘의 실패"가 구조적으로 "내일의 판단 근거"가 됨 |
| **패턴 수신** | ai-study 허브에서 검증된 패턴 이식 | 허브에서 3+ 세션 검증된 것만 프로덕션 투입 |

### 작전 4: tarosaju — AI 출력 방어

**FDE 미션**: 사용자에게 도달하는 AI 출력의 품질을 사람 없이 구조적으로 보장

| FDE 제어 포인트 | 구현 | 왜 필요했나 |
|----------------|------|-----------|
| **출력 구조 검증** | 타로 해석 응답 Zod 스키마 강제 | AI가 형식 무시하고 자유 텍스트 반환 → 프론트 파싱 실패 |
| **Anthropic SDK 통합** | Claude API 직접 호출 + 에러 핸들링 | API 타임아웃·rate limit 미처리로 UX 중단 |
| **보안 하드닝** | ai-study 허브에서 검증된 패턴 이식 | 각 프로젝트에서 개별 보안 설계는 비효율 |

## 4. 지휘 통제 체계(C2) — 크로스 프로젝트 운용

### 4-1. 허브-워커 지식 순환

```
ai-study (허브: 방법론 센터)
  │
  ├─→ 패턴 검증 (3+ 세션 검증 후)
  │     │
  │     ├─→ moneyflow ← /curate-inbound 이식
  │     ├─→ tarosaju  ← /curate-inbound 이식
  │     └─→ aidy      ← Spec-Driven 독립 운영
  │
  ├─← 워커 피드백 (인시던트, 새 패턴 발견)
  │     └─→ 허브에서 일반화 → 다시 전 프로젝트 배포
  │
  └── /projects-sync: 6레포 상태 동시 진단
  └── /cross-session-review: 다른 Claude 세션 결과 diff 검증
```

**핵심: 양방향 순환.** 허브→워커 일방향이 아니라, 워커에서 발견한 패턴이 허브로 올라오고 다시 전체에 배포된다.

### 4-2. 작전 커맨드 체계 (슬래시 커맨드 12종)

| 작전 단계 | 커맨드 | FDE가 하는 일 |
|----------|--------|-------------|
| **작전 개시** | `/wt-branch` | worktree 기반 안전 분기. squash merge 함정 사전 차단 |
| **정보 수집** | `/ingest` | 외부 URL(유튜브/블로그/논문) → 구조화된 학습 엔트리 변환 |
| **정보 수집** | `/projects-sync` | 6레포 동시 상태 진단. 충돌·드리프트 사전 탐지 |
| **품질 게이트** | `/validate-mdx` | MDX/Mermaid 5대 함정 사전 grep |

| **품질 게이트** | `/cross-session-review` | 다른 Claude 세션 결과물 diff 기반 5단계 검증 |

## 5. FDE 운용 원칙 7가지

현장에서 반복적으로 실패한 후 수립한 원칙들. 이론이 아니라 인시던트에서 나왔다.

| # | 원칙 | 어떤 실패에서 나왔나 |
|---|------|-------------------|
| 1 | **행동 가드 > 메모리 가드** | CLAUDE.md에 "하지 마라" 써놔도 AI가 무시 → 훅/슬래시 커맨드로 물리적 차단 |
| 2 | **N=3 승격** | 같은 솔루션 3번째 작성할 때 "왜 이게 아직 자동화 안 됐지?" → 코드 게이트 자동 승격 |
| 3 | **인시던트 = 다음 입력** | 장애 로그를 버리고 다음 날 같은 실수 반복 → Compound로 구조적 피드백 루프 강제 |
| 4 | **Spec = Law** | 4레포 AI가 각각 "돌아가는 코드" 만들었는데 통합하니 필드 불일치 → 스펙을 유일한 진실로 |
| 5 | **Compound 스킵 금지** | "급하니까 회고 생략" → 2주 후 동일 버그 재발 → 4단계 전부 실행 강제 |
| 6 | **Diff 기반 검증** | 커밋 메시지 "fix: 버그 수정"인데 실제 diff는 전혀 다른 파일 변경 → diff만 신뢰 |
| 7 | **LLM-First** | AI 에이전트가 위키 읽고 "해당 정보 없음" 반환 → 사람용이 아니라 에이전트용으로 구조 재설계 |

## 6. 토큰 경제학 — FDE의 운영 비용 제어

AI를 대규모 배치하면 비용이 폭발한다. FDE는 비용도 엔지니어링한다.

| 레버 | 전략 | 효과 |
|------|------|------|
| **A. Prefix 최적화** | CLAUDE.md 60줄 제한 + 스킬 파일 분산 | 매 턴 고정 토큰 최소화 |
| **B. 캐시 활용** | 0.28-turn 손익분기, cache read 0.1x 비용 | 반복 컨텍스트 90% 절감 |
| **C. 출력 절감** | JIT 검색 (331K → ~800 tokens), 서브에이전트 요약 | 99.8% 컨텍스트 절감 |
| **D. 모델 라우팅** | 컨텍스트 60% 임계값 → 세션 분리 또는 compact | 품질 저하 전 선제 대응 |
| **E. 비용 추적** | ccusage 프로젝트별 분리 (moneyflow+tarosaju 70%) | 레버 ROI 타겟 식별 |

**적용 기준 (4 AND 조건)**: 공식문서 근거 + 실측 데이터 + 부작용 관찰 가능 + 롤백 용이. 하나라도 불충분하면 보류.

## 7. 성숙도 현황 — 자동화 5단계

```
Stage 1: 수동 실행 (사람이 모든 명령 입력)          ✅ 완료
Stage 2: 훅 + 리마인더 (실수 시 자동 경고)          ✅ 완료 (2026-01)
Stage 3: /compound 통합 커맨드 (사이클 자동화)      ✅ 현재 (68%)
Stage 4: /autoceo N-라운드 자동 (무인 스프린트)     🔄 진행 중 (aidy 10-20라운드)
Stage 5: 완전 자율 운영 (FDE는 전략만)              ☐ 목표 2026-Q3
```

**현재 위치**: Stage 3→4 전환 중. aidy에서 `/autoceo` 10-20라운드 무인 스프린트 실행 중.
목표는 FDE가 **"실행"이 아니라 "배치 전략"만 결정**하는 Stage 5.

## 9. FDE 차별화 — "AI를 쓴다"가 아니라 "AI를 배치한다"

### 9-1. 프레임워크 소비자 vs 조합자

대부분의 개발자는 하나의 AI 프레임워크를 그대로 쓴다.
FDE는 **여러 프레임워크를 분해해서 현장에 맞는 모듈만 이식하고, 빈 곳은 직접 설계한다.**

- gstack에서 30+ 스킬, Superpowers에서 SDD, Perpetual Engine에서 3개 패턴 이식
- 나머지(Compound, Harness, Hub-Worker, LLM-First)는 현장에서 자체 설계

### 9-2. 단일 에이전트 vs 다중 에이전트 오케스트레이션

- **aidy**: tmux 4-pane에서 Claude 4개 동시 실행, 스펙 기반 동기화
- **허브-워커**: 프로젝트 간 비동기 메시지 큐로 지식 순환
- **크로스 세션 검증**: 다른 Claude 세션의 결과를 diff 기반으로 교차 검증

### 9-3. "AI 제어"의 구체적 의미

| 계층 | 제어 내용 | 도구 |
|------|---------|------|
| **입력** | 스키마 강제, 프롬프트 구조화 | Zod, CLAUDE.md, SPEC.md |
| **컨텍스트** | 필요한 지식만 정밀 주입 | JIT 벡터 검색, 쿼리 라우터 |
| **추론** | 모델 선택, 비용 라우팅 | 용도별 모델 배정, 60% 임계값 |
| **출력** | 구조 검증, 가짜 정보 탐지 | /validate-mdx, /validate-ai-output |
| **관측** | 성능·비용·적중률 추적 | shadow-benchmark, ccusage, search-hits |
| **행동** | 위험 행동 물리적 차단 | pre-commit hook, .claude hook, N=3 승격 |

### 9-4. 인시던트 기반 진화

이론에서 출발하지 않는다. **현장 인시던트 → 원칙 도출 → 코드 게이트 승격**의 순서.

```
인시던트 발생
  → docs/solutions/ 기록 (1회차)
  → 같은 패턴 2회차 발생 → 솔루션 업데이트
  → 3회차 발생 → /promote-solution → 훅/validator 코드 게이트 승격
  → 이후 자동 차단 (사람 개입 불필요)
```

모든 원칙과 가드는 이 경로를 거쳐 만들어졌다. **"좋은 관행"이 아니라 "반복된 실패의 구조적 대응".**
