이 엔트리는 Blake Crosley의 [Foundation Models Image Input in iOS 27](https://www.blakecrosley.com/blog/foundation-models-image-input-in-ios-27/)을 정독하고 핵심을 추출한 것이다. 원문은 Apple의 WWDC26 세션(237, 241, 319)을 직접 인용하며, 이 엔트리 또한 해당 공식 자료에 근거한다.

### 왜 중요한가

iOS 27에서 Foundation Models 프레임워크에 이미지 입력 기능이 추가된 것은 단순한 기능 확장을 넘어, 온디바이스 AI 앱 개발의 패러다임을 바꾼다. 핵심은 **API 통합 방식**에 있다. 별도의 비전 모델이나 세션을 사용하는 대신, 기존 텍스트 프롬프트에 이미지를 '첨부'하는 방식으로 멀티모달리티를 구현했다.

- **낮은 학습 곡선**: iOS 26의 `LanguageModelSession`에 익숙한 개발자라면, 별도 학습 없이 즉시 이미지 처리 기능을 추가할 수 있다. 세션, 스트리밍, 도구 호출(Tool Calling), `Generable` 타입 등 기존 모든 개념이 그대로 작동한다.
- **아키텍처 명확성**: Apple은 범용 LLM(Foundation Models)과 전문 CV(Vision Framework)의 역할을 명확히 구분하고, 이 둘을 'Tool Calling'으로 연동하는 아키텍처를 공식적으로 제시했다. 이는 "무엇이든 할 수 있는" LLM과 "특정 작업을 빠르고 정확하게 하는" Vision API의 장점을 모두 취하는 가장 실용적인 접근법이다.
- **확장 가능한 스케일**: 온디바이스(4K 컨텍스트)와 Private Cloud Compute(32K 컨텍스트) 모델이 동일한 API를 사용한다. 따라서 간단한 단일 이미지 처리부터 여러 이미지와 긴 문서를 함께 분석해야 하는 복잡한 작업까지, 앱의 요구사항에 맞춰 자연스럽게 확장할 수 있는 경로를 제공한다.

### 핵심 패턴

#### 패턴 1: 프롬프트는 이제 '컨테이너'다

iOS 27에서 프롬프트는 더 이상 단순한 문자열(`String`)이 아니다. 텍스트와 이미지 첨부파일(`ImageAttachment`)을 함께 담는 컨테이너 역할을 한다.

- **API 표면**: 개발자는 `UIImage`, `CGImage`, `CVPixelBuffer`, 파일 URL 등 다양한 소스로부터 `ImageAttachment`를 생성하여 프롬프트에 텍스트와 함께 추가하기만 하면 된다.
- **자율적인 이미지 처리**: 모델은 이미지의 크기나 종횡비에 제약을 두지 않는다. 개발자가 직접 이미지를 자르거나 패딩할 필요가 없다. 단, 큰 이미지는 더 많은 토큰을 소모하고 더 높은 지연 시간을 유발한다는 점은 명심해야 한다.
- **기존 기능과의 완벽한 호환**: 이미지 첨부가 포함된 프롬프트에서도 가이드 생성(`Generable`), 도구 호출(`Tool`), 스트리밍 응답이 모두 기존과 동일하게 작동한다.

```swift
import FoundationModels

// 1. 기존과 동일하게 세션 생성
let session = LanguageModelSession(model: .Default)

// 2. UIImage (또는 다른 소스)로부터 이미지 첨부파일 생성
guard let uiImage = UIImage(named: "origami-crane") else { return }
let imageAttachment = ImageAttachment(uiImage)

// 3. 프롬프트에 텍스트와 이미지를 함께 포함
let prompt = Prompt(
    "이 사진에 있는 종이접기 동물의 이름이 뭐야?",
    attachments: [imageAttachment]
)

// 4. 응답 생성 (스트리밍 또는 전체)
let stream = session.generate(prompt: prompt)
for await chunk in stream {
    print(chunk, terminator: "")
}
// 출력 예상: "사진 속 종이접기는 학(crane)입니다."
```

#### 패턴 2: 범용 LLM과 전문 Vision의 역할 분담 및 협력

Apple은 WWDC 세션 237에서 "무엇을 빌드할지 결정하는 데 가장 유용한" 가이드라인을 제시했다. 바로 Foundation Models와 Vision 프레임워크의 역할을 명확히 나누고, 필요할 때 협력시키는 것이다.

- **Foundation Models (LLM)**: 이미지에 대한 개방형, 서술형 질문에 적합하다. "이 방의 인테리어 개선 아이디어를 제안해줘" 또는 "냉장고 사진으로 만들 수 있는 레시피를 알려줘"와 같은 작업에 강하다.
- **Vision Framework**: 얼굴 인식, 텍스트 인식(OCR), 바코드 스캔, 자세 추정 등 특정 작업을 매우 빠르고 정확하게 수행하는 데 특화되어 있다. 실시간 영상 프레임 분석에 적합하다.

이 둘을 선택하는 대신, **Tool Calling을 통해 연동**하는 것이 Apple의 권장 아키텍처다.

위 다이어그램처럼, LLM은 전체적인 맥락을 이해하고 어떤 도구가 필요한지 판단하는 '지휘자' 역할을 한다. Vision API는 빠르고 정확한 '전문가'로서 특정 데이터를 추출해 LLM에게 돌려준다. LLM은 이 데이터를 바탕으로 최종적인 사용자 친화적 답변을 생성한다.

#### 패턴 3: 컨텍스트 창에 따른 스케일링 전략

온디바이스 모델과 서버 모델은 동일한 API를 공유하지만, 컨텍스트 창 크기에서 결정적인 차이를 보인다.

- **온디바이스 모델 (4K 컨텍스트)**: 단일 이미지와 짧은 텍스트 프롬프트에 이상적이다. 빠른 응답과 완전한 개인정보 보호가 장점이다.
- **Private Cloud Compute 모델 (32K 컨텍스트)**: 여러 장의 이미지와 긴 문서(예: Markdown 파일)를 함께 분석해야 할 때 필수적이다. 더 큰 컨텍스트를 통해 이미지와 텍스트 간의 복잡한 관계를 이해하고 요약하는 능력이 뛰어나다.

### 실전 적용: `moneyflow`에 영수증 자동 입력 기능 구현

개인 자산 관리 앱 `moneyflow`에 이 패턴을 적용하여 영수증 사진 한 장으로 지출 내역을 자동 입력하는 기능을 구현할 수 있다.

1.  **사용자 경험**: 사용자가 '영수증 스캔' 버튼을 누르고 영수증 사진을 찍는다.
2.  **프롬프트 구성**: 앱은 촬영된 영수증 `UIImage`로 `ImageAttachment`를 생성하고, 아래와 같은 텍스트와 함께 `LanguageModelSession`에 전달한다.
    ```swift
    let prompt = Prompt(
        """
        이 영수증 이미지를 분석해줘.
        1. OCRTool을 호출해서 모든 텍스트를 추출해.
        2. 추출된 텍스트에서 가게 이름, 총 결제 금액, 결제 날짜를 찾아내서 아래 JSON 형식으로 반환해줘.
        { "merchant": "가게이름", "totalAmount": 12345, "date": "YYYY-MM-DD" }
        """,
        attachments: [receiptImageAttachment]
    )
    ```
3.  **LLM + Vision 협력**:
    - 온디바이스 LLM이 프롬프트를 받는다.
    - LLM은 `OCRTool`을 호출해야 한다고 판단하고, 이미지 참조(`ImageReference`)를 인자로 넘겨 내장된 `OCRTool`을 실행시킨다.
    - `OCRTool`(Vision 프레임워크 기반)이 영수증 이미지에서 텍스트를 빠르고 정확하게 추출하여 LLM에게 반환한다.
    - LLM은 반환된 텍스트 덩어리에서 '가게 이름', '총액', '날짜'에 해당하는 정보를 추론하고, 요청받은 JSON 형식으로 최종 결과를 생성한다.
4.  **데이터 처리**: `moneyflow` 앱은 LLM으로부터 구조화된 JSON 응답을 받는다. 이 데이터를 파싱하여 검증한 후, 사용자의 확인을 거쳐 새로운 지출 내역으로 데이터베이스에 저장한다.

이 방식은 단순히 LLM의 시각 능력에만 의존하는 것보다 훨씬 안정적이고 정확하다. 텍스트 인식이라는 전문적인 작업은 가장 빠르고 정확한 도구인 Vision 프레임워크에 위임하고, LLM은 전체적인 의미 파악과 데이터 구조화라는 고차원적인 작업에 집중하기 때문이다. 이는 Apple이 제시한 아키텍처의 핵심 철학을 그대로 따르는 것이다.
