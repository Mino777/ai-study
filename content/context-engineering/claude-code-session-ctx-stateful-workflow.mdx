LLM 기반 코딩 워크플로우의 가장 큰 걸림돌 중 하나는 '상태 없음(Statelessness)'입니다. 새로운 채팅 세션을 시작할 때마다 우리는 같은 프로젝트 구조, 동일한 API 명세, 반복되는 디자인 시스템 규칙을 처음부터 다시 설명해야 합니다. 이는 마치 매일 아침 기억을 잃는 동료와 협업하는 것과 같습니다. 이 문제를 해결하기 위해 등장한 것이 바로 세션 간 컨텍스트를 영속화(persist)하는 `CTX`와 같은 도구입니다.

이 글에서는 `CTX`를 활용하여 Claude와의 코드 세션을 단발성 대화가 아닌, 지식이 축적되는 연속적인 협업으로 전환하는 방법을 iOS/프론트엔드 개발자 관점에서 구체적으로 다룹니다.

## 왜 세션 간 컨텍스트 지속성이 중요한가?

기존의 LLM 워크플로우는 각 세션이 독립적인 섬과 같습니다. 어제 SwiftUI 뷰의 데이터 바인딩에 대해 깊이 논의했더라도, 오늘 새로운 세션에서 관련 질문을 하려면 어제의 모든 맥락을 다시 주입해야 합니다. 이는 단순히 비효율적인 것을 넘어 개발자의 인지 부하를 가중시키고 창의적인 흐름을 방해합니다.

`CTX`는 이러한 단절된 세션들 사이에 다리를 놓아, LLM을 상태 저장(Stateful) 파트너로 만듭니다. 이는 LLM의 지능을 일회성으로 소비하는 것이 아니라, 프로젝트 라이프사이클에 걸쳐 '육성'하는 패러다임의 전환을 의미합니다.

Tools that manage and persist context across interactions are essential for moving from simple chat to complex, multi-session tasks. ## CTX의 작동 원리와 기본 사용법

`CTX`는 로컬 파일 시스템을 일종의 외부 메모리로 사용하여 명명된 컨텍스트 스냅샷을 저장하고 불러오는 간단한 CLI(Command-Line Interface) 도구입니다. `pip install ctx` 또는 Claude의 `/plugin install`을 통해 설치할 수 있습니다.

핵심 동작은 다음 다이어그램으로 요약할 수 있습니다.

```mermaid
graph TD
    subgraph "Session 1 (어제)"
        A["User: SwiftUI ViewModel 코드 제공"] --> B{Claude};
        B --> C["응답: View 로직 제안"];
        C --> D["User: ctx save moneyflow-viewmodel"];
        D --> E["(Local Storage: moneyflow-viewmodel.ctx)"];
    end

    subgraph "Session 2 (오늘)"
        F[User: "새로운 세션 시작"] --> G["User: ctx load moneyflow-viewmodel"];
        G --> E;
        E --> H{Claude};
        H -- "ViewModel 구조를 · 기억한 상태" --> I[User: "이 ViewModel에 · 에러 핸들링 로직 추가해줘"];
        I --> J["응답: 정확한 코드 제안"];
    end
```

### 기본 명령어

-   **`ctx save <name>`**: 현재 대화의 컨텍스트를 `<name>`으로 저장합니다.
-   **`ctx load <name>`**: `<name>`으로 저장된 컨텍스트를 현재 세션으로 불러옵니다.
-   **`ctx list`**: 저장된 모든 컨텍스트 목록을 보여줍니다.
-   **`ctx add <file_path>`**: 파일 내용을 현재 컨텍스트에 추가합니다.

### 실제 iOS 개발 예제

**상황**: `TransactionListViewModel.swift` 파일 작업을 하고 있습니다.

**1. 첫 번째 세션: 컨텍스트 저장**

먼저 Claude에게 ViewModel의 기본 구조를 제공합니다.

```swift
// TransactionListViewModel.swift
import Foundation
import Combine

// API Contract에 정의된 Transaction 모델
struct Transaction: Identifiable {
    let id: String
    let amount: Double
    let date: Date
    let description: String
}

class TransactionListViewModel: ObservableObject {
    @Published var transactions: [Transaction] = []
    @Published var isLoading = false
    
    private var cancellables = Set<AnyCancellable>()
    private let transactionService: TransactionServiceProtocol // 의존성 주입

    init(transactionService: TransactionServiceProtocol) {
        self.transactionService = transactionService
    }

    func fetchTransactions() {
        isLoading = true
        transactionService.fetchTransactions()
            .receive(on: DispatchQueue.main)
            .sink(receiveCompletion: { [weak self] completion in
                self?.isLoading = false
                // 에러 처리 필요
            }, receiveValue: { [weak self] transactions in
                self?.transactions = transactions
            })
            .store(in: &cancellables)
    }
}
```

이 코드를 Claude에게 주고 기본적인 피드백을 받은 후, 세션을 종료하기 전에 컨텍스트를 저장합니다.

```bash
/edit
# (위의 Swift 코드를 붙여넣기)

# Claude와 대화 후...
ctx save swiftui-transaction-vm
```

**2. 두 번째 세션: 컨텍스트 불러오기 및 작업 재개**

다음 날, 새로운 기능(Paging)을 추가하기 위해 새 세션을 시작합니다.

```bash
# 새로운 Claude 채팅창에서
ctx load swiftui-transaction-vm

# 이제 Claude는 ViewModel 구조를 이미 알고 있습니다.
# 바로 본론으로 들어갑니다.

```

이 워크플로우는 매번 `Transaction` 모델과 `TransactionListViewModel` 클래스 전체를 복사-붙여넣기 해야 했던 과정을 단 한 줄의 명령어로 대체합니다.

## CTX를 활용한 고급 컨텍스트 엔지니어링 패턴

`CTX`는 단순한 저장/불러오기를 넘어 더 정교한 워크플로우를 구축하는 기반이 될 수 있습니다.

| 패턴 이름                      | 설명                                                                                             | 사용 사례 (`ctx` 명령어 예시)                                                                     |
| ------------------------------ | ------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| **레이어드 컨텍스트 (Layered)**    | 기본 컨텍스트 위에 특정 기능 컨텍스트를 겹쳐서 로드하는 방식. Git의 브랜치와 유사.                 | `ctx load project-core`<br />`ctx add feature-x.swift`                                             |
| **역할 기반 컨텍스트 (Role-Based)** | '리팩토링 전문가', 'TDD 코치' 등 특정 역할에 맞는 원칙과 가이드라인을 컨텍스트로 저장.         | `ctx load tdd-coach-rules`<br />"이 코드에 대한 테스트 케이스를 작성해줘."                       |
| **프로젝트 스냅샷 (Snapshot)**     | 특정 시점의 주요 파일들(API 명세, 핵심 모델, 아키텍처 문서)을 묶어 하나의 컨텍스트로 관리.       | `cat api.md models.swift arch.md \| ctx save moneyflow-v1.0`                                      |
| **휘발성/영속성 분리 (Separation)** | 간단한 질문은 일반 대화로, 프로젝트 핵심 정보는 `CTX`를 통해 영속성 레이어에서 관리.            | 일반 채팅: "Swift에서 map과 flatMap의 차이?"<br />`CTX` 사용: `ctx load my-project-api-contract` |

## `moneyflow` 프로젝트 적용 시나리오

`moneyflow` iOS 앱의 '계좌 연동' 기능을 개발하는 상황을 가정해봅시다. 이 기능은 복잡한 API 상호작용과 여러 UI 상태(로딩, 성공, 실패, 재인증 필요 등)를 가집니다.

1.  **기반 컨텍스트 생성 (`ctx save moneyflow-core`)**:
    *   `api-contract.md`: 계좌 연동 관련 Plaid API 명세
    *   `DesignSystem.swift`: 앱의 기본 색상, 폰트, 컴포넌트 스타일
    *   `NetworkClient.swift`: 기본 네트워킹 레이어 코드
    *   위 파일들의 내용을 합쳐 `moneyflow-core` 라는 이름의 베이스 컨텍스트를 생성합니다.

2.  **기능 개발 시작 (`ctx load moneyflow-core`)**:
    *   새로운 세션에서 `ctx load moneyflow-core`를 실행합니다.
    *   이제 Claude는 Plaid API가 어떻게 동작하는지, 앱의 디자인 시스템이 어떤지 이미 알고 있습니다.
    *   "Plaid 연동을 위한 SwiftUI 뷰를 만들어줘. 로딩, 성공, 실패 상태를 표시해야 하고, 우리 앱의 primaryButton 스타일을 사용해줘." 와 같이 고차원적인 지시가 가능해집니다.

3.  **작업 중단 및 재개 (`ctx save account-linking-wip`)**:
    *   퇴근 전, 현재까지 작성된 `AccountLinkingView.swift`와 `AccountLinkingViewModel.swift` 파일의 내용을 기존 컨텍스트에 추가하고 `account-linking-wip` (Work-In-Progress)로 저장합니다.
    *   `ctx add AccountLinkingView.swift`
    *   `ctx add AccountLinkingViewModel.swift`
    *   `ctx save account-linking-wip`
    *   다음 날, `ctx load account-linking-wip` 만으로 어제의 모든 작업 맥락을 즉시 복원하고 "이제 재인증 필요 상태에 대한 UI를 추가하자"며 작업을 이어갈 수 있습니다.

이러한 상태 저장 워크플로우는 단순한 시간 절약을 넘어, LLM을 프로젝트의 장기적인 맥락을 이해하는 진정한 페어 프로그래밍 파트너로 격상시킵니다.
