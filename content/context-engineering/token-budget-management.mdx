## Token Budget의 3계층 구조

AI 에이전트의 토큰 비용을 제어하려면 단순히 "전체 비용을 줄여야 한다"는 추상적 목표가 아니라, **계층별 예산을 명확히 설정**해야 한다.

```mermaid

    A["프로젝트 Annual Budget · (전사 정책)"] --> B["프로젝트 Monthly Budget · (비용 운영)"]

    C --> D["작업 Task Budget · (태스크별 토큰 쿼터)"]
```

이 3계층은 **정책 → 운영 → 실시간**으로 내려가며, 각 레벨에서 다른 대응 전략을 써야 한다.

### Session Budget — 60% 임계값이 실제 사례

ai-study CLAUDE.md의 "60% 임계값" 규칙은 **이론이 아니라 실전 관찰의 산물**이다.

> **Context Window 관리 규칙**: 60% 임계값 — 컨텍스트 윈도우 60% 이상 소진 시 태스크 분리 또는 `/compact` 실행. **초과 상태에서의 의사결정은 품질이 떨어진다.**

이 규칙의 근거:
- **0~40%**: 최적 추론 구간. LLM이 new context와 충분한 "생각할 공간"을 가짐
- **40~60%**: 선형 품질 저하 구간. 정확도 저하가 보이지만 여전히 가능
- **60% 이상**: 비선형 품질 저하. 기억은 잘하지만 창의적 추론, 엣지 케이스 처리, 버그 감지 능력 급격히 떨어짐

따라서 **60% 도달 시 작업을 분리해서 새 Session을 시작**하는 게 최적 경로다.

### Task Budget — 작업별 토큰 쿼터 설정

개별 작업(예: "엔트리 생성", "PR 리뷰", "버그 디버깅")마다 예상 토큰 예산을 정한다.

```typescript
interface TaskBudget {
  taskId: string;
  estimatedInputTokens: number;    // 작업 설명 + 컨텍스트
  estimatedOutputTokens: number;   // 응답 기대 길이
  maxReruns: number;               // 재시도 횟수 한계
  fallbackModel?: string;          // 예산 초과 시 대체 모델
}

// 실례
const entryCreationBudget: TaskBudget = {
  taskId: "wiki-entry-create",
  estimatedInputTokens: 4000,      // 주제 + CLAUDE.md + 기존 엔트리 2개
  estimatedOutputTokens: 2000,     // MDX 본문 ~150줄
  maxReruns: 2,                    // 첫 시도 + 1회 수정
  fallbackModel: "claude-3-5-sonnet" // 초과 시 Sonnet으로 다운그레이드
};
```

**작업 단위 예산 책정의 이점:**
- 각 작업의 "예상 비용"을 미리 파악 → 월 예산과 대조
- 초과 시 대응 전략(재시도 금지, 모델 변경 등)을 미리 결정
- 여러 작업이 있을 때 우선순위 조정 가능

### Agent Budget — API 호출당 스펙

AI 에이전트(subagent, 마이크로에이전트)가 외부 API를 호출할 때의 토큰 스펙.

```typescript
interface AgentBudget {
  agentName: string;
  maxTokensPerCall: number;
  cachingStrategy: "ephemeral" | "persistent" | "none";
  rateLimitPerHour: number;
}

// ai-study 과외 선생님
const lessonAgentBudget: AgentBudget = {
  agentName: "ai-tutor",
  maxTokensPerCall: 8000,        // Gemini API 호출 당 최대 소비
  cachingStrategy: "ephemeral",   // 시스템 프롬프트 캐싱 (90% 절감)
  rateLimitPerHour: 10            // 하루 3개 주제만 생성
};
```

## Budget 초과 시 대응 전략

**우선순위 순서** (손상도 낮은 순):
1. **Context Pruning** — 현재 세션에서 불필요한 컨텍스트(log, old history, duplication) 제거
2. **Task Splitting** — 현재 작업을 여러 세션으로 분리 (격리되어 각 세션이 fresh context 시작)
3. **Model Downgrade** — Claude Opus → Sonnet (품질 저하가 크므로 마지막 수단)

### Context Pruning 체크리스트

```typescript
// ai-study CLAUDE.md의 JIT 검색 선택 기준
type PruningStrategy = {
  // 현재: 위키 331K tokens → JIT ~800 tokens (99.8% 절감)
  
  // 검색으로 충분: 패턴/방법론 찾기, 에러 해결책, 관련 엔트리
  // MDX 전체 필요: 핵심 개념 이해, 새로운 영역 학습
};
```

**실전 예시:**
- 세션이 50%에 도달 → 지난 10턴의 repetitive output 제거
- 61% 도달 → 작업을 2개로 split (예: "엔트리 생성" + "검증"을 별도 세션)
- Opus에서 재시도 실패 → Sonnet으로 재시도 (속도 3배, 가격 1/3)

## RTK — Token Budget 인프라 레이어

**Rust Token Killer(RTK)**는 실제로 Token Budget Management의 기반이다.

```bash
# RTK가 하는 일
bash command
↓ (30-40% 절감)
rtk <command>  # bash output 요약, 관련성 필터링
↓
Claude의 컨텍스트에 최소한만 들어옴
```

**RTK의 세 가지 절감 경로:**
1. **Output Filtering** — 불필요한 stdout 자동 제거 (예: "npm install" 시 dependency tree 요약)
2. **Command Rewrite** — 이미 알려진 명령 결과는 캐시 사용 (git status, ls 등)
3. **Error Context Compression** — 스택트레이스 → 핵심 에러 메시지만 추출

**RTK 없이는 Budget 관리 불가능:**
- 매 git status ~200 tokens → RTK 사용 시 ~50 tokens
- npm install의 npm WARN 로그 → 무시
- 빌드 에러 시 전체 log 대신 마지막 5줄만 제공

## Prompt Caching으로 90% 절감한 실측

실제 프로덕트(moneyflow, tarosaju)에서 cache_control: ephemeral 적용:

```typescript
// 시스템 프롬프트 캐싱
const request = {
  model: "claude-3-5-sonnet",
  max_tokens: 1024,
  system: [
    {
      type: "text",
      text: systemPrompt,        // 반복되는 대규모 프롬프트
      cache_control: { type: "ephemeral" }
    }
  ],
  messages: [{ role: "user", content: userQuery }]
};

// 첫 호출: system_prompt 캐시 write = 1.25x cost
// 2번째 이상: system_prompt 캐시 read = 0.1x cost
// 결과: 반복 호출 시 평균 90% 절감
```

**계산식:**
- System prompt 토큰: 10,000개
- Breakeven: (1.25 - 1) / (1 - 0.1) = 0.28번 (1번 미만!)
- 1달에 100번 호출: (10,000 × 1.25) + (100 × 10,000 × 0.1) = 112,500 tokens 절감

**moneyflow 실전 데이터:**
- 적용 전: 월 $400 API 비용
- 적용 후: 월 $40 API 비용
- 절감률: 90%

## ccusage로 Budget 기준점 설정

프로젝트별 비용 분석으로 Budget을 데이터 기반 설정:

```bash
# 전체 비용 보기
$ ccusage session

# 프로젝트별 분리
$ ccusage session --instances

# 출력 예시 (4주 합계)
Project        Input     Output    Total Cost    Usage Pattern
─────────────────────────────────────────────────────────────
ai-study       1.2M      0.3M      $18          Interactive (가볍)
moneyflow      8.5M      2.1M      $145         Batch processing (매일)
tarosaju       7.2M      1.9M      $138         Scheduled jobs (매주)
─────────────────────────────────────────────────────────────
Total                                $301        (moneyflow+tarosaju = 93%)
```

**Budget 설정 기준:**
- moneyflow: 월 $50 상한 (현재 $35 추이) — 여유 30%
- tarosaju: 월 $50 상한 (현재 $35 추이) — 여유 30%
- ai-study: 월 $10 상한 (현재 $4.50 추이) — 여유 55%

이 "여유"가 바로 **Task/Agent Budget의 기초**가 된다.

## TokenBudgetManager — 실전 구현

```typescript
class TokenBudgetManager {
  private sessionBudget = { maxTokens: 200000, warningAt: 0.6 };
  private currentUsage = 0;
  
  recordTokens(inputTokens: number, outputTokens: number) {
    this.currentUsage += inputTokens + outputTokens;
    
    const utilization = this.currentUsage / this.sessionBudget.maxTokens;
    
    if (utilization >= this.sessionBudget.warningAt) {
      return {
        status: "warning",
        action: "split-task | prune-context",
        utilization: Math.round(utilization * 100) + "%"
      };
    }
    
    return { status: "ok", utilization: Math.round(utilization * 100) + "%" };
  }
  
  estimateTaskBudget(taskName: string): TaskBudget {
    const estimates: Record<string, TaskBudget> = {
      "wiki-entry": { input: 4000, output: 2000, maxRetries: 2 },
      "pr-review": { input: 6000, output: 1500, maxRetries: 1 },
      "bug-debug": { input: 8000, output: 2500, maxRetries: 3 }
    };
    return estimates[taskName];
  }
}

const manager = new TokenBudgetManager();
manager.recordTokens(45000, 12000);  // { status: 'ok', utilization: '28%' }
manager.recordTokens(45000, 12000);  // { status: 'warning', action: '...' }
```

## 요약: Budget의 3계층

| 계층 | 담당 | 빈도 | 대응 |
|------|------|------|------|
| **Project Budget** | 월 운영 | 월 1회 | 기획/비용 조정 |
| **Session Budget** | 개발자 | 매 세션 | 60% → split/compact |
| **Task Budget** | 태스크 | 작업 단위 | 초과 → prune/downgrade |
| **Agent Budget** | API 호출 | 매 호출 | caching/rate-limit |

RTK + Prompt Caching + JIT 검색의 조합으로 이 4개 계층이 **자동으로 서로 강화**된다. 각 계층을 모니터링하지 않으면 한 달에 $1000+ 낭비할 수 있다.
