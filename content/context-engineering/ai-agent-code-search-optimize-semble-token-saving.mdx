## 왜 grep은 AI 에이전트에게 독(毒)인가?

`grep`은 특정 문자열 패턴을 찾는 데는 강력하지만, 코드의 구조나 의미를 전혀 이해하지 못합니다. AI 에이전트가 "사용자 인증 관련 로직을 찾아줘"라고 요청했을 때, `grep 'auth'` 와 같은 명령은 수많은 관련 없는 결과를 쏟아냅니다. 주석, 변수명, 심지어 다른 모듈의 함수 호출까지 무차별적으로 포함되기 때문입니다.

이 '오염된' 컨텍스트는 두 가지 치명적인 문제를 야기합니다.

1.  **토큰 예산 낭비**: LLM은 불필요한 코드 조각을 처리하느라 한정된 컨텍스트 윈도우와 연산 비용을 소모합니다. 이는 응답 품질 저하와 비용 증가로 직결됩니다.
2.  **에이전트 할루시네이션(Hallucination)**: 관련 없어 보이는 코드 조각들이 한 컨텍스트에 묶여 제공되면, LLM은 존재하지 않는 연관성을 상상해내거나 잘못된 추론을 할 가능성이 높아집니다.

한 연구에 따르면, `grep`을 사용한 코드 검색은 Semble과 같은 의미 기반 검색에 비해 최대 98% 더 많은 토큰을 사용할 수 있습니다. 이는 단순히 비용 문제를 넘어, 에이전트의 근본적인 성능을 저해하는 핵심 병목 지점임을 의미합니다.

## Semble의 접근법: AST와 시맨틱 분석의 결합

Semble은 `grep`의 문자열 매칭과 순수 벡터 검색의 한계를 극복하기 위해 코드의 구조적, 의미적 정보를 함께 활용합니다. 핵심은 추상 구문 트리(Abstract Syntax Tree, AST)를 분석하여 코드의 논리적 단위를 식별하고, 이를 기반으로 시맨틱 검색을 수행하는 것입니다.

### Swift 코드에서의 적용 예시

iOS 개발자가 `moneyflow` iOS 클라이언트에서 "결제 실패 시 사용자에게 알림을 보내는 로직"을 찾고자 한다고 가정해 봅시다.

`grep`은 `payment`, `failed`, `alert` 등의 키워드가 포함된 모든 파일을 검색하여 수많은 `UIViewController`의 UI 코드, 로깅 메시지, 테스트 코드까지 반환할 것입니다.

반면 Semble은 Swift 코드의 AST를 분석하여, `processPayment`와 같은 함수 시그니처, `completion` 핸들러의 에러 케이스, 그리고 그 내부에서 `UNUserNotificationCenter`나 `UIAlertController`를 호출하는 특정 코드 블록을 하나의 의미 단위로 묶어 식별합니다. 에이전트에게는 바로 이 핵심 로직만 전달되므로, 불필요한 UI 설정 코드나 테스트 케이스는 컨텍스트에서 원천적으로 배제됩니다.

## 비교 분석: Grep, 벡터 검색, 그리고 Semble

각 접근 방식은 명확한 장단점을 가지며, 해결하려는 문제에 따라 적합성이 달라집니다.

| 평가 기준 | `grep` (문자열 검색) | 표준 벡터 검색 (RAG) | Semble (AST + 시맨틱) |
| :--- | :--- | :--- | :--- |
| **핵심 원리** | 정규 표현식 기반 문자열 매칭 | 임베딩 벡터 간의 거리 계산 | 코드 구조(AST) 분석 + 시맨틱 임베딩 |
| **토큰 효율성** | 매우 낮음 (관련 없는 코드 다수 포함) | 중간 (의미는 유사하나 구조적으로 무관한 코드 포함 가능) | 매우 높음 (필요한 코드 블록만 정확히 추출) |
| **정확성 (Precision)** | 낮음 | 중간 | 높음 |
| **설정 복잡성** | 매우 낮음 (기본 제공) | 높음 (임베딩 모델, 벡터 DB 등 필요) | 중간 (전용 엔진 또는 라이브러리 필요) |
| **최적 사용 사례** | 특정 변수명, API 엔드포인트 등 정확한 문자열 검색 | 유사한 기능이나 개념을 가진 코드 조각 탐색 | 복잡한 로직 흐름, 함수 정의 등 코드의 의미/구조 기반 검색 |
| **치명적 단점** | 코드의 의미/구조를 전혀 이해하지 못함 | 코드의 계층 구조나 호출 관계를 파악하기 어려움 | 정확한 문자열 리터럴이나 주석 검색에는 부적합할 수 있음 |

### 언제 Semble을 사용하면 안 되는가?

Semble의 접근법이 항상 우월한 것은 아닙니다. 다음과 같은 상황에서는 오히려 전통적인 방식이 더 효과적일 수 있습니다.

*   **특정 설정 값이나 API 키 검색**: `grep "API_KEY"`처럼 정확한 문자열 리터럴을 찾아야 할 때는 시맨틱 분석이 불필요하며, 오히려 결과를 방해할 수 있습니다.
*   **코드베이스 초기 탐색**: 아직 코드 구조에 대한 이해가 전혀 없는 상태에서 광범위하게 특정 키워드가 어떻게 사용되는지 패턴을 파악하고 싶을 때는 `grep`의 광범위한 결과가 더 유용할 수 있습니다.
*   **매우 동적인 언어**: AST 분석이 어렵거나 불가능한 일부 동적 언어 환경에서는 Semble과 같은 도구의 효과가 제한될 수 있습니다.

결국 핵심은 문제의 성격에 맞는 올바른 도구를 선택하는 것입니다. AI 에이전트의 워크플로우 내에서 여러 검색 전략을 조합하는 하이브리드 접근법이 가장 현실적인 해결책이 될 수 있습니다.
