## 한 줄 요약

[Token Levers Catalog](/wiki/tokenomics/claude-code-token-levers-catalog)에서 검증된 레버 중 **즉시 적용 가능 4 조건**을 통과한 것만 ai-study / moneyflow / tarosaju 3 프로덕트에 박제. 적용 전 `ccusage` 베이스라인 측정 → 적용 → 재측정 → 로그 append.

---

## 즉시 적용 가능 4 조건 (AND — 모두 만족)

1. **공식 문서 또는 2+ 독립 소스로 검증**된 수치 (커뮤니티 단일 주장은 제외)
2. **실측 수치 재현 가능** — 적용 전/후 `ccusage` 또는 `/context`로 토큰 감소 측정 가능
3. **부작용이 *관찰 가능한* 레이어에 한정** — 응답 품질 저하 / 기능 삭제 / UX 퇴행이 없거나 명확히 트레이드오프 문서화됨
4. **롤백 비용이 낮음** — 설정 한 줄 추가 / env var / 커맨드 1개 수준. 코드 리팩터링이나 아키텍처 변경은 제외

## 보류 조건 (OR — 하나라도 해당 시 적용 X)

- 공식 문서에 명시 안 된 *경험칙*만 있음
- 실측이 단일 세션 샘플 — 세션당 변동성이 큼
- 응답 품질 저하가 *가능성으로* 언급된 레버 (`/fast`, 공격적 auto-compact, thinking budget 축소 등)
- 프로젝트 구조 변경이 필요 (파일 분할, 아키텍처 리팩터)
- 카탈로그 §안티패턴에 포함된 것

## 적용 순서 (안전한 것부터)

1. **베이스라인 측정** — 적용 전 `ccusage` 스냅샷, 어느 덩어리(A/B/C)가 가장 큰지 파악
2. **단일 레버 단위 적용** — 한 번에 한 레버만, 직후 `/cost` + `ccusage`로 검증
3. **3 사이클 후 재측정** — 며칠간 세션을 돌려본 후 효과 유지 여부 (transient 아닌지)
4. **프로젝트 간 이식** — 한 프로젝트에서 검증되면 박제 전이성 패턴으로 나머지에 이식 ([Journal 011 모델](/wiki/harness-engineering/harness-journal-011-concurrent-session-safety))

### 2026-04-13 — `.claudeignore` 신설 (moneyflow iOS)

- **대상**: iOS SPM 멀티모듈 프로젝트

- **모듈당**: 6~7GB
- **합산**: ~130GB
- **함정**: 루트의 `.build/`만 보면 99% 놓침 → **`**/.build/` 패턴 필수**
- **추가 배제**:
  - `vendor/` (CocoaPods 63MB)
  - `external-spm/` (외부 SPM 패키지 미러 디렉토리)
  - `*.xcodeproj/` (Xcode XML — generated)

> 자세한 SPM 함정 해부: [iOS Harness Journal 003 — .claudeignore + SPM 멀티모듈 함정](/wiki/ios-ai/ios-ai-journal-003-claudeignore-spm)

## 실측 결과 요약 (2026-04-14 기준)

| 레버 | 측정 방법 | 절감량 | 비고 |
|---|---|---|---|
| **RTK (Rust Token Killer)** | `rtk gain` | **47.4M tokens / 2,234 commands (99.5%)** | grep 43.7M (최대 기여), vitest 3.2M, read 154K. 누적 |
| **`.claudeignore` (ai-study)** | du + 토큰 추정 | **~194K tokens / 세션 context (~50%)** | package-lock.json 117K + src/generated 48K + docs/* 29K |
| **`.claudeignore` (moneyflow iOS)** | du 실측 | **~130GB 파일 탐색 차단** | 각 SPM 모듈 `.build/` 6~7GB × N + vendor/ 63MB |
| **Prompt Caching (자동)** | ccusage | **cache read 98.6% (7.22B / 7.33B)** | Anthropic 자동 캐싱. 실비용이 총 토큰 대비 낮은 핵심 이유 |

**종합**: RTK가 CLI 출력의 99.5%를 필터링하고, `.claudeignore`가 세션 context를 50% 축소하며, API cache read가 98.6%로 비용을 극적으로 낮춤. 3 레버가 서로 다른 레이어(CLI 출력 / context prefix / API 비용)에서 **독립적으로** 작동하여 중복 없이 누적 효과.

**일일 비용 베이스라인 (2026-04-14)**: ~305M tokens / ~$180 (24일 평균). 이전 베이스라인 대비 15% 감소. 3 레버가 *이미 적용된* 상태.

## AI Agent Directive

**Trigger**: 새로운 토큰 절감 레버를 발견하거나 프로덕트에 적용했을 때 / "토큰 비용을 줄이고 싶다" 요청 시
**Prerequisites**: [Claude Code Token Levers Catalog](/wiki/tokenomics/claude-code-token-levers-catalog) / [AI API Cost Tracking Pattern](/wiki/harness-engineering/ai-api-cost-tracking-pattern)

### Actionable Steps
1. **새 레버 발견 시**: 먼저 카탈로그의 4 조건 게이트 검증 (공식 문서 확인, 실측 가능, 부작용 관찰 가능, 롤백 비용 낮음)
2. **4 조건 모두 통과 시**: 프로덕트 적용 전에 이 로그에 entry 추가 예정 명시
3. **적용 직후**: 레버 번호 + before/after 수치 + 부작용 + 롤백 방법을 반드시 기록 (4가지 필수)
4. **베이스라인 재측정**: 적용 3 사이클 경과 후 ccusage 비교 · 지속성 확인
5. **안티패턴 회귀 방어**: 카탈로그 §안티패턴과 적용 로그 §보류 조건 재확인

### Anti-patterns
- **실측 없이 레버 적용** — "원리상 절감될 것 같은데" 추측만으로 박제하면 신뢰도 하락
- **한 번에 여러 레버 동시 적용** — 어느 것이 효과인지 식별 불가 · 롤백 복잡화
- **롤백 비용 무시** — "나중에 돌릴 수 있지" 마음으로 도입하면 기술 부채 증가

이 문서는 **append-only live document**다. 새 레버를 적용한 직후:

1. 카탈로그 §우선순위 표의 **레버 번호**를 적용 로그 헤더에 명시
2. **before/after 토큰 수** (`ccusage` 또는 `/context`)
3. **부작용** (있으면 명시, 없으면 "관찰된 부작용 없음")
4. **롤백 방법** (1줄)

이 4가지 없이 로그 append 금지. 실측 없는 박제는 카탈로그 신뢰도를 깎는다.
