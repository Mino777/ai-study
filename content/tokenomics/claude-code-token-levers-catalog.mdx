## 한 줄 요약

토큰 절감은 "적게 쓰기"가 아니라 **prefix × cache warmth × 모델 라우팅 × output 필터**의 *곱셈* 문제다. RTK는 마지막 축의 bash 부분만 덮는다. 나머지 세 축에 박히는 15개 레버를 5 카테고리로 정리. 계측(§계측) → 우선순위 표(§우선순위) → 본인 환경 검증 순으로 공략.

---

## 토큰은 4 곱셈 — 3 덩어리 모델 + cache 수학

한 세션의 토큰은 세 덩어리로 쪼개진다:

| 덩어리 | 정체 | RTK 커버? |
|---|---|---|
| **A. Prefix (매 턴 반복)** | 시스템 프롬프트 + CLAUDE.md + MCP 툴 정의 + skill 1줄 설명 | ❌ |
| **B. Tool output** | Bash / Read / Grep / MCP 결과 | ⭕ (bash만) |
| **C. Thinking + assistant output** | extended thinking + 응답 텍스트 | ❌ |

RTK는 B의 bash 서브 카테고리에만 꽂혀 있다. **A, C, B의 비-bash 부분은 여전히 손 안 댄 상태** — 여기가 남은 절감 여지.

### 베이스라인: 자동 prompt caching 수학

Claude Code는 prompt caching을 *기본값*으로 적용한다:

- **cache read = base input의 0.1x (90% 할인)**
- **cache write = base input의 1.25x**
- **TTL = 5분** (ephemeral)
- 네이티브 클라이언트에서 **90~96% cache hit rate** 보고

**Breakeven 턴 수 = (1.25 - 1) / (1 - 0.1) ≈ 0.28** — 딱 1번만 hit해도 이득. 짧은 세션도 자동으로 캐시 혜택을 받는다. 반대로 **prefix가 변하면 캐시 전체 무효화** + 1.25x write → 활성 세션 중 CLAUDE.md 수정은 손해. **5분 TTL은 체감보다 짧다** — 점심 먹고 오면 재캐시.

이 수학이 모든 레버의 우선순위를 결정한다: **prefix를 작게 + 세팅한 prefix를 깨지 않기**가 첫 번째 목표.

## 15 레버 — 5 카테고리

### A. Prefix 다이어트

매 턴 반복되는 덩어리 축소. 캐시 덕분에 한 번만 내면 되지만, *작은 prefix*는 그 한 번도 싸다.

#### A1. CLAUDE.md ≤ 200줄 + skill/rules/@imports 분할

- 공식 권고: [memory docs](https://code.claude.com/docs/en/memory) — "target under 200 lines"
- 커뮤니티 CLI 런타임 경고는 *40k 문자* 기준 (issue #2766, 비공식).
- **분할 전략**: 워크플로별 상세 지시문 → `.claude/skills/` / 파일타입별 → `.claude/rules/*.md`의 `paths:` frontmatter 스코프 / 대형 참조 → `@docs/foo.md` import.
- ⚠️ **극단 다이어트는 역효과** — 핵심 가드레일까지 지우면 매 세션 재탐색 → 오히려 증가. *"삭제"가 아니라 "로딩 시점 분리"*.
- 실전: [Dev Setup Tips 로그](/wiki/harness-engineering/dev-setup-tips-log)의 `mino-moneyflow` — 73.9KB → 27.7KB (−62%).

#### A2. MCP Tool Search (deferred tools)

- **문제**: MCP 툴 정의(개당 400~800 토큰)가 매 턴 prefix 상시 로드. **50개 툴 = 20~40K 토큰** 점유.
- **해결**: 이름만 컨텍스트에 넣고 스키마는 호출 시 fetch (이 세션에서도 `ToolSearch`로 로드).
- **공식 수치**: **51K → 8.5K (46.9% 감소)** (issue #13717).
- **기본값**: 네이티브 자동 ON. `ANTHROPIC_BASE_URL`이 프록시면 꺼지므로 `ENABLE_TOOL_SEARCH=true` 또는 `auto:5` 명시.
- ⚠️ Haiku는 `tool_reference` 블록 미지원 — D2/D3 조합 시 주의.
- 출처: [mcp — scale with tool search](https://code.claude.com/docs/en/mcp#scale-with-mcp-tool-search)

#### A3. CLI > MCP 선호 + `/mcp`로 안 쓰는 서버 disable

- `gh`, `aws`, `gcloud`, `sentry-cli` 같은 CLI는 툴 디스크립션 **0 토큰**.
- 극단: GitHub MCP 하나에 34K → 80K 튐 (github-mcp-server#1286).
- 규칙: MCP는 *CLI로 대체 불가능할 때만* (Playwright 등). 나머진 CLI.
- 출처: [costs — reduce MCP server overhead](https://code.claude.com/docs/en/costs)

#### A4. `includeGitInstructions: false`

- `settings.json`에 `"includeGitInstructions": false` 또는 env `CLAUDE_CODE_DISABLE_GIT_INSTRUCTIONS=1`.
- 시스템 프롬프트 내장 git 지시문 제거. 자체 git skill 있을 때만.
- 절감 폭 수백 토큰 수준이지만 *매 턴 반복* 누적.
- 출처: [settings docs](https://code.claude.com/docs/en/settings)

#### A5. `.claudeignore`

- `.gitignore` 문법으로 `node_modules/`, `dist/`, `*.lock`, 벤더 디렉토리 차단. 탐색 시 *아예 안 읽음*.
- RTK가 output side를 덮는다면 이건 **input 탐색을 차단**하는 상보 레버. 베이스라인 중 베이스라인.

**프로젝트 타입별 패턴:**

*Next.js / Node:*
```
node_modules/
.next/
src/generated/
package-lock.json
*.lock
```

*iOS / Swift (SPM 멀티모듈):*
```
# ⚠️ 루트의 .build/만 넣으면 놓친다 — 각 모듈마다 독립 .build/ 존재
**/.build/
.build/

vendor/          # CocoaPods
*.xcodeproj/     # Xcode 프로젝트 XML (generated)
```

> **iOS SPM 함정**: 각 SPM 모듈마다 `.build/`가 독립적으로 존재. 모듈당 6~7GB, 20개 모듈이면 130GB+. `**/.build/` 없이 루트 `.build/`만 넣으면 99%를 놓친다.

### B. Cache warmth 유지

자동 캐시를 *깨지 않는* 것만으로 90% 할인.

#### B1. 5분 TTL 존중: 리듬 관리

- 5분 이상 쉴 거면 세션 닫고 새로 열기가 유리할 때도 있다 (write는 이미 만료).
- 연속 작업 중엔 *5분 안에 다음 턴*을 때리는 리듬이 최적.
- 점심/회의 전엔 세션 명시적 종료, 재개 시 새 세션.

#### B2. 활성 세션 중 CLAUDE.md·툴 정의 수정 금지

- Prefix가 변하면 캐시 *전부* 무효화 + 1.25x write 오버헤드.
- 고쳐야 하면: 수정 → `/clear` 또는 세션 종료 → 새 세션에서 재수립.
- MCP 서버 add/remove도 동일. 세션 시작 *전*에 정돈.

#### B3. ccusage로 cache read/write 비율 관측

```bash
npx ccusage@latest
```

- 일/월/세션/모델별 토큰 + **cache read를 분리 표시**.
- RTK `gain`(명령별 절감)과 역할 다름 — 둘 다 돌릴 가치.
- MCP 서버 모드도 있어서 Claude Code에서 직접 조회 가능.
- 출처: [ryoppippi/ccusage](https://github.com/ryoppippi/ccusage)

### C. Output pipeline 필터 — RTK의 빈틈 메우기

RTK가 bash를 덮으니, 남은 Read/MCP/대형 파일 쪽을 공략.

#### C1. PreToolUse 훅을 Read/MCP로 확장

- RTK 패턴의 *공식 문서화 버전*: [costs — offload to hooks](https://code.claude.com/docs/en/costs).
- PreToolUse 훅은 v2.0.10부터 **tool input 수정**까지 가능 → Bash 명령을 투명하게 `cmd | head -200`으로 재작성.
- **확장 여지**: Read 큰 파일 → 자동 청크/요약 / MCP 출력 → 후처리 필터.
- 출처: [hooks docs](https://code.claude.com/docs/en/hooks), [disler/claude-code-hooks-mastery](https://github.com/disler/claude-code-hooks-mastery)

#### C2. `MAX_MCP_OUTPUT_TOKENS=5000`

- 공식 env var. 기본값 25000, 10K 초과 시 경고만 뜨고 실제 절단은 이 값 기준.
- 5K로 낮추면 MCP verbose dump 자동 절단.
- 트레이드오프: 너무 낮으면 재호출 비용 증가. 5K가 스위트스팟.

#### C3. Read/Grep 내장 옵션 활용

Read/Grep 툴 자체가 token-aware 옵션을 갖고 있다. CLAUDE.md에 명시:

```markdown
## Token hygiene

- 큰 파일은 `Read`의 `offset`/`limit`으로 청크 단위로 읽는다.
- `Grep`은 일단 `output_mode: "files_with_matches"`(디폴트)로 범위 좁히고, 필요할 때만 `content` 모드로 전환.
- `head_limit`을 적극 사용. `glob`/`type` 사전 필터로 검색 범위를 좁힌다.
```

트레이드오프 없음. 공식 권장인데 *안 쓰는 사람*이 많음.

### D. 모델 라우팅 — 싼 모델에 싼 일 주기

Haiku는 Opus 대비 약 **15x 저렴**.

#### D1. `/model haiku` + Plan 모드 트릭 (Boris Cherny 공유)

- `/model haiku` → `Shift+Tab Shift+Tab` (Plan 모드 진입).
- Claude Code가 **planning은 Sonnet, execution은 Haiku**로 자동 라우팅.
- Anthropic 직원(Boris Cherny) Threads 공유 **undocumented 패턴**.
- Haiku 4.5는 **SWE-bench Verified 73.3%** (Sonnet 4 수준).
- ⚠️ 복잡한 리팩토링/아키텍처 결정에선 품질↓ — *routine edit/반복 작업*에만.
- 출처: [Boris Cherny on Threads](https://www.threads.com/@boris_cherny/post/DP4Jk2HkUUP/), [Haiku 4.5 announcement](https://www.anthropic.com/news/claude-haiku-4-5)

#### D2. 서브에이전트 frontmatter `model: haiku`

```markdown
---
name: log-scanner
description: 로그 파일에서 에러 패턴 추출
model: haiku
---
```

- `~/.claude/agents/*.md`에 `model: haiku` → 해당 서브에이전트 호출 전부 Haiku.
- 적합: 로그 파싱, 테스트 실행, 단순 grep, 파일 리스팅.
- 관련: `CLAUDE_CODE_SUBAGENT_MODEL=haiku`로 기본 모델 강제 가능.
- 출처: [sub-agents — choose a model](https://code.claude.com/docs/en/sub-agents#choose-a-model)

#### D3. Explore subagent로 탐색 격리

- 빌트인 Explore는 **Haiku + read-only + fresh context**로 실행.
- 탐색 결과 수만 토큰이 **메인 컨텍스트에 안 쌓임** → 요약만 반환.
- **수학**: subagent는 새 컨텍스트라 CLAUDE.md·tool 정의를 *재로드*. 3~4 파일 이상 탐색부터 이득. 짧은 탐색은 직접이 쌈.
- Plan 모드에선 자동 스폰 많음.

#### D4. 환경 변수 모델 고정

```bash
# ⚠️ 과거 이름 (deprecated)
# export ANTHROPIC_SMALL_FAST_MODEL=...

# ✅ 현재 이름
export ANTHROPIC_DEFAULT_HAIKU_MODEL=claude-haiku-4-5-20251001

# 서브에이전트 기본 모델
export CLAUDE_CODE_SUBAGENT_MODEL=haiku
```

- `ANTHROPIC_SMALL_FAST_MODEL`은 deprecated. 새 이름으로 교체 (issue #27665).
- `DISABLE_NON_ESSENTIAL_MODEL_CALLS=1`로 suggestion/tip 같은 백그라운드 호출 차단.

### E. Thinking / 컨텍스트 관리

#### E1. Extended thinking 예산 제어

```bash
export MAX_THINKING_TOKENS=8000
```

- 또는 `/effort low` (Sonnet 4.6 / Opus 4.6), `/config` 토글.
- thinking은 **output 토큰 과금**. 기본 수만 토큰.
- ⚠️ **어려운 문제에서 thinking 끄면 재시도 루프 → 결국 더 씀**. 아키텍처 결정엔 high 유지, 반복 작업에만 낮춰라.
- 출처: [costs — adjust extended thinking](https://code.claude.com/docs/en/costs)

#### E2. `/compact <지시문>` 타겟 압축

지시문을 반드시 준다:

```
/compact Focus on API changes only
/compact Keep only the decision log, discard exploration
```

- 요약 품질이 *훨씬* 좋아진다. 70K → 4K (94% 압축) 사례.
- `Esc Esc` (`/rewind`)로 특정 메시지 지점부터 요약 가능.
- ⚠️ compact 자체가 LLM 호출 → 너무 자주는 역효과.

#### E3. 무관한 작업 전환 = `/clear` (not `/compact`)

- Prefix는 캐시 덕분에 재로드가 *싸다*. 메시지 히스토리만 버리면 됨.
- `/compact`는 같은 작업 계속 이어갈 때. `/clear`는 완전히 다른 작업 전환 시.
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50`으로 auto-compact 일찍 발동 가능하지만 **공격적 설정은 품질 저하** — 권장 안 함.

## 안티패턴 5 — "절감인 척"하는 것들

### 1) Fast mode (`/fast`)로 절약

- Opus 4.6 Fast는 **$30 / $150 per MTok**로 일반 Opus보다 **비쌈**.
- *속도*를 사는 것이지 *비용*을 줄이는 게 아니다. 토큰 절약 목적이면 정반대.
- 출처: [fast-mode 문서](https://code.claude.com/docs/en/fast-mode)

### 2) Extended thinking 완전 차단

- 어려운 문제 → 틀린 답 → 재시도 → 결국 더 소비. Opus 라우팅 task에서 특히 역효과.
- 정답: 반복 작업에만 낮추고, 복잡 작업은 high 유지.

### 3) Caveman 모드 (원시인 말투)

- Reddit 바이럴 "원시인 말투" 스킬. 벤치에서 output 61% 감소 보고.
- 하지만 **output 토큰은 세션의 5% 미만** (892K in / 12K out 사례). Input이 진짜 문제.
- 코딩에선 설명 부족 → 디버깅 품질 하락 → 재시도 → 손해.

### 4) Custom subagent 남용

- "PythonTests" 같은 걸 만들면 main agent가 테스트 컨텍스트를 *잃어* → holistic reasoning 불가 → 재진입 반복.
- Shrivu Shankar (ex-Anthropic): **Task tool의 general-purpose 복제("master-clone")**가 대부분 경우 더 낫다.
- 출처: [Shrivu Shankar — How I Use Every Claude Code Feature](https://blog.sshh.io/p/how-i-use-every-claude-code-feature)

### 5) Agent Teams 실험 플래그

- `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` — plan 모드 대비 **~7배** 토큰.
- subagent(단일 세션 내 격리) ≠ agent team(병렬 세션). 이름 혼동 주의.
- 출처: [costs — manage agent team costs](https://code.claude.com/docs/en/costs)

## 계측 — 어디부터?

계측 없는 최적화는 *신호 없이 쏘는 것*. 위 15개 중 어느 것부터 할지는 **본인 세션의 어느 덩어리가 제일 큰지**에 달려 있다.

### 스냅샷 3종

```bash
# 1. 전체 토큰 breakdown
npx ccusage@latest               # 세션/일/월/모델별
rtk gain                          # RTK가 잡은 명령별 절감량
rtk gain --history                # 명령 히스토리

# 2. 세션 내 실시간
/cost                             # 현 세션 누적 비용
/context                          # prefix 점유량 (MCP/skill 분해)

# 3. 레버 적용 전후 비교
# → CLAUDE.md 다이어트 전후 첫 턴 input 토큰 체크
# → MCP Tool Search 켠 직후 /context 비교
```

### 해석 공식

- **`/context`의 prefix가 크다** → A 카테고리 (A1 CLAUDE.md, A2 MCP Tool Search)
- **`ccusage`에서 cache read 비율이 낮다** → B 카테고리 (세션 리듬 재검토)
- **RTK `gain`이 이미 큰데 세션 비용이 여전히 크다** → C·D·E 공략 (훅 확장, 모델 라우팅, thinking 예산)

## 놀라운 발견 5

- **Claude.ai 웹계정 스킬 무음 주입** (issue #39686): 로컬 CLI 세션에 `claude.ai`/Cowork 스킬이 몰래 주입되어 **세션당 ~6K 토큰 낭비**. opt-out 없음.
- **Opus Max 유저의 실제 모델 분포** (issue #27665): 17일 실사용 **Opus 93.8%, Haiku 5.2%**. "지능형 자동 라우팅"이 사실상 작동 안 함 → 수동 `/model` 전환이 유일.
- **Task tool "master-clone" 패턴이 custom subagent보다 낫다** — 가장 저평가된 기법 (Shrivu Shankar).
- **Cache write 1.25x → 1 hit breakeven**: 짧은 세션도 무조건 이득. "긴 세션에만 의미 있다"는 오해 뒤집기.
- **iOS SPM `.build/` 함정**: SPM 멀티모듈은 각 모듈 하위에 `.build/` 독립 존재. `.claudeignore`에 `.build/`만 넣으면 루트 레벨만 차단. **`**/.build/` 패턴 필수**. 실측: 모듈당 6~7GB, 합산 130GB+.

## 핵심 한 문장

> **토큰 절감은 "적게 쓰기"가 아니라 "prefix 줄이기 × cache warmth 유지 × 싼 모델 라우팅 × output 필터"의 *곱셈* 문제.** RTK는 마지막 곱에서 이미 ~70%를 가져간다. 나머지 셋을 각각 30~50% 더 긁으면 세션 비용은 원래의 1/3까지 내려갈 여지.

네 축을 *동시*에 손대야 곱해진다. 한 축만 극단으로 밀면 안티패턴(CLAUDE.md 극단 다이어트 → 재탐색 증가, thinking 차단 → 재시도, caveman → 디버깅 품질 저하).

## AI Agent Directive

**Trigger**: 사용자가 "Claude Code 비용을 줄이고 싶다" / "세션이 비싼 것 같은데" / 토큰 절감 레버에 대해 물어볼 때
**Prerequisites**: [AI API Cost Tracking Pattern](/wiki/harness-engineering/ai-api-cost-tracking-pattern) / [Token Levers Applied Log](/wiki/tokenomics/claude-code-token-levers-applied-log) — 실제 적용 사례 확인 필수

### Actionable Steps
1. **계측 우선**: ccusage / rtk gain / /context 로 "어느 덩어리가 제일 큰지" 파악 — 측정 없이 추천 금지
2. **A/B/C 레이어 특정**: Prefix (A) · cache (B) · output (C) 중 어디가 문제인지 분류
3. **우선순위 표 참조**: 직교하는 레버만 추천 (RTK 사용자라면 A·D·E 카테고리 중심)
4. **본인 환경 검증**: 책의 수치는 참고만 — ccusage before/after 로 본인 세션에서 효과 재측정 강제
5. **안티패턴 절대 금지**: /fast · caveman · agent teams · aggressive E1 차단 — 역효과 입증됨
6. **새 레버 발견 시**: 4 조건 게이트 검증 후 카탈로그에 추가, applied-log에 박제는 별도

### Anti-patterns
- **계측 스킵하고 추천** — 사용자 환경의 병목이 다르면 추천이 역효과
- **안티패턴 레버 제안** — /fast 는 "빠르다" 해서 비용까지 낮은 게 아님 · 명시적 금지
- **모든 레버를 동시 적용** — 어느 게 효과인지 모르고 · 롤백 불가능해짐

## For AI Agents

1. 이 카탈로그는 *레버 사전*, [applied-log](/wiki/tokenomics/claude-code-token-levers-applied-log)는 *적용 결정 로그*. 새 레버 발견 시 여기에 추가, 프로덕트 적용 여부는 applied-log의 4 조건 게이트 통과 후 거기에 기록.
2. 사용자가 "내 세션 비용 줄이고 싶다"고 하면 계측(§계측) → 가장 큰 덩어리 식별 → §우선순위 표의 직교 레버 순서대로 적용. 계측 전 레버 추천 금지.
3. 안티패턴(§안티패턴 5) 레버는 *절대 추천하지 않는다* — 특히 `/fast`, caveman, agent teams.
