# iOS Harness Journal 017 — 워커 풀 운영 튜닝

> **TL;DR** — moneyflow 4-pane 워커 풀 운영에서 같은 결의 작은 마찰 4건을 한 저널에 박제. (1) opus/sonnet *자리별 모델 분리*, (2) 고정 sleep 25s를 pane Claude ready 폴링으로 교체, (3) Orchestrator 탐색 차단 훅을 정적 → 동적 큐 분기(WO ≤ 3 차단, > 3 직접 허용)로 진화, (4) post-commit swift-reviewer 자동 dispatch를 사용자 확인으로 *되돌린* 자동화 후퇴 회고. 보너스로 조건부 UI 비활성화 회귀의 3종 세트 패턴도 박제.

## 0. 한 결로 묶인 메타 원칙

네 가지 작은 패치였지만 회고하니 한 원칙으로 묶였다:

> **하네스 비용은 자리·시점·시스템 상태에 따라 다르다. 한 가지 설정을 전역으로 박지 말고 *조건에 따라 분기*하게 만들어라.**

- 모델 선택은 자리(orchestrator/lead vs impl/verify)별로 분리
- 초기화 대기는 *시간*이 아니라 *준비 상태*로 분기
- 가드 강도는 큐 적체 상태로 분기
- 자동 dispatch는 분류기 신뢰도로 분기 (못 믿으면 사람 확인으로 후퇴)

전부 "분기"가 키워드다.

---

## 1. 모델 분리 — orchestrator/lead opus, impl/verify sonnet

### 기존 (Journal 010)
4 pane 모두 sonnet 단일.

```bash
CLAUDE_CMD="claude --dangerously-skip-permissions --model sonnet"
tmux send-keys -t "${SESSION}:1.${i}" "${CLAUDE_CMD}" C-m
```

문제: orchestrator pane이 스프린트 우선순위, 워커 큐 분배, 컨플릭트 중재를 하는데 sonnet으로는 길고 모호한 멀티-팩터 결정에서 가끔 빗나갔다. 반대로 worker-3(verify) pane은 Maestro yaml 실행 + 결과 보고가 주 업무라 opus는 과투자.

### 변경

```bash
CLAUDE_CMD_OPUS="claude --dangerously-skip-permissions --model opus"
CLAUDE_CMD_SONNET="claude --dangerously-skip-permissions --model sonnet"

# pane 1: orchestrator (CTO 판단)         → opus
# pane 2: worker-1 / lead (사양 분해)      → opus
# pane 3: worker-2 (구현)                  → sonnet
# pane 4: worker-3 (검증)                  → sonnet
tmux send-keys -t "${SESSION}:1.1" "${CLAUDE_CMD_OPUS}"   C-m
tmux send-keys -t "${SESSION}:1.2" "${CLAUDE_CMD_OPUS}"   C-m
tmux send-keys -t "${SESSION}:1.3" "${CLAUDE_CMD_SONNET}" C-m
tmux send-keys -t "${SESSION}:1.4" "${CLAUDE_CMD_SONNET}" C-m
```

### 자리별 부하 곡선 — 왜 분리해야 하나

| 자리 | 부하 곡선 | 한 번 실수 비용 | 빈도 |
|------|-----------|-----------------|------|
| Orchestrator (CTO) | 멀티-팩터 결정 (우선순위 X 워커 X 시간) | **매우 큼** (스프린트 방향 오류) | 낮음 |
| Worker-1 (Lead/Advisor) | 사양 분해 + 모호 해소 | 큼 (잘못된 WO 1개 = 워커-2 1시간 낭비) | 중간 |
| Worker-2 (Impl) | 명확한 WO를 코드로 | 작음 (개별 함수 fix) | 높음 |
| Worker-3 (Verify) | smoke 실행 + 보고 | 작음 (재실행 가능) | 높음 |

**부하 곡선 다른 자리에 같은 모델을 박으면 항상 한쪽이 overpay 또는 underspec.** 자리별 분리는 같은 월간 예산으로 두 곡선을 동시에 최적화한다.

### 측정

본격 정량은 다음 회고에서. 체감은 orchestrator pane이 우선순위 충돌 중재에서 "왜 그렇게 결정했는지"를 1~2단계 더 정리해서 내놓기 시작 — sonnet 시절에는 결정만 던졌다.

### 전이성

- "에이전트 풀의 모든 자리에 같은 모델" 가정은 거의 항상 *둘 중 한쪽이 비최적*.
- Manager/Worker 같은 *역할 비대칭이 명확한* 토폴로지는 모델도 비대칭이어야 한다.

## 2. ready 폴링 — 고정 sleep을 준비 신호로 교체

### 기존
tmux 4 pane을 띄운 직후 25초 고정 sleep 후 역할 브리핑 paste-buffer 송신.

```bash
PRIME_WAIT="${PRIME_WAIT:-25}"
sleep "${PRIME_WAIT}"
prime_pane 1 ...   # 25초가 *너무 짧으면* Claude 부팅 중 키 입력 유실
                   # 25초가 *너무 길면* 시작이 매번 25초 페널티
```

**왜 25초였나**: 짧으면 부팅 중 키 유실로 브리핑이 *조용히* 사라지고 (Journal 007의 silent failure 패턴), 길면 매 세션 시작이 25초 손해. 마법 숫자는 *상한 추정*이라서 항상 과투자다.

### 변경 — pane별 `pane_current_command` 폴링

```bash
wait_pane_ready() {
  local pane="$1"
  local timeout="${PANE_READY_TIMEOUT:-45}"
  local waited=0
  while [[ "${waited}" -lt "${timeout}" ]]; do
    local cmd
    cmd=$(tmux display-message -t "${SESSION}:1.${pane}" -p '#{pane_current_command}' 2>/dev/null || echo "")
    case "${cmd}" in
      bash|zsh|-zsh|-bash|sh|login|"") ;;   # 아직 shell — 계속 대기
      *)
        sleep 1.5   # TUI 페인트 grace
        return 0 ;;
    esac
    sleep 1
    waited=$((waited + 1))
  done
  return 0  # 타임아웃이어도 시도는 함
}
```

- `pane_current_command`가 `bash`/`zsh`면 Claude Code 미부팅 → 1초 후 재확인
- shell 아닌 다른 명령(`node` 또는 Claude TUI runtime)으로 바뀌면 부팅 완료 → 1.5초 페인트 grace 후 송신
- 상한 45초 (Claude 업데이트 자동 적용 케이스 대비), 보통 8~12초에 ready

### 결과

- 4 pane 평균 부팅 ≈ 10초 → **세션 시작 페널티 25s → ~12s**
- 키 유실로 브리핑이 사라지는 사고 = 0 (이전엔 5% 정도 — 첫 키가 부팅 중 화면에 *삼켜졌다*)

### 전이성

- "고정 sleep"은 거의 항상 *준비 상태 신호*로 치환 가능. 신호의 후보:
  - 프로세스/명령 이름 (이 케이스)
  - HTTP `/healthz` (웹 서버)
  - 파일 존재 (lock/socket/pid)
  - 로그 grep
- 신호 없이 적당히 기다리는 코드는 자동화 흔적이지 자동화의 완성이 아니다.

## 3. Orchestrator 탐색 차단 훅 — 정적 → 동적 큐 분기 (v3)

### 배경 — v1/v2 (Journal 014에 박제)
Orchestrator pane(세션 `:1.1`)에서 `*.swift` 탐색을 PreToolUse 훅으로 정적 차단. 의도는 "탐색은 Worker-1에 위임, orchestrator는 결정만 한다".

```bash
# v2: 무조건 차단
echo "🚫 Orchestrator 탐색 차단 — Worker-1에 위임" >&2
exit 1
```

### 문제
Worker-1 큐가 이미 과부하(`work-orders/in-progress/` 4건 이상)인데도 정적 차단 → 위임이 적체 → Worker-1이 큐 다 비울 때까지 orchestrator는 아무것도 못 함. **가드가 보호하려던 자원(Worker-1 큐)이 이미 과부하인 상황에서, 가드가 시스템을 더 망친다.**

### v3 — 동적 큐 기반 분기

```bash
# .claude/hooks/pretooluse-bash-code-search-guard.sh
WO_COUNT=$(ls "$REPO_ROOT/work-orders/in-progress/" 2>/dev/null \
  | grep -v '\.gitkeep' | wc -l | tr -d ' ')

if [ "$WO_COUNT" -le 3 ]; then
    # Worker-1 큐 여유 → 위임 강제
    echo "🚫 Orchestrator 탐색 차단 — Worker-1에 위임. 현재 WO: ${WO_COUNT}개" >&2
    exit 1
fi
# WO > 3: 과부하 → 직접 허용 (exit 0)
exit 0
```

| WO 수 | 판단 | 가드 동작 |
|-------|------|-----------|
| ≤ 3 | Worker-1 여유 | **차단** → 위임 강제 |
| > 3 | Worker-1 과부하 | **허용** → orchestrator 직접 처리 |

임계값 3은 운영 측정으로 정한 Worker-1 동시 처리 comfortable 상한.

### 메타 원칙

> **가드의 강도는 보호 대상의 *상태*에 따라 분기해야 한다.**

가드가 *항상* 켜져 있는 게 정의가 아니다. 보호 대상(워커 큐, DB connection pool, rate limit budget)이 *지금 어떤지*를 가드가 직접 관찰하고 강도를 조정해야 한다. 정적 가드는 정상 상태 가정에서만 옳다.

### 전이성

- Circuit breaker, rate limiter, dispatch guard — 전부 같은 결.
- "차단 vs 허용"이 가드의 두 상태인 줄 알면 함정. 실제로는 *관측 → 조건부 분기*가 가드의 본질이다.

## 4. post-commit 자동 dispatch — *되돌린* 자동화 후퇴 회고

### 기존 (자동화)
post-commit 훅이 `*.swift` 커밋을 감지하면 *자동으로* Worker-3(swift-reviewer) pane에 dispatch.

```bash
# .githooks/post-commit (이전)
if git diff --name-only HEAD~1 HEAD | grep -q '\.swift$'; then
  bash .claude/scripts/notify-main.sh "REVIEW_REQUEST" "..."
fi
```

### 문제
- **브랜치명 필터로 dispatch 대상 구분 시도** → `feature/` prefix 강제했으나 `chore/`, hotfix 브랜치, prefix 없는 임시 브랜치에서 무용
- **타 브랜치 커밋이 리뷰 큐로 쏟아짐** — feature/MOB-XXXX에서 작업 중인데 사이드 hotfix 커밋 1개에 리뷰 큐 자리 뺏김
- **무단 백그라운드 실행** — 사용자 모르게 워커-3 pane에 작업이 들어가서 *다른 작업 중이던 워커-3 컨텍스트 오염*

### 후퇴 — orchestrator를 거치는 사용자 확인 단계 추가

```bash
# .githooks/post-commit (변경)
# 자동 dispatch 제거. orchestrator에 REVIEW_REQUEST만 통지.
bash .claude/scripts/notify-main.sh "REVIEW_REQUEST_PROPOSED" \
  "commit=$(git rev-parse --short HEAD), files=$N"
# orchestrator pane이 사용자에게 확인 → Worker-3로 디스패치
```

흐름:
```
[git commit *.swift]
  ↓
[post-commit] → orchestrator에 PROPOSED 통지
  ↓
[orchestrator pane] → 사용자에게 "이 커밋 리뷰 돌릴까?"
  ↓ (사용자 OK)
[orchestrator] → Worker-3 dispatch
```

### 이게 후퇴인가, 전진인가?

> **분류기가 신뢰 불가하면 자동화는 무자동화보다 나쁘다.**

- 잘못된 자동화 = 소음 증폭기 (타 브랜치 큐 폭주 + 컨텍스트 오염)
- 무자동화 = 사람 1회 확인 (낮은 처리량, 0 false positive)
- *현재 분류기 신뢰도*에서는 무자동화가 우수
- 더 신뢰할 분류기(예: 변경 파일 .swift 비율 + WO 메타데이터 + branch policy)를 만들면 다시 자동화로 전진하면 됨

**자동화는 ratchet이 아니다.** 분류기 신뢰도 하락이 발견되면 즉시 한 칸 되돌리는 게 정상 운영이다. *되돌린 결정* 자체를 회고에 박제해야 다음에 자동화 재시도할 때 같은 함정에 안 빠진다.

### 전이성

- "AI가 한 행동을 자동 승인/실행" 게이트 전반 — 분류기 신뢰도 측정 없이 자동화 켜지 말 것
- 모든 자동화는 *되돌릴 방법*과 *되돌리는 트리거*를 명시한 채로 켜야 함

## 5. 보너스 — 조건부 UI 비활성화 회귀의 3종 세트

본 스프린트에서 잡은 UI 회귀 1건이 패턴화될 가치가 있어서 같이 박제.

### 증상
조건 X가 참일 때 버튼을 비활성화해야 하는 사양. 안내 박스는 정확히 표시됐지만 버튼이 *시각적으로 활성처럼* 보임. 탭하면 내부적으로는 차단되어 "아무 반응 없음" — 가장 나쁜 UX(사용자는 화면이 죽었다고 인식).

### 근본 원인
이벤트 핸들러에서 `isHidden`만 토글하고 `isEnabled`를 누락. 추가로 `isEnabled = false`만 박아도 UIButton은 `.normal` state 색상을 유지하기 때문에 *시각 피드백이 없다*.

### 3종 세트 — 조건부 상태 변경 시 항상 확인

| 항목 | 처리 | 누락 시 증상 |
|------|------|--------------|
| `isHidden` / `alpha` | 요소 표시/숨김 | 비조건일 때도 박스가 표시됨 |
| `isEnabled` | 인터랙션 차단 | 탭이 먹힘 |
| 색상 (`setTitleColor(_:for:)` / `.disabled` state) | 시각 피드백 | 비활성인데 활성처럼 보임 |

UIButton의 `isEnabled = false`는 *기본 색상을 자동으로 바꾸지 않는다* — `.disabled` state 색상을 별도로 지정하거나 `setTitleColor` 호출이 필요. 이게 가장 자주 빠지는 함정.

### 패턴 박제

```swift
// 단일 메서드로 묶기 — 3종이 흩어지면 한 군데가 빠진다
func applyDisabledState(_ disabled: Bool) {
    infoBoxView.isHidden = !disabled
    actionButton.isEnabled = !disabled
    actionButton.setTitleColor(disabled ? .gray060 : .blue, for: .normal)
}
```

### 재발 방지

- **코드 리뷰 체크포인트**: `isHidden` 변경 코드 근처에 `isEnabled` + 색상이 같이 있는지 확인
- **유닛 테스트 템플릿**: 상태 조건당 3가지(`isHidden`, `isEnabled`, titleColor) 모두 assert
- **Maestro yaml**: 시각 검증만으로는 `enabled=false` 못 잡음. accessibility 속성 assert 추가 (`snapshot_ui` + `enabled: false`)
