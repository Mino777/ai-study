## 배경 — /ios-ui-test가 사람보다 느려지다

RIBs 기반 사내 iOS 앱에 `/ios-ui-test` 하네스를 붙인 이후 세션을 거듭할수록 사용 빈도가 떨어지고 있었다. 이유는 하나였다: **화면 1개 검증에 30초+**.

체감 측정(편도 핫딜 시간 선택 화면 기준):

```
시뮬레이터 부팅 + 앱 launch + DebugMode 통과 + deeplink 발사 + snapshot_ui 1회 = ~22초
```

이 22초도 이미 느린데, 위에 snapshot_ui 반복 호출, navigation 더듬기, 로그 캡처까지 더하면 30초가 넘는다. **사람이 직접 시뮬레이터 보면서 확인하는 게 더 빠른** 상황이었다.

### 5개 병목 진단

단일 원인이 아니었다. 외부 의견서(5병목 분석)를 같이 읽으면서 원인이 누적임을 확인:

| # | 병목 | 영향 |
|---|------|------|
| 1 | smoke와 forensic 한 모드로 통합 | 회귀 확인용도 매번 트리/로그/시나리오 전부 |
| 2 | deeplink 미등록 화면 多 | navigation 5-7단계로 더듬어 진입 |
| 3 | fixture/인증 상태 사전 복원 부재 | 매번 fresh install → 수동 로그인 반복 |
| 4 | snapshot_ui 단계마다 호출 | 각 5KB JSON, 토큰 폭발 |
| 5 | 고정 sleep N초가 곳곳에 | 누적 지연 + 이미 끝난 화면도 기다림 |

**병목 #1이 전제 조건**이었다. smoke/forensic 분리부터 시작하지 않으면 나머지 최적화 효과가 증발한다.

## 해결 1: 3-tier 등급 분리 (의견서 #5 완결)

### 설계

처음엔 smoke/forensic 2-mode로 시작했다가, 당일 추가 스프린트에서 **Level 2 interaction** 등급이 추가돼 3-tier로 완결됐다.

```
/ios-ui-test                               # Level 1 smoke (기본, 5-8s)
/ios-ui-test --interaction "<시나리오>"    # Level 2 interaction (10-20s)
/ios-ui-test --forensic                    # Level 3 forensic (30s+)
```

| Level | 별칭 | 무엇을 | 목표 시간 | 호출 |
|-------|------|--------|-----------|------|
| **1** | smoke | 도달 + 핵심 라벨 1-2개 매칭. 스크린샷/로그 X | **5-8s** | 기본값 |
| **2** | interaction | smoke + 사용자 명시 인터랙션 1-2개 + 결과 라벨 매칭. 실패 시 스크린샷 1장 | **10-20s** | `--interaction "<시나리오>"` |
| **3** | forensic | 트리 분석 + 다중 스크린샷 + 로그 캡처 + 시나리오 N개 + 리포트 | **30s+** | `--forensic` |

### 상황별 추천

```
회귀 확인 (코드 수정 후 "깨졌나 봐")       → Level 1
"버튼 잘 눌리나?" 인터랙션 검증            → Level 2
빈 화면 / 렌더 버그 / 디자인 정합성        → Level 3
```

Level 1 흐름:

```
S0-A: fixture restore (인증 게이트 뒤 화면만)
S0-B: build_run_sim (코드 수정 있으면 incremental 빌드)
S1:   deeplink 발사 → 화면 직진
S2:   wait-for-ui.sh match "라벨1,라벨2" → 도착 확인 후 종료
```

## 해결 2: 고정 sleep → condition wait (22s → 7s)

### 실측이 밝혀준 진짜 병목

"sleep만 줄이면 빨라질 것"이라 추측했는데 실제 측정은 달랐다:

```
Before: sleep 총합 줄임 → 22s → 21s  (1초 절감, 의미 없음)
After:  condition wait + fresh-launch 플래그 → 22s → 7s  (3배 절감)
```

진짜 병목은 sleep 자체가 아니라 **"이미 일어난 일을 계속 폴링하는 비용"** 이었다.

### 변경 내역

```bash
# Before: 고정 sleep
sleep 3   # 시뮬 부팅 대기
sleep 4   # 앱 launch 대기
sleep 3   # DebugMode 통과 대기

# After: condition wait
simctl bootstatus $SIM_ID -b           # 부팅 완료까지 block
wait_for_app_ready 12                  # 트리에 children > 1 보일 때까지 폴링 (cap 6s)
wait_for_label_gone "Continue" 10      # DebugMode 버튼 사라질 때까지 (cap 5s)
wait_for_label_gone "열기" 6           # 열기 다이얼로그 닫힐 때까지 (cap 3s)
```

### APP_FRESHLY_LAUNCHED 플래그 패턴

핵심 아이디어: **한 번만 나타나는 것을 매번 폴링하지 않는다**.

```bash
if [[ "$APP_FRESHLY_LAUNCHED" == "true" ]]; then
  MAX_RETRY=10   # DebugMode/열기 다이얼로그 기다림
else
  MAX_RETRY=2    # 이미 ready 상태, 1-2회만 확인
fi
```

DebugMode 버튼과 "열기" 다이얼로그는 **fresh launch일 때만 나타난다**. 이미 앱이 떠있는 재호출에서 MAX_RETRY=10을 유지하면 idb 호출 8-13초가 낭비된다. 플래그로 분기 → 재호출 시 ~10s로 단축.

### 최종 측정값

| 시나리오 | Before | After |
|----------|--------|-------|
| `launch-test-screen.sh` (앱 떠있음) | ~22s | ~7s |
| smoke 전체 (첫 호출, fixture 포함) | ~30초+ | ~60s (fixture 45s 포함) |
| smoke 전체 (재호출, fixture 유지) | ~30초+ | ~10s |
| Xcode-attached smoke | ~30초+ | ~6-7s |

## 해결 3: seed-fixture 자동 복원 (S0-A)

### 왜 필요한가

로그인 + 면허인증 + 기기인증 상태여야 진입 가능한 화면이 대부분이다. 미인증 상태에서 deeplink 발사 → 로그인 화면으로 fallback → 수동 로그인 1-2분 낭비.

**시뮬레이터 data 디렉토리(키체인 + UserDefaults + .app)를 tar.gz로 박제**해두면 즉시 복원된다.

### 복원 흐름

```bash
bash .claude/scripts/seed-fixture.sh restore logged_in_verified
# ~45초 (2.4GB tar.gz 풀기), 세션 첫 1회만 필요
```

### 주의: 순서가 중요하다

```
restore → build_run_sim  ✅  (키체인/UserDefaults 보존 + 최신 .app 덮어쓰기)
build_run_sim → restore  ❌  (restore가 build 결과를 박제된 .app으로 롤백)
```

`build_run_sim`은 `.app`만 교체하므로 fixture의 키체인/UserDefaults는 건드리지 않는다. 그래서 restore 먼저, build 나중이 정답이다.

### 권장 시드 목록

| 이름 | 인증 상태 | 주요 용도 |
|------|----------|----------|
| `logged_in_verified` | 로그인 + 면허 + 기기 | 인증 게이트 뒤 화면 전반 |
| `oh-with-permissions` | logged_in_verified + 위치/알림 권한 허용 | 편도 핫딜 등 위치 필요 화면 |

### fixture 파일 관리 원칙

- **git 비공유**: 2.4GB tar.gz를 레포에 넣으면 clone 불가. 팀 내 공유 스토리지(내부 서버/오브젝트 스토리지)에 올리거나 개인이 각자 생성.
- **토큰 만료 시 갱신**: fixture 안의 인증 토큰이 만료되면 수동으로 재로그인 후 `seed-fixture.sh save logged_in_verified` 재실행.

## 해결 4: deeplink lazy 등록 정책

### 정책

> 자주 쓰는 화면을 사전에 일괄 등록하지 않는다. `/ios-ui-test` 호출 시점에 미등록이면 **그 자리에서 lazy 등록 후 진행**.

사전 일괄 등록은 over-engineering이다: 어떤 화면을 자주 쓸지는 해봐야 안다. 반면 각 화면의 등록 작업은 30분 내 완료 가능하므로 lazy 방식이 전체 ROI가 높다.

### RIB 깊이별 패턴

RIBs 구조에서 deeplink로 진입하려면 화면 깊이에 따라 접근법이 달라진다.

#### 1단 패턴 — 탭/메뉴 직진

메뉴를 통해 바로 도달 가능한 화면. Workflow 파일 하나면 끝난다 (~30줄).

```swift
public class FooWorkflow: Workflow<RootActionableItem> {
  public init?(url: URL) {
    super.init()
    guard url.path.hasPrefix("/foo") else { return nil }

    onStep { rootItem in
      rootItem.waitBridge()
    }
    .onStep { (bridgeItem, _) in
      bridgeItem.moveBridgeMenu(menu: .foo)
    }
    .commit()
  }
}
```

#### 2-3단 패턴 — 중첩 RIB (ActionableItem 정공법)

Bridge 아래 중간 RIB을 거쳐야 하는 화면. `*ActionableItem` 프로토콜을 중간 RIB에 채택해야 한다.

```
작업량:
1. Bridge/Sources/Bridge/Workflow/<Group>/<Name>.swift 신규
2. AppRootInteractor+DeepLink.swift에 1줄 등록
3. 중간 RIB Interactor에 *ActionableItem 채택
4. wait*ActionableItem() 메서드 구현 (Observable 반환)
```

```swift
// 중간 RIB Interactor
extension FooInteractor: FooActionableItem {
  public func waitFooScreen() -> Observable<(FooActionableItem, ())> {
    // 화면 준비 완료 신호 emit
    return .just((self, ()))
  }
}
```

#### 깊이별 선택 기준

| RIB 깊이 | 참고 패턴 | 작업량 | 비고 |
|----------|----------|--------|------|
| 1단 | FreeOneWay 스타일 | Workflow 1개 | 가장 단순 |
| 2-3단 | MyInfoInvite 스타일 | Workflow + ActionableItem | 중간 RIB 수정 필요 |
| 4단+ | 권장 X | 과도한 coupling | navigation fallback 검토 |

## 해결 5: Xcode-attached 모드 (LLDB 공존)

### 발견 경로

사용자의 자연스러운 질문에서 함정을 발견했다:

> "어차피 Xcode로 런 해놓으면 그거 쓰는 거지?"

이 시나리오에서 기본 흐름은 즉시 깨진다:
- `fixture restore` → 시뮬레이터 data 교체 → 앱 프로세스 종료
- `build_run_sim` → 앱 재실행 → Xcode LLDB attach 끊김 → 콘솔 로그 stream 종료

### 해결

트리거 키워드 감지 → S0-A/S0-B 전부 스킵, S1(deeplink)만 발사:

```
트리거: "방금 Run 했어", "런 안해도 돼", "Xcode 띄워놨어", --xcode-attached 플래그

동작:
- build_run_sim 스킵 (코드 수정 있으면 "Xcode에서 직접 Cmd+R 하세요" 안내)
- fixture restore 스킵
- simctl openurl <deeplink> 만 발사
  → Xcode LLDB attach된 그 프로세스가 deeplink 수신
  → Xcode 콘솔 로그 stream 유지
```

| 기존 흐름 | Xcode-attached 모드 |
|---------|------------------|
| fixture restore → build → deeplink | deeplink만 발사 |
| LLDB attach 끊김 | LLDB attach 유지 |
| 콘솔 stream 끊김 | 콘솔 stream 유지 |
| ~60s | ~6-7s |

한계 명시: **코드 수정사항은 이 모드에서 반영 안 됨.** "rebuild 포함해줘"라고 명시하면 build_run_sim 모드로 자동 전환.

## 당일 추가 완료 (오후 스프린트)

Phase 1 완결 후 같은 날 오후에 "다음 스프린트 큐" 항목 4개를 즉시 처리했다.

### wait-for-ui.sh 헬퍼 신설 (smoke S2 헬퍼 추출)

`snapshot_ui` 대신 **라벨 매칭 전용 헬퍼**를 분리했다. snapshot_ui는 JSON 트리 전체를 덤프(~5KB 토큰)하지만, 헬퍼는 idb describe-all 결과에서 라벨만 grep한다.

```bash
bash .claude/scripts/wait-for-ui.sh match "설정,개인 계정,혜택 알림"
# match: 설정 ← 통과, 0.5s × 최대 6회 폴링
```

- **smoke 기본 경로**: `wait-for-ui.sh match` 사용 (snapshot_ui 호출 X)
- **디버깅 필요 시**: snapshot_ui 1회 → grep (헬퍼 사용 불가한 예외 케이스만)

### fixture zstd 압축 최적화

seed-fixture.sh에 **zstd 우선** 압축 추가. tar.gz 대비 3-4배 빠름.

```bash
# 이전: tar.gz (2.4GB, ~50s restore)
# 이후: tar.zst + Library/Caches 제외 → 800MB 목표, ~15s restore
bash .claude/scripts/seed-fixture.sh save logged_in_verified   # 자동으로 zstd 사용
bash .claude/scripts/seed-fixture.sh restore logged_in_verified # .zst 우선, .gz fallback
```

기존 `.tar.gz` fixture와 호환 유지 (restore는 양쪽 지원).

### DebugMode 통과 마커

`APP_FRESHLY_LAUNCHED` 플래그보다 한 단계 더 나아간 최적화.

```bash
PASSED_MARKER="/tmp/lts-debugmode-passed-${UDID}"
# 마커 있으면 DebugMode 폴링 자체를 스킵 (1-2초 추가 절약)
# fresh launch → rm -f 마커 → 다시 폴링
# 통과 후 → touch 마커
```

**APP_FRESHLY_LAUNCHED 플래그 vs 마커 차이:**
- 플래그: 폴링 횟수를 10 → 2회로 줄임 (여전히 idb 호출 2회)
- 마커: 폴링 자체를 0회로 스킵 (시뮬 재부팅 전까지 영구 유지)

### screenshot-diff.sh — 픽셀 diff opt-in

디자인 정합성 회귀 감지용. **opt-in** (`--diff <name>` 옵션)이라 기본 smoke 5-8s에 영향 없음.

```bash
# 호출
/ios-ui-test setting --diff setting_v1

# 흐름
# 1. baseline 없음 → save + "다음 호출부터 비교" 안내. 추가 시간 0s
# 2. baseline 있음 → 현재 vs baseline 픽셀 diff (+1s)
```

| 차이 | 판정 |
|------|------|
| < 2% | ✅ OK (정상 노이즈: status bar 시계, AA, fetch 타이밍) |
| 2-5% | 🟡 주의 (의도된 변경 가능, diff.png 확인) |
| 5%+ | 🔴 회귀 의심 (diff.png + bbox 출력, baseline 갱신 권장) |

> 임계값 2%는 dogfood로 실측 조정. 같은 화면 0.5초 간격 두 번 찍기에서 1.84% 차이가 측정됐다. 1% 임계값은 정상 화면도 경고를 내서 2%로 완화.

baseline 위치: `.claude/screenshots/<name>.png` (gitignore — fixture와 같은 로컬 전용 정책).

```bash
# 9항목 검증: Homebrew / SwiftLint / SwiftFormat / idb-companion / zstd

```
