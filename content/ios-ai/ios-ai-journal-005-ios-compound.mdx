## 한 줄 요약

iOS 배포 후 생기는 지식을 그냥 흘려보내지 않는다. `/ios-compound`가 5단계로 회고를 돌리고, 모든 해결책을 `docs/solutions/`에 박제한다. 다음 스프린트가 이걸 입력으로 받는다.

---

## iOS 특화 Compound가 필요한 이유

웹 배포와 iOS 배포의 사이클은 다르다:

| | 웹 | iOS |
|--|---|-----|
| 배포 속도 | push → 즉시 | 심사 → 수일 |
| 피드백 속도 | 실시간 | 심사 완료 후 |
| 롤백 | 즉시 가능 | 불가 (심사 재제출) |
| 지식 생산 시점 | 배포 직후 | 심사 중 / 배포 후 |

배포와 피드백 사이에 시간이 있다. 이 시간을 **지식 축적**으로 쓰는 것이 iOS Compound다.

---

## iOS Compound 사이클

```
코드 작성 → PR → /ios-ship (앱스토어 심사 제출)
                      ↓
              심사 중 (1~3일)
                      ↓
              /ios-compound (회고 + 지식 축적)
                      ↓
              사용자 피드백 / 크래시 리포트
                      ↓
         /ios-triage → /ios-investigate → 수정
                      ↓
              docs/solutions/ 업데이트
                      ↓
              다음 스프린트 NEXT.md 입력 ←────────┘
```

## /ios-compound 5단계

```markdown
# .claude/commands/ios-compound.md

## iOS Compound 회고

### Phase 1: 이번 스프린트 결과 정리
- 해결한 버그 목록
- 추가한 기능 목록
- 도입한 패턴/도구

### Phase 2: docs/solutions/ 기록
각 해결책을 카테고리별로 저장:

**포맷**:
- 문제 상황 (언제, 어디서, 어떤 증상)
- 원인 (트리아지 결과)
- 해결책 (코드 포함)
- 재발 방지 방법
- 관련 RIB / 모듈

### Phase 3: deferred 업데이트
미해결 이슈 → docs/solutions/ios/deferred/ 에 기록
(시도한 것, 실패 이유, 다음 시도 방향 포함)

### Phase 4: NEXT.md 업데이트
다음 스프린트 후보 3개 작성
우선순위: 이번 회고 발견 사항 > 기존 백로그

### Phase 5: CHANGELOG 업데이트
앱스토어 출시 노트 초안 작성
사용자 관점으로 기술 (기술 용어 최소화)
```

## docs/solutions/ 카테고리 구조

```
docs/solutions/
├── swiftui/         ← SwiftUI 관련 패턴 / 버그
├── networking/      ← API 통신 / Alamofire / async-await
├── storage/         ← UserDefaults / Keychain / CoreData
│                       (Persistent 모듈 관련 포함)
├── xctest/          ← 유닛 테스트 / UI 테스트 패턴
├── performance/     ← 성능 최적화 / 메모리 리크
├── build/           ← 빌드 에러 / 의존성 문제 / SPM
├── ios-compat/      ← iOS 버전별 호환성 이슈
└── appstore/        ← 심사 관련 이슈 / 거절 사유
```

## 솔루션 기록 예시

```markdown
# docs/solutions/ios/storage/aes256-userdefaults-migration.md

## 문제
Persistent 모듈의 AES256 암호화 UserDefaults에서 키 마이그레이션 시
기존 암호화된 데이터가 읽히지 않는 문제

## 원인
새 키로 암호화된 데이터를 쓰기 전에 기존 키로 읽기를 시도하지 않음.
마이그레이션 순서: 읽기(구 키) → 쓰기(신 키) → 삭제(구 키)

## 해결책
```swift
func migrateEncryptedValue<T: Codable>(key: String, oldKey: String) {
    guard let oldValue: T = Persistent.read(key: oldKey) else { return }
    Persistent.write(oldValue, key: key)
    Persistent.delete(key: oldKey)
}
```

## 재발 방지
마이그레이션이 필요한 경우 항상 이 순서: 읽기 → 쓰기 → 삭제
CLAUDE.md에 마이그레이션 패턴 추가
```

## 앱스토어 심사 거절도 자산

```markdown
# docs/solutions/ios/appstore/guideline-4-3-spam-rejection.md

## 심사 거절: Guideline 4.3 (Spam)

**상황**: 신규 기능 버전 심사 거절
**이유**: 기존 앱과 기능이 유사하다는 판정
**해결**: 차별화 기능 스크린샷 추가 + 심사 메모 작성

**다음 버전 적용**:
처음부터 스크린샷에 차별화 포인트를 넣는다.
심사 메모에 "기존 버전 대비 추가된 기능" 명시.
```

심사 거절 이유가 쌓이면, 다음 버전에서 같은 이유로 거절당하는 일이 없어진다.
