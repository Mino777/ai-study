## 한 줄 요약

iOS 배포 후 생기는 지식을 흘려보내지 않는다. **`/compound`** (`/ios-compound` 아님 — 실제 파일명은 `.claude/commands/compound.md`)가 **3 Agent를 병렬로** 돌려서 CHANGELOG 업데이트 / 솔루션 문서 생성 / 회고 작성을 동시에 수행하고, CLAUDE.md + Claude 메모리까지 동기화한 뒤 단일 커밋으로 박제한다. 다음 스프린트는 이 자산을 입력으로 받는다.

---

## 왜 iOS에 compound가 필요한가

웹 배포와 iOS 배포의 사이클은 다르다.

| | 웹 | iOS |
|--|---|-----|
| 배포 속도 | push → 즉시 | 심사 → 수일 |
| 피드백 속도 | 실시간 | 심사 완료 후 |
| 롤백 | 즉시 가능 | 불가 (심사 재제출) |
| 지식 생산 시점 | 배포 직후 | 심사 중 / 배포 후 |

배포와 피드백 사이에 물리적 시간이 있다. 이 시간을 **지식 축적**으로 쓰는 것이 `/compound`의 역할이다.

### Phase 1 — 변경사항 수집

```bash

### Phase 2 — 3 Agent 병렬 실행

> **동시 실행이 핵심.** 순차가 아니라 **3개를 동시에** 띄운다. CHANGELOG 쓰는 동안 솔루션도 쓰고, 회고도 같이 쓴다. 세 종류의 산출물은 서로 의존하지 않기 때문에 병렬로 묶여도 안전하다.

**Agent A — `CHANGELOG.md` 업데이트**

- `vX.X.X (YYYY-MM-DD)` 섹션 추가
- 섹션: **Added / Fixed / Changed / Performance**(있을 때만) / **Dependencies**(변경 시만)

**Agent B — `docs/solutions/[카테고리]/YYYY-MM-DD-[slug].md` 생성**

- 판단 기준: **"다음에 또 만날 수 있는가?"** → YES면 문서화, NO면 스킵
- 카테고리는 8개 (아래 섹션)
- 솔루션 포맷:
  ```markdown
  # [문제 제목]

  ## 문제
  (언제, 어디서, 어떤 증상)

  ## 근본 원인
  (트리아지 결과)

  ## 해결책
  (실제 수정 코드 포함)

  ## 예방
  (재발 방지 / 컨벤션)

  ## 관련
  - iOS 최소 버전: …
  - 영향 컴포넌트: …
  ```

**Agent C — `docs/retros/YYYY-MM-DD.md` 생성**

- 배포된 것
- 잘된 것
- 아쉬운 것
- 다음에 적용할 것
- Compound Assets (이번 사이클에서 생긴 재사용 자산)

## docs/solutions/ 카테고리 8개

실제 `compound.md`에 박제된 분류 그대로다.

```
swiftui/        ← SwiftUI 레이아웃, 상태 관리, 애니메이션
networking/     ← URLSession, Alamofire, WebSocket, 인증
storage/        ← CoreData, Keychain, UserDefaults, FileManager
xctest/         ← XCUITest, Unit Test, Mock 패턴
performance/    ← 메모리 누수, FPS, 배터리 최적화
build/          ← Xcode 설정, SPM, CocoaPods, 서명
ios-compat/     ← iOS 버전별 deprecation, API 차이
architecture/   ← MVVM 패턴, 레이어 분리, 의존성
```

새로운 문제가 이 8개 중 어디에도 맞지 않으면, 카테고리를 추가하는 게 아니라 "진짜 재사용 가능한가?"를 다시 묻는다. 억지로 끼워 넣은 솔루션은 다음 세션에서 절대 찾지 못한다.

## 솔루션 문서 예시

"다음에 또 만날 수 있는가?"가 YES인 전형적인 패턴 하나.

```markdown
# tableView 갱신이 간헐적으로 멈추는 문제

## 문제
네트워크 응답 후 tableView가 새 데이터를 그리지 않음.
실기기에서 재현. 시뮬레이터에서는 재현 안 됨.

## 근본 원인
`reloadData()` 호출이 background queue에서 이뤄짐.
UIKit은 모든 UI 업데이트를 main queue에서만 받음.

## 해결책
```swift
DispatchQueue.main.async { [weak self] in
    self?.tableView.reloadData()
}
```

## 예방
- UI 업데이트는 항상 main queue로 디스패치
- 네트워크 완료 핸들러 안에서 UI를 만지는 코드는 리뷰 시 main queue 체크

## 관련
- iOS 최소 버전: 15.0
- 영향 컴포넌트: 리스트 계열 화면 전체
```

이 문서가 `docs/solutions/performance/2026-02-24-tableview-main-queue.md`로 박제되면, 다음에 유사 증상이 보이자마자 `/ios-triage`가 이 케이스를 먼저 체크한다.
