## 현실적 오류 시나리오

1. **재고 조회 API 실패** → 캐시된 값? 없으면 "알 수 없음" 표시
2. **LLM timeout** → 기본 추천 (룰 기반) 반환
3. **Network 완전 단절** → Static context만 사용
4. **권한 부족** (사용자 프로필 조회 거부) → 익명 context 사용

## Fallback Chain 구현

```swift
enum ContextQuality {
  case complete     // 모든 데이터 수집 성공
  case partial      // 일부만 성공
  case static       // 고정 데이터만
  case degraded     // 최소 context
}

func collectContextWithFallback() async -> (context: DynamicContext?, quality: ContextQuality) {
  do {
    let complete = try await collectCompleteContext()
    return (complete, .complete)
  } catch {
    do {
      let partial = try await collectPartialContext()
      return (partial, .partial)
    } catch {
      let static = getStaticContext()
      return (static, .static)
    }
  }
}
```

## LLM 오류 처리

```swift
func askAI(query: String, context: DynamicContext) async -> String {
  do {
    // 5초 타임아웃
    let response = try await withTimeoutSeconds(5) {
      await llmService.generate(prompt: buildPrompt(query, context))
    }
    return response
  } catch LLMError.timeout {
    return getFallbackRecommendation()  // 미리 만든 추천
  } catch {
    return "죄송합니다. 지금은 추천을 드리기 어렵습니다."
  }
}
```

## 사용자에게 투명하게

```swift
// 앞에 작은 아이콘으로 정보 신선도 표시
- "🟢 실시간 재고" (업데이트 < 5분)
- "🟡 최근 재고" (업데이트 < 1시간)
- "⚪ 캐시된 정보" (업데이트 > 1시간)
```

moneyflow-ios의 통계:
- 99.2% context 수집 성공률
- 0.8% partial (API 일부 실패)
- 최악의 경우 static으로 graceful degrade

## 교훈

**완벽한 context가 없어도 앱은 동작해야 한다.**

Dynamic context injection은 "정확도 개선"이지 "필수 기능"이 아니다.
Fallback이 없으면 하나의 API 오류로 entire feature가 죽는다.
