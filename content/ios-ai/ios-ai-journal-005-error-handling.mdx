## iOS AI Journal 005: Graceful Degradation - 실패할 때의 UX

**가장 중요한 원칙**: 사용자가 AI 없어도 앱이 작동해야 한다.

### 실패의 종류와 대응

#### 1. 네트워크 실패
```swift
// Fallback: 로컬 템플릿 사용
struct FailoverStrategy {
    async func getResponse(userQuery: String) -> String {
        do {
            return try await callAI(userQuery)
        } catch NetworkError.timeout {
            // AI 없이 템플릿으로 응답
            return getLocalTemplate(for: userQuery)
        }
    }
}
```

#### 2. 권한 부족 (Location 거부)
```swift
// 위치 데이터 없이 동작
async func getRecommendation() -> [Recommendation] {
    var context = AIContext()
    
    if CLLocationManager.authorized {
        context.location = await getLocation()
    } else {
        // 위치 없이도 일반적인 추천은 가능
        context.location = nil
    }
    
    return await callAI(context)
}
```

#### 3. AI 응답 실패
```swift
// Tier-based fallback
func getAnswer(to question: String) -> String {
    // Tier 1: 최신 AI
    if let aiResponse = try? await callLatestModel(question) {
        return aiResponse
    }
    
    // Tier 2: Lite 모델 (온디바이스)
    if let liteResponse = try? runLiteModel(question) {
        return liteResponse + " (제한된 버전)"
    }
    
    // Tier 3: 사전 정의된 응답
    return getPredefinedAnswer(for: question) ?? "죄송합니다. 나중에 다시 시도해주세요."
}
```

### 우아한 성능 저하

사용자에게 투명하게:
```
AI 응답: "근처 베스트셀러 추천 (위치 기반)"

↓ (위치 거부시)

AI 응답: "전국 베스트셀러 추천"

↓ (네트워크 끊김)

앱 응답: "저장된 추천 목록입니다"

↓ (AI 완전 실패)

앱 응답: "현재 AI 서비스 이용 불가. 카테고리별 추천을 보시겠어요?"
```

### 실전: 체크리스트

- [ ] 인터넷 없어도 앱 동작
- [ ] AI 응답 실패해도 사용자는 목표 달성 가능
- [ ] 권한 부족 시에도 제한된 기능 제공
- [ ] Timeout은 짧게 (5초 이상 금지)
- [ ] Fallback 응답도 유용해야 함 (스텁 응답 금지)
