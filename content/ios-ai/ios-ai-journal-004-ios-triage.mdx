## 한 줄 요약

iOS 버그를 받았을 때 코드를 바로 건드리지 않는다. `/ios-triage`가 **탐정처럼** 5 라운드로 인터뷰하면서 **한 번에 최대 3개 질문**으로 원인을 좁히고, **충분한 정보 기준 5 체크리스트**가 모두 충족되면 브리프를 생성해 `/ios-bugfix`로 인계한다. `/ios-bugfix`는 이 브리프 덕분에 Phase 1(조사) 없이 바로 Phase 2(수정) + Phase 3(검증)으로 진입한다.

---

## 왜 "인터뷰"인가

맥락이 없으면 AI는 엉뚱한 곳을 수정한다. 증상만 받은 상태에서 곧장 코드를 뒤지기 시작하면, 이름이 비슷한 파일을 보다가 전혀 관계없는 코드를 건드린다.

```
나쁜 방식
"MainViewController에서 크래시납니다. 고쳐주세요."
→ AI가 MainViewController 전체를 뒤지다가
   전혀 관계없는 코드를 수정. 버그는 그대로.

/ios-triage 방식
Round 1 (증상): 어떤 화면에서? 어떤 증상?
Round 2 (재현): 항상? 간헐? 특정 조건?
Round 3 (증거): 크래시 로그? 최근 변경?
Round 4 (맥락): 디바이스/OS/사용자 상태?
Round 5 (가설 검증): 의심 가는 모듈은?
→ 5 체크리스트 모두 충족 → 브리프 → /ios-bugfix
```

수정에 들어가기 전에 "무엇을 고쳐야 하는지"가 확정되어 있다.

## 인터뷰 원칙 4 룰

`.claude/commands/ios-triage.md`에 박제된 원칙은 체크리스트 낭독이 아니라 탐정식 추궁이다.

1. **탐정처럼 물어라.** 고정된 체크리스트를 읊는 게 아니라, 답변에 따라 다음 질문을 동적으로 결정한다.
2. **한 번에 최대 3개 질문.** 10개를 한꺼번에 쏟아내지 않는다. 답변을 듣고 판단한 뒤 다음 묶음을 던진다.
3. **모호한 답변이 오면 즉시 파고든다.** "가끔 발생"이면 → 어떤 조건일 때? 재현 가능한가?
4. **로그/스택 트레이스를 줄 수 있다고 하면 반드시 받아내라.** 추측 대신 증거로 좁힌다.

> 사용자가 "잘 모르겠다"고 하면 대안 질문을 제시한다. 레이어 가설이 생기면 확인 질문으로 검증한다.

---

## 5 Round 인터뷰 흐름

```
Round 1 — 증상 파악
  "어떤 화면/액션에서 발생하나요?"
  "어떤 증상인가요? (크래시 / 무반응 / 깜빡임 / UI 깨짐)"

Round 2 — 재현 조건
  "항상 발생하나요, 간헐적인가요?"
  "특정 조건(로그인 상태, 딥링크 진입, 백그라운드 복귀 등)에서만?"

Round 3 — 증거 수집
  "크래시 로그나 스택 트레이스 있으신가요?"
  "최근 변경된 코드나 PR이 있나요?"

Round 4 — 맥락 파악
  "어떤 디바이스/OS 버전에서 발생하나요?"
  "사용자 상태(신규/기존/로그인/게스트)는?"

Round 5 — 레이어 가설 검증
  "증상으로 보면 OO 모듈이 의심되는데, 최근 그쪽 변경이 있었나요?"
  "이 가설이 맞다면 XX 로그가 떠야 하는데, 확인 가능할까요?"
```

각 라운드는 앞 라운드 답변에 따라 질문 내용이 바뀐다. Round 2에서 "딥링크 진입 시에만"이 나오면 Round 3은 "딥링크 처리 코드 최근 변경 있나?"로 구체화된다.

---

## 충분한 정보 기준 — 5 체크리스트

5개가 **모두 충족**되어야 브리프를 생성한다. 하나라도 비면 인터뷰를 계속한다.

- [ ] 버그 증상이 한 줄로 명확히 정의됨
- [ ] 재현 스텝이 구체적 (없으면 "재현 불가" 이유 확정)
- [ ] 증거 수집 (로그 / 스택 트레이스 / 스크린샷 중 하나 이상)
- [ ] 영향 범위 추정 (특정 사용자군 / 특정 OS / 전체)
- [ ] 레이어 가설 1~3개 (어느 모듈/파일이 의심되는지)

## 실제 트리아지 예시

```
[/ios-triage 시작]

Round 1 — 증상
Q: 어떤 화면/액션에서 발생하나요? 어떤 증상인가요?
A: 알림 탭에서 특정 알림 누를 때. 앱 크래시.

Round 2 — 재현
Q: 항상 발생하나요? 특정 진입 경로가 있나요?
A: 딥링크로 앱 진입한 직후에만. 앱을 직접 켜면 괜찮음.

Round 3 — 증거
Q: 크래시 로그 있나요? 최근 변경된 코드 있나요?
A: Thread 1: EXC_BAD_ACCESS
   AlertRouter.routeToDetail(notificationId:) + 48
   어제 딥링크 처리 코드를 수정했음.

Round 4 — 맥락
Q: 특정 OS/디바이스에서만인가요?
A: 전 OS. 로그인 상태에서만 재현됨.

Round 5 — 레이어 가설 검증
Q: 딥링크 처리 후 화면이 준비되기 전에 라우팅이 들어가는 패턴 같은데,
   딥링크 수신 시점과 화면 attach 시점이 분리되어 있나요?
A: AppDelegate에서 바로 Router를 호출함.

---

[5 체크리스트 전부 충족 → 브리프 생성]
```

## 브리프 포맷 → /ios-bugfix로 인계

인터뷰가 끝나면 아래 포맷으로 브리프를 작성하고 `/ios-bugfix`에 넘긴다.

```markdown
## Bugfix 브리프 (from /ios-triage)

증상: 딥링크 진입 직후 알림 상세 크래시 (EXC_BAD_ACCESS)
재현: 딥링크 앱 진입 → 알림 탭 → 특정 알림 탭 (로그인 상태 필수)
증거: AlertRouter.routeToDetail +48 스택 트레이스
영향 범위: 로그인 사용자 전체, 전 OS
1순위 가설: 라우팅 호출 시점이 화면 준비보다 앞섬
2순위 가설: routeToDetail 파라미터 강제 언래핑 실패
3순위 가설: 딥링크 처리 → 라우터 호출 순서 역전

제안 수정 경로: guard/대기 큐로 준비 시점 이후 라우팅
```

`/ios-bugfix`는 이 브리프 덕분에 **Phase 1(조사)을 건너뛰고 Phase 2(수정) + Phase 3(검증)으로 바로 진입**한다. 트리아지가 이미 "어디를 고칠지"를 확정해놨기 때문이다.

## 수작업 디버깅 vs /ios-triage

| | 수작업 디버깅 | /ios-triage |
|--|--------------|------------|
| 질문 방식 | 애드혹 / 고정 체크리스트 | 탐정식 동적 전개 |
| 질문 수 | 끝없이 쏟아짐 | 라운드당 최대 3개 |
| 종료 조건 | "이 정도면 됐겠지" | 5 체크리스트 모두 충족 |
| 다음 단계 연결 | 수정자가 다시 조사 | 브리프로 직접 인계 |
| 재수정자 혼선 | 잦음 | 브리프가 단일 소스 |
