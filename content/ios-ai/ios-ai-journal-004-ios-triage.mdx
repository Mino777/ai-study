## iOS AI Journal 004: ios-triage - 인터뷰식 버그 파악 패턴

iOS 버그는 맥락이 부족하면 AI가 엉뚱한 곳을 수정한다. **ios-triage**는 먼저 질문으로 버그를 좁혀가는 패턴이다.

### 왜 "인터뷰"인가?

의사가 진단하는 방식과 같다. 증상을 들고 바로 수술하지 않는다.

```
❌ 나쁜 방식:
"HomeViewController에서 크래시나요. 고쳐주세요."
→ AI가 HomeViewController 전체를 수정 시도

✅ 좋은 방식 (ios-triage):
Q: 언제 발생하나요?
A: 탭바에서 홈 탭 두 번 탭할 때
Q: 항상 발생하나요?
A: 특정 사용자만 (로그인 상태)
Q: 크래시 로그 있나요?
A: EXC_BAD_ACCESS at RootRouter.swift:42
→ 이제 정확한 원인 범위 특정
```

### ios-triage 슬래시 커맨드

```markdown
# .claude/commands/ios-triage.md

## iOS 버그 트리아지 인터뷰

다음 질문을 순서대로 사용자에게 묻고 답을 기록한다.
모든 질문에 답하기 전에 수정을 시작하지 않는다.

### 1단계: 버그 현상
- 어떤 화면/액션에서 발생하나요?
- 어떤 증상인가요? (크래시 / 화면 깨짐 / 동작 안 함 / 느림)

### 2단계: 재현 조건
- 항상 발생하나요, 간헐적인가요?
- 특정 조건이 있나요? (로그인 상태, 특정 데이터, iOS 버전)
- 디바이스 vs 시뮬레이터 차이가 있나요?

### 3단계: 증거 수집
- 크래시 로그나 스택 트레이스가 있나요?
- 발생 직전에 어떤 사용자 액션이 있었나요?
- 최근 변경된 코드가 있나요?

### 4단계: 가설 수립
수집된 정보로 가능한 원인 최대 3개를 나열하고
/ios-bugfix로 브리프 전달.
```

### 실제 트리아지 예시

```
[ios-triage 시작]

Q: 어떤 화면/액션에서 발생하나요?
A: 알림 탭에서 특정 알림 누를 때

Q: 어떤 증상인가요?
A: 앱 크래시

Q: 항상 발생하나요?
A: 딥링크로 진입한 경우에만

Q: 크래시 로그 있나요?
A: Thread 1: EXC_BAD_ACCESS
   NotificationRouter.routeToDetail(notificationId:)

[트리아지 결과]
- 범위: NotificationRouter
- 원인 가설:
  1. notificationId가 nil인데 강제 언래핑
  2. RIB이 아직 attach되기 전에 라우팅 시도
  3. 딥링크 처리 시 RIB 생명주기 순서 문제

→ /ios-bugfix에 전달
```

### bugfix 브리프 포맷

```markdown
## Bugfix 브리프 (ios-triage 결과)

**증상**: 딥링크 진입 시 알림 상세 화면에서 크래시
**재현 조건**: 딥링크로 진입 + 알림 탭 → 특정 알림 탭
**증거**: NotificationRouter.routeToDetail 스택 트레이스
**가설 1순위**: notificationId 강제 언래핑

**수정 순서**:
1. NotificationRouter.routeToDetail 확인
2. nil 체크 또는 optional binding으로 수정
3. 딥링크 진입 재현 테스트
4. docs/solutions/ios/2026-04-13-notification-deeplink-crash.md 기록
```
