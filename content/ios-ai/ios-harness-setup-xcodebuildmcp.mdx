## 왜 iOS에 특화된 하네스가 필요한가

[공통 하네스 템플릿](/wiki/harness-engineering/harness-template-setup-guide)은 모든 프로젝트에 적용할 수 있지만, iOS 개발에는 **고유한 함정**이 있다:

| iOS 고유 위험 | 일반 프로젝트와 다른 점 |
|---------------|----------------------|
| **.pbxproj 파손** | 5000줄+ XML을 AI가 건드리면 프로젝트가 열리지 않음 |
| **코드 사이닝** | Provisioning Profile, Entitlement 에러는 비구조화 |
| **xcodebuild 로그 폭탄** | 빌드 출력이 수천 줄 → 컨텍스트 오염 |
| **Retain Cycle** | AI가 [weak self] 없이 클로저를 생성 → 메모리 누수 |
| **Swift 6 Strict Concurrency** | @MainActor, Sendable 누락 → 런타임 크래시 |
| **시뮬레이터 상태** | 빌드는 성공하지만 시뮬레이터에서 크래시 |

공통 하네스의 deny/hook/rule 위에 **iOS 전용 레이어**를 얹어서 이 함정들을 구조적으로 차단한다.

## 셋업 방법 (2단계)

### Step 1: 공통 하네스 이식

```bash
bash harness-template/setup.sh /path/to/ios-project
```

이 단계에서 deny 7개 + GC 드리프트 + 시크릿 가드 + /review + /compound가 설치된다.

### Step 2: iOS 레이어 추가

```bash
bash harness-template/ios/setup-ios.sh /path/to/ios-project
```

이 단계에서 추가:
- `.mcp.json` (XcodeBuildMCP)
- `swift-build-gate.sh` (빌드 검증 Hook)
- `swiftlint-check.sh` (린트 Hook)
- `ios-swift.md` (Swift 규칙)
- `/build`, `/test`, `/run` 커맨드

### Step 3: CLAUDE.md에 iOS 섹션 추가

```bash
cat harness-template/ios/CLAUDE.md.ios-append >> CLAUDE.md
```

### Step 4: settings.json에 iOS Hook 등록

```json
{
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "bash .claude/hooks/swift-build-gate.sh",
        "timeout": 300000
      }]
    },
    {
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "if echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.pbxproj$|\\.xcodeproj/'; then echo '❌ .pbxproj 직접 수정 금지. Xcode에서 Add Files 사용.' >&2; exit 2; fi"
      }]
    }
  ],
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "bash .claude/hooks/swiftlint-check.sh",
        "timeout": 10000
      }]
    }
  ]
}
```

## MCP 서버 구성: 2개 서버 조합

### XcodeBuildMCP (Sentry, 오픈소스)

**80개 도구**. Xcode 없이 동작 (headless). 빌드, 테스트, 시뮬레이터, 실기기 배포, LLDB 디버깅.

```json
// .mcp.json
{
  "mcpServers": {
    "XcodeBuildMCP": {
      "command": "npx",
      "args": ["-y", "xcodebuildmcp@latest", "mcp"],
      "env": { "XCODEBUILDMCP_SENTRY_DISABLED": "true" }
    }
  }
}
```

**핵심 가치**: xcodebuild를 Bash로 실행하면 수천 줄 로그가 컨텍스트를 오염시킨다. XcodeBuildMCP는 **구조화된 JSON**(파일 경로, 줄 번호, 에러 카테고리)을 반환 → 에이전트가 즉시 수정 가능.

```
Before (Bash):
  xcodebuild build ... 2>&1
  → 3,000줄 로그 → 에이전트가 에러를 찾느라 컨텍스트 50% 소비

After (MCP):
  XcodeBuildMCP build_sim 호출
  → JSON { errors: [{ file: "ViewModel.swift", line: 42, message: "..." }] }
  → 에이전트가 즉시 해당 줄로 이동하여 수정
```

설치:
```bash
# Homebrew (권장)
brew tap getsentry/xcodebuildmcp && brew install xcodebuildmcp

# npm
npm install -g xcodebuildmcp@latest

# on-demand (설치 불필요)
npx -y xcodebuildmcp@latest mcp
```

### Apple Xcode MCP (Xcode 26.3+, 네이티브)

**20개 도구**. 실행 중인 Xcode에 XPC로 연결. SwiftUI Preview 렌더, 문서 검색, Swift REPL.

```bash
# Xcode가 열려있을 때만 동작
xcrun mcpbridge
```

| 기능 | XcodeBuildMCP | Apple Xcode MCP |
|------|---------------|-----------------|
| Xcode 필요 | X (headless) | O (실행 중이어야) |
| 도구 수 | ~80 | ~20 |
| 빌드/테스트 | O | O |
| SwiftUI Preview | X | O |
| 문서 검색 | X | O |
| Swift REPL | X | O |
| LLDB 디버깅 | O | X |
| CI/headless | O | X |
| 실기기 배포 | O | X |

**권장 조합**: XcodeBuildMCP를 기본으로 사용하고, SwiftUI Preview/문서 검색이 필요할 때만 Apple MCP 추가.

## iOS 전용 Rules: `.claude/rules/ios-swift.md`

`.swift` 파일을 편집할 때만 자동 로딩되는 스코프드 규칙.

### 절대 금지 3가지

| 규칙 | 이유 | 대안 |
|------|------|------|
| **.pbxproj 직접 수정** | LLM이 5000줄+ XML에서 구문 에러 필연 | SPM Package.swift / Xcode에서 수동 |
| **force unwrap (!)** | 크래시 원인 1위. 런타임에서만 발견 | guard let / if let |
| **하드코딩 시크릿** | API 키 유출 | xcconfig / Keychain |

### 메모리 관리

- 클로저 캡처: `[weak self]` 필수 (@escaping, Combine sink, 네트워크 콜백)
- delegate: `weak` 선언
- NotificationCenter: deinit에서 제거 또는 Cancellable

### Swift 6 Strict Concurrency

- UI 업데이트: `@MainActor` 필수
- `Sendable` 프로토콜 준수 확인
- `Task.detached`는 최소화 — 구조화된 동시성 선호

## iOS 전용 Hooks

### swift-build-gate.sh — 빌드 시스템 자동 감지

```
git commit 시도
  → Tuist 프로젝트? → tuist build
  → xcworkspace?    → xcodebuild -workspace
  → xcodeproj?      → xcodebuild -project
  → 없음?           → 빌드 체크 skip

빌드 실패 → exit 2 (커밋 차단 — exit 1은 비차단이므로 반드시 2!)
빌드 성공 → exit 0 (커밋 허용)
```

Tuist/xcworkspace/xcodeproj를 **자동 감지**하므로 프로젝트 구조에 관계없이 동작.

### swiftlint-check.sh — 린트 자동수정 루프

```
Swift 파일 편집
  → SwiftLint 해당 파일만 린트 (전체 X — 속도 우선)
    → error 감지: exit 2 (에이전트 자동수정)
    → warning만: 표시만 하고 통과 (exit 0)
    → SwiftLint 미설치: 조용히 skip
```

**해당 파일만 린트**하는 것이 핵심. 전체 프로젝트를 린트하면 느리고 불필요한 경고가 컨텍스트를 오염.

### pbxproj 방어 Hook

```bash
# settings.json PreToolUse에 추가
if echo "$CLAUDE_FILE_PATH" | grep -qE '\.pbxproj$|\.xcodeproj/'; then
  echo '❌ .pbxproj 직접 수정 금지. Xcode에서 Add Files 사용.' >&2
  exit 2  # 반드시 2! (exit 1은 비차단 — 파일이 수정됨)
fi
```

이 Hook이 있으면 에이전트가 pbxproj를 **시도조차 할 수 없다**. CLAUDE.md에 "금지"라고 적는 것보다 훨씬 안전 — Hook은 셸 레벨 차단이므로 AI가 무시할 수 없다.

> **주의**: `exit 1`은 비차단 경고이므로 파일이 **그대로 수정된다**. 차단하려면 반드시 `exit 2`.

## 트레이드오프

### XcodeBuildMCP를 쓰면

| 이점 | 비용 |
|------|------|
| 구조화된 에러 → 빠른 수정 | MCP 서버 프로세스 상주 (메모리 ~50MB) |
| 수천 줄 로그 파싱 불필요 | 도구 스키마가 컨텍스트에 삽입 (~80도구) |
| headless CI에서 동작 | Node.js 18+ 필요 |
| 시뮬레이터 자동 관리 | 풀 빌드 (증분 빌드 아직 불완전) |

### pbxproj Hook을 걸면

| 이점 | 비용 |
|------|------|
| 프로젝트 파손 원천 차단 | 파일 추가/제거 시 Xcode 수동 조작 필요 |
| 에이전트 작업 범위 축소 → 안전 | SPM 구조가 아니면 불편 |

**권장**: SPM(Package.swift) 기반 프로젝트가 AI 친화적. pbxproj 의존도를 줄이는 것이 장기 해법.

### SwiftLint Hook을 걸면

| 이점 | 비용 |
|------|------|
| 코딩 컨벤션 자동 강제 | Edit마다 ~2초 추가 |
| error는 자동수정 루프 | SwiftLint 설치 필요 |
| warning은 통과 (차단 안 함) | 규칙 커스텀 필요 (.swiftlint.yml) |

## XcodeBuildMCP 도구 카테고리 상세 (59+도구)

에이전트가 어떤 도구를 쓸 수 있는지 알아야 적절한 워크플로우를 설계할 수 있다.

| 카테고리 | 주요 도구 | 용도 |
|----------|-----------|------|
| **프로젝트 탐색** | `discover_projs`, `list_schemes`, `list_targets` | 프로젝트 구조 자동 파악 |
| **빌드** | `build_sim`, `build_device`, `build_mac` | 시뮬레이터/실기기/Mac 빌드 |
| **테스트** | `test_sim`, `test_device` | 유닛/UI 테스트 실행 |
| **시뮬레이터** | `list_sims`, `boot_sim`, `shutdown_sim` | 시뮬레이터 라이프사이클 관리 |
| **디버깅** | `debug_attach_sim`, `debug_stack`, `debug_variables` | LLDB 원격 디버깅 |
| **UI 자동화** | `tap`, `swipe`, `type_text`, `screenshot` | 시뮬레이터 UI 인터랙션 |

### 자율 빌드-수정 루프

에이전트가 MCP로 구현하는 자율 루프:

```
1. build_sim 호출 → 구조화된 에러 JSON 수신
2. 에러의 file + line으로 이동
3. 코드 수정
4. 다시 build_sim → 성공할 때까지 반복
5. test_sim으로 테스트 실행
6. screenshot으로 UI 확인
```

이 전체 루프에 **사람 개입 0**. 기존 Bash xcodebuild에서는 3000줄 로그 파싱이 필요했지만, MCP에서는 JSON 에러 구조를 직접 읽는다.

## 에이전트 강약점 맵 — iOS 영역별

| 영역 | 이유 |
|------|------|
| **SwiftUI 뷰 합성** | modifier 체인이 패턴화되어 있고 preview로 즉시 검증 |
| **SwiftData 모델 + @Query** | 선언적 스키마, 관계 설정이 명확한 패턴 |
| **유닛 테스트 생성** | given-when-then 패턴이 기계적으로 생성 가능 |
| **대규모 리팩토링** | ObservableObject → @Observable 같은 기계적 변환 |
| **빌드 에러 자율 수정** | MCP 구조화 에러 → 즉시 수정 가능 |
| **NavigationStack 구현** | 타입 안전 라우팅이 패턴화됨 |
| **@Bindable 폼 바인딩** | 반복적 구조, 에러 가능성 낮음 |

### 에이전트가 못하는 것 (사람 필수)

| 영역 | 이유 |
|------|------|
| **.pbxproj 수정** | 5000줄+ XML, 구문 에러 필연. 단일 최대 위험 |
| **Interface Builder / Storyboard** | 바이너리 XML, LLM이 해석 불가 |

## Swift 6.2 아키텍처 패턴 — 에이전트 친화적 코드

Blake Crosley 가이드에서 에이전트가 가장 잘 생성하는 패턴들.

### @Observable 패턴 (절대 ObservableObject 사용 금지)

```swift
@Observable @MainActor final class FeatureManager {
  var items: [Item] = []
  var isLoading = false

  func load() async throws {
    isLoading = true
    defer { isLoading = false }
    items = try await api.fetchItems()
  }
}

struct FeatureView: View {
  @State private var manager = FeatureManager()

  var body: some View {
    List(manager.items) { item in
      ItemRow(item: item)
    }
    .task { try? await manager.load() }
  }
}
```

**금지 목록**: `ObservableObject`, `@Published`, `@StateObject`, `@ObservedObject` — 전부 deprecated.

### NavigationStack 타입 안전 라우팅

```swift
NavigationStack {
  List(items) { item in
    NavigationLink(value: item) {
      ItemRow(item: item)
    }
  }
  .navigationDestination(for: Item.self) { item in
    ItemDetailView(item: item)
  }
}
```

**금지**: `NavigationView`, 인라인 `destination` 파라미터.

### SwiftData 관계 패턴

```swift
@Model final class Parent {
  @Relationship(deleteRule: .cascade) var children: [Child]
}

@Model final class Child {
  var parent: Parent?  // 양방향, nullify on delete
}
```

**주의**: `@Model`에 `@Observable` 추가 금지 (중복, 경고 발생).

## pbxproj 방어 확장 — .xib/.storyboard까지

기본 Hook은 `.pbxproj`만 차단하지만, 더 안전한 확장 버전:

```bash
# .pbxproj + .xcodeproj/ + .xcworkspace/ + .xib + .storyboard 전부 차단
if echo "$FP" | grep -qE '\.(pbxproj|xcworkspace|xib|storyboard)$|xcodeproj/|xcworkspace/'; then
```

Interface Builder 파일(.xib, .storyboard)도 **바이너리 XML**이므로 LLM이 편집하면 깨진다. SwiftUI로 새 UI를 만들고, 레거시 xib는 사람이 관리.

## LLDB 원격 디버깅 — MCP로 브레이크포인트 세팅

XcodeBuildMCP는 빌드/테스트 외에 **LLDB 디버깅**도 MCP 도구로 제공한다.

```
1. debug_attach_sim   ← 시뮬레이터 앱에 디버거 연결
2. debug_breakpoint   ← 특정 파일:줄에 브레이크포인트
3. debug_variables    ← 로컬 변수 검사
4. debug_stack        ← 콜스택 확인
5. debug_continue     ← 실행 재개
```

에이전트가 자율적으로: 빌드 → 실행 → 크래시 → 디버거 연결 → 변수 검사 → 원인 파악 → 수정. **사람은 최종 확인만.**

## SwiftFormat PostToolUse Hook — SwiftLint와 역할 분담

SwiftLint가 **규칙 위반 감지**(exit 2 차단)라면, SwiftFormat은 **자동 포맷팅**(silent 정리).

```json
{
  "PostToolUse": [{
    "matcher": "Edit|Write",
    "hooks": [{
      "type": "command",
      "command": "FP=$(echo \"$CLAUDE_TOOL_INPUT\" | python3 -c \"import sys,json; print(json.load(sys.stdin).get('file_path',''))\" 2>/dev/null); if echo \"$FP\" | grep -qE '\\.swift$'; then swiftformat \"$FP\" --quiet 2>/dev/null; fi"
    }]
  }]
}
```

**SwiftLint → error 차단 + warning 표시, SwiftFormat → 자동 정리.** 둘 다 PostToolUse에 걸되, SwiftFormat이 먼저 실행(정리 후 린트).

기본: `.pbxproj`만 차단. **확장 권장**: .xib, .storyboard, .xcworkspace까지.

```json
{
  "PreToolUse": [{
    "matcher": "Edit|Write",
    "hooks": [{

```

**exit 2 사용 필수** — exit 1은 비차단이므로 파일이 그대로 수정됨!
