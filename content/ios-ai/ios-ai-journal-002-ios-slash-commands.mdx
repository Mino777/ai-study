## 한 줄 요약

일반 `/review`, `/investigate`로는 RIBs 위반·메모리 누수·앱스토어 심사를 못 잡는다. iOS 전용 7개 슬래시 커맨드와 **병렬 Agent 패턴(2 Agent / 3 Agent)** 으로 버그→배포→지식 축적 사이클을 구조화한다.

---

## 7개 커맨드 전체 맵

```
버그 발생
  │
  ▼
/ios-triage ──── 인터뷰식 버그 심층 파악 (Round 1~5)
  │
  ▼
/ios-investigate ── 4단계: Reproduce → Isolate → Identify → Fix
  │
  ▼
/ios-bugfix ─── Phase 1~5 (병렬 분석/수정 → 리뷰 → commit, push 제외)
  │
  ▼
/ios-review ─── [2 Agent 병렬] Reviewer ⇄ Defender
  │
  ▼
/ios-ship ───── Phase 0~6 (버전 bump → fastlane qa_testflight | release_testflight)
  │
  ▼
/compound ──── [3 Agent 병렬] CHANGELOG / 솔루션 / 회고

별도 트랙:
/ios-arch ─── [3 Agent 토론] 옵션A 지지 · 옵션B 지지 · 판사
```

---

## `/ios-triage` — 버그 심층 인터뷰

버그 리포트 한 줄만 받고 바로 수정에 뛰어들지 않는다. **"탐정처럼 물어라"** 원칙으로 Round 1~5 인터뷰를 진행해 원인 레이어를 좁힌 뒤 브리프를 생성한다.

- Round 1: 증상
- Round 2: 재현
- Round 3: 증거 (로그/스택/최근 변경)
- Round 4: 맥락 (기기/OS/빌드 변종)
- Round 5: 레이어 가설 검증

제약: **한 번에 최대 3개 질문**. 세부 구조는 Ep.004에서 다룬다.

---

## `/ios-investigate` — 근본 원인 4단계

크래시·회귀·이상 동작의 근본 원인을 찾기 위한 4단계 프로토콜.

1. **Reproduce** — 재현 경로 확보
2. **Isolate** — 변수/레이어 분리
3. **Identify** — 원인 특정 (가설 검증)
4. **Fix** — 수정 경로 제시

**핵심 원칙**: *"원인을 못 특정하면 결과 리포트에 반드시 '원인 미확정'이라고 명시한다."* 추측으로 넘어가지 않는다.

---

## `/ios-bugfix` — 실행 워크플로

티켓 기반 버그 픽스를 Phase 1~5로 실행.

1. 티켓/이슈 파악
2. 병렬 분석 + 수정
3. 내부 리뷰
4. 테스트
5. **commit까지만. push는 사용자 직접**

단방향 커밋 원칙을 지켜서 AI가 원격 히스토리에 자동으로 손대지 못하도록 한다.

---

## `/ios-review` — 2 Agent 병렬 리뷰

RIBs/ReactorKit 특화 코드 리뷰. 다른 커맨드와 다르게 **Agent 2개를 동시에 띄운다**.

- **Agent A (Reviewer)** — 변경 파일에서 이슈를 발굴한다 (레이어 위반, weak self 누락, Reactor 순수성 위반 등).
- **Agent B (Defender)** — Reviewer가 찾은 이슈를 방어·반박하거나 **자동 수정 패치**를 제안한다.

Phase 1~3:
1. 변경 파일 파악
2. 두 Agent 병렬 실행
3. 결과 병합 + 요약

한 명의 리뷰어가 혼자 판단하는 것보다 *발굴 + 반박* 쌍으로 띄워서 거짓 양성을 걷어낸다.

---

## `/ios-arch` — 3 Agent 아키텍처 토론

아키텍처 결정이 필요할 때 **토론형 3 Agent**를 띄운다.

- **Agent A** — 옵션 A 지지
- **Agent B** — 옵션 B 지지
- **Agent C** — 판사 (두 입장을 듣고 판결 + 실행 계획)

Phase 1~3:
1. 컨텍스트 수집
2. 3 Agent 병렬 토론
3. 실행 계획 출력

아키텍처는 "정답"이 없어서 단일 Agent의 의견을 그대로 쓰면 편향이 그대로 코드가 된다. 강제로 두 입장을 세운 뒤 판사가 가른다.

---

## `/ios-ship` — TestFlight 배포

Fastlane 기반 배포 플로우. Phase 0~6.

0. 대상 확인 (stg / prd)
1. `/ios-review` + `/cso` 호출
2. `MARKETING_VERSION` / `CURRENT_PROJECT_VERSION` 업데이트
3. CHANGELOG 갱신
4. 커밋 + 태그
5. `bundle exec fastlane qa_testflight` (stg) 또는 `release_testflight` (prd)
6. 사후 확인

배포는 항상 버전 bump → 리뷰 → fastlane 순서로 강제된다. 순서를 바꾸면 Phase가 실패한다.

---

## `/compound` — 3 Agent 지식 축적

스프린트가 끝난 뒤 지식을 축적하는 복리 커맨드. **3 Agent 병렬**로 동시 실행.

- **Agent A** — CHANGELOG 작성
- **Agent B** — 솔루션 문서 작성 (`docs/solutions/`)
- **Agent C** — 회고 작성 (`docs/retros/`)

Phase 1~5:
1. 변경 수집
2. 3 Agent 병렬 실행
3. CLAUDE.md 동기화
4. Claude 메모리 갱신
5. **단일 커밋**으로 묶어서 기록

세부 구조는 Ep.005에서 다룬다.

## 공통 원칙

> **추측 금지** — 원인을 못 특정하면 반드시 "원인 미확정"으로 리포트한다. 그럴싸한 가설로 덮어쓰지 않는다.
>
> **병렬 Agent** — 독립적인 작업(리뷰 vs 방어, 옵션 지지 vs 판사, CHANGELOG vs 솔루션 vs 회고)은 반드시 동시에 띄운다. 순차 실행은 관성에서 나오는 낭비다.
>
> **단방향 커밋** — AI는 commit까지만. `git push`, 태그 푸시, 릴리즈는 사람이 직접 트리거한다.
