## 배경 — 메모리룰이 새고 있었다

iOS 워커 저장소와 ai-study(이 위키) 양쪽에서 같은 패턴이 동시에 발견됐다. **메모리에 박제된 룰을 AI가 silent하게 위반한다.**

사례 1 — 식별자 노출 (ai-study)

```
메모리 룰:  feedback_company_project_names.md
            "특정 식별자(워커 모듈명 등)는 ai-study 공개 글에 절대 노출 금지"

위반 1:   2026-04-13 iOS Journal 재작성 시 워커 모듈명 그대로 노출
위반 2:   2026-04-23 Journal 027에서 재발 — 훅이 envvar만 읽어서
                  stdin 스키마로 바뀐 하네스에서 silent pass됐음
```

사례 2 — 보호 브랜치 직접 커밋 (iOS 워커)

```
메모리 룰:  develop/main 직접 커밋 금지

위반:     `git commit --no-verify`로 우회. pre-commit 훅이 막아도
         Claude가 즉시 --no-verify 붙여 재시도하면 그대로 통과
```

두 사례 모두 **메모리만으론 부족했다.** 메모리는 다음 세션에서 모델이 *떠올려야* 작동한다. 떠올리지 못하면 silent하게 위반된다. 한 번 새면 계속 샌다.

이게 이번 스프린트의 가설이다 — **한 번이라도 위반된 룰은 즉시 deterministic 가드(PreToolUse 훅)로 escalate**.

```
[기존]  메모리 룰 → AI가 다음 세션에 상기 → 가끔 잊음 → silent 위반
[목표]  메모리 룰 + PreToolUse 훅 → exit 2로 차단 → 잊을 수 없음
```

## exit code 규약 — Claude Code 훅의 핵심

먼저 Journal 007에서 박제한 함정을 다시 짚는다. PreToolUse 훅에서 exit code는 단순한 성공/실패가 아니다:

| exit | 의미 | Claude 동작 |
|------|------|-------------|
| `0` | 통과 (silent) | 도구 실행 진행 |
| `1` | 비차단 실패 | **도구 실행 계속됨** (stderr만 사람에게 보임) |
| `2` | 차단 | 도구 실행 막힘 + stderr가 **에이전트 컨텍스트에 주입** |

**exit 1로 두면 사람은 메시지를 봐도 Claude는 그냥 다음 단계로 진행한다.** silent failure의 직접 원인이다. 모든 가드는 exit 2로 통일한다.

OpenAI Codex 원칙도 같은 줄기다 — *성공은 silent, 실패만 표면화*. 통과 시 출력 0줄, 차단 시 명확한 메시지 + 우회 경로.

## 가드 1: branch-commit-guard — `--no-verify` 자동 우회 차단

iOS 워커의 `.claude/hooks/branch-commit-guard.sh`. PreToolUse Bash 훅으로 등록.

### 1-A. Protected 브랜치 차단

```bash
PROTECTED_BRANCHES="develop|main|master"
BRANCH=$(git branch --show-current)

if echo "$BRANCH" | grep -qE "^($PROTECTED_BRANCHES)$"; then
  echo "🛑 protected 브랜치($BRANCH) 직접 커밋 금지" >&2
  echo "   강제 우회: CLAUDE_ALLOW_NO_VERIFY=1 git commit ..." >&2
  exit 2
fi
```

### 1-B. `--no-verify` 자동 우회 차단 — 1차 회피 시도 학습

이게 핵심이다. Claude가 1-A에서 차단되면 **자동으로 `--no-verify`를 붙여 재시도하는 패턴**이 발견됐다. pre-commit 훅을 우회해 protected 브랜치를 오염시킨다.

해결: Bash 명령에서 `--no-verify` 토큰을 검출하면 차단:

```python
# shlex 토큰 파싱 — -m 본문 false positive 방지
import sys, shlex
args = shlex.split(sys.stdin.read())
skip = False
for a in args:
  if skip: skip = False; continue
  if a in ('-m', '--message', '-F', '--file'):
    skip = True; continue
  if a == '--no-verify':
    print('1'); sys.exit()
print('0')
```

naive `grep` 으로 하면 `fix: --no-verify 우회 차단` 같은 정상 커밋 메시지가 false positive로 잡힌다. shlex로 토큰화 후 옵션 위치만 검사한다.

차단 메시지에 우회 경로 명시:

```
🛑 'git commit --no-verify' 차단 (Claude는 훅 우회 금지)

   이유: pre-commit 훅이 .claude/, docs/, CLAUDE.md 등 셋업 파일을 자동으로
   chore/claude-setup 브랜치로 라우팅하는데, 우회는 이걸 막아 피쳐 브랜치를 오염시킴.

   해결 (우선순위):
   1. 그냥 'git commit'으로 재시도 — 훅이 자동 처리
   2. 훅이 'worktree 생성 실패'로 fail하면 원인 fix 후 재시도
   3. 사용자가 직접 커밋 (사람은 본인 책임으로 우회 가능)

   강제 우회: CLAUDE_ALLOW_NO_VERIFY=1 git commit --no-verify ...
   (사용자 명시 결정 시만 사용, AI는 자율 사용 금지)
```

### 1-C. 이중 차단 계층

```
[1] Claude Code PreToolUse hook  ← AI가 명령 실행 *전* 차단 (exit 2)
[2] git pre-commit hook          ← git이 실제 커밋 *직전* 차단 (exit 1)
```

[1]이 새도 [2]가 잡는다. [2]가 우회되면 [1]이 더 위에서 잡는다. 두 계층 모두 protected 브랜치 + 셋업 파일 라우팅 검사를 중복 수행한다.

## 가드 3: ai-study `식별자 노출 차단` — 같은 패턴, dogfood

이게 결정적인 dogfood 사례다. iOS 워커에서 굳힌 패턴을 ai-study에 동일하게 적용했다 (`feedback_dogfood_in_same_session` 메모리룰의 실전 적용).

`.claude/hooks/` 의 식별자 노출 차단 훅 핵심 구조:

```bash
# 입력: stdin JSON (Claude Code PreToolUse 표준) + envvar fallback
INPUT_JSON=""
if [ -t 0 ]; then
  INPUT_JSON="${CLAUDE_TOOL_INPUT:-}"
else
  INPUT_JSON="$(cat)"
  [ -z "$INPUT_JSON" ] && INPUT_JSON="${CLAUDE_TOOL_INPUT:-}"
fi

# 입력 비어있으면 fail-closed (보수적 차단)
[ -z "$INPUT_JSON" ] && { echo "🚫 입력 비어있음 — 보수적 차단" >&2; exit 2; }

# 화이트리스트: 메모리 파일 자체, 훅 스크립트 자체, .gitignore, settings*.json
# 이외 경로에서 금지 식별자 패턴 발견 시 exit 2
```

### 박제된 함정 — stdin 스키마 변경

이 훅의 주석에 박혀 있는 사실:

> 2026-04-13 iOS Journal 재작성에서 식별자 노출. **2026-04-23 Journal 027에서 재발 — 훅이 envvar만 읽어서 stdin 스키마로 바뀐 하네스에서 silent pass됐음.**

Claude Code 하네스 스키마가 envvar(`CLAUDE_TOOL_INPUT`) → stdin JSON으로 바뀌면서 훅이 silent pass됐다. **하네스 스키마 변경이 가드를 무력화시킨 사례.** 이후 stdin + envvar 둘 다 처리하는 패턴으로 굳혔다 — 다른 모든 훅도 이 패턴 채택.

### 자기검증 — 이 글을 쓰는 도중 차단당함

이 저널 1차 초안을 작성할 때 본문에 금지 식별자 그대로 넣었다가 Write 툴이 **exit 2로 차단**됐다. 가드가 자기 dogfood로 정확히 작동한 순간이다. 차단 메시지가 stderr로 컨텍스트에 주입돼서 즉시 본문을 익명화 표현으로 재작성. 메모리만 있었으면 그대로 publish됐을 것.

## 가드 4: session-start GC drift detector — 가드 자체의 가드

`.claude/hooks/session-start.sh`에 GC(Garbage Collection) drift 3종 체크 추가. 세션 시작 시 1회 실행:

```bash
# GC drift 1: CLAUDE.md 비대화 (>250줄이면 캐시 효율 저하)
[ "$(wc -l < CLAUDE.md)" -gt 250 ] && echo "⚠️  CLAUDE.md drift" >&2

# GC drift 2: deny 규칙 미설정 (보안 후퇴)
grep -q '"deny"' .claude/settings.json || echo "⚠️  deny 규칙 없음" >&2

# GC drift 3: hooks/.sh 파일 없음 (가드 무력화)
[ -z "$(ls .claude/hooks/*.sh 2>/dev/null)" ] && echo "⚠️  hooks 비어있음" >&2
```

이게 메타 가드다 — **다른 가드들이 silent하게 사라지지 않았는지 매 세션 시작 시 확인**. 누군가 `.claude/hooks/`를 통째로 삭제하거나 deny 규칙을 지우면 다음 세션 시작 메시지에 경고가 박힌다.

세션 시작 메시지에 `✅ GC: 하네스 드리프트 없음`이 박힌 게 이 훅의 출력이다 (이 세션에서도 통과).

## 패턴 — 메모리룰을 행동가드로 escalate하는 5단계

이번 스프린트에서 굳어진 절차:

### 1. 메모리룰 위반 1회 발생

```
"식별자 노출 금지"가 메모리에 있는데
2026-04-13 Journal 재작성에서 위반
```

### 2. 위반 패턴을 grep 가능한 deterministic 룰로 변환

자연어 룰 → 정규식/AST 룰. 예:
- "식별자 노출 금지" → 금지 토큰 정규식 grep
- "protected 브랜치 직접 커밋 금지" → `git branch --show-current` 검사
- "API 키 커밋 금지" → 6개 토큰 정규식

### 3. PreToolUse 훅으로 등록 (exit 2)

```json
{
  "hooks": {
    "PreToolUse": [
      {"matcher": "Write|Edit", "hooks": [{"type": "command", "command": "..."}]},
      {"matcher": "Bash", "hooks": [{"type": "command", "command": "..."}]}
    ]
  }
}
```

### 4. 화이트리스트 정의 (의도적 self-reference 허용)

훅 스크립트 자체, 메모리 파일 자체, settings.json 등은 패턴이 등장해도 정상. 이걸 미리 허용해야 훅이 자기 자신을 못 고치는 deadlock이 없다.

### 5. 메타 가드(GC drift)로 가드 자체를 보호

훅 파일이 사라지거나 deny 규칙이 지워지면 다음 세션 시작 시 경고. 가드는 silent하게 사라지지 않는다.

## 박제된 함정

### 함정 1: exit 1 vs exit 2 — Journal 007 재발 방지

훅에서 exit 1을 쓰면 stderr만 사람에게 보이고 Claude는 그냥 진행한다. **모든 차단은 exit 2.** Journal 007에서 박제된 함정이지만, 새 훅 작성 시 매번 까먹어서 이번 스프린트에서 다시 박제.

### 함정 2: stdin vs envvar 스키마 차이

Claude Code 하네스가 시간이 지나면서 입력 스키마를 바꿀 수 있다. envvar만 읽는 훅은 stdin 스키마로 바뀌면 silent pass된다. **모든 훅은 stdin + envvar 둘 다 처리** + **둘 다 비어있으면 fail-closed (보수적 차단)**.

### 함정 3: -m 본문 false positive

naive grep 으로 검사하면 정상 커밋 메시지("--no-verify 차단 추가")가 false positive로 잡힌다. **shlex/argparse로 토큰화 후 옵션 위치만 검사**.

### 함정 4: 우회 자동화 학습

Claude가 차단되면 즉시 우회 시도(--no-verify, --force 등)를 *자동으로* 붙여 재시도하는 패턴이 보인다. **차단 메시지에 우회 경로를 명시**해서 환경변수(`CLAUDE_ALLOW_NO_VERIFY=1`)로만 가능하게 한다 — 사용자 명시 의사결정 외에는 우회 불가.
