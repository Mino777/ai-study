## iOS AI Journal 000: iOS AI 하네스 베이스라인

모든 iOS 프로젝트에 AI 에이전트를 붙이기 전에 먼저 **환경 설계**를 해야 한다. 에이전트가 코드를 잘못 건드리지 않도록 울타리를 치는 것이 핵심.

### 왜 iOS는 별도 하네스가 필요한가?

웹 프로젝트와 iOS 프로젝트는 AI 에이전트 관점에서 다르다:

| 구분 | 웹 | iOS |
|------|-----|-----|
| 아키텍처 | 자유로움 | RIBs / VIPER / Clean 강제 |
| 빌드 | npm run build (수초) | xcodebuild (수분) |
| 의존성 | node_modules | SPM + CocoaPods 혼재 |
| 테스트 | 즉각적 | 시뮬레이터 필요 |
| 컨텍스트 크기 | 보통 | .build/ 폴더가 130GB+ |

에이전트가 iOS 프로젝트를 그냥 읽으면 빌드 아티팩트까지 다 삼켜버린다.

### RIBs 아키텍처에서 AI 에이전트의 역할 범위

RIBs 구조에서 AI가 건드려도 되는 곳과 건드리면 안 되는 곳을 명확히 한다:

```
✅ 건드려도 되는 곳
├── Interactor (비즈니스 로직)
├── Presenter (뷰 로직)
├── Builder (의존성 조립)
└── Tests/

❌ 건드리면 안 되는 곳
├── .build/           ← SPM 캐시, 130GB+
├── Pods/             ← CocoaPods 의존성
├── *.xcodeproj       ← Xcode 프로젝트 파일
└── DerivedData/      ← 빌드 아티팩트
```

### CLAUDE.md 핵심 원칙

```markdown
## iOS 프로젝트 AI 에이전트 규약

### 절대 금지
- .build/, Pods/, DerivedData/ 파일 수정
- xcodeproj 파일 직접 수정
- 시뮬레이터 없이 UI 코드 "완성됐다" 주장

### 필수 확인
- PR 전 swiftlint 실행
- RIBs 레이어 경계 침범 여부 확인
- 새 파일 추가 시 타겟 멤버십 확인 (AI가 놓치는 부분)

### 컨텍스트 주입
- 현재 작업 중인 RIB 이름 명시
- 연관된 Interactor/Router/Builder 파일 미리 열기
```
