## WO-005: Interface 모듈 분리 — 의존성 방향 뒤집기

### 문제

`OdaPresentation`이 `OnewayPresentation`을 직접 import해서 `OnewayReturnSearchBuilder`를 받아쓴다. 한 Feature가 다른 Feature의 **구현 전체**를 링크해야 타입 계약(Buildable/Listener) 하나를 얻는 구조.

```
OdaPresentation ──import──→ OnewayPresentation (구현 전체)
                             ↑ 필요한 건 프로토콜 2개뿐
```

이게 왜 위험한가:
1. 한 Feature 수정이 무관한 Feature 재빌드를 유발
2. 모듈 그래프가 조금만 복잡해지면 순환 의존 발생
3. Tuist 전환 시 transitive 전파가 끊기면서 컴파일 에러 폭탄

### 해결: UIKit-free Interface 패키지

```
OnewayPresentationInterface/
├── Sources/
│   ├── ReturnZoneType.swift          # pure enum, no UIKit
│   ├── OnewayReturnSearchInterface.swift   # Buildable + Listener
│   └── OnewayReturnLocationInterface.swift # Buildable + Listener
└── Project.swift
```

**핵심 규칙**: Interface는 **프로토콜 + 값 타입만**. UIKit extension은 원본 모듈에 남긴다.

```swift
// OdaDependency.swift — 구체 Builder가 아니라 Interface 프로토콜을 요구
import OnewayPresentationInterface

public protocol OdaDependency: Dependency {
  var onewayReturnSearchBuildable: any OnewayReturnSearchBuildable { get }
  var onewayReturnLocationBuildable: any OnewayReturnLocationBuildable { get }
}
```

OdaPresentation에서 OnewayPresentation/OnewayRepository 직접 참조 6개를 제거하고, Buildable 2개 DI로 교체. Transport/BridgeComponent가 concrete builder를 주입.

46개 모듈을 한 번에 전환하면 어디서 깨졌는지 모른다. **리프(의존성 0)부터 올라가는 Wave 전략**:

| Wave | 모듈 | 특성 |
|---|---|---|

각 Wave 끝에 `tuist generate --no-open` (10초)으로 manifest 오류 smoke test 통과 후 커밋. **Wave 단위 배치 커밋이 토큰 예산도 관리**해준다 — 한 Wave가 하나의 컨텍스트 청크.

### Tuist 4.27 → 4.185 업그레이드

처음에 Tuist 4.27로 시도했다가 `targets[3].settings[0].name missing` 에러로 **1시간+ 삽질**. `brew info tuist`로 확인하니 4.185.1이 최신. 업그레이드 후 에러 소실.

**교훈**: `brew info <tool> && <tool> version` — 마이그레이션 시작 전 첫 번째 명령어로 고정.

### `productTypes` — 전부 framework는 틀렸다

```swift
// ❌ 이렇게 하면 Kakao/Firebase/Braze가 깨진다
productTypes: [
  "KakaoSDK": .framework,  // sub-target 간 상호의존 → symbol 누락
]

// ✅ 선택적 적용: 단일 target만 framework
productTypes: [
  "RIBs": .framework,
  "RxSwift": .framework,
  "SnapKit": .framework,
  // KakaoSDK, Firebase, Braze → static 유지 (주석으로 이유 명시)
]
```

**근본 원인**: `productTypes`의 granularity가 패키지 단위인데, 복합 SDK의 문제는 타겟 단위에서 발생. KakaoSDKCommon/Auth/User가 각각 별개 framework로 쪼개지면서 상호 참조 심볼이 누락.

**결정 트리**:
1. 단일 target이고 sub-product 없음 → `.framework` 추가
2. 내부 다중 target SDK → static 유지 + 소비자에서 `.external(name:)` 명시
3. 애매하면 static으로 두고 빌드 확인 후 승격

### Package.swift 충돌 회피

Tuist와 SPM이 같은 디렉토리의 `Package.swift`를 각자 해석하려 해서 워크스페이스 충돌. 기존 SPM `Package.swift`를 **`.bak`으로 rename** — 롤백이 필요하면 `.bak` → `Package.swift`로 복원.

## 빌드 체인 복구 — scan-imports 자동화

### 문제: Transitive Dependency 폭탄

SPM 시절에는 의존성이 transitive로 전파됐다. 모듈 A가 모듈 B를 import하면, B의 의존성 C도 자동으로 보였다. **Tuist 전환 후 이 자동 전파가 끊겼다.** 결과: Presentation 레이어 ~20개 모듈에서 `extension method link 실패` 에러 폭탄.

```
// 예: Common.SomeRepository.fetch()가
// (extension in RxUtil):ObservableType.debugRx를 찾지 못함
```

### 해결: 2단계 자동화

**1단계: `scan-imports.py`** (228줄) — Swift 소스의 `import` 구문과 `Project.swift`의 declared dependencies를 비교해서 **차집합 계산**.

```bash
python3 scripts/scan-imports.py --all --report
# 출력: 모듈별 누락 dependency + .project()/.external() 라인 제안
```

내부 인덱스는 `Package.swift.bak`의 `.library` + `Project.swift`의 `ModuleTarget.framework name`. 외부 인덱스는 `Tuist/Package.swift`의 `productTypes` 키 + `.product(name:)`.

**2단계: `patch-project-deps.py`** (148줄) — 스캐너 리포트를 받아서 각 모듈의 `Project.swift` `dependencies` 배열 끝에 라인 자동 삽입. `UNRESOLVED`는 주석으로 보존.

```bash
python3 scripts/scan-imports.py --report | python3 scripts/patch-project-deps.py
# 28개 모듈 일괄 패치 — 수동이었으면 모듈당 5분, 총 1.5시간+
```

### 한계: implicit extension은 못 잡는다

```swift
// AccountDomain/Sources/SomeEntity.swift
let formatted = Date().toyyyyMMddHHmmHypen  // CoreUtil extension
// ↑ import CoreUtil 문이 없다! 스캐너 감지 불가
```

SPM transitive로 "보이던" extension이 Tuist 전환 후 끊기면서 `undefined symbol`. 소스에 import가 없으니 스캐너가 잡을 수 없다. **7개 Domain 모듈에 CoreUtil을 수동 보강** — 빌드 에러를 보고 하나씩.

### Build Chain Status (2차 세션 시점)

스캐너 + 수동 보강으로 달성한 빌드 체인:

```
Network → Repository → Domain → BusinessWebView → Account → Menu ✅
                                                                  │
                                              OnewayPresentation ──┘ ❌ (Routing 미이전)
```

3차 세션 후 멀티타겟 7모듈(Transaction/Smartkey/Transport/Splash/NotificationHub/FreeOneWay) + FloatingPanel 등록으로 **빌드 체인이 전체 Presentation 레이어까지 확장**.

## WO-008: Routing 프로토콜 Interface 추출

### 빌드 체인의 마지막 벽

WO-005에서 Buildable/Listener만 Interface로 이전했다. **Routing 프로토콜은 OnewayPresentation 내부에 남겨뒀다.** 이게 문제가 됐다.

```swift
// Transaction/Sources/.../UsageHistoryDetailRouter.swift
private var onewayReturnLocationRouting: OnewayReturnLocationRouting?
// error: cannot find type 'OnewayReturnLocationRouting' in scope
```

Transaction이 `OnewayReturnLocationRouting`을 쓰려면 OnewayPresentation을 직접 import해야 하는데, 그러면 **순환 의존**. SPM 시절에는 transitive로 보였지만 Tuist가 그 환상을 깨뜨렸다.

### 해결: Routing도 Interface로

```swift
// Before: OnewayPresentation 내부
public protocol OnewayReturnLocationRouting: ViewableRouting { ... }

// After: OnewayPresentationInterface로 이전
// OnewayPresentation은 extension으로 conform
extension OnewayReturnLocationRouter: OnewayReturnLocationRouting { ... }
```

Transaction/Smartkey/OdaLocationConfirmPresentation이 `.project(target: "OnewayPresentationInterface")`만 의존하면 순환 없이 Routing 타입 참조 가능.

**판별 기준 — 어떤 Routing을 이전하나?**

```bash
grep -rn "OnewayReturn.*Routing" --include="*.swift" \
    Transaction/ Smartkey/ OdaLocationConfirmPresentation/ OdaPresentation/
```

외부 참조가 있으면 → Interface 이전 대상. 이번에 `OnewayReturnLocationRouting` + `OnewayReturnSearchRouting` 2개 이전 완료.

| Transaction | +16 | Presentation 타겟 집중 |
| Smartkey | +13 | Presentation 타겟 집중 |
| Transport | +16 | OdaPresentation 의존 누락 발견 |

`scan-imports --all --report` 기반이지만, 멀티타겟이라 **서브타겟별 import → dependency 매핑을 수동**으로 해야 했다. FreeOneWay만 4개 서브타겟에 총 38개 dependency 추가.

### 그림자 의존성 감사

스캐너가 `UNRESOLVED`로 남긴 항목들을 하나씩 추적:

- **FloatingPanel**: `Package.swift.bak`에도 `Tuist/Package.swift`에도 없었음. OnewayPresentation이 실제로 사용 중 → `Tuist/Package.swift`에 `scenee/FloatingPanel` (3.0.0+) 등록
- **netfox / FLEX**: 메인 앱 Xcode SPM 전용 디버깅 도구. Tuist 모듈에서는 미사용 → 등록 불필요
- **adrop-ads-backfill**: 코드에서 실제 import 없음 → 등록 불필요

**"그림자 의존성"** = 원본 SPM 매니페스트에 선언 없이 `Package.resolved`에만 존재하는 패키지. 이전 프로젝트 구성에서 우회로로 들어온 것들. 스캐너의 `UNRESOLVED` 보고가 이 그림자들을 수면 위로 올려준다.

### Scaffold 템플릿 기본 의존성 보강

implicit extension 누락 문제를 **예방 인프라**로 해결:

```stencil
// Tuist/Templates/feature/Project.stencil
// Feature 타겟 기본 deps
.external(name: "RIBs"),
.external(name: "ReactorKit"),
.external(name: "RxSwift"),
.external(name: "RxCocoa"),
.project(target: "CoreUtil", path: "../Util"),
.project(target: "Log", path: "../Util"),
.project(target: "Persistent", path: "../Persistent"),

// Interface 타겟 기본 deps  
.external(name: "RIBs"),  // Buildable/Listener 정의용
```

`tuist scaffold feature --name X`로 새 모듈을 만들면 **CoreUtil/Log/Persistent가 처음부터 포함**. Domain 레이어에서 `Date.toyyyyMMddHHmmHypen` 같은 CoreUtil extension을 import 없이 쓰다가 빌드 에러 나는 문제를 원천 차단.

`greencarModule()` settings도 자동 적용되어 4-config (Debug/QA/Release/Release_Debug) 통일.

## 함정 7종 박제

### 1. Tuist 버전 체크 지연

4.27로 시작 → `targets[3].settings[0].name missing` 에러로 1시간 삽질. `brew info tuist`로 4.185.1이 최신인 걸 확인하고 업그레이드하니 해소.

**규칙**: 마이그레이션 첫 명령어는 `brew info <tool> && <tool> version`.

### 2. productTypes 전역 적용 불가

위에서 상세 기술. 복합 SDK(Kakao/Firebase/Braze)의 sub-target 간 상호의존 때문에 전역 `.framework`는 불가. **선택적 적용 + 제외 이유 주석** 패턴.

### 3. Swift Framework Transitive Extension

SPM에서 Tuist로 전환하면 transitive extension 전파가 끊긴다. `import` 없이 쓰던 extension이 갑자기 `undefined symbol`. 스캐너로 잡히지 않아 빌드 에러 기반 수동 보강 필요.

### 4. Interface 추출 범위 오판

Buildable/Listener만 이전하고 Routing은 남겨뒀다가 빌드 체인이 멈춤. **Interface 추출 기준은 "외부 참조되는 모든 public protocol"**이어야 한다. grep으로 외부 참조 전수조사 후 한 번에 이전.

### 5. SPM/Tuist Package.swift 충돌

같은 디렉토리에 `Package.swift`가 있으면 Tuist generate와 SPM resolve가 동시에 동작해서 워크스페이스가 꼬인다. `.bak` rename으로 회피.

### 6. 자동 패처 멀티타겟 skip

`patch-project-deps.py`가 멀티타겟 모듈(Network, OCR, AddressSearch 등 10개)에서 서브타겟 경계를 추론 못해 skip. 3차 세션에서 7개를 수동 패치했지만, 서브타겟별 dependency가 달라 자동화 혜택이 없었다. FreeOneWay 하나에 4개 서브타겟 × 별도 deps = 38개 추가.

### 7. 그림자 의존성 — Package.resolved에만 있는 유령

`FloatingPanel`이 OnewayPresentation에서 실제로 사용되는데, 원본 `Package.swift`에도 `Tuist/Package.swift`에도 선언이 없었다. `Package.resolved`에만 존재 — 이전 프로젝트 구성에서 우회로로 들어온 "그림자". 스캐너의 `UNRESOLVED` 보고가 이걸 수면 위로 올렸고, 추적 결과 `netfox`/`FLEX`(디버그 전용)와 `adrop-ads-backfill`(미사용)은 등록 불필요, FloatingPanel만 등록 필요로 판별.

## 교훈

1. **마이그레이션 첫 명령어는 버전 확인이다.** `brew info <tool> && <tool> version` — 구버전으로 삽질하는 시간은 순수 낭비. Tuist 4.27 → 4.185에서 1시간을 태웠다.

2. **productTypes는 패키지 단위지만 문제는 타겟 단위다.** 복합 SDK의 sub-target 간 상호의존은 dynamic framework 전환으로 깨진다. 선택적 적용 + 제외 이유 주석이 유지보수 가능한 패턴.

3. **Interface 추출 기준은 "외부 참조되는 모든 public protocol"이다.** Buildable/Listener만 이전하고 Routing을 남기면 빌드 체인이 멈춘다. grep으로 외부 참조를 전수조사한 뒤 한 번에 이전.

4. **자동화 스캐너의 한계를 정직하게 박제한다.** scan-imports.py는 explicit import만 잡는다. implicit extension은 빌드 에러로만 발견된다. "100% 자동화"가 아니라 "80% 자동화 + 20% 수동"이 현실적 기대치.

5. **리프-업 빌드 검증 순서가 진도 가시성을 만든다.** 46개 모듈을 무작위로 빌드하면 어디서 막혔는지 모른다. 의존성 0인 리프부터 위로 올라가면 "한 모듈 고치면 체인이 한 단계 전진"하는 리듬이 생긴다.

6. **인프라 마이그레이션은 멀티 에이전트보다 단일 세션이 맞다.** "동일 작업 46회 반복"은 분산의 오버헤드가 이득보다 크다. Worker Pool의 진짜 무대는 다중 레이어 동시 작업(피쳐 개발)이다.

7. **발표 슬라이드와 이식 사이에는 물리적 장벽이 있다.** 토스의 Microfeatures 패턴은 설계로는 깔끔하지만, 40+ 모듈 레거시에 이식할 때 transitive extension, productTypes granularity, Package.swift 충돌 같은 빌드 시스템 레벨의 장벽이 있다. 이 장벽을 박제하는 게 실전 가치.

8. **scaffold 템플릿에 기본 의존성을 넣는 게 예방 인프라다.** implicit extension 문제를 "발생 후 수동 보강"에서 "발생 자체를 차단"으로 전환. CoreUtil/Log/Persistent를 scaffold 기본에 포함하면 새 모듈에서 같은 삽질이 원천 차단된다.

9. **스캐너의 UNRESOLVED 보고는 그림자 의존성 감사의 입구다.** 자동화가 "해결 못 한 것"을 정직하게 남기면, 그게 수동 추적의 시작점이 된다. FloatingPanel처럼 Package.resolved에만 숨어있던 유령을 찾는 유일한 경로.
