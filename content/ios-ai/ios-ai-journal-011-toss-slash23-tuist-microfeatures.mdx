## 배경 — 왜 토스 아키텍처인가

ios-study는 RIBs 기반 58개 SPM 패키지로 구성된 대규모 모듈러 앱이다. 문제:

| 증상 | 원인 |
|------|------|
| Feature 간 순환 참조 위험 | `OdaPresentation` → `OnewayPresentation` 직접 import |
| `.pbxproj` 머지 컨플릭트 빈발 | Xcode 프로젝트 파일 수동 관리 |
| 전체 빌드 필수 (UI 한 줄 수정에도) | Feature 독립 빌드 불가 |
| 새 모듈 생성에 10-15분 | Package.swift 복붙 + 수동 설정 |

토스 SLASH23에서 공개한 **Microfeatures 5계층 구조**가 이 문제들의 정확한 해답이었다. 하지만 발표 슬라이드와 실제 이식 사이에는 **물리적 장벽**이 7개 있다. 이 글은 그 장벽을 전부 박제해서, 다음에 같은 마이그레이션을 할 사람이 같은 삽질을 안 하게 만드는 게 목표다.

Journal 003(`.build/` 130GB `.claudeignore` 차단)이 "컨텍스트 위생"이었고, Journal 006(SPM 순환 해체)이 "그래프 수술"이었다면, 이 글은 그 연장선의 끝 — **빌드 시스템 자체를 SPM에서 Tuist로 전환**하고 Microfeatures 패턴을 얹는 작업이다.

## Microfeatures 5계층 구조

```
[Feature]Interface   ← 프로토콜 계약만 (Buildable/Listener/Entity)
    ↓
[Feature]            ← 실제 구현 (외부 비공개)
    ↓
[Feature]Testing     ← Mock 구현체 (테스트/Example용)
    ↓
[Feature]Tests       ← XCTest 코드
    ↓
[Feature]Example     ← 독립 실행 미니앱
```

**핵심 규칙**: 같은 계층의 Feature는 서로의 구현을 절대 import하지 않는다. **Interface만** 참조.

## WO-005: Interface 모듈 분리 — 의존성 방향 뒤집기

**Before** — Presentation 직접 참조:
```swift
// OdaPresentation/OdaBuilder.swift (위반)
import OnewayPresentation  // ❌ 구현 전체를 끌어옴

class OdaComponent: Component<OdaDependency> {
  private let builder = OnewayReturnSearchBuilder()  // ❌ 구체 타입 직접 생성
}
```

```
OdaPresentation ──import──→ OnewayPresentation (구현 전체)
                             ↑ 필요한 건 프로토콜 2개뿐
```

이게 왜 위험한가:
1. 한 Feature 수정이 무관한 Feature 재빌드를 유발
2. 모듈 그래프가 조금만 복잡해지면 순환 의존 발생
3. Tuist 전환 시 transitive 전파가 끊기면서 컴파일 에러 폭탄

**After** — Interface 프로토콜 주입:
```swift
// OdaPresentation/OdaBuilder.swift
import OnewayPresentationInterface  // ✅ 프로토콜만

public protocol OdaDependency: Dependency {
  var onewayReturnSearchBuildable: any OnewayReturnSearchBuildable { get }  // ✅ 프로토콜
  var onewayReturnLocationBuildable: any OnewayReturnLocationBuildable { get }
}

final class OdaRouter: ViewableRouter<OdaInteractable, OdaViewControllable> {
  private let searchBuilder: any OnewayReturnSearchBuildable
  private var searchRouting: ViewableRouting?  // ✅ 구체 타입 아닌 프로토콜
}
```

`OnewayPresentationInterface` 모듈은 **3개 파일만** 담는다:
- `ReturnZoneType.swift` — 순수 enum (UIKit 없음)
- `OnewayReturnSearchInterface.swift` — `Buildable` + `Listener` 프로토콜
- `OnewayReturnLocationInterface.swift` — `Buildable` + `Listener` 프로토콜

**핵심 규칙**: Interface는 **프로토콜 + 값 타입만**. UIKit extension은 원본 모듈에 남긴다.

부모 Component가 concrete builder를 주입:
```swift
// TransportComponent.swift (부모)
extension TransportComponent: OdaDependency {
  var onewayReturnSearchBuildable: any OnewayReturnSearchBuildable {
    OnewayReturnSearchBuilderForOda(dependency: self)  // ✅ 구체 타입은 부모만 안다
  }
}
```

각 Wave 끝에 `tuist generate --no-open` (10초)으로 manifest 오류 smoke test 통과 후 커밋. **Wave 단위 배치 커밋이 토큰 예산도 관리**해준다 — 한 Wave가 하나의 컨텍스트 청크.

### 핵심 산출물

```
Project.swift              ← 메인 앱 타겟 (4 빌드 config)
Workspace.swift            ← 56개 모듈 등록
Tuist/
  ├── ProjectDescriptionHelpers/
  │   └── ModuleSettings.swift   ← 공통 빌드 설정 + 헬퍼
  └── Templates/
      ├── feature/               ← 새 Feature 모듈 scaffold
      └── example/               ← Example 앱 scaffold
```

`ModuleSettings.swift` 헬퍼로 모듈 선언 1줄:
```swift
ModuleTarget.framework(name: "OnewayPresentation", dependencies: [...])
ModuleTarget.exampleApp(exampleName: "Oneway", featureTarget: "OnewayPresentation")
```

### `productTypes` — 전부 framework는 틀렸다

```swift
// ❌ 이렇게 하면 Kakao/Firebase/Braze가 깨진다
productTypes: [
  "KakaoSDK": .framework,  // sub-target 간 상호의존 → symbol 누락
]

// ✅ 선택적 적용: 단일 target만 framework
productTypes: [
  "RIBs": .framework,
  "RxSwift": .framework,
  "SnapKit": .framework,
  // KakaoSDK, Firebase, Braze → static 유지 (주석으로 이유 명시)
]
```

**결정 트리**:
1. 단일 target이고 sub-product 없음 → `.framework` 추가
2. 내부 다중 target SDK → static 유지 + 소비자에서 `.external(name:)` 명시
3. 애매하면 static으로 두고 빌드 확인 후 승격

## 빌드 체인 복구 — scan-imports 자동화

### 문제: Transitive Dependency 폭탄

SPM 시절에는 의존성이 transitive로 전파됐다. 모듈 A가 모듈 B를 import하면, B의 의존성 C도 자동으로 보였다. **Tuist 전환 후 이 자동 전파가 끊겼다.** 결과: Presentation 레이어 ~20개 모듈에서 `extension method link 실패` 에러 폭탄.

### 해결: 2단계 자동화

**1단계: `scan-imports.py`** (228줄) — Swift 소스의 `import` 구문과 `Project.swift`의 declared dependencies를 비교해서 **차집합 계산**.

```bash
python3 scripts/scan-imports.py --all --report
# 출력: 모듈별 누락 dependency + .project()/.external() 라인 제안
```

내부 인덱스는 `Package.swift.bak`의 `.library` + `Project.swift`의 `ModuleTarget.framework name`. 외부 인덱스는 `Tuist/Package.swift`의 `productTypes` 키 + `.product(name:)`.

**2단계: `patch-project-deps.py`** (148줄) — 스캐너 리포트를 받아서 각 모듈의 `Project.swift` `dependencies` 배열 끝에 라인 자동 삽입.

```bash
python3 scripts/scan-imports.py --report | python3 scripts/patch-project-deps.py
# 28개 모듈 일괄 패치 — 수동이었으면 모듈당 5분, 총 1.5시간+
```

### 한계: implicit extension은 못 잡는다

```swift
// AccountDomain/Sources/SomeEntity.swift
let formatted = Date().toyyyyMMddHHmmHypen  // CoreUtil extension
// ↑ import CoreUtil 문이 없다! 스캐너 감지 불가
```

SPM transitive로 "보이던" extension이 Tuist 전환 후 끊기면서 `undefined symbol`. 소스에 import가 없으니 스캐너가 잡을 수 없다. **7개 Domain 모듈에 CoreUtil을 수동 보강** — 빌드 에러를 보고 하나씩.

### Build Chain Status (2차 세션 시점)

```
Network → Repository → Domain → BusinessWebView → Account → Menu ✅
                                                                  │
                                              OnewayPresentation ──┘ ❌ (Routing 미이전)
```

## WO-008: Routing 프로토콜 Interface 추출

WO-005에서 Buildable/Listener만 Interface로 이전했다. **Routing 프로토콜은 남겨뒀다.** 이게 빌드 체인의 마지막 벽이 됐다.

```swift
// Transaction/Sources/.../UsageHistoryDetailRouter.swift
private var onewayReturnLocationRouting: OnewayReturnLocationRouting?
// error: cannot find type 'OnewayReturnLocationRouting' in scope
```

Transaction이 `OnewayReturnLocationRouting`을 쓰려면 OnewayPresentation을 직접 import해야 하는데, 그러면 **순환 의존**. SPM 시절에는 transitive로 보였지만 Tuist가 그 환상을 깨뜨렸다.

**판별 기준 — 어떤 Routing을 이전하나?**

```bash
grep -rn "OnewayReturn.*Routing" --include="*.swift" \
    Transaction/ Smartkey/ OdaLocationConfirmPresentation/ OdaPresentation/
```

외부 참조가 있으면 → Interface 이전 대상. `OnewayReturnLocationRouting` + `OnewayReturnSearchRouting` 2개 이전 완료.

이후 transitive dependency 누락 28개 모듈 일괄 패치. Firebase 초기화 크래시 8커밋에 걸쳐 해소 (`@main` ↔ `main.swift` 전환, ObjC category 링킹 `-ObjC` 플래그, 시뮬레이터 분기).

### 그림자 의존성 감사

스캐너가 `UNRESOLVED`로 남긴 항목들을 하나씩 추적:

- **FloatingPanel**: `Package.swift.bak`에도 `Tuist/Package.swift`에도 없었음. OnewayPresentation이 실제 사용 중 → 등록
- **netfox / FLEX**: 메인 앱 Xcode SPM 전용 디버깅 도구 → 등록 불필요
- **adrop-ads-backfill**: 코드에서 실제 import 없음 → 등록 불필요

**"그림자 의존성"** = 원본 SPM 매니페스트에 선언 없이 `Package.resolved`에만 존재하는 패키지. 스캐너의 `UNRESOLVED` 보고가 이 그림자들을 수면 위로 올려준다.

### Scaffold 템플릿 기본 의존성 보강

implicit extension 누락 문제를 **예방 인프라**로 해결:

```stencil
// Tuist/Templates/feature/Project.stencil
// Feature 타겟 기본 deps
.external(name: "RIBs"),
.external(name: "ReactorKit"),
.external(name: "RxSwift"),
.external(name: "RxCocoa"),
.project(target: "CoreUtil", path: "../Util"),
.project(target: "Log", path: "../Util"),
.project(target: "Persistent", path: "../Persistent"),
```

`tuist scaffold feature --name X`로 새 모듈을 만들면 **CoreUtil/Log/Persistent가 처음부터 포함**. implicit extension 빌드 에러 원천 차단.

## 함정 7종 박제

### 1. Tuist 버전 체크 지연

4.27로 시작 → `targets[3].settings[0].name missing` 에러로 1시간 삽질. `brew info tuist`로 4.185.1이 최신인 걸 확인하고 업그레이드하니 해소.

**규칙**: 마이그레이션 첫 명령어는 `brew info <tool> && <tool> version`.

### 2. productTypes 전역 적용 불가

복합 SDK(Kakao/Firebase/Braze)의 sub-target 간 상호의존 때문에 전역 `.framework`는 불가. **선택적 적용 + 제외 이유 주석** 패턴.

### 3. Swift Framework Transitive Extension

SPM에서 Tuist로 전환하면 transitive extension 전파가 끊긴다. `import` 없이 쓰던 extension이 갑자기 `undefined symbol`. 스캐너로 잡히지 않아 빌드 에러 기반 수동 보강 필요.

### 4. Interface 추출 범위 오판

Buildable/Listener만 이전하고 Routing은 남겨뒀다가 빌드 체인이 멈춤. **Interface 추출 기준은 "외부 참조되는 모든 public protocol"**이어야 한다. grep으로 외부 참조 전수조사 후 한 번에 이전.

### 5. SPM/Tuist Package.swift 충돌

같은 디렉토리에 `Package.swift`가 있으면 Tuist generate와 SPM resolve가 동시에 동작해서 워크스페이스가 꼬인다. `.bak` rename으로 회피.

### 6. 자동 패처 멀티타겟 skip

`patch-project-deps.py`가 멀티타겟 모듈에서 서브타겟 경계를 추론 못해 skip. FreeOneWay 하나에 4개 서브타겟 × 별도 deps = 38개 추가. "80% 자동화 + 20% 수동"이 현실적 기대치.

### 7. 그림자 의존성 — Package.resolved에만 있는 유령

`FloatingPanel`이 실제로 사용되는데, 원본 `Package.swift`에도 선언이 없었다. `Package.resolved`에만 존재하는 "그림자". 스캐너의 `UNRESOLVED` 보고가 수동 추적의 시작점.

## 교훈

1. **인제스트 → WO → 실행의 3단 파이프라인.** 토스 SLASH23 블로그를 `/ingest`로 분석 → 6개 패턴 추출 → 4개 WO 발행 → 하루 스프린트 완주. 외부 지식을 실행 가능한 Work Order로 변환하는 파이프라인이 핵심.

2. **Interface 먼저, Tuist는 그 위에.** WO-005(Interface 분리)를 WO-006(Tuist) 이전에 배치한 것이 정확한 순서. Interface 패턴이 확립된 상태에서 `Project.swift`에 의존성을 올바르게 선언할 수 있었다.

3. **L형 WO는 Wave 분할 필수.** 58개 모듈 대상의 마이그레이션을 Wave 없이 진행하면 중간 빌드 실패에서 복구 불가능. 리프-업 순서가 "한 모듈 고치면 체인이 한 단계 전진"하는 리듬을 만든다.

4. **productTypes는 패키지 단위지만 문제는 타겟 단위다.** 복합 SDK의 sub-target 간 상호의존은 dynamic framework 전환으로 깨진다. 선택적 적용 + 제외 이유 주석이 유지보수 가능한 패턴.

5. **Interface 추출 기준은 "외부 참조되는 모든 public protocol"이다.** Buildable/Listener만 이전하고 Routing을 남기면 빌드 체인이 멈춘다. grep으로 외부 참조를 전수조사한 뒤 한 번에 이전.

6. **자동화 스캐너의 한계를 정직하게 박제한다.** scan-imports.py는 explicit import만 잡는다. implicit extension은 빌드 에러로만 발견된다. "100% 자동화"가 아니라 "80% 자동화 + 20% 수동"이 현실적 기대치.

7. **인프라 마이그레이션은 멀티 에이전트보다 단일 세션이 맞다.** "동일 작업 46회 반복"은 분산의 오버헤드가 이득보다 크다. Worker Pool의 진짜 무대는 다중 레이어 동시 작업(피쳐 개발)이다.

8. **발표 슬라이드와 이식 사이에는 물리적 장벽이 있다.** Microfeatures 패턴은 설계로는 깔끔하지만, 40+ 모듈 레거시에 이식할 때 transitive extension, productTypes granularity, Package.swift 충돌 같은 빌드 시스템 레벨의 장벽이 있다. 이 장벽을 박제하는 게 실전 가치.

9. **scaffold 템플릿에 기본 의존성을 넣는 게 예방 인프라다.** implicit extension 문제를 "발생 후 수동 보강"에서 "발생 자체를 차단"으로 전환.

10. **Firebase 초기화는 8커밋의 삽질.** Tuist 전환과 무관한 **런타임 문제가 마이그레이션 완료를 가로막는 전형적 패턴**. `@main` ↔ `main.swift` 전환, ObjC category 링킹, 시뮬레이터 분기까지.
