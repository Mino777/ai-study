## 핵심 개념

프롬프트 엔지니어링은 "모델에게 잘 물어보는 기술"이다. Fine-tuning은 한 단계 더 나아가 **모델 자체를 내 데이터와 도메인에 맞추는 기술**이다.

```
프롬프트 엔지니어링: "이런 형식으로 답해줘" → 매번 지시
Fine-tuning: "이런 형식이 기본이야" → 한 번 학습, 이후 자동
```

## 언제 Fine-tuning이 필요한가

| 상황 | 프롬프트로 충분 | Fine-tuning 필요 |
|------|---------------|-----------------|
| 일반 QA | O | X |
| 특정 톤/스타일 | △ (프롬프트 길어짐) | O |
| 도메인 전문 용어 | △ (few-shot 한계) | O |
| 응답 형식 고정 | O (structured output) | △ |
| 지연 시간 줄이기 | X | O (작은 모델 사용 가능) |
| 비용 절감 | X | O (작은 모델로 대체) |

**판단 기준:** 프롬프트에 매번 10줄 이상의 지시를 반복하고 있다면, fine-tuning을 고려할 때다.

## Fine-tuning 방식 비교

### 1. Full Fine-tuning

모델의 모든 파라미터를 업데이트. 가장 강력하지만 비용이 크다.

```
GPT-4 급 모델: 수백 GPU 시간, 수만 달러
7B 모델: 8× A100 GPU, 수 시간
```

### 2. LoRA (Low-Rank Adaptation)

모델 가중치를 직접 수정하지 않고, **작은 어댑터 행렬**만 학습한다.

```python
# 개념적 설명
# 원래 가중치 W (변경 안 함) + 작은 행렬 A, B (학습)
# 출력 = W·x + A·B·x
# A: (d, r), B: (r, d) — r은 rank (보통 8~64)
```

장점:
- 원본 모델의 0.1~1% 파라미터만 학습
- GPU 메모리 대폭 절감
- 여러 LoRA 어댑터를 스왑 가능 (용도별 전환)

### 3. QLoRA (Quantized LoRA)

LoRA + 4비트 양자화. 16GB GPU 하나로 7B 모델 fine-tuning 가능.

```
7B 모델 Full: 28GB+ VRAM
7B 모델 LoRA: 16GB VRAM
7B 모델 QLoRA: 6GB VRAM ← 일반 GPU로 가능
```

## 데이터 준비가 핵심

Fine-tuning의 80%는 데이터 준비다. 모델 학습은 20%.

```typescript
// SFT (Supervised Fine-Tuning) 데이터 형식
interface TrainingExample {
  messages: [
    { role: "system", content: string },
    { role: "user", content: string },
    { role: "assistant", content: string },  // 이것이 정답
  ];
}

// 최소 100개, 이상적으로 1000개 이상
// 품질 > 양 — 잘못된 예시 10개가 좋은 예시 100개를 망친다
```

**데이터 품질 체크리스트:**
- 정답이 진짜 정답인가? (전문가 검증)
- 다양한 케이스를 커버하는가?
- 엣지 케이스가 포함되어 있는가?
- 편향이 있지 않은가?

## Eval이 없으면 Fine-tuning은 도박

학습 전후를 비교할 Eval 세트가 반드시 필요하다.

```
1. Eval 세트 준비 (학습 데이터와 겹치면 안 됨)
2. Base 모델로 Eval 실행 → 기준 점수
3. Fine-tuning 실행
4. Fine-tuned 모델로 Eval 실행 → 비교
5. 점수가 올랐으면 성공, 내렸으면 데이터/하이퍼파라미터 재검토
```

## AI Agent Directive

**Trigger**: 프롬프트만으로는 응답 품질 · 톤 · 형식이 일정하지 않을 때 / "매번 10줄 이상의 지시를 반복하고 있다" / 응답 지연이나 비용을 줄이고 싶을 때
**Prerequisites**: [Few-shot Prompting](/wiki/prompt-engineering/few-shot-prompting) — Fine-tuning 전에 시도할 대안 / [LLM Evaluation Fundamentals](/wiki/evaluation/llm-evaluation-fundamentals) — 학습 전후 품질 측정 방법

### Actionable Steps
1. **프롬프트 엔지니어링으로 충분한지 먼저 검증** — few-shot · structured output · 틀 강제 시도 (Fine-tuning은 마지막 수단)
2. **고품질 데이터 수집** — 최소 100개, 이상적으로 1000개 이상의 (user 입력, 정답) 쌍 · 전문가 검증 필수
3. **데이터 품질 체크** — 다양한 케이스 · 엣지 케이스 · 편향 검토 · 정답이 진짜 정답인지 재확인
4. **Eval 세트 분리** — 학습 데이터와 겹치지 않는 평가 데이터 준비 (최소 20개 이상)
5. **Base 모델 평가** — Fine-tuning 전에 Eval 세트로 기준 점수 측정
6. **LoRA/QLoRA로 시작** — Full fine-tuning은 비싸므로 7B 모델 + QLoRA로 시작
7. **Fine-tuning 후 재평가** — 같은 Eval 세트로 점수 비교 · 성능 향상 확인
8. **점수 향상 없으면 다시** — 데이터 품질 · 하이퍼파라미터 · 모델 크기 재검토

### Anti-patterns
- **프롬프트 없이 Fine-tuning** — 프롬프트 먼저 시도. 90%의 경우 충분함
- **데이터 없이 시작** — 100개 미만의 데이터로 학습하면 과적합. 고품질 데이터가 first-class
- **Eval 세트 없이 진행** — 개선이 있는지 알 수 없음. 도박이 됨
- **한 번에 전체 파라미터 fine-tune** — Full fine-tuning은 비쌈. LoRA/QLoRA로 충분한 경우 90%
- **학습 데이터와 평가 데이터 혼합** — overfitting 측정 불가
- **점수 향상 없어도 배포** — A/B 테스트 필수. Base 모델보다 못하면 선택지 오류
