## 핵심 개념

Vector Database는 고차원 벡터(임베딩)를 저장하고 **유사도 기반 검색**을 수행하는 전문 데이터베이스다. RAG 파이프라인에서 검색 단계의 핵심 인프라.

## 왜 일반 DB가 아닌 Vector DB인가

일반 DB는 exact match에 최적화되어 있다. "이 벡터와 가장 비슷한 벡터 100개를 찾아라"는 질의에는 전혀 다른 인덱싱 전략이 필요하다. ANN(Approximate Nearest Neighbor) 알고리즘이 핵심.

## 주요 ANN 알고리즘

- **HNSW** (Hierarchical Navigable Small World): 가장 널리 쓰임. 높은 recall + 빠른 검색.
- **IVF** (Inverted File Index): 클러스터 기반. 대규모 데이터셋에 적합.
- **PQ** (Product Quantization): 메모리 효율적. 정확도 트레이드오프.

## 주요 제품 비교

| 제품 | 타입 | 특징 |
|------|------|------|
| Pinecone | Managed | 가장 쉬운 시작, 서버리스 옵션 |
| Weaviate | Self-hosted/Cloud | 하이브리드 검색 (벡터+키워드) |
| Chroma | Embedded | 로컬 개발에 최적, Python 네이티브 |
| Qdrant | Self-hosted/Cloud | Rust 기반, 높은 성능 |
| pgvector | Extension | PostgreSQL 확장, 기존 스택 활용 |

## 선택 기준

1. **규모**: 백만 벡터 이하면 pgvector로 충분
2. **운영 부담**: 관리형 원하면 Pinecone, 직접 운영이면 Qdrant
3. **하이브리드 검색**: 벡터+키워드 필요하면 Weaviate
4. **프로토타입**: Chroma로 시작, 필요 시 마이그레이션

## AI Agent Directive

**Trigger**: RAG 파이프라인에서 **벡터 임베딩 저장 및 유사도 검색**을 구축해야 할 때. 특히 대규모 문서 집합(수천 개 이상)에서 빠른 검색이 필요한 경우.

**Prerequisites**:
- [rag/vector-search-basics](/wiki/rag/vector-search-basics) — ANN 알고리즘과 임베딩 기초
- [rag/rag-overview](/wiki/rag/rag-overview) — RAG 파이프라인 전체 흐름

### Actionable Steps
1. **프로젝트 규모 파악**: 저장할 벡터 개수 추정. 1,000개 미만이면 pgvector도 충분 (numpy brute-force 사용 가능). 100만 개 이상이면 Qdrant 또는 Pinecone 검토
2. **관리형 vs 자관형 결정**: 운영 부담을 피하고 빠르게 시작하려면 Pinecone/Weaviate 클라우드. 데이터 프라이버시/비용 중심이면 Qdrant 또는 pgvector 직접 운영
3. **Vector DB 선택**: 
   - **빠른 프로토타입**: Chroma (로컬, Python 네이티브)
   - **프로덕션 서버리스**: Pinecone
   - **하이브리드 검색 필요**: Weaviate (벡터 + 키워드 동시)
   - **높은 성능 + 자관**: Qdrant (Rust 기반)
4. **ANN 인덱스 선택**: HNSW(기본, 균형잡힘) > IVF(대규모) > LSH(레거시). 특별한 이유 없으면 HNSW로 시작
5. **메타데이터 필터링 설정**: 벡터 검색 후 문서 출처, 날짜 등의 메타데이터로 필터링 가능하게 설계
6. **검색 파라미터 튜닝**: 
   - `efConstruction` (인덱싱 시 그래프 연결 수): 높을수록 품질 ↑ / 시간 ↑
   - `efSearch` (검색 시 탐색 깊이): 높을수록 정확도 ↑ / 지연 ↑
7. **모니터링 & 평가**: Recall@K 측정 (정답이 상위 K 안에?). 부족하면 임베딩 모델 교체 또는 reranker 추가

### Anti-patterns
- 작은 규모 프로젝트에 과도한 Vector DB 도입 (운영 복잡성 증가, 비용 낭비)
- Vector DB 만들고 검색 정확도 평가 없이 사용 (쓸모없는 인덱스)
- 메타데이터 필터링 미지원 Vector DB 선택 (범위 기반 쿼리 불가)
- ANN 파라미터를 기본값 그대로 사용 (도메인 특성에 맞지 않음)
