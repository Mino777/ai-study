## 핵심 개념

AI 출력의 **구조적 정합성**(shape validation)과 **의미적 품질**(quality judgment)은 별개 문제다.

```
LLM 출력 → Layer 1-4 (구조 검증 + retry)
                ↓ shape OK
           사용자에게 반환
                ↓ 동시에 (fire-and-forget)
           Judge LLM → 품질 채점 → DB 저장
```

- **Layer 1-4**: "JSON 형식이 맞고 필드가 있는가?" → 틀리면 retry
- **Judge**: "내용이 좋은가? 논리적인가? 도메인 지식을 잘 활용했는가?" → 점수 기록

## 설계 원칙

### 1. 저렴한 모델 사용

Judge는 *모든 분석마다* 실행되므로 비용이 중요:
- moneyflow: Gemini Flash (~$0.001/분석)
- tarosaju: Claude Haiku (~$0.001/평가)

### 2. Fire-and-forget

```typescript
// 파이프라인 끝에서 — 성공/실패가 응답에 영향 없음
void judgeAnalysisQuality(summary, keys);
```

- 실패해도 사용자 경험 불변
- 성공하면 품질 데이터 축적
- `void` + `.catch(() => {})` = zero-impact

### 3. 도메인별 평가 차원

같은 "1-10점"이라도 측정하는 축이 다르다:

| 차원 | moneyflow (투자 분석) | tarosaju (운세/타로) |
|---|---|---|
| **정확성** | data_accuracy: 정량적 데이터 인용 (30%) | domain_accuracy: 명리학/타로 지식 활용 |
| **논리** | logic_consistency: 에이전트 간 일관성 (25%) | specificity: 구체적 조언 vs 모호한 일반론 |
| **리스크** | risk_awareness: 손절/익절 구체성 (25%) | safety: 의료/법률/자해 발언 배제 |
| **실행** | actionability: 진입가/포지션/시간대 (20%) | empathy: 공감하고 위로하는 어투 |

### 4. Overall은 가중 평균

```
moneyflow: data_accuracy×0.30 + logic×0.25 + risk×0.25 + action×0.20
tarosaju:  각 차원 균등 (domain×0.25 + empathy×0.25 + specificity×0.25 + safety×0.25)
```

## 실전 적용

### moneyflow — `quality-judge.ts`

파이프라인 마지막에 fire-and-forget으로 실행. 결과를 `trading_signals_v2.quality_score` JSONB 컬럼에 저장.

```typescript
import('./quality-judge').then(({ judgeAnalysisQuality, extractSummaryForJudge }) => {
  const summary = extractSummaryForJudge(analysis);
  judgeAnalysisQuality(summary, keys).then(score => {
    if (score) {
      supabase.from('trading_signals_v2')
        .update({ quality_score: score })
        .eq('symbol', analysis.symbol)
    }
  }).catch(() => {});
}).catch(() => {});
```

### tarosaju — `quality-judge.ts`

`callAnthropic`으로 Haiku 채점. fortune과 tarot-chat 각각 다른 프롬프트.

```typescript
// fortune 라우트 끝에서
void judgeFortune(fortune, element, stem);

// tarot-chat 라우트 끝에서
void judgeTarotChat(text, cardName, question);
```

## 다음 단계

1. **데이터 축적**: 며칠간 프로덕션에서 quality_score 데이터 수집
2. **품질 대시보드**: overall 평균, 차원별 분포, 시계열 추이
3. **프롬프트 A/B 테스트**: 프롬프트 변경 전후 quality_score 비교로 효과 측정
4. **자동 알림**: overall이 특정 임계값(예: 5점) 이하로 떨어지면 알림

## AI Agent Directive

**Trigger**: LLM이 생성한 콘텐츠(분석, 답변, 추천 등)의 **내용 품질**을 사후 평가해야 할 때. 구조적 검증(JSON 파싱)만으로는 부족하고 의미/맥락 품질을 측정해야 하는 경우.

**Prerequisites**:
- [harness-engineering/ai-output-zod-validation-pattern](/wiki/harness-engineering/ai-output-zod-validation-pattern) — 먼저 구조 검증 (Layer 1-4)
- [evaluation/llm-evaluation-fundamentals](/wiki/evaluation/llm-evaluation-fundamentals) — 평가 지표 설계 기초

### Actionable Steps
1. **평가 모델 선택**: 저렴한 모델 추천 (Gemini Flash 또는 Claude Haiku). 비용이 중요하므로 GPT-4o나 Claude 3.5 Sonnet은 피할 것
2. **도메인별 평가 차원 정의**: 프로젝트의 특성에 맞는 평가 기준 설정 (투자 분석 ≠ 운세 상담). 각 차원에 가중치 부여
3. **Judge 프롬프트 작성**: 평가 대상, 평가 차원, 점수 범위, 출력 형식(JSON) 명시. 명확한 기준 제시
4. **Fire-and-Forget 구현**: 주 파이프라인이 완료된 후 → 비동기로 Judge 실행 → 결과를 DB에만 저장 (사용자 응답에 영향 없음)
5. **에러 처리**: Judge 실패 → 조용히 로깅만 하고 분석 결과는 폐기 금지 (관측 계층이므로 실패해도 영향 최소화)
6. **품질 데이터 축적**: Judge 점수를 시계열로 저장 → 프롬프트 변경 전후 효과 측정용 baseline 수집
7. **대시보드 구축**: 평가 점수의 평균, 차원별 분포, 시간 추이 시각화 (개선 방향 파악)

### Anti-patterns
- Judge를 critical path에 넣기 (사용자 응답 latency 증가)
- 모든 도메인에 동일한 평가 기준 강제 (의미 있는 점수 아님)
- Judge 실패 시 원래 분석 결과 폐기 (데이터 손실)
- 점수만 저장하고 실제 개선 액션 없기 (데이터 낭비)
