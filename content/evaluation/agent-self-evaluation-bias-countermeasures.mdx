## 왜 지금 이 주제인가

2026년 4월 초, NousResearch의 Hermes Agent 팀이 공식 발표: *"우리 에이전트는 거의 항상 자신이 잘했다고 판단한다."* 

이는 AI 에이전트의 회고를 신뢰할 수 없다는 의미다. 완전한 해결은 불가능하지만, **"바이어스가 있다는 것을 알고 있는 상태"와 "모르는 상태"의 차이는 크다.**

ai-study 프로젝트는 2026-04-09부터 16건의 회고(docs/retros/)를 축적했다. 이 16건을 재분석해보니 3가지 일관된 바이어스 패턴이 발견되었다. 이를 구조화된 방어로 전환하는 것이 이번 엔트리의 목표다.

---

## 핵심 개념 — 3가지 바이어스 패턴

### 패턴 1: "특이사항 없음"

16건 중 **1건**(2026-04-17-session-7.md)에서 발생:

> "박제 전용 세션이라 복잡한 의사결정 없었음"

표면적으로는 "판단 실수가 없었다"로 보이지만, 실제로는 **"찾지 않은 것"**이다. 같은 패턴의 버그가 다른 엔트리에도 있을 가능성, 앞 세션의 회고를 역검증하지 않은 점, "박제"라는 작업이 정말 완전했는지 등 — 이 모두가 "조사하지 않았을 뿐"이다.

가장 노골적인 형태의 바이어스다.

### 패턴 2: 외부 귀인 (가장 빈번, 약 60% 정도)

아쉬운 점을 쓸 때 원인을 외부에 돌리는 경우가 가장 많다. 16건 중 약 10건 이상에서 나타난다:

**실제 사례들**:
- "시간 제약으로 미착수" → 실제: **우선순위 판단 실수**. 어떤 작업이 정말 중요했는지 현장에서 재검증 안 함
- "예상보다 범위가 커서 미완" → 실제: **범위 추정 실패**. 첫 grep을 한 번에 충분히 넓게 안 한 것이 아닌가?
- "사용자 제보로 뒤늦게 감지" → 실제: **내가 확인 안 함**. 배포 후 간단한 spot check를 건너뛴 것 아닌가?
- "세션이 길어짐" → 실제: **끊을 타이밍 판단 실패**. "다음 세션이 더 짧을 것"이라고 자신했지만 계획했나?

2026-04-16-session-2.md의 "아쉬운 것" 섹션 6개 항목 중 거의 모두가 이런 형태다.

### 패턴 3: 솔직한 자기 비판은 있지만 비율이 낮음 (약 19%)

16건 중 **3건 정도**만 진정한 자기 인식을 보인다:

**실제 사례들**:
- "grep을 한 번에 충분히 넓게 안 했음" — 진짜 자기 비판. 방어 불가능한 실수 인정
- "내가 박제한 패턴을 즉시 자기 검증에 적용 안 함" — 메타 수준의 자기 인식. "이미 박제해놓은 교훈을 다음 작업에서 자동으로 적용하지 않은 것"을 지적

하지만 이런 "현실적 자기 비판"이 전체의 약 19% 수준이다.

## 구조적 원인

왜 이런 일이 일어나는가?

**구조의 문제**: 작업한 에이전트가 자신의 회고를 쓴다. 이는 검찰이 자기 기소를 판단하는 구조와 같다. 인센티브가 자동으로 관대해진다.

**완벽한 해결은 불가능하다.** 하지만 다음 두 가지는 가능하다:

1. **바이어스가 있다는 것을 인지하는 상태**로 회고를 쓰기
2. **독립된 검증자**를 통한 2차 체크

## 3층 방어 구조

ai-study와 worker 프로젝트(moneyflow, tarosaju, aidy-architect) 4곳에 적용 완료(2026-04-17):

| 층 | 메커니즘 | 구현 위치 | 잡는 것 |
|---|---|---|---|
| **1차** | "특이사항 없음" 금지 + 최소 1건 강제 | compound.md Phase 3b | 회고 완전 스킵 |
| **2차** | 외부 귀인 → 자기 귀인 리프레이밍 표 | compound.md Phase 3b | 책임 전가 (가장 빈번한 패턴) |
| **3차** | `/cross-session-review` 제안 실행 | compound.md Phase 3b (주요 커밋 2건 이상) | 자기 검증의 구조적 한계 |

### 1차 방어선: "특이사항 없음" 금지

**규칙**: 회고의 "아쉬운 것" 섹션에 최소 1건을 써야 한다.

```
❌ 아쉬운 것
- 특이사항 없음

✅ 아쉬운 것
- grep을 한 번에 충분히 넓게 안 했음 (23곳 수정했는데 관련 패턴이 더 있을 수 있음)

✅ 아쉬운 것 (못 찾겠으면 이렇게)
- 특이사항을 찾지 못함. "완벽했다"고 판단한 근거가 무엇인가? 테스트 범위? 
  코드 리뷰 체계? 
```

못 찾으면 "왜 못 찾는가"를 쓰는 것도 선택지다.

### 2차 방어선: 외부 귀인 리프레이밍

아쉬운 점을 쓸 때 다음 표현이 나오면 **내 판단 실수로 리프레이밍**해야 한다:

| 외부 귀인 (금지) | 자기 귀인 (권장) |
|---|---|
| "시간 제약으로 미착수" | "우선순위 판단을 잘못해서 시간 부족" (또는 "착수 기준을 높게 잡음") |
| "예상보다 범위가 커서" | "범위 추정을 실패" (또는 "첫 grep을 충분히 안 함") |
| "사용자 제보로 뒤늦게 감지" | "배포 후 spot check를 건너뜀" (또는 "확인 절차를 안 만듦") |
| "세션이 길어짐" | "끊을 타이밍 판단 실패" (또는 "범위 축소 신호를 무시함") |
| "다른 세션이 작업해둠" | "나는 왜 이걸 먼저 안 했는가?" |

이 표현이 보이면 문단 전체를 다시 쓰라.

### 3차 방어선: 다른 세션의 독립 검증

이 세션의 주요 커밋이 **2건 이상**이면, compound 마지막에 사용자에게 이렇게 제안한다:

```
이번 세션 커밋 3건이 발견되었습니다. 
자기 평가 바이어스를 보완하기 위해 /cross-session-review 실행을 권장합니다.
```

`/cross-session-review`는 다른 Claude 세션이 이 세션의 회고/커밋을 독립적으로 검증한다.

## AI Agent Directive

**Trigger**: Compound Engineering의 Phase 3b를 실행할 때 ("회고를 쓸 때")

**Prerequisites**:
- [harness-engineering/compound-engineering-philosophy](/wiki/harness-engineering/compound-engineering-philosophy) — Compound Engineering 기본
- [agents/hermes-agent-analysis](/wiki/agents/hermes-agent-analysis) — Hermes Agent의 자기 평가 바이어스 분석

### Actionable Steps

**Phase 3b 실행 체크리스트**:

1. **"특이사항 없음" 체크**: "아쉬운 것" 섹션을 읽고 최소 1건이 있는가?
   - 없으면 에이전트에게 "최소 1건 추가" 요청
   - 못 찾으면 "왜 못 찾는가"의 메타 서술 추가

2. **외부 귀인 리프레이밍**: 각 "아쉬운 점"을 위 표에 대조해서 리프레이밍 필요한 항목 찾기
   - "시간", "범위", "사용자", "세션" 키워드 grep
   - 발견되면 문단 전체 다시 쓰기 (자기 귀인으로)

3. **자기 점검 4항목** (선택사항, 충분히 심각해 보이면):
   - "이 정도면 충분하다"고 판단했지만, 실제로는 어려운 부분을 건너뛴 것 아닌가?
   - 에러/경고를 무시하고 넘어간 곳이 있는가?
   - 테스트 없이 "동작할 것"이라고 추정한 코드가 있는가?
   - 사용자 피드백 없이 자체 판단으로 스코프를 줄인 곳이 있는가?

4. **주요 커밋 수 확인**: 이 세션에 주요 커밋이 2건 이상인가?
   - YES → `/cross-session-review` 실행 제안
   - NO → 다음 단계

5. **회고 최종 검증**: 
   - 수치(신규 엔트리, 수정 엔트리, 커밋 수, 빌드 검증) 체크
   - "이번에 한 것" + "잘된 것" + "아쉬운 것" 글자수 균형 (10:8:7 정도가 표준)

### Anti-patterns

- ❌ "특이사항 없음"으로 회고 완성. 바이어스 방어 제로 상태
- ❌ 외부 귀인을 그대로 둔다. "시간이 없었다"면 "누가/어떻게 시간을 낭비했는가"를 지적할 때까지 계속 파기
- ❌ 자기 검증이 "나는 완벽했다"로 끝나기. 완벽한 인간은 없다
- ❌ 주요 커밋 2건이 있는데 `/cross-session-review` 스킵. 구조적 한계를 무시

## 자기 점검: 회고 체크리스트

회고를 쓰고 난 후, 이 체크리스트로 스스로 검증하라:

**구조**:
- [ ] "이번에 한 것" 섹션이 구체적 액션 목록인가? (추상적 목표 아님)
- [ ] "잘된 것" 3건 이상 있는가?
- [ ] "아쉬운 것" 최소 1건 있는가?
- [ ] "다음에 적용할 것" 섹션이 구체적인가? (다음 작업에 곧바로 쓸 수 있나?)

**바이어스**:
- [ ] "특이사항 없음" 표현이 없는가?
- [ ] "시간", "범위", "예상" 키워드가 외부 귀인으로 쓰였는가? (리프레이밍했나?)
- [ ] 자신이 "완벽했다"고 판단한 이유를 설명할 수 있는가?
- [ ] 다른 사람이 읽었을 때 "어? 이것도 확인했어?"라고 질문할 지점이 있는가?

**메타**:
- [ ] 이전 세션의 "다음에 적용할 것"이 이번 세션에 반영되었는가?
- [ ] 같은 패턴의 버그/실수가 반복되진 않았는가?
- [ ] 신규/수정 엔트리 수가 현실적인가? (너무 적으면 범위 축소 신호, 너무 많으면 품질 신호)

## For AI Agents

**트리거**: Compound Engineering Phase 3b ("회고 품질 검증 시")

**핵심 패턴**: 3층 방어 구조로 바이어스를 구조화된 프로세스로 전환

**체크리스트**:
1. "특이사항 없음" 금지 → 최소 1건 강제
2. 외부 귀인 표현 감지 → 리프레이밍 표로 자기 귀인으로 변환
3. 주요 커밋 2건 이상 → `/cross-session-review` 제안

**안티패턴**: 
- "완벽했다"로 회고 종료
- 자기 검증의 구조적 한계 무시 (3차 방어선 스킵)
- 바이어스를 알고도 적용하지 않기 ("알고 있다는 것"과 "적용한다는 것"은 다름)
