## 핵심 개념

Tool Use(Function Calling)는 LLM이 **텍스트 생성 외에 실제 행동을 수행**할 수 있게 하는 메커니즘이다. 모델이 "이 함수를 이런 인자로 호출해야겠다"고 판단하면 구조화된 함수 호출을 출력하고, 시스템이 실행하여 결과를 돌려준다.

## 동작 흐름

```
User → LLM → "search_web(query='RAG best practices')" → System executes → Result → LLM → Final response
```

1. 사용 가능한 도구 목록을 시스템 프롬프트에 정의 (이름, 설명, 파라미터 스키마)
2. LLM이 질문 분석 후 적절한 도구 선택
3. 구조화된 JSON으로 함수 호출 출력
4. 시스템이 실행하고 결과를 다시 LLM에 전달
5. LLM이 결과를 해석하여 최종 응답 생성

## 주요 프로토콜

### OpenAI Function Calling
가장 널리 쓰이는 구현. `tools` 배열로 함수 정의, `tool_calls`로 호출 결과 반환.

### Anthropic Tool Use
Claude의 구현. XML 기반 도구 정의. 병렬 도구 호출 지원.

### MCP (Model Context Protocol)
Anthropic이 제안한 표준 프로토콜. 도구, 리소스, 프롬프트를 표준화된 방식으로 LLM에 연결.

## 설계 원칙

- **도구 설명이 핵심**: LLM은 설명을 보고 도구를 선택. 설명이 모호하면 잘못된 도구 사용.
- **최소 권한**: 필요한 도구만 노출. 너무 많으면 혼란.
- **에러 핸들링**: 도구 실행 실패 시 LLM에게 에러를 전달하여 대안 탐색.
- **확인 단계**: 위험한 작업(삭제, 결제 등)은 사용자 확인 필수.

## 한계

- 모델이 불필요하게 도구를 호출하는 "tool hallucination"
- 복잡한 도구 체이닝에서의 에러 전파
- 도구 응답이 너무 크면 컨텍스트 초과

## 실전 구현 예시: MCP를 통한 도구 통합

```typescript
// Claude 모델에서 MCP (Model Context Protocol) 사용
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

const tools: Anthropic.Messages.Tool[] = [
  {
    name: "search_web",
    description: "웹에서 정보를 검색합니다. 질문에 답하기 위해 현재 정보가 필요할 때 사용하세요.",
    input_schema: {
      type: "object" as const,
      properties: {
        query: {
          type: "string",
          description: "검색 쿼리 (예: 'TypeScript generics 2024년')",
        },
      },
      required: ["query"],
    },
  },
  {
    name: "execute_code",
    description: "코드를 실행하고 결과를 반환합니다. 계산이나 코드 검증에 사용하세요.",
    input_schema: {
      type: "object" as const,
      properties: {
        language: {
          type: "string",
          enum: ["python", "javascript", "typescript"],
          description: "프로그래밍 언어",
        },
        code: {
          type: "string",
          description: "실행할 코드",
        },
      },
      required: ["language", "code"],
    },
  },
];

// 도구 호출 루프
async function agentLoop(userQuery: string) {
  const messages: Anthropic.Messages.MessageParam[] = [
    {
      role: "user",
      content: userQuery,
    },
  ];

  while (true) {
    const response = await client.messages.create({
      model: "claude-opus-4-20250805",
      max_tokens: 2048,
      tools,
      messages,
    });

    // 도구 호출이 없으면 최종 응답 반환
    if (response.stop_reason === "end_turn") {
      return response.content
        .filter((block) => block.type === "text")
        .map((block) => (block as Anthropic.Messages.TextBlock).text)
        .join("\n");
    }

    // 도구 호출 처리
    const assistantMessage: Anthropic.Messages.MessageParam = {
      role: "assistant",
      content: response.content,
    };
    messages.push(assistantMessage);

    // 각 도구 호출 실행
    const toolResults: Anthropic.Messages.ToolResultBlockParam[] = [];
    for (const block of response.content) {
      if (block.type === "tool_use") {
        let result = "";
        try {
          if (block.name === "search_web") {
            // 실제 웹 검색 실행
            result = await searchWeb((block.input as { query: string }).query);
          } else if (block.name === "execute_code") {
            // 코드 실행
            const { language, code } = block.input as {
              language: string;
              code: string;
            };
            result = await executeCode(language, code);
          }
        } catch (error) {
          result = `에러: ${error instanceof Error ? error.message : "알 수 없는 에러"}`;
        }

        toolResults.push({
          type: "tool_result",
          tool_use_id: block.id,
          content: result,
        });
      }
    }

    // 도구 결과를 메시지에 추가
    messages.push({
      role: "user",
      content: toolResults,
    });
  }
}

// 테스트
agentLoop("2024년 최신 TypeScript 기능 3가지를 알려주고, 예시 코드로 보여줘");
```

## 도구 설명 작성 가이드라인

### Good: 명확한 설명
```
name: "fetch_document"
description: "URL에서 문서를 다운로드하여 텍스트 내용을 반환합니다. 
             사용자가 특정 웹페이지나 PDF의 내용을 알고 싶을 때 사용하세요."
```

### Bad: 모호한 설명
```
name: "get_data"
description: "데이터를 가져옵니다." ← 너무 모호함
```

## AI Agent Directive

**Trigger**: LLM이 외부 API, 데이터베이스, 파일 시스템 등의 도구를 호출하여 실제 작업을 수행해야 할 때. 정보 검색/수집 또는 상태 변경이 필요한 경우.

**Prerequisites**:
- [agents/agent-architectures](/wiki/agents/agent-architectures) — 에이전트 기본 구조
- [prompt-engineering/chain-of-thought](/wiki/prompt-engineering/chain-of-thought) — 도구 선택 추론

### Actionable Steps
1. **도구 목록 정의**: 에이전트가 사용할 함수들을 이름, 설명, 파라미터 스키마로 작성 (OpenAI/Anthropic 표준 형식)
2. **도구 설명 최적화**: LLM이 설명만 보고 올바른 도구를 선택하도록. "사용자의 요청을 분석하고 어떤 도구가 필요한지 판단" 능력에 직결
3. **최소 권한 원칙**: 필요한 도구만 노출. 너무 많으면 LLM의 선택 혼란 증가
4. **에러 핸들링**: 도구 실행 실패 시 → 에러 메시지를 LLM에 다시 전달 → 대안 도구 또는 다른 접근 시도 유도
5. **도구 응답 크기 제한**: 매우 큰 응답은 요약하거나 필터링 후 LLM에 전달 (컨텍스트 초과 방지)
6. **확인 단계 추가**: 위험한 도구(삭제, 결제, 접근 제어 변경 등)는 실행 전 사용자 승인 요청
7. **도구 체이닝**: 첫 번째 도구의 결과 → 메모리 저장 → LLM 재 추론 → 다음 도구 호출 흐름 구현

### Anti-patterns
- 도구 설명이 너무 기술적이거나 모호 (LLM hallucination 유발)
- "필요하다면 뭐든 호출해" 식으로 모든 도구 노출 (도구 오용)
- 도구 실패 시 전체 파이프라인 중단 (대체 전략 없음)
- 도구 응답 전체를 필터 없이 LLM에 전달 (토큰 낭비, 컨텍스트 오염)
