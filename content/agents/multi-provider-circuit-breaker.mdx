## 갭 / 맥락

AI 프로덕트가 단일 프로바이더(예: Claude만)에 의존하면 그 프로바이더의 장애가 곧 서비스 장애다. moneyflow는 13개 에이전트가 Gemini/Claude/OpenAI 3사를 역할별로 사용하는데, 피크 시간대에 Claude 429(Rate Limit)가 10분에 3회 발생하는 상황을 겪었다.

**해결하지 않으면**: 사용자가 분석 요청 → Claude 429 → 전체 파이프라인 중단 → 사용자 이탈.

**해결한 뒤**: Claude 429 → Circuit Breaker가 Claude를 2분간 차단 → 대체 프로바이더로 자동 라우팅 → 사용자는 지연만 느낌.

## 핵심 패턴

### Circuit Breaker 상태 머신

```
CLOSED (정상)
  │ 연속 3회 실패
  ▼
OPEN (차단) ── 2분 쿨다운 ──→ HALF-OPEN (시험)
  ▲                              │
  │ 1회 실패                    │ 1회 성공
  └──────────────────────────────┘
                                  │
                                  ▼
                              CLOSED (복구)
```

### 프로바이더별 독립 상태

```typescript
// 각 프로바이더마다 독립 상태 관리
interface CircuitState {
  failures: number;      // 연속 실패 횟수
  openUntil: number;     // open 상태 해제 시각 (ms)
}

// Gemini 429 ≠ Claude 차단. 독립 관리가 핵심
const circuitStates: Record<Provider, CircuitState> = {
  gemini:  { failures: 0, openUntil: 0 },
  claude:  { failures: 0, openUntil: 0 },
  openai:  { failures: 0, openUntil: 0 },
};
```

### 호출 흐름

```typescript
async function callWithCircuitBreaker(
  provider: Provider,
  fn: () => Promise<Response>
): Promise<Response> {
  const state = circuitStates[provider];
  const now = Date.now();

  // 1. OPEN 상태 → 즉시 거부 (프로바이더에 부담 안 줌)
  if (state.openUntil > now) {
    throw new CircuitOpenError(provider);
  }

  try {
    const result = await fn();
    // 성공 → 상태 리셋 (half-open에서도 closed로 복귀)
    state.failures = 0;
    state.openUntil = 0;
    return result;
  } catch (error) {
    state.failures++;

    // threshold 도달 → OPEN 전환 (2분 쿨다운)
    if (state.failures >= 3) {
      state.openUntil = now + 120_000;
    }
    throw error;
  }
}
```

### 에이전트별 최적 프로바이더 라우팅

Circuit Breaker와 결합하여, 에이전트 역할별로 최적 프로바이더를 지정하되 장애 시 대체:

| 에이전트 역할 | 1순위 | 2순위 (fallback) |
|-------------|-------|-----------------|
| 데이터 분석 (기술/펀더멘털) | Gemini | Claude |
| 비판적 분석 (Bull/Bear 토론) | Claude | GPT |
| 종합 판단 (Judge/CIO) | Claude | Gemini |
| 토론 중재 | GPT | Claude |

## 운영 데이터

- **Claude 429 빈도**: 피크 시간대(한국 21-23시) 10분 3회 수준
- **쿨다운 설정**: 2분 (120,000ms) — 피크 빈도 기반으로 경험적 설정
- **Threshold**: 3회 연속 실패 — 일시적 네트워크 이슈와 프로바이더 장애 구분
- **장애 격리 효과**: Claude 차단 중에도 Gemini/GPT 에이전트는 정상 동작

### 비용 추적 연동

프로바이더별 호출 비용을 `ai-cost-tracker.ts`로 실시간 추적:

## AI Agent Directive

### Trigger
- 멀티 AI 프로바이더를 사용하는 프로덕트에서 장애 대응 패턴이 필요할 때
- 특정 프로바이더의 rate limit이 서비스 장애로 이어지는 상황

### Prerequisites
- [Multi-Agent Pipeline](/wiki/agents/multi-agent-pipeline)

### Actionable Steps
1. **프로바이더별 독립 CircuitState 구현** — failures 카운터 + openUntil 타임스탬프
2. **Threshold 설정** — 연속 실패 횟수 (3회 권장, 프로바이더 SLA에 따라 조정)
3. **쿨다운 설정** — 장애 빈도에 맞춰 경험적 결정 (moneyflow: 2분)
4. **Fallback 라우팅 테이블 정의** — 에이전트별 1순위/2순위 프로바이더
5. **비용 추적 연동** — 프로바이더 전환 시 비용 변동 로깅

### Anti-patterns
- **공유 Circuit Breaker** — 한 프로바이더 장애가 전체를 차단. 반드시 독립 관리
- **무한 retry** — 과부하 프로바이더에 추가 요청으로 상황 악화
- **하드코딩 프로바이더** — 장애 시 대체 경로 없음. 항상 fallback 정의
