## 갭 / 맥락

2026년 2월, 모든 주요 AI 코딩 플랫폼이 **2주 내에 동시에** 멀티 에이전트 기능을 출시했다. 단일 에이전트 시대에서 멀티 에이전트 오케스트레이션 시대로의 전환이 확정된 순간이다.

핵심 변화:
- 소프트웨어 엔지니어 역할: "코드 작성" → **"에이전트 오케스트레이션"**
- GitHub 글로벌 커밋의 4%가 AI 에이전트 저자
- 자율성보다 **조율(coherence)**이 핵심 — Mike Mason (2026)

## 핵심 개념: Planner → Worker → Judge

```
┌─────────────────────────────────────────────┐
│              Orchestrator                    │
│                                             │
│  ┌──────────┐                               │
│  │ Planner  │ 코드베이스 탐색 → 태스크 생성    │
│  └────┬─────┘                               │
│       │ 태스크 분배                           │
│  ┌────┴────┬────────┬────────┐              │
│  │Worker A │Worker B│Worker C│  (병렬 실행)   │
│  │worktree │worktree│worktree│  (격리된 환경)  │
│  └────┬────┴────┬───┴────┬───┘              │
│       │         │        │                   │
│  ┌────┴─────────┴────────┴───┐              │
│  │         Judge             │ 진행 판정      │
│  └───────────────────────────┘              │
└─────────────────────────────────────────────┘
```

### 3역할 상세

| 역할 | 책임 | 추천 모델 | 비용 비율 |
|------|------|----------|----------|
| **Planner** | 코드베이스 분석, 태스크 분해, 의존성 파악 | Opus | 높음 (5%) |
| **Worker** | 독립적 태스크 실행 (코딩, 테스트, 문서화) | Sonnet/Haiku | 낮음 (70%) |
| **Judge** | 각 사이클 종료 시 진행 판정, 품질 검증 | Opus | 높음 (25%) |

### 모델 라우팅 — 40-60% 비용 절감

```
┌─────────────────────────────────────────┐
│           Model Routing Ladder           │
│                                         │
│  Opus ($5/$25)   ← 설계, 아키텍처 리뷰   │
│  Sonnet ($3/$15) ← 대부분 운영 업무 70%   │
│  Haiku ($1/$5)   ← 분류, 추출, 포매팅    │
└─────────────────────────────────────────┘
```

Planner/Judge에 Opus, Worker에 Sonnet/Haiku를 배치하면 **품질 손실 없이 40-60% 비용 절감**.

### git worktree 격리 — 충돌 제로

멀티 에이전트의 최대 위험은 **파일 충돌**. 해결책:

```bash
# 각 Worker가 독립 worktree에서 작업
git worktree add /tmp/worker-a -b feature/task-a
git worktree add /tmp/worker-b -b feature/task-b

# 작업 완료 후 merge
git merge feature/task-a
git merge feature/task-b
```

Claude Code의 `isolation: "worktree"` 옵션이 이를 자동화한다.

## 오케스트레이션 패턴 비교

| 패턴 | 구조 | 강점 | 약점 | 적합 상황 |
|------|------|------|------|----------|
| **Sequential** | A → B → C | 단순, 디버깅 쉬움 | 느림 | 의존성 높은 파이프라인 |
| **Concurrent** | A ∥ B ∥ C | 빠름, 비용 효율 | 충돌 위험 | 독립적 태스크 |
| **Hierarchical** | Manager → Sub-managers → Workers | 대규모 확장 | 복잡성 | 100+ 파일 리팩토링 |
| **Orchestrator-Worker** | Hub → Workers | 중앙 제어 + 분산 실행 | 단일 실패점 | 중규모 프로젝트 |

## ai-study Hub-Worker 모델과의 연결

이 프로젝트의 기존 Hub-Worker 아키텍처:

```
ai-study (허브)
  ├── moneyflow (워커) — 패턴 수집, 비용 분석
  ├── tarosaju (워커) — 패턴 수집, Supabase 실전
  └── aidy-architect (워커) — 멀티에이전트 오케스트레이션
```

이것은 **Orchestrator-Worker 패턴**의 실전 구현이다:
- 허브(ai-study)가 Planner 역할: 지식 그래프 분석, 다음 작업 큐 결정
- 워커가 실행: 각 프로젝트에서 독립 작업
- `/projects-sync`가 Judge 역할: 워커 상태 진단, 충돌 사전 탐지
- `/compound`가 피드백 루프: 결과를 허브에 박제

### 진화 방향

| 현재 | 다음 단계 |
|------|----------|
| 수동 `/projects-sync` | 자동 워커 상태 폴링 |
| 단일 Claude 세션 | 역할별 병렬 에이전트 |
| Hub에서 전체 컨텍스트 | Worker에 최소 컨텍스트만 전달 |

## AI Agent Directive

### Trigger
- 프로젝트가 3+ 독립 에이전트/세션으로 분산될 때
- 단일 에이전트로 처리하기엔 태스크가 너무 클 때 (100+ 파일)
- 워커 프로젝트 간 충돌이 발생할 때

### Prerequisites
- `agents/agent-architectures` — 에이전트 기초 아키텍처
- `harness-engineering/aidy-journal-000-architect-worker-baseline` — Hub-Worker 실전
- `harness-engineering/wt-branch-structural-safety` — worktree 안전 패턴

### Actionable Steps
1. Planner/Worker/Judge 3역할 정의 — 각 역할의 입출력 계약 명시
2. Worker 간 격리 전략 결정 — git worktree 또는 별도 브랜치
3. 모델 라우팅 설정 — Planner(Opus) + Worker(Sonnet) + Judge(Opus)
4. `/projects-sync`로 워커 상태 모니터링 자동화
5. 각 사이클 종료 시 `/compound`로 결과 박제

### Anti-patterns
- 모든 에이전트에 동일 모델 사용 — 비용 낭비 (Haiku로 충분한 작업에 Opus 사용)
- Worker 간 같은 파일 동시 수정 — git worktree 격리 없이 병렬 실행
- Orchestrator 없이 에이전트 자율 실행 — 방향성 상실, 중복 작업
- 3~5개를 넘는 초기 에이전트 수 — MVP는 소규모로 시작
