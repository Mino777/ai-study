## 갭 / 맥락

멀티에이전트 시스템(Architect + 3 Worker)으로 실제 프로덕트를 11세션 운영했다.
62개 Work Order를 완주하고, 테스트 198개 → 1,500+개로 성장시키는 과정에서
**반복되는 비효율 패턴**이 누적 데이터로 드러났다.

이 엔트리는 "멀티에이전트는 어떻게 구성하나"가 아니라
**"실제로 돌려보니 어디서 시간/토큰이 새는가"**에 대한 운영 리포트다.


## 핵심 개념 — 3대 병목과 해결 패턴

### 병목 1: Architect Idle 시간 (55%)

**발견**: 10라운드 스프린트에서 Architect 시간의 55%가 `sleep` + pane 확인 루프.
가장 비싼 모델(Opus)이 절반 이상 아무 것도 안 하고 있었다.

**시간 분배 (최적화 전)**:

| 활동 | 비중 | 비고 |
|------|------|------|
| 워커 대기 (polling) | **55%** | sleep 300 반복 |
| Gate-1 검증 | 17% | 서브에이전트 3개 |
| 스펙/WO 작성 | 11% | R1, R5에만 집중 |
| 디스패치 + 체크포인트 | 5% | 자동화 가능 |
| 이슈 대응 | 5% | 크래시, FAIL 수정 |
| Compound | 5% | 마무리 |

**해결: 라운드 파이프라이닝**

```
Before:
  [서버 dispatch] ──── [idle 8분] ──── [클라 dispatch] ──── [idle 8분]

After:
  [서버 dispatch] ── [다음 스펙 작성 + WO 준비] ── [서버 완료 확인] ── [클라 dispatch + 서버 Gate-1 동시]
```

핵심 규칙:
- **워커가 일하는 동안 Architect도 반드시 다음 작업을 수행**
- 서버 구현 중 → 다음 피처 스펙 초안, WO 상세화
- 클라 구현 중 → 완료된 워커부터 즉시 Gate-1 (전원 대기 안 함)
- 서버 빌드 직접 검증은 클라 대기 중에 실행

### 병목 2: Gate-1 서브에이전트 토큰 과다

**발견**: Gate-1을 실행할 때마다 3개의 sonnet 서브에이전트를 spawn.
각각 API contract 전체(1,200줄)를 다시 읽는다. 라운드당 6만+ 토큰.

**해결: 축약 Gate-1 모드**

```
신규 엔드포인트 ≤ 3개:
  Architect가 직접 검증
  1. git diff HEAD~N 으로 변경 범위 확인
  2. 해당 엔드포인트만 api-contract.md와 대조
  3. 에러 코드 + HTTP status 확인
  4. gates/reviews/ 에 결과 기록
  → 서브에이전트 0개, 토큰 67% 절감

신규 엔드포인트 ≥ 4개 또는 복잡한 구조 변경:
  풀 Gate-1 (서브에이전트 line-by-line)
```

### 병목 3: 빌드 미검증 반복 (4세션 연속)

**발견**: "Android 빌드 직접 검증 습관화" — 이 문장이 s20, s21, s22, s23 retro에
4번 연속 등장. 매번 "다음에 하겠다"만 반복.

**근본 원인**: 습관에 의존하면 영원히 안 된다. 자동화해야 한다.

**해결: `./architect-cli.sh verify` 명령**

```bash
./architect-cli.sh verify server    # gradlew build + test + 숫자 집계
./architect-cli.sh verify android   # assembleDebug + testDebugUnitTest + 숫자 집계
./architect-cli.sh verify all       # 전체
```

`/autoceo` Step 4에서 **필수 실행**으로 규정. 습관이 아닌 프로세스로 강제.

## 라운드 구조 최적화

### Before: 10R (기능 2개)

```
R1:  스펙1       R2:  서버1     R3:  클라1      R4:  Gate-1
R5:  스펙2       R6:  서버2     R7:  클라2      R8:  Gate-1
R9:  품질        R10: Compound
```

### After: 8R (동일 산출)

```
R1: 스펙1+2 한번에 (WO 6개 발행)
R2: 서버1 + (파이프라인: 피처2 WO 상세화)
R3: 서버2 + 클라1 병렬
R4: 클라2 + Gate-1 축약
R5: Gate-1 전체 + 수정
R6: 품질 + 테스트 보강
R7: verify all + 수정
R8: Compound
```

**효과**: 동일 산출에 2라운드(~20분) 절감.

## 반복 실패 패턴 아카이브 (운영 11세션)

실제 운영에서 발견한 패턴들. 다른 프로젝트에서도 동일하게 발생할 가능성이 높다.

### 워커 자기보고 신뢰 문제

| 세션 | 사건 | 교훈 |
|------|------|------|
| s4 | iOS 워커가 10라운드 동안 "테스트 PASS" 보고 → 실제 0회 실행 | 숫자 증거 의무화 |
| s20~s23 | Android 빌드 "통과" 자기보고 → Architect 미검증 | verify 자동화 |

**원칙**: 워커의 자기보고를 신뢰하지 않는다. 숫자 증거 또는 직접 검증만 인정.

### 워커 Stall (멈춤) 감지

| 세션 | 사건 | 교훈 |
|------|------|------|
| s11 | iOS 워커가 xcodebuild 루프 10분+ → Architect는 "working" 폴링만 | pane 직접 확인 필요 |
| s23 | iOS 워커 원인 불명 크래시 → 코드는 보존, 커밋 미완 | 변경사항 확인 후 재시작 |

**4단계 프로토콜**:
1. **0~5분**: 조기 실행 확인 (tmux capture → 스피너 확인)
2. **5~10분**: 2회 연속 working → pane 직접 진단
3. **10~15분**: 워커에게 탈출 지시 ("먼저 커밋, 테스트 별도")
4. **15분+**: Architect가 직접 코드 수정

### 토큰 버스트

| 세션 | 사건 | 교훈 |
|------|------|------|
| s6 | 3-way 병렬 × 10라운드 → org-level 429 | 2-way로 축소, 5분 폴링 |

**규칙**: 최대 2-way 병렬, 5분 폴링 간격, 429 감지 시 즉시 sequential 전환.

## AI Agent Directive

### Trigger
- 멀티에이전트(Architect + N Worker) 스프린트 루프를 설계하거나 운영할 때
- `/autoceo` 또는 유사한 자동 스프린트 루프를 구현할 때
- 에이전트 오케스트레이션의 효율성을 개선할 때

### Prerequisites
- [멀티에이전트 오케스트레이션 패턴 2026](/wiki/agents/multi-agent-orchestration-patterns-2026)
- [멀티 Claude 에이전트 관리: 컴파일 전략](/wiki/agents/multi-claude-agent-management-compile-strategy)

### Actionable Steps

1. **시간 분배를 측정하라** — 체감이 아닌 실측. Architect idle 비율이 40%를 넘으면 파이프라이닝 도입.

2. **파이프라이닝 적용** — 워커 대기 중 다음 라운드의 스펙/WO를 미리 작성. 완료된 워커부터 즉시 다음 단계 진행.

3. **검증을 자동화하라** — "습관화"는 4세션 연속 실패한다. CLI 명령으로 만들어서 프로세스에 강제 삽입.

4. **축약 vs 풀 검증을 구분하라** — 작은 변경에 풀 검증은 낭비. 규모에 비례하는 검증 레벨 도입.

5. **워커 자기보고를 불신하라** — 숫자 증거(테스트 N건 통과) 또는 직접 실행만 인정. "빌드 통과"는 "테스트 통과"가 아니다.

6. **Stall detection 프로토콜을 세워라** — polling ≠ monitoring. 실제 pane/output을 주기적으로 직접 확인.

7. **라운드 구조를 압축하라** — 스펙 2개를 한 라운드에 묶고, 서버와 클라를 파이프라인으로 겹쳐서 2라운드 절감.

### Anti-patterns

- **Sleep-and-check 루프**: Architect가 `sleep 300` → pane 확인만 반복하면서 idle 시간을 낭비
- **과도한 서브에이전트**: 3개 EP 검증에 서브에이전트 3개 spawn → API contract 전체를 3번 읽기
- **습관 의존 검증**: "다음에 꼭 하겠다"는 영원히 안 된다. 자동화하거나 프로세스에 강제
- **전원 대기 패턴**: 3개 워커 모두 완료될 때까지 대기 → 1개 완료되면 즉시 다음 단계
- **3-way 병렬 burst**: 3개 워커 + Architect = 4 Claude 인스턴스 동시 → rate limit 폭발
