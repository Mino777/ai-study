## 핵심 트레이드오프: 유연성 vs. 비용

두 접근 방식의 차이는 명확합니다.

| 기준 | Computer Use (UI 조작) | 구조화된 API 호출 |
| :--- | :--- | :--- |
| **비용** | 높음 (Vision 토큰, 다단계 추론) | 낮음 (텍스트 토큰, 단일 호출) |
| **유연성** | 매우 높음 (API 없는 모든 앱/웹 조작 가능) | 낮음 (미리 정의된 API만 사용 가능) |
| **신뢰성** | 낮음 (UI 변경에 취약) | 높음 (API 명세가 바뀌지 않는 한 안정적) |
| **개발 오버헤드** | 낮음 (별도 API 개발 불필요) | 높음 (API 설계, 개발, 문서화 필요) |
| **실행 속도** | 느림 | 빠름 |

Computer Use는 GPT-4o와 같은 멀티모달 모델이 화면을 '보고' 인간처럼 클릭할 위치, 입력할 텍스트를 추론하는 방식입니다. 이 과정은 스크린샷 이미지를 토큰으로 변환하고, 복잡한 추론을 거치므로 막대한 비용과 지연 시간을 발생시킵니다. 반면, API 호출은 명확한 요청과 응답 구조를 가지므로 토큰 사용량이 적고 안정적입니다.

결국 선택의 문제는 'API가 존재하지 않는 작업을 자동화하기 위해 45배의 비용을 지불할 가치가 있는가?'로 귀결됩니다.

## "API-First, UI-Fallback" 패턴

실무적인 해답은 두 가지를 모두 사용하는 하이브리드 전략에 있습니다. 에이전트가 특정 작업을 요청받았을 때, 항상 API를 먼저 시도하고, API가 없거나 실패했을 때만 차선책으로 Computer Use를 사용하도록 설계하는 것입니다.

이 패턴은 비용 효율성과 작업 성공률 사이의 균형을 맞추는 최적의 방법입니다.

```mermaid
graph TD
    A[Agent Receives Task: "GitHub 이슈 요약"] --> B{Decision Point};
    B -- "Attempt 1: Cost-Effective" --> C["Strategy: Structured API"];
    C --> D{"`get_github_issue()` API Available?"};
    D -- Yes --> E[Execute API Call];
    E --> F([Success: Low-cost, structured data]);
    D -- No --> G["Strategy: Computer Use (Fallback)"];
    G --> H[Open github.com in Browser];
    H --> I[Visually Identify & Click Issue Link];
    I --> J[Extract Title & Body Text from UI];
    J --> K([Success: High-cost, potentially fragile]);
```

### 코드 예제 (TypeScript)

프론트엔드 환경에서 이 패턴을 구현하는 가상 시나리오입니다. 특정 작업을 수행하는 `TaskExecutor`는 API 전략을 먼저 시도하고, 실패 시 UI 조작 전략으로 넘어갑니다.

```typescript
// strategies/apiStrategy.ts
export async function fetchWithApi(taskId: string): Promise<string | null> {
  console.log(`[API Strategy] Attempting to fetch data for task: ${taskId}`);
  try {
    // 가상의 API 호출
    const response = await fetch(`https://api.internal.com/tasks/${taskId}`);
    if (!response.ok) {
      throw new Error('API endpoint not found or failed.');
    }
    const data = await response.json();
    return `API Result: ${data.summary}`;
  } catch (error) {
    console.warn('[API Strategy] Failed:', error.message);
    return null; // 실패 시 null 반환하여 다음 전략 시도 유도
  }
}

// strategies/uiStrategy.ts
// Playwright나 Puppeteer와 같은 도구를 사용한다고 가정
import { computerVisionControl } from './computerVisionControl';

export async function fetchWithUi(taskUrl: string): Promise<string> {
  console.log(`[UI Strategy] Falling back to UI interaction for URL: ${taskUrl}`);
  // 1. 브라우저 실행 및 페이지 이동
  await computerVisionControl.launchBrowserAndGoTo(taskUrl);
  // 2. 화면을 보고 '본문' 영역을 찾아 텍스트 추출 (멀티모달 LLM 호출)
  const content = await computerVisionControl.extractTextFromMainContent();
  return `UI Scraped Result: ${content.slice(0, 100)}...`;
}

// TaskExecutor.ts
import { fetchWithApi } from './strategies/apiStrategy';
import { fetchWithUi } from './strategies/uiStrategy';

class TaskExecutor {
  async execute(task: { id: string; url: string }): Promise<string> {
    // 1. API-First
    const apiResult = await fetchWithApi(task.id);
    if (apiResult) {
      return apiResult;
    }

    // 2. UI-Fallback
    const uiResult = await fetchWithUi(task.url);
    return uiResult;
  }
}

// 사용 예시
const executor = new TaskExecutor();
executor.execute({ id: '123', url: 'https://example.com/some-article' })
  .then(console.log);
```

이 코드는 `fetchWithApi`가 성공하면 즉시 결과를 반환하여 비용과 시간을 절약합니다. API 호출이 실패(e.g., 404 Not Found)해야만 비싼 `fetchWithUi` 함수가 실행됩니다.

## 프로젝트 적용 사례: `tarosaju`의 운세 콘텐츠 파싱

`tarosaju` 프로젝트는 다양한 외부 웹사이트에서 운세 콘텐츠를 가져와 사용자에게 제공해야 합니다.

- **기존 방식 (UI-Fallback Only):** 초기 버전에서는 모든 외부 사이트를 대상으로 Playwright를 이용해 UI를 스크래핑했습니다. 이는 개발 속도는 빨랐지만, 사이트 리디자인 시 파서가 깨지는 일이 잦았고, 실행 비용도 높았습니다.
- **"API-First" 패턴 적용:**
    1. **API 탐색:** 먼저 대상 웹사이트들이 콘텐츠 제공을 위한 RSS 피드나 숨겨진 JSON API를 제공하는지 분석합니다. 많은 뉴스/콘텐츠 사이트는 모바일 앱을 위해 비공식 API를 열어두는 경우가 있습니다.
    2. **API 전략 구현:** API가 존재하는 사이트에 대해서는 `fetch`를 사용한 직접적인 API 클라이언트를 구현합니다. 이는 훨씬 안정적이고 비용이 저렴합니다.
    3. **UI-Fallback 유지:** API가 없는 레거시 웹사이트에 대해서만 기존의 Playwright 기반 UI 스크래핑 로직을 유지합니다.

이러한 전환을 통해 `tarosaju` 프로젝트는 콘텐츠 수집 비용의 약 80%를 절감하고, 주요 콘텐츠 소스의 파싱 안정성을 크게 향상시킬 수 있었습니다. (이론적 추정치)

## 2026년 트렌드: 자기 수정 에이전트와 OS 통합

이러한 비용 모델은 앞으로 더욱 중요해질 것입니다. 미래의 에이전트는 API 호출이 실패했을 때 단순히 UI로 넘어가는 것을 넘어, UI를 관찰하여 변경된 API 명세를 추론하고 스스로 API 클라이언트 코드를 수정하는 '자기 수정(Self-healing)' 능력을 갖추게 될 것입니다.

또한 Apple의 Private Cloud Compute나 온디바이스 AI처럼, OS 레벨에서 앱의 의도(Intents)를 구조화된 데이터로 제공하는 기능이 보편화되면, 비싼 화면 분석(Computer Use) 대신 저렴한 시스템 API 호출로 대부분의 작업을 처리할 수 있게 될 것입니다. 개발자들은 자신의 앱이 에이전트에게 'API처럼' 보이도록 만드는 데 더 집중해야 합니다.

결론적으로, 지금은 무조건적인 UI 자동화의 환상에서 벗어나, 비용과 안정성을 고려한 영리한 상호작용 전략을 설계해야 할 때입니다. "API-First, UI-Fallback" 패턴은 그 시작을 위한 훌륭한 청사진입니다.
