## 프롬프트 필터링을 넘어서: 와이어 레벨(Wire-Level)이란 무엇인가?

와이어 레벨 액션 게이팅은 AI 에이전트가 생성한 행동(action)이 실제 시스템(Tool/API)에서 실행되기 직전, 네트워크 "와이어"를 통과하는 최종 직렬화된(serialized) 명령어를 가로채 검사하고 승인/거부하는 보안 모델입니다. 이는 LLM의 자연어 출력이나 중간 계획을 검사하는 것이 아니라, `curl`, `psql`, `kubectl`과 같이 시스템이 직접 이해하는 언어로 번역된 최종 명령어를 검증 대상으로 삼습니다.

"An agent cannot be trusted to police itself"라는 Deno의 선언처럼, 이 접근법은 에이전트 내부의 판단을 신뢰하지 않고 외부에서 독립적인 강제 계층을 두는 것이 핵심입니다.

이 다이어그램은 LLM의 추상적인 계획이 어떻게 구체적인 API 호출로 변환되고, 이 호출이 방화벽의 정책 엔진을 통과하여 최종 실행 여부가 결정되는 과정을 보여줍니다. 핵심은 C 지점, 즉 에이전트의 경계를 넘어 시스템으로 향하는 바로 그 지점에서 검증이 이루어진다는 것입니다.

### 기존 보안 모델과의 비교

와이어 레벨 게이팅의 장점은 다른 접근법과의 비교를 통해 명확해집니다.

| 접근법 | 검증 대상 | 장점 | 단점 (Trade-offs) |
| :--- | :--- | :--- | :--- |
| **프롬프트 가드레일** | LLM의 입/출력 **자연어 텍스트** | 구현이 비교적 간단하고 빠름. | 프롬프트 주입 공격에 취약하며, LLM이 생성할 최종 명령어를 정확히 예측하기 어려움. |
| **인간 참여 루프 (HITL)** | 에이전트의 **행동 의도** | 치명적인 작업에 대해 가장 높은 수준의 보안 제공. | 확장성 부재, 사용자 승인 피로, 에이전트의 자율성 저해. Claw Patrol은 이를 선택적 기능으로 제공. |
| **와이어 레벨 게이팅** | 직렬화된 **시스템 명령어/API 호출** | 결정적이고, 프로토콜 의미론을 이해하며(e.g., SQL verb), 프롬프트 주입에 강함. | 구현 복잡성, 정책 관리 오버헤드, 모든 호출에 대한 검사로 인한 지연 시간(latency) 발생 가능성. |

## Claw Patrol 아키텍처 해부

Claw Patrol은 와이어 레벨 게이팅을 구현한 구체적인 오픈소스 프레임워크입니다. iOS 개발자가 이 패턴을 이해하고 적용하기 위해 핵심 구성 요소를 살펴보겠습니다.

1.  **게이트웨이 (Gateway)**: 모든 에이전트의 트래픽은 WireGuard나 Tailscale 같은 터널을 통해 중앙 게이트웨이로 라우팅됩니다. 이 게이트웨이는 TLS를 종료하고, 내부 프로토콜(Postgres, K8s API 등)을 파싱하는 역할을 합니다.
2.  **프로토콜별 사실 추출 (Fact Extraction)**: 게이트웨이는 각 프로토콜의 의미를 이해합니다. HTTP 요청에서는 `method`, `path`를 추출하고, Postgres 프로토콜에서는 SQL `verb`(e.g., `SELECT`, `DROP`)와 대상 `table` 이름을 추출합니다.
3.  **정책 엔진 (Policy Engine)**: 추출된 '사실(fact)'들은 HCL(HashiCorp Configuration Language)로 작성된 규칙과 비교하여 평가됩니다. HCL은 선언적으로 보안 정책을 명시하기에 적합합니다.
4.  **자격 증명 관리 (Credential Management)**: 에이전트는 실제 API 키나 DB 암호를 갖지 않습니다. 대신 플레이스홀더를 사용해 요청하면, 게이트웨이가 요청을 승인할 때 실제 자격 증명을 주입합니다. 이는 에이전트가 탈취되더라도 민감한 정보 유출을 막는 핵심 기능입니다.

### iOS/Swift 환경에서의 개념적 구현

iOS 앱 내에서 동작하는 에이전트가 로컬 함수(Tool)를 호출하는 시나리오를 생각해 봅시다. Claw Patrol의 아키텍처를 Swift로 단순화하여 개념을 구현할 수 있습니다.

먼저, 에이전트가 호출할 모든 액션을 직렬화 가능한 구조체로 정의합니다.

```swift
// 모든 도구 호출을 표현하는 직렬화 가능한 구조체
struct AgentAction: Codable {
    let toolName: String
    let parameters: [String: String] // 단순화를 위해 모든 파라미터를 문자열로 표현
    let timestamp: Date
}

// 보안 정책을 정의하는 프로토콜과 구체적인 규칙
protocol PolicyRule {
    func evaluate(action: AgentAction) -> Bool
}

struct BlockDestructiveCommandsRule: PolicyRule {
    let blockedCommands = ["deleteFileSystemObject", "executeShellCommand"]

    func evaluate(action: AgentAction) -> Bool {
        return !blockedCommands.contains(action.toolName)
    }
}
```

다음으로, 실제 도구 실행 전에 액션을 가로채 정책을 검증하는 `SecurityFirewall`과 `GuardedToolExecutor`를 만듭니다.

```swift
class SecurityFirewall {
    private let rules: [PolicyRule]

    init(rules: [PolicyRule]) {
        self.rules = rules
    }

    func approve(action: AgentAction) -> Bool {
        // 모든 규칙을 통과해야만 승인
        for rule in rules {
            if !rule.evaluate(action: action) {
                print("🔥 FIREWALL: Action denied by rule \(type(of: rule)). Action: \(action.toolName)")
                return false
            }
        }
        print("✅ FIREWALL: Action approved. Action: \(action.toolName)")
        return true
    }
}

class GuardedToolExecutor {
    private let firewall: SecurityFirewall

    init(firewall: SecurityFirewall) {
        self.firewall = firewall
    }

    // 에이전트는 이 메서드를 통해 도구를 호출
    func execute(toolName: String, parameters: [String: String]) {
        let action = AgentAction(
            toolName: toolName,
            parameters: parameters,
            timestamp: Date()
        )

        // 와이어 레벨 게이팅: 직렬화된 Action 객체를 방화벽에 전달
        guard firewall.approve(action: action) else {
            // 실행 거부
            return
        }

        // 실제 도구 실행 로직 (예시)
        print("🚀 EXECUTOR: Executing tool '\(toolName)'...")
        // ... 실제 tool_name에 해당하는 Swift 함수를 호출 ...
    }
}

// 사용 예시
let firewall = SecurityFirewall(rules: [BlockDestructiveCommandsRule()])
let executor = GuardedToolExecutor(firewall: firewall)

// LLM 에이전트가 생성한 호출 시뮬레이션
executor.execute(toolName: "fetchCalendarEvents", parameters: ["date": "today"])
// 출력:
// ✅ FIREWALL: Action approved. Action: fetchCalendarEvents
// 🚀 EXECUTOR: Executing tool 'fetchCalendarEvents'...

executor.execute(toolName: "deleteFileSystemObject", parameters: ["path": "/private/var"])
// 출력:
// 🔥 FIREWALL: Action denied by rule BlockDestructiveCommandsRule. Action: deleteFileSystemObject
```

이 Swift 코드는 실제 네트워크 통신 대신 함수 호출을 가로채지만, '실행 직전의 구체적인 액션을 독립된 보안 계층에서 검증한다'는 와이어 레벨 게이팅의 핵심 철학을 동일하게 보여줍니다.

## `moneyflow` 프로젝트에 적용 시나리오

`moneyflow` 같은 금융 트레이딩 에이전트 프로젝트에 이 아키텍처를 적용하면 치명적인 실수를 방지할 수 있습니다.

-   **상황**: 포트폴리오 리밸런싱 작업을 지시받은 에이전트가 환각(hallucination)을 일으켜 `DROP TABLE users;` SQL을 생성.
-   **방어**: Claw Patrol 게이트웨이는 Postgres 와이어 프로토콜을 파싱하여 SQL `verb`가 'DROP'인 것을 인지합니다. HCL 정책에 `"DROP"과 "DELETE" 동사는 human-in-the-loop 승인이 필요하다`는 규칙이 있다면, 이 쿼리는 자동으로 차단되고 슬랙(Slack)으로 승인 요청이 전송됩니다.
-   **상황**: 에이전트가 시장가 주문(`market order`)을 생성하여 과도한 슬리피지(slippage) 발생 위험.
-   **방어**: `create_order` API 호출을 게이팅하는 정책을 설정할 수 있습니다. 정책은 `order_type` 파라미터가 항상 `limit`인지, `limit_price` 파라미터가 존재하는지 검증하고, 그렇지 않으면 호출을 거부합니다. 이는 LLM의 실수를 시스템 레벨에서 보정합니다.

-   Claw Patrol 아키텍처에서 '자격 증명 주입(credential injection)' 방식이 에이전트의 보안을 어떻게 향상시키나요?
-   와이어 레벨 게이팅을 도입할 때 얻는 가장 큰 이점과 감수해야 할 가장 큰 운영 비용(trade-off)은 무엇이라고 생각하십니까?
-   현재 진행 중인 프로젝트(또는 구상 중인 앱)에서 AI 에이전트에게 위험한 권한(e.g., 파일 쓰기, 유료 API 호출, 데이터 삭제)을 부여해야 한다면, 어떤 종류의 '와이어 레벨' 규칙을 정의하여 잠재적 위험을 통제하시겠습니까?
