AI 에이전트 시스템이 복잡해질수록, 우리가 이미 알고 있는 소프트웨어 설계 원칙이 다시 중요해진다. SOLID 원칙은 40년간 검증된 소프트웨어 설계의 핵심이며, 멀티 에이전트 시스템에도 그대로 적용된다.

핵심 통찰: **에이전트는 먼저 소프트웨어 컴포넌트다.** 추론 능력보다 시스템 설계가 먼저다.

## 5가지 원칙의 에이전트 매핑

### S — Single Responsibility (단일 책임)

**OOP**: 클래스 하나는 변경 사유가 하나여야 한다.

**에이전트**: 에이전트 하나는 책임이 하나여야 한다.

- 계획(Plan), 검색(Search), 실행(Execute), 검증(Verify)을 **하나의 에이전트가 다 하면 안 된다**
- MoneyFlow의 13개 에이전트가 각각 한 가지 분석만 담당하는 것이 이 원칙의 실례
- 위반 시: "초과 책임 에이전트"가 예측 불가 행동 + 비용 폭증

### O — Open-Closed (개방-폐쇄)

**OOP**: 확장에는 열려 있고, 수정에는 닫혀 있어야 한다.

**에이전트**: 새 에이전트 추가 시 오케스트레이터를 수정할 필요가 없어야 한다.

- 결제 검증 에이전트, 사기 탐지 에이전트, 음성 확인 에이전트를 **독립적으로 추가**
- 오케스트레이터는 "에이전트 목록"만 관리, 각 에이전트의 내부 구현은 모름
- Aidy의 4-Agent 시스템에서 새 플랫폼(Android)을 추가해도 Architect는 변경 없음

### L — Liskov Substitution (리스코프 치환)

**OOP**: 자식 클래스는 부모의 인터페이스를 대체할 수 있어야 한다.

**에이전트**: 같은 인터페이스를 따르는 에이전트는 상호 교체 가능해야 한다.

- 추천 에이전트의 구현이 OpenAI API든, 로컬 ML이든, 규칙 기반이든 → **동일한 입출력 계약**
- MoneyFlow의 Bounded Retry: Claude 실패 → Gemini 자동 전환 → GPT 폴백. 같은 인터페이스라 가능

### I — Interface Segregation (인터페이스 분리)

**OOP**: 필요 없는 메서드에 의존하면 안 된다.

**에이전트**: 에이전트는 필요한 기능만 노출해야 한다.

- 뉴스 분석 에이전트가 차트 분석 에이전트의 API를 알 필요 없음
- 각 에이전트에 필요한 최소한의 인터페이스만 제공 → 결합도 감소
- Context Engineering의 핵심과 일치: 각 에이전트는 자기 영역의 데이터만 봐야 환각이 줄어든다

### D — Dependency Inversion (의존성 역전)

**OOP**: 고수준 모듈이 저수준 세부 구현에 의존하면 안 된다.

**에이전트**: 오케스트레이터는 구체적 AI 모델이 아닌 추상 계약에 의존해야 한다.

- "GPT-4를 호출한다"가 아니라 "분석 에이전트에 요청한다"
- 모델 교체(GPT → Claude → Gemini)가 오케스트레이터 코드 변경 없이 가능
- Harness의 폴백 패턴이 이 원칙 위에서 동작

## 실무 적용 체크리스트

에이전트 시스템을 설계할 때 이 질문을 던져라:

- **SRP**: 이 에이전트가 2가지 이상의 이유로 변경되는가? → 분리
- **OCP**: 새 에이전트를 추가할 때 기존 코드를 수정해야 하는가? → 추상화
- **LSP**: 이 에이전트를 다른 구현으로 교체할 수 있는가? → 인터페이스 통일
- **ISP**: 이 에이전트가 사용하지 않는 기능에 의존하는가? → 인터페이스 분리
- **DIP**: 오케스트레이터가 특정 모델/서비스에 하드코딩되어 있는가? → 추상화
