## Devils Advocate — 후처리 전 Raw Output 검증

이전 세션(Journal 016)이 남긴 노트:

> DevilsAdvocateResult는 이번 사이클에서 제외 — parseJSON 후 수동 후처리 (activated 수동 설정, counterarguments slice 등)가 있어서 기존 validator 패턴과 자연스럽게 맞지 않음.

### 해결: validator를 LLM raw output에 적용

```
LLM → parseJSON(raw, fallback, validateDevilsAdvocateRaw) → parsed (4 fields)
                                                              ↓
                                                    수동 후처리
                                                    - activated: true 추가
                                                    - counterarguments.slice(0, 5)
                                                    - confidence_penalty clamp(0, 15)
                                                              ↓
                                                    DevilsAdvocateResult 반환
```

`DevilsAdvocateRawOutput` 타입을 별도 정의 — `activated` 없이 LLM이 실제 반환하는 4 필드만:

```typescript
export interface DevilsAdvocateRawOutput {
  counterarguments: string[];
  risk_factors: string[];
  confidence_penalty: number; // 0-15
  summary: string;
}
```

validator가 실패하면 fallback(빈 배열, 0, 빈 문자열) → 후처리가 안전하게 빈 결과 반환. **기존 코드의 방어적 후처리(Array.isArray, typeof 등)도 그대로 유지** — validator는 *추가* 안전망이지 기존 방어를 *대체*하지 않는다.

## 나머지 3 에이전트 Validator

### validateDebateMessageRaw (bull/bear researcher)

가장 단순. research-debate.ts에서 bull과 bear 모두 같은 shape:

```typescript
{ argument: string, counter_points: string[] }
```

parseJSON 호출 2곳(bull/bear)에 동일 validator 주입.

### validateRiskPerspectives (risk debate)

3 관점 중첩 구조:

```typescript
{ aggressive: { argument, risk_assessment },
  conservative: { argument, risk_assessment },
  neutral: { argument, risk_assessment } }
```

`checkRiskPerspective` 헬퍼로 3 관점 검증 코드 중복 제거. 이 패턴이 **Zod 3차 재평가의 핵심 데이터** — 중첩이어도 헬퍼 하나로 깔끔하게 처리 가능.

### validatePortfolioManagerDecision (portfolio manager)

가장 복잡한 에이전트. 중첩 `price_targets` + 9 string/array 필드:

```typescript
{
  final_rating: SignalRating,
  confidence: number,
  executive_summary: string,
  entry_strategy: string,
  position_sizing: string,
  risk_management: string,
  time_horizon: string,
  key_catalysts: string[],
  key_risks: string[],
  price_targets: { bull_case: number, base_case: number, bear_case: number }
}
```

price_targets 중첩 검증: `isRecord` 체크 후 3 필드 각각 positive number 확인.

### 3차 재평가 입력

1. **실제 구현 결과**: `checkRiskPerspective` 헬퍼 패턴이 중첩을 깔끔하게 처리. Zod의 `.object().shape()` 대비 코드량 차이 미미.
2. **동시 세션 상태**: worktree 여전히 활성 (Conductor la-paz + 3 agent). package-lock.json 충돌 위험 지속.
3. **9 validator 패턴 일관성**: 모두 동일한 `ValidationResult<T>` 반환. Zod 도입 시 *패턴 이중화* 발생.
4. **성과**: 중첩 구조 포함 10개 validator를 vitest 92 케이스로 커버. type guard만으로 충분.

**Zod 불필요 확정.** 이 결정은 *교과서적*이 아니라 *실측 기반*:
- 3번 재평가해도 답이 같으면 확정이다
- Zod의 가치는 *더 복잡한 구조*에서 나오는데, moneyflow 에이전트의 최고 복잡도(PM)도 type guard로 충분했다
