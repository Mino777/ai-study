## 왜 지금 이 주제인가

이번 세션에서 55개 이상의 MDX 파일에 **"AI Agent Directive" 섹션**을 추가해야 했다. 순차적으로 하나씩 하면:

```
파일 1 수정 (3분) → 파일 2 수정 (3분) → ... → 파일 55 수정 (3분)
= 총 165분 (~3시간)
```

이렇게 하면 **세션 컨텍스트가 폭발**하고, 중간에 실수가 누적될 수밖에 없다. 더 중요한 문제는 **마지막 파일을 수정할 때쯤이면 처음 파일의 수정 패턴을 기억하지 못한다**는 것.

대신, 파일을 **카테고리별로 분리**하고 **6개 병렬 에이전트**를 동시에 띄웠다. 결과:

- **실제 소요 시간**: ~10분
- **빌드 실패**: 0회
- **merge 충돌**: 0건
- **모든 55개 파일**: 일관된 패턴으로 수정 완료

이 패턴이 어떻게 작동하는지, 어디서 함정이 숨어있는지 정리한다.

## 핵심 패턴: 파일 분리 → 병렬 에이전트 → 동시 커밋

### Phase 1 — 파일 분리 (직접 계산)

MDX 파일 55개를 6개 에이전트에 분배. **핵심: 카테고리 기준으로 깔끔하게 분리**.

| 에이전트 | 할당 카테고리 | 파일 수 | 예상 시간 |
|---------|-----------|--------|---------|
| Agent 1 | harness-engineering | 25 | 10분 |
| Agent 2 | context-engineering, prompt-engineering | 12 | 5분 |
| Agent 3 | agents, evaluation | 8 | 3분 |
| Agent 4 | rag, tokenomics, fine-tuning | 7 | 3분 |
| Agent 5 | frontend-ai, ios-ai | 9 | 4분 |
| Agent 6 | infrastructure, android-ai, backend-ai | 9 | 4분 |

각 에이전트에 할당된 파일 집합은 **완전히 겹치지 않음**. 같은 파일을 두 에이전트가 수정할 수 없으므로 merge 충돌이 구조적으로 불가능.

**실제 배분 로직**:
```bash
# content/ 디렉토리 내 각 카테고리별 .mdx 파일 수 세기
ls content/harness-engineering/*.mdx | wc -l     # 25개
ls content/context-engineering/*.mdx | wc -l     # 12개
# ... 반복
```

직접 세지 말고 **다음 명령으로 한 번에**:
```bash
for dir in content/*/; do 
  count=$(ls "$dir"*.mdx 2>/dev/null | wc -l)
  echo "$(basename "$dir"): $count"
done | sort -t: -k2 -rn
```

### Phase 2 — 프롬프트 템플릿 통일

모든 에이전트가 **동일한 지시사항**을 받아야 품질이 일관된다. 내가 사용한 프롬프트:

```markdown
## 작업: AI Agent Directive 섹션 추가

각 파일의 frontmatter 뒤, 본문 시작 전에 다음 섹션을 추가하세요:

### AI Agent Directive

**Trigger**: [파일이 다루는 주제가 사용되는 시점]

**Prerequisites**:
- 선수 자료 3개 링크

### Actionable Steps
1. 단계 1
2. 단계 2
...

### Anti-patterns
- ❌ 하면 안 될 것 1
- ❌ 하면 안 될 것 2

---

## 체크리스트

✅ frontmatter (quiz 포함) 검증
✅ "AI Agent Directive" 섹션 추가 여부 확인
✅ 모든 링크가 카테고리 형식 (/wiki/category/slug) 따르는가
✅ Actionable Steps가 5~7개인가
✅ Anti-patterns가 최소 3개인가
✅ npm run build (또는 로컬 빌드) 통과 여부 확인
```

이 프롬프트를 **6개 에이전트에 그대로 전달**. 각 에이전트는:
- "당신은 harness-engineering 카테고리 25개 파일만 수정하세요"
- "당신은 context-engineering 카테고리 12개 파일만 수정하세요"

형태로 분리된 지시사항을 받음.

### Phase 3 — 동시 작업 + 모니터링

모든 에이전트를 **동시에 시작**:

```bash
# 터미널 1: Agent 1 (harness-engineering)
claude code --session "batch-agent-1" --prompt "..."

# 터미널 2: Agent 2 (context-engineering)
claude code --session "batch-agent-2" --prompt "..."

# ... 터미널 3~6
```

또는 **Claude Code의 멀티 탭 기능** 활용:
- Tab 1: Agent 1
- Tab 2: Agent 2
- ...
- Tab 6: Agent 6

각 탭에서 **동일 프롬프트** (단, 카테고리 다름) 실행.

**모니터링 체크리스트**:
- 각 에이전트가 정해진 파일만 수정하는가?
- 한 에이전트가 다른 카테고리에 손댈 수 있는가? (안 된다)
- 모든 에이전트가 10분 내 완료했는가?

### Phase 4 — 동시 커밋 + 빌드 검증 한 번

모든 에이전트가 커밋을 완료한 후:

```bash
# 모든 브랜치를 main으로 merge
git checkout main
git merge batch-agent-1-branch
git merge batch-agent-2-branch
git merge batch-agent-3-branch
# ... 반복

# 빌드 한 번
npm run build

# 통과하면
git push
```

**왜 빌드가 한 번만 통과했는가?**
- 각 에이전트가 **다른 파일**을 수정했으므로 merge 충돌이 0
- 각 에이전트가 **동일 프롬프트**를 따랐으므로 구조 일관성 보장
- 개별 파일 구문 오류는 각 에이전트의 로컬 빌드 (또는 프롬프트 체크리스트)에서 이미 검증됨

## 실전 적용 — 내가 한 방식

### 1단계: 파일 분배 명시

55개 파일을 카테고리별로 정렬:

```markdown
## 배분 현황

### Batch 1 (harness-engineering, 25개)
harness-journal-001.mdx
harness-journal-002.mdx
...
(직접 나열하지 않아도 "harness-engineering 폴더의 모든 .mdx" 형태로 충분)

### Batch 2 (context-engineering, prompt-engineering, 12개)
...

### Batch 3~6
...
```

### 2단계: 각 에이전트에 프롬프트 전달

```
당신의 역할: Batch 1 담당 에이전트 (harness-engineering)

수정할 파일:
- content/harness-engineering/harness-journal-*.mdx (모든 25개)

작업:
1. 각 파일을 열기
2. frontmatter 검증 (quiz 포함)
3. 본문 시작 전 "AI Agent Directive" 섹션 추가
4. Trigger, Prerequisites, Actionable Steps, Anti-patterns 작성
5. 모든 링크 검증
6. npm run build로 검증
7. git commit -m "feat: add AI Agent Directive to harness-engineering (Batch 1)"
```

### 3단계: 모니터링

각 에이전트의 진행 상황을 **간단히만** 점검:
- ✅ 파일 열기 시작
- ✅ 프롬프트 이해
- ✅ 첫 파일 완료
- ✅ 커밋 완료

**절대 하면 안 될 것**:
- 한 에이전트의 작업을 다른 에이전트가 검토 (순서 대기 필요 → 병렬 이점 상실)
- 한 에이전트가 다른 에이전트 파일에 손대기 (병합 충돌 유발)
- 중간에 프롬프트 변경 (일관성 깨짐)

## 결과

| 지표 | 값 |
|-----|-----|
| 수정 파일 수 | 55+ |
| 병렬 에이전트 수 | 6 |
| 총 소요 시간 | ~10분 |
| Merge 충돌 | 0건 |
| 빌드 실패 | 0회 |
| 커밋 수 | 6개 (에이전트당 1개) |

각 에이전트의 커밋 메시지:
```
feat: add AI Agent Directive to harness-engineering (Batch 1)
feat: add AI Agent Directive to context/prompt-engineering (Batch 2)
feat: add AI Agent Directive to agents/evaluation (Batch 3)
...
```

나중에 원한다면 **6개 커밋을 1개로 squash**할 수 있지만, 각 배치가 **독립 검증 가능**하도록 분리 커밋으로 두는 게 역추적(audit trail) 측면에서 유리.

## 주의사항

### 1. 에이전트가 이미 커밋한 상태

만약 Agent 3이 작업을 마쳤을 때 Agent 1이 아직 중간이라면:

```bash
# Agent 3이 이미 git commit + git push했다면
git log --oneline | head
# batch-agent-3-commit이 보임

# Agent 1이 같은 main 브랜치에 push하려고 할 때
git push
# "Your branch is behind origin/main" 메시지 가능

# 해결: git fetch + git rebase (또는 git pull)
git fetch origin
git rebase origin/main
git push
```

**핵심**: 각 에이전트가 **자기 브랜치에서만 작업**하면 이 문제가 안 생긴다 (worktree 권장).

### 2. 빌드 검증 타이밍

```
Best practice:

1. 각 에이전트가 로컬에서 `npm run build` 실행 (커밋 전)
2. 통과하면 커밋
3. 모든 에이전트 커밋 완료 후, main에서 한 번 더 `npm run build`

이렇게 하면 "어느 에이전트의 실수가 전체를 깬" 상황을 **사전에** 방어.
```

### 3. 일부 에이전트 실패 시 대응

만약 Agent 4가 fail했다면:

```bash
# Agent 4 재실행
# 같은 프롬프트, 같은 파일 집합

# 다시 커밋
git commit -m "feat: add AI Agent Directive to rag/tokenomics (Batch 4 - retry)"

# merge 순서는 상관없음 (파일 겹침 없으므로)
git merge batch-agent-4-retry
```

### 4. 프롬프트 튜닝

만약 첫 번째 라운드 후 "AI Agent Directive 품질이 일정하지 않다"면:

```
원인: 프롬프트의 Trigger 작성 방식이 모호했음

해결:
1. 첫 라운드 결과 1개 파일 샘플로 검토
2. "Trigger를 더 구체적으로" 피드백 정제
3. 같은 카테고리의 나머지 파일은 다시 (프롬프트 v2)
4. 다른 카테고리는 프롬프트 v2로 다시 (새 에이전트)
```

이건 **한 번의 빌드**가 모든 함정을 발견하기 때문에 가능.

## AI Agent Directive

**Trigger**: 50+ 개의 파일을 동시에 일괄 수정해야 할 때. sequential 처리로는 세션 컨텍스트 또는 일관성이 깨질 위험이 있을 때.

**Prerequisites**:
- [Multi-Session AI Ops Patterns](/wiki/harness-engineering/multi-session-ai-ops-patterns) — 병렬 작업의 기본 안전
- [Compound Engineering Philosophy](/wiki/harness-engineering/compound-engineering-philosophy) — 빌드 검증과 원자적 커밋
- [Multi-Agent Pipeline](/wiki/agents/multi-agent-pipeline) — 에이전트 조율 기초

### Actionable Steps

1. **파일 분석**: 수정할 전체 파일 목록을 카테고리/모듈별로 그룹화 (겹침 0)
2. **에이전트 수 결정**: 병렬도 = `총 파일 수 / (한 에이전트의 처리 용량)`. 보통 5~8개 에이전트 권장
3. **프롬프트 템플릿 작성**: 모든 에이전트가 *동일한* 체크리스트와 형식을 따를 수 있게
4. **파일 할당**: 각 에이전트에 "당신은 카테고리 X의 파일 N개만 수정" 명시
5. **동시 시작**: 모든 에이전트를 같은 시간에 띄우기 (또는 순차 시작 후 병렬로 진행)
6. **각자 로컬 빌드**: 각 에이전트가 커밋 전 `npm run build` (또는 해당 프로젝트의 빌드 명령) 실행
7. **merge 이전 검증**: 모든 에이전트 커밋 완료 후, main에서 최종 빌드 한 번 (충돌 없음 보장)
8. **원자적 커밋**: 각 배치별 커밋 1개 또는 전체 배치 1개 squash (나중에 필요시)

### Anti-patterns

- ❌ **파일을 겹쳐서 할당**: 두 에이전트가 같은 파일을 수정하면 merge 충돌이 필연
- ❌ **프롬프트를 에이전트마다 다르게 주기**: 품질 일관성이 깨짐. 같은 프롬프트, 다른 파일 집합이 원칙
- ❌ **중간에 프롬프트 변경**: 나중에 추가된 에이전트와 처음 에이전트의 결과물이 다름
- ❌ **에이전트 간 의존성 만들기**: "Agent A의 결과가 Agent B의 입력" → 순서 필요 → 병렬 이점 상실
- ❌ **각 에이전트가 빌드 검증 생략**: 한 에이전트의 구문 오류가 merge 후에야 드러남
- ❌ **병렬 중에 main 손대기**: 내가 main을 수정하면 에이전트들의 커밋이 rebase 필요 → 혼란

## 변형: 의존성이 있는 경우

만약 수정 작업이 **순서가 필요**하다면 (예: Phase 1 API 설계 → Phase 2 구현):

```
병렬 대신 세미 시리즈:

Phase 1: Agent A만 실행 (선수 작업, 독립)
  ↓ (완료 기다림)
Phase 2: Agent B, C, D 동시 실행 (Phase 1 결과 이용)
  ↓ (완료 기다림)
Phase 3: Agent E 실행 (최종 정리)

=== 하이브리드 병렬 ===
병렬도는 낮지만, 완전 시리즈보다는 2~3배 빠름
```

이 경우 **각 Phase 완료 후 빌드 검증** 필수.

| 51~150 | 6~10 | 10~30분 | 이 패턴 설계 기준 |
| 151+ | 10+ | 20~60분 | 자동화 도구(스크립트) 권장 |

**주의**: 에이전트 수를 무작정 늘린다고 빨라지지 않음. 각 에이전트도 **컨텍스트 한계**가 있으므로, 15개 파일 × 10개 에이전트보다는 **5개 파일 × 10개 에이전트** 방식이 정확성 높음.

## 자기 점검

1. 내 수정 작업이 **카테고리별로 완전히 독립적**인가?
2. 100개 파일이라면, **프롬프트 일관성을 유지할 방법**이 있는가?
3. 병렬 작업 중 **한 에이전트만 실패**했을 때 **재실행 계획**이 있는가?
4. **최종 빌드**는 누가, 언제 하는가?
5. 나중에 **"왜 이 파일이 이렇게 수정됐나"** 추적할 때 **커밋 메시지**로 충분한가?
