### 왜 중요한가: 에이전트 모니터링은 사후 분석이 아닌 실시간 제어여야 한다

기존의 AI 에이전트 모니터링은 대부분 작업이 끝난 후 로그, 트레이스, 대시보드를 통해 "무슨 일이 있었는가"를 분석하는 '사후 법의학(after-the-fact forensics)'에 머문다. 이 방식은 문제의 원인을 파악하는 데는 유용하지만, 이미 발생한 피해를 막지는 못한다. 예를 들어, 에이전트가 위험한 셸 명령을 실행한 뒤에야 로그를 통해 이를 발견하는 것은 너무 늦다.

진정한 에이전트 안전성은 결과가 확정되기 전에 개입할 수 있는 **런타임 개입(Runtime Intervention)** 능력에 달려있다. 모니터링 시스템은 단순히 관찰자에 그치지 않고, 에이전트의 행동 궤적(trajectory)을 실시간으로 감시하며 정책 위반이나 결정적 오류가 예상될 때 즉시 실행을 중단, 차단, 격리, 또는 수정하는 '작동기(actuator)' 역할을 해야 한다. 이 패러다임 전환은 최종 결과물의 품질을 따지는 것에서, 실행 과정 전체의 안정성을 보장하는 것으로 무게 중심을 옮긴다.

### 핵심 패턴

#### 1. 약한 모니터링 vs. 강한 모니터링

에이전트 모니터링은 '무엇이 일어났는가?'에서 '다음에 무엇이 일어나야 하는가?'로 초점을 옮겨야 한다.

| 질문 | 약한 모니터링 (사후 분석) | 강한 모니터링 (런타임 개입) |
| --- | --- | --- |
| **무슨 일이 있었나?** | 실행이 끝난 후 이벤트를 기록한다. | 실행 중에 타입이 정의된 이벤트를 기록한다. |
| **다음에 무엇을 해야 하나?** | 판단을 나중의 리뷰를 위해 남겨둔다. | 계속, 경고, 일시중지, 차단, 격리, 복구, 에스컬레이션 중 하나를 결정한다. |

#### 2. 런타임 개입의 7가지 유형

단순히 로그를 남기는 것을 넘어, 시스템은 상황에 따라 구체적인 제어 액션을 취해야 한다.

- **Continue**: 이벤트가 정책 및 예상 계획 범위 내에 있다.
- **Warn**: 이벤트가 비정상적이지만 되돌릴 수 있다.
- **Pause**: 다음 단계는 사람이나 정책 검토가 필요하다.
- **Block**: 액션이 명백한 하드 룰을 위반한다.
- **Contain**: 실행은 축소된 샌드박스나 권한 내에서만 계속될 수 있다.
- **Recover**: 시스템이 알려진 보상 경로(compensating path)를 실행한다.
- **Escalate**: 보안, 제품, 또는 도메인 전문가의 검토가 필요하다.

#### 3. 최신 연구가 제시하는 4가지 핵심 개념

Blake Crosley는 최신 연구들이 모두 '런타임 개입'이라는 방향을 가리킨다고 강조한다. 이는 그의 주관적인 주장이 아니라, 아래와 같은 권위 있는 논문들을 근거로 한다.

1.  **Temporal Logic & Intervening Monitors** (Alamdari et al., 2026): 개별 이벤트가 아닌, "테스트 없이는 배포하지 말라"와 같이 순서와 관계가 중요한 '시간적 제약(temporal constraints)'을 적용한다. 위반이 '예측'될 때 선제적으로 개입하는 모니터 개념을 제시한다.
2.  **AgentForesight**: 긴 호흡의 작업에서 단 하나의 '결정적 오류(decisive error)'가 전체 실패로 이어지는 것을 막는다. 작업이 끝난 뒤 원인을 찾는 대신, 실시간 감사(online auditor)가 현재까지의 실행 경로(prefix)만 보고 첫 번째 결정적 오류 지점에서 즉시 경고를 발생시킨다.
3.  **AgentTrust**: 에이전트의 툴 호출을 실행 **전**에 가로채 `allow`, `warn`, `block`, `review` 같은 구조화된 판정(structured verdict)을 내린다. 파일 시스템, 셸, DB 쿼리 등 실제 부작용(side effect)을 일으키는 지점을 통제하는 것이 핵심이다.
4.  **AIR (AI Incident Response)**: 기존의 예방 중심 안전성 연구를 넘어, 사고 발생 시의 대응, 격리, 복구 능력을 에이전트 실행 루프에 통합한다.

#### 4. 구조화된 개입 기록

모든 개입은 투명성과 신뢰성을 위해 구조화된 기록을 남겨야 한다. 이 기록만 보면 누가 봐도 왜 런타임이 경로를 변경했는지 이해할 수 있어야 한다.

- **Run**: 실행 ID, 태스크, 단계, 소유자
- **Event**: 툴 호출, 네트워크 요청, 파일 쓰기 등
- **Rule**: 발동된 정책 또는 시간적 제약
- **Evidence**: 트레이스 일부, 인자, 대상 리소스, 이전 이벤트
- **Decision**: `Continue`, `Warn`, `Block` 등 7가지 유형 중 하나
- **Next allowed action**: 결정 후 에이전트가 할 수 있는 행동
- **Human path**: 검토, 오버라이드, 종료 권한이 있는 담당자
- **Outcome**: 개입이 문제를 예방했는지, 지연시켰는지, 복구했는지 등

#### `ai-study` 위키 에이전트에 적용 시나리오

`ai-study` 프로젝트에서 새로운 논문을 요약하고 위키 페이지를 자동으로 생성하는 에이전트를 운영한다고 가정하자.

- **상황**: 에이전트가 `arxiv.org`에서 논문 PDF를 다운로드하고, 내용을 요약한 뒤, 결과물을 `git`을 사용해 위키에 푸시하려고 한다.
- **위험**:
    1.  에이전트가 논문 요약 중 실수로 `rm -rf .` 같은 파괴적인 셸 명령을 생성할 수 있다.
    2.  `git push --force` 명령으로 중요한 히스토리를 덮어쓸 수 있다.
    3.  `summarize()` 함수가 `push_to_wiki()` 함수보다 먼저 실행되어야 한다는 순서 제약을 어길 수 있다.

- **런타임 개입 적용**:
    - `AgentTrust` 패턴: `execute_shell` 툴 호출을 가로채는 인터셉터를 구현한다. 명령어에 `rm -rf`, `sudo`, `git push --force` 등의 위험한 패턴이 포함되면 즉시 `Block` 판정을 내리고, 담당자에게 `Escalate`한다.
    - `Temporal Logic` 패턴: 에이전트의 액션 시퀀스를 추적한다. `push_to_wiki` 액션이 요청되었을 때, 이전 이벤트 로그에 `summarize` 액션이 없다면 `Pause` 상태로 전환하고 사람의 승인을 요구한다.

#### TypeScript 코드 예시: 툴 호출 인터셉터

```typescript
type AgentAction = {
  tool: 'execute_shell' | 'write_file' | 'git_push';
  args: Record<string, any>;
};

type InterventionDecision = {
  decision: 'Continue' | 'Block' | 'Pause' | 'Warn';
  reason?: string;
  evidence?: AgentAction;
};

// AgentTrust 패턴을 구현한 런타임 모니터
async function runtimeMonitor(action: AgentAction): Promise<InterventionDecision> {
  if (action.tool === 'execute_shell') {
    const command = action.args.command as string;

    // 위험한 셸 명령어 패턴을 차단
    if (command.includes('rm -rf') || command.includes('sudo')) {
      return {
        decision: 'Block',
        reason: 'Destructive shell command detected.',
        evidence: action,
      };
    }
  }

  if (action.tool === 'git_push') {
    // 강제 푸시를 경고하고 검토를 위해 일시 중지
    if (action.args.force === true) {
      return {
        decision: 'Pause',
        reason: 'Force push requires human review.',
        evidence: action,
      };
    }
  }

  // 문제가 없으면 계속 진행
  return { decision: 'Continue' };
}
```
