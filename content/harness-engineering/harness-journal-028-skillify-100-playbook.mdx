## 왜 쓰는가

[Journal 027](/wiki/harness-engineering/harness-journal-027-skillify-resolver-backfill)이 Step 8(check-resolvable)을 9 repo에 dogfood한 기록이라면, 이 저널은 **Step 7(resolver-eval) 자체의 정확도를 올린 방법**이다. 목적은 한 가지: 이 playbook을 **워커 6개(moneyflow / tarosaju / aidy×4)에 그대로 적용**해서 각자 `weekly-resolver-eval.yml`을 돌리게 만드는 것.

재사용을 위해 *설명보다 숫자·순서·실패 신호*를 우선한다.

## 시작 상태

허브 CLAUDE.md의 `## Skill routing` 섹션 21개 rule. Step 7 최초 실행:

```
규칙 21개 / 케이스 25개 / 통과 13개 / 정확도 52.0%
```

통과한 13개의 패턴: 영어 키워드가 trigger 설명과 intent 양쪽에 반복 등장 ("deploy", "review"). 실패한 12개의 패턴: 사용자가 **한국어로 말하는 순간** 매칭 실패.

## 3개 레버 (순서가 곧 투자 대비 효과)

### Lever 1 — Tokenizer 현실화 (52 → 68%)

결정론적 tokenizer가 "으로/로/은/는/이/가/을/를/의" 조사를 통째로 단어의 일부로 취급하는 상태였다. "배포로", "PR을", "버그가" 가 영영 매칭되지 않음.

변경:

```js
const KOREAN_SUFFIX = /(?:으로|로|은|는|이|가|을|를|의|에게|한테|에서|부터|까지|에|와|과|야|다|자|해줘|해주|해|하자|하기|한테|라서|면|다면)$/;

function tokenize(text) {
  return text
    .toLowerCase()
    .replace(/[,.!?():·"'`\/\\]/g, " ")  // slash/backslash 추가
    .split(/\s+/)
    .map((w) => w.replace(KOREAN_SUFFIX, ""))
    .filter((t) => t.length > 1);
}
```

두 가지가 동시에 발동:
1. **조사 제거**: "배포로" → "배포"
2. **slash split**: "법률/보안/마케팅" → 세 토큰

**결과**: 13/25 → 17/25 = 68%.

**왜 tokenizer 먼저인가**: 이걸 안 한 상태에서 CLAUDE.md를 고치면 "영어 키워드를 한국어로 직역해서 박아 넣기"로 끝난다. tokenizer가 조사만 벗기면 *한국어 원문 그대로* 키워드를 박을 수 있다. 엔진 → 데이터 순서가 곧 재사용성.

### Lever 2 — CLAUDE.md routing 한국어 키워드 보강 (68 → 100%)

영어로만 쓰여 있던 10개 엔트리에 한국어 말뭉치를 덧댔다. 원문을 훼손하지 않고 **꼬리에 덧붙이는** 형태.

| skill | 추가된 한국어 키워드 |
|---|---|
| office-hours | 아이디어 브레인스토밍 빌드 가치 고민 |
| investigate | 버그 에러 문제 원인 크래시 디버깅 |
| ship | 배포 푸시 릴리스 진행 |
| qa | QA 테스트 |
| review | 코드 리뷰 PR 리뷰 diff 리뷰 검토 |
| document-release | 릴리스 후 문서 업데이트 문서화 |
| design-consultation | 디자인 시스템 브랜드 |
| design-review | 비주얼 디자인 폴리싱 폴리쉬 |
| plan-eng-review | 아키텍처 플랜 락 엔지니어링 |
| compound | 컴파운드 |

**의도적 제외**:
- design-review에서 "감사" 제외 — infra audit(check-skills 등)과 의미 충돌.
- compound에서 "돌려줘" 제외 — 너무 generic (어떤 cli 실행에도 붙음).

**결과**: 17/25 → 25/25 = 100%.

### Lever 3 — Rule parser regex 꼬리 버그 (잠복 이슈 발견)

초기 실행에서 rules가 **20개로 집계**됐다. 테이블은 분명 21개.

원인: `review` 라인이 다음 형태였음.

```
- Code review, check my diff → invoke review (2-Stage: Plan 있으면 ...)
```

rule regex는 `^-\s+(.+?)\s*(?:→|->)\s*invoke\s+([a-z][a-z0-9-]*)\s*$` — 꼬리 괄호가 `$` 앵커에 걸려 전체 매치 실패. rule 자체가 eval에서 **증발**.

수정: 꼬리를 desc 내부로 이동.

```
- Code review, check my diff, 코드 리뷰 PR 리뷰 diff 리뷰 검토 (2-Stage: Plan 있으면 Spec compliance + Code quality) → invoke review
```

**결과**: rules 20 → 21, 그리고 review 관련 intent가 비로소 정상 집계.

**일반화된 신호**: eval 도구는 *파싱된 규칙 개수*를 매 실행에 찍어야 한다. 숫자만 보면 잠복 실패가 사각지대로 숨는다. → `check:skills`의 "로컬 13 / 글로벌 43 / 라우팅 21" 1줄 로그가 같은 이유.

## 유지 검증 — golden set 확장

3 레버 직후 25/25라는 건 **과적합 의혹**이었다. 그래서 즉시:

1. Step 5(`extract-failure-moments`) 산출물에서 *다른 말투* 23개 발굴
2. `data/resolver-eval-cases.json` 25 → 48로 확장
3. `expectedSkill: null` 케이스(=아무 skill도 안 발동해야 함) 3개 포함
4. 재측정: **48/48 = 100%**

과적합이었다면 신규 23개에서 실패가 터졌어야 함. 유지된 건 3 레버가 *구조적*으로 유효하다는 방증. 단, 다음 스프린트에서 난이도 상향 필요.

## 워커 6개 이식 체크리스트

각 워커(moneyflow / tarosaju / aidy-architect / aidy-server / aidy-ios / aidy-android)에 적용할 순서.

```bash
# 0. 사전 점검 — 허브 상태
(허브) npm run check:skills
(허브) npm run eval:resolver   # 100% 기준점 확인

# 1. 워커에 scripts/resolver-eval.mjs 복사 (tokenizer 포함 버전)
cp /Users/jominho/Develop/ai-study/scripts/resolver-eval.mjs <워커>/scripts/

# 2. data/resolver-eval-cases.json 워커용 축소 세트 작성 (5~10개)
#    - 실제 그 워커에서 자주 쓰는 말투만
#    - expectedSkill: null 케이스 1개 이상 포함

# 3. package.json 에 "eval:resolver": "node scripts/resolver-eval.mjs" 추가

# 4. 첫 실행 → 정확도 측정 → baseline 기록

# 5. 부족하면 CLAUDE.md routing 섹션에 워커 특화 한국어 키워드 보강
#    (허브처럼 꼬리에 덧붙이는 방식, 원문 훼손 금지)

# 6. 재측정 → 90%+ 달성 확인

# 7. .github/workflows/weekly-resolver-eval.yml 복사 + threshold 조정
#    (최초는 80%, 안정화 후 90%)

# 8. Journal 027식 1줄 보고: "워커 X — M/N = P%"
```

**예상 발생 이슈**:
- 워커의 routing 섹션이 **영어 단문만**인 경우 Lever 2 보강 필수.
- aidy-server/ios/android 3개는 애초 routing 섹션이 없었음(Journal 027 데이터). 섹션을 **만든 뒤** eval 돌려야 함. 그 전까진 parse exit 2.
- moneyflow/tarosaju는 워커 특화 스킬이 없어 골든 케이스도 얇음. 일반 skill(ship/investigate/review) 중심 5케이스로 시작 → 실 사용 데이터 쌓이면 확장.

## 회고용 체크

- ✅ 엔진(tokenizer) → 데이터(CLAUDE.md) → 골든셋 확장 순서는 재사용 가능.
- ✅ "잠복 파싱 실패"를 우연히 발견했음 — 다음 버전에서 parse 로그 1줄 의무 출력으로 가드.
- ⚠️ golden set이 여전히 *내 말투 편향*. 다른 세션의 transcript에서 뽑으면 난이도 올라갈 것.
- ⚠️ LLM eval은 아직 미구현. 구조적 eval이 "설명 어휘 겹침"을 잡지만 "의도 유사 skill 간 충돌"(예: qa vs qa-only)은 놓친다.
