> 이 엔트리는 Blake Crosley의 글 "MCP Tools Need Action-Level Authorization"의 핵심 주장과, 그 배경이 된 실제 CVE 사례를 정리한 것이다.

### 왜 중요한가: 유효한 토큰이 '만능 키'가 되는 순간

2026년 5월(공개 2026-05-18), WordPress 플러그인 'AI Engine'에서 권한 상승 취약점(CVE-2026-8719, CVSS 8.8)이 보고되었다. 이 플러그인은 MCP(Model Context Protocol)를 통해 AI 에이전트와 WordPress 사이트를 연동하는 기능을 제공했다. 문제는 OAuth 인증 과정에 있었다. 시스템은 요청에 담긴 OAuth bearer 토큰이 **유효한지(Authentication)**만 확인하고, 해당 토큰의 소유자가 관리자 도구를 사용할 **권한이 있는지(Authorization)**는 확인하지 않았다.

보도된 분석에 따르면, 이로 인해 Subscriber 이상 권한을 가진 유효한 토큰만으로 관리자급 MCP 도구를 호출하여 Administrator로 권한을 상승시키는 공격이 가능했다. 벤더는 3.5.0 버전에서 이를 수정했다.

이 사례는 AI 에이전트 시대에 서버 보안의 패러다임이 어떻게 변해야 하는지를 명확히 보여준다. 인간 사용자는 정해진 UI 경로를 따르지만, AI 에이전트는 계획(planning)을 통해 API 엔드포인트(도구)를 예상치 못한 방식으로 조합하고 호출할 수 있다. 따라서 도구 하나하나에 대한 **액션 레벨(action-level) 인가**는 이제 선택이 아닌 필수다.

또한 Crosley는 한 논문을 인용하며, 오픈소스 생태계의 '도구 복제(tool-cloning)' 현상이 취약한 인가 로직을 빠르게 확산시킬 수 있다고 경고한다. 잘 만들어진 템플릿은 모범 사례를 퍼뜨리지만, 검증이 누락된 템플릿은 취약점을 대규모로 복제할 수 있다.

### 핵심 패턴: 인증과 인가를 명확히 분리하는 계층적 검증

AI Engine의 실패는 인증(Authentication)과 인가(Authorization)를 동일시한 것에서 비롯됐다. 유효한 토큰은 단지 "누구인지"를 증명할 뿐, "무엇을 할 수 있는지"를 보장하지 않는다. MCP 서버는 OAuth 토큰 검증을 통과한 후에도, 반드시 별도의 인가 계층을 두어 구체적인 액션을 허용할지 결정해야 한다.

올바른 검증 체인은 다음과 같은 다단계 프로세스를 거친다.

이 다이어그램은 MCP 사양과 Crosley가 강조하는 두 가지 핵심 분리 지점을 보여준다.

1.  **401 vs 403 응답의 분리**:
    *   `401 Unauthorized`: 토큰이 없거나, 만료되었거나, 서명이 틀리는 등 '인증' 실패. "당신이 누구인지 모르겠습니다."
    *   `403 Forbidden`: 토큰은 유효하여 '인증'은 성공했지만, 해당 작업을 수행할 '권한'이 없음. "당신이 누구인지는 알지만, 이 작업은 할 수 없습니다."

2.  **OAuth 역할 vs 애플리케이션 역할의 분리**:
    *   OAuth와 같은 프로토콜은 요청이 신뢰할 수 있는 출처에서 왔는지 확인하는 **인증 게이트** 역할에 충실하다.
    *   실제 비즈니스 로직과 데이터를 보호하는 책임은 **애플리케이션의 인가 게이트**에 있다. 이 단계에서 `(주체, 액션, 리소스)` 튜플에 기반한 정책 결정을 내려야 한다.

### 실전 적용: `tarosaju` 프로젝트의 AI 상담사 권한 제어

`tarosaju` 프로젝트에 AI 상담사가 도입되어 사용자 데이터를 기반으로 상담을 진행한다고 가정해보자. AI 에이전트는 다음과 같은 도구를 사용할 수 있다.

*   `getUserProfile(userId: string)`: 사용자의 기본 프로필 조회 (민감도 낮음)
*   `getPastReadings(userId: string)`: 사용자의 과거 상담 내역 조회 (민감도 높음)
*   `createNextAppointment(userId: string, details: object)`: 다음 상담 예약 (쓰기 작업)

여기서 AI Engine과 같은 실수를 방지하기 위해 액션 레벨 인가 미들웨어를 TypeScript로 구현할 수 있다.

```typescript
// src/middleware/mcpAuthorization.ts

import { NextFunction, Request, Response } from 'express';
import { verifyToken, DecodedToken } from '../auth/jwt';
import { UserPermissionService } from '../services/permissionService';

/**
 * MCP 도구 호출에 대한 액션 레벨 인가를 수행하는 미들웨어
 */
export const canAccessTool = (toolName: string) => {
  return async (req: Request, res: Response, next: NextFunction) => {
    const authHeader = req.headers.authorization;

    // 1. 토큰 유효성 검사 (Authentication Gate)
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      return res.status(401).json({ error: 'Missing or malformed token' });
    }
    
    let decodedToken: DecodedToken;
    try {
      decodedToken = verifyToken(authHeader.split(' ')[1]);
    } catch (error) {
      return res.status(401).json({ error: 'Invalid token' });
    }
    
    // 2. 주체(Subject) 매핑 및 정책 로드 (Authorization Gate 시작)
    const subjectId = decodedToken.sub; // JWT의 subject 필드 (사용자 ID)
    const targetResourceId = req.params.userId || req.body.userId;

    // 3. 인가 규칙 적용
    // 규칙 1: 관리자가 아닌 경우, 자신의 리소스에만 접근 가능해야 한다.
    if (decodedToken.role !== 'admin' && subjectId !== targetResourceId) {
      console.warn(
        `Authorization Failed: Subject ${subjectId} attempted to access resource ${targetResourceId} for tool ${toolName}.`
      );
      return res.status(403).json({ 
        error: 'Forbidden',
        reason: 'User can only access their own resources.'
      });
    }

    // 규칙 2: 특정 도구에 대한 권한이 있는지 확인 (예: getPastReadings)
    const hasPermission = await UserPermissionService.checkPermission(subjectId, toolName);
    if (!hasPermission) {
        console.warn(
          `Authorization Failed: Subject ${subjectId} lacks permission for tool ${toolName}.`
        );
        return res.status(403).json({
          error: 'Forbidden',
          reason: `Insufficient scope for tool: ${toolName}`
        });
    }

    // 모든 인가 검사를 통과하면 다음 핸들러로 진행
    next();
  };
};

// Express 라우터에 적용 예시
// import { canAccessTool } from './middleware/mcpAuthorization';
//
// router.get(
//   '/users/:userId/profile',
//   canAccessTool('getUserProfile'), // getUserProfile 도구에 대한 인가 검사
//   userController.getProfile
// );
//
// router.get(
//   '/users/:userId/readings',
//   canAccessTool('getPastReadings'), // getPastReadings 도구에 대한 인가 검사
//   userController.getReadings
// );
```

이 코드는 Blake Crosley가 제시한 핵심 패턴을 `tarosaju`의 맥락에 맞게 구현한 것이다. 단순히 JWT 토큰이 유효한지를 넘어, 토큰의 `subject`와 접근하려는 리소스의 `userId`를 비교하고, 데이터베이스에 저장된 사용자별 권한을 확인하여 `getPastReadings`와 같은 민감한 도구에 대한 접근을 제어한다. 이를 통해 AI 에이전트가 권한을 넘어 다른 사용자의 정보를 조회하는 치명적인 버그를 원천 차단할 수 있다.
