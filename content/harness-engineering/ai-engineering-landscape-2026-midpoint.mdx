## 왜 이 정리가 필요한가

2026년 AI 엔지니어링은 6개월 전과 완전히 다른 세상이다. Prompt Engineering → Context Engineering → Harness Engineering으로 패러다임이 이동했고, 에이전트 시스템이 프로덕션에 진입했으며, 도구 생태계(MCP, Tool Use)가 폭발적으로 성장했다.

이 글은 **현재 시점의 AI 엔지니어링 전체 지형도**를 한 페이지에 압축한다. 용어 정의 → 권위자 인사이트 → 방법론 비교 → Claude Code 실무 개념 → 트렌드 정리 → 실무 적용 기준 순서로.

## 0. 권위자들이 말하는 2026 AI 엔지니어링

> **"The hottest new programming language is English."**
> — Andrej Karpathy, Software 3.0 발표

Karpathy는 소프트웨어 진화를 3단계로 정의한다: 1.0(명시적 코드) → 2.0(학습된 신경망) → **3.0(자연어 프롬프트)**. 2026년, 코드를 작성하는 것보다 AI에게 올바른 컨텍스트를 제공하는 능력이 더 중요해졌다. 동시에 Karpathy는 LLM의 **"Jagged Intelligence(들쭉날쭉한 지능)"**를 경고한다 — 복잡한 코드를 척척 짜면서도 단순한 산수를 틀리는, 인간과 전혀 다른 능력 분포.

> **"AI 코딩 에이전트가 '대체로 작동'에서 '진짜 작동'으로 넘어간 변곡점은 2025년 11월이었다. 지금은 코드의 95%를 휴대폰에서 작성한다."**
> — Simon Willison (Django 공동창시자)

Willison은 동시에 AI 에이전트의 보안 위협도 경고한다. **"치명적 삼위일체(Lethal Trifecta)"**: 개인 데이터 접근 + 신뢰할 수 없는 콘텐츠 노출 + 외부 통신 능력이 결합되면 심각한 취약점이 된다.

> **"Scaling laws have not hit a wall."**
> — Dario Amodei (Anthropic CEO)

코드 작성 → 시스템 이해 → 자기 효율을 개선하는 에이전트 순으로 능력이 진화 중.

> **"Deep Agents come with batteries included — planning, memory, subagents, file systems, and starter prompts."**
> — Harrison Chase (LangChain CEO, ODSC 2025)

단일 에이전트에서 **에이전트 팀**으로의 전환. 계획·메모리·서브에이전트를 내장한 "깊은 에이전트"가 2026 표준.

### 숫자로 보는 2026

- **Multi-Agent System** 관심이 폭발적으로 증가. 기업들의 에이전트 도입 선언은 많지만 **실제 프로덕션 운영은 극소수** — 병목은 능력이 아닌 거버넌스와 통합
- **Claude Code Max**: 1M 토큰 컨텍스트 윈도우로 전체 코드베이스를 한 세션에 처리 가능

## 1. 핵심 용어 사전

### 모델 & 추론

| 용어 | 정의 | 실무 맥락 |
|------|------|-----------|
| **LLM** | Large Language Model. 대규모 텍스트로 학습된 언어 모델 | GPT-4, Claude, Gemini 등 |
| **SLM** | Small Language Model. 경량화된 온디바이스 모델 | Phi-3, Gemma, Apple Intelligence |
| **Inference** | 학습된 모델에 입력을 넣어 출력을 생성하는 과정 | API 호출 1회 = Inference 1회 |
| **Token** | 모델이 처리하는 텍스트 최소 단위 (~4 chars 영문, ~1.5 chars 한글) | 비용·속도의 기본 단위 |
| **Context Window** | 모델이 한 번에 처리할 수 있는 최대 토큰 수 | Claude: 200K(기본)/1M(Max), GPT-4o: 128K, Gemini: 1M~2M |
| **Temperature** | 출력의 무작위성 조절 (0=결정적, 1=창의적) | 코드 생성: 0~0.2, 브레인스토밍: 0.7~1.0 |
| **Hallucination** | 모델이 사실이 아닌 내용을 확신있게 생성하는 현상 | RAG + 출력 검증으로 완화 |
| **Chain-of-Thought (CoT)** | 단계별 추론을 유도하여 정확도를 높이는 기법 | "단계별로 생각해보세요" |
| **Structured Output** | JSON Schema 등으로 출력 형식을 강제하는 기법 | API 응답 파싱 신뢰도 향상 |
| **System Prompt** | 모델의 행동 규칙을 정의하는 최상위 지시문 | "너는 iOS 시니어 개발자야" 같은 역할 부여 |
| **Few-shot / Zero-shot** | 예시를 주고(Few) 또는 안 주고(Zero) 지시하는 기법 | Few-shot: 정확도↑ 토큰↑, Zero-shot: 빠르고 저렴 |
| **Grounding** | 모델 출력을 외부 사실 데이터에 근거하게 만드는 기법 | RAG, 웹 검색, 문서 참조로 환각 방지 |
| **Prompt Caching** | 동일 System Prompt/컨텍스트를 캐시하여 재사용 | Anthropic: 최대 90% 비용 절감. 반복 호출에 필수 |
| **Vibe Coding** | AI에게 "분위기"만 전달하고 코드 생성을 맡기는 방식 | 빠르지만 위험. 70/30 Rule: AI 70% 생성 + 사람 30% 검증 |
| **Jagged Intelligence** | LLM의 들쭉날쭉한 능력 분포 (복잡한 건 잘하고 단순한 건 틀림) | Karpathy 정의. 맹신 금지의 근거 |

### 에이전트 & 도구

| 용어 | 정의 | 실무 맥락 |
|------|------|-----------|
| **AI Agent** | 도구를 사용하여 목표를 자율적으로 달성하는 시스템 | Claude Code, Cursor, Devin |
| **Tool Use / Function Calling** | LLM이 외부 함수/API를 호출하는 능력 | 파일 읽기, DB 쿼리, 웹 검색 |
| **MCP (Model Context Protocol)** | AI-도구 연결 표준 프로토콜 (Anthropic 제안) | USB-C for AI. 서버/클라이언트 구조 |
| **Agentic AI** | 에이전트가 자율적으로 계획→실행→검증하는 패러다임 | 단순 Q&A가 아닌 멀티스텝 작업 수행 |
| **Multi-Agent System** | 여러 에이전트가 역할 분담하여 협업하는 시스템 | Architect-Worker, 허브-워커 패턴 |
| **Orchestrator** | 여러 에이전트의 실행을 조율하는 상위 컴포넌트 | 작업 분배, 결과 취합, 폴백 처리 |
| **Guardrail** | AI 출력의 안전성/정확성을 보장하는 제약 시스템 | 입력 검증, 출력 필터, 독성 탐지 |
| **Human-in-the-Loop** | 중요 결정에 사람의 승인/검토를 포함하는 설계 | 금융 거래, 의료 판단 등 고위험 영역 |
| **Tool Poisoning** | 악의적 MCP 서버/도구가 AI를 조작하는 공격 | Simon Willison이 경고한 보안 위협 |
| **A2A (Agent-to-Agent)** | Google이 제안한 에이전트 간 통신 프로토콜 | MCP가 AI↔도구라면 A2A는 AI↔AI 통신 |

### 데이터 & 검색

| 용어 | 정의 | 실무 맥락 |
|------|------|-----------|
| **RAG (Retrieval-Augmented Generation)** | 외부 데이터 검색 → LLM 컨텍스트에 삽입 → 생성 | 환각 감소, 최신 정보 반영 |
| **Embedding** | 텍스트를 고차원 벡터로 변환하는 것 | 유사도 검색의 기반 기술 |
| **Vector Database** | 임베딩 벡터를 저장하고 유사도 검색하는 DB | Pinecone, Chroma, pgvector |
| **Chunking** | 긴 문서를 검색 가능한 단위로 분할하는 전략 | 크기·오버랩·의미 단위 선택 |
| **Reranking** | 1차 검색 결과를 더 정밀한 모델로 재정렬 | Cross-Encoder로 정확도 향상 |
| **Hybrid Search** | 벡터 검색 + 키워드(BM25) 검색 조합 | 한쪽의 약점을 보완 |

### 학습 & 최적화

| 용어 | 정의 | 실무 맥락 |
|------|------|-----------|
| **Fine-tuning** | 사전학습 모델을 특정 도메인 데이터로 추가 학습 | 응답 스타일 통일, 도메인 특화 |
| **LoRA / QLoRA** | 파라미터 일부만 학습하는 효율적 파인튜닝 기법 | GPU 메모리 절약 (풀 파인튜닝 대비 1/10) |
| **RLHF** | 인간 피드백으로 모델을 정렬하는 학습 방법 | ChatGPT의 핵심 기술 |
| **Distillation** | 큰 모델의 지식을 작은 모델로 전이하는 기법 | 추론 비용 절감 |
| **Quantization** | 모델 가중치를 낮은 정밀도(FP16→INT4)로 변환 | 온디바이스 배포용 |

## 2. 엔지니어링 방법론 비교

### Prompt Engineering → Context Engineering → Harness Engineering

```
2023          2024-2025        2026
Prompt Eng → Context Eng  → Harness Eng
(단일 지시)   (환경 설계)     (안전 울타리)
```

| 방법론 | 핵심 질문 | 범위 | 대표 산출물 |
|--------|-----------|------|-------------|
| **Prompt Engineering** | "어떻게 지시할까?" | 단일 프롬프트 최적화 | 프롬프트 템플릿 |
| **Context Engineering** | "무엇을 보여줄까?" | 시스템 전체 컨텍스트 설계 | CLAUDE.md, 메모리 시스템, 검색 파이프라인 |
| **Harness Engineering** | "어떻게 안전하게 달리게 할까?" | 입력→실행→출력 전 과정의 안전 보장 | 검증기, 폴백 체인, 모니터링 |
| **Compound Engineering** | "어떻게 복리로 성장시킬까?" | 매 사이클의 산출물이 다음 사이클 입력 | 회고 문서, 솔루션 DB, CHANGELOG |

### Harness Engineering 5가지 레버

1. **입력 검증** — 타입/범위/null 체크, 스키마 검증
2. **출력 검증** — 구조 확인, 내부 모순 탐지, 사실 검증
3. **Bounded Retry** — 실패 시 N회 재시도 → 다른 모델 폴백
4. **LLM-as-a-Judge** — 별도 AI가 생성 AI의 출력을 채점
5. **모니터링** — 비용·품질·속도 실시간 추적 + 알림

### 에이전트 아키텍처 패턴

| 패턴 | 구조 | 적합한 상황 |
|------|------|-------------|
| **Single Agent** | LLM + Tools | 단순 작업, 프로토타입 |
| **Router** | 분류기 → 전문 에이전트 | 다양한 요청 타입 처리 |
| **Orchestrator-Worker** | 상위가 계획 → 하위가 실행 | 복잡한 멀티스텝 작업 |
| **Pipeline** | A → B → C 순차 처리 | ETL, 콘텐츠 생성 파이프라인 |
| **Evaluator-Optimizer** | 생성 → 평가 → 개선 반복 | 품질 중시 작업 |

## 3. Claude Code 실무 개념 — 헷갈리기 쉬운 것들

AI 코딩 도구를 실제로 쓰다 보면 "이게 뭐지?" 싶은 개념들이 많다. Claude Code를 기준으로 정리하지만, 다른 AI 도구에서도 유사한 패턴이 존재한다.

### CLAUDE.md — 프로젝트의 AI 설명서

프로젝트 루트에 두는 마크다운 파일. **Claude가 매 세션 시작 시 자동으로 읽는다.** 코딩 컨벤션, 빌드 명령어, 아키텍처 규칙, 금지 사항 등을 적으면 매번 반복 설명할 필요 없음.

```
계층 구조:
~/.claude/CLAUDE.md          ← 글로벌 (모든 프로젝트 공통)
./CLAUDE.md                  ← 프로젝트 루트 (팀 공유)
./src/CLAUDE.md              ← 디렉토리별 (세부 규칙)
```

**핵심**: CLAUDE.md ≠ README. README는 사람을 위한 것, CLAUDE.md는 AI를 위한 것. AI가 실행할 수 있는 구체적 명령어, 금지 패턴, 파일 구조 맵을 담는다.

### Hooks — AI 행동에 거는 자동화 트리거

**특정 시점에 자동으로 실행되는 셸 명령어/HTTP 호출.** settings.json에 정의.

```
실행 시점:
PreToolUse   → 도구 사용 "전"에 실행 (차단 가능)
PostToolUse  → 도구 사용 "후"에 실행 (검증/로깅)
SessionStart → 세션 시작 시 실행
SessionEnd   → 세션 종료 시 실행
```

**실무 예시**:
- `PreToolUse(Edit)`: 회사명/시크릿이 코드에 들어가는 걸 grep으로 차단
- `PostToolUse(Bash)`: `git commit` 전에 `npm run build` 자동 실행
- `SessionStart`: 프로젝트 상태 체크 스크립트 실행

**훅 vs CLAUDE.md**: CLAUDE.md는 "이렇게 해줘"라는 부탁(AI가 무시할 수 있음), 훅은 "이건 반드시"라는 강제(셸 레벨 차단).

### MCP (Model Context Protocol) — AI의 USB-C

**AI 모델이 외부 도구/데이터에 접근하는 표준 프로토콜.** Anthropic이 제안, 2026년 사실상 업계 표준.

```
구조:
┌─────────────┐     MCP     ┌─────────────┐
│  AI Model   │ ──────────▶ │  MCP Server  │
│ (Client)    │ ◀────────── │  (도구 제공)  │
└─────────────┘             └─────────────┘
                              │
                    ┌─────────┼─────────┐
                    ▼         ▼         ▼
                 GitHub    Slack    Database
```

**왜 필요한가**: MCP 없이는 각 AI 도구마다 GitHub 연동, Slack 연동, DB 연동을 따로 만들어야 한다. MCP는 **하나의 표준으로 모든 도구를 연결**한다. 마치 USB-C가 충전/데이터/영상을 하나의 포트로 통일한 것처럼.

**설정**: `.mcp.json` 또는 `~/.claude.json`에 서버 정의. Claude가 해당 도구를 네이티브 명령처럼 사용.

### Memory — 세션을 넘어서는 학습

**Claude가 대화에서 배운 것을 파일로 저장하여 다음 세션에서도 기억하는 시스템.**

```
메모리 유형:
user      → 사용자 역할/선호도 ("시니어 iOS 개발자, 한글 커밋 선호")
feedback  → 작업 방식 교정 ("테스트 없이 커밋하지 마")
project   → 프로젝트 상태/맥락 ("v2.0 마이그레이션 진행 중")
reference → 외부 리소스 포인터 ("버그 트래킹은 Linear INGEST 프로젝트")
```

**CLAUDE.md vs Memory**: CLAUDE.md는 개발자가 직접 작성하는 "규칙서". Memory는 Claude가 대화 중 알게 된 것을 자동으로 기록하는 "학습 노트". 둘 다 세션 시작 시 로딩.

### Slash Commands / Skills — 재사용 가능한 워크플로우

`.claude/commands/` 디렉토리에 마크다운 파일로 정의하는 커스텀 명령어. `/commit`, `/review`, `/ship` 등으로 호출.

```
예: .claude/commands/review.md
→ /review 입력하면 이 파일의 프롬프트가 실행

장점:
- 10단계 프로세스를 매번 타이핑할 필요 없음
- 팀원 간 워크플로우 표준화
- 필요할 때만 로딩 (컨텍스트 절약)
```

### Sub-agents — 병렬 전문가 소환

**메인 대화를 유지하면서 독립적인 작업을 병렬 수행하는 하위 Claude 인스턴스.** 각자 별도 컨텍스트를 갖고, 결과만 메인에 반환.

```
메인 Claude ──┬── Sub-agent A (파일 탐색)     → 결과 반환
              ├── Sub-agent B (테스트 실행)    → 결과 반환
              └── Sub-agent C (웹 리서치)     → 결과 반환
```

**왜 쓰나**: ①메인 컨텍스트를 오염시키지 않음 ②병렬 처리로 속도 향상 ③비용 최적화(Haiku 모델로 간단한 작업 위임).

### Worktree — 격리된 평행 세계

**Git worktree를 활용한 독립적인 작업 환경.** 같은 레포에서 다른 브랜치를 동시에 작업할 때 충돌 없이 격리.

```
메인 디렉토리 (main 브랜치)
  └─ .worktrees/
       ├─ feature-a/ (feature-a 브랜치, 별도 Claude 세션)
       └─ hotfix/ (hotfix 브랜치, 별도 Claude 세션)
```

### Plan Mode — 먼저 보고, 나중에 행동

**읽기 전용 모드.** Claude가 코드를 분석하고 계획을 세우지만 파일 수정이나 명령 실행은 하지 않음. `Shift+Tab` 두 번으로 토글.

**언제 쓰나**: 대규모 리팩토링 전 영향 범위 파악, 낯선 코드베이스 탐색, 아키텍처 변경 방향 합의 후 실행.

### Permissions — AI의 권한 등급

```
4단계:
Default     → 매번 물어봄 (가장 안전)
Accept Edits → 파일 수정은 자동 승인
Yolo        → 거의 다 자동 승인 (개인 프로젝트용)
Plan Mode   → 읽기 전용 (수정 불가)
```

`settings.json`에서 세밀 조정 가능: `Bash(npm run *)` → npm 스크립트만 자동 허용.

### Context Window 관리 — 토큰은 돈이자 품질

```
컨텍스트 구성:
┌────────────────────────────────────┐
│ System Prompt + CLAUDE.md          │ ← 고정
│ Memory                             │ ← 고정
│ 대화 히스토리                       │ ← 압축 대상
│ 도구 호출 결과                      │ ← 압축 대상
│ 현재 작업 컨텍스트                   │ ← 최우선
└────────────────────────────────────┘
```

**80% 찰 때 자동 압축**: 오래된 도구 결과 제거, 대화 요약. CLAUDE.md와 Memory는 살아남음. `/compact`으로 수동 압축 가능.

**실무 규칙**: CLAUDE.md는 200줄 이하 유지 → 초과 시 상세 내용을 Skills로 분리.

4. **배치** — 실시간 불필요한 요청은 Batch API로 50% 할인

## 7. 흔한 오해 바로잡기

### "Fine-tuning하면 최신 정보를 알게 된다"

**틀림.** Fine-tuning은 응답 스타일/포맷을 바꾸는 것이지 새로운 지식을 주입하는 게 아니다. 최신 정보 반영은 **RAG**의 역할. Fine-tuning은 "어떻게 말할지", RAG는 "무엇을 말할지".

```
최신 정보 필요     → RAG (검색 + 컨텍스트 삽입)
응답 스타일 통일   → Fine-tuning (특정 포맷/톤 학습)
둘 다 필요        → RAG + Fine-tuned 모델 조합
```

### "Temperature 0이면 항상 같은 결과"

**거의 맞지만 완전히는 아님.** Temperature 0은 가장 확률 높은 토큰을 선택하지만, GPU 연산의 부동소수점 차이, 배치 처리 순서 등으로 미세한 차이가 발생할 수 있다. **결정론적 결과가 필요하면** seed 파라미터를 함께 사용.

### "Context Window가 크면 무조건 좋다"

**아님.** 컨텍스트가 길어지면: ①비용 증가(토큰당 과금) ②"Lost in the Middle" 현상(중간 정보를 놓침) ③응답 속도 저하. **필요한 정보만 정확히 넣는 Context Engineering**이 큰 윈도우를 무작정 채우는 것보다 효과적.

### "AI가 짠 코드는 테스트 안 해도 된다"

**가장 위험한 오해.** AI는 Jagged Intelligence — 복잡한 아키텍처를 잘 짜면서도 off-by-one 에러, retain cycle, race condition을 조용히 만든다. **Vibe Coding의 함정**: 작동하는 것처럼 보이지만 엣지 케이스에서 터진다. 70/30 Rule 필수.

### "프롬프트만 잘 쓰면 된다"

**2023년까지의 진실.** 2026년에는 프롬프트는 전체 시스템의 5%에 불과. 나머지 95%: 어떤 컨텍스트를 제공하는지(Context Eng), 출력을 어떻게 검증하는지(Harness Eng), 실패를 어떻게 다음에 반영하는지(Compound Eng).

### "MCP = API 래퍼 아닌가?"

**비슷하지만 다름.** API 래퍼는 특정 서비스를 특정 AI 도구에 맞게 감싸는 것(1:1). MCP는 **표준 프로토콜**로, 한 번 MCP 서버를 만들면 Claude Code, Cursor, 어떤 MCP 클라이언트든 연결 가능(1:N). 생태계 효과가 핵심.
