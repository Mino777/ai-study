정적이고 예측 가능한 에이전트 워크플로는 이제 한계에 도달했습니다. 개발자가 미리 정의한 ReAct 루프나 체인(Chain)은 정해진 경로를 벗어나는 복잡하고 예외적인 상황에 취약합니다. 이런 구조적 경직성은 에이전트가 예상치 못한 오류에 직면했을 때 쉽게 좌초되거나, 목표를 완수하기 전에 스스로 작업을 끝났다고 선언하는 '에이전트적 게으름(agentic laziness)'과 같은 실패 모드를 낳습니다. 이 문제를 해결하기 위해 Anthropic의 Claude Code는 새로운 패러다임을 제시합니다: 에이전트가 단순히 도구를 선택하는 것을 넘어, 작업 해결을 위한 전체 오케스트레이션 계획(Harness) 자체를 동적으로 생성하는 것입니다.

이는 하네스 엔지니어링의 근본적인 전환을 의미합니다. 하네스는 더 이상 LLM을 통제하는 외부의 정적 구조물이 아니라, LLM이 자신의 실행 환경과 절차를 스스로 설계하고 코딩하는 대상이 됩니다.

## 정적 하네스에서 동적 하네스로의 전환

기존의 에이전트 아키텍처에서 하네스는 LLM의 행동을 제어하는 '고삐' 역할을 했습니다. 개발자는 도구 목록, 실행 순서, 오류 처리 로직을 코드로 직접 정의했습니다. 이 접근법은 예측 가능성이 높다는 장점이 있지만, 모델의 추론 능력을 제한하고 새로운 문제에 대한 적응성을 떨어뜨리는 단점이 명확합니다.

Claude Code의 동적 워크플로는 이 모델을 뒤집습니다. 사용자가 복잡한 작업을 요청하면(예: "레거시 코드베이스 전체의 보안 취약점을 감사하고 수정안을 제시하라"), Claude는 먼저 해당 작업을 수행하기 위한 최적의 '실행 계획'을 JavaScript 스크립트 형태로 생성합니다. 이 스크립트는 단순한 도구 호출의 나열이 아니라, 다수의 서브에이전트를 생성하고, 작업을 병렬로 분산시키며, 심지어 특정 에이전트가 다른 에이전트의 결과를 검증하는 '적대적 검증(Adversarial verification)'과 같은 복잡한 패턴까지 포함할 수 있습니다.

> **이 글의 초점, 그리고 자매 글**: 동적 워크플로의 *메커니즘*(오케스트레이션 상태를 모델 컨텍스트가 아닌 스크립트 변수에 두어 컨텍스트 윈도우 제약을 푸는 원리)과 트리거·플랜 제약은 [Claude Code 동적 워크플로우: 오케스트레이션 스크립트 생성과 대규모 서브에이전트 분산](/wiki/harness-engineering/claude-code-dynamic-workflow-agent-orchestration)에서 다룬다. 이 글은 그 위에서 **개발자가 실제로 조립하는 4가지 워크플로 패턴(Fan-out / Adversarial / Tournament / Loop-until-done)을 카탈로그**로 정리하고, iOS/Swift 작업에 어떻게 매핑되는지에 집중한다.

> **실제 제약 (2026-05 리서치 프리뷰 기준)**: 한 워크플로는 **총 1,000개까지의 에이전트**를 조율할 수 있지만, **동시 병렬 실행은 최대 16개**로 제한된다(평균적인 로컬 머신 리소스 안에 머물도록 튜닝된 상한). "1,000개 분산"은 누적 처리량이고 순간 병렬도는 16이다. 트리거는 프롬프트에 단어 `workflow`를 포함하는 것이며, 유료 플랜 + Claude Code 최신 버전이 필요하다.

### 작동 방식: 계획 수립과 실행의 분리

동적 워크플로의 핵심은 **계획(Planning) 단계**와 **실행(Execution) 단계**의 명확한 분리입니다.

1.  **계획 단계 (스크립트 생성)**: 사용자의 요청을 받은 최상위 에이전트는 작업을 분석하여 가장 효율적인 해결 경로를 담은 JavaScript 오케스트레이션 스크립트를 작성합니다. 이 스크립트는 어떤 종류의 서브에이전트(예: 코드 분석가, 보안 전문가, 테스트 작성자)가 필요한지, 어떤 순서로 또는 어떻게 병렬적으로 실행할지, 결과는 어떻게 종합할지를 정의합니다.
2.  **실행 단계 (스크립트 실행)**: 생성된 JavaScript 스크립트는 Claude Code의 하네스(실행기)에 의해 실행됩니다. 하네스는 스크립트의 지시에 따라 서브에이전트들을 생성하고, 각 에이전트에게 격리된 컨텍스트와 작업 공간(worktree)을 제공하며, 전체 워크플로의 진행 상황을 추적합니다.

이러한 분리는 단일 컨텍스트 창에서 계획과 실행을 동시에 수행할 때 발생하는 '목표 표류(goal drift)'나 '자기 선호 편향(self-preferential bias)' 같은 문제를 구조적으로 방지합니다. 각 서브에이전트는 자신에게 할당된 명확하고 작은 목표에만 집중할 수 있기 때문입니다.

### 동적 워크플로 패턴

Anthropic은 동적 워크플로를 생성할 때 자주 사용되는 몇 가지 공통 패턴을 소개했습니다.

| 패턴 (Pattern) | 설명 | Swift/iOS 적용 예시 |
| :--- | :--- | :--- |
| **Fan-out-and-synthesize** | 큰 작업을 작은 단위로 나누어 병렬 처리한 후 결과를 종합합니다. | 모놀리식 iOS 앱을 20개의 Swift Package로 분리하는 작업. 각 모듈 분리를 별도 서브에이전트가 병렬로 처리 후, 최종 통합 에이전트가 종속성을 해결합니다. |
| **Adversarial verification** | 하나의 에이전트가 작업을 수행하면, 다른 에이전트가 그 결과를 비판적으로 검증합니다. | 새로운 기능 구현 에이전트가 코드를 작성하면, '까다로운 시니어 개발자' 페르소나를 가진 검증 에이전트가 잠재적 버그나 아키텍처 문제점을 찾아냅니다. |
| **Tournament** | 동일한 작업에 대해 여러 에이전트가 서로 다른 접근 방식으로 해결책을 제시하고, 판정 에이전트가 최상의 결과를 선택합니다. | CoreData 마이그레이션 성능 최적화. 한 에이전트는 경량 마이그레이션을, 다른 에이전트는 수동 매핑 모델을 제안하고, 벤치마크 결과를 바탕으로 최종안을 결정합니다. |
| **Loop until done** | 작업량이 불확실할 때, 특정 중단 조건이 충족될 때까지(예: 더 이상 수정할 버그가 없을 때) 에이전트를 반복적으로 실행합니다. | `XCTest`로 UI 테스트를 실행하고 실패하는 테스트를 자동으로 수정하는 루프를, 모든 테스트가 통과할 때까지 반복합니다. |

## 적응형 에이전트 설계의 핵심: 하네스의 역할 재정의

동적 워크플로 패러다임에서 하네스의 역할은 '결정론적 지시자'에서 '안전한 실행 환경 제공자'로 바뀝니다. 하네스 엔지니어링의 초점은 LLM의 모든 행동을 예측하고 제어하는 것에서, LLM이 생성한 계획을 안정적으로 실행하고 관찰하며 개입할 수 있는 런타임을 구축하는 것으로 이동합니다.

`aidy-ios` 프로젝트의 리팩토링 스프린트를 예로 들어보겠습니다. 기존 방식이라면 엔지니어가 수동으로 리팩토링 계획을 세우고, 각 단계마다 에이전트에게 명시적인 명령을 내려야 합니다. 하지만 동적 워크플로를 사용하면, "TMA 아키텍처를 26개의 Microfeature 모듈로 분리하라"는 단일 목표를 제시할 수 있습니다. 그러면 에이전트는 스스로 모듈 식별, 파일 이동, 종속성 업데이트, 테스트 실행의 전체 계획을 담은 스크립트를 생성하고, 하네스는 이 스크립트를 실행하며 수십 개의 서브에이전트가 병렬로 작업하는 것을 감독하기만 하면 됩니다.

### 트레이드오프와 한계

이 강력한 접근법에도 명확한 트레이드오프가 존재합니다.

1.  **토큰 비용 증가**: 워크플로 스크립트를 생성하고 다수의 서브에이전트를 호출하는 과정은 일반적인 단일 에이전트 세션보다 훨씬 많은 토큰을 소모합니다. Anthropic은 모든 작업에 이 기능이 필요한 것은 아니며, 비용을 신중히 고려해야 한다고 경고합니다.
2.  **실행 시간 및 복잡성**: 초기 계획 수립 단계는 추가적인 시간을 소요합니다. 또한, 생성된 워크플로의 실행을 디버깅하는 것은 정적 코드보다 복잡할 수 있습니다.
3.  **보안 위험**: LLM이 생성한 코드를 직접 실행하는 것은 잠재적인 보안 위험을 내포합니다. 따라서 매우 강력한 샌드박스 환경이 필수적입니다.

결론적으로, Claude Code의 동적 워크플로는 에이전트가 자신의 작업 방식을 스스로 설계하게 함으로써, 복잡하고 장시간이 소요되는 태스크에 대한 자동화의 지평을 넓혔습니다. 이는 AI 에이전트 아키텍처가 정적인 제어 구조에서 벗어나, 문제 자체에 적응하는 동적이고 생성적인 시스템으로 진화하고 있음을 보여주는 중요한 이정표입니다.
