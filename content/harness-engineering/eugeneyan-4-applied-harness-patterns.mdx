## ① 드리프트 2종 분리 — '올바른 작업?' vs '올바르게?'

Eugeneyan은 AI 에이전트의 드리프트를 한 덩어리로 다룬다. 워커 풀을 운영하면 이게 *두 개의 다른 문제*로 보인다:

- **방향 드리프트 (WHAT)** — 지금 작업이 올바른 작업인가? (요구사항 이탈, 다른 파일 수정, 범위 폭주)
- **실행 드리프트 (HOW)** — 작업 자체는 맞는데 실행이 올바른가? (테스트 누락, 회귀, 스타일 위반)

방향은 *advisor*가 잡아야 하고 실행은 *verifier*가 잡아야 한다. 한 워커가 둘 다 하면 둘 다 못한다.

### 워커 분담

```
Worker-1 (advisor) → 🧭 방향 드리프트 감시 ("지금 올바른 작업?")
Worker-3 (verify)  → ⚙️ 실행 드리프트 감시 ("올바르게 했나?")
```

CLAUDE.md 워커 계층 섹션에 명시:

```markdown
- 🧭 방향 드리프트 = Worker-1 advisor 의무
- ⚙️ 실행 드리프트 = Worker-3 verify 전담
```

이렇게 쪼개면 advisor가 "테스트가 빠졌네"를 잡지 않고, verifier가 "이게 PRD에 있던가?"를 묻지 않는다. 책임 경계가 한 줄로 정리된다.

## ② 트랜스크립트 마이닝 — 한국어 반복 신호 박제

Eugeneyan은 "사용자가 같은 질문을 반복하면 스킬화 후보"라고 썼다. 한국어 워커 프로젝트는 영어 동의어로 안 잡힌다.

### 시스템 노이즈 필터 (먼저 제외)

```
REVIEW_REQUEST
<task-notification>
워커 prime
```

이것들은 자동 디스패치다. 사용자 의도가 아니라 시스템이 만든 텍스트. 필터링하지 않으면 모든 트랜스크립트가 "반복"으로 잡힌다.

### 한국어 반복 신호

```
또      — "또 같은 에러 나"
다시   — "다시 한번 해줘"
왜 안   — "왜 안 돼"
아니   — "아니, 그거 말고"
엥     — "엥? 이상한데"
```

### 영어 반복 신호 (참고)

```
still
again
can you also
did you check
```

신호별로 분리 로그를 둔다:

```
.claude/logs/user-repeat-signals.jsonl
```

`/daily` 스킬이 매일 아침 이 로그를 요약해 "지난 24시간 가장 자주 나온 불만 N개"를 보고. 이게 다음 스킬화 후보 큐다.

> 한국어 신호는 *문화*다. "또"는 영어 "again"보다 강한 불만이고, "엥"은 영어 "huh"보다 약한 당혹이다. 직역하지 말고 *프로젝트 사용자가 실제로 쓰는 단어*를 박을 것.

## ③ tmux pane 상태 시각화 — 이모지 4종 + Stop hook 사운드

워커 4-pane 레이아웃에서 어느 pane이 일하는 중인지 한 눈에 안 보이면 결국 화면 전환 비용이 누적된다.

### 상태 이모지 4종

| 상태 | 이모지 | 의미 |
|------|--------|------|
| busy | ⚡ | 작업 중 |
| idle | 💤 | 대기 |
| blocked | 🚨 | ADVICE_REQUEST / 에러 |
| done | ✅ | 직전 작업 완료 (5초 후 자동 idle) |

### 역할 추정 (pane title 키워드 기반)

```bash
case "$CURRENT_TITLE" in
  *orchestrat*) ROLE="orchestrator"; ICON="🎩" ;;
  *worker-1*|*leader*)  ROLE="worker-1"; ICON="👨‍💼" ;;
  *worker-2*|*impl*)    ROLE="worker-2"; ICON="🛠" ;;
  *worker-3*|*verify*)  ROLE="worker-3"; ICON="👀" ;;
  *)                    ROLE="claude";   ICON="🤖" ;;
esac
NEW_TITLE="${STATE_ICON} ${ICON} ${ROLE}"
tmux select-pane -T "$NEW_TITLE"
```

### done은 pulse — 5초 후 idle 복귀

```bash
if [ "$STATE" = "done" ]; then
  ( sleep 5; tmux select-pane -T "💤 ${ICON} ${ROLE}" ) &
fi
```

영구 done이 아니라 *방금 끝났다*는 순간 신호. 다음 작업 대기 상태로 즉시 돌아온다.

### Stop hook 사운드 (AFK 알림)

```bash
# ~/.claude/settings.json
"hooks": {
  "Stop": [
    { "matcher": "", "hooks": [{ "type": "command", "command": "bash $CLAUDE_PROJECT_DIR/.claude/scripts/stop-sound.sh" }] }
  ]
}
```

스크립트:
```bash
SOUND_FILE="${STOP_SOUND_FILE:-/System/Library/Sounds/Glass.aiff}"
[ "${STOP_SOUND_MUTE:-0}" = "1" ] && exit 0
afplay "$SOUND_FILE" &
```

AFK 중 다른 작업하면서 완료 알림 받기. `STOP_SOUND_MUTE=1`로 회의 중 음소거 가능.

## ④ CLAUDE.md 306→210줄 분리 (31%↓)

[Frozen Snapshot 원칙](/wiki/harness-engineering/orchestrator-worker-tmux-communication-rules) — 활성 세션 중 CLAUDE.md를 수정하면 캐시 전체 무효화 + 1.25x 토큰 오버헤드. **본문은 매 turn 로드할 가치가 있는 룰만** 남긴다.

### 남긴 것 (본문 210줄)

- HARD RULE (절대 하지 말 것)
- 워커 계층 + 디스패치 흐름
- Skill routing 핵심 매핑
- 드리프트 2종 분리 (① 결과)

### 분리한 것 (필요 시 fetch)

| 파일 | 줄수 | 내용 |
|------|------|------|
| `docs/guides/build-and-test.md` | 86 | 스펙 검증, 테스트 정책, Think in Code, Build 가속 |
| `docs/guides/skill-catalog.md` | 58 | 워크플로우, 자동 활성, Agent 카탈로그 |

검증/정책/카탈로그는 *작업 중 한 번도 안 읽을 가능성*이 높다. 매 turn 로드는 낭비. 워커가 필요할 때 `Read` 도구로 가져오면 그게 곧 캐시 미사용 cost — 하지만 본문에 박힌 200줄이 매 turn 캐시되는 cost보다 *훨씬 작다*.

### 측정

```
변경 전: 306줄 (turn당 약 8.2K 토큰 캐시)
변경 후: 210줄 (turn당 약 5.6K 토큰 캐시)
차이:    -2.6K 토큰/turn × 200 turn/일 = -520K 토큰/일
```

> ⚠️ 분리 자체가 자동 정답은 아니다. *어떤 룰을 매 turn 보고 어떤 룰을 fetch할지* 판단이 핵심. "본문 = 절대 룰 + 의사결정 분기" / "분리 = 절차 + 카탈로그"가 좋은 분리 기준.

## 적용 순서 권장

1. **④ CLAUDE.md 분리 먼저** — 캐시 비용 즉시 절감. 다른 패턴 박을 때 본문 부풀어 오르는 걸 막는다.
2. **③ tmux pane 상태** — 즉시 효과. AFK 가능해지면 워커 풀 ROI 급상승.
3. **② 트랜스크립트 마이닝** — 신호 박제는 빠르지만 *2~3주* 누적돼야 패턴이 보인다. 일찍 시작.
4. **① 드리프트 2종 분리** — 워커 풀 안정화 후. 분담 룰은 운영 중 추가해도 늦지 않다.

## 안티패턴

- 영어 신호를 한국어로 직역해서 그대로 박제 → "또" 가 "again"의 한국어 번역이라고 생각하면 *문화적 강도*가 안 맞는다
- 모든 룰을 분리 파일로 빼고 CLAUDE.md를 포인터만 두기 → 매번 fetch가 발생해 분리 효과 상쇄
- pane 이모지를 4개 이상 추가 → 시각 노이즈. 5개 넘으면 한 눈에 안 들어옴
- 드리프트 분리를 3종 이상으로 쪼개기 → "범위 드리프트" "스타일 드리프트" 등 추가하면 책임 경계가 다시 흐려짐. 2종이 적정선
