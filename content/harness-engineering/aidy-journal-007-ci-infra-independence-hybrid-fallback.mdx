## ADR-010 — Hybrid Fallback 전략

### 워커별 차등 적용

| 워커 | 전략 | 근거 |
|---|---|---|
| iOS | self-hosted **only** | macOS 분 가중치 10x — ADR-009 유지 |
| Server | **Hybrid** (GitHub-hosted primary + self-hosted fallback) | Linux 계수 1x — 평상시 무료 쿼터(2000분/월) 활용, 장애 시 자동 fallback |
| Android | **Hybrid** (동일 + merge 가드) | 동일 논리 |

Hybrid를 선택한 이유: 사용자 의도("깃헙 먼저 찌르고 실패시 self-hosted") 100% 부합 + Linux 무료 쿼터 활용 극대화. self-hosted only로 가면 무료 2000분/월을 버리게 된다.

### continue-on-error masking 버그 발견

WO-016 최초 설계:

```yaml
# 설계 의도: primary fail → fallback 발동
primary:
  runs-on: ubuntu-latest
  continue-on-error: true
fallback:
  needs: primary
  if: needs.primary.result != 'success'   # ← 항상 false!
```

**server 1차 실행에서 발견**: primary fail + fallback **skipped**. 예상의 정반대.

**근본 원인**: `continue-on-error: true`가 job-level에 있으면 `needs.<job>.result`를 `'success'`로 마스킹. GitHub Actions 미문서화 동작. job-level에는 step의 `outcome`도 미제공이라 어떤 필드로도 실패 여부 확인 불가.

### Mark-step 우회 패턴 (ADR-010 §8)

```yaml
primary:
  runs-on: ubuntu-latest
  continue-on-error: true
  outputs:
    passed: ${{ steps.mark.outputs.passed }}
  steps:
    - # actual work ...
    - name: Mark primary success
      id: mark
      # if: success() (암묵 기본값) — 이전 step 실패 시 skip
      run: echo "passed=true" >> $GITHUB_OUTPUT

fallback:
  needs: primary
  if: ${{ always() && needs.primary.outputs.passed != 'true' }}
  runs-on: [self-hosted, macOS, ARM64, aidy-server]
```

**왜 작동하는가**:
1. `outputs`는 `continue-on-error` 마스킹 대상이 아님 (step-level 값 직접 전달)
2. Mark step의 `if: success()`(암묵) — 이전 step 실패 → skip → `passed` 미설정 → fallback 발동
3. billing 차단(job startup fail) → step 전부 미실행 → `passed` 미설정 → fallback 발동
4. `always()`로 needs 상태와 무관하게 평가 실행

2차 실행 결과: primary fail → fallback **green** (3초 지연). 실증 완료.

## 박제 교훈

### 1. 선행 투자의 복리 효과

iOS self-hosted(WO-010, Journal 006)는 macOS 분 절감이 주요 동기였다. 그런데 billing 차단이 재발했을 때 **iOS만 유일하게 즉시 green**. 인프라 독립성 투자는 예상 못한 시점에 배당을 지급한다. Compound Engineering의 본질 — 지금의 결정이 미래 세션의 옵션을 넓힌다.

### 2. `continue-on-error` + `needs.result` = 지뢰

GitHub Actions 미문서화 동작. `continue-on-error: true`가 job-level에 있으면 `needs.<job>.result`를 `'success'`로 마스킹. **fallback workflow 설계 시 `needs.result` 사용 금지** → Mark-step 패턴 필수.

### 3. spec-first-verify-first

WO-016 §1 예시 패턴 버그는 Architect가 spec 작성 시 dry-run을 안 한 것. workflow 패턴처럼 *실행해봐야 확인되는* spec은 "1 워커 실행 → 실증 → spec 확정 → 나머지 dispatch" 순서.

### 4. 크로스 워커 전파 채널 필수

tmux 알림이 없었다면 android 워커가 server와 같은 버그를 반복. 워커 간 실시간 전파 채널(tmux, 알림 스크립트)이 multi-agent 오케스트레이션의 비용 절감 인프라.

### 5. Hybrid는 쿼터 최적화 전략

Linux 계수 1x인 server/android는 self-hosted only가 아니라 Hybrid가 정답. 무료 2000분/월을 버리지 않으면서 billing 장애에 자동 대응. 경제성과 안정성의 양립.

## AI Agent Directive

### Trigger
- GitHub Actions workflow에서 `continue-on-error: true` 사용 + fallback job 설계 시
- billing 차단 재발 시 (iOS 외 워커)
- self-hosted runner 추가 등록 시
- Hybrid fallback 패턴 도입 검토 시
- 3 워커 동시 CI 인프라 변경 시

### Prerequisites
- [Journal 006](/wiki/harness-engineering/aidy-journal-006-ios-ci-self-hosted-runner-migration) — ADR-009 iOS self-hosted 선행
- [Journal 003](/wiki/harness-engineering/aidy-journal-003-parallel-dispatch-token-economics) — send-seq 직렬 dispatch + 토큰 경제성
- [tmux flush 자동화](/wiki/harness-engineering/tmux-flush-automation-pattern) — 크로스 워커 tmux 알림 채널

### Actionable Steps

1. **Hybrid fallback 설계 시 `needs.result` 사용 금지** — `continue-on-error` 마스킹 때문. 반드시 Mark-step 패턴(`outputs.passed` + `if: always() && != 'true'`)
2. **workflow 패턴 spec 작성 후 1 워커 실행 → 실증 → spec 확정** — spec-first-verify-first. WO-016처럼 실행해봐야 드러나는 버그가 있다
3. **self-hosted runner 추가 시 프로젝트 빌드 설정 1회 읽기** — `build.gradle.kts`의 `jvmTarget`/`compileSdk`, `Package.swift`의 tools version 등. spec 전제와 실제 불일치 방지
4. **multi-worker dispatch는 `send-seq` 직렬** — billing/rate-limit 리스크 회피. 1 워커 green 확인 후 나머지
5. **인프라 변경 후 모든 워크플로 스캔** — Journal 006 교훈 재적용. `ci-status.sh --watch`로 모든 워크플로 fail 탐지
6. **tmux 알림으로 크로스 워커 긴급 전파** — spec 정정/버그 발견 시 다른 워커에 즉시 알림

### Anti-patterns

- **`needs.result`로 fallback 트리거** — `continue-on-error` 마스킹으로 영원히 skipped
- **spec dry-run 없이 전체 워커 동시 dispatch** — 같은 버그 N회 반복
- **프로젝트 빌드 설정 미확인 상태로 환경 spec 작성** — JDK/SDK 버전 미스매치
- **self-hosted only로 Linux 워커 전환** — 무료 쿼터 2000분/월 낭비
- **크로스 워커 전파 채널 없이 multi-agent 운영** — 동일 버그 N회 반복
- **`actions/cache` 무조건 사용** — self-hosted에서 15분 hang 발생 가능. 로컬 캐시면 불필요
