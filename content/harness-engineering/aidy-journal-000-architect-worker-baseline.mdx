## 시리즈를 시작하며

Harness Journal(000~023)이 *단일 프로젝트 내 AI 운영 환경 개선*을 박제했다면, Aidy Journal은 한 단계 위 — **멀티레포 멀티플랫폼 AI 오케스트레이션**을 박제하는 시리즈다.

Aidy는 AI 다이어리 앱으로, 4개 레포가 동시에 움직인다:

| 레포 | 역할 | 스택 |
|------|------|------|
| `aidy-architect` | 컨트롤 센터 (스펙/WO/게이트/문서) | Claude Code 세션 |
| `aidy-server` | 백엔드 워커 | Spring Boot 3.5 + Kotlin + PostgreSQL |
| `aidy-ios` | iOS 워커 | Tuist + TCA + SwiftUI |
| `aidy-android` | Android 워커 | Jetpack Compose + MVVM + Kotlin |

핵심 질문: **4개 독립 Claude 세션이 동시에 코드를 작성할 때, 어떻게 동기화하고 품질을 보장하는가?**

이 0번 엔트리는 그 답의 베이스라인이다.

## 2. API Contract — 코드가 아니라 문서가 Source of Truth

일반 프로젝트: 코드가 source of truth → 문서가 뒤따라감 (항상 stale)
Aidy 모델: **api-contract.md가 source of truth** → 코드가 이를 구현 → Gate가 검증

### 계약 내용 (v0.2 — JWT 인증 활성화 후)

- Base URL, 인증: Bearer JWT (v0.1의 X-User-Id에서 마이그레이션 완료, ADR-006)
- **10개 엔드포인트**:
  - Auth: `POST /api/auth/signup`, `POST /api/auth/login`
  - Chat: `POST /api/chat`, `GET /api/chat/history`
  - Memory: `GET /api/memories`, `GET /api/memories/categories`, `GET /api/memories/search`, `GET /api/memories/people`, `POST /api/memories/{id}/feedback`
  - Health: `GET /api/health`
- Request/Response 스키마: 필드명과 타입 정확히 명시
- Error Codes 테이블: 8개 (EMPTY_MESSAGE, INVALID_CREDENTIALS, UNAUTHORIZED, ...)
- Memory 카테고리 enum: 7개 고정값 (schedule, finance, work, health, preference, people, general)

### 왜 이게 중요한가

서버가 `memoriesExtracted: MemoryItem[]`로 반환하는데 iOS가 `memories: [Memory]`로 모델링하면? **Gate 1 FAIL**. 이름이 다르면 파싱이 깨지고, 프로덕션에서 크래시가 난다.

계약을 중심에 놓으면 3개 플랫폼이 *동시에 독립적으로* 구현해도 통합 시점에 깨지지 않는다.

## 4. Gate 검증 프로토콜

### Gate 1: 스펙 준수 (라인별 코드 리뷰)

Architect가 워커의 git diff를 추출하여 api-contract.md와 **라인별** 대조:

| 체크 항목 | 검증 방법 |
|-----------|----------|
| 엔드포인트 URL | `@GetMapping`/`@PostMapping` vs 계약 |
| HTTP 메서드 | GET/POST/PUT/DELETE 일치 |
| Request body 필드명/타입 | DTO 필드 vs 계약 스키마 |
| Response body 필드명/타입 | DTO 필드 vs 계약 스키마 |
| Error codes | 계약의 Error Codes 테이블에 있는 것만 |
| 네이밍 컨벤션 | conventions.md 준수 |
| 시크릿 하드코딩 | grep으로 탐지 |

**결과**: PASS / CONDITIONAL (사소한 이슈) / FAIL (반드시 수정)

실전 예시 — WO-002 iOS 1차 Gate 1 FAIL:
- categories, health 엔드포인트 미구현
- 에러 파싱 미구현
- → 워커 재작업 → commit `6f78358` → 3건 수정 → Gate 1 재통과

### Gate 2: 통합 검증

Gate 1 PASS 후 실행. 3가지 축:

1. **빌드**: `./gradlew clean build` / `tuist build` / `./gradlew assembleDebug`
2. **테스트**: 각 플랫폼 테스트 스위트 실행
3. **크로스 플랫폼 필드 동기화**: server DTO vs iOS Model vs Android data class — 필드명 정확히 일치하는지 교차 비교

## 5. Cross-Session Review — 메타데이터 불신 원칙

다른 Claude 세션의 작업을 검증할 때, **절대 신뢰하지 않는 것**:

- 커밋 메시지
- PR 설명
- 코드 주석 ("I tested this")
- 타임스탬프

**대신 하는 것**:

1. `git diff HEAD~1` — 실제 diff만 추출
2. **4 Trap 탐색**:
   - 스펙에 없는 엔드포인트/필드 추가됨?
   - 선언만 되고 호출 없는 데드코드?
   - 계약에 없는 에러코드 사용?
   - 하드코딩된 시크릿/API 키?
3. 로컬 빌드+테스트로 CI 누락 잡기
4. 마크다운 리포트 → `gates/reviews/`에 아카이브

## 6. Inbox 비동기 통신

워커가 막히면 `aidy-architect/inbox/{platform}-request.md`에 요청 작성:

```markdown
# iOS 워커 요청
**시각**: 14:32
**유형**: 스펙변경 | 권한 | 터미널명령 | 질문 | 블로커
**내용**: DELETE /api/memories/{id}의 응답이 200인지 204인지 계약에 명시 안 됨
**현재상태**: Memory 삭제 구현 중, 이 결정 없이 진행 불가
```

워커는 요청 작성 후 **작업 중단**. Architect가 `inbox/{platform}-response.md`에 응답을 쓰면 워커가 읽고 재개.

## 7. Compound Flywheel — 지식이 순환하는 구조

```
WO 완료 + Gate 2 PASS
    ↓
/compound 실행
    ↓
3개 산출물:
├── docs/retros/WO-{num}-retro.md     ← 회고
├── docs/solutions/{date}-{slug}.md    ← 재사용 가능한 해결책
└── specs/decisions/ADR-{num}.md       ← 아키텍처 결정 (해당 시)
    ↓
다음 라운드의 /autoceo Research 단계가 이 문서들을 입력으로 읽음
    ↓
더 구체적인 Plan → 더 빠른 구현 → 더 나은 Compound
    ↓
반복 (라운드마다 품질 향상)
```

**핵심**: 회고의 "For AI Agents" 섹션은 사람이 아니라 *다음 Claude 세션*을 위한 것. 정확한 명령, 엣지 케이스, 검증된 코드 스니펫을 담아서 다음 세션이 같은 실수를 반복하지 않게 한다.

## 8. autoceo — 풀 자동 루프

`/autoceo N`으로 N라운드 자동 실행:

| 단계 | 내용 |
|------|------|
| Research | TODOs, 실패한 게이트, 이전 회고 분석 |
| Plan | 작업 결정, WO 생성, api-contract 변경 필요 시 수정 |
| Dispatch | 3개 워커에 동시 전송 |
| QA | /cross-session-review + /gate-1 + 빌드 체크 |
| Compound | 회고 + 솔루션 생성 |

**안전장치**:
- 라운드별 git tag 체크포인트 (`autoceo-round-N-before`)
- 보호 파일: api-contract.md, settings.json, .env (autoceo가 수정 불가)
- 커밋당 최대 10 파일
- QA 3회 실패 시 해당 워커 롤백
- 패키지 설치/의존성 변경 금지

## 10. 일반 CI/CD와의 차이

| 관점 | 일반 CI/CD | Aidy Architect-Worker |
|------|------------|----------------------|
| Source of Truth | 코드 | api-contract.md (문서) |

### 20라운드에서 확인한 것

1. **Spec-First가 멀티플랫폼 동기화 문제를 구조적으로 해결한다** — 3개 워커가 독립적으로 작업해도 Gate 2에서 필드 불일치를 잡는다
2. **Gate 1 FAIL은 초기에만 발생하고 이후 0건** — 워커(Claude 세션)가 계약을 숙지하면서 1차 통과율이 자연스럽게 올라감. 시스템이 *학습*하는 구조
3. **ADR에 '보류'를 명시하는 것도 아키텍처 결정** — ADR-005의 Phase 2 보류가 Phase 1 집중을 가능하게 했고, normalizedName 실전 검증을 Phase 2 착수 조건으로 써두니 autoceo가 함부로 Phase 2를 시작하지 않음
4. **기능 스프린트와 보안 스프린트를 분리하면 Gate 검증 초점이 명확해진다** — Sprint 1은 "동작하는가", Sprint 2는 "안전한가"
5. **Compound Engineering은 언어를 넘어 이식된다** — moneyflow/tarosaju의 5-Layer 검증이 TypeScript → Kotlin(Spring Boot)으로 패턴 자체가 그대로 이식

## AI Agent Directive

### Trigger
- 멀티레포 프로젝트에서 AI 에이전트 간 동기화 문제가 발생할 때
- 새 멀티플랫폼 프로젝트의 AI 운영 환경을 설계할 때
- 기존 단일 레포 Harness Journal 패턴을 멀티레포로 확장할 때

### Prerequisites
- [Harness Journal 000](/wiki/harness-engineering/harness-journal-000-baseline) — 단일 프로젝트 베이스라인 패턴
- [Compound Engineering Philosophy](/wiki/harness-engineering/compound-engineering-philosophy) — 왜 지식이 순환해야 하는가

### Actionable Steps
1. **api-contract.md 작성**: 모든 엔드포인트, 필드, 에러코드를 한 문서에 명시. 이 문서가 source of truth
2. **conventions.md 작성**: URL(kebab), JSON(camelCase), Error(UPPER_SNAKE) 등 플랫폼 공통 규약
3. **Work Order 템플릿 생성**: backlog/in-progress/done 디렉토리 + WO 파일 포맷
4. **Gate 체크리스트 정의**: 플랫폼별 검증 항목 (엔드포인트, 필드, 에러코드, 보안)
5. **Inbox 통신 규약 설정**: request/response 파일 포맷 + 유효한 요청 유형 정의
6. **Compound 산출물 규약**: 회고/솔루션/ADR 템플릿 + "For AI Agents" 섹션 필수

### Anti-patterns
- **코드를 source of truth로 두기** — 3개 플랫폼이 각자 "자기 코드가 맞다"고 주장하면 통합 불가
- **커밋 메시지만으로 검증** — 다른 세션의 AI가 정직해도 누락은 발생. diff 기반 검증 필수
- **iOS/Android를 다른 라운드에서 다른 화면 구현** — 드리프트 발생, ADR-003의 교훈
- **회고 생략** — Compound Flywheel이 끊기면 다음 라운드 품질이 올라가지 않음
