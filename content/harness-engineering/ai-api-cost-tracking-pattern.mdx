## 갭 (현시점)

[Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)에서 발견된 **공통 갭 #4**:

> AI API 비용 추적 0 — 양쪽 모두 호출 로깅·일/월 리포트 부재. *"이번 달 얼마 썼는지 모름"* 상태. 수익화 직전 단계라 더 시급.

이 글은 양쪽 모두 미적용 상태에서 *제안 엔트리*로 작성. 실제 적용은 다음 사이클(Harness Journal 006~009 큐). 패턴이 박제돼 있어야 적용 시 *처음부터 만들지 않고* 이식할 수 있다.

## 패턴 정의 — 4 컴포넌트

### 컴포넌트 1: 모델별 단가 표 (코드 한 곳에 박음)

```typescript
// src/lib/ai-cost-pricing.ts
// 가격은 1M tokens 기준 USD. 분기마다 검증 필요.

export const PRICING = {
  // Claude (Anthropic) — cached input은 uncached의 10% 가격
  'claude-sonnet-4-6': {
    input:  3.00,       // /1M tokens
    cached: 0.30,       // /1M tokens (cache hit)
    output: 15.00,
  },
  'claude-haiku-4-5': {
    input:  0.80,
    cached: 0.08,
    output: 4.00,
  },
  'claude-opus-4-6': {
    input:  15.00,
    cached: 1.50,
    output: 75.00,
  },

  // Gemini (Google)
  'gemini-2.5-pro': {
    input:  1.25,
    output: 5.00,
  },
  'gemini-2.5-flash': {
    input:  0.075,
    output: 0.30,
  },

  // OpenAI
  'gpt-4o': {
    input:  2.50,
    cached: 1.25,       // OpenAI 자동 캐싱 (50% 할인)
    output: 10.00,
  },
  'gpt-4o-mini': {
    input:  0.15,
    cached: 0.075,
    output: 0.60,
  },
} as const;

export function estimateCost(
  model: keyof typeof PRICING,
  usage: { input: number; cached?: number; output: number; thinking?: number }
): number {
  const p = PRICING[model];
  if (!p) return 0;

  const inputCost = ((usage.input - (usage.cached || 0)) / 1_000_000) * p.input;
  const cachedCost = p.cached ? ((usage.cached || 0) / 1_000_000) * p.cached : 0;
  const outputCost = (usage.output / 1_000_000) * p.output;
  const thinkingCost = usage.thinking
    ? (usage.thinking / 1_000_000) * p.output  // thinking은 output처럼 청구
    : 0;

  return inputCost + cachedCost + outputCost + thinkingCost;
}
```

핵심 결정:
- **단일 source of truth** — 여러 곳에 가격이 흩어지면 분기 시 update 누락
- **cached 분리** — 캐시 히트가 작동하는지 측정 가능
- **thinking tokens** — Gemini Thinking 모델은 별도 청구

### 컴포넌트 2: 호출 시점 로거 (ai-client에 통합)

```typescript
// src/lib/ai-client-with-cost.ts
import { logApiCall } from './ai-cost-logger';

export async function callAI(opts: CallOpts) {
  const start = Date.now();
  let usage: TokenUsage | null = null;
  let error: string | null = null;

  try {
    const result = await actualCall(opts);
    usage = result.usage;
    return result;
  } catch (err) {
    error = err.message;
    throw err;
  } finally {
    // 성공/실패 모두 로깅
    if (usage || error) {
      await logApiCall({
        timestamp: new Date().toISOString(),
        model: opts.model,
        agentRole: opts.agentRole,  // 어떤 에이전트가 호출했는지
        userId: opts.userId || null,
        input: usage?.input || 0,
        cached: usage?.cached || 0,
        output: usage?.output || 0,
        thinking: usage?.thinking || 0,
        cost: usage ? estimateCost(opts.model, usage) : 0,
        latencyMs: Date.now() - start,
        error,
      });
    }
  }
}
```

핵심:
- **try/finally로 실패 호출도 로깅** — 비용은 실패해도 발생할 수 있고, 어떤 호출이 자주 실패하는지가 중요
- **에이전트별 분류** — 어떤 에이전트가 가장 비싼지 측정 가능
- **userId 분류** — 사용자당 비용 추적 (수익화 단계)

### 컴포넌트 3: Supabase 저장 + 일별 집계 뷰

```sql
-- supabase/migrations/YYYYMMDD_api_cost_tracking.sql

CREATE TABLE api_calls_log (
  id BIGSERIAL PRIMARY KEY,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  model TEXT NOT NULL,
  agent_role TEXT,
  user_id UUID REFERENCES auth.users(id),
  input_tokens INTEGER NOT NULL DEFAULT 0,
  cached_tokens INTEGER NOT NULL DEFAULT 0,
  output_tokens INTEGER NOT NULL DEFAULT 0,
  thinking_tokens INTEGER NOT NULL DEFAULT 0,
  cost_usd NUMERIC(10, 6) NOT NULL,
  latency_ms INTEGER,
  error TEXT
);

CREATE INDEX api_calls_log_created_at ON api_calls_log (created_at DESC);
CREATE INDEX api_calls_log_model ON api_calls_log (model);
CREATE INDEX api_calls_log_user_id ON api_calls_log (user_id) WHERE user_id IS NOT NULL;

-- 일별 집계 뷰 (조회 빠르게)
CREATE OR REPLACE VIEW api_cost_daily AS
SELECT
  DATE(created_at) AS date,
  model,
  COUNT(*) AS calls,
  SUM(input_tokens) AS input_tokens,
  SUM(cached_tokens) AS cached_tokens,
  SUM(output_tokens) AS output_tokens,
  SUM(cost_usd) AS cost_usd,
  AVG(latency_ms) AS avg_latency_ms,
  COUNT(*) FILTER (WHERE error IS NOT NULL) AS error_count
FROM api_calls_log
GROUP BY DATE(created_at), model
ORDER BY date DESC, cost_usd DESC;

-- RLS
ALTER TABLE api_calls_log ENABLE ROW LEVEL SECURITY;

-- service role만 INSERT
CREATE POLICY "Service role insert" ON api_calls_log
  FOR INSERT WITH CHECK (auth.jwt()->>'role' = 'service_role');

-- admin만 조회
CREATE POLICY "Admin select" ON api_calls_log
  FOR SELECT USING (auth.jwt()->>'role' = 'service_role');
```

핵심:
- **api_calls_log는 raw, api_cost_daily는 집계 view** — 조회 빠르게
- **RLS** — 사용자 데이터라 권한 격리 필수
- **error_count** — 실패율 동시 측정

## 양쪽 프로젝트 적용 가이드

이 패턴은 양쪽에 *동일한 형태로* 적용 가능. moneyflow는 ai-client.ts에 통합, tarosaju는 fortune/ai와 tarot-chat 두 라우트에 통합.

### moneyflow 적용 step

1. `src/lib/ai-cost-pricing.ts` 신설 (위 컴포넌트 1)
2. `src/lib/trading/ai-client.ts`의 `callGemini`, `callClaude`, `callOpenAI`에 try/finally + logger 통합 (컴포넌트 2)
3. supabase migration 신설 (컴포넌트 3)
4. `.claude/commands/cost-check.md` 신설 (컴포넌트 4)
5. CLAUDE.md skill routing 추가
6. 첫 24시간 후 `/cost-check`로 데이터 검증

### tarosaju 적용 step

1~5는 moneyflow와 동일. *단*, tarosaju는 단일 모델(Claude Haiku)이라 모델 라우팅 부분은 단순. 그러나 *사용자별 비용*은 더 중요 (점술 사용자 구분).

## 부작용 검토

- **DB 쓰기 오버헤드**: 매 호출당 INSERT 1회. *비동기 로깅* (await 없이 fire-and-forget)으로 응답 latency에 영향 0
- **RLS 정책**: 사용자 데이터라 service_role 격리 필수. 잘못 설정하면 *다른 사용자의 비용 조회 가능* 보안 사고
- **분기마다 가격 표 검증**: Anthropic/Gemini/OpenAI가 가격을 변경하면 표를 업데이트해야 함. 자동화 안 되면 *측정값과 실제 청구의 갭*

## AI Agent Directive

**Trigger**: 프로덕션 AI API 호출 시스템에서 비용 가시성이 부족한 상태 / 사용량 기반 가격 책정 단계

**Prerequisites**:
- [AI 호출 패턴 — Circuit Breaker / 폴백](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback)
- 프로젝트의 데이터베이스(Supabase 또는 유사)

### Actionable Steps

1. **모델별 단가 표 생성** (`src/lib/ai-cost-pricing.ts`) — 모든 사용 모델(Claude/Gemini/OpenAI)의 현재 가격 정리, 1M tokens 기준 USD로 기록
2. **AI 호출 wrapper에 로깅 추가** — 호출 시점에 `logApiCall()` 비동기 호출 (응답 latency 영향 0)
3. **데이터베이스 테이블 설계** — `api_calls_log` raw 테이블 + `api_cost_daily` 집계 뷰
4. **`/cost-check` 슬래시 커맨드 구현** — 일/월 누적 비용 + 모델별 분류 + 임계치 경고
5. **주간 단가 표 검증** — Anthropic/Google/OpenAI 공식 문서와 대조 (분기마다 가격 변경 확인)

### Anti-patterns

- ❌ 여러 파일에 가격 하드코딩 (분기 update 누락 가능성)
- ❌ 모든 호출을 비동기 await (응답 latency 증가)
- ❌ 사용자별 비용 추적 없이 프로덕션 가격 책정 (수익화 불가능)

## 자기 점검

1. *지난 한 달*의 AI API 비용을 *모델별로* 답할 수 있는가?
2. 답할 수 없다면 — 어떤 모델/에이전트가 가장 비싼지 *추측*인가 *측정*인가?
3. 캐싱이 *작동하는지* 어떻게 확인하는가? 비용 차이 외에 다른 방법이 있는가?
4. 사용자당 평균 비용은? 그 위에 *가격을 책정*할 수 있는가?
5. (열린 질문) 비용 추적을 *적용한 직후*에 가장 충격적인 발견은 무엇일 것 같은가?

### 실습 과제

자신의 AI 호출 함수에 *최소 버전 로깅*을 30분 안에 추가:

```typescript
// 비동기 fire-and-forget 로깅
async function callAI(opts) {
  const result = await actualCall(opts);
  // 응답 latency에 영향 0
  void logApiCall({
    model: opts.model,
    cost: estimateCost(opts.model, result.usage),
  });
  return result;
}
```

DB 없으면 *콘솔 로그*만으로도 시작 — 24시간 후 어떤 모델/에이전트가 비싼지 보임.

## 출처

이 엔트리는 *제안* — 양쪽 프로젝트 모두 미적용 상태에서 패턴 박제. 다음 사이클(Journal 006~009 큐)의 입력.

- 가격 표: Anthropic / Google / OpenAI 공식 페이지 (분기마다 검증 필요)
- 패턴 구조: moneyflow `ai-client.ts` 패턴(이미 구현된 Circuit Breaker 등)에 *비용 로깅 step만 추가*하는 형식
- 적용 후 측정 데이터는 다음 Journal 사이클에서 박제

### 검증 메모

- 가격 표의 USD 수치는 *공식 페이지 기준 추정값*. 실제 청구와 정확히 일치하지 않을 수 있음 (지역 할인, 약정 할인 등)
- 코드 스니펫은 *직접 작성한 패턴*이며, 양쪽 프로젝트의 실제 구현은 아직 없음
- "수익화 단계가 가깝다"는 사용자의 두 프로젝트 운영 맥락에서의 추정이며 정확한 시점은 미정
