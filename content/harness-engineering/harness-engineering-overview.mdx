## 핵심 개념

AI 하네스(Harness)란 **LLM을 프로덕션에서 안전하고 효율적으로 사용하기 위해 감싸는 소프트웨어 계층**이다.

LLM API를 직접 호출하는 것은 프로토타입에서는 충분하지만, 프로덕션에서는 부족하다. 입력 검증, 출력 파싱, 에러 처리, 비용 제어, 품질 보장이 모두 필요하다.

```
┌─────────────────────────────────────────────────┐
│                  AI Harness                       │
│                                                   │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐     │
│  │ 입력 검증 │ → │ Context  │ → │  LLM     │     │
│  │ + 라우팅  │   │ Assembly │   │  호출    │     │
│  └──────────┘   └──────────┘   └──────────┘     │
│       ↑                              │           │
│       │         ┌──────────┐         ↓           │
│       │         │ 캐시     │   ┌──────────┐     │
│       │         └──────────┘   │ 출력 검증 │     │
│       │                        │ + 파싱    │     │
│       │                        └──────────┘     │
│       │                              │           │
│       └──────── 재시도 / 폴백 ◄──────┘           │
└─────────────────────────────────────────────────┘
```

## 왜 하네스가 필요한가

| LLM 직접 호출 | 하네스를 통한 호출 |
|--------------|------------------|
| 할루시네이션 그대로 노출 | 출력 검증 + 필터링 |
| API 장애 시 서비스 중단 | 폴백 모델로 자동 전환 |
| 불필요한 API 호출 = 비용 증가 | 시맨틱 캐싱으로 비용 절감 |
| 느린 응답 그대로 전달 | 스트리밍 + 타임아웃 관리 |
| 프롬프트 인젝션에 취약 | 입력 필터링 + 가드레일 |

## 하네스의 5가지 핵심 계층

### 1. 입력 계층 (Input Layer)

사용자 입력을 검증하고, 적절한 처리 파이프라인으로 라우팅한다.

```typescript
async function processInput(userMessage: string) {
  // 1. 입력 검증
  if (userMessage.length > MAX_INPUT_LENGTH) {
    throw new InputTooLongError();
  }

  // 2. 프롬프트 인젝션 감지
  if (detectInjection(userMessage)) {
    return { type: 'blocked', reason: 'injection_detected' };
  }

  // 3. 의도 분류 → 파이프라인 라우팅
  const intent = await classifyIntent(userMessage);
  return routeToPipeline(intent, userMessage);
}
```

### 2. 컨텍스트 조립 (Context Assembly)

사용자의 질문에 필요한 맥락을 수집하고 구조화한다. Context Engineering의 실행 단계.

```typescript
async function assembleContext(query: string, user: User) {
  const [docs, history, rules] = await Promise.all([
    vectorSearch(query, { topK: 5 }),
    getChatHistory(user.id, { limit: 10 }),
    getDomainRules(user.domain),
  ]);

  return buildPrompt({ docs, history, rules, query });
}
```

### 3. LLM 호출 계층 (Inference Layer)

실제 모델 호출. 타임아웃, 재시도, 폴백을 관리한다.

```typescript
async function callLLM(prompt: string, options: LLMOptions) {
  try {
    // 1차: 주 모델
    return await callWithTimeout(primaryModel, prompt, { timeout: 10000 });
  } catch (err) {
    if (err instanceof TimeoutError) {
      // 폴백: 더 빠른 모델로 전환
      return await callWithTimeout(fallbackModel, prompt, { timeout: 5000 });
    }
    throw err;
  }
}
```

### 4. 출력 계층 (Output Layer)

LLM 응답을 검증하고 파싱한다.

```typescript
async function processOutput(rawResponse: string, schema: ZodSchema) {
  // 1. 구조 파싱
  const parsed = safeJsonParse(rawResponse);
  if (!parsed) {
    return await retryWithFix(rawResponse, schema);
  }

  // 2. 스키마 검증
  const result = schema.safeParse(parsed);
  if (!result.success) {
    return await retryWithFix(rawResponse, schema);
  }

  // 3. 콘텐츠 필터링
  if (containsBlockedContent(result.data)) {
    return { blocked: true, reason: 'content_policy' };
  }

  return result.data;
}
```

### 5. 관측 계층 (Observability Layer)

모든 호출을 추적하고 모니터링한다.

```typescript
async function tracedCall(fn: Function, metadata: TraceMetadata) {
  const span = tracer.startSpan('llm.call', metadata);
  const start = Date.now();
  try {
    const result = await fn();
    span.setAttributes({
      'llm.tokens.input': result.usage.inputTokens,
      'llm.tokens.output': result.usage.outputTokens,
      'llm.latency_ms': Date.now() - start,
    });
    return result;
  } catch (err) {
    span.setStatus({ code: SpanStatusCode.ERROR });
    throw err;
  } finally {
    span.end();
  }
}
```

## AI Agent Directive

**Trigger**: AI 모델을 호출하는 새로운 코드를 작성하거나, 기존 LLM 호출 코드가 *단순 API 호출 수준*일 때. 프로덕션급 AI 시스템이 필요할 때.

**Prerequisites**:
- [Context Engineering Fundamentals](/wiki/context-engineering/context-engineering-fundamentals)
- [AI 호출 패턴 - Circuit Breaker/폴백/캐싱](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback) (선택)

### Actionable Steps
1. 5계층 하네스 구조 이해: Input → Context Assembly → Inference → Output → Observability
2. **입력 계층** — Zod 스키마로 입력 검증 + 프롬프트 인젝션 감지 + 의도 분류 라우팅
3. **컨텍스트 조립** — vectorSearch + history + domain rules 병렬 수집 후 `buildPrompt()`
4. **LLM 호출 계층** — timeout (60s) + retry (2회 지수 백오프) + 폴백 모델 구현
5. **출력 계층** — `safeJsonParse` → Zod schema.parse → 범위 검증 → 콘텐츠 필터
6. **관측 계층** — tracer.startSpan + 토큰 수 + latency + 에러 기록
7. 각 계층을 *독립 함수*로 분리 (테스트 + 재사용 가능)
8. 한 계층의 실패가 다른 계층에 전파되지 않게 설계

### Anti-patterns
- ❌ LLM API를 *직접 호출* (wrapper 없이)
- ❌ 입력 검증만 있고 *출력 검증* 없음
- ❌ Circuit Breaker 없이 retry만 (실패한 서비스에 더 많은 트래픽)
- ❌ 관측 계층 생략 (에러가 어디서 발생하는지 모름)
- ❌ maxTokens를 기본값 4096으로 둠 (비용 낭비)

## For AI Agents

**Trigger**: AI 호출 코드를 새로 작성하거나 기존 코드가 *단순 API 호출*에 머물러 있을 때

**Prerequisites**:
- [AI Agent Start Here](/wiki/harness-engineering/ai-agent-start-here)

**Actionable Steps (5 계층 하네스 구축)**:
1. **입력 계층** — Zod 스키마로 입력 검증 + 프롬프트 인젝션 감지 + 의도 분류 라우팅
2. **컨텍스트 조립** — vectorSearch + chat history + domain rules 병렬 수집 후 `buildPrompt()`
3. **LLM 호출 계층** — timeout (60s) + retry (2회 지수 백오프) + 폴백 모델 (AI 호출 패턴 엔트리 참조)
4. **출력 계층** — `safeJsonParse` → Zod schema.parse → 범위 검증 → 콘텐츠 필터 (5 layer, Zod 엔트리 참조)
5. **관측 계층** — tracer.startSpan + 토큰 수 + latency + 에러 기록

각 계층은 *독립 함수*로 분리해서 테스트 가능하게. 한 계층의 실패가 다른 계층에 전파되지 않게.

**Anti-patterns**:
- ❌ LLM API를 *직접 호출* (wrapper 없이). timeout 없으면 무한 대기, 폴백 없으면 SPOF
- ❌ 입력 검증만 있고 *출력 검증* 없음 (환각이 그대로 통과)
- ❌ Circuit Breaker 없이 retry만 — 실패한 서비스에 더 많은 트래픽
- ❌ 관측 계층 생략 — 에러가 *어디서* 발생하는지 모름
- ❌ maxTokens 기본값 4096 그대로 두기 (비용 낭비)
