## 갭 (현시점)

[Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)에서 발견된 **공통 갭 #2**:

> AI 출력 Zod 검증 부재 — 입력은 검증하지만 LLM 응답을 schema.parse로 거르지 않음. moneyflow 13 에이전트, tarosaju fortune/tarot-chat 모두. 환각이 그대로 통과.

### 양쪽 프로젝트의 *현재 상태* (audit 결과)

| 항목 | moneyflow | tarosaju |
|---|---|---|
| 입력 Zod 검증 | ❌ (호출자 책임) | ✅ `src/lib/schemas/api.ts` |
| **출력 Zod 검증** | ❌ | ❌ |
| 출력 형식 강제 | ✅ JSON Schema (Gemini), JSON mode (OpenAI) | ⚠️ Anthropic은 prompt-based만 |
| 범위 검증 | ❌ | ❌ |
| 내부 일관성 검증 | ❌ | ❌ |

입력 검증과 출력 검증은 *완전히 다른 작업*. 입력은 *받지 말 것*을 거르고, 출력은 *내보내지 말 것*을 거른다. 두 번째가 거의 항상 빠진다.

## 왜 출력 검증이 가장 어려운가

LLM 출력의 본질적 함정:

| 함정 | 어떻게 발생 | 형식 검증으로 잡히는가 |
|---|---|---|
| **JSON 형식 오류** | 마지막 콤마, 따옴표 누락 | ✅ |
| **타입 불일치** | string인데 숫자, array인데 object | ✅ (Zod) |
| **범위 환각** | 주가 $150인데 목표가 $15,000 | ❌ — Zod 기본형으론 안 잡힘 |
| **enum 외 값** | rating에 'BUY!'(원래 'BUY') 같은 변형 | ⚠️ Zod enum으론 잡힘 |
| **내부 모순** | STRONG_BUY + confidence 30 | ❌ — refine 필요 |
| **입력 무시** | 사용자가 'Apple 분석'을 요청했는데 응답이 'Microsoft' | ❌ — cross-input 검증 필요 |
| **신뢰도 과신** | 모든 응답에 confidence 95+ | ❌ — 통계적 보정 필요 |

LLM은 *자신감 있게 틀린다*. 문법적으로 valid한 JSON 안에 의미적 환각이 들어 있다. **출력 검증은 형식 + 범위 + 일관성 + 입력 정합성 + 통계 보정 5 layer**가 필요하다.

## 5 Layer 출력 검증 패턴

### Layer 1 — JSON 파싱 (구조)

```typescript
import { z } from 'zod';

function safeJsonParse<T>(raw: string, schema: z.ZodSchema<T>): T | null {
  // 1순위: 직접 파싱
  try {
    const parsed = JSON.parse(raw);
    return schema.parse(parsed);
  } catch {}

  // 2순위: 첫 {...} 추출 (Claude prompt-based의 흔한 case)
  const match = raw.match(/\{[\s\S]*\}/);
  if (match) {
    try {
      const parsed = JSON.parse(match[0]);
      return schema.parse(parsed);
    } catch {}
  }

  // 3순위: ```json ... ``` 코드 블록 추출
  const codeBlock = raw.match(/```(?:json)?\s*([\s\S]*?)```/);
  if (codeBlock) {
    try {
      const parsed = JSON.parse(codeBlock[1]);
      return schema.parse(parsed);
    } catch {}
  }

  return null;
}
```

핵심: *3단 fallback*. Gemini/OpenAI는 1순위로 끝나지만 Claude prompt-based는 2-3순위가 자주 작동.

### Layer 2 — Zod 스키마 (타입)

```typescript
const TradingAgentOutputSchema = z.object({
  rating: z.enum(['STRONG_BUY', 'BUY', 'HOLD', 'SELL', 'STRONG_SELL']),
  confidence: z.number().min(0).max(100),
  target_price: z.number().positive(),
  stop_loss_pct: z.number().min(0).max(50),
  reasoning: z.string().min(50).max(2000),
  signals: z.array(z.string()).min(1).max(10),
});
```

핵심: *모든 필드에 제약*. `z.string()`만 쓰지 말고 `z.string().min().max()` 같은 *경계 명시*. enum은 *허용 값을 모두 나열*해서 *변형 환각*('BUY!', 'STRONG-BUY' 등)을 차단.

### Layer 3 — 범위 / Domain 제약

```typescript
const TradingAgentOutputSchema = z.object({
  // ... layer 2 ...
})
.refine(
  (data) => data.target_price > 0 && data.target_price < 100000,
  { message: 'target_price must be between 0 and 100000 USD' }
)
.refine(
  (data) => data.stop_loss_pct >= 0.1 && data.stop_loss_pct <= 30,
  { message: 'stop_loss_pct must be 0.1% to 30%' }
);
```

핵심: *현실적 범위*를 명시. 주가가 $1M까지 갈 수는 있지만 *대부분의 종목*은 $0~$100k. *극단값*은 환각의 신호.

### Layer 4 — Cross-field consistency (내부 일관성)

```typescript
const TradingAgentOutputSchema = z.object({
  // ... ...
})
.refine(
  (data) => {
    // STRONG_BUY/BUY는 confidence 50+ 필요
    if (['STRONG_BUY', 'BUY'].includes(data.rating)) {
      return data.confidence >= 50;
    }
    // STRONG_SELL/SELL도 confidence 50+ 필요
    if (['STRONG_SELL', 'SELL'].includes(data.rating)) {
      return data.confidence >= 50;
    }
    // HOLD는 어떤 confidence도 OK
    return true;
  },
  { message: 'High-conviction rating requires confidence >= 50' }
)
.refine(
  (data) => {
    // STRONG_BUY는 target_price가 적어도 reasoning에 *언급*되어야
    if (data.rating === 'STRONG_BUY') {
      const priceStr = data.target_price.toFixed(0);
      return data.reasoning.includes(priceStr) || data.reasoning.includes('$');
    }
    return true;
  },
  { message: 'STRONG_BUY requires target_price reference in reasoning' }
);
```

핵심: *각 필드는 valid이지만 함께는 모순*인 케이스. LLM이 이걸 가장 자주 만든다 — 한 필드를 만들 때 다른 필드의 함의를 잊는다.

### Layer 5 — Cross-input consistency (입력 데이터와의 일치)

```typescript
async function validateAgainstInput(
  output: TradingAgentOutput,
  input: { symbol: string; current_price: number; pe_ratio: number }
): Promise<TradingAgentOutput> {
  // target_price가 current_price 대비 너무 멀면 환각
  const priceChange = (output.target_price - input.current_price) / input.current_price;
  if (Math.abs(priceChange) > 5) {
    // 500% 이상 변동은 의심
    throw new Error(
      `target_price ${output.target_price} too far from current ${input.current_price}`
    );
  }

  // reasoning이 input symbol을 언급하지 않으면 환각 (다른 종목 분석)
  if (!output.reasoning.toLowerCase().includes(input.symbol.toLowerCase())) {
    throw new Error(
      `reasoning does not reference input symbol ${input.symbol}`
    );
  }

  // PE ratio가 input과 다른 값을 *직접 인용*하면 환각
  const peStr = input.pe_ratio.toFixed(1);
  const peMatch = output.reasoning.match(/PE[:\s]*?([\d.]+)/i);
  if (peMatch && peMatch[1] !== peStr) {
    console.warn(
      `PE ratio in reasoning (${peMatch[1]}) differs from input (${peStr})`
    );
  }

  return output;
}
```

핵심: *LLM이 입력을 무시하거나 자기가 만든 숫자를 인용*하는 환각. 가장 미묘하지만 가장 위험. 사용자는 *valid한 응답을 받았다*고 믿지만 사실은 *다른 종목 분석*일 수 있다.

## 통합 — 5 Layer를 한 함수로

```typescript
async function callAIWithValidation<T>(
  opts: CallOpts,
  schema: z.ZodSchema<T>,
  input: unknown,
  options: { maxRetries?: number; safeDefault?: T } = {}
): Promise<T> {
  const maxRetries = options.maxRetries ?? 2;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      const raw = await callAI(opts);

      // Layer 1: JSON 파싱
      const parsed = safeJsonParse(raw, schema);
      if (!parsed) {
        throw new Error('JSON parse failed at layer 1');
      }

      // Layer 2-4: schema.parse가 자동 처리 (z.refine 포함)
      const validated = schema.parse(parsed);

      // Layer 5: cross-input
      const finalOutput = await validateAgainstInput(validated, input);

      return finalOutput;
    } catch (err) {
      console.warn(`Validation attempt ${attempt + 1} failed: ${err.message}`);
      // 재시도: 더 강한 instruction 추가
      opts.systemPrompt = (opts.systemPrompt || '') +
        `\n\nIMPORTANT: Previous attempt failed validation: ${err.message}. Be more careful.`;
    }
  }

  // 모든 재시도 실패 → 안전 기본값 또는 throw
  if (options.safeDefault !== undefined) {
    console.error('All validation attempts failed. Returning safe default.');
    return options.safeDefault;
  }
  throw new Error('AI output validation failed after retries');
}
```

핵심:
- **재시도 시 instruction 강화** — 같은 프롬프트 재시도가 아니라 *실패 이유를 알려주고* 재시도
- **안전 기본값** — 모두 실패 시 *환각 응답*보다 *명확하게 답을 못 함*이 더 안전
- **로깅** — 어떤 layer에서 실패하는지 측정 (어느 layer가 가장 자주 실패하는지가 시스템 개선의 입력)

## AI Agent Directive

**Trigger**: LLM 응답을 사용자 또는 다운스트림 시스템에 전달하는 코드 / *입력 검증은 있지만 출력 검증은 없는* 상태

**Prerequisites**:
- [AI 호출 패턴](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback)
- Zod 기본 사용법

### Actionable Steps (5 Layer 적용, 점진적)

*Layer 1+2만 먼저 (30분)*:
1. 출력 schema 정의 — `z.object({ ... })` with `min/max/enum` 경계 명시
2. `safeJsonParse` 구현 — 3단 fallback: 직접 파싱 → 첫 `{...}` 추출 → ` ```json``` ` 코드 블록
3. 호출자에서 `schema.parse(parsed)` 적용
4. 실패 시 **안전 기본값** 반환 (콘텐츠) 또는 **throw** (거래/결제)

*Layer 3-5는 다음 사이클*:
5. `.refine()`로 범위 제약 (target_price 0~100k 등)
6. `.refine()`로 cross-field consistency (STRONG_BUY + confidence 50+ 등)
7. 호출 후 `validateAgainstInput()` 별도 함수 — 입력 심볼을 reasoning에 인용했는지 등

### Anti-patterns

- ❌ `JSON.parse` 직접 사용 (문법 오류에 throw, Claude prompt-based에 흔함)
- ❌ schema에 제약 없는 `z.string()`, `z.number()` (환각 범위 무한)
- ❌ 재시도 시 *같은 프롬프트*로 재호출 (실패 이유를 프롬프트에 추가해야 함)
- ❌ 실패 시 *환각 응답을 그대로 반환* (안전 기본값 또는 throw 필수)
- ❌ layer별 실패율 측정 안 함 — 어느 layer가 병목인지 모름

## 자기 점검

1. 내가 호출하는 LLM의 출력 중 *형식 검증을 통과한 환각*을 본 적 있는가? 어떤 모양이었는가?
2. 5 layer 중 *내 시스템에 가장 부족한* layer는?
3. 안전 기본값과 throw 중 *내 도메인*에 더 적합한 것은? (콘텐츠는 안전 기본값, 거래/결제는 throw)
4. 검증 실패율을 *측정*하고 있는가? layer별로 어느 layer가 가장 자주 실패하는지 답할 수 있는가?
5. (열린 질문) 5 layer를 모두 통과한 응답이 *여전히 환각*인 경우는? (통계적 보정이 6번째 layer가 될 수 있는가?)

### 실습 과제

자신의 LLM 호출 한 곳에 *layer 1+2*만 30분 안에 추가:

```typescript
const Schema = z.object({ /* 출력 필드 */ });

async function callWithValidation(opts) {
  const raw = await callAI(opts);
  try {
    const parsed = JSON.parse(raw);
    return Schema.parse(parsed);
  } catch (err) {
    console.error('Validation failed:', err.message);
    return SAFE_DEFAULT;
  }
}
```

이게 layer 1+2. 적용 후 *얼마나 자주 fail하는지* 며칠 측정하면 layer 3-5의 필요성이 명확해짐.

## 출처

이 엔트리는 *제안 + 패턴 정의*. 양쪽 프로젝트 모두 미적용 상태.

- 갭 식별: [Harness Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)의 공통 갭 #2
- 보완 패턴: [AI 호출 패턴 엔트리](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback)의 *6번 패턴 (구조화된 출력 3단 fallback)*은 layer 1만 다룸. 이 엔트리는 layer 2-5까지 확장.
- Zod 공식 문서 (refine, schema chaining)

### 검증 메모

- 코드 스니펫은 *직접 작성한 패턴*. 양쪽 프로젝트의 실제 구현은 아직 없음.
- 5 layer 분류는 *내 분석*이며 공식적인 분류가 아님. 다른 분류도 가능.
- "STRONG_BUY + confidence 30" 같은 사례는 *Journal 000 audit 보고서*에서 직접 인용된 *실제 환각 사례*
- 안전 기본값 vs throw 결정은 *도메인 의존*. 거래/결제 시스템은 throw, 콘텐츠 시스템은 안전 기본값이 일반적.
