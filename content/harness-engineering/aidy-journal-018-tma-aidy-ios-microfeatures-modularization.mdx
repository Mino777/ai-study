## ios-study에서 aidy-ios로 — 같은 패턴, 다른 스택

[iOS Journal 011](/wiki/ios-ai/ios-ai-journal-011-toss-slash23-tuist-microfeatures)에서 ios-study(RIBs, 58모듈)에 토스 Microfeatures를 이식하며 **7개 함정을 박제**했다. 그 경험을 aidy-ios(TCA, SwiftUI)에 적용한 기록이다.

| 축 | ios-study (Journal 011) | aidy-ios (이 글) |
|---|---|---|
| **아키텍처** | RIBs + ReactorKit + UIKit | TCA + SwiftUI |
| **기존 상태** | 58개 SPM 패키지 (이미 모듈화) | 모놀리스 단일 타겟 (~200 파일) |
| **작업 성격** | SPM→Tuist 전환 + Interface 분리 | 모놀리스→26모듈 **처음부터** 구조 설계 |
| **DI 패턴** | RIBs Builder (Buildable 프로토콜) | TCA @DependencyClient |
| **Interface 내용물** | Buildable/Listener + 순수 enum | DependencyClient 계약 + 공유 UI 컴포넌트 |
| **고유 함정** | productTypes, transitive extension, Package.swift 충돌 | 모듈 이름↔타입 이름 충돌, @Reducer 매크로 모호성 |
| **규모** | 44커밋 198파일 3세션 | 4커밋 268파일 4시간 연속 |

## 스프린트 타임라인 — 3 Phase, 4시간

### Phase 1: Module.swift 헬퍼 + Core 12모듈 (709845d → 3b0d015)

Tuist `ProjectDescriptionHelpers/Module.swift`로 5-타겟 생성 자동화:

```swift
let chatModule = Module(
    name: "Chat",
    layer: .feature,
    dependencies: featureCoreDeps + [
        .module("Haptics"),        // Core 구현체 접근 OK
        .module("Draft"),
        .interface("Settings"),    // Feature는 Interface만! ←── TMA 핵심 규칙
        .interface("Memory"),
    ]
)
```

**의존성 규칙 (Module.swift가 구조적으로 강제)**:
- Feature → Feature: `.interface()` only — 구현체 직접 참조 차단
- Feature → Core: `.module()` 허용 (구현체 접근)
- Core → Core: `.module()` or `.interface()`

Core 12모듈 분리. `Network→Networking`, `Notification→AppNotification` rename (Foundation 충돌 회피).

### Phase 2: Feature 14모듈 분리 (c8f7f93)

14개 Feature 각각에 Interface/Sources/Testing/Tests 생성. Cross-Feature Interface:

| Interface | 노출 항목 | 사용처 |
|---|---|---|
| `MemoryInterface` | FlowLayout, ShimmerModifier, ColorHex | Chat, People, Settings |
| `PeopleInterface` | AvatarView | Memory, Chat, Search |
| `SettingsInterface` | ShareSheet, DashboardCard | Chat, HomeDashboard |

RIBs의 Interface가 Buildable/Listener 프로토콜만 담았다면, TCA의 Interface는 **공유 UI 컴포넌트도 포함**. SwiftUI의 합성 가능한 View 특성상 자연스러운 확장.

### Phase 3: 모듈 이름 충돌 해결 (28ac2cf)

**문제**: Swift에서 모듈 이름과 내부 타입 이름이 같으면 `@Reducer` 매크로가 생성하는 코드에서 스코프 충돌.

```swift
// 모듈 "Anniversary"에서 타입 "Anniversary"를 참조하면
// Anniversary.Anniversary — 모듈인지 타입인지 모호
```

**해법 2가지**:

| 전략 | 적용 모듈 | 예시 |
|------|----------|------|
| **모듈 이름 복수형** | SmartNotification→SmartNotifications, Anniversary→Anniversaries, Nudge→Nudges, RelationshipMap→Relations | 타입과 모듈 이름이 완전히 달라짐 |
| **Qualified typealias** | Digest, InteractionLog, Onboarding | `typealias DigestReminder = NetworkingInterface.DigestReminder` |

빌드 상태: `xcodebuild build-for-testing` ✅

## 고유 패턴: ClassifiableError 순환 의존 해체

ErrorLog와 Networking 사이 순환 의존을 **프로토콜 추상화**로 해체:

```swift
// ErrorLog (Core) — Networking을 모른다
public protocol ClassifiableError: Error {
    var errorCode: String? { get }
    var errorMessage: String { get }
}

@DependencyClient
public struct ErrorLogClient {
    public var log: @Sendable (_ error: Error, _ context: [String: String]) -> Void
}
```

```swift
// Networking (Core) — ErrorLog를 import
extension APIError: ClassifiableError {
    public var errorCode: String? { ... }
    public var errorMessage: String { ... }
}
```

의존 방향: `Networking → ErrorLog` (단방향). ErrorLog는 `any Error`를 받고 `ClassifiableError` 다운캐스트로 분류. RIBs의 Interface 패턴과 **같은 원칙**(프로토콜로 의존 역전)이 TCA에서 DependencyClient + 프로토콜로 구현된 형태.

## 교훈 — 하네스 엔지니어링 관점

### 1. 선행 프로젝트의 함정 박제가 후행 프로젝트의 속도를 만든다

ios-study에서 3세션 44커밋이 걸린 작업을, aidy-ios에서는 4시간 4커밋으로 완주. 핵심 차이: ios-study의 7개 함정 중 6개가 "SPM→Tuist 전환"에서만 발생하는 것임을 **이미 알고 있었기** 때문에, aidy-ios는 모놀리스에서 바로 Tuist로 설계하는 전략을 선택.

### 2. 같은 패턴이 다른 스택에서 다른 형태로 구현된다

| TMA 원칙 | RIBs (ios-study) | TCA (aidy-ios) |
|---|---|---|
| Interface 계약 | Buildable/Listener 프로토콜 | @DependencyClient 구조체 |
| DI 방식 | Component/Builder 트리 주입 | TCA @Dependency 매크로 |
| 순환 해체 | ViewableRouting 프로토콜 추상화 | ClassifiableError 프로토콜 추상화 |
| 공유 UI | Interface에 넣지 않음 (UIKit 의존 차단) | Interface에 SwiftUI View 포함 (합성 가능) |

원칙(Feature 간 직접 참조 금지)은 스택에 무관하게 동일. **구현체만 스택에 맞게 변환**.

### 3. 모놀리스→모듈화가 SPM→Tuist 전환보다 오히려 빠르다

ios-study는 "이미 모듈화된 58개 SPM 패키지"를 Tuist로 전환하면서 transitive dependency, Package.swift 충돌 등 **레거시 유산과 싸웠다**. aidy-ios는 "모놀리스에서 처음부터 올바른 구조로" 설계했기 때문에 레거시 함정이 없었다. **백지 설계가 마이그레이션보다 빠른 역설**.

### 4. @Reducer 매크로의 모듈/타입 이름 충돌은 TCA 고유 함정

이건 ios-study(RIBs)에서는 발생하지 않는다. TCA의 `@Reducer` 매크로가 타입 이름을 기반으로 보일러플레이트를 생성하는데, 모듈 이름과 같으면 Swift 컴파일러가 스코프를 혼동. **TCA + Tuist 조합의 고유 함정으로 박제**.
