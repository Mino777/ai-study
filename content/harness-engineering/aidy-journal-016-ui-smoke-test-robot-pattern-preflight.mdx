## Before — 수동 QA + 크래시 사후 대응

Journal 015(s30~s31)까지:

- **UI 테스트 부재**: 서버 API 테스트 + 유닛 테스트는 Gate로 강제하지만, 실제 UI 플로우(로그인→채팅→메모리→인물)는 수동 검증 의존
- **크래시 사후 대응**: s29에서 iOS 4회 크래시 발생. crash-log에 기록은 했지만 **사전 방지 메커니즘 없음**
- **워커 재시작 시 메모리 누수**: tmux pane 닫아도 Claude 프로세스가 고아로 남아 메모리 계속 점유

## After — UI Smoke Test + Preflight 사전 방지

### 1. Robot Pattern + Smoke Test (3플랫폼 동시)

s32에서 WO-154~156으로 3플랫폼에 동시 도입:

**서버 (WO-154)**: UI 테스트 전용 계정 시딩
- `TestAccountSeeder` (ApplicationRunner, @ConditionalOnProperty)
- 계정: `uitest@aidy.com` / `AidyTest2026!`
- 멱등: 이미 존재하면 스킵, 프로덕션에서는 미실행 (`aidy.test-account.enabled: false`)

**iOS (WO-155)**: Robot Pattern + Smoke Test
- Robot 클래스 5개: `BaseRobot`, `LoginRobot`, `ChatRobot`, `PeopleRobot`, `MemoryRobot`
- `SmokeTest_CoreFlow`: 로그인 → 채팅 전송 → AI 응답 → 기억 확인 → 인물 탭 → 프로필
- 기존 UITest `loginIfNeeded()` → `LoginRobot`으로 리팩터링

**Android (WO-156)**: Robot Pattern + Smoke Test
- Robot 클래스 5개 (ComposeTestRule 기반)
- `SmokeTest_CoreFlow`: iOS와 동일 플로우
- `createAndroidComposeRule<MainActivity>()` 사용

**3-way 병렬 dispatch**: 서버(계정 시딩) + iOS(Robot) + Android(Robot)를 동시에 보내고, 서버 완료 후 클라이언트가 테스트 계정으로 Smoke Test 실행.

### 2. Preflight 메모리 체크 강화

`architect-cli.sh preflight`에 4단계 시스템 점검 추가:

| 체크 | 임계값 | 경고 |
|------|--------|------|
| **Swap 사용률** | 80% 이상 | 워커 크래시 위험 |
| **Claude 인스턴스 수** | 4개 이상 | 16GB 권장 최대 3개 |
| **memory_pressure** | warning/critical | sudo purge 권장 |
| **Compressor** | 4GB 이상 | swap 임박, sudo purge 권장 |

s29 크래시 로그 분석에서 도출한 임계값:
- 크래시 시점: Swap 331MB 사용 + Claude 6~7 인스턴스
- **핵심 발견**: Swap 자체보다 **Claude 인스턴스 수 + compressor 크기**가 더 정확한 크래시 예측 지표

### 3. restart_workers 프로세스 그룹 kill

워커 재시작 시 tmux pane의 프로세스 그룹(PGID)을 kill한 뒤 pane을 닫는 순서로 변경:

```bash
# 기존: tmux kill-pane (foreground만 종료, 고아 발생)
# 변경: PGID kill → pane 정리 → 재시작
```

이전에는 스프린트 사이에 재시작할 때마다 고아 프로세스가 누적되어 다음 스프린트에서 swap 압박이 가중되었다.

### 4. watch-workers bash 3 호환

macOS 기본 bash(3.x)에서 `local -A`(associative array)가 지원되지 않아 watch-workers가 실패. 변수명 방식으로 우회:

```bash
# 기존: local -A worker_status  (bash 4+ only)
# 변경: worker_status_server=... worker_status_ios=... (bash 3 호환)
```

## 배운 것 / 다음 후보

1. **Robot Pattern은 AI 생성 UI 테스트의 핵심 전제**: AI가 UI 테스트를 작성할 때 Robot 없이 직접 UI 요소를 접근하면, UI 변경 시 모든 테스트가 깨진다. Robot이 변경 전파를 1곳으로 집중
2. **Preflight는 예방, crash-log는 사후 분석**: 두 도구는 상보적. preflight가 경고해도 무시하면 crash-log에 기록됨
3. **고아 프로세스가 swap의 숨은 원인**: tmux kill-pane만으로는 Claude 서브프로세스가 안 죽는다. PGID kill 필수
4. **bash 3 호환은 macOS에서 영구 과제**: Homebrew bash 4를 설치해도 시스템 스크립트는 `/bin/bash`(3.x)를 사용
5. **다음 후보**: Smoke Test 실 디바이스 실행 자동화, preflight → autoceo 시작 시 자동 실행 (현재는 수동 호출)

## AI Agent Directive

**Trigger**: 멀티 에이전트 환경에서 워커가 원인 불명으로 종료되거나, 스프린트 시작 전 시스템 상태를 점검할 때.

**Prerequisites**:
- [Aidy Journal 015 — Agent Teams + 파이프라이닝](/wiki/harness-engineering/aidy-journal-015-agent-teams-pipelining-8r-to-5r)
- [Test Strategy 3-Client](/wiki/evaluation/test-strategy-3-client-via-aidy)

**Actionable Steps**:
1. `preflight` 실행 — Swap/Claude 인스턴스/memory_pressure/compressor 4단계 점검
2. 경고 발생 시 `sudo purge` + 불필요 앱 종료 후 재점검
3. 워커 재시작 시 반드시 PGID kill → pane 정리 → 재시작 순서
4. UI Smoke Test 도입 시 Robot Pattern 먼저 — Robot 없는 UI 테스트는 유지보수 불가
5. 테스트 계정은 서버 시딩으로 — 클라이언트에 하드코딩하되, 서버가 멱등 생성

**Anti-patterns**:
- preflight 경고 무시하고 스프린트 시작 — s29 크래시 4회의 원인
- tmux kill-pane만으로 워커 재시작 — 고아 프로세스 누적
- Robot 없이 UI 테스트 직접 작성 — UI 변경 시 N개 테스트 전부 수정 필요
- macOS bash 4 기능(`local -A`, `${!array[@]}`) 사용 — 시스템 bash는 3.x

## 자기 점검

1. 스프린트 시작 전 preflight를 실행하고 있는가?
2. 워커 재시작 시 PGID kill을 하고 있는가, 아니면 tmux kill-pane만 하고 있는가?
3. UI 테스트에 Robot Pattern이 적용되어 있는가?
4. 테스트 계정이 서버 시딩으로 관리되는가, 아니면 각 플랫폼에 흩어져 있는가?
