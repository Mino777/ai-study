## 왜 이 사이클은 *메타*인가

Journal 005~009 다섯 사이클이 *실코드 작업*이었다면, 010은 *잠깐 멈춰서 상태를 박제*하는 사이클이다.

사용자의 원 요청: *"journal 010 까지 자율 진행해줘"*. 10 사이클이 끝난 시점에 *다음 행동 전에* 현재 상태를 *살아있는 베이스라인*에 append하는 것이 자연스러운 마무리.

이 엔트리는 *Journal 000의 세 번째 갱신 로그*를 이 파일에 박은 다음, 원본 Journal 000의 해당 섹션에도 같은 내용을 append한다. 중복처럼 보이지만 다름:
- 이 엔트리(010): *6 사이클 회고*가 주 메시지
- Journal 000의 갱신 로그: *베이스라인의 역사 추적*이 주 메시지

## 시리즈 자체 동력의 3가지 결정적 증거

005~009 사이클 동안 *시리즈가 단순 기록물이 아니라 운영 도구*임을 입증한 3가지:

### 1. 박제의 전이성 (Journal 005)

`/wt-branch` 슬래시 커맨드가 ai-study에 박제된 직후, *다른 세션*이 *별도 안내 없이* 발견해서 tarosaju에 이식함:

```
tarosaju main commit:
52e5cdc chore: /wt-branch slash command 추가 (ai-study에서 복사)
```

커밋 메시지의 *"ai-study에서 복사"*가 결정적 증거. 박제가 *문서 기반 소통 채널*로 작동했다는 첫 라이브 사례.

### 2. 자기 검증 구조 (Journal 002, 005)

도구가 *자기 자신에 적용*됨:
- Journal 002: ai-review.yml inline test gate가 *자기 자신의 변경*을 검증
- Journal 005: wt-branch가 *wt-branch 자체의 이식*에 사용됨

시리즈가 5 에피소드에 도달한 시점에 이 패턴이 *2회* 등장. 우연이 아닌 *시리즈의 본질적 속성*.

### 3. 계획 vs 현실의 라이브 교정 (Journal 007, 009)

계획 큐가 라이브로 *4번* 재정렬됨. 대표 사례:

- **Journal 007**: *"tarosaju prompt caching"* → Haiku cache minimum 고려해서 *비용 추적 인프라*로 전환
- **Journal 009**: *"tarosaju Zod 출력 검증"* → 응답이 자연어라서 *text guards*로 전환

계획을 *억지로 이행*하지 않고 *라이브 현실*에 맞춰 *도구를 바꾸는* 것이 올바른 판단. Compound Engineering의 *"사고(또는 발견)가 다음 사이클의 정확한 입력"* 원칙의 구체적 사례.

Journal 005~009 실코드 사이클과 *동시에* LLM-First Wiki 전환도 진행됨:

| Phase | 산출물 |
|---|---|
| Phase 1 | SPEC.md + CLAUDE.md 갱신 + llm-first-wiki-principles + ai-agent-start-here |
| Phase 2A | 3 핵심 엔트리 (compound, overview, diagnosis-checklist)에 For AI Agents 섹션 |

이게 *메타 인프라*. 실코드 사이클이 쌓일수록 *에이전트가 읽기 좋은 형태*로 누적되는 이중 레이어.

## 자기 점검

1. *시리즈가 시작된 이후* 사고 재발률을 측정해본 적 있는가? 내 시리즈는 라이브 사고 *0건*이 7 사이클 이어지고 있음.
2. 원본 큐 정의와 *실제 진행*의 괴리는 얼마인가? 내 경우 30% 재정렬.
3. *같은 메시지*를 여러 사이클에 걸쳐 반복하고 있지 않은가? (중복은 atomicity 위반)
4. 시리즈가 *멈출 때*와 *계속 돌 때*의 차이를 인식하는가? 정지 지점은 끝이 아님.
5. (열린 질문) Journal 011의 *첫 트리거*는 무엇에서 올까? 계획? 라이브 사고? 사용자 요청?

## For AI Agents

**Trigger**: 시리즈가 N 사이클 누적된 시점(예: 5, 10, 20) / 베이스라인을 갱신해야 할 때 / 시리즈 전체 회고가 필요할 때

**Prerequisites**:
- [Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline) — 시리즈의 출발점
- [LLM-First Wiki 설계 원칙](/wiki/harness-engineering/llm-first-wiki-principles)

**Actionable Steps (메타 회고 사이클)**:
1. 최근 N 사이클 작업 요약 표 작성 (Journal 번호 / 메시지 / 산출물)
2. Journal 000 원본 큐 대비 진행 상황 점검
3. *원본 큐 vs 실제 진행*의 괴리 사례 수집 (추측 큐 정확도 측정)
4. 회피 메커니즘 효과 측정 — 사고 재발률, CI 실패 횟수, dogfooding 성공
5. 시리즈 자체 동력 증거 확인 — 박제 전이 / 자기 검증 / 라이브 교정 사례
6. 다음 큐 재정의 — 완료된 것 제거, 새 발견 추가, 우선순위 재조정
7. Journal 000 베이스라인에 *append-only 갱신 로그* 섹션 추가 (덮어쓰기 X)

**Anti-patterns**:
- ❌ 메타 사이클을 *실작업 사이클과 같은 빈도*로 돌기 (매 사이클마다 회고는 과도)
- ❌ 원본 큐 정의를 *수정*(덮어쓰기) — 갱신 로그에 append
- ❌ 진행 상황을 *진짜보다 부풀리기* (정직 원칙)
- ❌ 시리즈 *정지 지점*을 *종료*로 착각
- ❌ 다음 큐 *추측만*으로 채우기 — 라이브 사고가 재정렬할 것
