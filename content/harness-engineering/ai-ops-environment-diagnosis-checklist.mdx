## 왜 진단인가

[Harness Journal 시작 가이드](/wiki/harness-engineering/harness-journal-bootstrap-guide)는 *시리즈 시작*을 다룬다. 그러나 시리즈를 시작하기 *전에* 자문해야 할 것: **내 환경이 시리즈를 시작할 만한가? 어떤 영역에 시리즈가 가장 가치 있을까?**

이 글은 그 질문에 30분 안에 답하는 *진단 체크리스트*다. [Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)의 10 영역을 *Yes/No 50문항*으로 응축. 각 영역 5분, 총 30분.

체크리스트의 결과:
- **0~20점**: 시리즈가 *반드시* 필요. 첫 사이클부터 임팩트 큰 영역이 명확.
- **21~35점**: 시리즈 가치 있음. 기존 강점 위에 *부족한 영역*을 보강.
- **36~50점**: 환경이 잘 갖춰짐. 시리즈는 *이미 잘하는 것을 더 잘하는* 데 쓰임.

## 진단 체크리스트 — 10 영역 × 5문항

각 문항에 *Yes(1점) / No(0점)* 로 답한다. 영역별 5점 만점, 총 50점.

### 영역 1 — Hook 시스템 (Pre/Post commit, push, edit)

- [ ] `.claude/settings.json`에 PreToolUse 또는 PostToolUse 훅이 *최소 1개* 정의되어 있다
- [ ] commit 전에 *자동으로* test가 실행된다 (사람이 매번 호출 X)
- [ ] commit 전에 *자동으로* build/lint가 실행된다
- [ ] push 후에 *자동 후속 작업*(알림, 회고 리마인더, 등)이 실행된다
- [ ] 훅 실패가 *작업을 차단*한다 (warning만이 아니라 hard block)

**5점 만점 / 영역 점수: ___ / 5**

### 영역 2 — 슬래시 커맨드 (`/.claude/commands/*.md`)

- [ ] 프로젝트 특화 슬래시 커맨드가 *최소 1개* 정의되어 있다
- [ ] *매 사이클 끝* 자동화 슬래시 커맨드가 있다 (예: `/compound`)
- [ ] *외부 입력 가공* 슬래시 커맨드가 있다 (예: `/ingest`)
- [ ] *전체 스프린트 자동화* 슬래시 커맨드가 있다 (예: `/autoceo`)
- [ ] 슬래시 커맨드들이 [CLAUDE.md skill routing](/wiki/harness-engineering/compound-automation-slash-commands)에 등록되어 있다

**5점 만점 / 영역 점수: ___ / 5**

### 영역 3 — AI 협업 헌장 (CLAUDE.md / AI-AGENT-GUIDE.md / ENGINEERING.md)

- [ ] CLAUDE.md(또는 동등한 파일)가 존재하고 *프로젝트 구조 + 규약*을 명시한다
- [ ] *Context Engineering 원칙* (어떤 컨텍스트를 어떻게 로드할지)이 박제되어 있다
- [ ] *Harness Engineering 원칙* (입력 검증, 출력 검증, 폴백 등)이 박제되어 있다
- [ ] *금지 사항*이 명시되어 있다 (보호 파일, 위험 명령 등)
- [ ] 이 문서들이 *최근 1개월 내* 갱신되었다 (정체된 문서는 0점)

**5점 만점 / 영역 점수: ___ / 5**

### 영역 4 — AI 호출 코드 패턴

- [ ] AI 호출 코드가 *한 곳에 집중*되어 있다 (`lib/ai/`, `lib/llm/` 등)
- [ ] *Circuit Breaker* 또는 *상태 기반 폴백*이 구현되어 있다
- [ ] *Timeout*이 모든 AI 호출에 명시되어 있다
- [ ] *멀티 프로바이더 폴백*이 있다 (또는 단일 모델만 쓰는 명확한 이유가 있다)
- [ ] *프롬프트 캐싱*이 적용되어 있다 (Claude `cache_control` 등)

**5점 만점 / 영역 점수: ___ / 5**

### 영역 5 — 입력/출력 검증

- [ ] 모든 API 입력에 *Zod*(또는 동등한) 스키마 검증이 있다
- [ ] *AI 응답*에 *schema.parse* 검증이 있다 (입력만이 아니라 출력도)
- [ ] 검증 실패 시 *재시도 또는 안전 기본값*이 있다
- [ ] *Cross-field consistency* 검증이 있다 (refine, 내부 모순 차단)
- [ ] *입력 데이터와 출력의 정합성* 검증이 있다 (cross-input)

**5점 만점 / 영역 점수: ___ / 5**

(자세한 패턴: [AI 출력 Zod 검증 패턴 5 layer](/wiki/harness-engineering/ai-output-zod-validation-pattern))

### 영역 6 — 테스트 / CI 게이트

- [ ] vitest/jest/pytest 등 *unit 테스트*가 존재한다
- [ ] PR 머지 *전*에 자동 검증 파이프라인이 있다 (GitHub Actions 등)
- [ ] *E2E 테스트*(Playwright, Cypress)가 1개 이상 있다
- [ ] *type check*(`tsc --noEmit`)가 별도 게이트로 있다
- [ ] 머지 게이트 실패 시 *PR이 차단*된다 (warning만이 아니라)
- [ ] *CI 장애 대응*이 있다 — GitHub-hosted 빌링 한도/장애 시 self-hosted runner fallback 또는 수동 검증 절차 (ADR-010 hybrid fallback 참조)
- [ ] `npm audit` / 보안 스캔이 CI에 포함되어 있다 (`--omit=dev`로 프로덕션 의존성만 검사)

**7점 만점 / 영역 점수: ___ / 7**

### 영역 7 — 회고 + 솔루션 박제 트랙

- [ ] `docs/retros/` 또는 동등한 디렉터리가 있다
- [ ] *최근 1주일 내* 회고가 1건 이상 작성되었다
- [ ] `docs/solutions/`에 *카테고리별 디렉터리*가 있다
- [ ] 매 사이클 회고에 *"다음에 적용할 것"* 섹션이 있다
- [ ] 회고에서 *"다음 후보"가 다음 사이클의 입력*이 된 사례가 있다

**5점 만점 / 영역 점수: ___ / 5**

### 영역 8 — 메모리 / 세션 연속성

- [ ] `MEMORY.md` 또는 Claude Code 메모리가 *프로젝트별*로 운영된다
- [ ] 매 세션 시작 시 *컨텍스트 자동 로드*가 정의되어 있다
- [ ] 핵심 결정/사고가 *메모리에 박제*되어 다음 세션이 *처음부터 시작하지 않는다*
- [ ] *어제 한 일*과 *오늘 할 일*이 git/문서/메모리 어딘가에 명시되어 있다
- [ ] *동일한 사고가 N번 반복*되지 않는다 (반복되면 0점)

**5점 만점 / 영역 점수: ___ / 5**

### 영역 9 — 비용 / 관측 / Observability

- [ ] *AI API 비용*을 일/월 단위로 *추적*한다
- [ ] *모델별 / 에이전트별 비용*을 분리해서 본다
- [ ] *에러*가 어딘가에 자동 수집된다 (Sentry, logger 등)
- [ ] *성능 메트릭*(LCP, latency 등)이 추적된다
- [ ] *임계치 초과*(비용 한도, 에러율 등) 시 알림이 작동한다

**5점 만점 / 영역 점수: ___ / 5**

### 영역 10 — 자율 진행 / Compound Loop

- [ ] [Compound Engineering 4단계 루프](/wiki/harness-engineering/compound-engineering-philosophy)(Plan → Work → Review → Compound)를 *명시적으로* 따른다
- [ ] [5단계 ladder](/wiki/harness-engineering/compound-engineering-philosophy) 중 *Stage 3 이상*(plan-first, PR-only review)에 도달했다
- [ ] *사람이 매번 의도해서 하는 작업*과 *자동화된 작업*의 경계가 명확하다
- [ ] *AI 사고*가 발생했을 때 *다음 사이클의 입력*으로 즉시 활용된다
- [ ] *전체 사이클을 사람 개입 없이 N라운드 자동 반복*하는 메커니즘이 있다 (`/autoceo` 등)

## 결과 해석

### 0~20점 — 시리즈가 *반드시* 필요

대부분의 영역에 자동화/박제가 없는 상태. AI는 *매 세션 처음부터* 시작하고, 같은 사고가 반복되며, 회고가 사람의 머릿속에만 남는다.

**다음 단계**: 가장 낮은 영역부터 시작. 보통 *영역 1(훅) → 영역 7(회고) → 영역 3(헌장)* 순서가 효과적. [Harness Journal Bootstrap Guide](/wiki/harness-engineering/harness-journal-bootstrap-guide)의 1단계부터.

### 21~35점 — 시리즈가 *임팩트 큼*

기본 인프라는 있지만 *연결*이 부족한 상태. 슬래시 커맨드는 있는데 회고와 연결 안 됨, 테스트는 있는데 머지 전 게이트 X, AI 호출은 있는데 검증 0 같은 *분리된 자산*들이 있다.

**다음 단계**: *연결*에 집중. 베이스라인을 박은 후 *큐가 자연스럽게* 만들어지는지 점검. 이 점수대가 시리즈 도입 시 *가장 빠른 ROI*를 보인다.

### 36~50점 — 환경이 잘 갖춰짐

대부분의 영역이 *작동*하는 상태. 시리즈는 *이미 잘하는 것을 더 잘하는* 데 쓰인다. 새 갭은 *이 체크리스트가 잡지 못하는 차원*에서 발견될 가능성 높음.

**다음 단계**: 체크리스트를 *자기 환경에 맞게 갱신*. 새 영역(예: *모델 selection 자동 라우팅*, *프롬프트 버전 관리*, *RAG 파이프라인 검증*)을 추가. 50점이 *상한*이 아니라 *지금 측정 중인 차원의 상한*.

## 영역별 점수가 *낮을 때*의 가장 가벼운 첫 단계

| 영역 | 점수 < 3 일 때 가장 가벼운 첫 단계 |
|---|---|
| 1. Hook 시스템 | `.claude/settings.json` 신설 + commit 전 test 자동 실행 1 step |
| 2. 슬래시 커맨드 | `/compound`만 도입 ([compound-automation-slash-commands](/wiki/harness-engineering/compound-automation-slash-commands)) |
| 3. AI 협업 헌장 | CLAUDE.md 신설 + 30줄 프로젝트 구조만 |
| 4. AI 호출 코드 | 한 곳에 ai-client.ts 모으기 + timeout 60s 추가 |
| 5. 입력/출력 검증 | 가장 자주 호출되는 API 1개에 Zod 입력 + safeJsonParse 출력 |
| 6. 테스트 / CI | GitHub Actions에 vitest 1 job 추가 |
| 7. 회고 + 솔루션 | `docs/retros/YYYY-MM-DD.md` 1개 작성 |
| 8. 메모리 | MEMORY.md 신설 + 핵심 결정 5줄만 |
| 9. 비용 / 관측 | console.log로 토큰 수만 출력 (DB 없이) |
| 10. Compound Loop | [Compound Engineering 엔트리](/wiki/harness-engineering/compound-engineering-philosophy) 읽고 4단계 루프를 *오늘 1번* 따라 보기 |

각 단계는 30분 안에 가능. 점수가 *1점만 올라도* 다음 사이클이 더 쉬워짐.

## AI Agent Directive

**Trigger**: 새 프로젝트의 AI 운영 환경을 평가하거나, Harness Journal 시리즈 시작 여부를 결정해야 할 때. 현재 환경의 강점과 갭을 30분 안에 정량화해야 할 때.

**Prerequisites**:
- [Harness Journal Bootstrap Guide](/wiki/harness-engineering/harness-journal-bootstrap-guide)
- [Compound Engineering Philosophy](/wiki/harness-engineering/compound-engineering-philosophy)

### Actionable Steps
1. 10개 영역 × 5문항 = 50문항을 Yes(1점) / No(0점)으로 답변 (30분)
2. 각 영역 점수 합산 → 총점 (0~52)
3. 결과 해석:
   - 0~20점: 시리즈 *반드시 필요*. 가장 낮은 영역의 *가장 가벼운 첫 단계*부터 시작
   - 21~35점: 시리즈 *가장 큰 ROI*. 연결이 부족한 부분 보강
   - 36~50점: *새 차원 발견*에 집중. 체크리스트 자체 갱신
4. 가장 낮은 영역의 *가장 가벼운 첫 단계*를 오늘 30분 안에 실행
5. 진단 결과를 파일로 박제 (회고, README, MEMORY.md, git 어딘가)
6. 첫 Harness Journal 엔트리로 결과 기록

### Anti-patterns
- ❌ 진단 결과를 기록하지 않고 지나가기 (살아있는 문서 원칙 위반)
- ❌ 점수를 부풀리기 (자기기만 — 정직한 진단이 핵심)
- ❌ 모든 영역을 동시에 보강 (한 사이클 = 한 영역)
- ❌ 50점을 최종 목표로 취급 (실제로는 현재 측정 차원의 상한)

## For AI Agents

**Trigger**: 새 프로젝트에서 AI 환경 구축을 시작해야 하거나, 기존 환경의 *강점/갭*을 빠르게 파악해야 할 때. Harness Journal 시리즈 시작 *전 단계*.

**Prerequisites**:
- [AI Agent Start Here](/wiki/harness-engineering/ai-agent-start-here)

**Actionable Steps (30분 진단)**:
1. 위 10 영역 × 5문항 = 50문항을 *Yes(1점) / No(0점)*으로 답
2. 각 영역 점수 합산 → 총점 (0~52)
3. 결과 해석:
   - 0~20점: *시리즈 반드시 필요*. 가장 낮은 영역의 *가장 가벼운 첫 단계*부터
   - 21~35점: *시리즈 가장 빠른 ROI*. 연결이 부족한 영역 보강
   - 36~50점: *새 차원 발견*에 집중. 체크리스트 자체 갱신
4. 가장 낮은 영역의 *가장 가벼운 첫 단계*를 *오늘 30분 안에* 실행
5. 결과를 [Harness Journal](/harness-journal) 첫 엔트리로 박제

**Anti-patterns**:
- ❌ 진단 결과를 *어디에도 기록하지 않고* 그냥 지나가기 (살아있는 문서 원칙 위반)
- ❌ 점수를 *부풀리기* (자기기만 — 진단은 *정직한 박제*가 핵심)
- ❌ 모든 영역을 *동시에* 보강하려고 시도 (한 사이클 = 한 영역)
- ❌ 50점 도달을 *상한*으로 취급 (실제로는 *현재 측정 차원의 상한*일 뿐)

**관련 다음 단계**:
- [Harness Journal 시작 가이드](/wiki/harness-engineering/harness-journal-bootstrap-guide) — 진단 후 시리즈 시작 방법

## 자기 점검

1. 위 50문항을 *지금* 답할 수 있는가? 답할 수 없는 문항이 있다면 *왜* — 측정 안 됨인가, 정의 모름인가?
2. 가장 낮은 영역의 점수를 *3주 안에* 2점 올릴 수 있는가? 그게 *가장 큰 임팩트*인가?
3. 가장 높은 영역의 점수가 *4점 이상*인 이유는 — 운인가, 의식적 설계인가?
4. 총점이 *너무 낮아서 부끄럽다*고 느낀다면 — 그게 시리즈 시작의 가장 강한 동기가 될 수 있다.
5. (열린 질문) 이 체크리스트의 *11번째 영역*은 무엇인가? 내 환경에서 이 50문항이 잡지 못하는 차원은?

### 실습 과제

1. *지금 30분*을 내서 위 50문항에 답하기
2. 점수표를 *어딘가에 박기* (회고, README, MEMORY.md, 어디든)
3. 가장 낮은 영역의 *가장 가벼운 첫 단계*를 *오늘* 30분 안에 실행
4. 결과를 *첫 Harness Journal 엔트리*로 박제

이 4단계가 끝나면 [시리즈가 시작된다](/wiki/harness-engineering/harness-journal-bootstrap-guide).

## 출처

- 10 영역은 [Harness Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)의 audit 항목 10가지를 *50문항으로 응축*
- 점수 해석 기준은 *내 두 프로젝트 + ai-study 운영 경험*에 기반한 추정 (다른 환경에서는 다른 임계치가 적합할 수 있음)
- 가장 가벼운 첫 단계는 *내 시리즈에서 실제로 효과를 본 시작점*들

### 검증 메모

- 50문항은 *제안*이며, 다른 영역/문항이 더 적합한 환경이 있을 수 있음
- "0~20점/21~35점/36~50점" 임계치는 *내 경험 기반 추정*. 더 큰 표본이 모이면 갱신
- *영역 8(메모리/세션 연속성)이 가장 자주 무시*된다는 관찰은 *내 audit 경험*에 기반 — 다른 환경에서는 다른 영역이 가장 자주 무시될 수 있음
- 50점이 *상한이 아니라 현재 측정 차원의 상한*이라는 것을 quiz 3번에서 명시 — 체크리스트가 *완성된 도구*가 아니라 *살아있는 도구*임을 박음
