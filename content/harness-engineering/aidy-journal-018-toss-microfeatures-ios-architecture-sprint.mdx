## 배경 — 왜 토스 아키텍처인가

ios-study 프로젝트는 RIBs 기반 58개 SPM 패키지로 구성된 대규모 모듈러 앱이다. 문제:

| 증상 | 원인 |
|------|------|
| Feature 간 순환 참조 위험 | `OdaPresentation` → `OnewayPresentation` 직접 import |
| `.pbxproj` 머지 컨플릭트 빈발 | Xcode 프로젝트 파일 수동 관리 |
| 전체 빌드 필수 (UI 한 줄 수정에도) | Feature 독립 빌드 불가 |
| 새 모듈 생성에 10-15분 | Package.swift 복붙 + 수동 설정 |

토스 SLASH23에서 공개한 **Microfeatures 5계층 구조**가 이 문제들의 정확한 해답이었다.

---

## Microfeatures 5계층 구조

```
[Feature]Interface   ← 프로토콜 계약만 (Buildable/Listener/Entity)
    ↓
[Feature]            ← 실제 구현 (외부 비공개)
    ↓
[Feature]Testing     ← Mock 구현체 (테스트/Example용)
    ↓
[Feature]Tests       ← XCTest 코드
    ↓
[Feature]Example     ← 독립 실행 미니앱
```

**핵심 규칙**: 같은 계층의 Feature는 서로의 구현을 절대 import하지 않는다. **Interface만** 참조.

## 스프린트 타임라인 — 4 Work Orders, 44 커밋

### WO-005: Interface 모듈 분리 (M형)

**Before** — Presentation 직접 참조:
```swift
// OdaPresentation/OdaBuilder.swift (위반)
import OnewayPresentation  // ❌ 구현 전체를 끌어옴

class OdaComponent: Component<OdaDependency> {
  private let builder = OnewayReturnSearchBuilder()  // ❌ 구체 타입 직접 생성
}
```

**After** — Interface 프로토콜 주입:
```swift
// OdaPresentation/OdaBuilder.swift
import OnewayPresentationInterface  // ✅ 프로토콜만

public protocol OdaDependency: Dependency {
  var onewayReturnSearchBuildable: any OnewayReturnSearchBuildable { get }  // ✅ 프로토콜
}

final class OdaRouter: ViewableRouter<OdaInteractable, OdaViewControllable> {
  private let searchBuilder: any OnewayReturnSearchBuildable
  private var searchRouting: ViewableRouting?  // ✅ 구체 타입 아닌 프로토콜
}
```

`OnewayPresentationInterface` 모듈은 **3개 파일만** 담는다:
- `ReturnZoneType.swift` — 순수 enum (UIKit 없음)
- `OnewayReturnSearchInterface.swift` — `Buildable` + `Listener` 프로토콜
- `OnewayReturnLocationInterface.swift` — `Buildable` + `Listener` 프로토콜

부모 Component가 concrete builder를 주입:
```swift
// TransportComponent.swift (부모)
extension TransportComponent: OdaDependency {
  var onewayReturnSearchBuildable: any OnewayReturnSearchBuildable {
    OnewayReturnSearchBuilderForOda(dependency: self)  // ✅ 구체 타입은 부모만 안다
  }
}
```

### WO-006: Tuist 마이그레이션 (L형)

58개 SPM `Package.swift` → Tuist `Project.swift` 전환. 9개 Wave로 나눠 진행:

| Wave | 대상 | 모듈 수 |
|------|------|---------|
| 1 | Tuist 워크스페이스 기반 구축 | 인프라 |
| 2 | Scaffold 템플릿 + 파일럿 | 2 |
| 3 | Util / UserInterface / FeatureSupport | 8 |
| 4-5 | Core / Domain / Repository | 20 |
| 6-9 | Presentation + 메인 앱 | 26 |

**핵심 산출물**:

```
Project.swift              ← 메인 앱 타겟 (4 빌드 config)
Workspace.swift            ← 56개 모듈 등록
Tuist/
  ├── ProjectDescriptionHelpers/
  │   └── ModuleSettings.swift   ← 공통 빌드 설정 + 헬퍼
  └── Templates/
      ├── feature/               ← 새 Feature 모듈 scaffold
      └── example/               ← Example 앱 scaffold
```

`ModuleSettings.swift` 헬퍼로 모듈 선언 1줄:
```swift
ModuleTarget.framework(name: "OnewayPresentation", dependencies: [...])
ModuleTarget.exampleApp(exampleName: "Oneway", featureTarget: "OnewayPresentation")
```

### WO-007: Example 앱 파일럿 (M형)

|---------|---------|------|

### WO-008: Routing Interface 확장 + 빌드 체인 안정화

WO-005에서 `OnewayReturnSearch`만 분리했던 것을 `OnewayReturnLocation`까지 확장. transitive dependency 누락 28개 모듈 일괄 패치. Firebase 초기화 크래시 8커밋에 걸쳐 해소.

## 교훈 — 하네스 엔지니어링 관점

### 1. 인제스트 → WO → 실행의 3단 파이프라인

토스 SLASH23 기술 블로그를 `/ingest`로 분석 → 6개 패턴 추출 → 4개 WO 발행 → 하루 스프린트 완주. **외부 지식을 실행 가능한 Work Order로 변환**하는 파이프라인이 핵심이었다.

### 2. L형 WO는 Wave 분할 필수

WO-006(Tuist 마이그레이션)은 58개 모듈 대상의 L형. 9개 Wave로 나눠 진행하지 않았으면 중간 빌드 실패에서 복구 불가능했을 것. 각 Wave 끝에 `compound` 커밋으로 체크포인트.

### 3. Interface 먼저, Tuist는 그 위에

WO-005(Interface 분리)를 WO-006(Tuist) 이전에 배치한 것이 정확한 순서. Interface 패턴이 확립된 상태에서 Tuist 전환 시 `Project.swift`에 의존성을 올바르게 선언할 수 있었다.

### 4. Firebase 초기화는 8커밋의 삽질

WO-008 마지막 단계에서 Firebase 초기화 순서 문제로 크래시가 8커밋 연속 발생. `@main` ↔ `main.swift` 전환, `AppDelegate.init()` vs `didFinishLaunching` 위치, ObjC category 링킹(`-ObjC` 플래그), 시뮬레이터 분기까지 — Tuist 전환과 무관한 **런타임 문제가 마이그레이션 완료를 가로막는 전형적 패턴**.

## Aidy Journal 018 vs iOS Journal 011 — 같은 스프린트, 다른 렌즈

이 스프린트는 두 저널에 기록됐다. **iOS Journal 011**은 iOS 실행 현장에서, **Aidy Journal 018**은 하네스 오케스트레이션 관점에서.

### 관점 비교

| 축 | Aidy Journal 018 (이 글) | iOS Journal 011 |
|---|---|---|
| **주 독자** | 멀티 레포 오케스트레이터, 하네스 엔지니어 | iOS 개발자, 빌드 시스템 관리자 |
| **핵심 질문** | "외부 지식을 WO로 변환하는 파이프라인이 작동하는가?" | "46개 모듈의 빌드 체인을 실제로 어떻게 고쳤나?" |
| **WO 서술 깊이** | 전체 흐름 요약 (What + Why) | 각 WO별 코드 레벨 상세 (How + Traps) |
| **교훈 성격** | 인제스트→WO 파이프라인, L형 분할, 순서 전략 | 7개 함정 박제 (productTypes, transitive extension 등) |
| **자동화 다룸** | 언급 수준 (28모듈 패치) | scan-imports.py 228줄 + patch-deps.py 148줄 상세 |
| **복리 체크** | Journal 017 dispatch 통합과의 연결 | Journal 010 오케스트레이션 인프라 미사용 회고 |

### 서로가 채우는 빈 칸

**018이 011에 없는 것**:
- `/ingest` → WO 발행 → 실행의 **3단 파이프라인 패턴** 명시화. "토스 블로그를 읽었다"에서 "6개 패턴 추출 → 4개 WO 발행"까지의 변환 과정을 하네스 방법론으로 정립
- 인프라 마이그레이션 vs 피쳐 개발에서 **단일 세션 vs 멀티 에이전트의 적합성 판단 기준**
- ccusage 베이스라인과 연결한 **토큰 경제성 관점** (Wave 단위 배치가 컨텍스트 청크 관리)

**011이 018에 없는 것**:
- `productTypes` granularity 불일치의 **근본 원인 + 결정 트리** (패키지 vs 타겟 단위)
- scan-imports.py의 **아키텍처와 한계** (explicit import만 잡고 implicit extension은 못 잡는 이유)
- 그림자 의존성(FloatingPanel) 추적의 **구체적 경로** (Package.resolved → 실사용 여부 판별)
- Routing 프로토콜 Interface 이전의 **판별 기준** (`grep`으로 외부 참조 전수조사)
- scaffold 템플릿 기본 의존성 보강이라는 **예방 인프라**
