## 🚨 Critical Discovery — iOS 테스트가 8라운드 동안 "통과"였지만 실제로는 무실행

### 증상

R2~R9 iOS 워커가 매 라운드 "테스트 통과" 보고. Gate는 PASS로 처리. 하지만 QA 단계에서 **실제 xcodebuild 출력을 확인하자**:

```
Test Suite 'All tests' started at ...
Test Suite 'All tests' passed at ...
Executed 0 tests, with 0 failures (0 unexpected) in 0.000 (0.001) seconds
** TEST SUCCEEDED **
```

**0 tests executed.** 테스트 코드는 있었고 컴파일도 됐지만 *실행되지 않았다*. xcodebuild는 `exit 0`을 반환했고, 워커도 Architect도 이를 "통과"로 해석했다.

### 근본 원인

1. **Tuist + SPM productTypes 미설정** — TCA/Sharing/Perception/Dependencies/CasePaths가 기본값으로 static library로 빌드됨. 테스트 번들이 리소스 번들 경로를 찾지 못해 로딩 실패.
2. **xcodebuild CLI 명령 — `-project` vs `-workspace`** — Tuist 환경에서 `-project` 사용 시 SPM 의존성이 누락된 상태로 실행.

### Fix

**`Tuist/Package.swift`**:
```swift
productTypes: [
  "ComposableArchitecture": .framework,
  "Sharing": .framework,
  "Perception": .framework,
  "Dependencies": .framework,
  "CasePaths": .framework,
]
```

**테스트 명령**: `-project`가 아닌 `-workspace Aidy.xcworkspace` 사용 의무화.

### 사후 조치

1. **Solution 문서**: `aidy-architect/docs/solutions/2026-04-16-ios-tests-never-ran.md` 작성
2. **테스트 정책 4개 신설**:
   - `gates/test-policy.md` — 범용 6원칙 (P1-P6)
   - `gates/test-policy-server.md` — Kotlin/Spring Boot
   - `gates/test-policy-ios.md` — Swift/TCA (productTypes fix 포함)
   - `gates/test-policy-android.md` — Kotlin/Compose/MVVM
3. **CLAUDE.md 참조**: 4 레포 모두 test-policy 참조 라인 추가
4. **Gate 1 체크리스트 보강 대기**: "테스트 실행 증거(숫자)" 요구사항 추가 예정

## 배운 것 / 다음 후보

### 이번 스프린트의 진짜 메시지

1. **"Tests passed"는 주장이 아니라 증거로 뒷받침돼야 한다** — 숫자 없는 통과는 무실행과 구분 불가
2. **침묵하는 실패가 가장 위험하다** — exit 0 + "no tests to run"은 *실패로 보고되지 않는 실패*
3. **테스트 정책을 문서로 박제하면 이식 가능** — 4 레포 모두 test-policy.md 참조 한 줄 추가로 Gate 체크 일관성 확보
4. **안정성 인프라는 기능과 분리된 스프린트로** — RateLimit/RequestId/CB/Headers는 자기들끼리 묶으면 관측 일관성 증가

## AI Agent Directive

### Trigger
- autoceo 스프린트에서 "테스트 통과" 보고를 받고 있지만 실제 실행 여부가 의심될 때
- 안정성/관측성 인프라를 기능 스프린트와 별도로 진행하려 할 때

### Prerequisites
- [Aidy Journal 000](/wiki/harness-engineering/aidy-journal-000-architect-worker-baseline)
- [Aidy Journal 001](/wiki/harness-engineering/aidy-journal-001-20round-autoceo-sprint)

### Actionable Steps
1. **Gate 1에 "테스트 실행 증거(숫자)" 요구** — 'Test run with NN tests, 0 failures' 패턴 없으면 FAIL
2. **iOS Tuist + SPM 환경**: `Package.swift`의 productTypes를 `.framework`로 명시
3. **xcodebuild는 `-workspace`로만 호출** — `-project`는 SPM 누락 위험
4. **테스트 정책 문서 4종 박제** (범용 + 플랫폼별 3) + CLAUDE.md에서 참조
5. **안정성 인프라 스프린트 분리** — RateLimit/RequestId/CB/Headers를 한 묶음으로

### Anti-patterns
- **"Tests passed"를 그대로 신뢰** — 숫자 없는 통과는 무실행일 수 있음
- **`-project`로 xcodebuild 실행** — Tuist 환경에서는 SPM 의존성 누락 위험
- **안정성 기능을 기능 스프린트에 섞어 넣기** — Gate 검증 초점이 흐려짐
