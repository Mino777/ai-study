## v2.9 — Smart Notifications

AI가 메모리/기념일/건강점수를 분석해서 능동적으로 알림을 생성하는 기능.

### 핵심 설계 결정

```kotlin
// SmartNotificationService.generateNotifications(userId)
// AI가 아닌 규칙 기반 분석 — 예측 가능성 > AI 자유도
data class SmartNotification(
    val type: NotificationType,  // ANNIVERSARY, HEALTH_CHECK, MEMORY_REMINDER
    val priority: Priority,      // HIGH, MEDIUM, LOW
    val triggerReason: TriggerReason,  // 알림 생성 근거 명시
    val expiresAt: Instant       // 7일 후 자동 만료
)
```

**기존 NotificationPreference와 분리**: WO 금지 항목에 명시. 스마트 알림은 별도 Entity, 기존 설정 코드 무침범. 두 시스템이 공존하면서 각자 역할 분담.

### 3개 엔드포인트

```
GET  /api/notifications/smart           # 미처리 알림 조회 (7일 내)
POST /api/notifications/smart/generate  # 즉시 생성 트리거
POST /api/notifications/smart/{id}/dismiss
```

## v3.0 — Relationship Map

인물 간 관계를 그래프로 시각화. 노드 = 인물, 엣지 = 관계 링크.

### strength 자동 산출

직접 입력 없음. 코드가 두 인물의 공동 등장 메모리 수를 쿼리해 1~5로 환산:

```kotlin
// PersonLink.strength 자동 계산
// "지난달 민수랑 같이 간 여행" → personId A와 B가 공동 등장
// 공동 등장 횟수가 많을수록 strength 높아짐
val strength = minOf(5, coAppearanceCount / 2 + 1)
```

### 클라이언트 그래프 렌더링

iOS: `RelationshipMapView` — 외부 라이브러리 금지, **커스텀 Canvas/Path로 그래프 직접 그림**.

Android: `RelationshipMapScreen` — Compose Canvas 기반.

외부 라이브러리를 쓰지 않은 이유: 번들 사이즈 통제 + 앱 의존성 최소화. 노드 20~30개 수준이라 custom Canvas로 충분.

### 3개 엔드포인트

```
GET    /api/people/map              # 노드 + 엣지 전체
POST   /api/people/map/link         # 관계 연결 생성
DELETE /api/people/map/link/{id}    # 관계 연결 삭제
```

## v3.1 — Interaction Log

채팅과 별개로 실제 만남/통화/이벤트를 기록하는 기능.

```kotlin
data class Interaction(
    val type: InteractionType,  // CALL, MEETING, MESSAGE, EVENT, OTHER
    val title: String,
    val note: String?,
    val occurredAt: Instant,    // 과거 날짜 소급 입력 가능
    val duration: Int?          // 분 단위, nullable
)
```

`occurredAt`이 과거 날짜를 허용한다는 점이 중요. 메모리 시스템에서 "3주 전에 만났었는데"를 소급 입력할 수 있어야 한다. Relationship Timeline(v2.7)에 연동되어 `INTERACTION` 타입으로 통합 표시.

## 바선생 L5.5 → L6.0

s28에서 `architect-cli.sh`에 `crash-log` 명령 추가. 바선생(Architect 워커 제어 CLI)의 6번째 마일스톤.

### crash-log 명령 동작

```bash
./architect-cli.sh crash-log ios
# → docs/crash-logs/2026-04-21_115925_ios.md 생성
```

워커 세션이 죽었을 때:
1. `tmux capture-pane` — pane 마지막 500줄 캡처
2. 시스템 상태 수집 (swap, memory pressure, claude 인스턴스 수)
3. 원인 자동 분류 (OOM / TOKEN_OVERFLOW / RATE_LIMIT / NETWORK / ERROR / SESSION_END / UNKNOWN)
4. 마크다운 파일로 저장

### FAIL/META/VERIFY 3축

바선생 진화의 핵심 프레임. 각 축은 독립적으로 성장:

| 축 | 목적 | L5.5→L6.0 개선 |
|----|------|---------------|
| FAIL | 실패 수집/분류 | crash-log 명령으로 자동화 |
| META | 패턴 학습/문서화 | crash-log docs 디렉토리 체계화 |
| VERIFY | 검증 의무화 | Android testDebugUnitTest 의무 실행 |

크래시가 발생할 때마다 docs/crash-logs/에 기록이 쌓이고, 다음 세션에서 패턴을 분석해 대응 로직을 개선한다. 사후 분석이 아니라 **사전 방어의 데이터 축적**.

## 패턴: 서버 우선 → 클라 병렬

s28에서 완전히 정착한 dispatch 패턴:

```
[R2-server] Smart Notifications API (WO-103) 구현
      ↓
[Gate-1 서버 PASS 확인]
      ↓
[R2-ios] WO-104 + [R2-android] WO-105  ← 2-way 동시 dispatch
```
