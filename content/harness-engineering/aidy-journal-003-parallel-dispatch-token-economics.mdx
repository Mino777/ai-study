## 구조적 원인 — architect 1 : worker 3 소비 비율

### 라운드 1회의 실제 소비 구조

```
Architect 턴 (분석/dispatch)     — 짧음, 프롬프트 1~2회
  ↓
Worker server (백그라운드)       — 10~30분, 파일 수정+테스트+커밋
Worker iOS    (백그라운드, 동시)  — 10~30분, 파일 수정+테스트+커밋
Worker android(백그라운드, 동시)  — 10~30분, 파일 수정+테스트+커밋
  ↓
Architect 턴 (gate-1 검증)       — 중간
```

Architect 본인이 소비하는 토큰은 스프린트 운영에서 **가장 작은 조각**이다. 실제 소모는 병렬 워커 3명이 각자 수행하는 장시간 작업 — 라운드당 **4 Claude 인스턴스**가 동시에 org에 붙어있다.

### 수치

| 구간 | architect 소비 | 워커 3 합산 소비 | 비율 |
|------|---------------|------------------|------|
| 라운드 1회 | ~1 | ~15 | 1 : 15 |
| 10라운드 | ~10 | ~150 | 1 : 15 |

(정확한 토큰 수치는 `architect-cli.sh` 기록에 없음 — 상대 비율만 경험적 관찰. 수치 보강은 Phase 2에서 pane별 토큰 추정치 누적 파일로 가능.)

### 리밋 리셋 타이밍의 영향

- **Session 5 (리밋 여유 많을 때 시작)** — R1~R10 완주, 문제 없음
- **Session 6 (리밋 리셋 직후 시작)** — 15% 즉시 소비 + R9 429 유실

리셋 직후는 **큐에 밀려있던 리소스 + 병렬 burst 충돌**. 3-way 병렬은 리밋 여유의 함수다.

## 박제한 교훈

### 1. 3-way 병렬이 항상 최적은 아니다

- **리밋 여유 많음** → 3-way 최대 스루풋
- **리셋 직후 / 대규모 스프린트 중반** → 부분/완전 순차가 안전

### 2. dispatch 후 폴링 간격

- 2분 → **너무 공격적** (워커가 응답 시작도 전에 상태 체크)
- **5~7분 권장** (wait_for_idle 기준)

### 3. 429 감지 + backoff 런타임 적응

`architect-cli.sh tmux_send` 에 4-Layer 방어 박제 — dispatch 후 30초 윈도우로 pane 감시 → 429 패턴 매칭 시 5분 backoff → 1회 재시도. 구현 세부(환경변수 제어, 재귀 재시도 로직, 매칭 패턴)는 [tmux flush 자동화 패턴](/wiki/harness-engineering/tmux-flush-automation-pattern#layer-3--429--rate-limit-감지--backoff) 참조.

**판단 기준**:
- 리밋 여유 많음 + 단일 실패 → 자동 backoff로 충분
- 리밋 여유 부족 + 반복 실패 → `send-seq` 순차 모드로 수동 전환

### 4. Sequential 모드 (send-seq)

`architect-cli.sh tmux_send_sequential` 로 각 워커 dispatch 후 idle 대기. `is_pane_idle` 이 `"esc to interrupt"` 부재 + 프롬프트 시그니처 존재를 감지. 구현은 [tmux flush 자동화 패턴 Layer 4](/wiki/harness-engineering/tmux-flush-automation-pattern#layer-4--sequential-dispatch-send-seq) 참조.

### 5. 429 중단 후 재개는 짧은 continue

- 전체 프롬프트 재전송은 **낭비**
- 워커는 partial 파일 저장 상태로 유지됨
- **200자 내외 continue 프롬프트**로 문맥 이어붙이기 가능

## AI Agent Directive

### Trigger
- autoceo 스프린트 기획 시 병렬 워커 수 결정이 필요할 때
- 3-way dispatch 후 429 / usage limit 메시지를 관측했을 때
- 토큰 리밋 리셋 직후 대규모 스프린트를 돌리려 할 때

### Prerequisites
- [Aidy Journal 001](/wiki/harness-engineering/aidy-journal-001-20round-autoceo-sprint) — 20R autoceo 구조
- [Aidy Journal 002](/wiki/harness-engineering/aidy-journal-002-stability-observability-test-integrity) — Gate 체크리스트

### Actionable Steps

1. **스프린트 시작 전 리밋 체크** — `ccusage` 또는 사용량 화면으로 여유 확인
2. **여유 많음** → 3-way 병렬 유지 (`tmux_send` 기본 모드)
3. **리셋 직후 / 스프린트 중반** → `send-seq` 로 순차 또는 2-way 혼합
4. **dispatch 후 폴링 간격을 5~7분으로** — 2분은 너무 공격적
5. **429 감지 시** — `AIDY_SEND_429_DETECT=1` 기본 활성, 5분 backoff + 1회 재시도
6. **재시도 실패 시** — 순차 재개로 폴백, 짧은 continue 프롬프트로 이어붙이기

### Anti-patterns

- **워커 수 고정 규칙** ("항상 3-way") — 리밋 여유 무시한 경직 운영
- **재디스패치로 재시작** — 파일 저장된 partial 상태 버리고 처음부터 → 토큰 낭비
- **tmux_send 후 1~2분 폴링** — 워커가 응답 시작도 전에 상태 판정 불가
- **architect 단일 인스턴스 토큰만 추적** — 실제 소비의 1/16에 해당
