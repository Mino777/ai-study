## 컨텍스트

[Aidy Journal 000](/wiki/harness-engineering/aidy-journal-000-architect-worker-baseline) 의 Architect-Worker 모델에서 Architect는 `architect-cli.sh send <target> "<prompt>"` 로 워커에게 dispatch한다. 프롬프트는 tmux pane에만 존재하고 휘발 — 세션 종료 후 재현 불가.

Session 6 R1 (`5a75faf`) 에서 이 프롬프트를 파일로 자동 기록하는 인프라 도입. 목적:
- 프롬프트 엔지니어링 학습 (잘 작동한 패턴 발굴)
- 과거 WO 재현 (같은 작업 다시 시켜볼 때)
- 회고/문서화 자동화 (retro 작성 시 프롬프트 참조)

---

## 구현 — architect-cli.sh 내부 기록

```bash
# architect-cli.sh tmux_send() 발췌 (L82~L94)

# 프롬프트 히스토리 로깅 (프롬프트 엔지니어링 학습용)
# 파일명: docs/worker-prompts/YYYY-MM-DD.md (날짜별 append)
local log_dir="$ARCH_DIR/docs/worker-prompts"
mkdir -p "$log_dir"
local log_file="$log_dir/$(date '+%Y-%m-%d').md"

# 파일 없으면 헤더 생성
if [[ ! -f "$log_file" ]]; then
    printf '# 워커 프롬프트 로그 — %s\n\n> architect-cli.sh `send` 로 워커에게 보낸 프롬프트 원문. 프롬프트 엔지니어링 학습/회고용.\n\n' \
        "$(date '+%Y-%m-%d')" > "$log_file"
fi

# append: 구분선 + timestamp → target 헤더 + fenced block
{
    printf '\n---\n\n## %s → %s\n\n' "$(date '+%H:%M:%S')" "$target"
    printf '```\n%s\n```\n' "$prompt"
} >> "$log_file"
```

## 자동화 범위

### 기록됨
- **dispatch 프롬프트 원문** — `architect-cli.sh send <target> "<prompt>"` 호출 시 자동
- **timestamp + target** — 언제 누구에게
- **컨텍스트 전체** — `작업`, `금지`, `실행 증거`, `커밋 규칙` 등 프롬프트 내 섹션 그대로

### 기록 안 됨 (의도적)
- **워커 응답** — git diff + docs/retros + 커밋 메시지로 이미 박제
- **429 backoff 재전송** — 같은 프롬프트 재전송은 첫 기록으로 충분
- **사용자 stdin 입력** — tmux_send 호출 외 경로는 로깅 안 함

**비대칭 로깅**: 프롬프트는 휘발 → 기록 필수, 응답은 영속 → 재기록 불필요.

## 학습 사이클

### Step 1 — 축적

매 dispatch가 자동 기록. 몇 주 지나면 수백 개 프롬프트.

### Step 2 — 패턴 발굴

스프린트 종료 후 로그 스캔:

```bash
# 테스트 증거를 요구한 프롬프트만 추출
grep -l "실행 증거 필수" docs/worker-prompts/*.md

# 특정 타겟만
grep -A 30 "→ ios" docs/worker-prompts/2026-04-*.md

# 금지 섹션 포함 프롬프트
grep -B 2 -A 5 "^금지:" docs/worker-prompts/*.md
```

### Step 3 — README에 잘 작동한 패턴 박제

`docs/worker-prompts/README.md`:

```markdown
## 잘 작동하는 패턴
- 번호 매긴 작업 단계 (1, 2, 3…)
- `금지:` 섹션 — 범위 이탈 방지
- `실행 증거 필수:` — 테스트 숫자 요구
- 커밋 메시지 형식 지정 `[Rn-target] type: 설명`
- 파일 수 제한 `파일 N개 이하`

## 안티 패턴
- 모호한 "개선해줘" 지시 — 구체적 파일명/함수명 명시
- 검증 불가능한 요구 ("깔끔하게") — 측정 가능한 기준으로 치환
- 너무 많은 의존 작업 한 라운드에 — WO 분할 권장
```

### Step 4 — 다음 스프린트 프롬프트에 패턴 재활용

Compound 루프. 아키텍트가 build_prompt 할 때 README의 "잘 작동하는 패턴" 을 baseline 으로 사용.

## Backfill — 자동화 이전 스프린트

R1 도입 이전(s1~s5 + s6-R0)의 프롬프트는 수동 백필:

```
docs/worker-prompts/
├── 2026-04-17.md                      # 자동 기록 (R1~)
├── 2026-04-16.md                      # 자동 기록
└── autoceo-s6-backfill.md             # 수동 백필 (R1 이전)
```

백필은 commit message + retro + 메모리 조각으로 복원 — 완벽하지 않지만 주요 패턴은 보존.

**교훈**: 로깅 인프라는 가능한 일찍 도입. 늦게 도입하면 복원 비용 큼.

## 이식 가능성

이 패턴을 다른 멀티 에이전트 시스템에 이식할 때 **핵심 조건**:

### 필수
1. **Dispatch 유일 경로** — 모든 agent 호출이 한 함수/경로를 거쳐야 함
2. **파일 시스템 접근** — 로그 append 가능한 환경
3. **Fenced block 지원** 포맷 (마크다운/JSON/YAML 등)

### 권장
4. **날짜별 파일 분리** — 단일 파일 누적은 수 MB 도달 시 GitHub 렌더 느려짐
5. **README로 패턴 큐레이션** — 원본 로그 + 정제된 패턴을 함께
6. **git 포함** — retro/PR 에서 참조 가능

### 비권장
- 응답까지 기록 — 중복 + 비용
- DB/Redis 같은 외부 스토리지 — git history가 이미 충분한 아카이빙

## AI Agent Directive

### Trigger
- Multi-agent / Architect-Worker 하네스 설계 착수 시
- 프롬프트 엔지니어링 학습을 체계화하려 할 때
- Architect 세션 종료 후 프롬프트 복원이 필요해진 경험

### Prerequisites
- Dispatch 유일 경로 (tmux_send 같은 단일 함수)
- 파일 append 가능 환경
- [Aidy Journal 000](/wiki/harness-engineering/aidy-journal-000-architect-worker-baseline) — Architect-Worker 모델

### Actionable Steps

1. **Dispatch 함수에 직접 로깅 추가** — 호출자 우회 불가능한 위치
2. **날짜별 파일** — `docs/<path>/YYYY-MM-DD.md` 로 분리
3. **헤더 자동 생성** — 파일 없으면 `# 워커 프롬프트 로그 — YYYY-MM-DD` + 설명
4. **포맷**: `## HH:MM:SS → target` + fenced code block
5. **Backfill** — 자동화 이전 스프린트는 retro/커밋으로 수동 복원
6. **README로 패턴 큐레이션** — "잘 작동하는 패턴" 섹션
7. **이식 시 응답은 기록 안 함** — 비대칭 로깅 원칙 유지

### Anti-patterns

- **README 지침만 두고 호출자 자발적 로깅** — 누구는 하고 누구는 안 함, 커버리지 깨짐
- **응답까지 기록** — git diff + retro 중복, 비용 대비 ROI 낮음
- **단일 파일 누적** — 수 MB 도달 시 렌더 느려짐
- **DB/외부 스토리지** — git 포함 안 되어 PR 리뷰 어려움
- **프롬프트 정제 후 기록** — 원문 소실, 학습 가치 감소
