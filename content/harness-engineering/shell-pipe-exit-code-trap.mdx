## 한 줄 요약

Shell 파이프에서 `$?`는 마지막 명령의 종료 코드만 반환한다. 빌드 명령을 grep/tail에 파이프하면 빌드 실패를 감지할 수 없다.

## 라이브 사고 사례 — moneyflow 2026-04-14

moneyflow cron-helpers.ts에서 `PromiseLike` 타입 불일치 에러가 발생했다. 해당 코드는 AI 에이전트가 수정한 파일이었고, QA gate pre-commit 훅이 빌드를 검증하도록 설계되어 있었다.

그런데 훅이 통과했다. Vercel 배포까지 에러가 흘러들어갔다.

원인은 훅 스크립트였다:

```bash
# AS-IS (문제 있는 코드)
npm run build | grep -E "(error|Error)" | tail -5
if [ $? -ne 0 ]; then
  echo "Build failed"
  exit 1
fi
```

`npm run build`가 타입 에러로 exit code 1을 반환했다. 하지만 `$?`가 잡은 것은 `tail -5`의 종료 코드 — 항상 0이다. 게이트가 실패를 보지 못했다.

---

## 왜 이런 일이 생기나

Shell 파이프 `cmd1 | cmd2 | cmd3`에서 각 명령은 병렬로 실행된다. `$?`는 파이프라인의 **마지막 명령**(`cmd3`) 종료 코드만 반영한다.

`cmd1`이 exit code 1로 실패해도, `cmd2`와 `cmd3`이 정상 종료하면 `$?`는 0이다. `tail`은 표준 입력만 읽고 항상 성공하므로, 앞에서 무슨 일이 벌어지든 0을 반환한다.

## 수정 3패턴

### 패턴 A — 파일 리다이렉트 (가장 명확)

파이프를 완전히 없애고 종료 코드를 바로 캡처한다.

```bash
npm run build > /tmp/build.log 2>&1
BUILD_EXIT=$?
cat /tmp/build.log | grep -E "(error|Error)" | tail -5
if [ $BUILD_EXIT -ne 0 ]; then
  echo "Build failed (exit $BUILD_EXIT)"
  exit 1
fi
```

종료 코드와 출력 필터링을 분리한다. 가장 읽기 쉽고 오해가 없다.

### 패턴 B — `set -o pipefail` (가장 간결)

파이프 내 어느 명령이든 실패하면 파이프 전체가 non-zero를 반환하도록 설정한다.

```bash
set -o pipefail
npm run build | grep -E "(error|Error)" | tail -5
if [ $? -ne 0 ]; then
  echo "Build failed"
  exit 1
fi
```

단, `grep`이 매칭 결과를 찾지 못하면 exit code 1을 반환하므로, 에러가 없는 성공 빌드도 실패로 판정될 수 있다. grep에 `|| true`를 붙이거나 패턴 A를 권장한다.

### 패턴 C — `PIPESTATUS` 배열 (bash 전용)

bash는 파이프 내 각 명령의 종료 코드를 `PIPESTATUS` 배열에 저장한다.

```bash
npm run build | grep -E "(error|Error)" | tail -5
BUILD_EXIT=${PIPESTATUS[0]}
if [ $BUILD_EXIT -ne 0 ]; then
  echo "Build failed (exit $BUILD_EXIT)"
  exit 1
fi
```

`${PIPESTATUS[0]}`은 파이프 첫 번째 명령(`npm run build`)의 종료 코드다. bash 전용 — `sh`나 `zsh`에서는 동작이 다를 수 있다.

## 실전 수정 — moneyflow Before / After

```bash
# Before
npm run build | grep -E "(error|Error)" | tail -5
if [ $? -ne 0 ]; then exit 1; fi

# After
npm run build > /tmp/build.log 2>&1
BUILD_EXIT=$?
tail -20 /tmp/build.log
if [ $BUILD_EXIT -ne 0 ]; then exit 1; fi
```

## 이 함정이 자주 발생하는 곳

| 위치 | 패턴 |
|------|------|
| CI 게이트 | `npm run build \| grep error; if [ $? -ne 0 ]` |
| pre-commit 훅 | `tsc \| tail -10; exit $?` |
| health check 스크립트 | `curl ... \| jq '.status'; if [ $? -eq 0 ]` |
| 로그 파싱 | `docker logs app \| grep FATAL; if [ $? -ne 0 ]` |

공통점: 실제 검증 대상 명령의 종료 코드를 파이프 뒤 명령이 덮어쓴다.

## For AI Agents

**Trigger**: shell 스크립트에서 빌드/테스트 결과를 파이프로 필터링한 후 `$?`로 성공 여부를 판단하는 코드를 발견했을 때.

**Actionable**:
1. 파이프를 제거하고 출력을 파일로 리다이렉트한 뒤 종료 코드를 변수에 저장 (패턴 A)
2. 또는 스크립트 상단에 `set -o pipefail` 추가 후 grep 결과 처리에 주의 (패턴 B)
3. bash 확정 환경이면 `${PIPESTATUS[0]}` 사용 (패턴 C)

**Anti-pattern**:
```bash
# 절대 이렇게 쓰지 말 것
cmd | grep | tail
if [ $? -ne 0 ]; then ...  # $? 는 tail의 코드. cmd 실패를 감지 못함
```

**AI 에이전트 환경에서 특히 위험한 이유**: AI가 생성하거나 수정한 코드는 미묘한 타입 에러나 빌드 실패를 포함할 수 있다. QA gate가 이 함정으로 인해 통과시키면, 검토되지 않은 코드가 프로덕션에 배포된다. 하네스의 신뢰도가 코드 자체의 신뢰도를 결정한다.
