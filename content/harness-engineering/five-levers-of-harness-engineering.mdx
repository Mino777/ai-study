## 왜 지금 하네스 엔지니어링인가

영상 제목 자체가 논지를 요약해 준다 — **"프롬프트 엔지니어링은 끝났습니다: 이제 '하네스'의 시대입니다"**.

이 프레임을 여러 한국어 정리글이 공통적으로 이렇게 풀이한다: 2024년은 프롬프트 엔지니어링의 해였고, 2025년은 컨텍스트 엔지니어링으로 관심이 이동했으며, 2026년은 "하네스 엔지니어링"이 새로운 중심이 되고 있다는 흐름이다. 내가 이 위키에 쌓고 있는 카테고리 순서([prompt-engineering](/wiki/prompt-engineering/chain-of-thought) → [context-engineering](/wiki/context-engineering/context-compression) → harness-engineering)가 정확히 그 진화 궤적과 겹친다. 그래서 이 글은 내가 위키에 이미 기록한 [Harness Engineering 개요](/wiki/harness-engineering/harness-engineering-overview)의 **다음 단계 — 실전 레버 — 를 한 층 더 쌓는 작업**이다.

### 하네스(Harness)의 본뜻

하네스는 원래 말에게 채우는 고삐와 안장이다. 말의 통제가 이 장비의 설계에 의존한다는 점에서 이 비유는 AI 에이전트 맥락에도 그대로 옮겨진다 — 여러 정리글과 트위터/Threads 포스트가 공통적으로 이 "말과 고삐" 비유를 쓰고 있다.

내 식으로 옮기면 이렇다: LLM은 야생마 같아서 혼자 두면 멀리 갈 수 있지만, 내가 원하는 방향으로 가지는 않는다. **프롬프트**는 채찍에 가깝다 — 순간의 지시. **컨텍스트**는 시야를 정해주는 안대에 가깝다. **하네스**는 말과 기수 사이의 물리적 연결 자체다. 2026년의 관심이 여기로 모이는 이유는, 전자 두 개가 이미 "해결된 문제"에 가까워지고 있어서다.

## "보고-잡고-되살리기" — 3단계 제어 프레임

이 프레임은 내가 교차 확인한 여러 정리글에서 가장 실용적이었던 것이다. 하네스가 무엇을 해결하려는지 한 문장으로 표현해 준다.

```mermaid
flowchart LR
    A[에이전트 실행] --> B["보고<br/>Observe"]
    B --> C["잡고<br/>Intercept"]
    C --> D["되살리기<br/>Recover"]
    D --> A

    B -.SSOT<br/>progress.md.- B
    C -.2-part agent<br/>sub-agent.- C
    D -.checkpoint<br/>git worktree.- D
```

| 단계 | 해결하는 문제 | 구현 |
|---|---|---|
| **1. 보고 (Observe)** | 긴 작업 중 에이전트가 현재 위치를 잊는 컨텍스트 드리프트 | SSOT(Single Source of Truth) 파일 — `progress.md`, `architecture.md` |
| **2. 잡고 (Intercept)** | 에러 발생 시 에이전트가 폭주 | 훅(hooks) + 서브에이전트 분업 |
| **3. 되살리기 (Recover)** | 망친 상태에서 안전하게 복원 | 체크포인트 + git worktree |

여러 정리글이 공통적으로 강조하는 핵심 원칙은 이렇다 — **에이전트에게 큰 작업을 시키기 전에, 지금 이 작업이 망했을 때 안전하게 돌아갈 수 있는 지점이 어디인지 먼저 확정해 두어야 한다.** 바이브 코딩으로 2시간 신나게 달린 뒤 5번째 파일에서 파이프가 고장나면 처음부터 다시 해야 한다는 공포를 모두가 한 번쯤 겪었기 때문에 이 원칙이 설득력이 있다.

## 5가지 레버

교차 확인한 정리글들이 공통적으로 언급하는 실전 하네스 구성 요소는 아래 5개로 수렴한다. 각 레버는 독립적이고, 작은 것부터 쌓아올릴 수 있다는 점이 중요하다.

### 레버 1 — 시스템 프롬프트 (CLAUDE.md)

**원칙: 60줄 이하로 유지한다.**

정리글들이 반복해서 제시하는 기준점이 "60줄"이다. 처음에는 납득이 안 됐다 — 내가 아는 건 많은데 왜 다 못 적어?

이유는 이거다:

- 시스템 프롬프트는 **매 턴 전체가 입력 컨텍스트에 들어간다.** 길수록 토큰 비용 + 주의 희석.
- 에이전트는 긴 규칙 목록의 **중간을 잘 지키지 못한다** (중간 소실 — lost in the middle).
- 디렉터리 구조를 나열해봐야 파일명이 바뀌면 거짓말이 된다. 코드를 직접 읽게 하는 편이 정확하다.

**들어가야 할 것:**

```markdown
# 프로젝트 이름

한 줄 요약.

## 기술 스택
- Next.js 15 (App Router)
- TypeScript
- Tailwind CSS 4

## 테스트/빌드 명령
- `npm run build` — 프로덕션 빌드
- `npm run test` — 단위 테스트

## 코딩 컨벤션
- 한글 커밋 메시지
- 컴포넌트는 함수형 + TypeScript strict

## 금지 사항
- `git push --force` 금지
- 프로덕션 환경변수 커밋 금지
```

**들어가면 안 되는 것:** 파일 트리, 모든 라이브러리 사용법, "어떻게 생각하세요" 같은 감성.

### 레버 2 — 스킬 (점진적 지식 공개)

시스템 프롬프트가 60줄로 부족할 때 해답은 "늘리기"가 아니라 **"분산시키기"**다. 태스크별 별도 파일에 지식을 쪼개 넣고, 에이전트가 필요할 때만 불러오게 한다.

```
.claude/skills/
├── db-migration.md       # DB 마이그레이션 규칙만
├── api-endpoint.md       # API 라우트 작성 규칙만
├── frontend-component.md # 리액트 컴포넌트 규칙만
└── mdx-content.md        # MDX 작성 규칙만
```

이 글이 올라가는 ai-study 프로젝트에 대입하면:

- `mdx-content.md` — "HTML void 태그는 반드시 self-closing (`<br />`)", "Mermaid subgraph 이름에 공백이 있으면 `id [\"label\"]` 형식"
- `new-entry.md` — 새 MDX 엔트리 작성 시 frontmatter 필드 가이드
- `compound.md` — 스프린트 마무리 시 CHANGELOG/retro/solution 작성 규칙

이게 프롬프트 엔지니어링과 결정적으로 다른 점은, **실패에서 배운 규칙이 체계적으로 축적되는 저장소**가 된다는 것이다. [compound engineering](/projects)의 핵심 아이디어와 정확히 같다.

### 레버 3 — MCP 서버 (도구 확장)

에이전트가 세상에 손을 뻗는 채널. Linear 이슈 읽기, Sentry 에러 조회, DB 쿼리 실행, Figma 디자인 읽기 등. 각각이 하나의 MCP 서버다.

정리글들이 입을 모아 제시하는 가이드는 간단하다: **"2~3개로 시작해라."**

MCP 서버가 많아질수록:
- 에이전트가 어떤 도구를 언제 쓸지 고민하는 시간이 길어짐
- 각 도구의 설명문이 시스템 프롬프트에 누적됨 (토큰 폭발)
- 디버깅 지옥 — 어느 도구가 잘못 반환했는지 추적 난이도 증가

**시작 세트 추천:**

```typescript
// 프로젝트 초기엔 이 셋만
const essentialMCPs = [
  "filesystem",   // 파일 읽기/쓰기 (기본)
  "git",          // 브랜치/커밋 조회
  "web-fetch",    // 외부 문서 읽기 (이번 영상 정리처럼)
];
```

필요가 생기면 그때 추가. "혹시 모르니까 켜두자"는 하네스 과적합의 첫걸음이다.

### 레버 4 — 서브에이전트 (컨텍스트 방화벽)

긴 작업에서 가장 무서운 건 메인 컨텍스트의 오염이다. 에이전트가 실패한 시도, 디버그 출력, 중간 파일 탐색 결과로 컨텍스트를 다 채워버리면 진짜 중요한 판단을 할 용량이 남지 않는다.

**해결:** 세부 작업을 서브에이전트에 위임하고, **결과만 요약해서 메인에 돌려준다.**

```mermaid
sequenceDiagram
    participant Main as 메인 에이전트
    participant Sub as 서브에이전트
    participant FS as 파일시스템

    Main->>Sub: "src/components/ 전체 구조 조사하라"
    Sub->>FS: ls, read, grep (수십 번)
    FS-->>Sub: 원시 결과 (수천 토큰)
    Sub->>Sub: 분석 + 요약
    Sub-->>Main: "Button.tsx / Card.tsx / ... 총 17개. 공통 패턴: X"
    Note over Main: 메인 컨텍스트는 요약만 받음
```

Claude Code의 `Agent` / `Explore` 서브에이전트가 바로 이 패턴을 구현한다. 검색 · 대규모 파일 탐색 · 별도 리서치 같은 작업을 서브에이전트에 위임하면, 메인 컨텍스트는 최종 판단에만 집중할 수 있다.

### 레버 5 — 훅 (자동 체크포인트)

가장 저평가된 레버. 훅은 에이전트가 특정 동작을 할 때 **자동으로 실행되는 스크립트**다.

```mermaid
flowchart TD
    A[에이전트 파일 편집] --> B{pre-commit hook}
    B -->|빌드 실패| C[커밋 차단]
    B -->|빌드 성공| D[커밋 통과]
    D --> E{post-push hook}
    E --> F[/compound 리마인더]
```

**이 프로젝트의 실제 훅 구성:**

- `pre-commit`: `npm run build` 자동 실행 → 실패 시 커밋 차단. 이 훅 덕에 AI 과외 선생님이 자동 생성한 MDX에서 `<br>` 태그나 Mermaid subgraph 공백 이름 같은 이슈가 production까지 가기 전 여러 번 차단됐다. 관련 솔루션 문서는 `docs/solutions/mdx/`에 축적되고 있다.
- `post-push`: `/compound 실행하세요` 리마인더 표시 → 스프린트 마무리를 잊지 않게 함.

핵심은 **같은 실수를 두 번 하지 않게 만드는 것**이 사람의 기억이 아니라 시스템 규칙이라는 점이다.

## 매주 금요일 5분

여러 정리글에서 반복해서 등장하는 운영 팁 하나가 있다. 매주 금요일에 5분만 투자해서, 그 주에 에이전트가 저지른 실수 중 하나를 골라 하네스에 반영하라는 것. 딱 5분이면 되는 이유는 할 수 있는 일이 셋뿐이기 때문이다:

1. CLAUDE.md에 한 줄 금지 규칙 추가
2. `.claude/skills/` 중 하나의 파일에 경고 블록 추가
3. 훅 스크립트에 체크 하나 추가

나는 여기서 멈추지 않고, 이 원칙을 compound engineering과 결합시켰다. 이 ai-study 프로젝트의 `docs/solutions/` 디렉터리에 쌓이는 솔루션 문서들이 사실상 "그 주의 실패 아카이브"가 되고, 프로젝트 내부 `/compound` 슬래시 커맨드가 5분 금요일을 자동화한다.

### 왜 이게 작동하는가

정리글들이 일관되게 짚어주는 핵심 원리는 이렇다. 에이전트가 시간이 갈수록 신뢰할 만해지는 이유는 **모델이 더 좋아져서**가 아니라 **하네스 시스템 자체가 실수를 반영하며 학습하기 때문**이라는 것.

모델은 Anthropic/OpenAI가 키운다. 하지만 **당신의 하네스는 당신이 키운다.** 1년 뒤 같은 모델을 써도, 하네스가 두꺼워진 당신의 프로젝트는 완전히 다른 생산성을 낸다. 복리 효과.

## 안티패턴 — 하네스 과적합

여러 자료가 공통적으로 경고하는 초보자 함정. 규칙을 많이 쓸수록 좋다는 오해가 가장 큰 적이다.

| 안티패턴 | 증상 | 해결 |
|---|---|---|
| **CLAUDE.md를 성경처럼 길게 쓴다** | 에이전트가 중간 규칙을 무시하기 시작 | 60줄 원칙 엄수 + 스킬 파일로 분산 |
| **모든 MCP 서버를 다 켠다** | 첫 응답까지 30초+, 도구 선택 실수 증가 | 2-3개로 시작, 필요시 추가 |
| **훅을 너무 많이 건다** | 커밋 한 번에 10초+ 대기 | 진짜 차단해야 하는 것만 (빌드 실패 수준) |
| **스킬 파일을 미리 다 만들어둔다** | 대부분 참조되지 않고 낡은 규칙이 됨 | 실제 실수가 발생한 뒤 작성 |

GPTers 정리글이 이 아이디어를 한 줄로 요약한다: **"최소한의 규칙으로 최대 효과"**. 규칙의 양이 아니라 **실제로 실수를 막는 최소 세트**를 찾는 것이 기술이다.

나는 이 경고가 특히 필요했다. 뭐든지 최대화하려는 기질이라, 처음에 CLAUDE.md에 프로젝트의 모든 것을 담고 싶은 충동을 참아야 했다.

## AI Agent Directive

**Trigger**: 에이전트 통제 체계를 새로 설계하거나, 기존 CLAUDE.md/훅/MCP 구성을 평가해야 할 때. 5가지 레버의 *우선순위*를 결정해야 할 때.

**Prerequisites**:
- [Harness Engineering Overview](/wiki/harness-engineering/harness-engineering-overview)
- [Five-Levers 원문 유튜브/블로그](/www.youtube.com/watch?v=6gvnDSAcZww) (선택)

### Actionable Steps
1. 현재 프로젝트에서 5가지 레버의 상태 진단 (각각 exists?/mature?)
2. 레버 1 (CLAUDE.md) — 60줄 체크. 초과면 스킬 파일로 분산
3. 레버 2 (Skill) — 2-3개부터 시작. "혹시 모르니 다 켜자" 금지
4. 레버 3 (MCP) — 필수 MCP 3개 결정 후 필요시 추가
5. 레버 4 (Subagent) — 긴 탐색을 분리하고 요약만 메인으로 받기
6. 레버 5 (Hook) — pre-commit build 1개, post-push 알림 1개부터
7. "보고-잡고-되살리기" 3단계 반영 확인 (SSOT → 훅 → 체크포인트)
8. 안티패턴 4가지 체크리스트로 자가진단 (CLAUDE.md 길이, MCP 과다, 훅 과다, 스킬 미리 다 만들기)

### Anti-patterns
- ❌ CLAUDE.md를 240줄 이상으로 길게 작성 (중간 규칙 무시)
- ❌ MCP 서버 10개 다 켜기 ("혹시 모르니")
- ❌ 훅 10개 동시에 거기 (커밋 10초+ 대기)
- ❌ 스킬 파일 미리 다 작성 (실제 실수 없이 만든 규칙은 낡음)
- ❌ 레버 5개를 1주일 안에 다 구축 (한 번에 한 레버)

## 자기 점검

1. 내 프로젝트의 CLAUDE.md는 몇 줄인가? 60줄 원칙을 위반한다면 무엇을 스킬 파일로 옮길 수 있나?
2. 지금 에이전트에게 큰 작업을 시킨다면, "망했을 때 안전하게 돌아갈 지점"을 먼저 정해둘 수 있는가?
3. 지난 한 주 동안 에이전트가 같은 실수를 두 번 이상 한 적이 있나? 그 실수는 하네스에 반영되었는가?
4. 내 프로젝트에서 가장 먼저 추가할 MCP 서버 3개는 무엇이고, 왜 그것들인가?
5. **(열린 질문)** 이 개념을 동료 개발자에게 설명한다면, 어떤 비유를 쓸 것인가? 여러 자료가 공통적으로 쓰는 "말과 고삐" 비유가 최선인지, 더 나은 것이 있을지 생각해보자.

### 실습 과제

이 위키 프로젝트의 `CLAUDE.md`를 실제로 열어 **60줄 버전의 초안**을 만들어 보자. 지금의 내용 중 무엇이 "진짜 규칙"이고 무엇이 "참고 정보"인지 구분하는 연습 자체가 하네스 엔지니어링의 핵심 근육이다.

## 출처

### 정보 상태 고지

이 글은 **영상 직접 시청 없이** 작성되었다. 나는 유튜브 메타데이터(제목·채널)만 확보한 상태에서, 이 주제에 대한 한국어 정리 블로그·커뮤니티 포스트 3개 이상을 교차 확인해 **일관되게 등장하는 공통 내용만** 엔트리에 반영했다. 따라서 본문의 모든 주장은 "아래 보강 자료들의 교집합에서 나온 것"이며, 원본 영상에만 고유하게 있는 내용이나 특정 인용구는 포함하지 않았다.

만약 원본 영상을 직접 시청할 기회가 생기면, 이 엔트리를 업데이트하면서 영상에서만 나오는 고유한 내용·구체 예시·직접 인용구를 추가할 예정이다.
