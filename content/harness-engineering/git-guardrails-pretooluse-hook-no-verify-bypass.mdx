## 왜 쓰는가

> 메모리 룰 위반 시 행동 레벨로 escalate. 메모리는 *상기*에 의존하므로 한 번이라도 위반되면 즉시 PreToolUse hook으로 올린다.

[mattpocock/skills](https://github.com/mattpocock/skills) (MIT)의 `git-guardrails-claude-code`를 한국어 메시지로 이식하면서 두 가지를 추가했다:

1. **`--no-verify` 차단 패턴 추가** — pre-commit/pre-push hook 우회 시도 차단 (mattpocock 원본은 미포함)
2. **grep 옵션 해석 버그 픽스** — `--no-verify`를 그대로 패턴으로 넘기면 grep이 자기 옵션으로 해석해 차단 자체가 무력화됨

## 1단계: PreToolUse hook 스크립트

```bash
#!/bin/bash
# .claude/hooks/block-dangerous-git.sh
# PreToolUse hook (Bash matcher) — 위험 git 명령 실행 전 차단.

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command' 2>/dev/null)

DANGEROUS_PATTERNS=(
  "git push"
  "git reset --hard"
  "git clean -fd"
  "git clean -f"
  "git branch -D"
  "git checkout \\."
  "git restore \\."
  "push --force"
  "reset --hard"
  "--no-verify"        # ← 우회 차단 핵심
)

for pattern in "${DANGEROUS_PATTERNS[@]}"; do
  if echo "$COMMAND" | grep -qE -e "$pattern"; then    # ← -e 분리 필수
    echo "🚫 BLOCKED: '$COMMAND'" >&2
    echo "  → 차단 패턴: '$pattern'" >&2
    echo "  → 사용자가 명시적으로 이 명령을 차단함 (git guardrails)" >&2
    echo "  → 우회 방법: 사용자에게 직접 실행 요청" >&2
    echo "" >&2
    echo "참고:" >&2
    echo "  • git push 금지 — CLAUDE.md '커밋 후 개발자에게 push 요청'" >&2
    echo "  • --no-verify 금지 — pre-commit 우회 차단 (테스트/lint 통과 의무)" >&2
    echo "  • git reset --hard / git clean — 작업 손실 위험. 사용자 확인 필수" >&2
    exit 2
  fi
done

exit 0
```

## 2단계: settings.json 등록

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "bash $CLAUDE_PROJECT_DIR/.claude/hooks/block-dangerous-git.sh" }
        ]
      }
    ]
  }
}
```

`~/.claude/settings.json`(글로벌) 또는 프로젝트 `.claude/settings.json`(로컬). 글로벌이면 모든 프로젝트에 적용.

## 핵심 버그: `grep -qE "$pattern"` 우회

처음 박았을 때 차단이 동작 안 했다. 이유:

```bash
# 잘못된 버전
echo "git push --no-verify" | grep -qE "--no-verify"
# → grep이 "--no-verify"를 자기 옵션으로 파싱 시도
# → grep 옵션 아님 → 패턴 인자 비어버림
# → 매칭 0, 차단 실패
```

```bash
# 픽스된 버전
echo "git push --no-verify" | grep -qE -e "--no-verify"
# → -e 뒤가 명시적 패턴 인자
# → 정상 매칭 → exit 2 차단
```

`--`로 시작하는 패턴이 하나라도 있으면 **모든 패턴**에 `-e`를 붙여야 한다. (DANGEROUS_PATTERNS 배열 중 하나만 `--`로 시작해도 해당 iteration에서 우회됨)

> ⚠️ 이 버그는 *조용한 실패*다. 차단 메시지가 안 떠도 사용자는 "오, 막혔겠지" 가정하고 넘어간다. 실제로는 모든 `--no-verify` 가 통과. **자동 수정 도구는 idempotent + 자기 검증이 없으면 silent 누적 손상을 만든다** ([feedback_self_modifying_tools](/wiki/harness-engineering) 메모리 패턴과 동일).

## 차단 대상 정당화

| 패턴 | 차단 사유 |
|------|----------|
| `git push` | 커밋 후 사용자에게 push 요청 (CLAUDE.md 룰) |
| `git push --force` | 리모트 history 파괴 — 사용자 확인 필수 |
| `git reset --hard` | 작업 손실. 머지 충돌 해결 중 untracked 손실 빈발 |
| `git clean -f` / `-fd` | untracked 파일 영구 삭제 |
| `git branch -D` | 강제 브랜치 삭제 (`--delete` 우회) |
| `git checkout .` / `git restore .` | 모든 unstaged 변경 폐기 |
| `--no-verify` | pre-commit/pre-push hook 우회 — 테스트·lint 우회 금지 |

## 화이트리스트 (false positive 방지)

기본은 *전체 차단*이지만 다음 경로는 grep 자체를 스킵:

```bash
# 훅 파일 자신 + 메모리 파일
if [[ "$COMMAND" == *.claude/hooks/* ]] || [[ "$COMMAND" == *memory/* ]]; then
  exit 0
fi
```

훅 파일 안에 차단 패턴 문자열이 들어있는 게 자연스러우니, 그 파일을 편집하는 cat/grep/edit 자체는 통과시켜야 한다.

## 우회가 필요할 때

사용자가 명시적으로 `git push`를 원할 때:

1. 에이전트가 자동으로 push 시도 → exit 2 차단
2. stderr 메시지가 "사용자에게 직접 실행 요청" 안내
3. 사용자가 직접 터미널에서 push (혹은 `! git push`로 슬래시 실행)

이게 곧 *의도된 마찰* — 자동화는 안 되지만 수동은 1초. 사고는 안 나고 정상 워크플로우는 멀쩡.

## 안티패턴

- 메모리에 "`--no-verify` 쓰지 마"만 박고 hook 없이 신뢰 → 잠시는 작동, 결국 어느 세션에서 우회
- 화이트리스트를 path 기반이 아닌 명령 *내용* 기반으로 → 정규식 우회 위험
- 차단 메시지를 영어로만 → 한국어 워커에서 LLM이 메시지 의도를 약하게 해석
- `exit 1`로 차단 → Claude Code는 1=일반 에러로 해석. 차단 의도면 명시적 `exit 2`
