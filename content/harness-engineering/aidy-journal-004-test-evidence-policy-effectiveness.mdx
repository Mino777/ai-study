## 왜 자발적 준수가 일어났나

### 3층 박제의 효과

```
Layer 1: gates/test-policy*.md (문서)
          ↓ (CLAUDE.md에서 참조)
Layer 2: 세션 시작 시 자동 로드
          ↓ (architect-cli.sh build_prompt에서 인용)
Layer 3: 매 dispatch 프롬프트에 기본 포함
```

워커가 **매 라운드** 프롬프트에서 "실행 증거(숫자) 제출" 요구를 본다. 정책이 *체크리스트 문서*로만 있었다면 참조되지 않았을 것. **행동 경로(프롬프트) 위에 박았기 때문에** 내재화됐다.

### 유관성 (coupling)

정책과 Gate 검증이 **같은 단위**를 공유:

- 정책: "`NN tests, 0 failures` 형식 증거"
- Gate 1: "숫자 패턴 없으면 FAIL"
- 프롬프트: "실행 숫자 필수"

셋이 같은 문법을 쓴다. 워커 입장에서 정책 준수 = Gate 통과 = 다음 라운드 dispatch 가능. 분리돼 있으면 준수 인센티브가 약해진다.

## 이식 가능성 체크

이 패턴을 다른 하네스(moneyflow / tarosaju 등)에 이식할 때:

### 필수 요소

1. **관측 가능한 증거** — 숫자, 파일 경로, 해시처럼 기계가 검증 가능한 것
2. **정책의 프롬프트 노출** — dispatch/build_prompt에서 매번 인용
3. **CLAUDE.md 참조** — 세션 시작 context 로드 단계에서 로드
4. **Gate와 정책의 문법 일치** — 준수 = 통과 인센티브

### 주의

- **문서만 추가**는 효과 없음 — 행동 경로 위에 없으면 참조되지 않음
- **체크리스트 복붙**은 인센티브 결여 — 정책 준수가 다음 단계 블로커여야 함
- **숫자 없는 정성적 기준** ("깔끔하게", "충분히") → 검증 불가 → 무력화

## AI Agent Directive

### Trigger
- 워커가 "Tests passed" 자체 보고만 하고 숫자 증거 없을 때
- 정책 문서를 추가했지만 워커가 참조 안 하는 것 같을 때
- 새 하네스에 Gate 정책을 이식하려 할 때

### Prerequisites
- [Aidy Journal 002](/wiki/harness-engineering/aidy-journal-002-stability-observability-test-integrity) — iOS 무실행 사건
- [Numeric Execution Evidence](/wiki/evaluation/numeric-execution-evidence)
- [Test Strategy 3-Client via Aidy](/wiki/evaluation/test-strategy-3-client-via-aidy)

### Actionable Steps

1. **정책을 문서로 먼저 박제** — `gates/test-policy*.md` 4종(범용 + 플랫폼별 3)
2. **관측 가능한 증거 형식 지정** — `Test run with NN tests, 0 failures`
3. **dispatch 프롬프트에 자동 인용** — `architect-cli.sh build_prompt` 수정
4. **CLAUDE.md 참조 라인 추가** — 세션 시작 로드 경로에 노출
5. **Gate 1 체크리스트에 숫자 검증 추가** — 매칭 실패 시 FAIL
6. **첫 스프린트 후 실효성 관찰** — 자발적 준수 여부 = 정책이 행동 경로 위에 있는지의 지표

### Anti-patterns

- **체크리스트 문서만 추가** → 참조되지 않음
- **정성적 기준** ("충분히 테스트되어야 함") → 검증 불가
- **Gate와 정책의 형식 불일치** → 인센티브 분리
- **정책 이식 후 실효성 관찰 생략** → 이식 실패를 2~3 스프린트 후 발견
