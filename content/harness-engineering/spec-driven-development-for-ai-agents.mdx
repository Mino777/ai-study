## 갭 / 맥락

AI 에이전트가 코드를 생성하는 비율이 급증하면서 (GitHub 글로벌 커밋의 4%가 Claude Code 저자), 새로운 문제가 등장했다:

- LLM이 생성한 코드의 취약도: **9.8~42.1%** (벤치마크별)
- 유닛 테스트가 놓치는 결함 클래스 존재 (시스템 수준 일관성)
- Code churn: 3.1%(2020) → 5.7%(2024) — AI 코드와 상관관계 확인

**Gartner 예측**: 2027년까지 agentic AI 프로젝트의 **40%가 취소**될 것. 주원인은 평가 부족 + 데이터 품질.

이 문제의 해답으로 **Spec-Driven Development (SDD)**가 부상했다. 30+ SDD 프레임워크가 이미 존재한다 (Kiro, Spec Kit, OpenSpec, GSD, Devika, Tessl 등).

## 핵심 개념

### 코드 ≠ 진실, Spec = 진실

SDD의 제1원칙:

```
Spec (계약) ─── 검증 ───→ 코드 (구현)
     ↑                        │
     │     충돌 시 코드 수정    │
     └────────────────────────┘
```

- **Spec**은 에이전트와의 **계약** = 무엇을 만들지 + 무엇을 배제할지 + 검증 방법
- 코드와 Spec 충돌 시 → **코드를 수정** (Spec 수정 아님)
- Spec은 시스템 수준에서 작동 → 유닛 테스트가 놓치는 결함 클래스 포착

### 분 단위 피드백 루프

```
Spec 작성 (5분)
  ↓
AI 코드 생성 (5~15분)
  ↓
Spec 대비 리뷰
  ↓
Spec 수정 or 코드 수정
  ↓
재생성 → 반복
```

기존 주 단위 피드백이 **분 단위**로 단축된다. 이것이 SDD가 AI 시대에 강력한 이유.

### 3단계 Spec 구조

| 단계 | 내용 | 예시 |
|------|------|------|
| **Requirements Spec** | 무엇을 만드는가 (기능 요구사항) | "위키 엔트리 CRUD + 자동 그래프 생성" |
| **Design Spec** | 어떻게 만드는가 (아키텍처) | "MDX + Zod 검증 + content-manifest.json" |
| **Acceptance Spec** | 언제 완료인가 (검증 기준) | "빌드 통과 + 23/23 테스트 + dangling connection 0" |

### 강화 포인트

현재 부족한 영역:

1. **Acceptance Spec 명시화** — SPEC.md에 검증 가능한 acceptance criteria 섹션 추가
2. **Spec vs 코드 자동 비교** — Spec 변경 시 코드 일관성 자동 검증
3. **에이전트별 계약** — 멀티 에이전트 시 각 에이전트의 입출력 계약 명시

## 6-Layer Test Harness (에이전트 테스트)

SDD와 결합되는 에이전트 테스트 프레임워크:

| Layer | 대상 | 검증 내용 |
|-------|------|----------|
| 1. Unit | 각 하네스 컴포넌트 | tool selection, argument construction, response format |
| 2. Integration | 에이전트 간 상호작용 | 데이터 전달, 상태 동기화 |
| 3. E2E | 전체 workflow | Spec 대비 end-to-end 결과 |
| 4. Production | 실제 데이터 | 모니터링 + 이상 감지 |
| 5. Adversarial | 의도적 실패 | 엣지 케이스, 악의적 입력 |
| 6. Compliance | 규제 준수 | 보안, 개인정보, 라이선스 |

ai-study 현재 Layer 1~3 구현 (Zod 검증 + vitest + npm run build). Layer 4~6은 추후 확장 후보.

## AI Agent Directive

### Trigger
- 새 AI 에이전트 프로젝트를 시작할 때
- 기존 프로젝트에서 AI 생성 코드의 품질 문제가 반복될 때
- 멀티 에이전트 환경에서 에이전트 간 계약이 필요할 때

### Prerequisites
- `harness-engineering/harness-engineering-overview` — 하네스 기초
- `harness-engineering/compound-engineering-philosophy` — Compound 피드백 루프
- `harness-engineering/ai-output-zod-validation-pattern` — 출력 검증

### Actionable Steps
1. SPEC.md (또는 동등한 문서)에 Requirements/Design/Acceptance 3단계 Spec 작성
2. Acceptance Spec에 **검증 가능한 기준** 명시 (빌드 통과, 테스트 수, 성능 임계값)
3. AI 에이전트에게 Spec을 컨텍스트로 제공 후 코드 생성
4. 생성된 코드를 Spec 대비 리뷰 — 충돌 시 코드 수정
5. 반복 결함은 `/promote-solution`으로 코드 게이트 승격

### Anti-patterns
- Spec 없이 "그냥 만들어줘" 지시 — 검증 기준 없음 = 품질 측정 불가
- 코드에 맞춰 Spec 수정 — 인과관계 역전
- Spec을 한 번 쓰고 갱신하지 않음 — 살아있는 문서여야 함
- 유닛 테스트만으로 충분하다고 판단 — 시스템 수준 결함 놓침
