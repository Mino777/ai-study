## 왜 지금 이 주제인가

외부 프레임워크(Superpowers, Hermes)와 자체 에이전트 오케스트레이션(aidy-architect)의 패턴을 관찰하면서 한 가지 패턴이 드러났다:

> **좋은 패턴은 한 곳에만 존재하지 않는다.** 도구별, 프로젝트별로 흩어져 있다.

Superpowers는 "No-Placeholder Scan", "Anti-Rationalization Guard"라는 강력한 평가 체크를 운영한다. Hermes는 "경험→스킬 자동 생성"이라는 학습 루프를 박제한다. aidy-architect는 "process improvement collection"이라는 자동 수집 파이프라인을 실행한다.

문제는: 이들이 서로 독립적으로 존재하면, 각 프로젝트에서 각각 재발명된다. 따라서 **외부 패턴을 평가→이식→배포하는 체계적 방법론**이 필요하다.

이 엔트리는 2026-04-17 세션에서 8개 패턴을 평가하고 7개를 이식한 실적을 기반으로 방법론을 정리한 것이다.

## 핵심 개념 — 3단계 프로세스

### 1단계: 평가 (Evaluate)

외부 패턴을 수집한 후, **갭 분석과 ROI 필터링**을 수행한다.

#### 1a. 분석 엔트리 박제

외부 프레임워크는 `/ingest` 커맨드로 학습 엔트리로 변환한다. 예: [superpowers-framework-analysis](harness-engineering/superpowers-framework-analysis), [hermes-agent-analysis](agents/hermes-agent-analysis)

이 분석 엔트리는 단순 요약이 아니라, **사용 가능한 패턴을 명시적으로 열거**한다:

```
## Superpowers의 7개 스킬
1. No-Placeholder Scan — 입력 피드백 검증 (권장)
2. Anti-Rationalization Guard — 합리화 방어 (권장)
3. SDD 2-Stage Review — 설계 문서 리뷰 (권장)
...
```

#### 1b. 갭 분석 — 대조표 작성

각 패턴에 대해:
- **현재 상태**: ai-study 또는 해당 프로젝트에 이미 구현되어 있는가?
- **판정**: "이미 있음" / "갭 있음" / "불필요" / "관찰만"
- **근거**: 왜 이 판정인가?

이 단계에서 가장 중요한 발견은 "이미 있음"이다. Progressive Disclosure는 이미 Sidebar + Mermaid + Quiz로 구현되어 있었다. 이 발견이 **중복 구현을 방지**했다.

#### 1c. ROI 필터링

"갭 있음" 판정을 받은 패턴을 2개 축으로 분류:
- **구현 복잡도**: 즉시 구현 가능 vs 구조 변경 필요 vs 시기상조
- **임팩트**: 다수 프로젝트 적용 vs 특정 프로젝트만

우선순위: **즉시 구현 가능 + 높은 임팩트** > **구조 변경 필요** > **시기상조**

---

### 2단계: 이식 (Port)

ROI 필터링을 통과한 패턴부터 시작한다.

#### 2a. 프로젝트 특성 파악

같은 패턴이라도 프로젝트마다 **적용 대상이 다르다**:

| 프로젝트 | 워크플로 구조 | 검증 대상 |
|--------|-----------|---------|
| aidy-architect | WO (Workflow Object) + api-contract | WO 매니페스트 + Contract 스키마 |
| moneyflow / tarosaju | Plan (구조화된 태스크 계획) | Plan YAML + 실행 결과 |
| ai-study | 자유형 (Markdown 엔트리) | Frontmatter + MDX 파싱 |

#### 2b. 코드 변경 — validate + build 통과 필수

각 패턴의 코드 구현은 최소 단위로 분리:
- **validate**: `scripts/validate-content.mjs` 통과 (MDX, Mermaid, 링크 검증)
- **build**: `npm run build` 통과 (manifest 생성, 타입 체크)
- **test**: 기존 회귀 테스트 통과 (TS 컴파일, ESLint)

#### 2c. 원자적 커밋

패턴당 1커밋. 예:
```
[harness] Add No-Placeholder Scan to validate-content.mjs

- Detect unquoted special chars in template literals
- Flag ${placeholder} patterns in frontmatter
- Warn-only mode for legacy entries
```

---

### 3단계: 배포 (Deploy)

원본 프로젝트에서 검증한 후, 워커 프로젝트로 확산한다.

#### 3a. 원본 프로젝트 검증 (ai-study)

ai-study에서 먼저 패턴을 적용하고 2-3 세션 검증한다. 이유:
- ai-study가 가장 복잡한 위키 시스템 (MDX + graph + manifest)
- 여기서 통과하면 다른 프로젝트는 더 간단함
- 롤백이 가장 비용이 적음 (문서만 변경)

#### 3b. 워커 프로젝트 확산

1. **moneyflow / tarosaju** (같은 구조)
2. **aidy 오케스트레이터** (aidy-architect)
3. **aidy 워커** (aidy-ios / aidy-server / aidy-android)

각 프로젝트에서 프로젝트 특성에 맞게 조정:
- aidy-architect: WO 매니페스트 기반 검증 추가
- moneyflow: Plan YAML 기반 검증 추가
- aidy-ios/server/android: WO 수신 구조에 맞게 적용

#### 3c. 문서 동기화

배포가 완료되면 각 프로젝트의 CLAUDE.md / SPEC.md를 업데이트하여 **신규 개발자/에이전트가 즉시 적용**할 수 있도록 한다.

## 핵심 교훈

### 교훈 1: "이미 있음" 판정의 가치

Progressive Disclosure는 Hermes에서 "좋은 패턴"으로 소개되었지만, ai-study에는 **이미 구현되어 있었다**:
- Sidebar의 카테고리 그룹 (정보 층화 공개)
- Quiz의 선택형 문제 (낮은 인지 부하)
- Mermaid 다이어그램 (복잡성 숨기기)

이 발견이 **중복 구현을 방지**하고, 대신 "왜 우리 시스템에서 Progressive Disclosure가 이미 자연스럽게 구현되었는가?"라는 질문으로 이어졌다.

→ **"이미 있음"은 자기 시스템에 대한 이해도를 높이는 가장 중요한 판정이다.**

### 교훈 2: 프로젝트별 조정의 필수성

같은 SDD 2-Stage Review를 다르게 적용해야 한다:

| 프로젝트 | 1단계: 설계 검토 | 2단계: 코드 검토 |
|--------|---------------|----|
| aidy-architect | WO 매니페스트 + api-contract 스키마 대조 | WO 실행 로그 + 반환값 검증 |
| moneyflow / tarosaju | Plan YAML 구조 + 데이터 모델 대조 | 실행 결과 + 출력 검증 |
| ai-study | Frontmatter + 링크 스키마 대조 | MDX 파싱 + 렌더링 결과 |

**같은 개념도 구체적 구현은 프로젝트마다 다르다.**

### 교훈 3: 원본→워커 순서의 중요성

ai-study에서 먼저 패턴을 검증하는 이유:

1. **복잡도**: ai-study (MDX + graph + manifest) > moneyflow (Plan YAML) > aidy-ios (WO 수신)
2. **롤백 비용**: 문서 엔트리 > YAML 파일 > 코드
3. ** 발견 가능성**: 복잡한 시스템에서 엣지 케이스 발견 확률이 높음

즉시 moneyflow에 적용하지 않고 ai-study에서 2-3 세션 검증한 후 확산하면, 엣지 케이스로 인한 다중 프로젝트 롤백을 방지할 수 있다.

## AI Agent Directive

이 엔트리는 **패턴 이식 작업을 수행하는 에이전트**를 위해 작성되었다.

### Trigger

- 외부 프레임워크/도구 분석 이후 (`/ingest` 완료)
- 자체 프로젝트의 패턴과 갭을 발견했을 때
- "이 패턴을 다른 프로젝트에도 적용하면 좋을 것 같은데" → 이 엔트리 참조

### Prerequisites

1. 외부 패턴을 **분석 엔트리로 변환** (connections 명시)
2. 현재 프로젝트에 **이미 같은 패턴이 있는지 확인**
3. ROI 필터링: 즉시 구현 vs 구조 변경 필요 vs 시기상조 분류

### Actionable Steps

1. **대조표 작성** (갭 분석)
   ```
   | 패턴 | 출처 | 현재 상태 | 판정 | 근거 |
   ```

2. **즉시 구현 가능한 패턴부터 이식**
   - 코드 변경 (validate + build 검증)
   - 원자적 커밋
   - 테스트 통과

3. **ai-study에서 먼저 검증** (2-3 세션)
   - 문서 엔트리는 rollback 비용이 낮음
   - 복잡한 시스템에서 엣지 케이스 발견

4. **워커 프로젝트로 확산**
   - 프로젝트 특성에 맞게 조정
   - CLAUDE.md / SPEC.md 업데이트
   - 각 프로젝트별 커밋 + PR

### Anti-patterns

❌ **All-or-nothing 사고**
- "8개 패턴을 모두 동시에 이식해야 한다" → 롤백 리스크 증가
- **대신**: ROI 필터링으로 우선순위 지정

❌ **원본 검증 건너뛰기**
- "moneyflow에 바로 적용하면 되지 않나?" → 엣지 케이스로 인한 다중 프로젝트 롤백
- **대신**: ai-study에서 먼저 검증, 안정화 후 확산

❌ **프로젝트 특성 무시**
- "aidy-architect의 WO 기반 review를 moneyflow에 그대로 복사" → Plan 구조 미스매치
- **대신**: 각 프로젝트의 워크플로 구조 파악 후 조정

❌ **"이미 있음" 판정 무시**
- "좋은 패턴이니 이미 있든 없든 강화해야 한다" → 과도한 구현, 유지보수 비용 증가
- **대신**: "이미 있음"이라면 왜 자연스럽게 구현되었는가를 분석

## 자기 점검 + 실습

### 질문 1: 갭 분석

자신의 프로젝트에서 사용 중인 패턴 3개를 나열하고, 각각에 대해 "이미 있음/갭 있음/불필요" 판정을 내려보세요.

예:
- Anti-Rationalization Guard: 현재 compound Phase 3에 부분 구현됨 → "부분 갭" → 메모리 보강
- No-Placeholder Scan: validate-content에 없음 → "갭 있음" → 구현
- Progressive Disclosure: Sidebar+Quiz로 이미 구현 → "이미 있음" → 스킵

### 질문 2: ROI 필터링

발견한 "갭 있음" 패턴 중 **즉시 구현 가능한 것 1개**를 선택하고:
- 구현 복잡도 (낮음/중간/높음)
- 예상 임팩트 (다수 프로젝트 vs 특정 프로젝트)
- 3단계 프로세스 로드맵

작성해보세요.

### 질문 3: 문서화

패턴을 프로젝트에 이식한 후, **다른 팀원/에이전트가 즉시 사용할 수 있도록** CLAUDE.md / SPEC.md에 1-2문장 추가하는 연습을 해보세요.
