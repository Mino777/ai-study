## ADR-009 — self-hosted runner 전환 결정

### 대안 비교

| 옵션 | 비용 | 의존성 | 폐기 사유 |
|---|---|---|---|
| GitHub billing 정상화 | 분당 10x 가중치 유지 | 결제 상태 상시 유지 | 같은 사고 재발 가능 + 토큰 경제성 안 풀림 |
| Xcode Cloud | 25h/월 무료 | Apple Developer 가입 + Tuist 호환성 검증 | 도입 비용 큼, Tuist 호환성 불확실 |
| 워크플로 disable | 0 | — | CI 자체 포기 — 거부 |
| **self-hosted (사용자 Mac)** | **0** (전기료만) | 사용자 Mac 가용성 | **선택** |

### 환경 구성

- Runner: `jominhoui-mba-ios` / labels: `self-hosted, macOS, ARM64, aidy-ios`
- 사용자 Mac: macOS 26.3.1 + Xcode 26.3 + tuist 4.180.0 (모두 이미 설치됨)
- 실행: launchd user agent (`~/actions-runner/svc.sh install + start`) — 상시 실행
- 적용 워크플로: `test.yml` + `ai-review.yml` **동시 전환** (정정 2에서 배운 것)

### workflow 변경 요지

```yaml
# Before
runs-on: macos-14   # test.yml
runs-on: macos-latest  # ai-review.yml

# After
runs-on: [self-hosted, macOS, ARM64, aidy-ios]
```

`setup-xcode@v1` step 제거(시스템 Xcode 26.3 사용), `tuist` 설치는 `command -v tuist || brew install tuist` 가드로 idempotent 유지. `path filter` 도입(README/docs 변경 시 skip)은 self-hosted라 경제성 효과는 작지만 불필요한 run 억제 가치 존치.

## 박제 교훈 (ADR-009 인용)

### 1. "한도 초과 아님" ≠ "결제 정상"

분 카운트와 결제 상태는 별개 변수다. `gh api /users/<owner>/settings/billing/actions` 로 한도는 확인 가능하지만 spending limit/카드 상태는 또 별도. **둘 다 확인**해야 GitHub-hosted 기동 여부 판정 가능.

### 2. PR 1개 = 워크플로 1개 가정 금지

같은 push 트리거에 다른 워크플로가 묶여 있을 수 있다. `test.yml` 만 고쳤는데 `ai-review.yml` 이 매 push마다 fail → 알림 폭탄. `ci-status.sh --watch` 가 **모든 워크플로** fail을 잡아냈기에 발견 가능. 단일 워크플로만 보는 도구였다면 놓쳤다.

### 3. 인프라 변경 후 알림 폭탄은 진단 신호

무시하면 같은 사고가 반복된다. "뭔가 잘못된 상태"를 알려주는 신호로 받아들이기.

### 4. self-hosted = 토큰 경제성 + billing 안전 동시 해결

macOS CI에서 특히 효과 크다(분 가중치 10x). 단일 머신 의존이라는 위험은 launchd 자동 부팅 + 정기 점검으로 완화 가능. Tokenomics 4 조건([Token Levers Catalog](/wiki/tokenomics/claude-code-token-levers-catalog)) 에 비추면 공식문서 있음 + 실측 있음 + 부작용 관찰 가능 + 롤백 용이(`runs-on` 한 줄) 모두 충족.

## AI Agent Directive

### Trigger
- CI 워크플로가 `exit 134` / `Library not loaded` 로 즉시 abort 될 때
- `The job was not started because recent account payments have failed` 관측
- 인프라 변경 후 GitHub 알림이 폭증할 때
- macOS runner 분 가중치(10x) 절감 기회를 찾을 때
- `ci-status.sh` 가 연속 실패 패턴을 보고할 때

### Prerequisites
- [Journal 003](/wiki/harness-engineering/aidy-journal-003-parallel-dispatch-token-economics) — 토큰 경제성 판단 기준
- [tmux flush 자동화](/wiki/harness-engineering/tmux-flush-automation-pattern) — `ci-status.sh` 와 함께 도입된 도구군
- [AI Ops Environment Diagnosis Checklist](/wiki/harness-engineering/ai-ops-environment-diagnosis-checklist) — 진단 정정 체크리스트 연결

### Actionable Steps

1. **CI 실패 시 `gh run view <id> --log-failed` 부터** — exit code + dyld/library 메시지 확인
2. **OS 버전 ↔ 도구 체인 ABI 매핑 검증** — `brew info <tool>` 로 빌드 타겟 확인. tuist 처럼 SDK 강제 의존 패턴은 흔함
3. **결제 상태 별도 확인** — `gh api .../settings/billing/actions` 는 분만 보여줌. 결제/spending limit은 GitHub UI 에서 별도
4. **같은 push 트리거에 묶인 모든 워크플로 스캔** — `gh workflow list` + `on:` 블록 검토. 단일 파일만 고치는 함정 회피
5. **self-hosted 전환 판단 기준** — macOS CI + 10x 가중치 + 사용자 Mac 가용 → 도입. 단일 머신 의존은 launchd 자동 부팅 + 정기 점검으로 완화
6. **`ci-status.sh --watch` 를 상시 모니터링** — 연속 실패 패턴을 자동 감지하는 루프를 만들면 수동 `gh run list` 반복 제거 가능

### Anti-patterns

- **1차 진단으로 끝내기** — "exit 134는 OS 버전 문제야" → 결제 차단은 별개. 진단은 정정 가능
- **`test.yml` 단일 파일만 고치기** — `on: push` 트리거에 묶인 다른 워크플로 간과
- **self-hosted runner를 보안 이유로 기각** — launchd user agent + short-lived token + private repo 조합이면 실제 위험 낮음
- **알림 폭탄 무시** — 인프라 변경 후 이상 신호는 *진단의 연장선*
- **billing 정상화로 일시 해결** — 같은 사고 재발 + 토큰 경제성 미해결
- **`ci-status.sh` 없이 `gh run list` 반복** — 수동 관찰은 누락 가능
