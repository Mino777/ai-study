## 

정적이고 예측 가능한 자동화의 시대는 저물고 있습니다. 기존 AI 에이전트 워크플로우는 대부분 사전에 정의된 계획, 즉 정적 DAG(방향성 비순환 그래프)에 의존했습니다. 이는 마치 잘 짜인 각본처럼 에이전트가 정해진 순서대로 도구를 실행하게 하지만, 실제 소프트웨어 개발의 혼돈 앞에서는 무력합니다. 예기치 않은 API 변경, 테스트 실패, 새로운 요구사항 등 예측 불가능한 변수가 발생하면 전체 워크플로우는 쉽게 좌초됩니다. 이 지점에서 Claude Code의 '동적 워크플로우'는 패러다임의 전환을 제시합니다.

동적 워크플로우의 핵심은 **계획 자체를 실행 시점에 수정 가능한 대상으로 취급**하는 것입니다. 에이전트는 더 이상 수동적인 실행자가 아니라, 변화하는 상황에 맞춰 자율적으로 계획을 수정하고, 작업을 분산하며, 심지어 새로운 하위 에이전트를 생성하여 위임하는 능동적인 오케스트레이터가 됩니다. 이 글은 Anthropic이 공개한 동적 워크플로우 기능을 중심으로, iOS 시니어 개발자가 AI 엔지니어링의 관점에서 대규모 작업을 분산하고 오케스트레이션하는 구체적인 전략과 그에 따른 트레이드오프를 심도 있게 분석합니다.

## 정적 워크플로우를 넘어서: 동적 오케스트레이션의 필요성

전통적인 AI 에이전트 시스템은 '계획 후 실행(plan-and-execute)' 모델을 따릅니다. 이는 복잡한 목표를 더 작은 단계로 분해하고 순서대로 처리하는 방식입니다. 하지만 이 접근법은 다음과 같은 근본적인 한계를 가집니다.

-   **경직성(Rigidity):** 초기 계획은 실행 중 얻게 되는 새로운 정보를 반영하지 못합니다. 예를 들어, 코드 리팩토링 중 예상치 못한 의존성을 발견해도 계획은 수정되지 않아 비효율적인 작업을 계속하거나 실패합니다.
-   **순차적 병목(Sequential Bottleneck):** 대부분의 작업이 순차적으로 진행되어 병렬화의 이점을 누리기 어렵습니다. 이는 대규모 작업(예: 수백 개 파일 마이그레이션) 시 엄청난 시간 소요로 이어집니다.
-   **컨텍스트 윈도우 한계:** 모든 중간 결과와 계획을 단일 모델의 컨텍스트 윈도우에 유지해야 하므로, 작업이 길어질수록 컨텍스트가 유실되거나 비용이 기하급수적으로 증가합니다.

Claude Code의 동적 워크플로우는 이 문제를 정면으로 해결합니다. 핵심적인 아키텍처 변화는 **오케스트레이션 로직을 LLM의 컨텍스트 윈도우에서 외부 스크립트(JavaScript)로 분리**한 것입니다. 이제 Claude는 작업 지시를 받으면, 그 작업을 수행할 오케스트레이션 스크립트를 동적으로 생성합니다. 이 스크립트가 별도의 런타임에서 실행되며 하위 에이전트들을 조율하고, LLM은 최종 결과를 보고받는 역할만 수행합니다. 이로써 컨텍스트 윈도우의 제약에서 벗어나 수백 개의 에이전트를 병렬로 실행하는 것이 가능해졌습니다.

## 전략 1: 실행 계획을 코드로 추상화하고 동적으로 수정하기

동적 워크플로우의 첫 번째 핵심 전략은 '계획'을 데이터가 아닌 '코드'로 다루는 것입니다. Swift 개발자에게 익숙한 개념으로 비유하자면, `Codable`로 직렬화된 JSON 설정 파일을 넘어, `NSOperation`과 `NSOperationQueue`를 사용하여 동적으로 의존성을 추가하고 작업을 취소하는 것과 유사합니다.

생성된 JavaScript 오케스트레이션 스크립트는 단순한 명령어 목록이 아닙니다. 이 스크립트는 분기, 루프, 에러 핸들링, 병렬 실행 로직을 포함하는 완전한 프로그램입니다. 오케스트레이터(런타임)는 이 코드를 실행하며, 각 하위 에이전트의 결과를 변수로 받아 다음 행동을 결정합니다.

예를 들어, "앱의 모든 모듈에 대해 SwiftLint를 실행하고, 경고가 10개 이상인 모듈만 리팩토링 에이전트에게 전달하라"는 요청이 있다면, 동적 워크플로우는 다음과 같이 작동합니다.

1.  **계획 생성:** Claude가 JavaScript 코드를 생성합니다. 이 코드는 먼저 모든 모듈 디렉토리를 찾아 배열로 만듭니다.
2.  **병렬 실행 (Fan-out):** `Promise.all`과 같은 구문을 사용하여 각 모듈에 대해 `swiftlint` 명령을 실행하는 하위 에이전트를 병렬로 생성합니다.
3.  **결과 수집 및 분기:** 모든 Linter 에이전트가 완료되면, 오케스트레이션 스크립트는 각 결과를 순회하며 경고 개수를 확인합니다.
4.  **동적 태스크 생성:** 경고가 10개를 초과하는 모듈에 대해서만 새로운 `RefactoringAgent`를 동적으로 생성하여 작업을 위임합니다.

이러한 접근 방식을 Swift 코드로 개념화하면 다음과 같은 구조를 상상할 수 있습니다. 이는 실행 가능한 코드는 아니지만, iOS 개발자가 아키텍처를 이해하는 데 도움을 줄 것입니다.

```swift
// 개념적 Swift 의사코드 (Conceptual Swift Pseudocode)
// 실제 Claude Code는 JavaScript 런타임에서 실행됩니다.

// 1. 작업을 정의하는 구조체 (Task Definition)
protocol AgentTask {
    associatedtype ResultType
    func execute() async throws -> ResultType
}

// 2. 오케스트레이션 로직을 담은 워크플로우 (Workflow with Orchestration Logic)
class LintAndRefactorWorkflow {
    let modules: [String]
    private var lintingResults: [String: Int] = [:] // [ModuleName: WarningCount]

    init(modules: [String]) {
        self.modules = modules
    }

    // 동적 워크플로우의 핵심: 실행 흐름이 코드 자체에 있음
    func run() async {
        // Phase 1: Fan-out - 병렬로 Linting 실행
        await withTaskGroup(of: (String, Int).self) { group in
            for module in modules {
                group.addTask {
                    let lintAgent = LintingAgent(moduleName: module)
                    let warningCount = await lintAgent.execute()
                    return (module, warningCount)
                }
            }

            for await (module, count) in group {
                lintingResults[module] = count
            }
        }

        // Phase 2: Aggregate & Branch - 결과 취합 후 동적 작업 생성
        for (module, count) in lintingResults where count > 10 {
            print("High warning count in \(module) (\(count)). Starting refactoring agent...")
            let refactorAgent = RefactoringAgent(moduleName: module)
            await refactorAgent.execute() // 실제로는 별도 큐에 등록
        }
    }
}
```
이 Swift 예시는 `TaskGroup`을 사용하여 병렬 실행(fan-out)을 흉내 내고, 그 결과를 기반으로 후속 작업을 동적으로 결정하는 과정을 보여줍니다. 실제 Claude Code의 JavaScript 런타임은 이와 유사한 원리로 작동합니다.

## 전략 2: 에이전트 전문화 및 동적 라우팅

대규모 작업은 단일 유형의 에이전트로는 해결할 수 없습니다. 코드 분석, 버그 수정, 테스트 작성, 문서 업데이트 등 각기 다른 전문성이 필요합니다. 동적 워크플로우는 이러한 **전문 에이전트(Specialized Agents) 간의 작업을 동적으로 라우팅**하는 오케스트레이터 역할을 수행합니다.

| 라우팅 전략 (Routing Strategy) | 장점 (Pros) | 단점 (Cons) | `aidy-ios` 적용 시나리오 (aidy-ios Use Case) |
| :--- | :--- | :--- | :--- |
| **중앙 집중형 오케스트레이터 (Centralized Orchestrator)** | 일관된 결정, 전체 워크플로우 상태 추적 용이, 디버깅 단순화. | 오케스트레이터가 단일 실패 지점(SPOF) 및 병목이 될 수 있음. | `aidy-architect`가 모놀리스 분석 후, `ModuleExtractorAgent`, `DependencyFixerAgent`, `UITestWriterAgent`에게 명시적으로 작업을 순차/병렬 할당. |
| **분산형 코레오그래피 (Decentralized Choreography)** | 확장성 높음, SPOF 없음, 에이전트 간의 느슨한 결합(Loose Coupling). | 시스템 전체 상태 파악 어려움, 예측 불가능한 연쇄 반응 발생 가능, 디버깅 복잡. | `CodeCommitAgent`가 코드를 푸시하면, 이를 트리거로 `BuildAgent`와 `TestAgent`가 독립적으로 반응하여 CI/CD 파이프라인을 자율적으로 수행. |
| **하이브리드 (Hybrid)** | 중앙 제어의 예측 가능성과 분산형의 확장성을 결합. | 아키텍처 복잡성 증가, 두 모델 간의 상호작용 지점 설계가 어려움. | `aidy-architect`가 초기 리팩토링 계획을 수립하고 작업을 할당하지만(중앙 집중), 테스트 실패 시 `TestFixerAgent`와 `CodeReviewAgent`가 서로 직접 통신하며 문제를 해결(분산형). |

Anthropic의 동적 워크플로우는 현재 **중앙 집중형 오케스트레이터** 모델에 가깝습니다. Claude가 생성한 단일 JavaScript 스크립트가 전체 워크플로우를 통제하고 모든 하위 에이전트의 생명주기를 관리합니다. 이는 예측 가능성과 제어 용이성 측면에서 장점이 있지만, 스크립트 자체의 복잡성이 증가할수록 유지보수가 어려워질 수 있다는 트레이드오프가 존재합니다.

## `aidy-ios` 프로젝트 적용: 모놀리스를 26개 모듈로 분해하기

`aidy-journal-018`에서 언급된 "모놀리스 → 26모듈 Microfeatures" 작업을 동적 워크플로우로 수행하는 시나리오를 가정해 봅시다.

1.  **초기 프롬프트:** "aidy-ios 프로젝트를 분석해서 26개의 독립적인 피처 모듈로 리팩토링해줘. 각 모듈은 자체적으로 빌드 및 테스트가 가능해야 해."
2.  **워크플로우 생성:** Claude는 이 요청을 받고, 다음과 같은 로직을 포함한 `refactor-monolith.js` 스크립트를 생성합니다.
    *   **Phase 1: 분석 (Analysis):** `CodeAnalyzerAgent`를 실행하여 프로젝트 전체의 의존성 그래프를 생성하고, 모듈화 후보군을 식별한다.
    *   **Phase 2: 병렬 추출 (Parallel Extraction):** 의존성이 없는 Leaf 모듈들부터 시작하여, 각 모듈에 대해 `ModuleExtractorAgent`를 병렬로 실행한다. (예: 16개 동시 실행)
    *   **Phase 3: 동적 의존성 해결 (Dynamic Dependency Resolution):** 한 모듈의 추출이 완료되면, 해당 모듈에 의존했던 다른 모듈들의 소스 코드를 업데이트하는 `DependencyFixerAgent`를 동적으로 트리거한다.
    *   **Phase 4: 병렬 검증 (Parallel Verification):** 각 모듈이 생성될 때마다 `TestWriterAgent`와 `BuildVerifierAgent`를 병렬로 실행하여 모듈의 무결성을 검증한다.
    *   **Phase 5: 종합 및 보고 (Aggregation & Reporting):** 모든 작업이 끝나면, 성공/실패한 모듈 목록, 주요 변경 사항, 그리고 후속 작업 제안을 담은 최종 보고서를 생성하여 사용자 세션에 반환한다.
3.  **실행 및 적응:** 이 과정에서 `DependencyFixerAgent`가 예상치 못한 컴파일 에러를 발견하면, 워크플로우는 실패로 중단되는 대신 **재시도(Retry) 루프를 돌거나, `BugFixerAgent`라는 새로운 유형의 에이전트를 동적으로 호출하여 문제를 해결**하도록 설계될 수 있습니다. 이것이 바로 정적 워크플로우가 제공할 수 없는 동적 워크플로우의 핵심적인 회복탄력성(Resilience)입니다.

이러한 접근법은 단순히 작업을 자동화하는 것을 넘어, 복잡하고 긴 엔지니어링 작업을 거의 사람의 개입 없이 자율적으로 수행하는 시스템으로의 진화를 의미합니다.
