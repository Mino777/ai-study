## s25 — Swap 조기 종료 → 재개 + iOS 자발 통합

### Swap 80%+ 금지 규칙 수립

s25 시작 시 Swap 90%로 autoceo 시작 → 서버 R2에서 3.5시간 stall → 조기 종료.

유저 요청으로 재개 시 물리 메모리 2GB 여유 확인 후 2-way 병렬 dispatch → 완주 성공.

**이 사고 직후 메모리 기록**: "Swap 80%+ 상태에서 autoceo 하면 안 됨."

```bash
# s25 이후 autoceo preflight 체크
sysctl vm.swapusage
# used 80%+ → 앱 종료 후 재확인 (시작 금지)
```

### iOS 워커 자발 통합

v2.0(Health Score) WO만 보냈는데 iOS 워커가 v2.0과 v2.1(Daily Digest)를 한 번에 구현:

```
WO-078 (iOS v2.0 Health Score) dispatch
→ iOS 워커가 v2.1 Daily Digest WO까지 자발적으로 구현
→ 별도 dispatch 불필요, 라운드 1개 절감
```

패턴: "다음 WO를 미리 제공하면 워커가 자발적 통합 가능."

## s26 — 3가지 최적화 동시 적용 + 인프라 확충

### 폴링 간격 5분→20분

```
Before: 5분 × 워커 평균 10분 = 라운드당 폴링 6~10회
        tmux capture-pane × 6~10 = 컨텍스트의 ~60% 폴링 소비

After: 20분 간격 (워커 완료 시간 × 1.5~2배)
       대부분 1회 폴링에 결과 확인
       컨텍스트 절감 → 10라운드 완주 가능
```

### iOS 게이트 변경

```
Before: xcodebuild test (시뮬레이터 부팅 2분+, 전체 테스트 실행 시간)
After:  tuist build 통과 = 커밋 OK
        xcodebuild test는 Gate-2에서만 (또는 선택적)
```

### 스펙 4개/라운드 배치

```
Before: R1에 v2.2 스펙, R3에 v2.3 스펙 (2개 나누어 처리)
After:  R1에 v2.2+v2.3+v2.4+v2.5 한 번에 처리
        스펙 전용 라운드를 1개로 압축 → 라운드 수 절감
```

### /compact 2라운드 주기

매 2라운드마다 `/compact` 실행. 컨텍스트 누적으로 인한 폴링 토큰 낭비 사전 차단.

### 인프라 확충 (s26)


## s27 — 1-way 순차 안정 + 플래키 테스트 발견

### 구현 피처 (v2.7~v2.8)

| v2.8 | Quick Notes | 채팅 없이 직접 메모리 생성 (단건 + batch 10개) |

### 6인스턴스 환경에서 1-way 순차 안정

6개 Claude 인스턴스(architect + server + ios + android + ai-study + α) 동시 실행 환경에서도 1-way 순차 dispatch로 워커 크래시 0건.

보수적 전략의 효과: 병렬도를 줄이는 대신 안정성 확보.

### 플래키 테스트 감지

서버 첫 verify에서 FAIL → 재실행 PASS. 근본 원인 미파악 채로 다음 단계 진행.

retro: "어떤 테스트가 flaky한지 특정하지 않았다. 다음 세션에서 `./gradlew test --rerun-tasks` 3회 반복으로 재현 시도 필요."
