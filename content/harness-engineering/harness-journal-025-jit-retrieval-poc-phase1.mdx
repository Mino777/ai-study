## Before — 무엇이 없었나

[Context Scaling 3-레이어 아키텍처](/wiki/context-engineering/context-scaling-3-layer-architecture) 가 방향성을 제시했다. Layer 1(Prompt Caching · 즉시 적용 가능) · Layer 2(Pruning · 단계적) · Layer 3(JIT Retrieval · POC 필수). Layer 3는 **롤백 비용 높음** 으로 분류돼 4 조건 게이트의 즉시 적용에서 제외 → "단일 프로덕트 POC 후 단계적 도입" 권고.

Phase 1 목표: **인프라가 작동하는가**. 검색 품질은 그 다음 변수.

### 설계 결정 (4 조건 게이트 통과 기준)

| 항목 | 결정 | 근거 |
|------|------|------|
| 벡터 DB | **JSON 파일 + brute-force 코사인** | [vector-search-basics §6](/wiki/rag/vector-search-basics): 1000 미만 문서는 ANN 불필요. 923 청크 brute force = 1~2ms |
| 임베딩 모델 | **`@xenova/transformers` Xenova/all-MiniLM-L6-v2** | 100% 로컬, 외부 API 0, ~80MB 일회성 다운로드, 384차원 |
| 청킹 | **H2(`## `) 단위** | 엔트리 §10 안티패턴 "단일 .mdx 임베딩" 회피. 평균 8.7 청크/file |
| 메타 | slug · title · category · tags · date · confidence · h2_title | 검색 후 필터링 + 디버깅 |
| 인덱스 위치 | `public/embeddings.json` (gitignore) | search-index.json 패턴 따름 |
| 청크 크기 | min 200자 / max 2000자 | 너무 작으면 합침, 너무 크면 분할 |

### 인프라 메트릭 (실측)

```

```

## 발견 — 3가지 변수가 다음 사이클을 정의한다

### 변수 1: 모델 (가장 큰 변수)

`all-MiniLM-L6-v2` 는 영어 위주 학습. 한국어 처리는 가능하지만 의미 분리 정밀도가 영어 대비 현저히 낮음.

Phase 2 후보:
- `Xenova/multilingual-e5-small` (94 언어, 384차원)
- `Xenova/paraphrase-multilingual-MiniLM-L12-v2`
- API 기반: Voyage `voyage-multilingual-2`, OpenAI `text-embedding-3-small` (둘 다 한국어 강함)

### 변수 2: 인덱싱 범위 누락 — `docs/solutions/`

Layer 3의 핵심 사용처는 "에러 메시지 → 과거 해결법". 그런 텍스트는 `docs/solutions/<category>/*.md` 에 집중. POC v0는 `content/` 만 인덱싱 — 가장 high-value 자산을 통째로 누락.

Phase 2 첫 작업: `embed-content.mjs` 의 인덱싱 대상에 `docs/solutions/` + `docs/retros/` 추가.

### 변수 3: 라우터 부재

현재는 *모든* 쿼리에 검색 실행. 엔트리 §5 안티패턴 "모든 쿼리에 RAG → Layer 3 가 선형 비용으로 회귀" 와 일치. 라우터(에러 키워드/명시 트리거)가 Phase 2 필수.

## 배운 것 / 다음 후보

### POC Phase 1 의 진짜 메시지

1. **인프라 자체는 단순하다** — JSON + brute force + 로컬 임베딩으로 50줄 + 50줄. ANN/Chroma/Pinecone 도입은 명확한 트리거(1000+ 문서)까지 미룸
2. **모델 선택이 압도적 변수** — 영어 쿼리는 완벽 적중, 한국어는 무관한 결과. 인프라 재설계가 아니라 모델 한 줄 교체로 해결 가능
3. **첫 인덱싱 범위 결정이 운명** — `content/` 만 인덱싱했더니 가장 가치 있는 `docs/solutions/` 누락. Phase 2 0순위
4. **POC = 가설 검증 데이터** — "한국어 검색이 잘 될까?" 의 답을 7 쿼리로 실측. "잘 안 됨, 모델 변수" 라는 명확한 결론

### 다음 사이클 후보 (Phase 2)

| # | 작업 | 핵심 질문 |
|---|------|----------|
| A | 다국어 모델 비교 | multilingual-e5-small vs MiniLM-L6 한국어 적중률 차이 (같은 7 쿼리 재측정) |
| B | `docs/solutions/` + `docs/retros/` 인덱싱 추가 | 가장 high-value 자산 포함 후 적중률 변화 |
| C | 쿼리 라우터 v0 (규칙 기반) | "에러 키워드/스택 트레이스" 포함 시만 검색 실행 |
| D | 섀도우 모드 + 측정 | 기존 전체 로드 vs JIT 주입 출력 차이 (LLM-as-Judge) |
| E | 인덱싱 자동화 (pre-commit 또는 CI) | 새 .mdx 커밋 시 재임베딩 자동 |

A·B 가 적중률 직접 영향 — Phase 2 의 가장 큰 ROI.

## AI Agent Directive

### Trigger
- Layer 3 (JIT Retrieval) POC를 시작하려는 모든 프로젝트
- 한국어 콘텐츠가 많은데 영어 위주 임베딩 모델로 검색 품질 부족 경험

### Prerequisites
- [Context Scaling 3-레이어 아키텍처](/wiki/context-engineering/context-scaling-3-layer-architecture)
- [vector-search-basics](/wiki/rag/vector-search-basics)

### Actionable Steps
1. **인프라 최소 시작** — JSON + brute force + 로컬 임베딩 (Chroma/Pinecone 미루기)
2. **첫 인덱싱 범위 신중 결정** — `content/` 만 보지 말고 `docs/solutions/` `docs/retros/` 등 *재발 패턴이 누적된 곳* 우선
3. **다양한 쿼리 7~10개로 실측** — 한국어/영어/추상/구체 골고루. POC = 가설 검증 데이터 수집
4. **인프라 작동 vs 검색 품질 분리 평가** — 인프라 OK인데 품질 부족이면 모델 변수, 둘 다 부족이면 설계 재검토
5. **모델 후보 미리 정리** — 영어 위주 시작 후 다국어 비교는 Phase 2 의 첫 작업

### Anti-patterns
- **Chroma/Pinecone부터 도입** — 1000 미만 문서에서는 ANN 불필요. 인프라 복잡도만 증가
- **`content/` 만 인덱싱하고 끝** — 가장 high-value인 `docs/solutions/` 누락
- **쿼리 1~2개로 적중률 판단** — 모델 약점이 특정 패턴에서만 드러남. 최소 7개
- **POC 결과를 "성공/실패" 로 이분법** — Phase 1은 가설 검증. "어느 변수가 다음 사이클을 정의하는가" 의 답을 얻는 것이 목적
