## 갭 (Before)

[Journal 003](/wiki/harness-engineering/harness-journal-003-squash-merge-trap-pattern)에서 두 라이브 사고로 입증된 패턴:

> AI 자동 squash merge 환경에서 *로컬 git 상태와 origin이 갈라지는 것*이 시스템 기본값이고, 그 갭을 *사람이 매번 fetch + reset으로 메워야* 하는데, 사람의 기억력 의존은 100% 실패. 사고 발생 확률이 0이 아니라 *시간 함수*.

회피 메커니즘 4가지 후보 비교에서 **후보 C (`/wt-branch` 슬래시 커맨드)** 가 가장 강력하다는 결론. 이유는 *행동 자체를 안전하게* 만들어 *기억력 의존을 0으로* 만들기 때문.

이 사이클(004)는 후보 C를 ai-study에서 먼저 정의한다. 양쪽 프로젝트(mino-moneyflow, mino-tarosaju) 이식은 *다른 세션이 PR #93 정리 후*. 동시 작업하면 또 사고.

## 만든 환경 (After) — `/wt-branch` 슬래시 커맨드

### 위치

```
/Users/jominho/Develop/ai-study/.claude/commands/wt-branch.md
```

ai-study의 `.claude/commands/`에 신설. 같은 형식으로 `compound`, `autoceo`, `ingest`가 이미 존재. `wt-branch`가 네 번째 슬래시 커맨드.

### CLAUDE.md skill routing 등록

```markdown
- 외부 URL(유튜브/블로그/논문/스레드) 정리, 학습 엔트리로 가공 → invoke ingest
- 새 작업 시작, 안전한 브랜치 분기 (squash merge 함정 회피) → invoke wt-branch  ← 신규
```

### 동작 요약

```
/wt-branch <branch-name>

→ Phase 0: 사전 점검 (인자, 브랜치 이름 valid, 중복 없음)
→ Phase 1: git rev-parse --show-toplevel으로 project-root 자동 감지
→ Phase 2: git -C <root> fetch origin
→ Phase 3: git -C <root> worktree add -b <branch-name> /tmp/<project>-<branch> origin/main
→ Phase 4: 사용자에게 작업 디렉터리 + cleanup 명령 안내
→ Phase 5: 이후 작업은 worktree 안에서 진행, 머지 후 worktree remove
```

### 핵심 설계 결정

| 결정 | 이유 |
|---|---|
| `git rev-parse --show-toplevel`로 project-root **자동 감지** | 한 파일이 어디에 복사되든 *변경 없이 작동* — 이식성 100% |
| `/tmp/<project>-<branch>` 위치 | 원본 디렉터리와 *완전 분리* — 충돌 0 |
| `origin/main`을 **항상 fetch 후** 분기 기준점으로 | 회피 메커니즘 자체의 무력화 방지 |
| 5가지 **금지 사항** 명시 | AI가 자율로 임의 변경하지 않도록 명문화 |
| **Phase 5 작업 지속** 안내 | 이후 모든 git 명령이 worktree를 가리키도록 |

## 배운 것 / 다음 후보

### 핵심 통찰 1 — 자율 진행의 경계는 *동시 작업 충돌 위험*까지 포함

자율 모드라도 *다른 세션이 같은 영역에서 작업 중*이면 멈춰야 한다. 이건 *코드 결정*만의 문제가 아니라 *물리적 충돌* 회피. 안전한 자율은 *물리적 경계를 인식하는 것*도 포함.

### 핵심 통찰 2 — 박제 ≠ 단순 PR

*완성 패치를 박제*하는 것은 *PR을 만드는 것*과 다르다. PR은 코드만 옮기지만 박제는 *코드 + 맥락(왜) + 검증 방법(어떻게 확인하는가)*을 함께 가진다. 다른 세션 또는 미래의 본인이 적용할 때 *왜 이렇게 하는지*를 알 수 있어야 진짜 자산.

### 핵심 통찰 3 — 이식성은 *변경 0*이 목표

증명된 자산이 다른 프로젝트로 옮겨질 때 *한 줄도 변경 없이* 작동해야 진짜 복리. project-root 자동 감지(`git rev-parse --show-toplevel`)가 그 핵심. 하드코딩은 매 이식마다 사람의 기억력에 의존.

## For AI Agents

**Trigger**: 새 작업을 시작해야 할 때 *항상* (코드/문서 변경 무관). 특히 squash merge가 있는 환경.

**Prerequisites**:
- [Harness Journal 003 — squash merge 함정](/wiki/harness-engineering/harness-journal-003-squash-merge-trap-pattern)

**Actionable Steps (슬래시 커맨드 호출)**:
```bash
/wt-branch <branch-name>
```

이 커맨드가 내부적으로 수행:
1. `git rev-parse --show-toplevel` — 프로젝트 루트 자동 감지 (이식성)
2. `git fetch origin` — 최신 origin/main 확보
3. `git worktree add -b <branch-name> /tmp/<project>-<branch> origin/main`
4. 작업 디렉터리 안내 + cleanup 명령
5. 이후 모든 git 명령은 worktree 안에서 (`git -C /tmp/...` 또는 `cd`)
6. 작업 완료 + 머지 후: `git -C <project-root> worktree remove /tmp/<project>-<branch>`

**수동 호출이 필요한 경우** (슬래시 커맨드 사용 불가 환경):
```bash
git -C <project-root> fetch origin
git -C <project-root> worktree add -b <branch-name> /tmp/<project>-<branch> origin/main
cd /tmp/<project>-<branch>
# 작업
git push -u origin <branch-name>
# 머지 후
cd -
git -C <project-root> worktree remove /tmp/<project>-<branch>
```

**Anti-patterns**:
- ❌ 원본 working tree에서 `git checkout -b` (stale changes + 옛 SHA 상속)
- ❌ 원본 working tree의 modified 파일을 worktree로 이동 (의도 불명)
- ❌ worktree 작업 중 원본 working tree도 동시 수정
- ❌ 작업 끝나고 worktree 정리 안 하기 (디스크 누적)
- ❌ `<project-root>`를 추측해서 하드코딩
