## 배경

Journal 012~017에서 Layer 1(JSON 파싱) + Layer 2(runtime shape validation)를 10/10 에이전트에 적용 완료. Layer 2가 실패하면 fallback을 반환하는데, 이는 *안전*하지만 *정보 손실*이다. LLM이 거의 맞는 JSON을 반환했지만 필드 하나가 빠졌을 때, fallback 대신 *무엇이 잘못됐는지 알려주고 다시 시도*하는 게 더 나은 전략.

## 설계

### retryWithValidation (순수 함수)

```typescript
async function retryWithValidation<T>(
  caller: (prompt: string) => Promise<string>,
  prompt: string,
  fallback: T,
  validator: Validator<T>,
  maxRetries: number = 1,
): Promise<T>
```

`caller`를 주입받아 테스트 가능. `callAI` 같은 외부 의존성 없이 retry 로직만 담당.

### callAIAndValidate (편의 래퍼)

```typescript
async function callAIAndValidate<T>(
  prompt: string,
  keys: AIKeys,
  options: CallAIAndValidateOptions,
  fallback: T,
  validator: Validator<T>,
): Promise<T>
```

`callAI`를 `caller`로 바인딩해서 `retryWithValidation`에 전달. 에이전트 코드에서는 이것만 호출.

### Instruction Augmentation (retry 시 프롬프트 추가)

```
attempt 0: 원본 prompt 그대로
attempt 1+: prompt + "\n\n⚠️ 이전 응답의 JSON 형식 오류:\n- issue1\n- issue2\n위 문제를 수정하여 올바른 JSON을 반환하세요."
```

validator의 `issues` 배열이 그대로 LLM에 전달되므로 "어떤 필드가 어떻게 잘못됐는지" 구체적.

## 테스트 — caller 주입으로 mock 문제 해결

같은 모듈 내 `callAI`를 `vi.mock()`으로 가로채려 했으나 ESM 내부 참조 문제로 실패. `retryWithValidation`을 순수 함수로 분리한 뒤 `vi.fn()`을 `caller`로 주입:

```typescript
const caller = vi.fn()
  .mockResolvedValueOnce('{"name":"bad","value":-1}')  // 1차: 실패
  .mockResolvedValueOnce('{"name":"fixed","value":10}'); // 2차: 성공

const result = await retryWithValidation(caller, 'prompt', FALLBACK, validator, 1);
expect(result).toEqual({ name: 'fixed', value: 10 });
expect(caller).toHaveBeenCalledTimes(2);
// 2번째 호출에 augmented prompt 확인
expect(caller.mock.calls[1][0]).toContain('value: non-negative number required');
```

vitest 8 케이스: 첫 시도 성공, retry 성공, 전체 실패 fallback, maxRetries 0, 기본 1, JSON 파싱 실패 복구, 다중 issues augmentation, 코드블록 JSON.

## 5 Layer 완료 현황

| Layer | 내용 | 상태 |
|---|---|---|
| 1 | JSON 파싱 (3단계 regex fallback) | ✅ 기존 |
| 2 | Runtime shape validation (type guard) | ✅ Journal 012~017 |
| **3** | **Retry (maxRetries)** | **✅ Journal 018** |
| **4** | **Instruction augmentation (issues → prompt)** | **✅ Journal 018** |
| 5 | Schema evolution (버전 관리) | 미착수 |

Layer 1~4 완료 = *실용적 완성*. Layer 5(schema evolution)는 에이전트 응답 구조가 실제로 변경될 때 필요.

## 핵심 메시지

> *fallback은 안전하지만 정보를 버린다. retry + augmentation은 "거의 맞는 답"을 "완전히 맞는 답"으로 고치는 마지막 기회를 준다. 비용은 최대 2x API 호출이지만, 실패율이 낮으면 평균 비용 증가는 미미하다.*
