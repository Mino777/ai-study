# Harness Journal 026 — AI 엔지니어링 중간 점검

> Journal 000 베이스라인부터 025까지, 26회 사이클을 거쳤다.
> 지금 시점에서 **"무엇을 만들었고, 어떻게 제어하고 있으며, 다음엔 뭘 해야 하는가"**를 정리한다.

---

## 전장 개요 — 1인이 4개 프로덕트를 동시 운영하는 구조

핵심 문제: **"혼자서 이걸 다 어떻게 돌리는가?"**

| 프로젝트 | 도메인 | 기술 스택 | 풀어야 할 문제 |
|----------|--------|----------|--------------|
| **ai-study** | AI 엔지니어링 위키 | Next.js 15 + MDX | 방법론을 다른 프로젝트에 즉시 배포 가능한 형태로 박제 |
| **moneyflow** | AI 트레이딩 | Next.js 16 + Supabase + Zustand | AI 매매 판단의 실패를 구조적으로 다음 사이클에 반영 |
| **tarosaju** | AI 타로 서비스 | Next.js 16 + Supabase + Anthropic SDK | AI 출력 품질을 사람 없이 검증·방어 |
| **aidy** | AI 다이어리 (4레포) | Spring Boot + Swift/TCA + Compose | 서버·iOS·Android를 AI 에이전트 4개로 동시 진행 |

**해답: AI를 "도구"가 아니라 "배치 가능한 병력"으로 운용한다.**

---

## 배치한 AI 전력

### 에이전트 편성

| 역할 | 모델 | 배치 방식 | 작전 범위 |
|------|------|----------|----------|
| 주력 개발 에이전트 | Claude Opus/Sonnet (Claude Code) | 터미널 직접 배치, tmux 다중 세션 | 전 프로젝트 코드 작성·리뷰·디버깅 |
| 콘텐츠 생성 에이전트 | Gemini 2.5 Flash | GitHub Actions CI 파이프라인 | ai-study 일일 레슨 자동 생성 |
| QA 에이전트 | Claude + gstack browse | 헤드리스 브라우저 자동화 | 배포 후 시각적 QA + 성능 체크 |
| 전문가 에이전트 | Claude (에페메럴) | `/consult`로 온디맨드 소환 | 법률·보안·마케팅 등 도메인 자문 |

### 외부 프레임워크 — 통째로 쓰지 않고 분해 이식

| 출처 | 가져온 것 (현장 검증 완료) | 버린 것 | 판단 근거 |
|------|--------------------------|---------|----------|
| **gstack** | QA, browse, ship, investigate, compound 등 30+ 스킬 | — | 가장 넓은 커버리지, 즉시 사용 가능 |
| **Superpowers** (obra) | SDD 2-stage review, git-worktrees, brainstorming | finish-branch | 워크플로우 충돌 — 우리 compound 사이클과 안 맞음 |
| **Perpetual Engine** (greatsk55) | MetricsEvaluator, ConsultantFactory, MessageQueue | 전체 프레임워크 | 부분 이식이 더 가벼움 |

### 자체 설계 제어 체계 — 현장에서 만든 것

외부 프레임워크로 안 풀리는 문제를 직접 설계했다.

| 체계 | 풀어야 했던 문제 | 설계한 해법 |
|------|----------------|-----------|
| **Harness Engineering** | AI가 틀린 걸 자신있게 말한다 | 5-Layer 래퍼: Input → Context → Inference → Output → Observability |
| **Compound Engineering** | 같은 실수가 다음 세션에서 반복된다 | Plan → Work → Review → Compound 4단계 + 인시던트 → 다음 입력 변환 |
| **Hub-Worker Model** | 4개 프로젝트 간 지식이 파편화 | 허브 1개(ai-study) + 워커 N개, 파일 기반 비동기 메시지 큐 |
| **LLM-First Wiki** | AI 에이전트가 위키를 못 읽는다 | 모든 엔트리에 AI Agent Directive 강제 (트리거·전제조건·실행 단계·안티패턴) |

## 프로젝트별 AI 운용 상세

### ai-study — 방법론 허브

**미션**: 현장에서 검증한 패턴을 박제하고, 다른 프로젝트에 즉시 배포

```
[정찰] 지식 그래프 분석 → 빈 영역·약한 연결 탐지
  → [생성] Gemini Flash 일일 레슨 (09:00 KST 자동)
  → [검증] Zod 스키마 + /validate-mdx + /validate-ai-output
  → [박제] MDX 엔트리 (AI Agent Directive 포함)
  → [배포] JIT 벡터 검색으로 에이전트가 즉시 호출 (99.8% 토큰 절감)
  → [측정] search-hits.json + shadow-benchmark
```

| 제어 포인트 | 구현 | 인시던트 출처 |
|------------|------|-------------|
| 입력 검증 | Zod frontmatter (13 카테고리, confidence 1-5) | Gemini가 잘못된 카테고리 넣어서 빌드 반복 실패 |
| 출력 검증 | `/validate-mdx` (5대 함정), `/validate-ai-output` (4대 함정) | AI 생성 MDX가 Shiki+Mermaid 충돌, 가짜 URL 인용 |
| 컨텍스트 절감 | JIT 검색 (multilingual-e5-small, 384d) | 위키 전체 331K tokens → ~800 tokens |
| 관측 | search-hits.json + weekly shadow-benchmark | "검색이 실제로 작동하는가"를 숫자로 확인 |

### aidy — 1인 4레포 동시 개발

**미션**: 서버·iOS·Android를 AI 4개로 동시 개발. 사람은 스펙만 쓰고 게이트만 검증

```
[스펙] api-contract.md (단일 진실 원천)
  → [명령] aidy-architect가 Work Order 발행
      ├── Claude #1: aidy-server (Spring Boot + Kotlin)
      ├── Claude #2: aidy-ios (Tuist + TCA + SwiftUI)
      └── Claude #3: aidy-android (Compose + MVVM)
  → [게이트] Gate 1: 스펙 준수 / Gate 2: 크로스 플랫폼 통합
  → [보고] 파일 기반 비동기 메시지 큐
```

| 제어 포인트 | 구현 | 인시던트 출처 |
|------------|------|-------------|
| Spec = Law | `api-contract.md` 단일 진실 | 4레포에서 각각 다른 해석 → 통합 시 필드 불일치 |
| 2-Stage Review | Stage 1: 스펙 compliance / Stage 2: 코드 품질 | AI가 "돌아가는 코드"를 만들지만 스펙과 어긋남 |
| tmux 4-pane 병렬 | 4개 Claude Code 세션 동시 실행 | 순차 실행 시 하루 걸릴 작업을 2-3시간으로 단축 |
| 비동기 메시지 큐 | messages/ JSON (Perpetual Engine 이식) | 세션 간 상태: "서버 완료" → "iOS 연동 시작" |

### moneyflow — AI 트레이딩

**미션**: AI 매매 판단의 실패를 구조적으로 다음 사이클 입력으로 변환

| 제어 포인트 | 구현 | 인시던트 출처 |
|------------|------|-------------|
| Circuit Breaker | AI 호출 실패 시 자동 차단 + fallback | API 장애 시 무한 재시도 → 비용 폭발 |
| 비용 추적 | ccusage 베이스라인 + 트레이드당 비용 로깅 | AI API 비용이 수익 초과하는 순간 감지 |
| 일일 Compound | 매매 결과 → 다음날 시그널 입력 | "오늘의 실패"가 "내일의 판단 근거"로 자동 변환 |
| 패턴 수신 | ai-study 허브에서 3+ 세션 검증된 패턴만 이식 | 미검증 패턴 직접 적용 → 롤백 비용 |

### tarosaju — AI 출력 방어

**미션**: 사용자에게 도달하는 AI 출력 품질을 사람 없이 보장

| 제어 포인트 | 구현 | 인시던트 출처 |
|------------|------|-------------|
| 출력 구조 검증 | 타로 해석 응답 스키마 강제 | AI가 형식 무시 → 프론트 파싱 실패 |
| 3-Layer API 방어 | tarosaju → moneyflow → aidy-server 캐스케이드 | 단일 레이어는 한 곳 뚫리면 끝 |
| 보안 하드닝 | 허브에서 검증된 패턴 이식 | 프로젝트마다 개별 보안 설계는 비효율 |

## 지휘 통제 — 크로스 프로젝트 운용

### 허브-워커 지식 순환

```
ai-study (허브: 방법론 센터)
  │
  ├─→ 패턴 검증 (3+ 세션 검증 후)
  │     ├─→ moneyflow  ← /curate-inbound
  │     ├─→ tarosaju   ← /curate-inbound
  │     └─→ aidy       ← Spec-Driven 독립 운영
  │
  ├─← 워커 피드백 (인시던트, 새 패턴 발견)
  │     └─→ 허브에서 일반화 → 전 프로젝트 재배포
  │
  └── /projects-sync: 6레포 상태 동시 진단
  └── /cross-session-review: 다른 Claude 세션 diff 검증
```

핵심: **양방향 순환.** 허브 → 워커 일방향이 아니라, 워커 인시던트가 허브로 올라오고 일반화되어 전체에 배포된다.

## 운용 원칙 7가지 — 전부 인시던트에서 나왔다

이론이 아니라 현장 실패에서 도출한 원칙.

| # | 원칙 | 어떤 실패에서 나왔나 |
|---|------|-------------------|
| 1 | **행동 가드 > 메모리 가드** | CLAUDE.md에 "하지 마라" 써도 AI가 무시 → 훅으로 물리적 차단 |
| 2 | **N=3 승격** | 같은 솔루션 3번째 작성 → "왜 아직 자동화 안 됐지?" → 코드 게이트 |
| 3 | **인시던트 = 다음 입력** | 장애 로그 버리고 다음 날 같은 실수 → Compound로 구조적 피드백 루프 |
| 4 | **Spec = Law** | 4레포 AI가 각자 "돌아가는 코드" → 통합하니 필드 불일치 → 스펙이 유일한 진실 |
| 5 | **Compound 스킵 금지** | "급하니까 회고 생략" → 2주 후 동일 버그 재발 → 4단계 전부 강제 |
| 6 | **Diff 기반 검증** | 커밋 메시지 "fix: 버그 수정"인데 diff는 전혀 다른 파일 → diff만 신뢰 |
| 7 | **LLM-First** | AI가 위키 읽고 "해당 정보 없음" → 사람용이 아니라 에이전트용으로 구조 재설계 |

## 토큰 경제학 — 비용도 엔지니어링한다

AI를 대규모 배치하면 비용이 폭발한다. 비용 제어는 운영의 일부다.

| 레버 | 전략 | 효과 |
|------|------|------|
| A. Prefix | CLAUDE.md 60줄 제한 + 스킬 파일 분산 | 매 턴 고정 토큰 최소화 |
| B. 캐시 | 0.28-turn 손익분기, cache read 0.1x | 반복 컨텍스트 90% 절감 |
| C. 출력 절감 | JIT 검색 (331K → ~800 tokens), 서브에이전트 요약 | 99.8% 컨텍스트 절감 |
| D. 모델 라우팅 | 컨텍스트 60% 임계값 → 세션 분리 or compact | 품질 저하 전 선제 대응 |
| E. 비용 추적 | ccusage 프로젝트별 분리 (moneyflow+tarosaju 70%) | 레버 ROI 타겟 식별 |

**적용 4 AND 조건**: 공식문서 근거 + 실측 데이터 + 부작용 관찰 가능 + 롤백 용이. 하나라도 불충분하면 보류.

## 차별화 — "AI를 쓴다" vs "AI를 배치하고 제어한다"

### 프레임워크 조합자

대부분은 하나의 AI 프레임워크를 그대로 쓴다.
여기서는 **여러 프레임워크를 분해해서 필요한 모듈만 이식하고, 빈 곳은 직접 설계한다.**

- gstack 30+ 스킬 + Superpowers SDD + Perpetual Engine 3개 패턴 = 선택적 이식
- Compound, Harness, Hub-Worker, LLM-First = 현장 자체 설계

### 다중 에이전트 오케스트레이션

- **aidy**: tmux 4-pane에서 Claude 4개 동시 실행, 스펙 기반 동기화
- **허브-워커**: 프로젝트 간 비동기 메시지 큐로 지식 순환
- **크로스 세션 검증**: 다른 Claude 세션 결과를 diff 기반 교차 검증

### 6계층 AI 제어

| 계층 | 제어 내용 | 도구 |
|------|---------|------|
| **입력** | 스키마 강제, 프롬프트 구조화 | Zod, CLAUDE.md, SPEC.md |
| **컨텍스트** | 필요한 지식만 정밀 주입 | JIT 벡터 검색, 쿼리 라우터 |
| **추론** | 모델 선택, 비용 라우팅 | 용도별 모델 배정, 60% 임계값 |
| **출력** | 구조 검증, 가짜 정보 탐지 | /validate-mdx, /validate-ai-output |
| **관측** | 성능·비용·적중률 추적 | shadow-benchmark, ccusage, search-hits |
| **행동** | 위험 행동 물리적 차단 | pre-commit hook, .claude hook, N=3 승격 |

### 인시던트 기반 진화

이론에서 출발하지 않는다. **인시던트 → 원칙 → 코드 게이트**의 순서.

```
인시던트 발생
  → docs/solutions/ 기록 (1회차)
  → 같은 패턴 2회차 → 솔루션 업데이트
  → 3회차 → /promote-solution → 훅/validator 코드 게이트 승격
  → 이후 자동 차단 (사람 개입 불필요)
```

모든 원칙과 가드는 이 경로를 거쳐 만들어졌다. "좋은 관행"이 아니라 **"반복된 실패의 구조적 대응"**.

## For AI Agents

### Trigger
- 프로젝트 간 AI 운용 현황을 파악해야 할 때
- 새 프로젝트에 AI 에이전트 배치 구조를 설계할 때
- "지금까지 뭘 만들었지?"라는 질문에 답해야 할 때

### Prerequisites
1. 허브-워커 모델 이해 (→ cross-project-pattern-porting-methodology)
2. 5-Layer 하네스 구조 이해 (→ five-levers-of-harness-engineering)
3. Compound 사이클 이해 (→ compound-engineering-philosophy)

### Actionable Steps
1. 이 엔트리의 "프로젝트별 AI 운용 상세" 섹션에서 대상 프로젝트 찾기
2. 해당 프로젝트의 제어 포인트 표에서 현재 상태 확인
3. "다음 점검까지 해야 할 것"과 대조해서 미완료 항목 식별
4. 관련 Journal 엔트리로 상세 맥락 추적

### Anti-patterns
- ❌ 이 엔트리를 "최신 현황"으로 맹신 — 작성 시점(2026-04-22) 이후 변경 있을 수 있음
- ❌ 개별 프로젝트 상세를 여기서만 파악 — 각 프로젝트의 CLAUDE.md/SPEC.md가 진실
- ❌ 정량 지표를 고정값으로 인용 — `npm run search`, `git log`, ccusage로 최신 수치 확인
