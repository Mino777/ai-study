## 왜 중요한가: 문서(Documentation)에서 운영 정책(Operations)으로

AI 에이전트에게 작업을 지시하는 `AGENTS.md` 파일은 인간 개발자를 위한 `README.md`가 아니다. 가장 흔한 실패 원인은 에이전트에게 '문서'를 읽히려는 시도에서 비롯된다. 에이전트는 "깨끗한 코드를 지향한다"와 같은 산문적 선언이나 "주의해서 마이그레이션 하세요" 같은 모호한 지침을 이해하고 실행하지 못한다.

`AGENTS.md`는 AI를 위한 **운영 정책(Operational Policy)**이어야 한다. 즉, 실행 가능한 명령어, 명확한 성공/실패 기준, 그리고 컨텍스트에 따른 규칙의 집합이다. Crosley는 이 주장을 뒷받침하기 위해 GitHub의 2,500개 이상 저장소 분석 결과를 인용한다: "대부분의 에이전트 파일은 기술적 한계가 아니라 너무 모호해서 실패한다." 또한 ICLR 2026의 AMBIG-SWE 연구를 언급하며, 모호한 지침을 받은 에이전트는 질문하는 대신 조용히 실패하는 경향이 있어 작업 해결률이 48.8%에서 28%로 급락했다고 지적한다.

따라서 효과적인 `AGENTS.md` 작성의 핵심은 사고의 전환에 있다. '이해'시키는 것이 아니라 '실행하고 검증'하게 만들어야 한다.

## 핵심 패턴: 에이전트가 '실행'하는 지시

Crosley는 실험을 통해 에이전트의 행동을 실질적으로 변화시키는 패턴들을 식별했다. 이 패턴들의 공통점은 '검증 가능성'이다.

### 1. 명령어 우선 (Command-First Instructions)

에이전트에게 무엇을 원하는지 설명하는 대신, **실행할 정확한 셸 명령어**를 제공해야 한다.

- **Bad**: "코드를 제출하기 전에 린트를 실행하고 포맷을 맞춰주세요."
- **Good**:
  ```bash
  # 빌드 및 테스트 명령어
  - Install: `npm install`
  - Lint: `npx eslint . --fix`
  - Format: `npx prettier . --write`
  - Test: `npm test`
  - Full verify: `npm run lint && npm run format && npm test`
  -```

에이전트는 이 명령어들을 직접 실행하고, 종료 코드(exit code)를 통해 성공 여부를 명확히 판단할 수 있다. 모든 지침은 "이 작업이 올바르게 완료되었음을 증명하는 명령어는 무엇인가?"라는 질문에 답해야 한다.

### 2. 완료 조건의 명시적 정의 (Definition of Done)

에이전트가 "작업이 끝났다"고 섣불리 보고하는 것을 막기 위해, '완료'의 기준을 명확히 정의해야 한다.

```markdown
## 완료의 정의 (Definition of Done)
하나의 태스크는 아래 **모든** 조건이 충족되었을 때 완료된 것으로 간주한다:
1. `npx eslint .`가 종료 코드 0으로 성공
2. `npm test`가 실패 없이 종료 코드 0으로 성공
3. 변경된 파일이 스테이징되고 커밋됨
4. 커밋 메시지가 Conventional Commits 형식을 따름: `type(scope): description`
```

이런 정의가 없으면 에이전트의 '완료'는 "내 생각에 끝난 것 같다"는 자의적 판단에 불과하며, 이는 버그의 주요 원인이 된다.

### 3. 작업별 섹션 구성 (Task-Organized Sections)

모든 명령어를 나열하기보다, 에이전트의 현재 컨텍스트에 맞춰 지침을 그룹화한다. `When...` 접두사는 에이전트가 자신의 작업 상태를 파악하고 관련 지침을 선택하는 데 직접적인 도움을 준다.

```markdown
## 코드를 작성할 때 (When Writing Code)
- 모든 새 함수에 JSDoc 타입 힌트 추가
- 테스트 명령어: `npm test -- --testPathPattern=&quot;test_&lt;module&gt;&quot;`

## 코드를 리뷰할 때 (When Reviewing Code)
- 보안 취약점 확인: `npx snyk test`
- 테스트 커버리지 검증: `npm test -- --coverage --coverageReporters=&quot;text&quot; --coverageThreshold='{&quot;global&quot;: {&quot;branches&quot;: 80}}'`

## 릴리즈할 때 (When Releasing)
- `package.json` 버전 업데이트
- 전체 테스트 실행: `npm run lint && npm test`
- 태그 생성: `git tag -a v&lt;version&gt; -m &quot;Release v&lt;version&gt;&quot;`
```

### 4. 에스컬레이션 규칙 (Escalation Rules)

에이전트가 막혔을 때 해야 할 일과 절대 하지 말아야 할 일을 명시한다. 이는 에이전트가 lock 파일을 삭제하거나 테스트를 건너뛰는 등 파괴적인 방식으로 문제를 '해결'하려는 시도를 막는다.

```markdown
## 막혔을 때 (When Blocked)
- 테스트가 3번 시도 후에도 실패하면: 중단하고 실패한 테스트의 전체 결과 보고
- 의존성 문제 발생 시: `package.json`을 먼저 확인한 후 질문
- 병합 충돌 발생 시: 중단하고 충돌 파일 목록 표시

## 절대 금지 (Never)
- 에러 해결을 위해 파일 삭제
- 강제 푸시 (force push)
- 테스트 건너뛰기
```

## 실전 적용: `ai-study` 프로젝트의 위키 엔트리 작성

`ai-study` 프로젝트에 새 위키 엔트리를 추가하는 작업을 AI 에이전트에게 맡기는 시나리오를 가정해보자. 효과적인 `AGENTS.md`는 다음과 같이 작성될 수 있다.

```typescript
// ai-study/AGENTS.md

## 완료의 정의 (Definition of Done)
새 위키 엔트리 추가 작업은 아래 조건이 모두 통과되어야 완료된다:
1. `npm run lint:mdx`가 종료 코드 0으로 성공
2. `npm run spellcheck`가 종료 코드 0으로 성공
3. 추가된 MDX 파일이 `git add`로 스테이징됨
4. 커밋 메시지가 `docs(wiki): Add new entry for [TOPIC]` 형식을 따름

## 위키 엔트리 작성 시 (When Writing a New Wiki Entry)
- 모든 코드 예시는 TypeScript 또는 Swift로 작성
- Mermaid 다이어그램을 1개 이상 포함. Mermaid 5대 함정 규칙 준수.
- 외부 자료 인용 시, 원문 링크와 저자를 명시.
- 작업 완료 후 `npm run lint:mdx`와 `npm run spellcheck`를 순서대로 실행하여 검증.

## 막혔을 때 (When Blocked)
- `lint:mdx` 에러 발생 시: 중단하고 에러 메시지를 포함하여 보고
- 내용에 대한 확신이 없을 때: "[CLARIFICATION NEEDED]" 태그를 남기고 중단
```

이 `AGENTS.md` 파일은 에이전트에게 추상적인 품질 기준("글을 잘 써라")을 요구하는 대신, 실행 가능한 검증 명령어(`npm run lint:mdx`)와 명확한 완료 조건(종료 코드 0, 커밋 메시지 형식)을 제공한다.

아래는 이 운영 정책에 따라 에이전트가 작업을 수행하는 과정을 나타낸 Mermaid 다이어그램이다.

```mermaid
graph TD
    A[Start: "새 위키 엔트리 작성"] --> B{Read AGENTS.md: "When Writing..."};
    B --> C[1. 엔트리 초안 작성];
    C --> D["2. 검증 명령어 실행: `npm run lint:mdx`"];
    D --> E{Exit Code == 0?};
    E -- Yes --> F["3. 검증 명령어 실행: `npm run spellcheck`"];
    F --> G{Exit Code == 0?};
    G -- Yes --> H["4. 커밋 메시지 형식에 맞춰 커밋"];
    H --> I([Task Complete]);
    
    E -- No --> J{Read AGENTS.md: "When Blocked"};
    J --> K["중단 후 에러 보고"];
    K --> L([Task Failed]);
    
    G -- No --> J;
```
