## ADR-008 — SSE 채택 결정

### 선택 이유

| 기준 | SSE | WebSocket |
|------|-----|-----------|
| 방향성 | 서버→클라 단방향 | 양방향 |
| Spring 지원 | SseEmitter 기본 | 추가 의존성 |
| 프록시/LB | HTTP 1.1 친화 | Upgrade 헤더 처리 |
| 재연결 | 프로토콜 기본 제공 | 수동 구현 |
| 바이너리 | 텍스트 전용 | 지원 |
| 채팅 응답 | ✅ 적합 | 과다 |

채팅 토큰 흐름은 단방향. `SseEmitter` 로 충분. ADR-008에 **"단방향이면 SSE"** 원칙 박제, BACKLOG P-002 완료 처리.

### Phase 분리 선언

ADR-008에 **Phase 1/2 경계 명시**:

- **Phase 1**: fake-stream — 완성된 응답 텍스트를 공백/문장 단위로 분할해 `onToken` 호출. Anthropic API는 기존 non-stream 그대로.
- **Phase 2**: Anthropic Messages API `stream=true` 실연동 — SSE 라인 파싱 + `content_block_delta` 이벤트 처리.

Feature Flag 미사용. Phase 경계는 **ADR 문서 + 커밋 히스토리**로 표현.

## Phase 1 (Session 5 R7) — Fake-Stream

### 범위

- **서버만** — POST /api/chat/stream 엔드포인트 + SseEmitter
- **클라 미변경** — 기존 POST /api/chat 유지, 스트리밍 UI는 Phase 2에서

### 구현 요약

```kotlin
// ChatController — SSE 엔드포인트
@PostMapping("/chat/stream")
fun chatStream(...): SseEmitter {
    val emitter = SseEmitter(0L)
    // 응답 완성 후 공백 분할해 토큰씩 전송 (Phase 1)
    aiService.chat(...).reply.split(" ").forEach { token ->
        emitter.send(SseEmitter.event().data(token))
    }
    emitter.complete()
    return emitter
}
```

- 테스트 158 passed (R7)
- rate limit: chat rpm=20 **버킷 공유** (별도 버킷 안 만듦)

## Phase 2 (Session 6 R2~R4) — Anthropic 실연동

### R2 — 서버 실연동

`AiService.kt` 에 `chatStream` 추가 (기존 `chat` 은 유지):

```kotlin
fun chatStream(
    user: User, history: List<ChatMessage>, memories: List<Memory>,
    userMessage: String,
    onToken: (String) -> Unit,
    onDone: (AiReply) -> Unit,
    onError: (Throwable) -> Unit,
) {
    try {
        val aiReply = circuitBreaker.execute {
            executeStreaming(user, history, memories, userMessage, onToken)
        }
        onDone(aiReply)
    } catch (e: Throwable) { onError(e) }
}

private fun executeStreaming(...): AiReply {
    val request = Request.Builder()
        .url("https://api.anthropic.com/v1/messages")
        .addHeader("accept", "text/event-stream")
        .post(/* stream=true body */)
        .build()

    return chatClient.newCall(request).execute().use { response ->
        val source = response.body!!.source()
        while (!source.exhausted()) {
            val line = source.readUtf8Line() ?: break
            if (!line.startsWith("data:")) continue
            val json = objectMapper.readTree(line.substring(5).trim())
            when (json["type"]?.asText()) {
                "content_block_delta" -> {
                    val text = json["delta"]?.get("text")?.asText() ?: ""
                    accumulated.append(text); onToken(text)
                }
                "message_stop" -> break
                "error" -> throw ApiException(ErrorCode.INTERNAL_ERROR)
            }
        }
        // accumulated → extractMemories → AiReply
    }
}
```

**Circuit Breaker 스트리밍 확장**: 기존 `chat()` 과 동일하게 `circuitBreaker.execute { ... }` 래핑. OPEN 상태면 스트림 시작도 전에 즉시 `onError`.

### R3 — iOS SSE 구독

`URLSession.bytes(for:)` + AsyncLineSequence:

```swift
let (bytes, response) = try await URLSession.shared.bytes(for: request)
for try await line in bytes.lines {
    guard line.hasPrefix("data:") else { continue }
    let json = try JSONDecoder().decode(StreamChunk.self, from: data)
    switch json.type {
    case "content_block_delta": await onToken(json.delta?.text ?? "")
    case "message_stop": return
    }
}
```

### R4 — Android SSE 구독

OkHttp `BufferedSource.readUtf8Line()`:

```kotlin
val response = client.newCall(request).execute()
val source = response.body!!.source()
while (!source.exhausted()) {
    val line = source.readUtf8Line() ?: break
    if (!line.startsWith("data:")) continue
    when (val type = parseType(line)) {
        "content_block_delta" -> onToken(parseText(line))
        "message_stop" -> return
    }
}
```

3 플랫폼 **모두 외부 SSE 라이브러리 0**. ADR-003 (클라이언트 동기화) 원칙 유지.

### R5 — SSE 회복성

- 연결 끊김 시 재시도 (2회, backoff 2s → 4s)
- latency 계측 (P50/P95 메모리 상주)

## 점진 도입의 효과

### Feature Flag 없이도 안전

```
Phase 1 배포 (R7)
  ↓ 엔드포인트 존재, 클라 미사용 (dark launch)
  ↓ 서버 측 성능/안정성 관찰 — 158 tests, 0 failures
Phase 2 서버 (R2)
  ↓ 동일 엔드포인트의 내부 구현만 교체 — 클라 계약 불변
  ↓ Circuit Breaker가 기존 흐름과 동일 보호
Phase 2 클라 (R3, R4)
  ↓ SSE 구독 UI 활성화
Phase 2 회복성 (R5)
  ↓ 재시도 + latency 계측
```

**동일 엔드포인트 계약**을 유지했기 때문에 Phase 경계는 *ADR 문서 + 구현 교체* 만으로 관리 가능. Feature Flag 토글 로직/운영 오버헤드 0.

### Circuit Breaker 재사용

기존 non-stream `chat()` 에서 검증된 `AiCircuitBreaker` 를 스트림에서도 **동일 래핑**. 새 CB 구현 불필요. 스트림이 에러면 OPEN 상태 반영 → 다음 요청에서 즉시 차단.

### Phase 3 대기

- Anthropic 공식 event_type 전수 처리 (error, ping, usage) — R5 에서 부분 도입, 전체는 다음 세션
- 클라 SSE `AsyncThrowingStream` 배리어 통합 (iOS 17+)

## AI Agent Directive

### Trigger
- 스트리밍 기능을 도입하려 할 때 프로토콜 선택 판단 필요
- 한 세션에 전체 쌓기 vs 여러 세션 나눠 Phase 도입 결정 필요
- 외부 SSE/WebSocket 라이브러리 도입 유혹

### Prerequisites
- [Aidy Journal 002](/wiki/harness-engineering/aidy-journal-002-stability-observability-test-integrity) — Circuit Breaker Phase 1 구조
- [Backend Flow Map](/wiki/backend-ai/backend-flow-map-via-aidy-server) — AiService 위치

### Actionable Steps

1. **방향성 체크** — 단방향(서버→클라)이면 SSE, 양방향이면 WebSocket
2. **ADR에 Phase 경계 명시** — Phase 1/2 범위 + 승격 조건 선언
3. **Phase 1 dark launch** — 엔드포인트만 배포, 클라는 후속 Phase
4. **동일 엔드포인트 계약 유지** — 내부 구현 교체 방식으로 Feature Flag 회피
5. **외부 SSE 라이브러리 사용 금지** — iOS `URLSession.bytes`, Android `BufferedSource.readUtf8Line()`, Spring `SseEmitter` 로 충분
6. **Circuit Breaker 스트림 적용** — 기존 CB를 `execute { ... }` 래핑으로 재사용

### Anti-patterns

- **스트리밍=WebSocket 기본값** — 채팅 응답은 SSE로 충분
- **Phase 1/2 구별 없이 한 번에 전부** — 회귀 위험 + 롤백 단위 커짐
- **Feature Flag 과용** — Phase 경계는 ADR로 표현 가능
- **외부 SSE 라이브러리 도입** — iOS EventSource/Android OkHttp-SSE 등 불필요, 보안/업데이트 리스크 증가
- **Circuit Breaker 재구현** — 기존 CB 래핑 패턴으로 재사용
