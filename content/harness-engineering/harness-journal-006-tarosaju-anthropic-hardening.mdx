## 만든 환경 (After)

### 추가: `src/lib/anthropic-client.ts` — `callAnthropic()` wrapper

moneyflow [`ai-client.ts` 패턴 6가지](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback) 중 적용 가능한 것을 선별:

| 패턴 | 적용 | 이유 |
|---|---|---|
| **timeout** (60s AbortSignal) | ✅ | 단일 프로바이더에도 필수 |
| **지수 백오프 재시도** (2회, 에러별 multiplier) | ✅ | 일시적 네트워크/rate limit 대응 |
| **에러 종류별 분류** | ✅ | client_error는 즉시 throw (재시도 무의미) |
| **prompt caching 지원** (system array 형식) | ✅ | 인프라만 준비 — 실제 적용은 다음 사이클 |
| **usage 필드 반환** | ✅ | 다음 사이클 비용 추적 준비 |
| **Circuit Breaker** | ❌ | *단일 프로바이더라 의미 제한* |
| **Multi-Provider 폴백** | ❌ | 단일 프로바이더 |

### 핵심 코드

```typescript
export async function callAnthropic(opts: CallAnthropicOpts): Promise<CallAnthropicResult> {
  const timeoutMs = opts.timeoutMs ?? 60_000;
  const maxRetries = opts.maxRetries ?? 2;

  let lastError: unknown;

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), timeoutMs);

    try {
      const response = await client.messages.create(
        { model: opts.model, max_tokens: opts.maxTokens, system: opts.system, messages: opts.messages },
        { signal: controller.signal },
      );
      clearTimeout(timeoutId);
      return { text, usage };
    } catch (err) {
      clearTimeout(timeoutId);
      lastError = err;
      const kind = classifyError(err); // timeout/rate_limit/server_error/client_error/unknown

      if (kind === "client_error") throw err; // 재시도 무의미
      if (attempt >= maxRetries) throw err;

      const delay = backoffDelay(attempt, kind); // rate_limit 4x, server_error 2x, timeout 1x
      await sleep(delay);
    }
  }
  throw lastError;
}
```

### 두 라우트 refactor

**fortune/ai**: `client.messages.create` → `callAnthropic` 래핑. 로직 변경 0.

**tarot-chat**: 동일 래핑 + **try/catch 신설**. 기존엔 SDK 에러가 사용자에게 그대로 전파됐는데, 이제 `"상담사를 잠시 불러올 수 없어요. 다시 시도해주세요."` 안전 메시지로 대체.

### 자동 검증 게이트의 가치

이 사이클은 *로컬 브라우저 QA 없이* push. 이유:
- API 라우트 변경 (UI 렌더링 아님)
- wrapper의 동작은 *기존과 동일*한 결과 + 실패 시 더 안전
- tarosaju `test.yml`이 [Journal 001](/wiki/harness-engineering/harness-journal-001-ci-merge-gate)에서 박은 자동 게이트 — PR 생성 즉시 5 jobs 돌고 결과 노출

이게 *자동 게이트가 쌓이면* 사람이 매 PR마다 브라우저를 띄우지 않아도 된다는 것을 라이브 증명. 직전 사이클의 그래프 사고(*브라우저 검증 건너뛴 대가*)와 대비: *UI 변경은 브라우저 필수*, *API 라우트 변경은 자동 게이트로 충분*. 구분이 중요.

## 배운 것 / 다음 후보

### 핵심 통찰 1 — 사이클 분할의 경계

원래 Journal 000 큐에서 006/007/008로 분할했던 작업:
- 006: timeout + retry
- 007: cache_control
- 008: Circuit Breaker

실전에서 006+007은 *같은 wrapper 파일 한 곳*에 구현되어서 분리 비용이 통합 비용보다 큼. 한 사이클로 묶음. 008(Circuit Breaker)는 단일 프로바이더 환경에서 *가치 제한*이라 의도적 제외.

**사이클 분할은 메시지 단위이지 기능 단위가 아니다**. 같은 파일에서 서로를 전제로 하는 기능들은 묶는 게 맞다.

### 핵심 통찰 2 — 비대칭 이식은 *전체 복사*가 아닌 *선별*

moneyflow의 6가지 패턴 중 tarosaju에 *적용 가능한 4가지*만 선별. Circuit Breaker와 Multi-Provider 폴백은 *환경 차이*로 가치 제한 → 제외. 복사-붙여넣기보다 *환경에 맞는 선별*이 더 좋은 이식.

### 핵심 통찰 3 — 자동 게이트의 누적 가치

[Journal 001](/wiki/harness-engineering/harness-journal-001-ci-merge-gate)에서 박은 test.yml CI 게이트가 *이 사이클의 검증자*. 내가 로컬 브라우저 QA를 *못 하더라도* 자동으로 5 jobs가 돈다. 이게 Compound Engineering이 말하는 *"다음 작업이 더 쉬워진다"*의 구체적 형태: *한 사이클의 인프라가 다음 사이클의 검증 부담을 줄인다*.

단, *UI 변경*은 여전히 브라우저 QA 필수. 직전 그래프 사고가 그 경계를 명확히 가르침.

## 자기 점검

1. 내가 비대칭 이식을 할 때 *전체 복사* vs *선별*을 어떻게 결정하는가? 환경 차이를 충분히 고려하는가?
2. *같은 파일 한 곳*에 구현되는 기능들을 억지로 여러 사이클로 쪼갠 적 있는가? 그게 진짜 가치 있었는가?
3. 자동 게이트 누적이 *내 매일 작업의 검증 부담*을 얼마나 줄여주는가? 줄여주지 않는다면 게이트가 *잘못 설정*된 것일 수 있다.
4. UI 변경 vs API 변경의 *브라우저 QA 필요성*을 명확히 구분하는가?
5. (열린 질문) Circuit Breaker가 *단일 프로바이더 환경*에서도 가치 있는 경우가 있는가? (힌트: 자신의 DB나 내부 서비스로 폴백)

### 실습 과제

자신의 AI 호출 코드 한 곳에:

1. 60초 timeout 추가 (`AbortSignal.timeout(60_000)` + `{ signal }` 옵션)
2. try/catch로 에러 감싸고 사용자용 안전 메시지 반환
3. 여기까지 30분 안에 가능
4. 2회 재시도는 그 다음 사이클에

이 한 사이클만으로도 *무한 대기 + 무방비 에러 노출*이 사라짐.
