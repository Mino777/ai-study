## 목적

Harness Journal에 넣기에는 *한 사이클*을 이루지 않는 소형 환경 수정들을 한 곳에 박제한다. 증상을 발견한 날짜 기준으로 섹션을 append하는 *살아있는 카탈로그*.

같은 문제를 두 번째 마주쳤을 때 "어제 고쳤는데 어떻게 했더라"로 30분을 태우지 않기 위한 장치다. Journal이 *서사*라면 이 로그는 *인덱스*다.

## For AI Agents

이 문서를 발견한 에이전트가 즉시 사용할 수 있도록:

- **언제 append할까** — 사용자가 환경/툴 설정 관련 불편을 고쳐달라고 했고, 그 해결책이 (a) 특정 프로젝트에 종속되지 않고 (b) 1 사이클짜리 Journal을 이룰 만큼 크지 않을 때.
- **어떻게 append할까** — 아래 "로그" 섹션 하단에 *새 섹션*을 추가. 기존 엔트리는 **절대 수정/삭제 금지** (append-only). 연관 엔트리가 있으면 *Related* 라인에 링크.
- **엔트리 포맷** (아래 템플릿 준수):

```
## YYYY-MM-DD — <한 줄 증상>

**환경**: <OS / 에디터 / 툴 버전 등>
**증상**: <관찰 가능한 현상 1~2 문장>
**근본 원인**: <진단 — 확실하지 않으면 "가설:"로 표시>
**수정**: <복붙 가능한 명령/설정>
**검증**: <고쳐졌는지 확인하는 방법>
**Related**: <연관 엔트리 링크, 없으면 생략>
```

- **Journal로 승격해야 할 때** — 같은 수정이 3개 이상의 프로젝트에 이식되거나, 아예 슬래시 커맨드/훅으로 자동화할 가치가 생기면 Harness Journal 엔트리로 승격. 승격 후 이 로그의 원래 섹션은 **지우지 않고** Related에 Journal 링크만 추가.

- **중복 방지** — append 전에 이 파일을 grep하여 같은 증상이 이미 박제됐는지 확인. 이미 있으면 해당 섹션에 *재발 노트*를 보조 bullet으로 추가하되 본문은 수정하지 않는다.

### 워커 세션이 허브에 꿀팁 PR을 쏘는 흐름 (Journal 014에서 확립)

각 워커 프로젝트 세션(moneyflow/tarosaju 직접 작업 세션)이 *자기 프로젝트에서 발견한 환경 팁*을 발견하면, 그것을 이 로그에 append하는 PR을 **ai-study(허브)에 직접 올린다**. 허브 세션이 그걸 받아서 자동 머지 + (필요 시) 편집으로 히스토리를 누적한다.

**워커 세션의 PR 제작 단계**:

1. `git -C /Users/jominho/Develop/ai-study worktree add -b tips/<branch-name> /tmp/ai-study-tips-<branch-name> origin/main` — 허브 리포에 worktree 분기
2. `content/harness-engineering/dev-setup-tips-log.mdx`의 `# 로그` 섹션 하단에 새 섹션 append (위 템플릿 준수)
3. `git commit -m "tips: <날짜> <한 줄 증상>"` + `git push -u origin tips/<branch-name>`
4. 끝. **PR을 직접 만들 필요 없음** — ai-study의 `ai-review.yml`이 push 이벤트를 잡아서 자동 PR 생성 → Test Gate(`npm test + npm run build`, prebuild에서 frontmatter zod + manifest 검증) → Squash Merge 까지 일괄 처리.
5. 워크트리 정리: `git -C /Users/jominho/Develop/ai-study worktree remove /tmp/ai-study-tips-<branch-name>`

**허브 세션의 처리**:

- 기본값: **아무것도 안 함**. 워커 세션이 보낸 PR이 ai-review.yml에 의해 자동 머지되므로 허브는 *사후 검토*만 할 수 있으면 됨.
- 예외: Test Gate 실패 → 허브 세션이 원인 파악 후 재작성 PR을 본인이 올릴지, 워커 세션에게 수정 요청할지 결정. 자동 머지 실패는 ai-review.yml이 PR에 comment를 남기므로 `rtk gh pr list -R Mino777/ai-study --state open`으로 감지.
- 예외: 같은 증상 재발 감지 → 허브 세션이 별도 PR로 기존 섹션에 *재발 노트 보조 bullet* 추가 (본문은 건드리지 않음).
- **승격 판단**: 로그가 누적되다가 "3 프로젝트 이식" 또는 "자동화 가치" 기준을 넘으면 허브 세션이 해당 섹션을 *Harness Journal로 승격*하는 사이클을 시작. 이 로그의 원래 섹션은 삭제하지 않고 Related에 Journal 링크만 추가 (append-only 유지).

**PR 제목 컨벤션**: `tips: YYYY-MM-DD <한 줄 증상>`. 이 prefix로 dev-setup-tips-log 전용 PR임을 허브 세션이 빠르게 식별.

### `patterns/` 브랜치 — 하네스 박제 후보 (Journal 020에서 확장)

`tips/`가 *소형 QoL*이라면, 더 큰 하네스 박제 후보는 **`patterns/` 브랜치 컨벤션**으로 별도 처리한다. 허브의 `/curate-inbound` 슬래시 커맨드가 자동 분류 + 커레이션 + 재포맷 + 익명화까지 일괄.

| Prefix | 용도 | 크기 | 처리 |
|---|---|---|---|
| `tips/` | 소형 환경 QoL 팁 | ≤ 30줄 | `ai-review.yml` 자동 PR → Test Gate → Squash Merge. 허브 사후 검토만 |
| **`patterns/`** | **하네스 박제 후보** (Journal / solution / pattern entry) | 20~500줄 | 자동 PR 생성 후 **`patterns` 라벨** 부착. 허브가 `/curate-inbound`로 분류 + 커레이션 |

**구분 기준**: *"3 프로젝트 중 2개 이상에 이식할 가치가 있는가?"* → YES면 `patterns/`, NO면 `tips/`.

**워커 세션의 `patterns/` PR 제작 단계**:

1. `git -C /Users/jominho/Develop/ai-study worktree add -b patterns/<slug> /tmp/ai-study-patterns-<slug> origin/main`
2. 자유 포맷 raw 설명을 `patterns/<slug>.md` (루트 또는 `patterns/inbox/`)에 작성:
   - **문제**: 어떤 상황에서 발견
   - **진단**: 근본 원인
   - **해결**: before/after 코드
   - **일반화**: 왜 재사용 가능한가
   - **관련 링크**: PR 번호 / 커밋 / 관련 엔트리 후보
3. `git commit -m "patterns: <한 줄 요약>"` + `git push -u origin patterns/<slug>`
4. ai-study `ai-review.yml`이 자동 PR 생성 + **`patterns` 라벨** 부착 (자동 머지 X)
5. 허브 세션이 PR을 발견하면 `/curate-inbound <PR-num>`으로 처리
6. 커레이션 결과가 새 PR(`curate/...`)로 생성됨. 원본 `patterns/` PR은 닫힘
7. 워크트리 정리: `git -C /Users/jominho/Develop/ai-study worktree remove /tmp/ai-study-patterns-<slug>`

**PR 제목 컨벤션**: `patterns: <한 줄 요약>`. 허브는 `gh pr list --search "patterns:" -R Mino777/ai-study`로 큐 확인.

**허브의 분류 6 카테고리** (`/curate-inbound` Phase 2):
- **reject** — 하네스 관련 없음 (사용자 UX 코드 등)
- **tips** — 30줄 이하 환경 QoL → `dev-setup-tips-log.mdx` append
- **solution** — 20~100줄 구체적 문제→해결 → `docs/solutions/<cat>/`
- **journal** — 200~500줄 1 사이클 서사 → 새 `harness-journal-NNN.mdx`
- **pattern-entry** — 100~400줄 재사용 가능한 추상 패턴 → 새 `content/<category>/<slug>.mdx`
- **needs-review** — 분류 자신 없음 → 허브 세션 사용자 결정 요청

자세한 커레이션 룰: [`.claude/commands/curate-inbound.md`](https://github.com/Mino777/ai-study/blob/main/.claude/commands/curate-inbound.md)

### tips vs patterns — 선택 기준 요약

| 상황 | 사용 |
|---|---|
| "Turbopack이 심볼릭 링크에서 터지네, 한 줄 env var 추가로 해결" | `tips/` |
| "Promise 타임아웃 래퍼 15 API routes 적용 + tsc 타입 붕괴 fix 사이클" | `patterns/` |
| "CLAUDE.md 40k 초과 시 FEATURES.md로 분리" | `tips/` |
| "validate-content.mjs 자가손상 버그 + 회귀 테스트 시스템" | `patterns/` (journal 수준) |
| "silent catch 8건에 logError 추가 패턴" | `patterns/` (solution 수준) |

**핵심**: *재사용성 + 서사 단위*로 판단. 한 프로젝트만의 팁이면 `tips/`, 3 프로젝트 공통 가치 있는 패턴이면 `patterns/`.

**허브-워커 model과의 관계**: 이 흐름은 [Journal 011](/wiki/harness-engineering/harness-journal-011-concurrent-session-safety)의 허브-워커 모델을 *반대 방향*으로 사용하는 첫 사례다. Journal 011~013에서 허브가 워커를 *분석/작업*했다면, 이 흐름은 워커가 허브에 *증언*을 기록한다. 양방향이 다 작동해야 허브가 진정한 *지식 저장소* 역할을 한다.

## 왜 append-only인가

이 로그의 가치는 *과거의 나*가 당시 환경에서 어떻게 문제를 해결했는지를 그대로 보여주는 데 있다. 나중에 "더 나은 방법"이 나와도 옛 섹션을 고치면 *그 시점의 증상-해결 쌍*이 사라진다. 새로운 해결책은 *새 섹션*으로 추가하고 옛 섹션에 "2026-MM-DD 업데이트: X 섹션 참조"만 보조로 붙인다.

이것은 git log의 축약본이 아니라 *증상-해결 인덱스*다. git log는 *변경*을 기록하지만 이 로그는 *왜 그 변경이 필요했는지*를 재현 가능한 형태로 기록한다.

## 2026-04-11 — VS Code 통합 터미널 스크롤이 최상단으로 튀는 현상

**환경**: macOS, VS Code 통합 터미널(zsh), Claude Code 세션 장시간 유지.

**증상**: Claude Code 세션을 열어두고 있으면 터미널 스크롤이 자꾸 *최상단*으로 튀어올라 가서, 최신 출력을 보려면 매번 수동으로 맨 아래로 내려야 했다. 타이핑 중에도 포커스와 별개로 뷰포트만 점프.

**근본 원인**: 최근 Claude Code의 "flicker fix" 업데이트가 전통적 라인-추가 렌더링 경로를 쓰면서, 긴 세션에서 스크롤 앵커가 상단으로 튀는 부작용이 있었다. Claude Code는 이를 우회하기 위해 *alternate screen* 기반 풀스크린 렌더링 경로를 ENV 변수로 제공하는데, 이 경로는 스크롤 위치 보존 + 메모리 절감까지 같이 따라온다.

**수정**: `~/.zshrc`에 다음 한 줄 추가.

```bash
export CLAUDE_CODE_NO_FLICKER=1
```

적용: `source ~/.zshrc` 또는 VS Code 완전 종료 후 재시작 (VS Code 통합 터미널은 새 세션 시작 시 `~/.zshrc`를 다시 읽는다).

**검증**:

```bash
echo $CLAUDE_CODE_NO_FLICKER   # → 1
```

그 다음 Claude Code를 다시 실행해 10분 이상 세션 유지 → 스크롤이 최신 출력 위치에 머무는지 확인.

**관련 삽질 기록**: 처음에는 VS Code `settings.json`에 `terminal.integrated.scrollback`, `smoothScrolling`, `fastScrollSensitivity`, `mouseWheelScrollSensitivity`를 넣어봤는데 *근본 수정은 아니었다*. VS Code에는 `terminal.integrated.scrollOnUserInput` 같은 설정이 존재하지 않으므로 에디터 레벨에서 막을 수 없고, Claude Code의 출력 경로 자체를 바꿔야 한다. 위 4개 설정은 남겨도 무해하지만 실제 해결은 ENV 변수가 담당.

**Related**: 없음 — 첫 박제.

## 2026-04-11 — `CLAUDE.md`가 40k 자 경고를 넘어감 (mino-moneyflow 케이스)

**환경**: 모든 Claude Code 프로젝트. 특정 케이스는 `mino-moneyflow/CLAUDE.md` 73.9KB / 1108 줄.

**증상**: Claude Code가 세션 시작 시 경고 출력 — `Large CLAUDE.md will impact performance (47.8k chars > 40.0k)`. 단순 경고지만 실제로 매 세션 시작마다 그 크기만큼의 토큰이 자동 로드되고 있었다.

**근본 원인**: "기능 상세" 섹션이 파일의 60% 이상(라인 155~836, ~45K)을 차지. *모든 세션에 필요한 규약/스택*과 *특정 모듈을 건드릴 때만 필요한 기능 상세*가 한 파일에 섞여 있었다. 전자는 auto-load, 후자는 on-demand여야 한다.

**수정**: 기능 상세를 별도 파일로 추출하고 CLAUDE.md에는 포인터만 남긴다.

1. 기능 상세 블록을 `docs/FEATURES.md`로 이동 (파일 신설).

## 2026-04-12 — `git pull --ff-only origin main`이 엉뚱한 브랜치에서 실패

**환경**: macOS, zsh, git 2.x, Claude Code 병행 세션 다수 (허브 ai-study + 별도 docs/* 브랜치 작업 세션).

**증상**: `/wt-branch`로 작업 후 `worktree remove` + `git pull --ff-only origin main`을 순차 실행했는데 pull이 다음 에러로 실패:

```
fatal: Not possible to fast-forward, aborting.
```

Local main은 깨끗하고 origin/main이 앞선 상태라 fast-forward 성공해야 할 상황. 그런데도 거부.

**근본 원인**: 현재 *checkout된 브랜치가 main이 아님*. 다른 세션이 `docs/dev-setup-tips-log`나 `docs/claude-code-token-levers` 같은 feature 브랜치로 local을 checkout해둔 채 작업 중이었고, 허브 세션이 worktree를 정리한 직후에도 원본 working tree는 여전히 그 feature 브랜치에 있었다. `git pull --ff-only origin main`은 *현재 브랜치*로 `origin/main`을 merge 시도 → feature 브랜치와 main이 diverge해서 fast-forward 불가.

*왜 발견 어려운가*: `git status`가 "clean"을 출력해서 추가 변경이 없어 보이고, `git log`는 local의 최근 main 히스토리와 feature 브랜치가 공통 조상을 공유하므로 linear로 보인다. 즉 *"local main이 origin/main 뒤에 있는데 fast-forward가 왜 안 되지?"* 라는 잘못된 멘탈 모델로 디버깅을 시작하게 된다.

**수정**: pull 전에 반드시 현재 브랜치 확인 + 필요 시 `checkout main`.

```bash
# Pull 전 현재 브랜치 확인
rtk git branch --show-current
# main이 아니면 먼저 체크아웃
rtk git checkout main
# 그 다음 fast-forward pull
rtk git pull --ff-only origin main
```

**검증**:

```bash
# "clean — nothing to commit" 상태에서 다음 3 명령이 모두 성공해야 함
rtk git branch --show-current     # → main
rtk git pull --ff-only origin main # → "Already up to date." 또는 fast-forward OK
rtk git log --oneline -1           # → 가장 최근 squash merge commit
```

- `../../docs/solutions/workflow/2026-04-12-hub-worker-concurrent-session-safety.md` — 3층 방어 체크리스트의 다음 갱신 때 "pull 전 branch --show-current" 한 줄 추가 필요
- 이 패턴은 [Harness Journal 012 작성 중](./harness-journal-012-market-analyst-runtime-validation.mdx) 실제 발생 (한 번) → [Journal 014 작성 후 cleanup 시](./harness-journal-014-hub-auto-merge-inbound-tips.mdx) 재발 (두 번). 2회 재발이므로 박제.

**승격 조건**: 이 tips가 3회 이상 재발하거나 `/wt-branch` 커맨드에 자동 `checkout main` 단계를 붙일 가치가 생기면 Journal로 승격 후보. 현재는 이 카탈로그에 유지.

## 2026-04-13 — `/wt-branch` 워크트리에서 Next.js 16 Turbopack 프로덕션 빌드 실패

**환경**: macOS, Next.js 16 (Turbopack), `/wt-branch`로 만든 `/tmp/<project>-<branch>` 워크트리. 실제 발생 케이스: `mino-tarosaju` 프로젝트 `fix/mcapp-cleanup` 워크트리에서 `npm run build` 실행 중.

**증상**: `npm run build` 실행 시 Turbopack이 다음과 같이 실패:

```
- Execution of Project::get_all_endpoint_groups_with_app_route_filter failed
- Execution of find_package failed
- Symlink [project]/node_modules is invalid, it points out of the filesystem root
  at ignore-listed frames {
  type: 'TurbopackInternalError',
}
```

빌드 직후에도 비슷하게, `npm run build`가 env validation 단계에서 터짐:

```
Error: [env] 필수 NEXT_PUBLIC_* 환경 변수 누락:
  - NEXT_PUBLIC_SUPABASE_URL: Invalid input: expected string, received undefined
  - NEXT_PUBLIC_SUPABASE_ANON_KEY: Invalid input: expected string, received undefined
```

**근본 원인**: 두 가지가 겹쳐 발생한다 — 둘 다 `/wt-branch`로 만든 `/tmp/<project>-<branch>` 워크트리가 *원본 워킹트리와 다른 파일시스템 위치*에 있다는 점에서 비롯.

1. **Turbopack의 filesystem root 제약**: 워크트리를 빨리 띄우려고 `ln -s /Users/<me>/Develop/<project>/node_modules /tmp/<project>-<branch>/node_modules`로 심볼릭 링크를 걸면, Turbopack은 "심볼릭 링크가 프로젝트 루트 *밖*을 가리킨다"고 판단해 resolve 자체를 거부한다. Webpack은 이 케이스를 묵인하지만 Turbopack은 내부 가드(`Symlink points out of the filesystem root`)가 걸려 있다. 테스트(vitest)와 린트는 filesystem root 가드가 없어서 심볼릭으로도 돌아가므로 *빌드 단계에서만* 에러가 보인다. "테스트/린트 pass → 빌드 pass 아닐 것" 라는 안이한 가정이 깨지는 지점.

2. **`.env.local`이 gitignored**: `/wt-branch`는 origin/main에서 *트래킹된 파일만* 체크아웃하므로 `.env.local`이 복사되지 않는다. 이 프로젝트들은 `next.config.js` 또는 `src/lib/env.ts`에서 `NEXT_PUBLIC_SUPABASE_URL` 같은 변수를 *빌드 타임에 zod로 validation*하기 때문에, 값이 없으면 프로덕션 빌드가 즉사. Dev 서버(`npm run dev`)는 runtime validation이므로 페이지 진입 시에만 터지고, 빌드는 통과하는 경우가 있어 "dev는 되는데 build는 안 되네?"로 헷갈릴 수 있다.

**왜 발견 어려운가**: 두 증상은 서로 *독립적*이라 "한 개 고치면 다음 에러가 나오는" 양파 디버깅이 된다. 심볼릭 링크를 먼저 고치면 env 에러가 나오고, env를 먼저 고치면 심볼릭 에러가 나온다. 또 둘 다 *"워크트리에서만"* 재현되고 원본 워킹트리에서는 `node_modules`가 진짜고 `.env.local`도 있으므로 로컬 빌드가 잘 되어서 문제를 예측하기 어렵다. Turbopack 에러 메시지도 내부 스택 트레이스만 뿌리고 사용자 조언(`filesystem root` 용어)이 없어서 구글링으로 찾기 어렵다.

**수정**: 워크트리에서 프로덕션 빌드가 필요하면 *두 단계*를 모두 밟는다.

```bash
# 1. 심볼릭 링크 대신 실제 설치 (dedupe 덕에 보통 30~60초, lockfile이 있으면 빠름)
cd /tmp/<project>-<branch>
npm install

# 2. 원본 워킹트리의 .env.local을 복사 (gitignored라 워크트리엔 없음)
cp /Users/<me>/Develop/<project>/.env.local /tmp/<project>-<branch>/.env.local

# 3. 이제 빌드
npm run build
```

**검증**: `mino-tarosaju/fix/mcapp-cleanup` 워크트리에서 이 순서대로 실행 — 691/0 test pass + lint clean + Turbopack 프로덕션 빌드 exit 0 확인. 이전 심볼릭 링크 방식은 build에서 즉사했지만 `npm install` + `cp .env.local` 조합은 성공.

**주의**: `cp .env.local`한 파일은 워크트리 내부에 남아있지만 `.gitignore`에 `.env.local`이 이미 등록돼 있으므로 *실수로 커밋되지 않는다*. 그래도 `git status`에 untracked로 뜨지 않는지 한 번 확인할 것.

- [Harness Journal 014 — inbound tips](./harness-journal-014-hub-auto-merge-inbound-tips.mdx): 이 팁이 들어오는 경로
- 향후 `/wt-branch` 커맨드에 선택적 `--with-node-modules` / `--with-env` 플래그를 붙이는 것도 고려할 수 있음 — 현재는 명시적 2단계로 두는 편이 "왜 이 단계가 필요한가"의 맥락을 잊지 않게 함.

## 2026-04-13 — Next.js 16 RSC: Server Component에서 Client Component로 함수 prop 전달 시 프로덕션 500 + `digest` 해시만 노출

**환경**: Next.js 16.2.1 (Turbopack), `@vercel/analytics ^2.0.1`, App Router, Server Component로 작성된 루트 `app/layout.tsx`. 실제 발생 케이스: `mino-moneyflow` 프로덕션 (`mino-moneyflow.vercel.app`).

**증상**: 프로덕션 모든 public 페이지(`/`, `/login`, `/signup` 등) HTTP 500. 사용자 화면은 `global-error.tsx` fallback이 렌더링하는 "문제가 발생했습니다" 페이지 + `오류 코드: 2912116883`(`error.digest`) 노출. `/api/*` 엔드포인트는 정상 200. 로컬 `next build`는 성공, 로컬 `next start`에서만 재현.

```
# 로컬 next start 서버 stderr
⨯ Error: Functions cannot be passed directly to Client Components unless
  you explicitly expose it by marking it with "use server". Or maybe you
  meant to call this function rather than return it.
  {beforeSend: function beforeSend}
               ^^^^^^^^^^^^^^^^^^^
    at stringify (<anonymous>) {
  digest: '3379263931'
}
```

사용자가 보는 `digest`(`2912116883`)와 로컬 서버 로그의 `digest`(`3379263931`)는 *다르다* — 같은 에러 클래스라도 Next.js가 프로덕션 환경의 스택 트레이스 컨텍스트를 섞어 해싱하기 때문에 배포 환경마다 다른 숫자가 나온다. 즉 프로덕션 digest만 보면 로컬에서 같은 해시를 만들어낼 수 없고, *서버 stderr에서 실제 메시지를 뽑아야* 원인이 드러난다. 프로덕션 응답 HTML에는 digest만 노출되고 실제 에러 메시지는 *숨겨져 있다*(보안 기본값). Vercel Function Logs나 Sentry 없이는 원격 상태에서 원인을 알 수 없다.

**근본 원인**: 문제의 코드는 다음과 같다.

```tsx
// app/layout.tsx — Server Component
import { Analytics } from '@vercel/analytics/next'

export default async function RootLayout({ children }) {
  // ...
  return (
    <html>
      <body>
        {/* ... */}
        <Analytics
          beforeSend={(event) => {
            const url = event.url.replace(/[?&](key|token|secret)=[^&]*/gi, '')
            return { ...event, url }
          }}
        />
      </body>
    </html>
  )
}
```

Server Component가 렌더링될 때 Next.js는 children/props 트리를 JSON으로 직렬화해서 Client Component인 `<Analytics>`에 전달한다. 그런데 `beforeSend`는 클로저(함수)이므로 JSON.stringify 대상이 될 수 없다 — 함수는 직렬화 불가. React Server Components의 규칙은 "Server → Client 경계를 넘는 모든 값은 직렬화 가능해야 한다 (primitives, plain objects, Arrays, Promises, Server Actions만 허용)"이다. Next.js 16의 RSC 런타임이 이 규칙을 *엄격히* 강제하면서 렌더 시점에 `stringify`가 던지는 에러가 곧 SSR 500이 된다.

특히 악질적인 부분 3가지:

1. **`next build`는 성공한다.** 이 검증은 *렌더 타임*에 일어나는 런타임 체크라서 빌드 단계에서는 잡히지 않는다. `npm run build` exit 0을 보고 "배포해도 되겠네"로 넘기면 프로덕션에서 터진다.
2. **Dev 서버는 이를 콘솔 경고로만 표시**하고 페이지는 계속 렌더링해줄 수 있어서, 개발 중에는 정상처럼 보인다. 프로덕션 `next start`에서만 500으로 치명화된다.
3. **에러 메시지가 "beforeSend"를 명시적으로 지목**하는데도, 프로덕션에서는 그 메시지가 사용자/관리자에게 안 보이고 `digest` 숫자만 남는다. Vercel Function Logs 없이 로컬에서 `next start`로 *재현*하지 않으면 근본 원인 도달 불가.

이전 `@vercel/analytics ^1.x` + Next.js 15 조합에서는 이 패턴이 *묵인*됐다 — 함수 prop이 직렬화 불가라는 건 같지만, 런타임 체크가 느슨해서 "Warn + 무시"로 동작했을 가능성이 높다. Next.js 16 / `@vercel/analytics ^2.x` 조합에서 체크가 throw로 승격되면서 기존 코드가 일제히 깨진다. 즉 *업그레이드 시 RSC 함수 prop 패턴을 전부 검사*해야 한다.

**수정**: 함수 prop을 사용하는 Client Component는 *그 자체를 또 하나의 Client Component로 래핑*한다. 함수가 Client 경계를 넘어가지 않게 *클라이언트 내부에서 정의*해야 한다.

```tsx
// src/components/AnalyticsWrapper.tsx — 신규 Client Component
'use client'

import { Analytics } from '@vercel/analytics/next'

export function AnalyticsWrapper() {
  return (
    <Analytics
      beforeSend={(event) => {
        // 이 함수는 클라이언트 번들에서만 정의/실행된다 — Server→Client 경계를 건너지 않음
        const url = event.url.replace(/[?&](key|token|secret)=[^&]*/gi, '')
        return { ...event, url }
      }}
    />
  )
}
```

```tsx
// app/layout.tsx — Server Component (기존 import 교체)
import { AnalyticsWrapper } from '@/components/AnalyticsWrapper'

export default async function RootLayout({ children }) {
  return (
    <html>
      <body>
        {/* ... */}
        <AnalyticsWrapper /> {/* 함수 prop 없음 — 직렬화 가능 */}
      </body>
    </html>
  )
}
```

**검증**:

```bash
# 로컬 재현 (수정 전) — 500 나와야 함
rm -rf .next && npm run build
NEXT_PUBLIC_SUPABASE_URL=https://placeholder.supabase.co \
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJ...placeholder... \
PORT=3458 node_modules/.bin/next start &
sleep 3 && curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3458/
# → 500 (서버 stderr에 beforeSend 에러 출력)

# 수정 후 — 200 나와야 함 (AnalyticsWrapper 적용)
rm -rf .next && npm run build
PORT=3459 node_modules/.bin/next start &
sleep 3 && curl -s -o /dev/null -w "%{http_code}\n" http://localhost:3459/
# → 200

# 프로덕션 검증
for path in / /login /signup /api/health; do
  curl -s -o /dev/null -w "$path = %{http_code}\n" "https://<your-app>.vercel.app$path"
done
# → 200, 200, 200, 200 (또는 /dashboard = 307 auth redirect)
```

`mino-moneyflow` 실제 결과: 수정 전 `/` `/login` 500, 수정 후 둘 다 200 + 208KB 정상 응답(`__next_error__`·`digest` 마커 없음).

**일반화 — 함수 prop을 쓰고 있을 가능성이 높은 Client Component 체크리스트**:

이 에러가 나올 때 한 파일만 고쳐서 끝내지 말고 레포 전역 검사:

```bash
# 1) Server Component(layout.tsx, page.tsx 등)에서 import한 Client Component에
#    함수 prop(=>)이 들어가는 패턴 찾기
rg -n "on[A-Z]\w*=\{|beforeSend=\{|serialize=\{|transform=\{|render=\{" \
   --glob '!**/node_modules/**' \
   src/app src/layouts 2>/dev/null

# 2) '라는 이름의 함수 prop'이 보편적으로 함정인 라이브러리
#    - @vercel/analytics: beforeSend
#    - 차트 라이브러리: formatter, tooltip.content, label.formatter
#    - 커스텀 provider: onError, onSettled, transform
#    - i18n provider: getMessageFallback, onError
```

## 2026-04-13 — `ai-review.yml`의 squash merge는 `deploy.yml`을 트리거하지 않는다 (GitHub의 GITHUB_TOKEN 정책)

**환경**: 모든 워커 프로젝트에서 사용 중인 [Harness Journal 002 inline Test Gate](./harness-journal-002-inline-test-gate.mdx) 기반 `ai-review.yml` + 별도 `deploy.yml` 워크플로우 조합. 실제 발생: `mino-moneyflow`에서 PR #113 ~ #117을 자동 머지하는 동안 `deploy.yml`이 한 번도 트리거되지 않아 프로덕션이 *수 시간 동안 stale 상태*로 남음.

**증상**: main 브랜치에 squash merge는 성공적으로 적용되는데 (GitHub UI에서 "Merged" 표시 + `git log origin/main`에 새 커밋 보임), `.github/workflows/deploy.yml`의 `on: push: branches: [main]` 트리거가 작동하지 않음. 최근 성공한 `deploy.yml` 실행이 *직접 main 푸시*였던 오래된 커밋(mino-moneyflow 케이스: 04:48 UTC `3682016`)에 머물러 있고, 이후 자동 PR 머지된 4개 커밋(#114 ~ #117)에 대해서는 deploy 워크플로우가 한 번도 뜨지 않음.

```bash
# 증상 재현 확인
gh run list --workflow "🚀 Deploy to Vercel" --limit 5 \
  --json headSha,createdAt,conclusion

# 결과 (mino-moneyflow, 2026-04-13):
# [
#   { createdAt: "2026-04-13T04:48:35Z", headSha: "3682016...", conclusion: "success" },  # 직접 main 푸시
#   { createdAt: "2026-04-13T04:37:33Z", headSha: "c2ef57a...", conclusion: "success" },  # 직접 main 푸시
#   ...
# ]
# → PR #114 (05:43), #115 (06:13), #116 (06:26), #117 (07:41) 4건은 deploy 트리거 없음

gh run list --limit 20 --json name,headBranch,event | grep -c "push.*main"
# → ai-review.yml만 돌았을 뿐 deploy.yml은 0회
```

**근본 원인**: GitHub의 [GITHUB_TOKEN 정책](https://docs.github.com/en/actions/writing-workflows/choosing-when-your-workflow-runs/triggering-a-workflow#triggering-a-workflow-from-a-workflow)은 명시적으로 다음과 같이 규정한다:

> *"When you use the repository's `GITHUB_TOKEN` to perform tasks, events triggered by the `GITHUB_TOKEN`, with the exception of `workflow_dispatch` and `repository_dispatch`, will not create a new workflow run."*

`ai-review.yml`은 PR을 squash merge한 뒤 `git push origin HEAD:main`을 `GITHUB_TOKEN`으로 실행한다. 이 push는 git 관점에서는 정상 push지만, GitHub Actions 관점에서는 *downstream 워크플로우(`deploy.yml`의 `on: push`)를 의도적으로 깨우지 않는다*. 이 정책의 의도는 *무한 워크플로우 루프 방지*다 — workflow A가 push → workflow B → workflow B의 push → workflow A → ... 를 막기 위해 `GITHUB_TOKEN`으로 만들어진 push는 `push` 이벤트를 발행하지 않는다.

이 정책은 *잠재적으로 알려진 사실*이지만 실제로 마주하기 전까지 인지되지 않는 경우가 대부분이다 — `ai-review.yml`은 머지를 성공시켜 main을 업데이트하지만 deploy를 *조용히* 건너뛰고, 개발자는 "main에 머지됐으니 배포됐겠지"라는 멘탈 모델로 움직인다. 로컬 `git log origin/main`은 최신 커밋을 보여주고, GitHub UI에는 "Merged" 배지가 달린다. 어디에도 *"배포가 안 됐다"는 신호*가 없다. 프로덕션에 직접 접속해서 새 코드가 서빙되고 있는지 확인하기 전까지는 문제가 보이지 않는다.

이 사건에서는 운 나쁜 타이밍이 겹쳤다: v0.9.37에서 들어간 `beforeSend` 함수 prop 버그(위 섹션)가 존재했지만, *실제로 프로덕션에 반영된 마지막 deploy는 그 버그를 포함한 뒤에*(`3682016`, 04:48 UTC) 이뤄졌다. 그 후 PR로 올라간 여러 fix/compound 커밋들은 deploy 없이 main에만 누적됐고, 프로덕션은 *버그 있는 버전에 고정*된 채 수 시간 동안 노출됐다. 사용자가 실제 페이지를 열고 나서야 500이 드러났다.

**수정** — 3가지 옵션, 가장 침습도 낮은 순:

```yaml
# 옵션 1 (권장): ai-review.yml 맨 마지막에 workflow_dispatch로 deploy 명시 호출
# 가장 작은 변경 — 파일 하나 추가 step만 수정
- name: Trigger deploy workflow (bypass GITHUB_TOKEN push event silence)

        repo: context.repo.repo,

```

```yaml

# 주의: workflow_run도 성공한 run만 감지. 또 workflow_run은 conclusion을 검사해야 함.
on:
  workflow_run:
```

```yaml

# GITHUB_TOKEN 제약을 완전히 우회하지만 PAT 관리(로테이션/권한 최소화)가 비용

```

각 옵션의 트레이드오프:

- **옵션 1**: 가장 명시적. `ai-review.yml` 쪽에 "deploy를 일부러 깨운다"는 의도가 코드로 남아서 다음 사람이 읽을 때 체인을 추적하기 쉽다. 실패 모드: `workflow_dispatch` 호출 자체가 실패하면 무음 drop.
- **옵션 2**: `deploy.yml` 쪽만 고치면 되므로 여러 프로젝트에 이식할 때 `ai-review.yml`의 유사도를 깨지 않는다. 단점: `workflow_run`은 *독립 실행*이라 `ai-review.yml` 컨텍스트(PR 번호 등)를 기본으로 못 받음. 필요하면 `github.event.workflow_run.head_sha` 등으로 다시 lookup 필요.
- **옵션 3**: 근본 해결이지만 PAT 관리 오버헤드가 붙고, PAT 만료 시 *조용히* 전체 체인이 깨진다는 새 실패 모드가 생긴다.

## 2026-04-14 — `grep -c "error TS"`가 Next.js Turbopack 빌드 에러를 못 잡음

**환경**: Next.js 16 + Turbopack + Claude Code 서브에이전트 위임 워크플로우

**증상**: `npm run build` 후 `grep -c "error TS"` 결과 `0` → 빌드 통과로 판단 → 커밋/푸시 → Vercel 빌드 실패. Supabase `PromiseLike` 타입의 `.then().catch()` 체이닝 에러가 있었는데 로컬에서 못 잡음.

**근본 원인**: Next.js Turbopack 빌드는 TypeScript 에러를 `"error TS2345:"` 포맷이 아닌 `"Type error: ..."` 포맷으로 출력한다. `grep -c "error TS"`는 이 포맷을 매칭하지 못한다. 두 번째로, 에이전트에 코드 수정을 위임한 후 빌드 검증을 생략하고 에이전트의 "성공" 보고만 믿고 커밋한 것이 직접적 원인.

**수정**:
```bash
# AS-IS (틀림 — Turbopack "Type error:" 못 잡음)
npm run build 2>&1 | grep -c "error TS"

# TO-BE (맞음 — 종료 코드로 판단)
npm run build
# $? == 0 이면 통과, 아니면 실패

# 또는 grep 패턴 확장
npm run build 2>&1 | grep -E "error TS|Type error|Failed to"
```

핵심: **빌드 성공 판단은 grep 패턴이 아닌 종료 코드(`$?`)로 해야 한다.** grep 패턴은 프레임워크/번들러마다 출력 포맷이 달라서 false negative가 발생한다. 그리고 **서브에이전트가 수정한 코드도 반드시 직접 `vitest run` + `npm run build` 를 실행해서 검증**해야 한다. 에이전트는 타입 체크를 실행하지 않는다.

**검증**: `cron-helpers.ts`의 `.then().catch()` → `try/catch await` 패턴으로 수정 후 `npm run build` 종료 코드 0 확인. Vercel 배포 성공.

## 2026-04-14 — Zod `datetime()` vs Supabase 타임스탬프 포맷 불일치

**발견 프로젝트**: moneyflow

**증상**: Supabase에서 읽어온 데이터가 Zod 검증에서 실패 → API 500 에러. 로그: `"Expected string matching ISO datetime, received ..."`

**원인**: Supabase는 타임스탬프를 `2026-04-14T12:30:00.123456+00:00` 포맷으로 반환한다. 하지만 Zod의 `.datetime()`은 **`Z` 접미사만 허용** (`2026-04-14T12:30:00.123Z`). `+00:00` 오프셋이나 마이크로초 6자리가 포함되면 검증 실패.

**수정**:

```typescript
// Before (깨짐)
created_at: z.string().datetime()

// After (안전)
created_at: z.string()
```

타임스탬프 형식 검증이 필요하면 `.datetime()` 대신 `.regex()` 사용:

```typescript
created_at: z.string().regex(/^\d{4}-\d{2}-\d{2}T/)
```

**추가 발견**: 같은 맥락에서 `currency` 필드가 DB에서 `null` 가능한데 Zod 스키마에 `.nullable()` 누락 → 추가 500 에러. DB 컬럼의 nullable 여부와 Zod 스키마를 반드시 대조할 것.

**교훈**: Zod 스키마를 DB 응답에 적용할 때, **Supabase/PostgreSQL이 반환하는 실제 포맷을 `console.log`로 먼저 확인** 후 스키마 작성. 특히 `datetime()`, `uuid()`, `email()` 같은 strict validator는 DB 반환값과 미묘하게 다를 수 있다.

## 2026-04-14 — withCronTracking: Cron 실행 이력 자동 추적 래퍼

**발견 프로젝트**: moneyflow

**문제**: 11개 cron route가 각각 인증 로직을 복붙하고, 실행 성공/실패를 로깅만 하고 있었음. "어제 cron이 돌았는지?" 확인하려면 Vercel 로그를 뒤져야 했음.

**해결**: `withCronTracking(jobName, request, handler)` 래퍼 도입.

```typescript
// Before — 각 route마다 인증 + try/catch 반복
export async function POST(request: NextRequest) {
  const auth = request.headers.get('authorization')
  if (auth !== 'Bearer ' + process.env.CRON_SECRET) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }
  try {
    const result = await doWork()
    return NextResponse.json({ ok: true, ...result })
  } catch (e) {
    return NextResponse.json({ error: e.message }, { status: 500 })
  }
}

// After — 래퍼가 인증 + 이력 + 에러를 전부 처리
export async function POST(request: NextRequest) {
  return withCronTracking('alerts-check', request, async () => {
    const checked = await processAlerts()
    return { success: true, metadata: { checked } }
  })
}
```

**래퍼가 자동으로 하는 것**:
1. `CRON_SECRET` 인증 (safeCompare로 타이밍 공격 방지)
2. `cron_execution_history` 테이블에 실행 시작 기록
3. handler 완료 후 성공/실패 + `duration_ms` + metadata 기록
4. 에러 시 `error_message` 저장 (1000자 제한)

## 2026-04-15 — React #418 Hydration Mismatch: Date + localStorage 초기값 함정

**발견 프로젝트**: moneyflow

**증상**: 프로덕션에서 React Warning #418 "Text content does not match server-rendered HTML". 대시보드 날짜 표시가 깜빡이거나 1일 차이.

**원인 2가지**:

1. `useState(new Date())` — 서버(UTC)와 클라이언트(KST)에서 `new Date()`가 다른 값 반환. 서버 렌더 시점과 클라이언트 hydration 시점의 시간도 다름.

2. `useState(localStorage.getItem('key'))` — SSR에서 `localStorage`는 존재하지 않아 `null` 반환, 클라이언트에서는 저장된 값 반환. `false` vs `true` 불일치.

**수정**:

```typescript
// Before (hydration mismatch)
const [date, setDate] = useState(new Date())
const [saved, setSaved] = useState(localStorage.getItem('pref'))

// After (SSR-safe)
const [date, setDate] = useState<Date | null>(null)
const [saved, setSaved] = useState<string | null>(null)

useEffect(() => {
  setDate(new Date())
  setSaved(localStorage.getItem('pref'))
}, [])
```

**규칙**: 서버/클라이언트에서 값이 다를 수 있는 초기값은 **항상 `null`로 시작** → `useEffect`에서 실제 값 설정. `new Date()`, `localStorage`, `window.innerWidth`, `navigator.language` 등 모두 해당.

## 2026-04-15 — Zod transitive dependency 함정

**발견 프로젝트**: moneyflow

**증상**: 로컬에서는 동작하지만 clean install (`npm ci`)에서 `Cannot find module 'zod'` 에러.

**원인**: `schemas.ts`에서 `import { z } from 'zod'`를 사용하지만, `package.json`에 zod가 직접 의존성으로 선언되지 않음. 다른 패키지(예: `@supabase/supabase-js`)가 zod를 transitive로 가져오고 있어서 우연히 동작.

**수정**: `npm install zod` — package.json에 직접 선언.

**교훈**: `import`하는 패키지는 **반드시 package.json에 직접 선언**. transitive dependency에 의존하면:
- 상위 패키지가 zod를 제거하면 갑자기 빌드 실패
- npm 버전/lockfile 차이로 환경마다 동작이 달라짐
- CI에서만 실패하고 로컬에서 재현 불가
