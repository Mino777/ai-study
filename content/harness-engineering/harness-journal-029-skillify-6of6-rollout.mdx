## 왜 쓰는가

[Journal 027](/wiki/harness-engineering/harness-journal-027-skillify-resolver-backfill)이 Step 8을 9 repo에 dogfood한 기록, [Journal 028](/wiki/harness-engineering/harness-journal-028-skillify-100-playbook)이 허브 eval을 52→100%로 올린 playbook. 이 저널은 **028의 playbook을 같은 세션에 6 워커에 그대로 돌렸을 때** 실제로 재사용 가능한가를 실측으로 답한다.

*방법론 해설이 아니라 숫자와 순서.*

## 실측 요약

| repo | baseline | after | rules | cases | CI |
|---|---:|---:|---:|---:|:---:|
| mino-moneyflow | 54.5% | 100% | 16 | 11 | ✅ |
| mino-tarosaju | (100%†) | 100% | 16 | 11 | ✅ |
| aidy-architect | 81.8% | 100% | 23 | 11 | ✅ |
| aidy-server | 36.4% | 100% | 14 | 11 | ✅ |
| aidy-ios | 27.3% | 100% | 15 | 11 | ✅ |
| aidy-android | 27.3% | 100% | 14 | 11 | ✅ |

† tarosaju는 moneyflow와 routing 섹션이 거의 동일해 Lever 2를 선제 적용 후 측정. 별도 baseline 숫자 없음.

**총 소요**: 6 워커 end-to-end ≈ 45분 (worktree 생성 → 파일 복사 → 측정 → 보강 → commit → merge → push, 각 ~7분).

## Playbook 준수 순서 (Journal 028 그대로)

각 워커에 다음 *동일* 순서를 적용했다. 편차가 발생한 지점만 특기.

### Step 1 — worktree 생성

```bash
cd /Users/jominho/Develop/<worker>
git fetch origin
git worktree add -b skillify-resolver-eval /tmp/<worker>-skillify-resolver-eval origin/main
```

편차: 없음. 6/6 모두 main 깨끗한 상태여서 `/wt-branch` 규칙 그대로 적용.

### Step 2 — 파일 복사 (3개 + 1 script)

```bash
cp ai-study/scripts/resolver-eval.mjs <wt>/scripts/
cp mino-moneyflow/data/resolver-eval-cases.json <wt>/data/
cp ai-study/.github/workflows/weekly-resolver-eval.yml <wt>/.github/workflows/
```

편차:
- aidy-server/ios/android는 `.github/workflows/` 디렉터리 자체가 없어 `mkdir -p` 선행 필요.
- aidy-server는 `scripts/`도 없었음 (기존 스크립트가 루트에 산재). 같이 `mkdir -p`.

### Step 3 — package.json script 추가

```json
"eval:resolver": "node scripts/resolver-eval.mjs"
```

편차:
- **3개 aidy 워커에서 생략**. 이유: aidy-server(Python)·aidy-ios(Swift)·aidy-android(Kotlin)는 package.json 자체가 없음. resolver-eval.mjs가 zero-deps(`node:fs`/`node:path`만 사용)라 로컬은 `node scripts/resolver-eval.mjs` 직접 호출로 충분. CI는 `setup-node@v4` 스텝이 node 공급. **이식 단위 하나가 통째로 비어도 Playbook이 무너지지 않음** — 이것이 재사용성의 증거.

### Step 4 — baseline 측정

6 repo 분포:

```
moneyflow  54.5% (6/11)
tarosaju   (100%, Lever 2 선적용됨)
architect  81.8% (9/11)
aidy-server 36.4% (4/11)
aidy-ios    27.3% (3/11)
aidy-android 27.3% (3/11)
```

**관찰**: aidy-ios/android는 "crash reports" (영문) 때문에 "500 에러 난다 봐줘" intent가 전혀 매칭 안 됨. trigger 설명이 실제 말투와 원격.

### Step 5 — Lever 2 (한국어 키워드 보강)

6개 공통 rule에 동일한 한국어 말뭉치를 꼬리에 덧댔다(허브 CLAUDE.md 포맷 그대로).

```
- Product ideas, ..., 아이디어 브레인스토밍 빌드 가치 고민 → invoke office-hours
- Bugs, errors, ..., 버그 에러 문제 원인 크래시 디버깅 → invoke investigate
- Ship, deploy, ..., 배포 푸시 릴리스 진행 → invoke ship
- QA, test ..., QA 테스트 → invoke qa
- Code review, ..., 코드 리뷰 PR 리뷰 diff 리뷰 검토 → invoke review
- Update docs ..., 릴리스 후 문서 업데이트 문서화 머지 후 README 반영 → invoke document-release
```

편차:
- **aidy-server/ios/android는 autoceo + compound routing 자체가 누락**. golden case에 "자동으로 풀 스프린트 돌려봐"(autoceo)와 "스프린트 정리하자 회고 쓰자"(compound)가 있어 2개 rule 신규 추가 필요. 3줄 추가 후 한 번에 100%.
- **aidy-architect는 compound↔autoceo 토큰 충돌**. 기존: `멀티에이전트 풀 자동 스프린트 루프 → autoceo` + `WO 완료 후 복리형 지식 축적, 회고 → compound`. "스프린트 정리하자 회고 쓰자" → "스프린트" 토큰이 autoceo로 먼저 매칭(score 0.5 > compound 0.408). 수정: autoceo에서 "스프린트" 제거(“풀 자동 개발 루프 자동화”), compound에 "스프린트 정리 컴파운드" 덧댐. 동시에 wt-branch routing이 통째로 누락이던 것도 발견해서 추가. → 23 rules로 증가, 100%.

**이 두 편차가 Journal 028 playbook의 "예상 발생 이슈" 리스트를 실시간으로 검증**했다. playbook이 예측한 이슈들이 정확히 재현됐다는 뜻.

### Step 6 — commit / merge / push

```bash
git add -A
git commit -m "skillify: resolver-eval 이식 (<before> → 100%)"
cd <parent> && git merge --ff-only skillify-resolver-eval
git worktree remove <wt>
git branch -D skillify-resolver-eval
git push origin main
```

편차:
- aidy-android push 첫 시도에서 pre-push hook이 `❌ BLOCK` 출력 후 재시도 시 성공(transient). 원인 미확인 — fail-closed 훅의 한계(false-positive 한 번). 재발 시 훅 self-test 스킬(NEXT.md Medium #5)에서 다룰 것.
- ai-review.yml 워크플로우는 *6 워커 모두 PR 없이 main 직접 push* 이므로 트리거 안 됨. Journal 003의 squash-merge 트랩이 발생하지 않는 경로.

## 통계 비교 — 허브 vs 워커 평균

| 항목 | 허브(ai-study) | 워커 평균 |
|---|---:|---:|
| rules 수 | 21 | 16.3 |
| cases 수 | 48 | 11 |
| baseline | 52% | 45.5% (5개 측정치 중앙값) |
| after | 100% | 100% |
| 튜닝 소요 | 세션 1 (Step 7 Day) | 7분/워커 × 6 = 45분 |

**해석**: playbook이 있을 때 per-worker 비용이 세션 수준에서 분(minute) 수준으로 내려온다. Journal 028의 *시간* 대비 *재사용 회수*가 곧 playbook의 ROI.

## 골든 케이스 재사용 결정

11 케이스 템플릿(moneyflow)을 6/6 전원이 재사용. workspace-특화 케이스 없이도 100% 달성. 이는 *워커별 스킬 셋이 핵심 6개(office-hours / investigate / ship / qa / review / document-release) + 자동화 루프 2개(autoceo / compound) + 분기 1개(wt-branch)로 수렴*한다는 의미. 특화 스킬(dispatch, gate-1, ios-test, tca-check, compose-check, api-test)은 본 golden set에서 **평가하지 않음** — 다음 스프린트 과제.

Journal 028이 경고한 "내 말투 편향" 리스크 그대로. 다음 스프린트 Low 큐 #8(워커 eval 난이도 상향)이 정확히 이 지점을 향한다.
