> 이 엔트리는 Blake Crosley의 [Build Custom Skills](https://blakecrosley.com/blog/building-custom-skills)을 정독하고 핵심을 추출한 것이다.

### 왜 중요한가: 반복적 컨텍스트 주입의 자동화

개발자는 세션마다 동일한 보안 체크리스트나 코딩 스타일 가이드 같은 '도메인 지식'을 AI에 반복적으로 설명한다. Blake Crosley는 이 문제를 해결하기 위해 Claude Code의 'Skill' 기능을 소개한다. Skill은 사용자가 매번 명시적으로 호출하지 않아도, 대화의 맥락을 AI가 스스로 이해하고 관련 지식을 자동으로 활성화하는 **모델 호출 확장(Model-Invoked Extension)**이다.

이는 단순한 프롬프트 저장을 넘어, 특정 작업(코드 리뷰, 보안 감사 등)에 필요한 전문 지식을 AI 에이전트의 '기본 소양'으로 내재화하는 접근 방식이다. 결과적으로 생산성이 극대화되고, 팀 전체가 일관된 표준을 손쉽게 적용할 수 있게 된다.

### 핵심 패턴: LLM 추론 기반의 자동 활성화 및 선택적 적용

Claude Skill의 핵심은 키워드 매칭이 아닌, **LLM의 의미론적 추론(Semantic Reasoning)**을 통해 활성화된다는 점이다. 이는 Skill의 `description` 필드를 어떻게 작성하는지가 성공의 관건임을 의미한다.

#### 패턴 1: `description`이 활성화의 전부다

Skill의 YAML frontmatter에 있는 `description` 필드는 Claude가 "이 Skill을 언제 사용해야 하는가?"를 판단하는 유일한 근거다. Crosley는 너무 광범위한 `description`의 위험성을 경고한다. 예를 들어, `git-rebase-helper` Skill의 `description`을 "git 관련 프롬프트 처리"와 같이 넓게 잡으면, `git status` 같은 무관한 명령어에도 활성화되어 컨텍스트를 오염시키고 예산을 낭비할 수 있다.

- **Bad:** "git 관련 작업을 돕습니다."
- **Good:** "대화형 rebase 세션을 단계별로 안내하고, 충돌 해결 전략을 제안하며, rebase 완료 후 정리 명령어를 추천합니다. `git rebase` 명령어 사용 시에만 활성화하세요."

#### 패턴 2: Skill, Slash Command, Subagent의 명확한 구분

모든 반복 작업을 Skill로 만들어서는 안 된다. Crosley는 명확한 의사결정 프레임워크를 제시한다.

| 상황 | 해결책 | 이유 |
| :--- | :--- | :--- |
| 세션마다 동일한 **지식/체크리스트**를 주입 | **Skill** | 자동 활성화되는 도메인 전문 지식 |
| 명시적으로 특정 **액션**을 실행 | **Slash Command** | 사용자가 직접 호출하는 예측 가능한 트리거 |
| 주 대화와 **격리된 분석**이 필요 | **Subagent** | 컨텍스트 오염을 막는 별도의 작업 공간 |

"Claude가 알아서 적용해야 하는가, 아니면 내가 직접 실행 시점을 정해야 하는가?"라는 질문이 둘을 구분하는 핵심 기준이다.

#### 패턴 3: `SKILL.md`와 보조 파일로 전문 지식 구조화

Skill은 핵심 로직을 담은 `SKILL.md`와 상세 자료를 담은 보조 파일들로 구성할 수 있다.

```bash
~/.claude/skills/code-reviewer/
├── SKILL.md                  # Required: Frontmatter + 핵심 지침
├── SECURITY_PATTERNS.md      # Referenced: 상세한 취약점 패턴
└── PERFORMANCE_CHECKLIST.md  # Referenced: 쿼리 최적화 가이드
```

`SKILL.md` 본문에서는 `[SECURITY_PATTERNS.md](SECURITY_PATTERNS.md)`와 같이 상대 경로로 파일을 참조할 수 있다. Claude는 Skill 활성화 시 필요에 따라 해당 파일을 읽어 컨텍스트에 포함시킨다. Crosley는 Claude 공식 문서를 인용하며, `SKILL.md` 파일을 500줄 미만으로 유지하여 컨텍스트 주입 오버헤드를 줄이고, 상세 내용은 보조 파일로 분리할 것을 권장한다.

#### Skill 활성화 및 컨텍스트 관리 흐름

```mermaid
flowchart TD
    A[User Prompt: "이 코드 좀 리뷰해줘"] --> B(Claude: Prompt 분석)
    B --> C{Skill 저장소 스캔}
    C --> D["`description` 필드와 의미론적 일치도 검사"]
    D -- 일치 --> E{Skill: code-reviewer 활성화}
    E --> F["Inject: SKILL.md 컨텍스트 주입"]
    F --> G["Inject: SECURITY_PATTERNS.md 참조 주입"]
    G --> H["Response: 보안 체크리스트 기반 리뷰 생성"]
    D -- 불일치 --> I["Response: 일반적인 응답 생성"]
```

### 실전 적용: `ai-study` 위키 포맷팅 Skill 구축

`ai-study` 프로젝트는 위키 엔트리의 구조(왜/핵심패턴/실전적용), Mermaid 규칙, 출처 명시 등 고유한 포맷팅 규칙을 가지고 있다. 새로운 기여자가 이 규칙을 매번 기억하기는 어렵다. 이 문제를 해결하기 위해 Claude Code Skill을 적용할 수 있다.

#### 1. Skill 디렉토리 생성

```bash
mkdir -p .claude/skills/ai-study-wiki-formatter
```
프로젝트 레벨(`.claude/`)에 생성하여 `git`을 통해 팀원들과 공유한다.

#### 2. `SKILL.md` 작성

YAML frontmatter의 `description`을 통해 활성화 조건을 명확히 정의한다. `allowed-tools`는 파일 읽기/검색으로 제한하여 안전성을 확보한다.

```yaml
---
name: ai-study-wiki-formatter
description: ai-study 위키 엔트리 초안을 작성하거나 기존 문서를 수정할 때 활성화됩니다. MDX 포맷, '왜-핵심-실전' 구조, Mermaid 다이어그램 규칙, 출처 명시 등 ai-study의 고유한 문서 작성 가이드라인 준수 여부를 검토하고 제안합니다.
allowed-tools: [Read, Grep, Glob]
---

# ai-study 위키 엔트리 포맷팅 가이드

문서 검토 시 아래 규칙을 반드시 확인하고 제안해 주세요.

## 1. 필수 구조: 왜/핵심/실전

모든 문서는 다음 3단 구조를 따라야 합니다.
- `### 왜 중요한가`: 이 기술/패턴이 해결하는 문제와 그 중요성을 설명합니다.
- `### 핵심 패턴`: 기술의 핵심 원리, 동작 방식, 주요 개념을 구조화하여 설명합니다.
- `### 실전 적용`: 실제 프로젝트(예: moneyflow, tarosaju)에 적용할 수 있는 구체적인 TypeScript/Swift 코드 예시를 포함합니다.

## 2. 품질 가드라인 준수
- **실측 기반**: 추측성 서술("~할 수 있을 것이다")을 지양하고, 실제 코드와 결과에 기반하여 작성되었는지 확인합니다.
- **프로젝트 연결**: 일반론이 아닌, `ai-study`, `moneyflow`, `tarosaju`, `aidy` 중 하나 이상의 프로젝트에 실제 적용 가능한 시나리오가 포함되었는지 확인합니다.
- **Mermaid 5대 함정**: 다이어그램이 [MERMAID_RULES.md](MERMAID_RULES.md)에 명시된 5가지 함정(괄호/특수문자 처리 등)을 준수하는지 확인합니다.
```

#### 3. 보조 파일 추가

`MERMAID_RULES.md`와 같은 상세 가이드는 별도 파일로 분리하여 관리한다.

```bash
.claude/skills/ai-study-wiki-formatter/
├── SKILL.md
└── MERMAID_RULES.md
```
