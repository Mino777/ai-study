## 이 가이드의 목적

이 문서는 **AI 에이전트가 읽고 바로 실행할 수 있는 하네스 셋업 가이드**다. 사람이 읽어도 되지만, 주 독자는 새 프로젝트에 투입된 Claude Code 세션이다.

`harness-template/` 폴더를 새 프로젝트에 복사하면 다음이 **즉시 동작**한다:

| 기능 | 동작 시점 | 효과 |
|------|-----------|------|
| 위험 명령어 차단 | rm -rf, force-push 시도 시 | 돌이킬 수 없는 실수 원천 차단 |
| 빌드 검증 | git commit 전 | 깨진 코드 커밋 방지 |
| 타입 에러 자동수정 | 파일 편집 후 | exit 2로 에이전트가 스스로 수정 |
| 시크릿 차단 | 코드 작성 시 | API 키 유출 방지 |
| 하네스 드리프트 감지 | 세션 시작 시 | 규칙 부패 조기 발견 |
| 스프린트 회고 | push 후 /compound | 매 작업의 교훈이 다음 작업의 입력 |

## 왜 이 하네스가 필요한가

### 하네스 없는 AI 코딩의 위험

```
에이전트에게 "이 버그 고쳐줘" 요청
  → 에이전트가 rm -rf로 폴더 삭제 (복구 불가)
  → 에이전트가 .env 파일을 읽고 API 키를 코드에 포함 (보안 사고)
  → 에이전트가 타입 에러 있는 코드를 커밋 (빌드 실패 배포)
  → 에이전트가 git push --force로 팀원 커밋 덮어씀 (작업 손실)
```

이 모든 것이 **실제로 발생한 사고**다. 하네스는 이런 사고를 **구조적으로 불가능**하게 만든다.

### 근거가 되는 4편의 연구

| 출처 | 핵심 원칙 | 템플릿에 적용된 형태 |
|------|-----------|---------------------|
| [Martin Fowler](/wiki/harness-engineering/martin-fowler-harness-engineering-framework) | Computational Sensor — 매 변경마다 결정론적 검증 | build-gate, type-check Hook |
| [Claude Code arxiv](/wiki/harness-engineering/claude-code-architecture-arxiv-deep-dive) | Deny-First — 위험한 도구는 시야에서 제거 | settings.json deny 7개 |
| [Anthropic 3-Agent](/wiki/agents/anthropic-three-agent-harness-pattern) | 생성과 평가 분리 + Few-shot calibration | /review 커맨드 |
| [OpenAI Codex](/wiki/harness-engineering/openai-codex-harness-engineering-agent-first) | Garbage Collection — 지속적 드리프트 스캔 | gc-drift.sh Hook |

## 셋업 방법

### 방법 1: setup.sh (권장)

```bash
bash /path/to/harness-template/setup.sh /path/to/new-project
```

이 한 줄이면:
1. `.claude/` 디렉토리 복사 (기존 파일 덮어쓰기 안 함)
2. CLAUDE.md 복사 (이미 있으면 건너뜀)
3. harness-fitness 테스트 복사
4. 훅 실행 권한 설정
5. docs/retros, docs/solutions 디렉토리 생성

### 방법 2: 수동 복사

```bash
cp -r harness-template/.claude /path/to/new-project/
cp harness-template/CLAUDE.md /path/to/new-project/
cp -r harness-template/scripts /path/to/new-project/
chmod +x /path/to/new-project/.claude/hooks/*.sh
```

### 셋업 후 필수 커스터마이징 (3가지)

**1. CLAUDE.md 변수 치환**

```markdown
# {{PROJECT_NAME}} → 실제 프로젝트명
## Tech Stack
- **Framework**: {{FRAMEWORK}} → Next.js 16 / Spring Boot / etc.
- **Language**: {{LANGUAGE}} → TypeScript / Kotlin / Swift
- **Deploy**: {{DEPLOY_TARGET}} → Vercel / AWS / etc.

## Key Commands
- `{{DEV_CMD}}` → npm run dev / ./gradlew bootRun
- `{{BUILD_CMD}}` → npm run build / ./gradlew build
- `{{TEST_CMD}}` → npm test / ./gradlew test
```

**2. secret-guard.sh 금지어 추가**

```bash
FORBIDDEN=(
  # 공통 (이미 포함)
  'sk-[a-zA-Z0-9]{20,}'         # OpenAI
  'sk-ant-[a-zA-Z0-9]{20,}'     # Anthropic
  'ghp_[a-zA-Z0-9]{36}'         # GitHub
  'AKIA[0-9A-Z]{16}'            # AWS
  'AIza[0-9A-Za-z_-]{35}'       # Google

  # 프로젝트별 추가 ← 여기에
  'MyCompanyName'
  'internal-codename'
)
```

**3. 프로젝트 고유 rules 추가 (선택)**

프로젝트에 특화된 규칙이 있으면 `.claude/rules/`에 추가:

```markdown
<!-- .claude/rules/my-project.md -->
---
description: 프로젝트 고유 규칙
globs: src/**/*.ts
---
# 규칙 내용
```

`globs` 패턴에 매칭되는 파일을 편집할 때만 자동 로딩되므로 **평소에는 토큰 0 소비**.

## 각 구성 요소 상세

### settings.json — 보안 + 자동화의 뼈대

```
deny 규칙 (7개):
├── Read(**/.env*)              ← 시크릿 파일 읽기 차단
├── Read(**/secrets*)           ← secrets 디렉토리 차단
├── Write(**/.env*)             ← 시크릿 파일 쓰기 차단
├── Bash(rm -rf *)             ← 재귀 삭제 차단
├── Bash(git push --force*)    ← 강제 푸시 차단
├── Bash(git reset --hard*)    ← 하드 리셋 차단
└── Bash(DROP TABLE*)          ← DB 테이블 삭제 차단

hooks (4개):
├── SessionStart: gc-drift.sh   ← 세션 시작 시 하네스 건강 체크
├── PreToolUse(Bash): build-gate ← git commit 전 빌드 검증
├── PreToolUse(Edit|Write): secret-guard.sh ← 시크릿 패턴 차단
├── PostToolUse(Edit|Write): type-check    ← 타입체크 자동수정
└── PostToolUse(Bash): compound reminder   ← push 후 /compound 안내
```

**설계 원칙: Deny-First with Human Escalation**

deny 규칙은 항상 allow보다 우선. 인식 못하는 위험 액션은 사용자에게 확인 요청. Claude Code arxiv 논문에서 확인된 내부 설계와 동일한 원칙.

### gc-drift.sh — 하네스가 썩지 않게

OpenAI Codex 팀이 사용하는 "Garbage Collection" 패턴의 경량 버전.

```
세션 시작 시 4가지 체크:
1. CLAUDE.md 줄 수 > 250? → 경고 (Skills로 분리 필요)
2. settings.json에 deny 있나? → 없으면 보안 경고
3. hooks/ 에 .sh 파일 있나? → 없으면 경고
4. .claude/ 에 빈 .md 파일? → 있으면 정리 권장
```

**모두 경고만** — 차단하지 않음. Martin Fowler의 "Computational Sensor"로 드리프트를 **조기 발견**하는 것이 목적.

### secret-guard.sh — 시크릿 유출 원천 차단

5대 클라우드 서비스의 API 키 패턴을 정규식으로 감지:

| 패턴 | 서비스 |
|------|--------|
| `sk-[a-zA-Z0-9]{20,}` | OpenAI |
| `sk-ant-[a-zA-Z0-9]{20,}` | Anthropic |
| `ghp_[a-zA-Z0-9]{36}` | GitHub |
| `AKIA[0-9A-Z]{16}` | AWS |
| `AIza[0-9A-Za-z_-]{35}` | Google |

프로젝트별 금지어(회사명, 코드네임)도 `FORBIDDEN` 배열에 추가 가능.

**화이트리스트**: `.claude/` 내부 파일은 검사 제외 — 훅 스크립트 자체에 정규식 패턴이 들어있으므로 자기 차단 역설 방지.

### PostToolUse 타입체크 — 자동수정 루프의 핵심

```
에이전트가 파일 편집
  → PostToolUse Hook 발동
    → tsconfig.json 존재 확인
      → 있으면: npx tsc --noEmit
        → 성공: silent (출력 없음, 컨텍스트 오염 방지)
        → 실패: 에러 10줄만 표시 + exit 2
          → 에이전트가 에러를 읽고 자동 수정
            → 다시 Hook 발동 → 성공할 때까지 반복
      → 없으면: skip (JS 프로젝트에서도 동작)
```

**exit code 2**가 핵심. Claude Code에서 exit 2는 "에이전트에게 다시 시도하라"는 시그널. 사람 개입 없이 타입 에러가 자동 수정되는 루프.

**"성공은 silent"**: OpenAI Codex 팀의 교훈 — 4,000줄 성공 테스트 출력이 컨텍스트를 오염시키면 에이전트가 본 작업을 잃어버린다. 실패만 표면화.

### /review — Few-shot Calibration 리뷰

Anthropic 3-Agent 패턴에서 배운 핵심: **평가에 점수 기준 예시를 제공하면 일관성이 올라간다.**

```
PASS 예시: 모든 항목 구현, 보안 이슈 0, 테스트 존재
WARNING 예시: 사소한 누락 1-2건, any 타입 1-2건
FAIL 예시: XSS 취약점, 빌드 실패, 하드코딩 시크릿
```

에이전트가 이 예시를 보고 자기 판정의 "기준점"을 잡을 수 있다.

### /compound — 복리 성장 루프

```
push 완료
  → /compound 실행
    → Phase 1: CHANGELOG 업데이트
    → Phase 2: 회고 작성 (잘된 것 / 아쉬운 것 / 다음에 적용)
    → Phase 3: 솔루션 박제 (재발 방지 문서)
    → Phase 4: 메모리 업데이트
    → Phase 5: 커밋
```

이 루프가 돌 때마다 **하네스 자체가 개선**된다. 회고에서 발견된 패턴이 Hook이나 Rule로 승격되고, 다음 세션은 그 개선된 하네스 위에서 작업한다.

## Hook 3가지 타입 — command만 쓰면 하네스의 30%만 쓰는 것

Hook은 **3가지 타입**이 있다. 대부분 command만 알지만 prompt/agent Hook이 더 강력한 경우가 있다.

| 타입 | 실행 | 토큰 비용 | 적합한 용도 |
|------|------|-----------|-------------|
| **command** | 셸 스크립트 | 0 | 포맷팅, 린트, 빌드 검증 |
| **prompt** | 시스템 프롬프트 주입 | 소량 | 특정 파일 편집 시 규칙 강화 |
| **agent** | 별도 AI가 평가 후 허용/차단 | 높음 | 의미적 안전성 검사 |

### Exit Code — 직관과 다르니 주의

| Exit Code | 의미 | 결과 |
|-----------|------|------|
| **0** | 성공 | 계속 진행 |
| **2** | **차단** | 도구 실행 중단 + stderr가 에이전트에게 에러 전달 |
| **1, 3+** | **비차단 경고** | 도구 실행 계속 (!), stderr는 verbose에서만 표시 |

> **가장 흔한 실수**: 보안 Hook에서 exit 1을 쓰면 위험한 명령이 **그대로 실행된다**. 반드시 **exit 2**를 써야 차단.

### Dispatcher 패턴

같은 이벤트에 여러 Hook을 순차 실행할 때 race condition 방지:

```bash
#!/bin/bash
INPUT=$(cat)
for hook in "$HOME/.claude/hooks/pre-tool-use.d/"*.sh; do
  [ -x "$hook" ] || continue
  echo "$INPUT" | "$hook"
  [ $? -eq 2 ] && exit 2  # 차단 전파
done
```

## Context 열화 곡선 — 30분마다 /compact

MS Research + Salesforce 연구 (15개 LLM, 200,000+ 시뮬레이션): 멀티턴 대화에서 평균 **39% 성능 하락**.

```
0분:   정밀한 멀티파일 편집 (전체 토큰)
30분:  아직 생산적 (~150K)
60분:  성능 저하 시작 (~100K)
90분:  심각한 저하 (~50K)
120분+: 에러 누적
```

**열화 원인은 토큰 한계가 아니라 턴 경계.** 전체 대화를 단일 프롬프트로 넣으면 95.1% 성능 유지.

| 세션 길이 | 전략 |
|-----------|------|
| < 60분 | 단일 세션 유지 |
| 60-90분 | `/compact` 후 계속 |
| > 90분 | Fresh Context — 새 세션 + Handoff 문서 |

### Session Handoff 5줄 템플릿

```markdown
**Status:** 완료된 것
**Files:** 변경된 파일
**Decision:** 내린 결정
**Blocked:** 막힌 것
**Next:** 다음 할 것
```

## Evidence Gate — "다 했어요"를 믿지 마라

에이전트가 "완료"라고 할 때 **증거를 요구**하는 패턴. Blake Crosley 데이터: 적용 후 false completion rate **35% → 4%**.

| 기준 | 요구 증거 |
|------|-----------|
| 패턴 준수 | 패턴명 + 존재하는 파일 경로 |
| 최단 해법 | 거부한 대안과 이유 |
| 엣지 케이스 | 구체 케이스 + 처리 방법 |
| 테스트 통과 | 테스트 출력 (0 failures) |
| 회귀 없음 | 확인한 파일/기능 목록 |

| 항목 | 오버헤드 | 이점 |
|------|----------|------|
| build-gate (PreToolUse) | commit마다 빌드 실행 (~30초) | 깨진 코드 커밋 0건 |
| type-check (PostToolUse) | Edit마다 tsc 실행 (~5초) | 타입 에러 자동 수정 |
| gc-drift (SessionStart) | 세션 시작 시 ~2초 | 하네스 드리프트 조기 발견 |
| secret-guard (PreToolUse) | Edit마다 grep (~100ms) | API 키 유출 0건 |

**총 세션당 추가 시간**: ~3초 (SessionStart) + Edit당 ~5초 (type-check)
**총 토큰 오버헤드**: ~2% (성공 시 silent, 실패 시만 출력)

### 하지 않기로 한 것

| 결정 | 이유 |
|------|------|
| **MCP 서버 미포함** | 프로젝트마다 필요한 도구가 다름. 범용 템플릿에 넣으면 불필요한 토큰 소비 |
| **Stop hook 미포함** | Claude Code API에서 안정적 지원 확인 안 됨. false completion 방지는 /review로 대체 |
| **CI 연동 미포함** | CI 환경은 프로젝트마다 다름 (GitHub Actions / GitLab CI / Jenkins) |
| **Language-specific linter 미포함** | TypeScript만 tsc 체크. Python/Go/Rust는 프로젝트에서 추가 |
| **복잡한 rules 언어 미채택** | 마크다운 + glob이면 충분. YAML/DSL은 오버엔지니어링 |

### 확장 포인트

프로젝트가 성장하면서 추가할 수 있는 것들:

```
Day 1 (이 템플릿):
  settings.json + hooks 4개 + rules 1개 + commands 3개

Week 2+ (필요할 때 추가):
  + 프로젝트 고유 rules (content-mdx.md, api-patterns.md 등)
  + 프로젝트 고유 commands (/deploy, /migrate 등)
  + MCP 서버 연결 (GitHub, Slack, DB 등)
  + 추가 hooks (lint, format, coverage 등)

Month 2+ (성숙기):
  + harness-fitness 테스트 확장
  + /compound Phase에 프로젝트 고유 메트릭 추가
  + 멀티 에이전트 워크플로우 (/projects-sync 등)
```

**핵심**: Day 1에 모든 것을 세팅하지 마라. 에이전트가 **실제로 실패하는 지점**을 관찰한 후 그 실패에 맞는 Hook/Rule을 추가.

## 기계적 검증: harness-fitness 테스트

Martin Fowler의 "Computational Sensor"를 하네스 자체에 적용. `npm test`에 포함되어 매 커밋 시 하네스 건강 상태를 자동 검증.

| 테스트 | 검증 내용 | 실패 시 의미 |
|--------|-----------|-------------|
| CLAUDE.md ≤ 250줄 | 모놀리식 매뉴얼 방지 | Skills/Rules로 분리 필요 |
| deny 규칙 ≥ 4개 | 최소 보안 보장 | 보안 규칙 추가 필요 |
| rm -rf deny 존재 | 재귀 삭제 차단 | 치명적 보안 갭 |
| hooks ≥ 2개 | 자동화 최소 보장 | 빌드 검증/시크릿 차단 누락 |
| rules 디렉토리 존재 | 스코프드 규칙 체계 | 규칙 체계 미수립 |
| commands ≥ 2개 | 워크플로우 코드화 | 반복 작업 수동 처리 중 |

## 다른 에이전트를 위한 셋업 체크리스트

이 섹션은 **새 프로젝트에 투입된 AI 에이전트가 하네스를 셋업할 때** 따라야 할 순서다.

```
□ Step 1: setup.sh 실행
  bash /path/to/harness-template/setup.sh $(pwd)

□ Step 2: CLAUDE.md 커스터마이징
  - {{PROJECT_NAME}} → 실제 프로젝트명
  - {{FRAMEWORK}} → 사용 중인 프레임워크
  - {{DEV_CMD}}, {{BUILD_CMD}}, {{TEST_CMD}} → 실제 명령어
  - 프로젝트 고유 규칙 추가

□ Step 3: secret-guard.sh 커스터마이징
  - FORBIDDEN 배열에 프로젝트별 금지어 추가
  - 회사명, 프로젝트 코드네임 등

□ Step 4: 검증
  - npx vitest run scripts/__tests__/harness-fitness.test.mjs
  - 6개 테스트 전부 PASS 확인

□ Step 5: 첫 커밋
  - git add .claude/ CLAUDE.md scripts/__tests__/harness-fitness.test.mjs
  - git commit -m "harness: 공통 하네스 템플릿 이식"

□ Step 6: 1주일 사용 후
  - 에이전트가 실패한 패턴 관찰
  - 해당 패턴에 맞는 Hook 또는 Rule 1개 추가
  - harness-fitness 테스트에 새 검증 추가
```
