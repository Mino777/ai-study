## Before — 솔루션은 박제했지만 가드는 없었다

`docs/solutions/mdx/` 에 Mermaid 관련 솔루션이 누적:

| # | 날짜 | 패턴 |
|---|------|------|
| 1 | 2026-04-09 | `<br>` self-closing 누락 → MDX 컴파일 에러 |
| 2 | 2026-04-11 | subgraph 이름 공백 → 파서 fail |
| 3 | 2026-04-16 | 노드 label 의 `<br/>` + 특수문자 따옴표 누락 (런타임-only) |
| 4 | 2026-04-16 | cylinder `[(label)]` 안의 `()` 중첩 → validator 오탐 + 파서 모호 |

3·4번째는 **같은 날** 연속 발생. validator 가 잡지 못하는 패턴이 27 노드 분량으로 한꺼번에 풀렸다 (Flow Map 시리즈 작성 중).

이때 두 솔루션 문서 모두 끝에 "validator 확장 후보" 섹션을 남겼지만, 실제로 구현되지 않은 채 *읽기만 하는 메모* 로 남아 있었다. 다음 세션이 같은 실수를 반복하면 또 솔루션 문서만 추가될 위험.

## After — Solution → Validator Warning 승격

### 결정 사항

`scripts/lib/mermaid-fix.mjs` 의 `fixAndValidateMermaid` 반환값에 **warnings** 필드 신설:

```js
return {
  fixed: ...,
  autoFixed: ...,
  errors,    // 빌드 실패 — 이미 있던 필드
  warnings,  // 빌드 통과하지만 ⚠️ 출력 — 신규
};
```

`detectUnquotedSpecialCharLabels(code)` 로 다음 패턴 검출:

```
SPECIAL_CHAR = /(<br\s*\/?>)|→/

일반 노드 [label]   — 따옴표 없으면 warning
rhombus {label}    — 따옴표 없으면 warning
cylinder [(label)] — 매치 안 함 (별도 솔루션으로 처리)
```

### auto-fix 가 아니라 warning-only 인 이유

**3 가지 위험 회피**:

1. **idempotency 깨짐 위험** — 정규식 자동 치환은 이미 처리된 입력에 다시 실행되면 누적 손상 (Bug 2 의 트라우마)
2. **shape 다양성** — cylinder `[(label)]`, rhombus `{label}`, circle `((label))` 등 정규식이 부적절하게 매칭되는 케이스 다수
3. **memory rule "자동 수정 도구 신뢰 X"** — 자기 검증 없이 silent 변형하는 도구는 다음 세션이 디버깅하기 가장 어려운 함정

→ 사람이 warning 메시지 보고 *수동 수정* 하는 게 가장 안전.

### 즉시 가치 입증

빌드 한 번 만에 **9건** 잠재 함정 발견:

```
⚠️ five-levers-of-harness-engineering.mdx Line ~26: "보고<br/>Observe"
⚠️ five-levers-of-harness-engineering.mdx Line ~27: "잡고<br/>Intercept"
⚠️ five-levers-of-harness-engineering.mdx Line ~28: "되살리기<br/>Recover"
⚠️ five-levers-of-harness-engineering.mdx Line ~210~213: 4건 (CLAUDE.md 60줄...)
⚠️ rag/vector-search-basics.mdx Line ~48: "ANN 인덱스<br/>HNSW"
```

이 9건은 *런타임에만* 드러나는 함정 — `Mermaid 에러` 메시지로 실제 사용자 화면이 깨지기 전에는 빌드가 통과한다. 솔루션 문서로만 박제했던 시기에는 이 9건이 *언제든 사용자 사고로 표면화* 될 수 있는 잠재 부채였다.

수정 후 빌드: warnings 0건.

### 회귀 테스트 — Bug 2 트라우마의 산물

`scripts/__tests__/validate-content.test.mjs` 에 **9 케이스** 추가 (총 16):

```
warnings — <br/> · → 따옴표 누락 탐지 (3·4번째 재발 패턴)
  ✓ 일반 노드 [label] 안에 <br/> 가 있고 따옴표 없으면 warning
  ✓ 일반 노드 [label] 안에 → 가 있고 따옴표 없으면 warning
  ✓ rhombus {label} 안에 <br/> 가 있고 따옴표 없으면 warning
  ✓ 이미 따옴표 있는 라벨은 warning 없음 (idempotency)
  ✓ 이미 따옴표 있는 rhombus 도 warning 없음
  ✓ 정상 라벨은 warning 없음
  ✓ cylinder [(label)] 은 일반 정규식 매치 안 함 (별도 솔루션)
  ✓ 여러 라인에 여러 패턴이 섞여 있으면 모두 잡음
  ✓ auto-fix 가 트리거되어도 warnings 는 같이 반환됨
  ✓ warning 함수도 idempotent — 5회 실행해도 warning 수 동일
```

**idempotency 검증 케이스가 별도로 들어간 이유**: 정규식 변경은 silent 손상 위험이 가장 큰 변경 유형. 미래의 누군가가 정규식을 확장할 때 *이 테스트가 가드 역할*. Bug 2 트라우마의 산물.

## 배운 것 / 다음 후보

### 이 사이클의 진짜 메시지

1. **Solution 문서 N=3+ 재발 = validator/훅으로 승격 신호** — 1~2번은 `docs/solutions` 에 기록으로 충분. 3+번은 사람 기억 대신 코드 가드 필요
2. **승격 시점에서 자동 발견되는 잠재 함정이 있다** — 이번처럼 9건 즉시 검출. solution 박제 시점에는 *기존 콘텐츠* 도 같은 패턴 가능성 점검해야 함
3. **warning-only 가 auto-fix 보다 안전한 케이스 존재** — shape 다양성 + idempotency 위험 + memory rule 모두 충족 시 warning-only 우선
4. **idempotency 테스트는 별도 케이스로 박제** — 정규식 확장 시 가장 흔한 회귀 유형. 새 정규식 추가 = 새 idempotency 케이스 추가

### 일반화

이 패턴은 mermaid 만의 이야기가 아니다:
- **JSX trap detector** (`detectJsxTraps`) 도 같은 패턴 — `{worker}` 같은 본문 중괄호 잡음. 빌드 실패는 안 시키지만 컴파일 에러 원인 표면화
- **앞으로의 후보**: AI 생성 MDX 의 반복 패턴(self-closing 누락 등)이 N=3+ 누적되면 validator 로 승격

## AI Agent Directive

### Trigger
- `docs/solutions/<category>/` 에 동일 패턴이 N=3+ 누적됐을 때
- AI 생성 콘텐츠에서 반복 패턴이 빌드/런타임 함정을 만들고 있을 때

### Prerequisites
- [Compound Engineering Philosophy](/wiki/harness-engineering/compound-engineering-philosophy)
- [Numeric Execution Evidence](/wiki/evaluation/numeric-execution-evidence) — warning 도 숫자 증거 원칙 적용

### Actionable Steps
1. **N=3 룰 적용** — 같은 docs/solutions 카테고리에 3+ 번 누적 시 validator/훅 승격 검토
2. **warning-only 우선** — auto-fix 는 idempotency · shape 다양성 · false positive 다 통과해야 채택
3. **승격 시점에 기존 콘텐츠 점검** — validator 추가 = 잠재 부채 발견. 빌드 한 번 돌려서 즉시 수정
4. **idempotency 케이스 별도 박제** — 정규식 확장 시 5회 실행해도 결과 동일한지 회귀 테스트
5. **솔루션 문서에 "validator 확장 후보" 섹션 명시** — 미래 승격 트리거가 자기 자신에 박혀 있어야 누락 안 됨

### Anti-patterns
- **솔루션 문서 박제만으로 종료** — 1~2번은 OK, 3+번은 시스템 가드 필요
- **검증 없는 auto-fix** — Bug 2 처럼 매 실행마다 손상 누적
- **승격 후 기존 콘텐츠 점검 누락** — 잠재 부채가 그대로 남아 미래 사용자 사고 표면화
