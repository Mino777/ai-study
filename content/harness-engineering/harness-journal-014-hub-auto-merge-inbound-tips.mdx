## 사이클 트리거

사용자의 요청:

> *"각 프로젝트에서 뭔가 개발하면서 생기는 꿀팁들을 너한테 PR로 계속 쏴주면 너가 하나의 엔트리로 관리하면서 계속 글로 히스토리를 남겨줬으면 좋겠어서 해봤어 ... 앞으로 이런 pr 오면 자동 머지 + 글 작성 해주면 돼~!"*

이미 [PR #15](https://github.com/Mino777/ai-study/pull/15)에서 `dev-setup-tips-log.mdx` 엔트리를 신설해놓고, 이제 그 로그에 *워커 세션들이 계속 기여할 수 있는 인프라*를 붙이라는 요청.

## 작업 스코프

한 PR에 3개를 묶었다:

1. **ai-review.yml 이식** (moneyflow → ai-study) — 박제 전이성 4회째.
2. **dev-setup-tips-log.mdx의 For AI Agents 섹션 보강** — 워커 → 허브 PR 제작 단계, 허브 처리 룰, PR 제목 컨벤션, 승격 판단 기준.
3. **이 Journal 014 박제** — 메타 사이클 자체 기록.

묶은 이유: 셋 다 "허브가 꿀팁 PR을 받는 인프라"라는 한 주제이고, 분리하면 *3개의 연속 수동 머지*가 필요해서 비효율.

## ai-review.yml 이식 — 바꾼 부분 정확히

원본 (moneyflow `ai-review.yml`)에서 바꾼 부분만 기록:

| 요소 | moneyflow | ai-study |
|---|---|---|
| Test Gate 명령 | `npx vitest run` + `npm run build` (with env placeholder) | `npm test` + `npm run build` (env 없이) |
| env vars | `NEXT_PUBLIC_SUPABASE_URL/ANON_KEY` placeholder | 없음 (ai-study는 build 시점에 Supabase env 불필요) |
| 주석 | Journal 002 | Journal 014 (첫 허브 이식) |

나머지 — Find-or-Create PR 로직, Rebase onto main, Comment on conflict, Squash Merge, Reset branch to main — **1:1 그대로**. 이식 사이클의 엄격성은 *원본이 검증된 구조를 그대로 유지*하는 것.

## Chicken-and-egg — 이번에도 수동 머지

`ai-review.yml`이 main에 *없는 상태*에서 이 PR이 만들어지므로 첫 PR은 수동 머지. [Journal 011의 tarosaju PR #6과 동일 구조](/wiki/harness-engineering/harness-journal-011-concurrent-session-safety).

단, tarosaju는 feature 브랜치에 ai-review.yml이 있는 *이 PR*을 봇이 먼저 돌려서 권한 부족 403을 맞은 뒤 re-run 으로 통과한 *예상 못한* 경로였던 반면, ai-study는 이미 `default_workflow_permissions: write` + `can_approve_pull_request_reviews: true` 상태라 바로 작동할 가능성이 높다. *dogfooding 여부*는 이 PR의 실제 작동으로 확인.

## ci.yml과 중복 우려

ai-study에는 이미 `ci.yml`이 있다 (PR/push → `npm test + npm run build`). ai-review.yml이 추가되면 Test Gate가 *중복*으로 돈다:

- `ci.yml` on `pull_request: main` → PR 생성/업데이트 시 돔
- `ai-review.yml` inline Test Gate → push 시 돔 (PR과 무관)

**결정**: 중복 유지. moneyflow도 `test-gate.yml` + `ai-review.yml` inline 둘 다 유지한다 (Journal 011). 중복 커버리지가 *우발적 회귀*를 한 번 더 잡아준다. 게다가 두 이벤트 트리거가 다르므로(`pull_request` vs `push`) 완전 동일한 실행이 아니다.

## 워커 → 허브 PR 흐름 — For AI Agents 섹션에 박제

`dev-setup-tips-log.mdx`의 For AI Agents 섹션에 *워커 세션이 PR을 제작하는 5단계 + 허브 세션의 처리 규칙*을 추가. 요점:

- 워커 세션은 **`tips/<branch-name>`** 브랜치로 push → `ai-review.yml`이 자동 PR 생성 + Test Gate + Squash Merge까지 일괄
- PR 제목 prefix **`tips:`** 로 식별
- 허브 세션의 기본 동작은 **no-op** (ai-review.yml이 전부 처리) — 허브는 *사후 검토자*로만 참여
- 예외: Test Gate 실패 → 허브가 수습, 재발 감지 → 별도 PR로 재발 노트 추가 (본문 보존), 3 프로젝트 누적 → Journal로 승격
