## 갭 (현시점)

[Journal 000 베이스라인](/wiki/harness-engineering/harness-journal-000-baseline)에서 발견된 **공통 갭 #8**:

> `.claude/skills/` 디렉터리 미활용 — 양쪽 모두 슬래시 커맨드 7개(`/compound`, `/autoceo`, `/wt-branch`, `/ingest`, `/cross-session-review`, `/curate-inbound`, `/projects-sync`)만. *작은 재사용 단위*인 skill 형식 미사용.

이 글은 양쪽 미적용 상태에서 *제안 + 도입 가이드*. Skill은 슬래시 커맨드보다 진입 비용이 낮은 *작은 능력 단위*인데, 양쪽 프로젝트 모두 *경로 의존성*으로 미사용 상태.

## 슬래시 커맨드 vs Skill — 본질적 차이

| 속성 | 슬래시 커맨드 | Skill |
|---|---|---|
| **호출 방식** | 사용자/AI가 *명시적*으로 호출 (`/compound`) | AI가 *작업 맥락*에 자동 매칭 |
| **단위 크기** | 큰 워크플로 (다단계, 여러 도구) | 작은 능력 (한 가지 작업) |
| **비유** | *동사* — "이 작업 해줘" | *조사할 수 있는 명사* — "이 능력이 있다" |
| **재사용** | 다른 슬래시 커맨드 안에서 호출 *어려움* | 다른 작업의 *부품*으로 자연스럽게 |
| **AI 의도성** | 사용자가 *의도*를 가지고 호출 | AI가 *맥락*에서 자동 발견 |
| **파일** | `.claude/commands/<name>.md` | `.claude/skills/<name>/SKILL.md` |
| **메타데이터** | 가벼움 (제목 + 본문) | 풍부함 (description, allowed-tools, examples) |

### 한 줄로 정리

> 슬래시 커맨드 = *사용자가 부르는 도구함*, Skill = *AI가 사용하는 손*

`/compound`처럼 *사용자가 의도해서 호출하는 것*은 슬래시 커맨드. *AI가 작업하다가 자동으로 가져다 쓰는 능력*은 Skill.

## 두 시스템의 적합한 작업 매핑

| 작업 종류 | 어디에 적합 | 이유 |
|---|---|---|
| 매 스프린트 후 회고 박제 (`/compound`) | 슬래시 커맨드 | 사용자가 의도, 큰 워크플로 |
| 새 작업 시 worktree 분기 (`/wt-branch`) | 슬래시 커맨드 | 사용자가 *명시적으로* 부름 |
| 외부 URL 학습 정리 (`/ingest`) | 슬래시 커맨드 | 사용자가 의도 + 다단계 |
| 자동 스프린트 (`/autoceo`) | 슬래시 커맨드 | 사용자가 부름, 매우 큰 워크플로 |
| **git status 깨끗 확인** | **Skill** | AI가 *모든 작업 시작 시* 자동 호출 |
| **TypeScript 에러 분류** | **Skill** | 빌드 fail 시 AI가 자동 호출 |
| **PR 코멘트 작성** | **Skill** | PR 작업 중 AI가 자동 |
| **rate limit 헤더 파싱** | **Skill** | API 응답 처리 중 AI가 자동 |
| **MDX 문법 검증** | **Skill** | 콘텐츠 작성 중 AI가 자동 |

좋은 휴리스틱: *사용자가 매번 의도해서 호출하는가, AI가 작업 맥락에서 자연스럽게 가져다 쓰는가*. 후자라면 Skill.

## 양쪽 프로젝트 도입 가이드 — 첫 한 가지

### Skill: `git-status-clean-check`

**왜 이게 첫 후보인가**:
- 모든 작업의 *맨 처음 단계*에 필요
- *여러 슬래시 커맨드*에서 재사용 (`/compound`, `/wt-branch`, 새 작업 시작 등)
- 단순 동작 (한 가지 작업)
- 양쪽 프로젝트에 *동일하게* 적용 가능

### 디렉터리 + 파일

```
.claude/skills/git-status-clean-check/
└── SKILL.md
```

### `SKILL.md` 형식

```markdown
---
name: git-status-clean-check
description: Verify git working tree is clean before starting new work. Returns the status, blocks the workflow if there are uncommitted changes that aren't from the current task.
allowed-tools: Bash
---

# Git Status Clean Check

Use this skill at the start of any new git-touching task.

## When to invoke

- Before creating a new branch
- Before running /wt-branch
- Before /compound (user expects clean state)
- Before any rebase or force-push

## Steps

1. Run `git status --short` to check working tree state
2. Classify the result:
   - **Clean** (no output): proceed
   - **Stale changes** (modified files unrelated to current task): warn user, ask if proceed
   - **Untracked files** (might be from another session): list them, ask if include
3. Return one of: `clean`, `stale`, `untracked`, `mixed`

## Output format

\`\`\`
✅ git working tree clean — safe to proceed
\`\`\`

or

\`\`\`
⚠️ git working tree has changes:
  M src/lib/foo.ts
  ?? src/lib/bar.ts
Are these from the current task? (y/n)
\`\`\`

## Why this exists

Single-developer + AI auto-merge environments often leave stale changes in working tree (worktrees from other sessions, uncommitted work). Touching them by accident creates unrelated commits or loses work. This skill makes "clean state confirmation" a habit, not a memory.
```

### 적용 step (양쪽 프로젝트 동일)

1. 양쪽 프로젝트에 `.claude/skills/git-status-clean-check/SKILL.md` 신설
2. `AI-AGENT-GUIDE.md`에 *"새 작업 시작 시 항상 이 skill을 호출"* 한 줄 추가
3. 다음 새 작업에서 *AI가 자동으로 호출하는지* 확인 (dogfooding)
4. 결과를 다음 Harness Journal 사이클에 박제

## 다음 후보 Skill (도입 큐)

`git-status-clean-check` 다음에 만들 만한 작은 능력들. 각각 *작은 단위*이고 *여러 슬래시 커맨드에서 재사용*되는 것:

| Skill 후보 | 무엇을 하는가 | 어디서 호출 |
|---|---|---|
| `typescript-error-classify` | tsc 출력을 파싱해서 에러를 카테고리(missing import, type mismatch, etc)로 분류 | 빌드 fail 시 |
| `mdx-frontmatter-validate` | MDX frontmatter가 zod 스키마에 맞는지 확인 | `/ingest`, `/compound` 솔루션 작성 시 |
| `pr-description-template` | 변경 파일 목록 + 커밋 메시지로 PR 본문 자동 생성 | PR 생성 시 |
| `vercel-deploy-status` | Vercel 최근 배포 상태 + URL 확인 | `/compound` 끝, 배포 검증 |
| `rate-limit-header-parse` | Anthropic/Gemini/OpenAI rate limit 헤더 파싱 + 다음 호출 가능 시간 계산 | 429 에러 처리 |
| `docs-solution-find` | docs/solutions/에서 키워드로 기존 솔루션 검색 | 새 사고 발생 시 |

## 부작용 검토

- **Skill 발견 비용**: AI가 매 작업 시작 시 *어떤 skill이 있는지* 알아야 함. 너무 많아지면 발견 비용 증가
- **유지보수**: skill이 늘어날수록 *의미 중복*이나 *오래된 skill* 정리 필요
- **테스트**: skill은 markdown이라 unit test 어려움. 동작은 *호출 후 결과*로만 검증

## 자기 점검

1. 내가 *AI에게 매 세션 반복적으로 시키는 작은 작업*은 무엇인가? (이게 첫 Skill 후보)
2. *슬래시 커맨드 안에 박혀 있는 단계 중* 다른 워크플로에서도 재사용되는 것은? (분리 후보)
3. Skill을 도입한 후 *몇 주 후* 얼마나 자주 자동 매칭되는가? (사용 빈도 측정)
4. (열린 질문) Skill과 슬래시 커맨드의 경계가 *애매한 작업*은? (예: `/cost-check`은 어느 쪽인가?)

### 실습 과제

자신의 .claude/ 디렉터리에:

1. `.claude/skills/git-status-clean-check/SKILL.md` 신설 (위 형식 그대로)
2. AI-AGENT-GUIDE.md에 *"새 작업 시작 시 호출"* 한 줄 추가
3. 다음 새 작업 시 AI가 *자동으로* 이 skill을 가져다 쓰는지 관찰
4. 작동하면 *다음 작은 능력*을 추가 (위 큐 참조)

## AI Agent Directive

**Trigger**: 여러 워크플로에서 반복되는 작은 작업 발견 / 슬래시 커맨드가 너무 커지기 시작 / AI가 특정 작업 타입에서 자동으로 도움 제공할 필요

**Prerequisites**:
- Claude Code `.claude/skills/` 디렉터리 구조 이해
- Markdown + YAML frontmatter

### Actionable Steps

1. **반복 작업 식별** — 모든 워크플로에서 공통으로 필요한 작은 작업 (예: git status 확인)
2. **Skill frontmatter 작성** — name, description, allowed-tools 명시
3. **`.claude/skills/<name>/SKILL.md` 신설** — 상황, 실행 단계, 출력 형식 정의
4. **AI-AGENT-GUIDE에 추가** — "어떤 작업 시 이 skill을 호출해야 하는가" 명시
5. **Dogfooding 실행** — 다음 작업에서 AI가 실제로 skill을 자동으로 사용하는지 확인

### Anti-patterns

- ❌ 이미 작동하는 슬래시 커맨드를 쪼개서 Skill로 변환 (위험)
- ❌ Skill이 너무 커서 하나의 워크플로에서만 사용 (그냥 슬래시 커맨드로)
- ❌ Skill을 만들었지만 AI-AGENT-GUIDE에 기록 안 함 (AI가 모를 수 있음)
