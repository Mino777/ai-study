### moneyflow — Journal 008 (PR #95)

파일: `src/lib/trading/ai-cost-tracker.ts` + `src/lib/trading/ai-client.ts` 수정

- **단가 표**: 3-provider (Claude 3 모델 + Gemini 2 모델 + OpenAI 3 모델)
- **프로바이더별 usage 파싱** — 각기 다른 response 구조:

| 프로바이더 | 토큰 경로 | Cache 경로 |
|---|---|---|
| Gemini | `data.usageMetadata.promptTokenCount / candidatesTokenCount` | (없음) |
| Claude | `data.usage.input_tokens / output_tokens` | `data.usage.cache_read_input_tokens` |
| OpenAI | `data.usage.prompt_tokens / completion_tokens` | `data.usage.prompt_tokens_details.cached_tokens` |

- **공통 출력**: 각 함수 내부에서 `TokenUsage` 정규화 후 `logApiCall(...)` 호출 (adapter 패턴)

### 양쪽 공통 구조

```typescript
// ai-cost-tracker.ts (양쪽 모두)
export function estimateCost(model: string, usage: TokenUsage): number { ... }
export function logApiCall(log: ApiCallLog): void {
  console.log(`[ai-cost] ${route} ${provider}:${model} $${cost} (...)`);
}
```

호출 측은 항상:
```typescript
void logApiCall({ route, provider, model, usage, costUsd: estimateCost(model, usage), latencyMs });
```

## 배운 것 / 다음 후보

### 핵심 통찰 1 — 같은 메시지의 양쪽 이식은 *한 엔트리*로

Journal 007(tarosaju)과 008(moneyflow)을 분리하지 않고 묶은 이유: *메시지가 동일*(비용 가시성 확보)하고 *패턴이 동일*(프로바이더별 파싱 + 공통 로깅). 분리하면 같은 메시지를 두 번 박제하는 중복.

[Journal 006](/wiki/harness-engineering/harness-journal-006-tarosaju-anthropic-hardening)이 006+007(원래 계획)을 묶은 것과 같은 판단. **사이클 분할의 경계는 기능이 아니라 메시지**.

### 핵심 통찰 2 — *로깅만*이 가장 큰 단일 이득

DB 저장도 대시보드도 없이 *console 출력*만으로 하루 청구서 예측 가능. 2차 가치(장기 트렌드, 집계 리포트)에 시간 쓰기 전에 *1차 가치(측정 자체)*를 먼저 확보.

이게 [AI Ops 진단 체크리스트](/wiki/harness-engineering/ai-ops-environment-diagnosis-checklist)의 영역 9(비용/관측)에서 *"console.log로 토큰 수만 출력 (DB 없이)"*을 첫 단계로 권장한 이유.

### 핵심 통찰 3 — 프로바이더별 adapter 패턴

*추상화는 출력 쪽에만*. 입력 쪽(response 파싱)은 프로바이더별로 다른 길을 인정. 억지로 통일하려고 하면 모든 프로바이더의 *최소공배수*만 남게 되어 *cache token 같은 프로바이더 특화 정보*가 사라진다.

## For AI Agents

**Trigger**: 새 프로젝트에서 AI 호출을 추가했거나 *비용 가시성이 0인 상태*에서 AI API를 계속 쓰고 있을 때

**Prerequisites**:
- [AI 호출 패턴 — Circuit Breaker / 폴백 / 캐싱](/wiki/harness-engineering/ai-call-patterns-circuit-breaker-fallback)
- [AI API 비용 추적 패턴](/wiki/harness-engineering/ai-api-cost-tracking-pattern)

**Actionable Steps (30분 내 적용)**:
1. `src/lib/ai-cost-tracker.ts` (또는 적절한 경로) 신설
2. 사용하는 프로바이더의 단가 표 작성 (공식 페이지에서 확인, 날짜 명시)
3. `estimateCost(model, usage)` 함수 — 토큰 → USD
4. `logApiCall(log)` 함수 — console.log fire-and-forget
5. AI 호출 함수 각각의 *성공 return 직전*에 `void logApiCall(...)` 추가
6. 에러 시에도 `{ usage: {0,0,0}, error: kind }`로 logApiCall
7. 프로덕션 배포 후 *하루 모니터링* — 어느 모델/라우트가 비싼지 확인

**Anti-patterns**:
- ❌ 로깅을 *호출자*에 퍼뜨리기 (wrapper 없이는 일관성 없음)
- ❌ `await logApiCall()` (응답 latency 증가) — 반드시 `void` 또는 fire-and-forget
- ❌ 프로바이더별 usage 구조를 *억지로 통일* (cache token 정보 소실)
- ❌ 단가 표 없이 `console.log(tokens)` (비용 계산 없으면 측정 가치 반감)
- ❌ 분기마다 단가 표 재검증 안 하기 (프로바이더 가격 변동)

## 자기 점검

1. 내가 호출하는 AI API의 *지난 7일 비용*을 답할 수 있는가?
2. 답할 수 없다면 — console.log로만이라도 로깅이 있는가?
3. 가장 비싼 *호출 경로*는 무엇인가? (fortune/ai vs tarot-chat? trading/analyze?)
4. Cache hit rate를 *모델별*로 측정하고 있는가?
5. (열린 질문) 로깅 인프라 다음 단계 — DB? 대시보드? `/cost-check`? 무엇이 가장 큰 가치를 만드는가?
