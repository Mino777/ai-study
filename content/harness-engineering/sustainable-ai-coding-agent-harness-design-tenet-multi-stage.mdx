## 선형 체인에서 DAG(방향성 비순환 그래프) 실행으로

기존의 에이전트는 "A 다음 B, B 다음 C"와 같은 선형적인 작업 흐름(Chain)을 따릅니다. 하지만 실제 개발 프로세스는 이보다 복잡합니다. 예를 들어, 새로운 UI 컴포넌트를 만드는 작업은 데이터 모델 정의와 API 클라이언트 구현이 선행되어야 하며, 이 두 작업이 완료된 후에야 UI 컴포넌트 개발과 통합 테스트를 진행할 수 있습니다.

이러한 의존성 관계를 명확하게 표현하고 실행 순서를 최적화하는 데 DAG가 매우 효과적입니다. 에이전트는 초기 요구사항("인터뷰")을 바탕으로 전체 작업을 여러 개의 노드(Node)로 분해하고, 노드 간의 의존성을 엣지(Edge)로 연결한 DAG를 생성합니다.

```mermaid
graph TD
    subgraph Feature_Implementation_DAG ["Feature Implementation DAG"]
        A[1. 요구사항 분석 및 Spec 확정] --> B[2. 데이터 모델 정의 (Typescript Interface)];
        A --> C[3. API Client 스텁(Stub) 구현];
        B --> D[4. UI 컴포넌트 (React/SwiftUI) 개발];
        C --> D;
        D --> E[5. 유닛/통합 테스트 작성];
        E --> F[6. 최종 코드 리뷰 및 승인 요청];
    end
```

이 구조의 장점은 명확합니다.
1.  **병렬 실행**: `B`와 `C`는 서로 의존성이 없으므로 동시에 처리할 수 있습니다. 이는 전체 작업 시간을 단축시킵니다.
2.  **명확한 의존성 관리**: `D`는 `B`와 `C`가 모두 완료되어야 시작할 수 있다는 점이 명시적으로 관리됩니다.
3.  **실패 지점 격리**: 만약 `C` 단계에서 실패하면, `A`와 `B`의 결과물은 보존된 채 `C`만 재시도하거나 수정할 수 있습니다. 선형 체인에서는 전체 과정을 처음부터 다시 시작해야 할 수도 있습니다.

## 3중 Critic: 단순 통과가 아닌 다층적 품질 보증

AI가 생성한 코드가 "작동한다"는 것만으로는 충분하지 않습니다. 우리는 코드가 **정확하고(Correct), 효율적이며(Efficient), 유지보수 가능(Maintainable)**하기를 바랍니다. 이를 위해 Tenet은 세 가지 다른 관점에서 코드를 평가하는 '3중 Critic' 모델을 제안합니다.

| Critic Layer | 목적 | 검증 수단 (예시) | 비용/속도 |
| :--- | :--- | :--- | :--- |
| **Critic 1: Syntactic** | 구문적 유효성 | Linter (ESLint), Type Checker (TypeScript Compiler), 컴파일러 (Swiftc) | 매우 저렴 / 매우 빠름 |
| **Critic 2: Semantic** | 기능적 정확성 | Unit Tests, Integration Tests (Jest, XCTest) | 중간 / 중간 |
| **Critic 3: Qualitative** | 정성적 품질 | LLM-as-a-Judge (가독성, 클린 코드 원칙, 보안 취약점, 프로젝트 컨벤션 준수 여부 평가) | 비쌈 / 느림 |

이 3단계 검증은 비용이 저렴하고 결정론적인 검사부터 순차적으로 진행됩니다. 타입 에러나 린팅 오류처럼 명백한 문제는 비싼 LLM을 호출하기 전에 빠르게 걸러낼 수 있어 전체 시스템의 효율성과 경제성을 높입니다.

## Steer 메시지: 실패를 학습의 기회로

검증(Critic) 단계에서 문제가 발견되었을 때, 단순히 "실패"라고 보고하는 것은 에이전트의 개선에 큰 도움이 되지 않습니다. 중요한 것은 **무엇이, 왜, 어떻게** 잘못되었는지를 알려주는 구체적인 피드백입니다. 이것이 바로 'Steer 메시지'의 역할입니다.

Steer 메시지는 단순한 에러 로그가 아니라, 다음 행동을 지시하는 교정 지침입니다.

-   **나쁜 피드백 (Raw Error):** `TypeError: Cannot read properties of undefined (reading 'map')`
-   **좋은 피드백 (Steer Message):** `[Critic 2: Semantic] 'ProductList' 컴포넌트의 유닛 테스트가 실패했습니다. API 응답이 비어있는 경우 'products' prop이 undefined가 되어 'map' 함수 호출 시 에러가 발생합니다. 'products'가 배열인지 확인하는 옵셔널 체이닝(?.)이나 기본값 `([])`을 추가하여 렌더링 코드를 수정하세요.`

이러한 구체적인 Steer 메시지를 받은 에이전트는 막연히 코드를 재작성하는 대신, 명확한 목표를 가지고 문제점을 수정하게 됩니다. 이는 자기 교정 루프의 성공률을 극적으로 높입니다.

## TypeScript를 이용한 의사 코드 구현

이 개념을 TypeScript로 간단하게 표현하면 다음과 같은 구조를 상상할 수 있습니다.

```typescript
// harness-tenet.ts

interface CriticReport {
  passed: boolean;
  feedback: string; // 실패 시 Steer 메시지의 기반이 됨
}

interface SteerMessage {
  criticLayer: 'Syntactic' | 'Semantic' | 'Qualitative';
  instruction: string;
}

// 각 Critic은 이 인터페이스를 구현
interface Critic {
  validate(code: string): Promise<CriticReport>;
}

class TenetHarness {
  private critics: Critic[];
  private maxRetries: number = 3;

  constructor(critics: Critic[]) {
    this.critics = critics; // [SyntacticCritic, SemanticCritic, QualitativeCritic] 순서
  }

  async run(initialPrompt: string) {
    let currentPrompt = initialPrompt;
    let lastGeneratedCode = '';

    for (let i = 0; i < this.maxRetries; i++) {
      console.log(`[Attempt ${i + 1}/${this.maxRetries}]`);
      
      // Planner & Executor 단계 (LLM 호출)
      const generatedCode = await this.generateCode(currentPrompt);
      lastGeneratedCode = generatedCode;

      let allPassed = true;
      for (const critic of this.critics) {
        const report = await critic.validate(generatedCode);
        if (!report.passed) {
          // Steer 메시지 생성
          const steerMessage = this.createSteerMessage(critic.constructor.name, report.feedback);
          console.error(`Validation failed: ${steerMessage.instruction}`);

          // 다음 시도를 위한 프롬프트 수정
          currentPrompt = this.refinePrompt(generatedCode, steerMessage);
          allPassed = false;
          break; // 실패 시 다음 critic은 실행하지 않고 재시도 루프로
        }
      }

      if (allPassed) {
        console.log("All critics passed. Task complete.");
        return generatedCode;
      }
    }

    console.error("Failed to generate valid code after max retries.");
    return lastGeneratedCode; // 마지막으로 생성된 코드 반환
  }

  private async generateCode(prompt: string): Promise<string> {
    // LLM API 호출하여 코드 생성 (e.g., Anthropic, OpenAI)
    console.log("Generating code with prompt:", prompt);
    // ... 실제 API 호출 로직 ...
    return `// Generated code for: ${prompt}`; 
  }

  private createSteerMessage(criticName: string, feedback: string): SteerMessage {
    const layer = criticName.replace('Critic', '') as any;
    return {
      criticLayer: layer,
      instruction: `[${layer}] ${feedback}. Please fix this issue.`
    };
  }
  
  private refinePrompt(code: string, steer: SteerMessage): string {
    return `
      The following code has an issue:
      \`\`\`typescript
      ${code}
      \`\`\`
      Feedback: ${steer.instruction}
      
      Please provide a new version of the code that fixes this issue.
    `;
  }
}
```

이 코드는 Tenet의 핵심 루프를 보여줍니다. `run` 메소드는 코드를 생성하고, 순차적으로 Critic들을 실행하며, 실패 시 Steer 메시지를 포함한 새로운 프롬프트를 구성하여 재시도합니다. DAG 실행을 완전히 구현하려면 `taso`나 `graph-execution` 같은 라이브러리를 사용하거나 직접 스케줄러를 구현해야 하지만, 이 기본 구조만으로도 에이전트의 신뢰성을 크게 향상시킬 수 있습니다.

    현재 진행 중인 iOS 또는 프론트엔드 프로젝트에서 간단한 기능(예: 사용자 프로필 카드 표시)을 구현하는 작업을 가정해봅시다. 이 작업을 Tenet의 DAG로 표현하는 Mermaid 다이어그램을 작성하고, 각 단계에 적용할 수 있는 '3중 Critic'의 구체적인 검증 항목(예: Syntactic - "SwiftLint 규칙 위반 없음", Semantic - "사용자 이름과 이미지가 정확히 표시되는지 UI Test 통과", Qualitative - "재사용성을 고려한 컴포넌트 분리가 적절한가?")을 각각 2가지 이상씩 나열해보세요.
