## 왜 이 가이드인가

[Harness Journal](/harness-journal) 시리즈는 내 두 프로젝트(mino-moneyflow, mino-tarosaju)에서 시작했지만, *방법론으로서는* 누구의 프로젝트에도 적용 가능하다. 이 글은 *다른 사람이 자기 프로젝트에 같은 시리즈를 시작하는 법*을 단계별로 정리한 메타 가이드다.

전제 조건:
- AI 코딩 에이전트(Claude Code 등)와 함께 작업 중인 프로젝트
- 사람이 코드를 직접 치는 비중이 줄고 있음
- *AI가 안전하게 더 많이 달릴 수 있는 환경*을 만들고 싶다는 동기
- 그 과정을 *글로 박제*해서 다음 사이클을 더 쉽게 만들고 싶음

위 4가지가 맞으면 Harness Journal은 *당신의 시리즈*가 될 수 있다.

## 1단계 — 시리즈 박제 인프라 준비 (10분)

### 1.1 카테고리 결정

콘텐츠를 어디에 박을지. 옵션:
- **기존 문서/위키 안에 카테고리 신설** — 가장 가벼움
- **별도 콘텐츠 저장소** — 분리 관리
- **블로그 시리즈** — 외부 노출

권장: 첫 시작은 *기존 문서/위키*. 분리는 시리즈가 10편 이상 도달한 후 결정.

### 1.2 슬러그 규약

```
harness-journal-NNN-<short-name>.mdx
```

3자리 zero-padded(`000`, `001`, ...). 시리즈가 1000편 도달하면 4자리로 — 이 시점에 베이스라인 갱신.

### 1.3 태그 + 인덱스 페이지 (선택)

- 모든 시리즈 엔트리에 `harness-journal` 태그
- 인덱스 페이지(`/harness-journal` URL)는 *episode 번호 자동 인식 + 시간순 정렬*. [내 인덱스 페이지](/harness-journal) 참조 — 자동화 패턴.

## 2단계 — Journal 000 베이스라인 박기 (1-2시간)

이 단계가 *시리즈 전체의 출발점*. 추측 0, 직접 확인 100%로 작성.

### 2.1 무엇을 박을 것인가

각 프로젝트의 *현재 상태*를 다음 항목으로 박제:

1. **Hook 시스템** (`.claude/settings.json`이 있다면)
2. **슬래시 커맨드** (`.claude/commands/*`)
3. **Skill 시스템** (`.claude/skills/*`이 있다면)
4. **AI 협업 헌장** (CLAUDE.md, AI-AGENT-GUIDE.md, ENGINEERING.md 등)
5. **AI 호출 코드 패턴** (Circuit Breaker, 폴백, 캐싱 등)
6. **scripts/ 자동화**
7. **docs/retros/ 와 docs/solutions/**
8. **테스트 / CI / 자동 검증 게이트**
9. **메모리 시스템**
10. **고유 인프라** (프로젝트만의 특수한 것)

### 2.2 어떻게 박을 것인가

**단일 프로젝트라면** — 그냥 위 10가지를 자기 프로젝트에 매핑해서 표로 정리.

**여러 프로젝트라면** — *병렬 감사*를 사용. AI 코딩 에이전트의 서브에이전트로 각 프로젝트를 *동시에* 감사하면 한 번에 베이스라인 확보. ([내 베이스라인 작성 사례](/wiki/harness-engineering/harness-journal-000-baseline) 참조 — Claude Code Explore 서브에이전트 2개 병렬 사용)

### 2.3 무엇을 박지 *말 것*인가

- **추측** — 직접 파일 안 본 것은 박지 X
- **꾸밈** — 좋은 것만 박지 X. *갭과 부재*도 같이.
- **수치 박제** — 측정 안 된 비용/효율은 *측정 안 됨*이라고 명시. 가짜 수치 만들지 X.

## 3단계 — 첫 5 사이클 큐 정의 (30분)

베이스라인의 갭 + 비대칭으로 *대략적인 큐 5개*를 정한다. 추측이지만 *대략으로 OK*. 실제 사이클에서 *재정렬*된다.

큐 정의 패턴:
- **임팩트 + 의존성 순서** — 다른 사이클의 *전제*가 되는 작업을 먼저
- **작은 단위로 쪼개기** — 한 사이클 = 한 가지 명확한 변화
- **양쪽/한쪽** 명시 — 양쪽 프로젝트 동시 적용인지, 한쪽만인지
- **작업량** — 작음/중간/큼

내 첫 큐 사례:

| Journal | 작업 | 양쪽 적용 | 작업량 |
|---|---|---|---|
| 001 | CI 머지 전 자동 게이트 | 양쪽 동시 | 중간 |
| 002 | strict 승격 | 양쪽 | 중간 |
| 003 | Anthropic Circuit Breaker tarosaju 이식 | tarosaju 우선 | 중간 |
| 004 | AI API 비용 추적 | 양쪽 | 큼 |
| 005 | AI 출력 Zod 검증 | moneyflow 우선 | 큼 |

이 큐는 *세 번 재정렬*됐다. 라이브 사고가 정확한 다음 항목을 만들었음. 첫 큐가 가짜라는 게 아니라 *큐는 살아있는 가설*이라는 뜻.

## 4단계 — 첫 사이클 진행 + 박제 (1-3시간)

각 사이클은 동일한 구조로 박제:

```markdown
## 갭 (Before)
무엇이 부족했는가, 어떤 사고가 어떻게 보였는가

## 만든 환경 (After)
정확히 어떤 파일·훅·게이트를 추가했는가 (코드 인용 OK, 직접 만든 것만)

## 운영 데이터
적용 후 어떤 지표가 변했는가. 데이터 없으면 *데이터 없음*이라고 명시.

## 배운 것 / 다음 후보
다음 Journal로 이어지는 큐

## 출처
어느 프로젝트의 어느 retro/solution/PR과 연결되는가

## 검증 메모
무엇을 직접 확인했고 무엇이 추측인지 명시
```

이 구조가 시리즈의 *형식*. 매 에피소드가 같은 구조여야 *비교 가능*.

## 5단계 — 라이브 사고 환영하기

가장 어려운 부분: **사고가 발생하면 *기뻐해야* 한다**.

이유:
- 사고 = *추측 큐의 검증*. 추측이 틀렸다는 것을 빠르게 알게 해줌.
- 사고 = *다음 사이클의 정확한 입력*. 추측 1시간보다 라이브 사고 1번이 더 정확.
- 사고 = *시스템 결함의 가시화*. 안 보이던 결함이 드러남.

내 시리즈에서 [Journal 001 라이브 사고](/wiki/harness-engineering/harness-journal-001-ci-merge-gate)가 [Journal 002 inline test gate](/wiki/harness-engineering/harness-journal-002-inline-test-gate)를 만들었고, [그 002의 부수 사고](/wiki/harness-engineering/harness-journal-003-squash-merge-trap-pattern)가 [Journal 004 wt-branch](/wiki/harness-engineering/harness-journal-004-wt-branch-command)를 만들었다. **사고가 시리즈를 굴러가게 한다**.

박제 시 사고를 *숨기지 말 것*. 오히려 *제목*에 사고를 박는 게 더 가치 있다. 내 002의 제목: *"inline Test Gate, dogfooding 성공, 그리고 squash merge가 만든 두 번째 라이브 사고"*.

## 6단계 — 베이스라인 갱신 (살아있는 문서)

매 N 사이클(예: 5 사이클마다) Journal 000에 *갱신 로그* 섹션 추가:

```markdown
## 갱신 로그

### YYYY-MM-DD — Journal 001~004 진행 후

새 갭:
| # | 갭 | 라이브 입증 |
|---|---|---|
| 9 | (새로 발견된 것) | (어느 Journal에서) |

회피 메커니즘 진행:
| 갭 | 회피 메커니즘 | 진행 |
|---|---|---|
| ... | ... | ✅ 완료 / 🚧 진행 / ⏳ 대기 |

큐 변경:
| Journal | 원본 정의 | 실제 진행 |
|---|---|---|
| 002 | (원본) | 🔄 변경 — (라이브 사고 이유) |
```

이게 *살아있는 베이스라인*. 원본 갭과 큐는 *그대로 보존*하고, 갱신은 새 섹션으로 추가.

## 7단계 — 시리즈가 *자체 동력으로 굴러가는지* 점검 (3 사이클 후)

3 사이클 후 자문:

- [ ] 매 사이클의 *다음 후보*가 *직전 사이클의 라이브 사고*에서 나왔는가?
- [ ] 추측으로 큐를 정의한 비율이 줄고 *사고로* 큐가 정의된 비율이 늘었는가?
- [ ] 사이클당 작업 시간이 줄어들고 있는가? (작업이 더 쉬워지고 있는가)
- [ ] 같은 종류 사고가 *재발*하는가? (재발하면 회피 메커니즘 부재)
- [ ] 박제 시간 / 작업 시간 비율이 *50:50 정도*인가? (Compound 단계가 충분한가)

7가지 중 *4가지 이상이 yes*면 시리즈가 *자체 동력*으로 굴러간다. 미만이면 큐 운영을 점검.

## 운영 규칙 (안 지키면 시리즈가 깨짐)

1. **추측 박제 금지** — 직접 보지 않은 것을 *본 것처럼* 쓰면 신뢰도 0
2. **운영 데이터 없으면 *없다고* 명시** — 가짜 수치는 시리즈 전체를 오염
3. **사고 숨기지 X** — 사고가 다음 사이클의 입력이라는 메시지가 핵심
4. **한 사이클 = 한 가지** — 여러 변화를 한 사이클에 묶으면 메시지가 흐려짐
5. **베이스라인은 살아있는 문서** — 원본 보존 + 갱신 누적, 덮어쓰기 X
6. **이식성을 박제** — 다른 프로젝트로 *변경 0*으로 옮길 수 있는 자산을 명시
7. **양쪽 프로젝트 운영이라면 비대칭이 자동 큐** — 한쪽의 자산을 다른 쪽으로 이식하는 작업이 자연 사이클

## AI Agent Directive

**Trigger**: 새 프로젝트 또는 새 환경에서 Harness Journal 시리즈를 *처음 시작*할 때

**Prerequisites**:
- [AI 운영 환경 진단 체크리스트](/wiki/harness-engineering/ai-ops-environment-diagnosis-checklist) — 시작 가치 확인
- [LLM-First Wiki 설계 원칙](/wiki/harness-engineering/llm-first-wiki-principles) — 엔트리 작성 표준

**Actionable Steps (2주 안에 시리즈 시작)**:
1. 진단 체크리스트 30분 (50문항)
2. 시리즈 박제 인프라 준비 10분 (카테고리, 슬러그 규약 `harness-journal-NNN-...`, 태그 `harness-journal`)
3. Journal 000 베이스라인 박기 1~2시간 — 10 영역 audit (병렬 서브에이전트 활용)
4. 첫 5 사이클 큐 정의 30분 — 추측이지만 *대략으로 OK*
5. 첫 사이클 진행 1~3시간 — 한 가지 명확한 변화 + 박제
6. 사고 환영 — 라이브 사고를 다음 사이클의 *입력*으로 받아들임
7. 매 N 사이클마다 베이스라인 갱신 로그 append

**Anti-patterns**:
- ❌ 추측 큐에 너무 많은 시간 투자 (라이브 사고가 재정렬하므로 대략 OK)
- ❌ 사고를 *숨기거나* *실패*로 취급 (다음 사이클의 정확한 입력)
- ❌ 한 사이클에 *여러 변화*를 묶기 (메시지 흐려짐)
- ❌ 베이스라인을 덮어쓰기 갱신 (원본 보존 + append)
- ❌ 운영 데이터 없는데 *가짜 수치* 박제

## 자기 점검

1. 내가 운영 중인 프로젝트에 *시리즈를 시작할 가치*가 있는가? (위 4가지 전제 조건 점검)
2. 베이스라인을 박는 데 *몇 시간*이 필요한가? 그 시간을 들일 가치가 있는가?
3. 추측 큐 5개를 *지금 당장* 적을 수 있는가? 적을 수 없다면 베이스라인이 부족.
4. 라이브 사고가 발생했을 때 *기뻐할 수 있는가*? 본능이 *숨기려고* 하는가?
5. (열린 질문) 시리즈가 50편에 도달했을 때 *어떤 모습*이어야 성공인가?

### 실습 과제

자신의 프로젝트에 Harness Journal을 시작하고 싶다면:

1. 위 1단계를 *지금* 30분 안에 실행 — 인프라 준비
2. 2단계를 *오늘* 1-2시간 들여 베이스라인 박기
3. 3단계 큐 5개를 *대략적으로* 정의 (정확하지 않아도 OK)
4. 4단계 첫 사이클을 *내일 또는 이번 주말*에 진행
5. 5단계 라이브 사고를 만나면 *축하*하고 다음 사이클로

이 5단계를 *2주 안에* 끝내면 시리즈가 시작된다.

### 검증 메모

- 이 가이드는 내 시리즈가 *4 에피소드*에 도달한 시점에 작성. 5단계 자체 동력 점검이 *3 사이클 후*라고 한 것은 *내 운영 경험*에 기반한 추정이며 다른 사이클 수에서도 작동할 수 있음
- "사고가 시리즈를 굴러가게 한다"는 결론은 *내 4 에피소드의 표본*에 기반. 더 긴 시리즈에서 다른 패턴이 나타날 수 있음
- 운영 규칙 7가지는 *내가 4 사이클 동안 의식적으로 따른 것*이며, 다른 운영자에게는 *조정이 필요*할 수 있음
