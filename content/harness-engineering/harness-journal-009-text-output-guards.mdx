## 갭 (Before)

[Journal 000 공통 갭 #2](/wiki/harness-engineering/harness-journal-000-baseline):

> AI 출력 Zod 검증 부재 — 입력은 검증하지만 LLM 응답을 schema.parse로 거르지 않음. 환각이 그대로 통과.

tarosaju에서 첫 적용을 하려고 했으나 *라이브에서* 한 가지를 발견:

> **tarosaju의 fortune/ai와 tarot-chat 응답은 *자연어 텍스트*이지 JSON 객체가 아니다.**

Zod의 `schema.parse`는 *구조화된 객체*에 적합. 자연어 문자열에 *Zod schema*를 억지로 씌우는 것은 가치 없음. 도구가 데이터 형식에 맞지 않으면 *다른 도구*가 필요.

## 만든 환경 (After) — Text Output Guards

### 추가: `src/lib/ai-output-guards.ts`

자연어 응답에 대한 *3가지 검증 축*:

1. **길이 범위** — `checkLength(min, max)`
   - 너무 짧으면 의미 없음 (응답 실패 가능)
   - 너무 길면 토큰 낭비 + 사용자 피로

2. **금지 용어 contains** — `checkForbiddenTerms(terms)`
   - Fortune: `"병원", "진료", "약을", "처방", "우울증", "불안장애"` 등 (의학/진료 용어)
   - Tarot-chat: `"우울증", "정신과", "변호사", "소송"` (의학/법률 조언)
   - 공통: `"죽어", "자해", "극단적"` (위기 제안)

3. **위기 제안 re-check**
   - 입력 `crisis-detector`와 별개
   - *출력에* 위기 단어가 나오면 AI가 사용자에게 *트리거 유발* 가능성

### 통합 함수

```typescript
export function guardFortuneOutput(text: string): GuardResult {
  const lengthCheck = checkLength(text, 30, 1000);
  if (!lengthCheck.safe) return lengthCheck;

  const forbiddenCheck = checkForbiddenTerms(text, FORTUNE_FORBIDDEN_TERMS);
  if (!forbiddenCheck.safe) return forbiddenCheck;

  const crisisCheck = checkForbiddenTerms(text, CRISIS_SUGGESTION_TERMS);
  if (!crisisCheck.safe) return crisisCheck;

  return { safe: true };
}
```

### 라우트 통합

```typescript
// fortune/ai/route.ts
const { text: fortune } = await callAnthropic({ ... });

const guardResult = guardFortuneOutput(fortune);
if (!guardResult.safe) {
  logger.warn("[fortune/ai] output guard failed", {
    reason: guardResult.reason,
    matched: "matched" in guardResult ? guardResult.matched : undefined,
  });
  // 환각을 사용자에게 노출하지 X — 안전 기본값
  return Response.json({ fortune: SAFE_FALLBACK.fortune, cached: false, guarded: true });
}

// 가드 통과한 응답만 캐시 저장
await supabase.from("ai_fortune_cache").upsert({ ... });
```

핵심:
- **가드 실패를 숨기지 않음** — `logger.warn`으로 가시화
- **SAFE_FALLBACK 반환** — *throw가 아님*. 콘텐츠 도메인은 *부드러운 대체*가 더 안전
- **클라이언트 플래그** — `{ guarded: true }`로 알림
- **캐시 저장 skip** — 가드 실패한 응답은 캐시하지 않음 (미래 재발 방지)

## 배운 것 / 다음 후보

### 핵심 통찰 1 — 계획 도구가 데이터에 맞지 않으면 *도구를 바꾼다*

Journal 000 큐에 "tarosaju Zod 출력 검증"이 있었지만 라이브에서 *응답이 자연어*라는 것을 발견. 계획을 *억지로 이행*하는 대신 *다른 도구*(text guards)를 만드는 것이 옳은 판단. 

이건 *큐는 살아있는 가설*이라는 [Journal 000 갱신 로그](/wiki/harness-engineering/harness-journal-000-baseline#갱신-로그) 원칙의 구체적 사례. 추측 큐가 라이브로 재정렬되는 *네 번째* 사례.

### 핵심 통찰 2 — 도메인별 fail 전략

| 도메인 | 가드 실패 시 | 이유 |
|---|---|---|
| 거래 / 결제 | **throw** | 잘못된 결과가 *더 큰 손해* |
| 콘텐츠 / 점술 / 상담 | **SAFE_FALLBACK** | 무응답보다 *부드러운 대체*가 사용자 경험 유지 |
| 내부 시스템 호출 | **재시도 + 최종 throw** | 시스템은 에러 처리 가능 |

*공통*: 환각 응답을 *그대로 노출하지 X*. 전략은 달라도 *환각 노출 금지*는 동일.

### 핵심 통찰 3 — 가드 실패도 *측정*

`logger.warn` + `{ guarded: true }` 플래그로 가드 발동 시 가시화. *얼마나 자주* 발동하는지가 *LLM 품질 측정*의 한 축. 측정 없는 가드는 *sanity check*도 못 함.

## For AI Agents

**Trigger**: 자연어 텍스트를 반환하는 AI 엔드포인트 / JSON 응답이 아닌 LLM 출력 검증이 필요할 때

**Prerequisites**:
- [AI 출력 Zod 검증 5 layer](/wiki/harness-engineering/ai-output-zod-validation-pattern) — *JSON 응답*에 적합한 패턴 (이 엔트리는 *text 응답* 대안)

**Actionable Steps**:
1. 응답 형식 확인: *JSON 객체*면 Zod, *자연어 텍스트*면 이 엔트리의 text guards
2. 금지 용어 리스트 작성 — system prompt의 "금지 사항" 섹션과 일치
3. 길이 범위 정의 — 너무 짧으면 실패 감지, 너무 길면 토큰 낭비
4. 위기/트리거 단어 re-check (도메인에 따라)
5. 가드 실패 시 전략 결정: throw (거래) vs SAFE_FALLBACK (콘텐츠) vs 재시도 (내부 시스템)
6. `logger.warn`으로 가시화 + 클라이언트 플래그 (`{ guarded: true }`)
7. 프로덕션 배포 후 *가드 발동률 모니터링*

**Anti-patterns**:
- ❌ Zod를 *자연어 응답*에 억지로 씌우기 (형식 불일치)
- ❌ 가드 실패를 *silent*하게 숨김 (측정 불가)
- ❌ 모든 도메인에 *같은 fail 전략* (throw vs fallback은 도메인별)
- ❌ 환각 응답을 *그대로 노출* (어떤 fail 전략이든 환각 노출 금지는 공통)
- ❌ 금지 용어 리스트를 *system prompt*와 분리 (일관성 없음)

## 자기 점검

1. 내 AI 엔드포인트의 응답 형식은 *JSON* 인가 *자연어 텍스트* 인가?
2. 자연어 응답에 *길이 + 금지 용어 + 위기 재검증* 중 몇 개가 적용돼 있는가?
3. 가드 실패 시 *어떤 전략* (throw / fallback / retry)을 쓰는가? 도메인에 적합한가?
4. 가드 발동률을 *측정*하고 있는가? 실제로 얼마나 자주 발동하는가?
5. (열린 질문) *system prompt의 금지 사항*과 *코드의 가드 리스트*를 *단일 source*에서 유지할 방법이 있는가?
