## 왜 DESIGN.md가 필요한가?

GitHub Copilot, Cursor, 혹은 자체 제작한 AI 코딩 에이전트를 사용할 때 우리는 종종 사소하지만 반복적인 수정 작업에 시간을 쏟게 됩니다. AI는 프로젝트의 디자인 시스템을 완벽히 이해하지 못하기 때문입니다. 예를 들어, "장바구니에 담기 버튼을 만들어줘"라고 요청하면, AI는 `class="btn-primary"` 같은 일반적인 코드를 생성하거나 존재하지 않는 `<PrimaryButton>` 컴포넌트를 상상해냅니다. 우리 프로젝트의 실제 버튼 컴포넌트가 `<ShopButton variant="primary" isLoading={false}>` 형태라는 사실을 알지 못합니다.

이러한 'UI 환각(Hallucination)' 현상은 AI가 프로젝트의 컨텍스트, 특히 시각적 컴포넌트의 'API 명세'를 모르기 때문에 발생합니다. 개발자는 AI가 생성한 코드를 일일이 디자인 시스템에 맞게 수정해야 하며, 이는 생산성을 저해하는 주요 요인입니다.

`DESIGN.md`는 이 문제를 해결하기 위한 제안입니다. 이는 디자인 시스템의 모든 컴포넌트, 토큰, 사용법을 기계가 읽을 수 있는(machine-readable) 단일 파일 포맷으로 표준화하는 전략입니다. 마치 백엔드 개발자가 OpenAPI/Swagger 명세로 API 계약을 정의하듯, 프론트엔드 개발자는 `DESIGN.md`를 통해 AI에게 UI 컴포넌트의 계약을 명확히 전달할 수 있습니다. AI는 이 명세를 "읽고" 우리 프로젝트의 규칙에 맞는 정확한 코드를 생성하게 됩니다.

## DESIGN.md의 핵심 구조

`DESIGN.md`는 마크다운 파일 형식을 제안하지만, 그 내용은 구조화된 데이터(JSON, YAML 등)를 포함하는 형태가 이상적입니다. 핵심은 AI가 파싱하기 쉬운 일관된 스키마를 갖는 것입니다.

다음은 JSON 기반 `DESIGN.md` 스키마의 예시입니다.

| 키 | 타입 | 설명 |
|---|---|---|
| `schemaVersion` | String | 명세의 버전. 향후 변경 사항 추적에 용이. |
| `metadata` | Object | 프로젝트 이름, 마지막 업데이트 날짜 등 메타 정보. |
| `tokens` | Object | 색상, 타이포그래피, 간격 등 디자인 토큰 정의. |
| `components` | Array | 디자인 시스템의 모든 컴포넌트 목록. |

### 컴포넌트 상세 스키마

`components` 배열의 각 요소는 다음과 같은 상세 구조를 가집니다.

```json
{
  "name": "ShopButton",
  "description": "사용자 액션을 유도하기 위한 주요 버튼 컴포넌트. 장바구니 담기, 구매하기 등에 사용됩니다.",
  "usage": {
    "do": "핵심적인 긍정적 액션에 사용하세요.",
    "dont": "취소나 보조적인 액션에는 'ghost'나 'secondary' variant를 사용하세요."
  },
  "props": [
    {
      "name": "variant",
      "type": "'primary' | 'secondary' | 'ghost'",
      "required": false,
      "defaultValue": "'primary'",
      "description": "버튼의 시각적 스타일을 결정합니다."
    },
    {
      "name": "isLoading",
      "type": "boolean",
      "required": false,
      "defaultValue": "false",
      "description": "로딩 상태를 표시하며, 클릭을 비활성화합니다."
    },
    {
      "name": "children",
      "type": "React.ReactNode",
      "required": true,
      "description": "버튼 내부에 표시될 텍스트나 아이콘입니다."
    }
  ],
  "examples": [
    {
      "description": "기본적인 primary 버튼 사용 예시",
      "code": "<ShopButton variant='primary'>구매하기</ShopButton>"
    },
    {
      "description": "로딩 상태의 secondary 버튼",
      "code": "<ShopButton variant='secondary' isLoading={true}>처리 중...</ShopButton>"
    }
  ]
}
```

이 구조에서 `name`, `props`, `examples` 필드는 AI에게 가장 중요합니다. AI는 이를 통해 컴포넌트의 이름, 사용 가능한 속성, 그리고 실제 코드에서 어떻게 사용되는지를 학습하여 정확한 코드를 생성할 수 있습니다.

## AI 코딩 도구와의 연동 흐름

`DESIGN.md` 파일이 프로젝트에 존재할 때, AI 코딩 도구는 다음과 같은 흐름으로 동작합니다. 이는 RAG(Retrieval-Augmented Generation) 패턴의 일종으로 볼 수 있습니다.

```mermaid
graph TD
    A[사용자 프롬프트 입력 · "결제 페이지에 로딩 버튼 추가"] --> B{AI Agent};
    B --> C["1. DESIGN.md 파일 검색 및 파싱"];
    C --> D["2. 프롬프트와 관련된 · 'ShopButton' 컴포넌트 정보 추출"];
    subgraph Injected Context
        D -- "isLoading, variant props, examples" --> E[컨텍스트 보강];
    end
    A -- "원본 프롬프트" --> E;
    E --> F[LLM API 호출];
    F --> G["`// 결제 처리 중...` · <ShopButton variant='primary' isLoading={true}>결제하기</ShopButton>"];
    G --> H[코드 생성 및 제안];
```

1.  **컨텍스트 검색(Retrieval)**: AI 에이전트는 사용자 프롬프트를 받으면 먼저 프로젝트 루트의 `DESIGN.md` 파일을 찾습니다.
2.  **정보 추출(Extraction)**: 파일 내용을 파싱하고, 사용자 요청과 관련된 컴포넌트(`ShopButton`)의 명세를 추출합니다.
3.  **컨텍스트 주입(Injection)**: 추출된 컴포넌트 명세(특히 `props`와 `examples`)를 원래 프롬프트와 함께 LLM의 컨텍스트 윈도우에 주입합니다.
4.  **코드 생성(Generation)**: 보강된 컨텍스트를 기반으로 LLM은 디자인 시스템에 100% 부합하는 코드를 생성합니다.

## 실전 예제: TypeScript와 AI 에이전트 연동

실제 AI 에이전트의 일부가 될 수 있는 TypeScript 함수 예제입니다. 이 함수는 `DESIGN.md`를 읽고 시스템 프롬프트를 구성합니다.

```typescript
import * as fs from 'fs/promises';
import path from 'path';

// DESIGN.md 파일의 타입을 정의 (실제로는 더 복잡할 것)
interface DesignSystemSpec {
  components: Array<{
    name: string;
    description: string;
    props: Array<{ name: string; type: string; description: string }>;
    examples: Array<{ description: string; code: string }>;
  }>;
}

// AI 에이전트가 LLM 호출 전 컨텍스트를 보강하는 함수
async function buildPromptWithDesignSystemContext(
  userPrompt: string,
  projectRoot: string
): Promise<string> {
  const designSpecPath = path.join(projectRoot, 'DESIGN.md');
  let designSpec: DesignSystemSpec;

  try {
    const specContent = await fs.readFile(designSpecPath, 'utf-8');
    // 실제 구현에서는 마크다운에서 JSON 블록을 파싱하는 로직이 필요
    designSpec = JSON.parse(specContent); 
  } catch (error) {
    console.warn('DESIGN.md not found or failed to parse. Proceeding without design context.');
    return userPrompt;
  }
  
  // 간단한 키워드 매칭으로 관련 컴포넌트 찾기 (실제로는 embedding 기반 검색이 더 효과적)
  const relevantComponents = designSpec.components.filter(c =>
    userPrompt.toLowerCase().includes(c.name.toLowerCase())
  );

  if (relevantComponents.length === 0) {
    return userPrompt;
  }

  const contextHeader = `
You are a frontend developer working on a project with a specific design system.
Adhere to the following component specifications. Do NOT invent props or component names.

--- DESIGN SYSTEM CONTEXT ---
`;

  const contextBody = relevantComponents.map(c => `
Component: <${c.name}>
Description: ${c.description}
Props:
${c.props.map(p => `- ${p.name} (${p.type}): ${p.description}`).join('\n')}
Examples:
${c.examples.map(e => `// ${e.description}\n${e.code}`).join('\n')}
`).join('\n---\n');

  const finalPrompt = `
${contextHeader}
${contextBody}
--- END OF CONTEXT ---

User Request: "${userPrompt}"

Generate the code now.
`;

  return finalPrompt;
}

// 사용 예시
async function main() {
    const userPrompt = "Add a loading button for the checkout page.";
    const finalPromptForLLM = await buildPromptWithDesignSystemContext(userPrompt, process.cwd());
    console.log(finalPromptForLLM);
    // 이 finalPromptForLLM을 OpenAI, Anthropic 등의 API로 전송
}

main();
```

## 2026년 이후의 전망: 멀티모달과 자동 동기화

`DESIGN.md`는 현재의 LLM 기술을 위한 훌륭한 인터페이스지만, 기술이 발전함에 따라 그 형태도 진화할 것입니다.

*   **Figma와의 자동 동기화**: 현재는 `DESIGN.md`를 수동으로 관리해야 할 수 있지만, 앞으로는 Figma의 Dev Mode나 API를 활용해 디자인 시스템이 변경될 때마다 `DESIGN.md` 파일을 자동으로 업데이트하는 CI/CD 파이프라인이 보편화될 것입니다. 디자인 변경이 코드 생성 컨텍스트에 실시간으로 반영되는 것입니다.
*   **멀티모달 LLM의 직접 참조**: GPT-4o와 같은 멀티모달 모델은 텍스트 명세뿐만 아니라 시각적 정보를 직접 이해할 수 있습니다. 미래의 AI 코딩 도구는 `DESIGN.md`를 참조하는 대신, Figma 링크나 UI 스크린샷을 직접 "보고" 컴포넌트의 형태와 사용법을 파악할 수 있게 될 것입니다. 이때 `DESIGN.md`는 AI가 시각적으로 파악한 내용을 검증하는 보조적인 'Ground Truth' 역할을 하게 될 수 있습니다.

결국 `DESIGN.md`의 핵심 철학은 "AI에게 명확한 계약(contract)을 제공한다"는 것입니다. 그 형태가 텍스트 기반 JSON이든, Figma API 응답이든, AI가 우리의 의도대로 정확하게 작동하도록 제어하는 'Harness'의 중요한 일부로 남을 것입니다.
