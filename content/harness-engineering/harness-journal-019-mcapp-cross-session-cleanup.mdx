### tarosaju — PR #17/#18 리뷰 결과: 5 심각 + 2 정리

자세한 회고: [`docs/retros/2026-04-13-mcapp-cleanup.md`](https://github.com/Mino777/mino-tarosaju/blob/main/docs/retros/2026-04-13-mcapp-cleanup.md) (tarosaju repo)

**심각 1 — 마이그레이션 안전 패턴 위반** (`20260413_finalize_to_user_id.sql`)
- 원본 `20260409_chat.sql` 주석 *"백필 확인 후 수동 실행"*을 무시하고 무조건 `ALTER ... NOT NULL`로 만들어져 있었음
- NULL row 1개라도 있으면 배포 시 PostgreSQL이 거부 → 프로덕션 중단 직전
- 교정: **(1) 재-backfill (2) 남은 NULL 카운트 기반 조건부 ALTER (3) 남아있으면 `RAISE WARNING`으로 경고만 + ALTER 스킵**
- 박제: [`docs/solutions/supabase/2026-04-13-not-null-finalize-safe-pattern.md`](https://github.com/Mino777/mino-tarosaju/blob/main/docs/solutions/supabase/2026-04-13-not-null-finalize-safe-pattern.md)

**심각 2 — Realtime retry 과잉설계 + dead code** (`useChatMessages.ts`, `useNotifications.ts`)
- 새로 추가된 `error state` + `AbortController` + retry 로직 모두 **세 가지 결함이 동시에**:
  1. `supabase.channel(channelName).unsubscribe()`는 **새 채널 인스턴스를 만들어 unsubscribe** — 기존 채널은 그대로 살아있음 (no-op 안티패턴)
  2. `setTimeout` retry가 cleanup에서 clear 안 됨 → **좀비 구독**
  3. `error` 필드를 return하지만 소비자 2곳 (`notifications/page.tsx`, `chat/[roomId]/page.tsx`) **어느 곳도 destructure 안 함** → dead code
- 결정: 부분 패치 vs **전체 롤백** → 사용자에게 합의 → 전체 롤백 채택
- 박제: [`docs/solutions/supabase/2026-04-13-realtime-channel-retry-pitfall.md`](https://github.com/Mino777/mino-tarosaju/blob/main/docs/solutions/supabase/2026-04-13-realtime-channel-retry-pitfall.md)

## 6 공통 함정 패턴

### 1. PR 메타데이터 거짓말

PR 설명 / 커밋 메시지는 깔끔한데 실제 diff는 망가져 있는 패턴.
- tarosaju #17: "P0 안정성 이슈 3개 수정" 메시지 → 실제로 5개 심각 + 2 정리 발견
- moneyflow #114: 깔끔한 변경사항 리스트 → 빌드 실패 + tsc 11 errors

**처방**: 다른 세션 PR 이어받을 때 **메시지 무시하고 line-by-line diff 검증**.

### 2. 과잉설계 + 소비자 0건 = 즉시 롤백

retry / error state / fallback을 추가했는데 **사용 측 코드에서 destructure조차 안 함**.
- tarosaju Realtime: `error` 필드 return → 소비자 2곳 모두 무시 → dead code

**처방**: 새 필드/state 추가 시 **소비자 grep으로 사용처 1건 이상 확인**, 0건이면 추가 안 함.

### 3. 빌드/타입 검증 누락 머지

`npm run build` / `tsc --noEmit` 안 돌리고 PR 머지.
- moneyflow #114: 빌드 실패 후 30분 만에 #115로 fix
- ai-study (이전 세션 누적): apple-intelligence-api.mdx MDX 컴파일 에러가 production 배포까지 도달

**처방**: PR 머지 전 inline test gate (Web Harness Journal 002 패턴) **반드시** — 이게 없으면 맥앱 세션 자가 fix loop 발생.

### 4. Codex 오인용 (모델 자기 정체성 혼동)

같은 날 두 프로젝트에서 모델이 자기를 Codex로 칭하는 텍스트가 commit message + CLAUDE.md에 박혀 머지됨.
- tarosaju CLAUDE.md "Output Review" 섹션
- moneyflow #114 commit message

**가설**: 맥앱 환경 변수 또는 시스템 프롬프트가 Codex/Cursor 관련 컨텍스트를 누설.

**처방**: CLAUDE.md / commit message에 "Codex" 문자열 grep 가드. PreToolUse 훅으로 Edit/Write 검사.

### 5. Stale worktree 환경 오염 (테스트 숫자 거짓말)

`.claude/worktrees/<name>` 안의 과거 코드 스냅샷이 vitest pickup되어 가짜 실패.
- tarosaju: 691 → 1294 (가짜 +603) 16 failed → worktree 제거 후 691/0

**처방**:
- `vitest.config.ts` / `jest.config.js`에 `.claude/worktrees/**` exclude 기본값
- *맥앱 Claude를 비난하기 전에 환경 변수를 의심* — 숫자 보고 즉시 `git worktree list` 확인

### 6. 자동 수정 도구의 silent 파일 손상

`validate-content.mjs`처럼 *수정 후 writeBack*하는 도구의 offset 버그.

**처방**:
- 자동 수정 도구는 항상 **수정 후 컴파일/검증 1회 더** 돌려서 손상 자기 검출
- writeBack 전 git diff 출력 옵션
- offset 계산은 항상 raw(전체 파일) 기준, 부분 추출 결과를 offset으로 쓰지 말 것

## 처방 — 크로스 세션 리뷰 프로토콜

다른 Claude 세션(맥앱 / 웹 / 다른 터미널) 결과물을 이어받을 때 **표준 5단**:

```
1. PR 메타데이터 무시 — 절대 신뢰 X
2. git worktree list → stale 검출
3. line-by-line diff 검증
   - 새 코드 vs 삭제 누락 (특히 try/catch 블록 교체 케이스)
   - 새 필드/state → 소비자 grep
   - 자기 정체성 오인용 grep ("Codex", "Cursor", "Cline", "Aider")
4. 검증 파이프라인 3단 (CI에 안 맡기고 로컬에서)
   - npm test (또는 vitest)
   - lint clean
   - 실제 프로덕션 빌드 (tsc --noEmit + 빌드)
5. 발견 사항 분류 → 부분 패치 vs 전체 롤백 → 사용자 합의 → 교정 PR
```

이걸 슬래시 커맨드 후보로: `/cross-session-review` (대상 PR/커밋 입력 → 5단 자동 실행).

## For AI Agents

이 엔트리를 읽는 에이전트가 *"맥앱 Claude가 작업한 코드를 이어받아 검증해야 한다"* 상황이면:

1. **메타데이터 신뢰 금지** — PR 설명/커밋 메시지가 깨끗해도 diff를 line-by-line 봐라
2. **6 함정 grep**:
   - "Codex" / "Cursor" / "Cline" 자기 정체성 오인용 (CLAUDE.md / commit message / docs)
   - 새 `error` 필드 / state → 소비자 grep, 0건이면 즉시 롤백 후보
   - try/catch 블록 변경 → 이전 블록 잔재 (`if(...Result)` 같은 패턴) 검색
   - allSettledWithTimeout 같은 제네릭 헬퍼 → 호출부 명시 annotation 검사
3. **환경 오염 먼저 의심**: 테스트 숫자 / 빌드 결과가 이상하면 `git worktree list` + `vitest --listFiles` 먼저
4. **부분 패치 vs 전체 롤백**: 한 번 결정 못 하면 사용자에게 명시적으로 "패치할까요 / 롤백할까요" 물어보기
5. **자동 수정 도구는 검증 1회 더**: write-back 후 컴파일/lint 재실행으로 자기 손상 검출
6. **로컬 빌드 파이프라인 3단** (test + lint + 실제 빌드) — CI에 위임 금지
