## 왜 이 글이 중요한가

Martin Fowler가 하네스 엔지니어링을 정리했다. 개별 팁 수준("CLAUDE.md 잘 쓰세요")이 아니라 **제어 이론(Cybernetics) 기반의 체계적 프레임워크**를 제시한다. 핵심 공식:

> **Agent = Model + Harness**

하네스의 목적: ①에이전트가 처음부터 좋은 결과를 낼 확률을 높이고 ②사람이 리뷰하기 전에 스스로 교정하는 피드백 루프를 만드는 것.

---

## 핵심 프레임워크: 4분면 매트릭스

Fowler는 하네스의 구성 요소를 **2개 축 × 2개 유형**으로 분류한다:

### 방향축: Guide vs Sensor

| | Guide (Feedforward) | Sensor (Feedback) |
|---|---|---|
| **언제** | 행동 **전**에 조향 | 행동 **후**에 관찰 |
| **목적** | 좋은 결과의 확률을 높임 | 나쁜 결과를 감지하고 교정 |
| **비유** | 자동차 내비게이션 | 차선이탈 경고 |

### 실행축: Computational vs Inferential

| | Computational | Inferential |
|---|---|---|
| **실행** | CPU 기반, 결정론적 | GPU/NPU 기반, 확률적 |
| **속도** | 밀리초~초 | 초~분 |
| **신뢰도** | 100% (같은 입력 → 같은 결과) | 확률적 (환각 가능) |
| **비용** | 거의 무료 | 토큰 비용 발생 |

### 4분면 조합

```
                Guide (사전)          Sensor (사후)
            ┌──────────────────┬──────────────────┐
Computa-    │ Language Server   │ 린터, 타입체커    │
tional      │ CLI 도구          │ 테스트 스위트     │
            │ Codemod          │ 커버리지 추적     │
            │ OpenRewrite      │ 구조 분석(ArchUnit)│
            ├──────────────────┼──────────────────┤
Inferen-    │ CLAUDE.md        │ 코드 리뷰 에이전트 │
tial        │ 아키텍처 문서     │ LLM-as-a-Judge   │
            │ 코딩 컨벤션 규칙  │ Mutation Testing  │
            │ Bootstrap Skills │ 시맨틱 중복 탐지   │
            └──────────────────┴──────────────────┘
```

**핵심 통찰**: Computational은 저렴하고 신뢰 가능 → **매 변경마다 실행**해도 됨. Inferential은 강력하지만 비싸고 확률적 → **선택적으로 사용**.

## Steering Loop — 하네스를 개선하는 루프

> "문제가 여러 번 발생하면, 그 문제의 재발 확률을 낮추거나 아예 방지하도록 Guide와 Sensor를 개선해야 한다."

```mermaid
flowchart LR
    A["에이전트 작업"] --> B["Sensor가 문제 감지"]
    B --> C["사람이 패턴 인식"]
    C --> D["Guide 또는 Sensor 추가/개선"]
    D --> A

    style D fill:#10b98120,stroke:#10b981
```

이것이 [Compound Engineering](/wiki/harness-engineering/compound-engineering-philosophy)과 정확히 같은 원리. 매 사이클의 실패가 다음 사이클의 하네스를 개선한다.

---

## 안티패턴 5가지

| 안티패턴 | 증상 | 해법 |
|----------|------|------|
| **Feedback-Only** | 같은 실수 반복 (Sensor는 있지만 Guide 없음) | Guide 추가: 규칙·컨벤션·문서 선행 |
| **Feedforward-Only** | 규칙은 있지만 지켜지는지 모름 | Sensor 추가: 린터·테스트·리뷰 |
| **AI 생성 테스트 맹신** | 테스트가 있지만 의미 없는 assertion | 사람 검증 또는 Approved Fixtures |
| **하네스 드리프트** | 시간이 지나면서 Guide/Sensor가 코드와 어긋남 | 정기 감사 + 자동 드리프트 탐지 |
| **비일관 하네스** | Guide와 Sensor가 서로 모순 | 우선순위 명시 + 정기 정합성 체크 |

## Harnessability — 모든 코드베이스가 똑같지 않다

**강타입 + 명확한 모듈 경계 + 프레임워크 규약** → Harnessability 높음. 에이전트의 문제 공간이 자연스럽게 줄어듦.

**레거시 + 약타입 + 모놀리스** → 하네스가 가장 필요하지만 적용이 가장 어려운 역설.

Fowler는 이를 **"Ambient Affordances"**라고 부른다 — 코드베이스의 구조적 속성이 에이전트의 성공을 암묵적으로 결정한다.

우리 프로젝트에 매핑하면:
- **aidy-ios (TMA 26모듈)**: Module.swift가 의존성 규칙을 강제 → Harnessability 높음
- **moneyflow (Next.js 모놀리스)**: 타입은 있지만 모듈 경계 약함 → 중간
- **ai-study (Next.js + MDX)**: Zod 스키마 + 빌드 검증 → 중간~높음

## 실전 사례: 기업들의 하네스

| 기업 | Guide | Sensor |
|------|-------|--------|
| **OpenAI** | 커스텀 린터 + 구조 테스트 + 레이어 아키텍처 강제 | "Garbage Collection" 스캔으로 데드코드/드리프트 정기 탐지 |
| **Stripe** | "Blueprints"로 Sensor를 에이전트 워크플로우에 통합 | Pre-push hook에 휴리스틱 기반 관련 린터 자동 선택 |
