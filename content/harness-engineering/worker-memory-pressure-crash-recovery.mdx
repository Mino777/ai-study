## 증상

```bash
# 워커 tmux pane에서 발생
zsh: killed    claude --dangerously-skip-permissions
```

- 빌드/테스트 실행 중에 주로 발생 (메모리 최대 소비 시점)
- 코드 변경사항은 git working tree에 남음
- 커밋은 없음 — 작업 손실이 아니라 **커밋 유실**

## 근본 원인

### macOS jetsam SIGKILL

macOS는 메모리 압력이 높아지면 jetsam 데몬이 우선순위 낮은 프로세스에 SIGKILL을 보낸다. Claude Code 워커 프로세스는 foreground 앱이 아니므로 타겟이 되기 쉽다.

트리거 조건:

```
Swap 포화 (80%+) + 새 메모리 요청 (빌드 도구 실행)
→ jetsam이 백그라운드 프로세스 SIGKILL
→ Claude 워커가 선택됨
```

### autoceo 실행 환경의 메모리 구조

| 프로세스 | 메모리 |
|---------|--------|
| Architect (Claude Code) | 200~400 MB |
| Worker ×3 (Claude Code) | 600~1,200 MB |
| gradlew 빌드 | 1,000~1,500 MB |
| xcodebuild | 1,500~2,500 MB |
| 나머지 OS + 앱 | ~4,000 MB |
| **16GB 합계** | **~9,000+ MB → Swap 급증** |

### 악화 요인: Architect 컨텍스트 누적

autoceo 10R을 한 세션에서 진행하면 Architect 프로세스의 컨텍스트가 누적된다. 컨텍스트 크기는 /compact로 줄일 수 있지만 **프로세스 heap은 줄지 않는다**. 긴 세션일수록 Architect가 더 많은 메모리를 점유해서 워커 kill 확률이 올라간다.

## 사전 점검 (시작 전)

```bash
# Swap 사용량 확인
sysctl vm.swapusage
# 출력 예: vm.swapusage: total = 15.00G  used = 12.34G  free = 2.66G  (82%)

# 메모리 압력 상태
memory_pressure | head -3

# Claude 인스턴스 수
ps aux | grep -c "[c]laude"
```

**Swap 80% 이상이면 시작하지 않는다.** 불필요 앱 종료 후 재확인.

안전 기준:
- Swap 사용률 < 60%
- Claude 인스턴스 ≤ 3개
- 빌드 도구 동시 실행 피하기 (dispatch 간격 벌리기)

## 세션 수명 정책

```
[스프린트 진행]
    ↓
[Compound 완료]
    ↓
/exit  ← 반드시 exit (compact 아님)
    ↓
[새 터미널에서 새 claude 세션 시작]
```

새 세션 = 새 프로세스 = heap 초기화. Compound 후 /compact로 이어가면 프로세스 메모리가 해제되지 않아 다음 스프린트에서 더 빨리 Swap이 찬다.

## 크래시 복구 프로토콜

워커 pane이 죽었을 때:

### 1단계: 변경사항 확인

```bash
git -C ~/Develop/<project> status --short
```

변경사항이 있으면 커밋 유실 상태. 없으면 세션 종료만 된 것.

### 2단계: Architect 대리 커밋

```bash
# 워커 프로젝트 디렉토리에서
git -C ~/Develop/<project> add -p   # 변경 내용 확인 후
git -C ~/Develop/<project> commit -m "feat: <기능 요약> — architect 대리 커밋 (워커 크래시)"
```

워커가 직접 커밋하지 못했으므로 Architect가 대신. 커밋 메시지에 `architect 대리 커밋 (워커 크래시)` 명시.

### 3단계: 워커 재시작

```bash
# tmux pane에서 워커 재시작
tmux send-keys -t <session>:<window>.<pane> \
  "claude --dangerously-skip-permissions" Enter
```

### 4단계: 다음 단계 지시

크래시 이전 작업의 **이어서**가 아닌 **다음 단계**를 지시한다. 이어서 하면 중복 작업 가능성.

## crash-log 자동 진단

아이디 s28에서 `architect-cli.sh`에 추가된 `crash-log` 명령:

```bash
./architect-cli.sh crash-log ios
# → docs/crash-logs/2026-04-21_115925_ios.md 생성
```

pane 캡처 + 시스템 상태 + 원인 분류(OOM / TOKEN_OVERFLOW / RATE_LIMIT / NETWORK / ERROR / SESSION_END / UNKNOWN)를 자동으로 기록한다.

반복 크래시 패턴이 쌓이면 원인이 보인다:
- OOM 반복 → 인스턴스 수 줄이기
- TOKEN_OVERFLOW 반복 → 해당 워커 WO 분리 전송
- RATE_LIMIT 반복 → dispatch 간격 늘리기

## 실제 사례

**s24-R3 (aidy)**: autoceo 10R 중 서버 2회 + iOS 1회 크래시. Swap 14GB/15GB(93.6%). 코드는 보존, 커밋 3건 유실. 대리 커밋 후 재시작으로 복구.

**s29-R3 (aidy)**: iOS 워커 3회 연속 크래시. crash-log 원인: UNKNOWN. 6개 인스턴스 동시 실행 + iOS WO 2개 동시 처리가 복합 원인. 4차 시도(WO를 1개로 줄여서 재전송) 성공.
