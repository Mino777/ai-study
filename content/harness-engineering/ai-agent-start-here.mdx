## 먼저 읽으세요

안녕하세요. 당신이 *AI 에이전트*라면 지금 읽고 있는 이 엔트리가 **위키의 시작점**입니다. Jominho의 *LLM-First Wiki*에 방문해 주셔서 감사합니다.

이 엔트리는 **30초 안에 당신의 역할을 정의**합니다. 끝까지 읽으세요.

## 30초 요약

- **저장소**: Jominho의 `ai-study` 위키 ([SPEC.md](../../SPEC.md) 참조)
- **목적**: Jominho의 두 프로젝트(mino-moneyflow, mino-tarosaju)에서 검증된 AI 엔지니어링 방법론을 *당신이 즉시 실행할 수 있는 형태*로 박제

- **당신의 역할**: (a) 위키를 읽고 *현재 작업에 적용*, (b) 작업 결과를 *다음 에이전트가 읽을 수 있게* 박제, (c) 워커 프로젝트에서 발견한 패턴을 허브로 전달 (`/curate-inbound`)

## iOS 하네스 환경 구축 — 실전 베스트 프랙티스

회사 iOS 프로젝트에 AI 에이전트 하네스를 처음 도입할 때의 **실전 가이드**. ai-study 위키와 3개 워커 프로젝트에서 검증된 패턴만 수록.

### iOS 하네스 5대 구성요소

#### 1. CLAUDE.md — 프로젝트 컨텍스트 (최우선)

iOS 프로젝트의 CLAUDE.md에 반드시 포함할 항목:

- **아키텍처 패턴**: RIBs, MVVM, TCA, Clean Architecture 등. 에이전트가 새 파일을 만들 때 *어디에 어떤 레이어로* 배치할지 결정하는 기준
- **모듈 구조**: SPM multi-module인지, CocoaPods인지, 혼합인지. 모듈 간 의존 방향 명시
- **금지 패턴**: force unwrap 정책(`!` 사용 범위), Rx vs `async`/`await` 혼용 규칙, `@objc` 남용 금지 등
- **배포 흐름**: fastlane인지 Xcode Cloud인지, TestFlight 배포 단계, 인증서 관리 방식
- **핵심 의존성과 용도**: `"Moya — 네트워크 추상화"`, `"SnapKit — Auto Layout DSL"` 같은 1줄 설명
- **이 프로젝트에서 AI가 절대 하면 안 되는 것** 섹션: 예) `"Podfile 직접 수정 금지"`, `"Core Data 마이그레이션 자율 실행 금지"`, `"인증서/프로비저닝 프로파일 건드리지 말 것"`

CLAUDE.md가 없으면 에이전트는 *아키텍처를 추측*한다. 추측은 반드시 실패한다.

#### 2. `.claudeignore` — 컨텍스트 오염 차단

iOS 프로젝트에서 필수로 제외할 패턴:

```
**/.build/          # SPM 모듈별 빌드 캐시 (모듈당 6-7GB)
vendor/             # CocoaPods vendor
*.xcodeproj/        # Xcode XML (generated, 수천 줄)
Pods/               # CocoaPods dependencies
DerivedData/        # Xcode derived data
.bundle/            # Ruby bundler
external-spm/       # 외부 SPM 패키지 미러
Gemfile.lock        # Ruby lockfile
```

`**/.build/`가 `.build/`가 아닌 이유: SPM multi-module 프로젝트에서는 *각 모듈 디렉토리 안에* `.build/`가 생긴다. 루트의 `.build/`만 제외하면 `Modules/FeatureA/.build/`, `Modules/FeatureB/.build/` 등이 전부 컨텍스트에 유입된다. 모듈이 20개면 120GB 이상이 컨텍스트를 오염시킨다. glob 패턴 `**/.build/`는 *모든 깊이의* `.build/` 디렉토리를 재귀적으로 제외한다.

#### 3. `.githooks/pre-commit` — 코드 규칙 자동 게이트

에이전트가 작성한 코드가 팀 규칙을 통과하는지 *커밋 시점에* 자동 검증:

```bash
#!/bin/bash
# SwiftFormat: 자동 교정 + 재스테이지
swift run swiftformat . --swiftversion 5.9
git diff --name-only --cached -- '*.swift' | xargs git add

# SwiftLint: 에러 0 + 경고 100 이하
LINT_ERRORS=$(swift run swiftlint lint --quiet 2>/dev/null | grep "error:" | wc -l)
LINT_WARNINGS=$(swift run swiftlint lint --quiet 2>/dev/null | grep "warning:" | wc -l)
if [ "$LINT_ERRORS" -gt 0 ]; then
  echo "SwiftLint errors: $LINT_ERRORS"
  exit 1
fi
if [ "$LINT_WARNINGS" -gt 100 ]; then
  echo "SwiftLint warnings: $LINT_WARNINGS (max 100)"
  exit 1
fi
```

팀에 맞게 `setup.sh`를 만들어 온보딩 시 한 번 실행:

```bash
#!/bin/bash
git config core.hooksPath .githooks
chmod +x .githooks/*
echo "Git hooks activated"
```

이 훅이 없으면 에이전트는 SwiftLint 위반을 *무한 반복*한다. 훅이 커밋을 차단하면 에이전트가 *스스로 교정*한다.

#### 4. 슬래시 커맨드 7종 — 워크플로우 구조화

`.claude/commands/` 디렉토리에 배치하는 iOS 전용 커맨드:

| 커맨드 | 역할 | 핵심 |
|---|---|---|
| `/ios-triage` | 5 라운드 탐정식 인터뷰 | 라운드당 최대 3 질문, 브리프 생성 |
| `/ios-investigate` | 4단계 근본 원인 추적 | 재현 - 격리 - 식별 - 수정 |
| `/ios-bugfix` | 5 Phase 버그 수정 | 티켓 - 분석 - 수정 - 리뷰 - 테스트 |
| `/ios-review` | 2 Agent 병렬 리뷰 | Reviewer + Defender 동시 실행 |
| `/ios-arch` | 3 Agent 아키텍처 토론 | A옹호 + B옹호 + Judge |
| `/ios-ship` | TestFlight 배포 6 Phase | 버전 - CHANGELOG - fastlane |
| `/compound` | 3 Agent 지식 축적 | CHANGELOG + 솔루션 + 회고 |

각 커맨드는 `.claude/commands/ios-*.md` 파일로 정의한다. 단순 프롬프트가 아니라 **Phase를 정의**하는 것이 핵심 — 에이전트에게 "분석해줘"가 아니라 "Phase 1에서 재현, Phase 2에서 격리, Phase 3에서 근본 원인 식별"처럼 *구조화된 사고 흐름*을 강제한다.

#### 5. 테스트 하네스 — 5단계 체크리스트

1. **SwiftLint/SwiftFormat pre-commit 훅** (위 3번) — 코드 스타일 자동 교정
2. **XCTest 유닛** — Protocol 기반 DI + `async`/`await` 테스트
3. **SPM 모듈 간 계약 테스트** — 모듈 경계의 public API가 변경되면 테스트 실패
4. **Snapshot 테스트** — `swift-snapshot-testing`으로 UI 회귀 탐지
5. **CI 게이트** — GitHub Actions에서 `xcodebuild test` 실행, exit code 직접 검사

상세: [iOS AI 코드 품질 보증 전략](/wiki/ios-ai/ios-ai-code-quality-test-strategy)

새 iOS 프로젝트에 하네스를 세팅할 때 에이전트가 순서대로 실행:

1. **`CLAUDE.md` 작성** — 아키텍처 + 모듈 구조 + 금지 패턴 + 배포 흐름 (10분)
2. **`.claudeignore` 생성** — `**/.build/` 필수, 위 패턴 목록 복사 (2분)
3. **`.githooks/pre-commit` + `setup.sh`** — SwiftLint/SwiftFormat 게이트 (5분)
4. **`.claude/commands/`에 7개 슬래시 커맨드 복사** — `ios-*.md` 파일 배치 (5분)
5. **`.claude/settings.json` hooks 설정** — pre-commit 훅 경로 지정 (3분)
6. **첫 `/compound` 실행으로 베이스라인 문서 생성** — 현 상태 스냅샷 (5분)
7. **`swift package show-dependencies`로 모듈 그래프 확인** — 즉시

7단계 완료 후 에이전트는 *구조화된 환경*에서 작업을 시작한다. 구조 없이 시작하면 위 5대 함정에 *반드시* 빠진다.

## Web 하네스 환경 구축 (Next.js 기준)

이 위키 자체가 Web(Next.js) 프로젝트이므로 간결하게 핵심만 정리.

### 핵심 구성요소

1. **`CLAUDE.md`** — Tech stack, 페이지 목록, API routes, 배포 흐름. 이 위키의 CLAUDE.md가 *레퍼런스 구현*
2. **`.claudeignore`** — `node_modules/`, `package-lock.json`, `.next/`, `src/generated/` 등 생성 파일
3. **`ai-review.yml`** — Push 시 자동 PR + Test Gate + Squash Merge. 3개 프로젝트에서 운용 중
4. **`validate-content.mjs`** — MDX/Mermaid 문법 사전 검증 + JSX 함정 탐지 (중괄호, void tag 등)
5. **`vitest` + `npm run build`** — CI 게이트. exit code *직접* 검사, pipe 금지 (Shell pipe 함정 참조)

### Web 하네스 핵심 차이점 (vs iOS)

| 차이점 | Web (Next.js) | iOS (Swift/SPM) |
|---|---|---|
| 빌드 검증 | `npm run build` (SSG 프리렌더) | `xcodebuild test` (시뮬레이터) |
| 린트 게이트 | ESLint + Prettier | SwiftLint + SwiftFormat |
| 배포 | Vercel auto-deploy (push = 배포) | fastlane + TestFlight |
| 모듈 시스템 | npm packages | SPM multi-module |
| 컨텍스트 폭발 원인 | `node_modules/` (거대하지만 단일) | `**/.build/` (모듈당 6-7GB, 분산) |
| 콘텐츠 검증 | MDX 문법 + JSX 함정 | Swift 컴파일러가 대부분 잡음 |

Web은 *배포가 push = 즉시 반영*이므로 pre-commit 게이트가 더 중요하다. iOS는 TestFlight 리뷰 단계가 있어 *빌드 실패를 한 번 더 잡을 기회*가 있지만, Web은 그 기회가 없다.

## 기본 계약 — 10가지

1. **세션 시작 3개 파일 로드**: CLAUDE.md → SPEC.md → 이 엔트리
2. **추측 금지**: 직접 확인한 것만 박제
3. **인용구 규칙**: `"..."`는 *원본에서 글자 단위 직접 확인*한 것만
4. **새 작업**: `/wt-branch`로 worktree 분기 (squash merge 함정 회피)
5. **매 사이클 끝**: `/compound`로 박제
6. **외부 자료**: `/ingest`로 교차 검증 후 박제
7. **UI 변경**: 브라우저 QA 필수 (claude-in-chrome javascript_tool로 errorBoundary 확인)
8. **API 변경**: 자동 게이트로 충분 (test.yml / test-gate.yml)
9. **보호 파일 금지**: `.env*`, `supabase/migrations/*`, `package.json` deps, `.claude/settings.json` (자율 수정 X)
10. **append-only 갱신**: 원본 섹션 보존, 갱신 로그 append

## 자율 모드의 경계 — 반드시 인식

자율 모드가 활성화되어 있어도 다음은 *자율 처리 영역이 아님*:

| 영역 | 이유 |
|---|---|
| 다른 세션이 동시에 만지는 파일 | 물리적 충돌 위험 |
| 코드 의도 결정이 필요한 충돌 해결 (cherry-pick conflict 등) | 사용자 판단 필요 |
| 보호 파일 수정 | 모든 세션 공유 룰 |
| 외부 서비스 설정 (GitHub branch protection 등) | 사용자 GitHub UI 작업 필요 |
| 민감 데이터 접근 (`.env` 값) | 보안 경계 |

*내가 처리할 수 없는 것*을 *사용자에게 알리는 것*도 자율의 일부.

## For AI Agents

**Trigger**: 모든 새 세션의 시작 지점. 특정 작업을 시작하기 *전*에 이 엔트리의 상황별 라우팅을 먼저 확인한다.

**Prerequisites**: 없음. *이 엔트리가 프리리퀴짓의 시작*.

**Actionable Steps**:
1. CLAUDE.md 읽기 (프로젝트 규약)
2. SPEC.md 읽기 (엔티티 + 계약)
3. 이 엔트리의 상황별 라우팅에서 *현재 작업 상황*에 해당하는 섹션 찾기
4. 해당 섹션의 엔트리 목록을 *순서대로* 읽기
5. 각 엔트리의 `For AI Agents` 섹션이 있으면 *체크리스트*로 사용
6. iOS 프로젝트면 **iOS 하네스 5대 구성요소** + **Day 1 체크리스트** 참조
7. Web 프로젝트면 **Web 하네스 환경 구축** 참조
8. 기본 계약 10가지 + 자율 모드 경계 확인
9. 작업 진행 중 의사결정이 필요한 순간마다 *관련 엔트리*를 다시 참조
10. 결과를 박제 (Harness Journal 또는 새 엔트리 또는 기존 엔트리 갱신 로그)

**Anti-patterns**:
- 이 엔트리만 읽고 CLAUDE.md / SPEC.md 생략
- 상황에 맞지 않는 라우팅 임의 선택
- 자율 모드 경계를 무시하고 *모든 것*을 자율 처리
- 박제 없이 작업 끝내기 (다음 에이전트가 학습 불가)
- 이 엔트리를 *한 번만* 읽고 잊기 (새 세션마다 반드시 재방문)
- iOS 하네스에서 `.claudeignore` 없이 시작 (컨텍스트 폭발 보장)

## 이 엔트리가 *살아있는 문서*입니다

위키가 성장하면 이 엔트리의 상황별 라우팅도 갱신됩니다. 새 엔트리가 추가되면 *관련 상황*에 링크 추가. 새 시리즈가 시작되면 *새 섹션* 추가.

갱신은 append-only. 원본 섹션은 *그대로 보존*. 변경 시 `## 갱신 로그` 섹션에 날짜 + 사유 명시.

## 자기 점검

1. 당신이 지금 *어떤 작업*을 시작하려는가? 위의 상황별 라우팅 중 *어디에 해당*하는가?
2. 세션 시작 3개 파일을 *모두* 로드했는가? (CLAUDE.md / SPEC.md / 이 엔트리)
3. 기본 계약 10가지 중 *가장 자주 위반*하는 것은 무엇인가? (박제하고 교정)
4. *자율 모드의 경계*를 5개 모두 외울 수 있는가? 못 외우면 이 엔트리를 북마크.
5. (열린 질문) 당신이 작업을 끝낼 때 *다음 에이전트가 읽을 엔트리*는 어느 것인가?

### 실습 과제

세션을 시작한 직후, 다음을 *각 2분 안에* 수행:

1. CLAUDE.md 읽기 (2분)
2. SPEC.md 읽기 (2분)
3. 이 엔트리 읽기 (2분)
4. 현재 작업 상황에 해당하는 라우팅 섹션의 *첫 번째 엔트리*로 이동 (1분)
5. 해당 엔트리의 `For AI Agents` 섹션 확인 (1분)

**총 8분**. 이 8분이 *사고 발생률*을 절반으로 줄인다. 측정된 것이 아니라 Compound Engineering의 *사람의 기억력에 의존하는 지식은 반드시 실패* 원칙의 직접 결과.
