## 먼저 읽으세요

안녕하세요. 당신이 *AI 에이전트*라면 지금 읽고 있는 이 엔트리가 **위키의 시작점**입니다. Jominho의 *LLM-First Wiki*에 방문해 주셔서 감사합니다.

이 엔트리는 **30초 안에 당신의 역할을 정의**합니다. 끝까지 읽으세요.

## 30초 요약

- **저장소**: Jominho의 `ai-study` 위키 (SPEC.md 참조)
- **목적**: Jominho의 3 프로덕트(mino-moneyflow, mino-tarosaju, aidy 4레포)에서 검증된 AI 엔지니어링 방법론을 *당신이 즉시 실행할 수 있는 형태*로 박제
- **주요 시리즈**: [Harness Journal](/harness-journal) — AI 운영 환경 개선 사이클 기록 (000~024, 24 에피소드) + [iOS Journal](/harness-journal) (000~009, 10 에피소드) + [Aidy Journal](/harness-journal) (000~009, 10 에피소드)
- **당신의 역할**: (a) 위키를 읽고 *현재 작업에 적용*, (b) 작업 결과를 *다음 에이전트가 읽을 수 있게* 박제, (c) 워커 프로젝트에서 발견한 패턴을 허브로 전달 (`/curate-inbound`)

```
1. CLAUDE.md                                  — 프로젝트 규약 + LLM-First 방향성
```

## iOS 하네스 환경 구축 — 실전 베스트 프랙티스

iOS 프로덕션 프로젝트에 AI 에이전트 하네스를 처음 도입할 때의 **실전 가이드**. ai-study 위키와 3개 워커 프로젝트에서 검증된 패턴만 수록.

### iOS 하네스 5대 구성요소

#### 1. CLAUDE.md — 프로젝트 컨텍스트 (최우선)

iOS 프로젝트의 CLAUDE.md에 반드시 포함할 항목:

- **아키텍처 패턴**: RIBs, MVVM, TCA, Clean Architecture 등. 에이전트가 새 파일을 만들 때 *어디에 어떤 레이어로* 배치할지 결정하는 기준
- **모듈 구조**: SPM multi-module인지, CocoaPods인지, 혼합인지. 모듈 간 의존 방향 명시
- **금지 패턴**: force unwrap 정책(`!` 사용 범위), Rx vs `async`/`await` 혼용 규칙, `@objc` 남용 금지 등
- **배포 흐름**: fastlane인지 Xcode Cloud인지, TestFlight 배포 단계, 인증서 관리 방식
- **핵심 의존성과 용도**: `"Moya — 네트워크 추상화"`, `"SnapKit — Auto Layout DSL"` 같은 1줄 설명
- **이 프로젝트에서 AI가 절대 하면 안 되는 것** 섹션: 예) `"Podfile 직접 수정 금지"`, `"Core Data 마이그레이션 자율 실행 금지"`, `"인증서/프로비저닝 프로파일 건드리지 말 것"`

CLAUDE.md가 없으면 에이전트는 *아키텍처를 추측*한다. 추측은 반드시 실패한다.

#### 2. `.claudeignore` — 컨텍스트 오염 차단

iOS 프로젝트에서 필수로 제외할 패턴:

```
**/.build/          # SPM 모듈별 빌드 캐시 (모듈당 6-7GB)
vendor/             # CocoaPods vendor
*.xcodeproj/        # Xcode XML (generated, 수천 줄)
Pods/               # CocoaPods dependencies
DerivedData/        # Xcode derived data
.bundle/            # Ruby bundler
external-spm/       # 외부 SPM 패키지 미러
Gemfile.lock        # Ruby lockfile
```

`**/.build/`가 `.build/`가 아닌 이유: SPM multi-module 프로젝트에서는 *각 모듈 디렉토리 안에* `.build/`가 생긴다. 루트의 `.build/`만 제외하면 `Modules/FeatureA/.build/`, `Modules/FeatureB/.build/` 등이 전부 컨텍스트에 유입된다. 모듈이 20개면 120GB 이상이 컨텍스트를 오염시킨다. glob 패턴 `**/.build/`는 *모든 깊이의* `.build/` 디렉토리를 재귀적으로 제외한다.

#### 3. `.githooks/pre-commit` — 코드 규칙 자동 게이트

에이전트가 작성한 코드가 팀 규칙을 통과하는지 *커밋 시점에* 자동 검증:

```bash
#!/bin/bash
# SwiftFormat: 자동 교정 + 재스테이지
swift run swiftformat . --swiftversion 5.9
git diff --name-only --cached -- '*.swift' | xargs git add

# SwiftLint: 에러 0 + 경고 100 이하
LINT_ERRORS=$(swift run swiftlint lint --quiet 2>/dev/null | grep "error:" | wc -l)
LINT_WARNINGS=$(swift run swiftlint lint --quiet 2>/dev/null | grep "warning:" | wc -l)
if [ "$LINT_ERRORS" -gt 0 ]; then
  echo "SwiftLint errors: $LINT_ERRORS"
  exit 1
fi
if [ "$LINT_WARNINGS" -gt 100 ]; then
  echo "SwiftLint warnings: $LINT_WARNINGS (max 100)"
  exit 1
fi
```

팀에 맞게 `setup.sh`를 만들어 온보딩 시 한 번 실행:

```bash
#!/bin/bash
git config core.hooksPath .githooks
chmod +x .githooks/*
echo "Git hooks activated"
```

이 훅이 없으면 에이전트는 SwiftLint 위반을 *무한 반복*한다. 훅이 커밋을 차단하면 에이전트가 *스스로 교정*한다.

#### 4. 슬래시 커맨드 7종 — 워크플로우 구조화

### iOS 하네스 고급 — 딥리서치 베스트 프랙티스 (2026)

커뮤니티와 공식 문서에서 검증된 iOS + Claude Code 꿀팁. 우리 프로젝트 경험과 외부 리서치를 결합.

#### `.pbxproj` 절대 수정 금지 (가장 중요한 규칙)

Xcode 프로젝트 파일(`.pbxproj`)은 수천 줄의 복잡한 XML이다. AI가 이 파일을 직접 수정하면 **프로젝트가 열리지 않는** 치명적 손상이 발생한다. 반드시 CLAUDE.md에 명시:

```
## 절대 금지
- *.pbxproj 파일 직접 수정 금지. 파일은 Claude가 생성하되, Xcode에서 수동으로 Add Files to Project.
- SPM Package.swift는 수정 가능하나, pbxproj는 절대 불가.
```

대안: SPM 기반 프로젝트 구조를 채택하면 `Package.swift`로 모든 의존성을 관리할 수 있어 AI 친화적.

#### XcodeBuildMCP — 2개 MCP 서버 조합이 최적

상세 가이드: [iOS 하네스 셋업 완전 가이드](/wiki/ios-ai/ios-harness-setup-xcodebuildmcp)

| 서버 | 도구 | 특징 | 적합한 경우 |
|------|------|------|-------------|
| **XcodeBuildMCP** (Sentry) | 59+ | headless, Xcode 불필요 | CI/headless, 자율 빌드-수정 루프 |
| **Apple Xcode MCP** (26.3+) | 20 | Xcode 실행 필요 | SwiftUI Preview, 문서 검색, REPL |

**핵심 가치**: xcodebuild Bash 호출 시 수천 줄 로그가 컨텍스트를 오염시킴. XcodeBuildMCP는 **구조화된 JSON**(파일:줄번호:에러)을 반환 → 에이전트가 즉시 수정 가능. 빌드 루프: 8-10분 → 90초.

**설치 불필요한 경우**: 대규모 프로젝트(SPM 30+모듈)에서 cold build가 구조적으로 느리거나, 개발자가 수동 빌드+결과 붙여넣기 정책인 경우.

#### CLAUDE.md 500줄 이하 유지

CLAUDE.md는 **매 턴마다** 컨텍스트에 로드된다. 500줄을 넘으면 중요한 지시가 희석된다. 핵심만 남기고, 상세 가이드는 `.claude/commands/` 슬래시 커맨드나 별도 문서로 분리.

```
# CLAUDE.md 크기 기준
- 이상적: 100-300줄 (핵심 규칙만)
- 허용: 300-500줄 (상세 포함)
- 위험: 500줄 초과 (지시 희석 + 토큰 낭비)
```

#### `settings.json` `permissions.deny` — `.claudeignore`보다 강력한 보호

`.claudeignore`는 자동 파일 탐색에서 제외하지만, 명시적 검색으로는 여전히 접근 가능하다. 진짜 민감한 파일은 `settings.json`의 `permissions.deny`로 **읽기 자체를 차단**:

```json
{
  "permissions": {
    "deny": [
      "Read(*.env*)",
      "Read(Secrets.swift)",
      "Edit(*.pbxproj)",
      "Write(*.pbxproj)"
    ]
  }
}
```

이렇게 하면 Claude가 해당 파일을 읽으려고 시도하면 에러가 반환된다.

#### PostToolUse 자동 린트 훅

`Write`/`Edit` 후 자동으로 SwiftLint를 실행하면 에이전트가 *수정 즉시* 규칙 위반을 인지한다. pre-commit 훅과 다른 레이어 — pre-commit은 커밋 시점이지만 PostToolUse는 *파일 수정 직후*.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [{
          "type": "command",
          "command": "f=$(echo $CLAUDE_TOOL_INPUT | jq -r '.file_path // empty'); [ -n \"$f\" ] && [[ \"$f\" == *.swift ]] && RESULT=$(swiftlint lint --path \"$f\" --quiet 2>/dev/null); if echo \"$RESULT\" | grep -q 'error:'; then echo \"$RESULT\" | grep 'error:' | head -5 >&2; exit 2; fi; exit 0"
        }]
      }
    ]
  }
}
```

**exit code 규칙**: error 감지 시 `exit 2`(차단 + 자동수정 루프). warning만이면 `exit 0`(통과). `exit 1`은 비차단 경고이므로 보안/품질 Hook에서는 **반드시 exit 2** 사용.

#### SwiftUI 프로젝트가 AI와 가장 궁합이 좋다

외부 벤치마크에 따르면:
- **SwiftUI only**: ~60% 개발 시간 단축
- **UIKit + SwiftUI 혼합**: ~35-40% 단축
- **UIKit + Storyboard**: AI가 `.storyboard` XML을 정확하게 수정하기 어려워 효과 제한적

신규 화면은 가능하면 SwiftUI로 작성하도록 CLAUDE.md에 명시하면 AI 생산성이 극대화된다.

#### Swift 버전 + 최소 배포 타겟 명시

Swift 6의 strict concurrency는 기존 코드와 호환성 이슈를 만든다. CLAUDE.md에 반드시:

```
## 환경
- Swift: 5.9 (Swift 6 strict concurrency 미적용)
- 최소 배포: iOS 16.0
- Xcode: 16.2
```

이것만 써도 에이전트가 `@Sendable` 강제, `nonisolated` 요구 등 Swift 6 전용 패턴을 추가하는 실수를 방지한다.

#### `swift package show-dependencies` — 1초 그래프 검증

SPM 모듈 의존성 수정 후 **빌드를 돌리지 않고** 의존 그래프를 즉시 검증:

```bash
cd Modules/Common
swift package show-dependencies --format flatlist 2>/dev/null | sort -u
# 여기에 제거하려는 모듈이 없으면 성공
```

빌드는 수 분 걸리지만 이 명령은 1초. 특히 transitive dependency 제거 작업에서 **대체 불가능한 검증 도구**.

### 핵심 구성요소

1. **`CLAUDE.md`** — Tech stack, 페이지 목록, API routes, 배포 흐름. 이 위키의 CLAUDE.md가 *레퍼런스 구현*

| 차이점 | Web (Next.js) | iOS (Swift/SPM) |
|---|---|---|

| 모듈 시스템 | npm packages | SPM multi-module |

|---|---|---|
| `last_verified` | 마지막으로 내용이 검증된 날짜 | 오래된 날짜면 적용 전 현재 코드와 교차 확인 |
| `applicable_to` | 적용 범위 (`["any"]` / `["aidy", "any"]` 등) | 현재 작업 프로젝트와 맞지 않으면 패턴만 참고 |
| `generated_by` | `gemini` 등 AI 자동 생성 표시 | 자동 생성 엔트리는 사람 검토 여부 불확실 — 비판적으로 읽기 |
| `confidence` | 저자의 체화 수준 (1~5) | 3 미만이면 보조 자료 교차 확인 권장 |

이 상태는 *스냅샷*이다. 최신 큐는 [Harness Journal 000 베이스라인의 갱신 로그](/wiki/harness-engineering/harness-journal-000-baseline#갱신-로그) 확인.

## AI Agent Directive

**Trigger**: 새 세션 시작 직후, 특정 작업을 시작하기 *전*에 현재 상황에 맞는 라우팅 경로를 찾아야 할 때. CLAUDE.md와 SPEC.md 로드 후 가장 먼저 참조.

**Prerequisites**: 없음. *이 엔트리가 모든 프리리퀴짓의 시작 지점*.

### Actionable Steps
1. CLAUDE.md (프로젝트 규약) 읽기 - 5분
2. SPEC.md (엔티티 + 계약) 읽기 - 5분
3. 이 엔트리의 상황별 라우팅 12개 섹션 중 *현재 작업*에 해당하는 것 찾기
4. 해당 섹션의 엔트리 목록을 *순서대로* 읽기
5. 각 엔트리의 AI Agent Directive 섹션 확인 - 체크리스트로 활용
6. 플랫폼별(iOS/Web) 하네스 5대 구성요소 + Day 1 체크리스트 확인
7. 기본 계약 10가지 + 자율 모드 경계 5가지 인식
8. 작업 진행 중 의사결정 마다 관련 엔트리 재참조
9. 작업 완료 후 결과 박제 (Journal 또는 엔트리 갱신)
10. 새 세션마다 이 엔트리를 *다시 읽기*

### Anti-patterns
- ❌ CLAUDE.md / SPEC.md 생략하고 이 엔트리만 읽기
- ❌ 상황에 맞지 않는 라우팅 임의로 선택하기
- ❌ 자율 모드 경계 무시하고 모든 것을 자율 처리
- ❌ 박제 없이 작업 끝내기 (다음 에이전트 학습 불가)
- ❌ 한 번 읽고 잊기 - 새 세션마다 재참조 필수

## For AI Agents

**Trigger**: 모든 새 세션의 시작 지점. 특정 작업을 시작하기 *전*에 이 엔트리의 상황별 라우팅을 먼저 확인한다.

**Prerequisites**: 없음. *이 엔트리가 프리리퀴짓의 시작*.

**Actionable Steps**:
1. CLAUDE.md 읽기 (프로젝트 규약)
2. SPEC.md 읽기 (엔티티 + 계약)
3. 이 엔트리의 상황별 라우팅에서 *현재 작업 상황*에 해당하는 섹션 찾기
4. 해당 섹션의 엔트리 목록을 *순서대로* 읽기
5. 각 엔트리의 `For AI Agents` 섹션이 있으면 *체크리스트*로 사용
6. iOS 프로젝트면 **iOS 하네스 5대 구성요소** + **Day 1 체크리스트** 참조
7. Web 프로젝트면 **Web 하네스 환경 구축** 참조
8. 기본 계약 10가지 + 자율 모드 경계 확인
9. 작업 진행 중 의사결정이 필요한 순간마다 *관련 엔트리*를 다시 참조
10. 결과를 박제 (Harness Journal 또는 새 엔트리 또는 기존 엔트리 갱신 로그)

**Anti-patterns**:
- 이 엔트리만 읽고 CLAUDE.md / SPEC.md 생략
- 상황에 맞지 않는 라우팅 임의 선택
- 자율 모드 경계를 무시하고 *모든 것*을 자율 처리
- 박제 없이 작업 끝내기 (다음 에이전트가 학습 불가)
- 이 엔트리를 *한 번만* 읽고 잊기 (새 세션마다 반드시 재방문)
- iOS 하네스에서 `.claudeignore` 없이 시작 (컨텍스트 폭발 보장)

## 이 엔트리가 *살아있는 문서*입니다

위키가 성장하면 이 엔트리의 상황별 라우팅도 갱신됩니다. 새 엔트리가 추가되면 *관련 상황*에 링크 추가. 새 시리즈가 시작되면 *새 섹션* 추가.

갱신은 append-only. 원본 섹션은 *그대로 보존*. 변경 시 `## 갱신 로그` 섹션에 날짜 + 사유 명시.

## 자기 점검

1. 당신이 지금 *어떤 작업*을 시작하려는가? 위의 상황별 라우팅 중 *어디에 해당*하는가?
2. 세션 시작 3개 파일을 *모두* 로드했는가? (CLAUDE.md / SPEC.md / 이 엔트리)
3. 기본 계약 10가지 중 *가장 자주 위반*하는 것은 무엇인가? (박제하고 교정)
4. *자율 모드의 경계*를 5개 모두 외울 수 있는가? 못 외우면 이 엔트리를 북마크.
5. (열린 질문) 당신이 작업을 끝낼 때 *다음 에이전트가 읽을 엔트리*는 어느 것인가?

### 실습 과제

세션을 시작한 직후, 다음을 *각 2분 안에* 수행:

1. CLAUDE.md 읽기 (2분)
2. SPEC.md 읽기 (2분)
3. 이 엔트리 읽기 (2분)
4. 현재 작업 상황에 해당하는 라우팅 섹션의 *첫 번째 엔트리*로 이동 (1분)
5. 해당 엔트리의 `For AI Agents` 섹션 확인 (1분)

**총 8분**. 이 8분이 *사고 발생률*을 절반으로 줄인다. 측정된 것이 아니라 Compound Engineering의 *사람의 기억력에 의존하는 지식은 반드시 실패* 원칙의 직접 결과.
