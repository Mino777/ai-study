# JSON 프롬프팅 vs 구조화된 마크다운 프롬프팅

> 같은 내용이라도 **어떤 형식으로 프롬프트를 작성하느냐**에 따라 LLM의 이해도, 토큰 비용, 출력 품질이 달라진다.
> "형식은 중요하지 않다"는 통념은 틀렸다.

---

## 핵심 비교

| 차원 | Markdown (+XML tags) | JSON |
|------|---------------------|------|
| **토큰 효율** | 34-38% 적음 | 구문 문자(중괄호, 따옴표, 콜론)가 토큰 소비 |
| **지시 명확성** | XML 태그로 섹션 구분 명확 | 스키마가 명시적이라 파싱 모호성 없음 |
| **가독성** | 사람이 읽기 쉬움 (GitHub/Notion 표준) | 구조적이지만 장황함 |
| **출력 검증** | 후처리 필요 | 스키마 기반 검증 가능 |
| **모델 선호** | GPT-4, Claude가 선호 | GPT-3.5가 선호 |
| **성능 편차** | 안정적 | 모델에 따라 최대 40% 편차 |
| **창의적 작업** | 자연스러움 유지 | 기술 모드로 전환되어 품질 저하 |

---

## 왜 형식이 성능에 영향을 주는가

LLM은 학습 데이터의 형식 분포에 의존한다.

- **Markdown**: GitHub README, 기술 문서, 포럼 등 대량의 학습 데이터가 마크다운 형식
- **JSON**: API 문서, 설정 파일 등 상대적으로 좁은 도메인
- **XML**: 웹 페이지 소스, 기술 문서에서 빈번

모델이 자주 본 형식일수록 해당 형식의 의미를 더 정확하게 파악한다. Markdown은 대부분의 LLM이 가장 많이 학습한 구조화 형식이다.

---

## 모델별 형식 선호도

동일 작업에서 형식만 바꿨을 때의 성능 차이가 유의미하게 존재한다.

| 모델 | 선호 형식 | 비고 |
|------|----------|------|
| **GPT-4** | Markdown | 대형 모델은 형식에 강건하지만 Markdown에서 약간 우위 |
| **GPT-3.5-turbo** | JSON | 소형 모델은 형식 민감도 높음, 형식에 따라 최대 40% 성능 차이 |
| **Claude** | Markdown + XML tags | Anthropic 공식 권장. XML 태그로 섹션 구분 |
| **Gemini 2.5 Flash** | YAML | JSON 대비 17.7%p 성능 우위 |
| **Llama 3.2 3B** | 형식 무관 | 모든 형식에서 유사한 성능 |

**핵심**: 모든 모델에 같은 형식을 적용하는 것은 차선이다. 모델별 테스트가 필요.

---

## 각 형식의 구체적 장단점

### Markdown + XML tags

**강점**:
- LLM 학습 데이터에서 가장 빈번한 구조화 형식
- 토큰 효율 34-38% 우위 (같은 내용 기준)
- 기존 문서를 변환 없이 프롬프트에 직접 사용 가능
- git diff에서 사람이 읽기 쉬움
- RAG에서 헤딩 기반 청킹 시 검색 정밀도 최대 35% 향상

**약점**:
- 출력 형식을 엄격하게 강제할 수 없음 (후처리 필요)
- 모호성을 피하려면 프롬프트 엔지니어링에 더 신경 써야 함

**Anthropic 권장 패턴**:

```xml
<instructions>
## 역할
당신은 코드 리뷰어입니다.

## 규칙
- 보안 취약점을 최우선으로 확인
- 성능 문제는 구체적 수치와 함께 지적
</instructions>

<context>
{{코드 내용}}
</context>

<output_format>
## 발견 사항
(심각도별 정리)
</output_format>
```

### JSON

**강점**:
- 파싱 모호성 제로 — 스키마가 곧 계약
- Structured Outputs(OpenAI)로 99%+ 스키마 준수 가능
- 기계 간 데이터 교환에 최적
- 다운스트림 시스템과 직접 연결 가능

**약점**:
- 토큰 비효율 (중괄호, 따옴표, 콜론이 각각 토큰)
- 모델이 기술 모드로 전환 — 창의적 작업 품질 저하
- 유지보수 시 스키마 + 프롬프트 이중 관리
- JSON 이스케이핑이 복잡해지면 사람이 읽기 어려움

**적합한 사용 예**:

```json
{
  "response_format": {
    "type": "json_schema",
    "json_schema": {
      "name": "review_result",
      "schema": {
        "type": "object",
        "properties": {
          "severity": { "enum": ["critical", "warning", "info"] },
          "description": { "type": "string" },
          "line_number": { "type": "integer" }
        },
        "required": ["severity", "description"]
      }
    }
  }
}
```

### YAML

**강점**:
- JSON과 유사한 구조성, Markdown에 가까운 가독성
- Gemini 2.5 Flash에서 JSON 대비 17.7%p 성능 우위
- 들여쓰기 기반이라 괄호/따옴표 오버헤드 없음

**약점**:
- 들여쓰기 민감 — 공백 하나로 파싱 실패
- LLM이 YAML을 생성할 때 들여쓰기 에러 빈번
- JSON 대비 에코시스템 지원이 약함

## 하이브리드 접근법 — 실전 권장

현실에서는 하나의 형식만 쓰지 않는다. **입력은 Markdown, 출력은 JSON**이 가장 실용적인 조합이다.

```
┌─────────────────────────────────────────────┐
│  프롬프트 입력 (Markdown + XML tags)         │
│  → 토큰 효율적, 사람이 읽기 쉬움             │
│  → 모델이 의미를 잘 파악                     │
├─────────────────────────────────────────────┤
│  LLM 추론                                   │
├─────────────────────────────────────────────┤
│  출력 (JSON / Structured Output)             │
│  → 다운스트림 시스템이 파싱 가능              │
│  → 스키마 검증으로 신뢰성 보장                │
└─────────────────────────────────────────────┘
```

**왜 이 조합인가:**
- 입력: 토큰 절감 + 모델 이해도 최대화 (Markdown의 강점)
- 출력: 파싱 신뢰성 + 검증 가능성 (JSON의 강점)
- 각 형식의 강점만 취하고 약점을 상쇄

## 벤더별 공식 권장 비교

| 벤더 | 프롬프트 입력 형식 | 출력 형식 | 핵심 가이드 |
|------|------------------|----------|-----------|
| **Anthropic (Claude)** | Markdown + XML tags | 자유 / JSON | XML 태그로 섹션 구분, 최소한의 프롬프트에서 시작 후 실패 기반 추가 |
| **OpenAI (GPT)** | 자연어 + 예시 | Structured Outputs (스키마 강제) | 스키마 + 1개 예시 + 검증 지시 4계층 |
| **Google (Gemini)** | 자연어 | JSON Schema (responseSchema) | 스키마는 responseSchema에만, 프롬프트에 중복 금지 |

**공통점**: 세 벤더 모두 프롬프트 입력에 JSON을 권장하지 않는다. JSON은 출력 형식으로만 권장.

## 판단 기준 — 언제 어떤 형식을 쓸 것인가

### Markdown + XML tags를 쓸 때
- 시스템 프롬프트, 에이전트 지시사항 작성
- 컨텍스트 주입 (문서, 코드, 배경 정보)
- 토큰 효율이 중요한 대량 처리
- Claude 또는 GPT-4 사용
- 사람이 프롬프트를 읽고 수정해야 하는 팀 환경

### JSON / Structured Outputs를 쓸 때
- 출력을 다운스트림 시스템이 파싱해야 할 때
- 스키마 준수가 비즈니스 요구사항일 때
- OpenAI Structured Outputs를 사용할 때
- 데이터 추출 작업 (temperature 0.0-0.1 권장)

### YAML을 쓸 때
- Gemini 2.5 Flash가 주력 모델일 때
- 중첩 구조가 있지만 스키마 강제까지는 불필요할 때
- JSON의 구조성과 Markdown의 가독성 중간이 필요할 때

---

## 실전 체크리스트

1. **모델 확인**: 사용 모델의 형식 선호도를 먼저 파악
2. **입출력 분리**: 입력은 Markdown, 출력이 파싱 필요하면 JSON
3. **토큰 계산**: 같은 프롬프트를 JSON과 Markdown으로 작성해서 토큰 수 비교
4. **A/B 테스트**: 형식만 바꿔서 동일 작업 수행 → 품질 비교
5. **벤더 가이드 확인**: 모델 업데이트마다 공식 권장이 바뀔 수 있음

## For AI Agents

### Trigger
- 프롬프트 형식을 결정해야 할 때
- 토큰 비용 최적화가 필요할 때
- 새 모델로 전환 시 프롬프트 형식 재검토

### Prerequisites
1. 사용 모델 확인
2. 입력/출력 요구사항 파악 (파싱 필요 여부)

### Actionable Steps
1. 위 판단 기준 표에서 현재 상황에 맞는 형식 선택
2. 하이브리드 접근(입력 Markdown + 출력 JSON) 우선 고려
3. 모델별 형식 선호도 표 참조하여 형식 조정
4. 토큰 수 비교로 비용 영향 추정

### Anti-patterns
- ❌ 모든 모델에 동일 형식 적용 — 모델별 최대 40% 성능 차이
- ❌ 프롬프트 입력에 JSON 사용 — 세 벤더 모두 비권장
- ❌ 형식 테스트 없이 감으로 선택 — A/B 테스트 필수
- ❌ 출력에 Markdown 강제 후 정규식 파싱 — Structured Outputs 사용
