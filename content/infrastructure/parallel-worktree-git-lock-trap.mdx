## 갭

Claude Code의 Agent tool에서 `isolation: "worktree"`로 N개의 에이전트를 동시에 띄우면, 모두 `git worktree add`를 실행한다. 이때 `.git/config.lock` race condition으로 **첫 번째만 성공하고 나머지 전부 실패**하는 함정이 있다.

에러 메시지:
```
error: could not lock config file .git/config: File exists
error: unable to write upstream branch configuration
fatal: unable to write to database
```

## 근본 원인

Git은 `.git/config` 수정 시 atomic write를 위해 lock file(`.git/config.lock`)을 생성한다. 하지만:

1. Process A가 `.git/config.lock` 획득 → 수정 시작
2. Process B, C, D가 동시에 같은 lock을 찾으려 함
3. Lock이 이미 존재하면 **retry 없이 즉시 실패** (exponential backoff 없음)

Git은 single-threaded config write를 가정하고 설계됨. OS 레벨의 flock만 사용하며, 동시 접근 시 lock manager/MVCC 같은 보호가 없다.

## 회피 패턴

### 패턴 1: 순차 생성 + 병렬 작업 (권장)

Worktree 생성은 순차, 작업은 병렬:

```bash
# ✅ 권장: worktree만 순차 생성
git worktree add /tmp/agent-1 -b feat-1 && \
git worktree add /tmp/agent-2 -b feat-2 && \
git worktree add /tmp/agent-3 -b feat-3

# 이후 각 worktree에서 병렬 작업 (lock 경합 없음)
```

### 패턴 2: non-worktree 모드

Agent tool에서 `isolation: "worktree"`를 쓰지 않고 로컬 브랜치로 작업:

```bash
# worktree 대신 브랜치만 생성 (config.lock 불필요)
git checkout -b feat-1
```

### 패턴 3: 사전 정리 + 순차 재시도

```bash
# 1. stale lock 확인 + 제거
ls -la .git/config.lock  # 존재 확인
git worktree prune        # dangling 참조 정리
rm -f .git/config.lock    # stale lock 제거

# 2. worktree 순차 생성
git worktree add /tmp/agent-1 -b feat-1
sleep 1
git worktree add /tmp/agent-2 -b feat-2
```

## 예방 체크리스트

병렬 에이전트 실행 전 매번:

- [ ] `.git/config.lock` 존재 여부 확인
- [ ] `git worktree list`로 stale worktree 점검
- [ ] `git worktree prune` 실행 (dangling 정리)
- [ ] **Sequential vs Parallel 의사결정**: worktree 생성은 반드시 순차
- [ ] CI/CD matrix job은 별도 runner라 이 문제 해당 없음 (로컬만 위험)

## 교훈

- **Concurrent git worktree creation은 unsupported** — git 내부 설계가 lock 경합 시 재시도 없이 즉시 실패
- **Stale lock은 신호** — `.git/config.lock` 발견 시 이전 프로세스가 비정상 종료된 것
- **CI matrix vs 로컬 병렬**: GitHub Actions matrix job은 각각 별도 runner라 충돌 없음. 로컬 병렬 실행만 위험

## AI Agent Directive

**Trigger**: Agent tool로 N개의 에이전트를 `isolation: "worktree"`로 동시에 띄우려 할 때. 또는 `.git/config.lock` 에러 발생 시.

**Prerequisites**: [wt-branch 슬래시 커맨드](/wiki/harness-engineering/harness-journal-004-wt-branch-command)

### Actionable Steps
1. 병렬 에이전트가 필요하면 worktree 생성은 **순차**로 실행
2. 생성 완료 후 각 worktree에서의 작업은 **병렬** 진행
3. `.git/config.lock` 에러 발생 시: prune → lock 제거 → 순차 재시도
4. 대안: `isolation: "worktree"` 대신 일반 브랜치 모드 사용

### Anti-patterns
- N개 에이전트를 동시에 `isolation: "worktree"`로 띄우기 (첫 번째만 성공)
- `.git/config.lock`을 확인 없이 즉시 삭제 (진행 중인 프로세스일 수 있음)
- lock 에러 후 같은 명령 재시도 (원인 해결 없이 반복)
