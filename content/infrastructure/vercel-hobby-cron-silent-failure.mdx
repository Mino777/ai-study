## 사건

mino-moneyflow에서 Telegram 아침 브리핑, 장중 모니터링, 시그널 워치 등 **모든 cron 알림이 오지 않았다.** "요즘 텔레그램으로 뭐 안 오던데" 라는 신고로 조사 시작.

결론: **3개의 독립적 실패가 겹쳐서 진단이 불가능한 상태였다.**

## 함정 1: Vercel Hobby Cron 2개 제한

`vercel.json`에 12개 cron을 등록했다:

```json
{
  "crons": [
    { "path": "/api/alerts/check", "schedule": "0 9 * * *" },
    { "path": "/api/scanner/cron", "schedule": "0 8 * * *" },
    { "path": "/api/digest/morning", "schedule": "0 0 * * *" },
    // ... 총 12개
  ]
}
```

**Hobby 플랜은 프로젝트당 2개만 실행한다.** 나머지 10개는:
- 빌드 에러 없음
- 대시보드에 12개 모두 표시됨
- 실행 기록만 없음 (확인하지 않으면 모름)
- 에러 로그 0

처음 2개(`alerts/check` + `scanner/cron`)는 Telegram을 안 보내는 route → **Telegram 알림이 한 번도 온 적 없을 가능성.**

### 교훈

> **등록 ≠ 실행.** 배포 후 Vercel Dashboard에서 각 cron의 "Last Run" 타임스탬프를 반드시 확인해야 한다.

| 플랜 | Cron 제한 |
|---|---|
| Hobby | 2개/프로젝트 (일 1회) |
| Pro | 40개/프로젝트 |
| Enterprise | 무제한 |

## 함정 2: Telegram HTML Escape 누락

Pro로 전환해 12개 cron이 모두 돌아도, 메시지 전송은 여전히 실패했을 것이다:

```typescript
// 문제의 데이터
const INDICES = [
  { symbol: '^GSPC', name: 'S&P 500' },  // & 문자!
]

// parse_mode: 'HTML'로 전송
await fetch('https://api.telegram.org/bot.../sendMessage', {
  body: JSON.stringify({
    text: `📊 S&P 500: $4,500 (+0.5%)`,  // & escape 안 됨
    parse_mode: 'HTML',
  }),
})
```

Telegram HTML 파서 규칙: 태그 밖의 `<`, `>`, `&`는 **반드시** `&lt;`, `&gt;`, `&amp;`로 escape.

`S&P 500` → `&P`가 유효하지 않은 entity → **400 Bad Request**.

### 수정

```typescript
export function escapeHtml(text: string): string {
  return text
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
}

// 사용: 모든 동적 텍스트에 적용
lines.push(`${escapeHtml(q.name)}: ${priceStr}`)
```

## 함정 3: fetch resp.ok 미체크

함정 2가 있어도, 에러가 Sentry에 기록됐다면 빨리 발견했을 것이다. 하지만:

```typescript
// 🔴 위험: fetch는 HTTP 에러를 throw하지 않는다
try {
  await fetch('https://api.telegram.org/...')
  // 400 Bad Request도 여기서 정상 완료!
} catch (e) {
  logError(e)  // 네트워크 에러만 잡힘. HTTP 400은 안 잡힘.
}
```

**400 에러가 조용히 무시됨 → Sentry 기록 0 → "에러 없는데 왜 안 올까?"**

### 수정

```typescript
// ✅ 안전: resp.ok 체크 + 에러 로깅
const resp = await fetch('https://api.telegram.org/...')
if (!resp.ok) {
  const body = await resp.text().catch(() => '')
  logError(new Error(`Telegram ${resp.status}: ${body.slice(0, 200)}`))
  return false
}
```

## 일반화: "에러가 없다 ≠ 정상이다"

이 사건의 핵심은 **silent failure의 겹침**이다. 각 레이어가 독립적으로 조용히 실패하면 진단이 exponential하게 어려워진다.

### 안전한 패턴

1. **등록 후 실행 확인**: cron, webhook, scheduled job → 배포 후 1회 실행 확인 필수
2. **fetch 후 resp.ok**: HTTP 에러 ≠ network 에러. 구분해서 처리
3. **외부 서비스 응답 로깅**: 성공만 로깅하지 말고, 실패도 명시적으로 로깅
4. **복붙 코드 → 중앙 모듈**: 같은 함수가 N곳에 있으면 버그도 N곳에 있다

### fetch 안전 패턴 (모든 외부 호출에 적용)

```typescript
async function safeFetch(url: string, options: RequestInit, scope: string) {
  try {
    const resp = await fetch(url, {
      ...options,
      signal: AbortSignal.timeout(10_000),  // 무한 hang 차단
    })
    if (!resp.ok) {
      const body = await resp.text().catch(() => '')
      logError(new Error(`HTTP ${resp.status}: ${body.slice(0, 200)}`), { scope })
      return null
    }
    return resp
  } catch (e) {
    logError(e, { scope })  // network / timeout
    return null
  }
}
```

## AI Agent Directive

**Trigger**: "모든 cron이 제대로 작동하는데 실제로는 일부만 실행 중" / Telegram · Slack 같은 외부 알림이 간헐적으로 안 올 때 / "에러가 없는데 왜 결과가 없지?" 상황
**Prerequisites**: [Cron Execution Tracking Pattern](/wiki/infrastructure/cron-execution-tracking-pattern) — silent failure 감지 방법 / [Fixture Silent Failure Trap](/wiki/harness-engineering/fixture-silent-failure-trap) — 에러를 삼키는 패턴

### Actionable Steps
1. **Vercel Hobby 제약 인식** — 2개 cron만 실행. `vercel.json`에 12개 등록해도 3~12번째는 조용히 무시
2. **배포 후 'Last Run' 확인** — Vercel 대시보드에서 각 cron의 마지막 실행 시각 확인 (계획과 실제 다를 수 있음)
3. **fetch resp.ok 필수 체크** — `fetch()` 호출 후 HTTP 4xx/5xx는 throw되지 않음 · 반드시 `if (!resp.ok)` 검증
4. **외부 서비스 응답 로깅** — 성공만 로깅하지 말고, 실패 시 상태 코드 + 응답 본문도 기록
5. **복붙 코드 금지** — 같은 함수가 여러 route에 있으면 버그도 N곳에 퍼짐 · 중앙 모듈로 통합
6. **HTML escape 강제** — Telegram HTML parse_mode 사용 시 `&`, `<`, `>` 반드시 `&amp;`, `&lt;`, `&gt;`로 변환
7. **DB 이력 기록** — Vercel 로그는 신뢰 불가 · `cron_execution_history` 테이블에 모든 실행을 기록

### Anti-patterns
- **Hobby 플랜에 10개 이상 cron** — 일부가 조용히 무시됨. Pro 플랜 필수
- **배포 후 Last Run 확인 안 함** — 등록 ≠ 실행. 실제 동작 확인 필수
- **fetch 에러 처리 없음** — HTTP 에러도 catch되지 않으므로 에러 처리 코드 불필요 착각 → 버그
- **Telegram/Slack 전송 후 응답 무시** — 400 에러도 로그가 안 남음 · resp.ok 체크 필수
- **인증 · 로깅이 각 route마다 다름** — 표준 패턴 부재 · 일부는 에러 삼키고 있을 가능성
