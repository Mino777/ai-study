## 왜 이 글이 Flow Map 이 아닌가

Flow Map 시리즈 5편의 원래 기획은 "aidy 풀스택 배포 파이프라인 — 로컬 → Neon/Vercel → 프로덕션" 이었지만, **2026-04-16 현재 aidy 3-client 중 어느 것도 실제 프로덕션 배포가 구축돼 있지 않다**. aidy-server · aidy-ios · aidy-android 모두 로컬 개발 단계에 머물러 있고, 각 레포의 `.github/workflows/` 에는 `ai-review.yml` (Claude AI code review) 만 존재 — 실 배포 워크플로 0건.

**Flow Map 장르의 전제** 는 "실재하는 코드/파이프라인을 읽으며 큰 그림을 잡는 역방향 학습" 이다. 존재하지 않는 파이프라인을 Flow Map 으로 쓰면 *aspirational 공상* 이 되어 "실 코드 기반" 이라는 시리즈 계약을 위반한다.

대신 이 글은 **설계 로드맵** 장르로 — "앞으로 어떻게 쌓을 것인가" 의 의사결정 순서를 미리 박제. **ADR(Architecture Decision Record) 과 같은 성격의 선제적 기록**. Confidence 를 2 로 유지해서 "아직 실측 아님" 을 명시.

## 한 줄 요약

aidy 3-client (Server Kotlin + iOS SwiftUI/TCA + Android Compose) 가 로컬 단일 머신 개발에서 다중 환경 프로덕션으로 전환할 때의 **의사결정 순서** 와 각 단계의 **4 조건 게이트** (토큰 카탈로그와 동일 원리) 적용을 정리. 핵심 결론 3가지:

1. **DB → Server → CI → Secrets → Observability → Client 채널** 순서로 1개씩 도입 · 병렬 금지
2. **managed service 기본값** (단일 운영자 체제에서 자체 호스팅은 학습 프로젝트로 분리)
3. **각 단계에서 베이스라인 측정 + 롤백 경로 명문화** — "일단 도입" 금지

## 1. 현재 상태 — 정확한 스냅샷

### aidy-server (Spring Boot + Kotlin)
- 로컬 개발: `docker compose up -d` (PostgreSQL 17) + `./gradlew bootRun` (localhost:8080)
- DB: PostgreSQL 17 Docker 컨테이너 (Flyway V1~V8 마이그레이션)
- Secrets: shell env (`DB_PASSWORD`, `CLAUDE_API_KEY`, `JWT_SECRET` 등)
- 배포 상태: **로컬만**. 프로덕션 URL 없음. TLS 없음
- 관측: logback + Request-Id + 구조화 로그 · 외부 로그 수집 없음

### aidy-ios (Swift/TCA + Tuist)
- 로컬 개발: `tuist generate` + Xcode 시뮬레이터
- 서버 URL: `http://localhost:8080` (Info.plist ATS 예외로 cleartext 허용)
- 배포 상태: **로컬 시뮬레이터만**. TestFlight 등 배포 채널 미구축
- Secrets: Keychain (런타임 JWT 저장)

### aidy-android (Kotlin/Compose)
- 로컬 개발: Android Studio + 에뮬레이터
- 서버 URL: `http://10.0.2.2:8080` (에뮬레이터 → 호스트 loopback) 또는 로컬 IP
- 배포 상태: **로컬 디바이스만**. Play Console · Firebase App Distribution 미구축
- Secrets: EncryptedSharedPreferences

### aidy-architect (관제)
- 로컬 markdown specs · tmux 4-pane · 각 워커 세션
- 배포 개념 없음 (개발 조정 저장소)

### 결론
*"프로덕션" 이라는 단어가 이 상태에서 무의미* — 아무도 쓸 수 없다. 모든 의사결정은 아직 미래 항목.

## 2. 목표 상태 — 최소 생존 프로덕션 (MSP) 정의

"엔지니어 1명이 혼자 운영 · 사용자 0 ~ 10 명 초기 베타 규모" 에서 지속 가능한 MSP 를 먼저 정의. 이상적 구성이 아니라 **운영 가능한 최소 구성**.

### 서버
- HTTPS 도메인 (e.g. `aidy.example.com`)
- 24/7 가동 (하루 몇 번 flaky 재시작 허용되지 않음)
- 로그 → 외부 수집기 (grep 가능한 곳)
- 에러 알림 (운영자 iPhone 에 알림)

### DB
- managed PostgreSQL
- 자동 백업 (최소 일 1회)
- 최소 1 복구 시나리오 검증 완료

### iOS
- TestFlight 배포 (본인 + 1~2 명 지인 초대)
- 외부 베타 불필요 (당분간)
- 크래시 리포팅

### Android
- Firebase App Distribution (가장 빠른 경로) 또는 Play Internal Testing
- 크래시 리포팅

### 공통 운영
- Secrets 관리 (shell env 탈피 — Vault · Doppler · 1Password CLI · 또는 managed service 기본 secret store)
- 최소 1 dashboard 로 3-client 의 건강 상태 1 화면에 보임
- "운영자가 8시간 잠드는 동안 무슨 일이 일어났는지" 아침에 1분 내 파악 가능

> 💡 **핵심 관찰**: 목표는 "프로덕션 급 완벽한 운영" 이 아니라 "로컬 개발이 끝난 직후 붙일 수 있는 **가장 얇은 운영 레이어**". 복잡한 것은 사용자가 생긴 다음에.

## 3. 컴포넌트별 선택지와 판단 기준

### 3.1 DB 호스팅

| 옵션 | 장점 | 단점 | 단일 운영자 적합도 |
|------|------|------|------|
| **Neon** (서버리스 PostgreSQL) | branching (브랜치별 DB 사본) · 자동 스케일 · generous free tier · 설정 거의 0 | 소수 쿼리 패턴 특이성 (cold start) | ⭐⭐⭐⭐⭐ |
| **Supabase** | Postgres + Auth + Storage 번들 | 필요 없는 기능까지 번들 · vendor lock-in | ⭐⭐⭐ |
| **자체 PostgreSQL (VPS)** | 학습 가치 · 완전 제어 | 백업 · 업데이트 · 장애 모두 본인 부담 | ⭐⭐ |
| **AWS RDS** | 안정적 · enterprise 표준 | 가격 · 복잡도 · overkill | ⭐⭐ |

**추천**: **Neon**. 4 조건 게이트 통과:
- 공식/커뮤니티 검증 ✅
- 실측 가능 (연결 레이턴시 · 쿼리 속도) ✅
- 부작용 관찰 가능 (cold start 는 로그에 드러남) ✅
- 롤백 비용 낮음 (pg_dump 로 덤프 → 다른 Postgres 로 이동) ✅

**이식 고려**: Flyway 마이그레이션은 그대로 작동. JDBC URL 만 교체.

### 3.2 서버 호스팅

| 옵션 | 장점 | 단점 | 단일 운영자 적합도 |
|------|------|------|------|
| **Fly.io** | JVM 앱 native 지원 · 멀티 리전 · Dockerfile 기반 · free tier | 리소스 한계 · 빌드 외부 CI 필요 | ⭐⭐⭐⭐ |
| **Railway** | deploy 극단적으로 쉬움 · Postgres 내장 옵션 · generous free | 요금 예측 어려움 (usage 기반) | ⭐⭐⭐⭐ |
| **Render** | clean UI · auto-scaling · managed Postgres 옵션 | 콜드 스타트 존재 (free) | ⭐⭐⭐ |
| **Vercel** (서버리스 함수) | 프런트엔드 통합 · edge | **Spring Boot 부적합** (Vercel 은 JS/Go/Python 중심) | ⭐ |
| **AWS Elastic Beanstalk** | enterprise | 복잡도 · 가격 | ⭐⭐ |
| **자체 VPS (Hetzner/DigitalOcean)** | 값싸고 자유 | OS 유지보수 · TLS · nginx 설정 본인 부담 | ⭐⭐ |

**추천**: **Fly.io** (1차) 또는 **Railway** (2차). 둘 다 Dockerfile 기반이라 이식성 높음. **Vercel 은 aidy-server 에 부적합** — Spring Boot 는 JVM 지속 실행 서버라 서버리스 함수 모델과 mismatch.

**4 조건 게이트**:
- Fly.io: ✅✅✅✅ 네 조건 모두 통과
- Railway: ✅✅✅✅ 동일

선택 차이는 **빌드 정책** — Fly 는 CI 에서 빌드 후 push 모델이 표준, Railway 는 git push 트리거. 로컬 개발자에겐 Railway 가 약간 더 쉬움. 안정성/예측성은 Fly 가 약간 우위.

### 3.3 iOS 배포

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|------|
| **TestFlight** (Apple 공식) | App Store Connect 기본 · iOS 개발자에게 익숙 | Apple 리뷰 지연 (외부 베타만) | ⭐⭐⭐⭐⭐ |
| **Firebase App Distribution** | 즉시 배포 · 리뷰 없음 | Apple 서명 설정 별도 | ⭐⭐⭐ |
| **Diawi / Enterprise distribution** | 특수 경우 | 복잡 · 가격 | ⭐⭐ |

**추천**: **TestFlight 내부 테스터** — 본인 + 지인 10명까지는 리뷰 없이 즉시. 외부 100명 베타 필요해지면 그때 외부 테스트 그룹 신청.

**파이프라인**:
- Xcode `archive` → `upload to App Store Connect` (자동 처리 권장: **Fastlane + Match** 또는 **Tuist signing** + CI)
- 1차엔 Xcode 수동 업로드 → 10+ 배포 누적되면 CI 자동화 도입

### 3.4 Android 배포

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|------|
| **Firebase App Distribution** | 즉시 배포 · APK/AAB 모두 · 테스터 invite 링크 | 구글 계정 필요 | ⭐⭐⭐⭐⭐ |
| **Play Console Internal Testing** | Play Store 네이티브 · 서명 키 통합 | 초기 설정 복잡 · 처음 검수 | ⭐⭐⭐⭐ |
| **Direct APK 배포** | 완전 자유 | 보안 위험 · 사이드로딩 | ⭐⭐ |

**추천**: **Firebase App Distribution** 1차. Play Store 출시 단계 오면 Internal Testing 추가.

### 3.5 Secrets 관리

| 옵션 | 장점 | 단점 | 적합도 |
|------|------|------|------|
| **호스팅 플랫폼 기본 secret store** (Fly secrets · Railway env · Vercel env) | 플랫폼 통합 · 설정 1줄 | 플랫폼 lock-in | ⭐⭐⭐⭐ |
| **1Password CLI** | 개발자 경험 탁월 · 팀 공유 | 러닝 커브 | ⭐⭐⭐⭐ |
| **Doppler** | multi-env · 로테이션 | 별도 서비스 | ⭐⭐⭐ |
| **HashiCorp Vault** | enterprise | overkill | ⭐ |

**추천**: **Fly/Railway 내장 secrets** 1차. 팀 규모 3+ 되면 **1Password CLI** 도입. 지금은 운영자 혼자이므로 플랫폼 secrets 가 적절.

### 3.6 Observability

| 레이어 | 옵션 | 추천 |
|--------|------|------|
| **로그 수집** | Logtail / Axiom / Datadog / 자체 ELK | **Logtail** (가격 · 설정 간단) |
| **APM** | Datadog APM / New Relic / Honeycomb / Sentry Tracing | **Sentry Performance** (에러와 통합) |
| **에러 알림** | Sentry / Rollbar / Bugsnag | **Sentry** (3-client 모두 SDK 있음) |
| **Uptime** | Better Stack / UptimeRobot / Pingdom | **Better Stack** (free tier 충분) |
| **대시보드** | Grafana / Datadog / 자체 웹 | **Grafana Cloud** (free tier) |

**추천 스타터 번들**: **Sentry + Logtail + Better Stack** — 3 서비스로 에러 · 로그 · uptime 커버. 각 free tier 로 초기 운영 가능. 복잡해지면 하나씩 교체.

## 4. 도입 순서 — 4 조건 게이트 적용

**원칙**: 한 번에 한 컴포넌트. 매 단계 베이스라인 측정 + 롤백 경로 문서화.

### Stage 0: 사전 준비 (1~2일)
- [ ] 도메인 구입 (`aidy.example.com` 같은 주 도메인)
- [ ] DNS 관리 선택 (Cloudflare 권장)
- [ ] TLS 인증서 전략 (플랫폼 자동 · Let's Encrypt)
- [ ] 배포 체크리스트 문서 작성 (이 글의 Week 1~N 로드맵 기반)

### Stage 1: DB 마이그레이션 (3~5일)
- [ ] Neon 계정 생성 + Free tier 프로젝트 생성
- [ ] Flyway 마이그레이션 로컬에서 덤프 → Neon 으로 임포트 테스트
- [ ] **Shadow mode**: 로컬 앱이 local Postgres 와 Neon 양쪽에 write · 결과 비교 1주
- [ ] 전환 스위치: JDBC URL 환경변수만 교체
- [ ] 베이스라인 측정: 쿼리 latency · cold start 빈도 · connection pool 안정성
- [ ] 롤백 문서: pg_dump 복구 절차 · local 로 복귀 방법 1문단

### Stage 2: 서버 호스팅 전환 (5~7일)
- [ ] Fly.io 계정 + `fly launch` 로 첫 deploy
- [ ] Dockerfile 최적화 (Spring Boot native image 고려 vs 일반 JVM)
- [ ] `fly secrets set` 으로 환경변수 이관
- [ ] HTTPS · 도메인 연결
- [ ] Health check 엔드포인트 `/api/health` 가 작동하는지 확인
- [ ] **Shadow mode**: 클라가 local 과 production 양쪽 호출 · 응답 비교
- [ ] 전환: 클라이언트 `baseURL` 교체 (단 하나의 설정 변경)
- [ ] 베이스라인: Rate Limit 실측 · Request-Id 추적 production 에서 작동 확인
- [ ] 롤백: `fly scale count 0` 으로 비활성화 · 클라 baseURL 로컬 복귀

### Stage 3: CI 통합 (3~5일)
- [ ] GitHub Actions 워크플로 작성 (`deploy.yml`)
- [ ] Branch 전략: `main` push → 자동 deploy vs 수동 approve
- [ ] 테스트 Gate: `./gradlew test` PASS 시에만 deploy 진행
- [ ] 베이스라인: deploy 시간 · 롤백 시간 · 빈도
- [ ] 롤백: 이전 커밋 SHA 로 `fly deploy --image` 롤백 절차

### Stage 4: Secrets 공식화 (2~3일)
- [ ] 모든 env var 를 `fly secrets list` 로 감사
- [ ] 로컬 `.env.example` 과 production secrets 비교표 작성
- [ ] rotation 정책: `CLAUDE_API_KEY` · `JWT_SECRET` 3개월 1회 교체 루틴
- [ ] 베이스라인: secret 누출 감사 (GitHub secret scanning · gitleaks)
- [ ] 롤백: secret 교체 실패 시 이전 값 즉시 재주입 절차

### Stage 5: Observability 도입 (5~7일)
- [ ] Sentry 통합 (서버 + iOS + Android 3 SDK)
- [ ] Logtail 연결 (서버 logback 설정)
- [ ] Better Stack uptime monitor 2개 (`/api/health` + 랜딩 도메인)
- [ ] 운영자 iPhone 에 Sentry · Better Stack 알림 설정
- [ ] 베이스라인: MTTD (Mean Time To Detect) · 첫 알람 수신 확인
- [ ] 롤백: Sentry 통합 제거는 SDK init 호출 주석 처리 수준

### Stage 6: iOS TestFlight (5~7일)
- [ ] Apple Developer Program 멤버십 (연 $99)
- [ ] App Store Connect 앱 생성
- [ ] Archive + Upload 수동 1회 체험
- [ ] 내부 테스터 초대 (본인 + 지인 2~3명)
- [ ] 빌드 자동 증가 규약 (`agvtool` 또는 Tuist)
- [ ] 베이스라인: archive 시간 · Apple 처리 시간 · 첫 설치까지
- [ ] 롤백: 이전 빌드 버전으로 테스터 재설치

### Stage 7: Android Firebase App Distribution (3~5일)
- [ ] Firebase 프로젝트 생성 + Android 앱 등록
- [ ] Firebase CLI 로 APK/AAB 업로드
- [ ] 테스터 그룹 생성 + invite
- [ ] 베이스라인: 업로드 시간 · 테스터 설치 경험
- [ ] 롤백: 이전 빌드 재배포

### Stage 8: 통합 대시보드 (2~3일)
- [ ] Better Stack heartbeat 페이지 (서버 + 2 클라 빌드 최신 정보)
- [ ] Grafana Cloud 대시보드: Rate Limit 발동 · Circuit Breaker 상태 · DB connection pool
- [ ] 운영자 북마크 1개로 전체 상태 1 화면 파악 가능

**총 소요 (예상)**: ~35일 (주 10 시간 작업 가정). 각 Stage 는 독립적 — 한 Stage 실패 시 다음 Stage 진행 안 함.

## 5. Architect 세션이 이 로드맵을 지휘하는 법

aidy 의 Architect-Worker 모델에서 배포는 "코드 작성" 이 아니라 "인프라 결정 + 워커 지시" 영역이라 Architect 책임:

| 단계 | Architect 역할 | 워커 역할 |
|------|--------------|---------|
| **Stage 결정** | 옵션 비교 · 선택 근거 ADR 작성 | 없음 |
| **WO 발행** | `WO-INFRA-NNN-{stage}-{component}.md` 발행 | 각 스택 워커가 해당 설정 변경 구현 |
| **Shadow mode** | 데이터 비교 로직 지시 | server 워커 구현 |
| **전환 스위치** | 결정 · 실행 명령 | 워커가 설정 변경 커밋 |
| **롤백 문서** | 절차 작성 | 없음 |
| **베이스라인 측정** | 측정 지표 정의 | 워커가 데이터 수집 |

**ADR 템플릿 사용** — 각 Stage 의 선택(Neon vs Supabase, Fly vs Railway 등) 은 `specs/decisions/ADR-INFRA-NNN-*.md` 에 박제.

## 6. 학습 로드맵 (Week 1~N)

### Week 1: 현재 상태 정확히 재확인
- [ ] 각 레포의 README · CLAUDE.md 읽고 "지금 뭐가 되고 뭐가 안 되는지" 문서화
- [ ] 이 글의 "현재 상태 스냅샷" 섹션을 본인 레포 기준으로 다시 작성
- [ ] 3-client 가 프로덕션에서 작동하려면 반드시 있어야 할 요소 10개 나열 (이 글 목차 기반)

### Week 2: 목표 MSP 명세 확정
- [ ] 사용자 수 · 예산 · 운영자 capa · SLA 요구사항 한 페이지로 정리
- [ ] 위 표에서 각 옵션의 4 조건 게이트 통과 여부 직접 평가
- [ ] ADR 1호 작성: "왜 managed service 기본값인가" 근거 박제

### Week 3: Stage 1 (DB) POC
- [ ] Neon 또는 Supabase 계정 생성
- [ ] 로컬 Flyway 마이그레이션 하나를 매니지드에 적용
- [ ] JDBC URL 교체 실험 (앱 재기동 필요한지)
- [ ] 결과 실측 + 다음 Stage 진행 여부 판단

### Week 4+: Stage 2~8 순차 진행
각 Stage 종료 시 `/compound` 로 retro · solution · ADR 박제.

## 7. 안티패턴

| 안티패턴 | 왜 실패 | 대안 |
|---------|--------|------|
| **한 주말에 Neon + Fly + TestFlight + Sentry 모두 셋업** | 어느 요소가 이슈인지 식별 불가 · 롤백 어려움 | Stage 1 → 7 순차 |
| **'나중에 Vercel 로 옮기면 되지' 로 Spring Boot → Vercel 고려** | 플랫폼 특성 mismatch · 재작성 비용 | Fly/Railway 등 JVM 친화 |
| **Shadow mode 없이 프로덕션 전환** | 문제 발생 시 사용자에게 직접 노출 | 로컬 · 프로덕션 동시 호출 · 차이 비교 후 전환 |
| **자체 호스팅 PostgreSQL 로 시작** | 새벽 장애 · 백업 실수 · 업데이트 부담 | managed (Neon/Supabase) |
| **Secrets 를 `.env` 커밋** | 기밀 노출 · 로테이션 불가 | 플랫폼 secrets · 1Password CLI |
| **Observability 를 마지막에** | 초기 버그를 놓침 · 사용자가 먼저 발견 | Stage 5 위치 고수 — 서버 트래픽 받기 전에 관측 완비 |
| **'사용자 100명 되면 그때 우려'** | 첫 1명이 첫 버그 발견 시 대응 채널 0 | MSP 부터 관측 · 알림 필수 |
| **여러 observability 도구 동시 도입** | 설정 중복 · 알림 노이즈 | Sentry → Logtail → Better Stack 순서 |
| **iOS Fastlane · CI 를 처음부터** | 아직 TestFlight 업로드 2회도 안 해봄 · 과잉 | 10회 수동 업로드 후 자동화 도입 |
| **'완벽한 Kubernetes 프로덕션' 목표** | 단일 운영자에 overkill · 배우는 시간 > 만드는 시간 | PaaS (Fly · Railway) 기본값 |

## AI Agent Directive

**Trigger**: 풀스택 프로젝트가 "로컬만 된다" 상태에서 실제 배포를 고민하기 시작할 때 / "플랫폼 선택이 맞나?" 의문 / 운영 체계 미확립
**Prerequisites**: [Architect Flow Map (Part 4)](/wiki/harness-engineering/architect-flow-map-via-aidy-architect) — Architect 책임 영역 / [Vercel Hobby Cron Silent Failure](/wiki/infrastructure/vercel-hobby-cron-silent-failure) — 플랫폼 제약 실패 사례

### Actionable Steps
1. **현재 상태 문서화** — 3 레포 README와 `.github/workflows/` 훑고 '무엇이 실제로 되고 안 되는지' 1페이지 정확히 기록
2. **MSP 먼저 정의** — 사용자 수 · SLA · 예산 · 단일 운영자 capa를 명문화 (목표 없으면 옵션 비교 불가)
3. **Stage별 순차 진행** — DB(Stage 1) → Server(2) → CI(3) → Secrets(4) → Observability(5) → 클라 배포(6-7) 순서 고수
4. **Shadow mode 강제** — 각 Stage 전환 시 로컬과 프로덕션 동시 실행 후 차이 비교
5. **각 Stage의 베이스라인과 롤백 절차 명문화** — "일단 도입" 금지 · 언제든 복귀 가능해야 함
6. **ADR 작성** — 각 선택(Neon vs Supabase, Fly vs Railway)의 근거와 Trade-off 박제
7. **Observability는 Stage 5에서 고수** — 첫 사용자 받기 전에 반드시 완비

### Anti-patterns
- **한 번에 여러 Stage 병행** — 어느 요소가 실패했는지 식별 불가
- **관측 없이 프로덕션 전환** — 첫 사용자가 버그 발견 (downtime)
- **Vercel에 Spring Boot 올리기** — 플랫폼 특성 mismatch · 재작성 비용
- **자체 호스팅 PostgreSQL로 시작** — 새벽 장애 운영 부담 > 프로젝트 가치
- **Shadow mode 스킵** — 차이를 프로덕션에서 발견
- **Secrets를 .env 또는 CI env로 관리** — 로테이션 불가 · 노출 위험
