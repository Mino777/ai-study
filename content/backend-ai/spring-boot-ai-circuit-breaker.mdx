## 갭 / 맥락

Aidy 서버는 Claude API에 의존한다. 외부 API가 일시적으로 실패할 때 무한 재시도는 상황을 악화시킨다 (rate limit 증폭, 비용 증가, 사용자 대기).

Spring Boot 생태계에는 [Resilience4j](https://resilience4j.readme.io/) 같은 성숙한 CB 라이브러리가 있지만, **ADR-007에서 Phase 1 범위를 "단일 인스턴스 + 0 dependency"로 명시**하고 in-memory 구현을 선택했다.

## 핵심 패턴

### 3상태 머신

```
CLOSED (정상)
  │ sliding window에서 failure ≥ threshold
  ▼
OPEN (차단) ── cooldown 경과 ──→ HALF_OPEN (시험)
  ▲                                  │
  │ 1회 실패                        │ 1회 성공
  └──────────────────────────────────┘
                                      ▼
                                  CLOSED (복구)
```

### Kotlin 구현 (Spring `@Component`)

```kotlin
@Component
class AiCircuitBreaker(
    private val clock: Clock = Clock.systemDefaultZone(),
) {
    enum class State { CLOSED, OPEN, HALF_OPEN }

    private val state = AtomicReference(State.CLOSED)
    private val failures = CopyOnWriteArrayList<Instant>()
    private var openUntil: Instant? = null

    companion object {
        const val FAILURE_THRESHOLD = 5
        val WINDOW_DURATION: Duration = Duration.ofMinutes(1)
        val COOLDOWN: Duration = Duration.ofSeconds(30)
    }

    fun <T> execute(block: () -> T): T {
        val currentState = transitionIfNeeded()
        if (currentState == State.OPEN) {
            throw CircuitOpenException("AI service circuit is OPEN")
        }

        return try {
            val result = block()
            onSuccess()
            result
        } catch (e: Exception) {
            onFailure()
            throw e
        }
    }

    private fun transitionIfNeeded(): State {
        val now = clock.instant()
        val current = state.get()

        if (current == State.OPEN && openUntil?.isBefore(now) == true) {
            state.compareAndSet(State.OPEN, State.HALF_OPEN)
            return State.HALF_OPEN
        }
        return current
    }

    private fun onSuccess() {
        val current = state.get()
        if (current == State.HALF_OPEN) {
            failures.clear()
            openUntil = null
            state.compareAndSet(State.HALF_OPEN, State.CLOSED)
        }
    }

    private fun onFailure() {
        val now = clock.instant()
        failures.add(now)

        // Sliding window 정리
        val windowStart = now.minus(WINDOW_DURATION)
        failures.removeIf { it.isBefore(windowStart) }

        val current = state.get()

        when (current) {
            State.CLOSED -> {
                if (failures.size >= FAILURE_THRESHOLD) {
                    openUntil = now.plus(COOLDOWN)
                    state.compareAndSet(State.CLOSED, State.OPEN)
                }
            }
            State.HALF_OPEN -> {
                openUntil = now.plus(COOLDOWN)
                state.compareAndSet(State.HALF_OPEN, State.OPEN)
            }
            else -> Unit
        }
    }
}

class CircuitOpenException(message: String) : RuntimeException(message)
```

## Phase 1 vs Phase 2 경계

| 항목 | Phase 1 (현재) | Phase 2 (미래) |
|------|--------------|---------------|
| 저장소 | in-memory (ConcurrentHashMap) | Redis (분산) |
| 범위 | 단일 인스턴스 | 멀티 인스턴스 |
| 의존성 | 0 | Resilience4j + Redis client |
| 트리거 시점 | 지금 | 2nd API key 추가 or 멀티 인스턴스 배포 |

ADR-007에 이 경계가 명시되어 있어, autoceo가 Phase 2 구현을 자의적으로 시작하지 않는다.

Phase 2에서 2nd API key가 추가되면 moneyflow 패턴을 참조하여 프로바이더별 독립 CB로 확장 예정.

## AI Agent Directive

### Trigger
- Spring Boot + Kotlin 환경에서 외부 API 의존성에 CB 적용 필요 시
- 단일 서버 + 단일 API key 상황에서 조기 복잡성을 피하고 싶을 때

### Prerequisites
- [Multi-Provider Circuit Breaker](/wiki/agents/multi-provider-circuit-breaker) — 프로바이더별 독립 CB 패턴

### Actionable Steps
1. **3상태 머신 구현** — CLOSED/OPEN/HALF_OPEN, AtomicReference로 상태 관리
2. **Sliding Window failure counting** — CopyOnWriteArrayList + 시간 기반 정리
3. **Clock 주입** — 테스트에서 시간 조작 가능하도록 `Clock` 파라미터화
4. **CircuitOpenException** — Controller에서 503 SERVICE_UNAVAILABLE로 변환
5. **ADR에 Phase 1/2 경계 명시** — 분산 CB는 명확한 트리거(2nd key, multi-instance) 있을 때만

### Anti-patterns
- **OPEN → CLOSED 직접 전이** — HALF_OPEN 없이 복구하면 프로바이더 부담 증가
- **Sliding window 없이 누적 카운트** — 오래된 실패가 영구 남아 불필요한 OPEN 발생
- **Phase 1에 Redis 도입** — 단일 인스턴스 환경에서는 과설계
- **스트리밍용 CB 별도 구현** — 스트림 전체를 1 요청으로 보고 기존 `execute { ... }` 래핑

## 스트리밍 확장 — SSE에도 동일 CB 재사용

Aidy Session 6 R2([Journal 005](/wiki/harness-engineering/aidy-journal-005-sse-phase1-to-phase2-progressive-rollout))에서 Anthropic streaming 실연동 시 **새 CB 구현 없이** 기존 인스턴스 재사용:

```kotlin
fun chatStream(..., onToken, onDone, onError) {
    try {
        val aiReply = circuitBreaker.execute {       // ← 기존 CB 그대로
            executeStreaming(...)                     // ← 스트림 전체가 1 요청
        }
        onDone(aiReply)
    } catch (e: Throwable) { onError(e) }
}
```

CB의 단위는 "요청 1건"이다. 스트리밍도 1 요청이므로 전체를 래핑 — OPEN이면 스트림 시작도 전에 `onError`, 스트림 중 `IOException` / `ApiException` 은 failure로 집계. 자세한 구현은 [Spring Boot SSE 0-dep](/wiki/backend-ai/spring-boot-sse-streaming-0-dep) 참조.
