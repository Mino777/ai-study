## 보안 핵심 5가지

### 1. 토큰 — SecureRandom 24 byte → URL-safe Base64 no-padding

```kotlin
companion object {
    private const val TOKEN_BYTES = 24  // 24 byte → Base64 32자, 192bit 엔트로피
}
private val secureRandom = SecureRandom()

private fun generateToken(): String {
    val bytes = ByteArray(TOKEN_BYTES)
    secureRandom.nextBytes(bytes)
    return Base64.getUrlEncoder().withoutPadding().encodeToString(bytes)
}
```

**왜**:
- `SecureRandom` — 암호학적 난수(일반 Random 금지)
- 24 byte = 192bit — UUID v4 (122bit) 보다 강
- URL-safe Base64 = `+/` 대신 `-_` → 이메일 링크/쿼리스트링에 직접 사용
- no-padding = `=` 없음 → URL 인코딩 불필요

DB 컬럼: `VARCHAR(64) NOT NULL UNIQUE` (여유 넉넉히).

### 2. 만료 — 30분 + `expires_at TIMESTAMPTZ`

```kotlin
private const val EXPIRES_MINUTES = 30L

val expiresAt = now.plusSeconds(EXPIRES_MINUTES * 60)
passwordResetTokenRepository.save(
    PasswordResetToken(userId = user.id, token = token, expiresAt = expiresAt)
)
```

**왜 30분**:
- 짧으면: 메일 지연/사용자 늦게 확인 → 불편
- 길면: 토큰 유출 시 공격 윈도우 증가
- 일반 권장 범위 15~60분, 30분이 중간값

`confirmReset` 에서 `now.isAfter(expiresAt)` 체크 → `PASSWORD_RESET_TOKEN_INVALID`.

### 3. 1회용 — `used_at TIMESTAMPTZ NULL`

```kotlin
@Transactional
fun confirmReset(token: String, newPassword: String) {
    val resetToken = passwordResetTokenRepository.findByToken(token)
        ?: throw ApiException(ErrorCode.PASSWORD_RESET_TOKEN_INVALID)

    if (resetToken.usedAt != null)     throw ApiException(ErrorCode.PASSWORD_RESET_TOKEN_INVALID)
    if (now.isAfter(resetToken.expiresAt)) throw ApiException(ErrorCode.PASSWORD_RESET_TOKEN_INVALID)

    // 비밀번호 변경...

    resetToken.usedAt = now   // ← 사용 마킹
    passwordResetTokenRepository.save(resetToken)
}
```

**왜 DELETE가 아니라 used_at**:
- 감사 로그 유지 (언제 reset 됐는지)
- "used token + 만료" 차이를 응답에 노출 안 함 (둘 다 동일 에러 코드)
- 동시성 — `@Transactional` + unique constraint 로 race 방어

### 4. 쿨다운 — 5분, 기존 토큰 유지 방식

```kotlin
private const val COOLDOWN_SECONDS = 5L * 60  // 5분

@Transactional
fun requestReset(email: String) {
    val user = userRepository.findByEmail(email) ?: run {
        // 유출 방지: no-op but 응답은 성공
        log.info("password reset requested for unknown email, ignoring (email={})", email)
        return
    }

    val now = Instant.now()
    val recent = passwordResetTokenRepository.findFirstByUserIdOrderByCreatedAtDesc(user.id)
    if (recent != null &&
        recent.usedAt == null &&
        recent.expiresAt.isAfter(now) &&
        recent.createdAt.isAfter(now.minusSeconds(COOLDOWN_SECONDS))
    ) {
        log.info("password reset cooldown active for {} — keeping existing token", email)
        return   // ← 신규 발급 안 함, 기존 토큰 유지
    }

    // 신규 토큰 발급...
}
```

**왜 '기존 유지'**:
- 사용자 double-click / 재시도에도 첫 이메일의 링크 작동
- 이메일/로그 플러딩 방지
- "기존 토큰 무효화 + 에러" 방식은 사용자가 이메일 클릭해도 실패 → UX 깨짐

**대비: '무효화 + 에러'**

```
User: [요청] → 토큰 A 발급 + 이메일 A
User: [실수로 재요청] → 토큰 A 무효화 + 토큰 B 발급 + 이메일 B
User: 이메일 A의 링크 클릭 → 에러 (이미 무효화)
```

사용자 관점에서 억울한 실패. '기존 유지' 방식은 이메일 A 링크가 계속 작동.

### 5. 사용자 존재 유출 방지 — 존재하지 않는 이메일도 성공 응답

```kotlin
// Controller
@PostMapping("/api/auth/password/reset/request")
fun requestReset(@RequestBody req: ResetRequestRequest): ResponseEntity<Unit> {
    passwordResetService.requestReset(req.email)
    return ResponseEntity.ok().build()   // ← 존재 여부 무관 200
}

// Service (발췌)
val user = userRepository.findByEmail(email) ?: run {
    log.info("password reset requested for unknown email, ignoring")
    return   // ← throw 금지
}
```

**왜**:
- `404 Not Found` 또는 `400 "이메일이 존재하지 않습니다"` 는 enumeration 게이트웨이
- 공격자가 이메일 목록으로 `/request` 쓸어보며 계정 목록 수집 가능
- OWASP 명시 금지: [Forgot Password Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Forgot_Password_Cheat_Sheet.html)

내부 처리는 log로 남기되 외부 응답은 **유효 이메일과 동일**.

## 보조 방어

### Flyway — additive-only 마이그레이션

```sql
-- V11__create_password_reset_tokens.sql
CREATE TABLE password_reset_tokens (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id),
    token VARCHAR(64) NOT NULL UNIQUE,
    expires_at TIMESTAMPTZ NOT NULL,
    used_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_password_reset_tokens_user
    ON password_reset_tokens(user_id);
```

- `UNIQUE` on `token` — DB 레벨 충돌 방어
- `REFERENCES users(id)` — 사용자 삭제 시 FK 정책 (앱 로직에서 CASCADE 여부 결정)
- H2 테스트는 `flyway.enabled=false` + JPA create-drop 우회 (기존 패턴)

### Rate Limit

`RateLimitInterceptor` 에서 `/api/auth/password/reset/request` 버킷 — 기존 auth rpm=10 공유 또는 별도 버킷 결정. Aidy 는 기존 auth 버킷 공유.

### 에러 코드 단일화

```kotlin
enum class ErrorCode {
    PASSWORD_RESET_TOKEN_INVALID,  // 무효/만료/사용됨 전부 동일
    ...
}
```

"이미 사용됨" / "만료됨" / "잘못된 토큰" 을 구별해 응답하면 공격자 정보 수집 — 단일 에러 코드.

## 다음 Phase (SMTP 통합)

Phase 1에서 의도적으로 분리:

```kotlin
// Phase 1
log.info("password reset token for {}: {}", email, token)  // ← 로그만

// Phase 2
emailService.sendResetEmail(user.email, resetLinkWith(token))  // ← 실제 발송
```

SMTP 통합 단계에서:
- 이메일 템플릿 (링크 + 만료 안내 + "본인 아닐 시 무시")
- 발송 실패 retry (SES/SendGrid)
- Bounce/Complaint 처리

**보안 핵심 5가지**는 Phase 1에서 전부 확정 — SMTP 변경이 보안 계층에 영향 없음.

## AI Agent Directive

### Trigger
- Password Reset 기능 구현 착수 시
- 기존 reset 엔드포인트가 `404` / `"존재하지 않음"` 응답하는 것 발견 시
- 토큰을 UUID로 생성하는 레거시 발견 시

### Prerequisites
- [Security Hardening Checklist](/wiki/harness-engineering/security-hardening-checklist)
- [Backend Flow Map](/wiki/backend-ai/backend-flow-map-via-aidy-server)

### Actionable Steps

1. **토큰**: `SecureRandom 24 byte → URL-safe Base64 no-padding` (32자 / 192bit)
2. **만료**: 30분, `expires_at TIMESTAMPTZ`, `confirm` 에서 체크
3. **1회용**: `used_at TIMESTAMPTZ NULL`, DELETE 아님
4. **쿨다운**: 5분 이내 유효 토큰 있으면 신규 발급 안 함 (기존 유지)
5. **유출 방지**: 존재하지 않는 이메일도 200 응답, 내부 log만
6. **에러 코드 단일화**: 무효/만료/사용됨 전부 `PASSWORD_RESET_TOKEN_INVALID`
7. **Rate Limit**: 기존 auth 버킷 공유
8. **Flyway additive-only**: V_n__create_password_reset_tokens.sql

### Anti-patterns

- **토큰 = UUID** → 122bit, 하이픈 포함, URL 인코딩 필요 — 약하고 불편
- **`java.util.Random`** → 예측 가능, 암호학 금지
- **미존재 이메일 404/400** → enumeration 게이트웨이
- **DELETE 사용 토큰** → 감사 로그 손실
- **'무효화 + 에러' 쿨다운** → 사용자 UX 깨짐
- **에러 코드 구별** (`TOKEN_EXPIRED` / `TOKEN_USED` 분리) → 공격자에게 정보 제공
- **Phase 1에 SMTP 통합** → 보안 로직 + 이메일 인프라 변경이 섞임
